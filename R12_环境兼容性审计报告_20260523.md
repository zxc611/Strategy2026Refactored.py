## 五、跨平台多进程审计

### P0-010: ProcessPoolExecutor未设置spawn/fork启动方式，Windows/Linux行为不一致

| 项目 | 详情 |
|:---|:---|
| **文件路径** | `参数池/task_scheduler.py`, `参数池/preprocess_ticks.py`, `参数池/l2_optimizer.py`, `参数池/delay_time_sharpe_3d.py` |
| **行号范围** | task_scheduler.py: L5644; preprocess_ticks.py: L847; l2_optimizer.py: L326; delay_time_sharpe_3d.py: 使用ThreadPoolExecutor替代（但L19导入ProcessPoolExecutor） |
| **问题描述** | 所有使用 `ProcessPoolExecutor` 的文件均未调用 `multiprocessing.set_start_method()` 指定启动方式。Windows默认使用spawn模式（需pickle数据和函数），Linux默认使用fork模式（共享内存）。这导致：(1) 在Windows上，所有worker函数和传递的数据必须可pickle。task_scheduler.py的`_worker_init`传递了pd.DataFrame，大数据量pickle可能内存爆炸；(2) `_TRAIN_DATA` 和 `_TEST_DATA` 全局变量在spawn模式下不会被共享，必须通过initargs传递，代码确实使用了initargs，这点是正确的；(3) 如果代码在Linux上开发测试后部署到Windows，或反之，可能遇到完全不同的并发行为。未设置start_method意味着在不同平台上的行为不可预测。 |
| **严重等级** | P0 |
| **影响的系统** | 测试系统（参数优化）、生产系统（数据预处理） |
| **修复建议** | 在 `if __name__ == "__main__":` 块中添加 `multiprocessing.set_start_method('spawn', force=True)` 统一行为，或显式检测平台并记录日志 |

| 项目 | 详情 |
|:---|:---|
| **文件路径** | 全局 |
| **问题描述** | 全量扫描未发现 `ctypes`、`cdll`、`LoadLibrary`、`.dll`、`.so` 等动态库直接加载代码。所有依赖均通过Python包管理（pip）安装，由各自包的wheel提供编译好的扩展模块。 |
| **严重等级** | P2 |
| **影响的系统** | 不适用 |

### P2-021: 隐含DLL依赖未声明

| 项目 | 详情 |
|:---|:---|
| **文件路径** | 全局 |
| **问题描述** | 以下依赖库隐含Windows DLL依赖：(1) **duckdb**: 需要C++运行时库，Windows上通过wheel自带；(2) **numpy/scipy**: 依赖BLAS/LAPACK，Windows wheel自带；(3) **psutil**: Windows上需要特定系统API访问权限。这些依赖在标准Python安装中通常不是问题，但在精简版Python环境（如InfiniTrader内置Python）中可能需要额外安装 Visual C++ Redistributable。代码中未检查这些运行时依赖是否可用。 |
| **严重等级** | P2 |
| **影响的系统** | 全部系统 |

## 补充发现

### P2-022: `if __name__ == "__main__":` 保护不完善

| 项目 | 详情 |
|:---|:---|
| **文件路径** | 多处 |
| **问题描述** | 多个使用ProcessPoolExecutor的文件有 `if __name__ == "__main__":` 保护（task_scheduler.py L6368, preprocess_ticks.py L928等），这符合Windows spawn模式的要求。但部分文件（如 `l2_optimizer.py`）虽使用ProcessPoolExecutor，其 `__main__` 保护在调用方而非本模块内，需确保调用方正确保护。 |
| **严重等级** | P2 |
| **影响的系统** | 测试系统 |

### P2-023: 验证脚本中的中文路径字符串分隔符混用

| 项目 | 详情 |
|:---|:---|
| **文件路径** | `strict_verify_v9.py`, `final_verify.py`, `full_chain_verify_v9.py` |
| **问题描述** | 路径字符串中使用正斜杠 `/`（如 `'参数池/task_scheduler.py'`），在Windows上通常可工作，但非标准做法。Python的os模块在Windows上会自动转换，但某些第三方库（如某些版本的pathlib）可能不处理。 |
| **严重等级** | P2 |
| **影响的系统** | 评判系统 |

## 审计方法说明

1. **全量正则扫描**：对全部120个.py文件执行正则表达式匹配，覆盖路径、编码、进程、环境变量等维度
2. **逐行读取交叉验证**：对匹配到的关键代码段进行逐行读取，确认问题真实存在
3. **调用链追踪**：对发现的问题，沿方法定义→参数传递→实际消费链路进行追踪，确认影响范围
4. **全链路核查**：确保每个问题的描述包含完整的调用链和影响分析

---

**审计结论：** ali2026v3_trading系统在环境兼容性方面存在4个P0级问题（主要集中在跨平台路径、进程启动模式和依赖库导入），这些问题在特定环境下可能导致系统无法启动或评判系统无法运行。建议按优先级逐一修复。
