## 一、收益归因（Performance Attribution）

> **标准要求（手册5.2节）：** PnL必须按Delta/Gamma/Vega/Theta分解至Greek字母贡献，残差>15%触发E7。收益不仅可解释，且必须在实时仪表盘中解剖至Greek字母贡献。

### 发现1-1 [P0] PnL归因缺乏按策略/品种/方向/时间段的分解

- **文件：** `risk_service.py`
- **行号：** 698-778 (`_compute_pnl_attribution`)
- **问题描述：** `_compute_pnl_attribution()`将所有持仓的Greek贡献聚合为单一全局总量（delta_contrib/gamma_contrib/vega_contrib/theta_contrib），未按以下维度拆分：
  - **策略维度：** 无法区分S1高频趋势共振 vs S2分钟趋势共振 vs S3箱体极值 vs S4箱体弹簧各自的PnL贡献
  - **品种维度：** 无法区分IH期权 vs IF期权 vs 商品期权各自的盈亏来源
  - **方向维度：** 无法区分多头 vs 空头各自的盈亏贡献
  - **时间段维度：** 无法区分日内时段（早盘/午盘/尾盘）的归因差异
- **影响系统：** 生产系统 + 评判系统
- **手册依据：** 5.2节明确要求"每个持仓的delta_pnl/gamma_pnl/vega_pnl/theta_pnl"应被分别计算，"残差>15%触发E7"。当前聚合方式无法回答"哪一策略赚钱/哪一策略亏钱"。

### 发现1-2 [P1] PnL归因中使用硬编码乘子近似而非实际价格变化

- **文件：** `risk_service.py`
- **行号：** 721-724
- **问题描述：** Greek贡献计算公式使用了固定的近似乘子：
  ```python
  delta_contrib += greeks.get('delta', 0.0) * multiplier * size * 0.01
  gamma_contrib += 0.5 * greeks.get('gamma', 0.0) * multiplier * size * 0.0001
  vega_contrib += greeks.get('vega', 0.0) * multiplier * size * (iv * 0.01)
  ```
  Delta贡献乘以固定0.01（假设1%标的价格变化），而非实际期货价格变化量。这导致：
  - 在标的价格波动大时，归因严重失真
  - Gamma贡献的0.0001假设价格变化²约等于0.0001，在极端行情下不成立
  - Vega贡献基于IV×0.01而非实际IV变化量
- **影响系统：** 生产系统
- **手册依据：** 5.2节标准实现（行592-619）中`delta_pnl`/`gamma_pnl`/`theta_pnl`/`vega_pnl`是由各持仓的`getattr(p, 'delta_pnl', 0)`等属性提供的，说明应使用实际计算值而非近似值。

### 发现1-3 [P1] 未解释收益(unexplained PnL)计算逻辑错误

- **文件：** `risk_service.py`
- **行号：** 726-729
- **问题描述：** `unexplained`的计算方式存在概念错误：
  ```python
  total_explained = delta_contrib + gamma_contrib + vega_contrib + theta_contrib
  total_pnl = abs(net_delta * 0.01) + abs(net_gamma * 0.0001) + abs(net_vega * 0.01) + abs(net_theta)
  unexplained = total_pnl - abs(total_explained) if total_pnl > 0 else 0.0
  ```
  - `total_pnl`使用绝对值（abs）计算，而非实际的已实现/未实现收益
  - 手册标准的`actual_pnl = sum(getattr(p, 'realized_pnl', 0) for p in positions)`在实际代码中被替换为对Greek分量的绝对值聚合
  - 这导致残差永远无法反映真正的"实际收益 - 解释收益"差异
- **影响系统：** 生产系统（E7告警可能误触发或漏触发）
- **手册依据：** 5.2节标准实现中明确使用`actual_pnl = sum(realized_pnl)`与`explained_pnl = sum(delta_pnl + gamma_pnl + theta_pnl + vega_pnl)`比较。

### 发现1-4 [P1] RiskDashboardService.attribute_pnl()归因维度单一

- **文件：** `risk_service.py`
- **行号：** 2256-2264 (`RiskDashboardService.attribute_pnl`)
- **问题描述：** `attribute_pnl()`仅按`close_reason`维度归因，文档声称"按策略/状态/持仓时长分组归因"，但实际实现：
  ```python
  def attribute_pnl(self, closed_trades: List[Any]) -> Dict[str, float]:
      """PnL归因：按策略/状态/持仓时长分组归因"""
      attr: Dict[str, float] = {}
      for t in closed_trades:
          reason = getattr(t, 'close_reason', 'unknown')
          pnl = getattr(t, 'pnl', 0.0)
          attr[reason] = attr.get(reason, 0.0) + pnl
      return attr
  ```
  实际只有`close_reason`一个维度，与其docstring自述的"按策略/状态/持仓时长"不符。`RiskDashboardService`是独立仪表盘服务（手册P3-2修复），但实际利用率为零。
- **影响系统：** 测试系统（仅定义，未被调用消费）
- **手册依据：** 5.2节仪表盘应包含"组合Greeks聚合+PnL归因+压力测试"。

### 发现1-5 [P2] PnL归因未区分已实现/未实现收益

- **文件：** `risk_service.py`
- **行号：** 698-778 全体
- **问题描述：** `_compute_pnl_attribution()`仅对Greek贡献做计算，不区分：

## 二、风险归因（Risk Attribution）

> **标准要求（手册5.3节）：** 11维度统一风险评分体系中，风险应分层分解（核心信号层/市场状态层/组合风控层），权重总和=1.00，仓位缩放规则由综合评分驱动。

### 发现2-1 [P1] 缺乏因子化风险分解（Factor-based Risk Decomposition）

- **文件：** `risk_service.py`
- **行号：** 1260-1375 (`compute_decision_score`)
- **问题描述：** 11维度统一风险评分体系（D1-D11）是一个"决策辅助评分"系统，而非"风险归因"系统。它告诉系统"是否应该交易"，但不告诉系统"风险来自哪里"：
  - 没有市场风险(market risk)归因 —— 标的价格波动贡献了多少组合风险？
  - 没有风格风险(style risk)归因 —— 大小盘/价值成长等风格因子贡献了多少风险？
  - 没有特质风险(idiosyncratic risk)归因 —— 品种特有风险占比多少？
  - 没有VaR或CVaR的因子分解
- **影响系统：** 生产系统 + 评判系统
- **手册依据：** 5.3.1节三层架构（核心信号0.45+市场状态0.30+组合风控0.25）定义的是评分权重，而非风险分解权重，但手册铁律4要求"收益不仅可解释，且必须在实时仪表盘中解剖至Greek字母贡献"，暗示风险也应可解剖。

### 发现2-2 [P1] 风险贡献与收益贡献之间缺乏一致性校验

- **文件：** `risk_service.py` + `shadow_strategy_engine.py`
- **行号：** risk_service L698-778（PnL归因）、shadow L590-689（Alpha计算）
- **问题描述：** 系统有独立的PnL归因（风险侧）和Alpha计算（收益侧），但两者从未交叉验证：
  - 高Delta贡献的PnL是否与高Delta暴露的风险成正比？
  - Alpha比率高的策略是否其风险调整后收益也一致高？
  - 没有"风险贡献 vs 收益贡献"的一致性散点图或秩相关检验
  - 这导致可能出现"高风险低收益"与"低风险高收益"混杂而未被识别的场景
- **影响系统：** 评判系统
- **手册依据：** 铁律3（无量化来源的参数不得锁定），5.3.4节仓位缩放规则需要基于综合评分，但不应忽略风险/收益一致性。

### 发现2-3 [P1] SPAN保证金模拟器未做风险因子归因

- **文件：** `risk_service.py`
- **行号：** 2311-2374 (`SimplifiedSPAN`)
- **问题描述：** `SimplifiedSPAN`使用16情景风险阵列计算组合保证金，但其输出仅为单一保证金数字，未将风险分解为：
  - 标的波动风险 vs 波动率曲面风险 vs 基差风险
  - 各情景对总保证金的边际贡献
  - 各品种对组合保证金的加法性贡献
- **影响系统：** 生产系统
- **手册依据：** 5.2节压力测试跳空模拟应"集成归因"。

### 发现2-4 [P2] 11维度中D10/D11权重分配无量化依据

- **文件：** `risk_service.py`
- **行号：** 1243-1258 (`RISK_DIMENSION_DEFAULTS`)
- **问题描述：** 11维度的权重分配：
  ```python
  'alpha_decay':         0.10,  # D10
  'cross_correlation':   0.08,  # D11
  ```
  D10（Alpha衰减）和D11（跨策略相关性）的权重(0.10+0.08=0.18)在组合风控层占比72%(0.18/0.25)，但这两个权重来自直觉而非量化研究。手册自身也标记这些参数来源为"直觉"。
- **影响系统：** 生产系统 + 评判系统
- **手册依据：** 手册5.3.2节表格标记多个维度"待量化"，铁律3要求"无量化来源的参数不得锁定为生产值"。

## 三、Alpha/Beta分离

> **标准要求（手册铁律5 + 19.4节）：** 策略的Alpha必须经受影子策略的持续拷问。Alpha比率 = 主策略Sharpe - max(影子A Sharpe, 影子B Sharpe)。市场友善度基准需t检验验证。

### 发现3-1 [P0] Alpha比率计算公式非标准定义

- **文件：** `shadow_strategy_engine.py`
- **行号：** 612-613 (`compute_alpha_metrics`)
- **问题描述：** Alpha比率计算公式：
  ```python
  best_shadow_sharpe = max(shadow_a_sharpe, shadow_b_sharpe)
  alpha_ratio = (master_sharpe - best_shadow_sharpe) / master_max_dd
  ```
  - 公式是 `(Sharpe差) / MaxDrawdown`，而非资产管理行业标准的Alpha定义（通常为 Jensen's Alpha = Rp - [Rf + β(Rm - Rf)]）
  - 除以`master_max_dd`使Alpha比率变成"每单位最大回撤的超额Sharpe"，这混淆了Alpha与风险调整指标
  - 手册5.2节和铁律5中仅表述为"Alpha比率"概念，未精确定义公式，但此公式无法与行业Alpha/Epsilon方法论对接
- **影响系统：** 生产系统 + 评判系统（所有依赖alpha_ratio的决策）
- **手册依据：** 铁律4要求"收益不仅可解释"，当前Alpha定义无法直接与市场基准对标。

### 发现3-2 [P1] 缺乏显式市场Beta估计

- **文件：** `shadow_strategy_engine.py`
- **行号：** 590-689 全体 (`compute_alpha_metrics`)
- **问题描述：** 系统通过影子策略A（反向逻辑）和影子B（随机动作）间接验证Alpha，但从未直接估计：
  - 策略对市场指数的Beta系数（CAPM回归）
  - 策略收益中Beta驱动部分占比
  - Alpha = Total Return - Beta × Market Return 的分解
  - `generate_daily_summary()`中的`alpha_beta_attribution`字段名含"beta"，但内部数据是影子策略对比，非真实Beta计算
- **影响系统：** 评判系统
- **手册依据：** 19.4节市场友善度基准要求"基准收益t统计量"，暗示应区分市场Beta与策略Alpha。

### 发现3-3 [P1] Alpha衰减检测仅基于窗口级比较

- **文件：** `shadow_strategy_engine.py`
- **行号：** 633-645 (`compute_alpha_metrics`)
- **问题描述：** Alpha衰减检测逻辑：
  ```python
  decline_pct = (prev_metrics.alpha_ratio - alpha_ratio) / abs(prev_metrics.alpha_ratio) * 100.0
  if decline_pct > self.ALPHA_DECLINE_THRESHOLD_PCT:
      consecutive_decline = prev_metrics.consecutive_decline_windows + 1
  ```
  仅检测连续窗口级别（configurable window）的下降，缺少：
  - 日内级别的Alpha衰减检测（如开盘Alpha vs 尾盘Alpha）
  - 不同市场状态(regime)下的Alpha稳定性检测
  - Alpha衰减的统计显著性检验（下降是否仅是随机波动？）
- **影响系统：** 生产系统
- **手册依据：** 铁律5要求"Alpha比率连续两窗下降>20%自动降级"，但未要求统计检验。

### 发现3-4 [P2] generate_daily_summary中Beta贡献计算有歧义

- **文件：** `shadow_strategy_engine.py`
- **行号：** 762-834 (`generate_daily_summary`)
- **问题描述：** 日汇总中的`alpha_beta_attribution`字段，其内部的"Beta贡献"实际映射到影子策略的Sharpe贡献，而非市场Beta。这种命名——在摘要中使用了行业术语"Beta"但含义不同——会造成用户误读。
- **影响系统：** 评判系统
- **手册依据：** 手册P1-裂缝48标记"Beta贡献修正"，说明此问题已知但仅修复了计算值，未澄清命名。

### 发现3-5 [P2] Alpha置信区间检验的统计方法局限性

- **文件：** `shadow_strategy_engine.py`
- **行号：** 1159-1223 (`alpha_confidence_overlap_test`)
- **问题描述：** `alpha_confidence_overlap_test()`通过比较主策略Sharpe CI与影子B Sharpe CI的重叠度来判断Alpha是否显著。但：
  - Sharpe比率的CI计算基于渐近正态性假设，在小样本下不可靠（手册21.2节表明确认了CI宽度→处理策略映射中S4弹簧策略仅有50个信号，CI宽度0.576）
  - 未进行Bootstrap重抽样验证，CI可能过于乐观或悲观
  - 手册V7.2补丁（行212-231）定义了`compute_alpha_confidence_interval()`和CI宽度分级处理策略，但实际代码中使用了不同的方法
- **影响系统：** 评判系统
- **手册依据：** V7.2补丁（行212-231）的CI宽度→策略→资金权重映射规则。

## 四、交易成本归因

> **标准要求（手册19.6节 + 4.1节）：** 交易成本需区分显性成本(手续费)和隐性成本(滑点/冲击成本)。滑点需按到期日/波动率动态调节。换月成本需单独建模追踪。

### 发现4-1 [P1] 缺乏系统性交易成本归因框架

- **文件：** `strategy_ecosystem.py` (L957-1022) + `order_service.py` (L1353-1373)
- **行号：** 分散于多个文件
- **问题描述：** 系统在多个位置独立处理交易成本，但不存在统一的"交易成本归因"模块：
  - `strategy_ecosystem.record_strategy_pnl()`接受`commission`和`slippage`参数，但仅做简单减法：`net_pnl = pnl - commission - slippage`
  - `order_service.py`的`SmartOrderSplitter`有`estimated_slippage_bps`（行1372），但仅为估算值，不反映实际成交滑点
  - `order_flow_analyzer.evaluate_execution()`有滑点评估（行345-423），但与PnL归因无连接
  - 这三个独立的成本计算路径互不通信，无法归总
- **影响系统：** 生产系统 + 评判系统
- **手册依据：** 手册行1388-1389标记"bid_ask_spread NaN降级(P0-2)"为阻塞项，说明成本归因对S1/S4策略至关重要。

### 发现4-2 [P1] 未区分显性成本与隐性成本

## 十、审计结论与建议

### 10.1 总评

ali2026v3_trading系统的归因能力呈现**"冰火两重天"**格局：

- **"火"：** Alpha/Beta监控体系通过影子策略+参数独立锁定+期望值底线+治理编码(E7-E13)构建了相当扎实的Alpha验证基础设施。这是系统的核心优势。
- **"冰"：** 从代码层面看，PnL归因（按策略/品种/方向/时间四维分解）、交易成本归因（显/隐分拆）、盈亏曲线分析（水下期/稳定性）、市场基准对标四大能力板块**全部缺失或严重不完整**。

当前系统能回答"策略是否在赚钱"（通过total_pnl），能在一定程度上回答"是否有Alpha"（通过与影子策略对比），但**几乎完全不能回答"策略为什么赚钱/亏钱"**——而这恰恰是本轮审计的核心问题。
