## 一、共享可变状态无保护访问（维度1）

### P0-01：StrategyCoreService._ensure_order_service / _ensure_position_service 双重检查锁缺失

**文件**：`strategy_core_service.py:180-200`  
**严重级别**：P0 — 确定竞态导致服务初始化不一致

```python
# Line 180-189
def _ensure_order_service(self) -> None:
    if self._order_service is not None:   # 第一次检查（无锁）
        return
    try:
        from ali2026v3_trading.order_service import get_order_service
        self._order_service = get_order_service()  # 第二次写入（无锁）
```

**问题描述**：
- `_order_service` 和 `_position_service` 的惰性初始化使用了 **经典双重检查锁反模式**——第一次判断和写入操作之间没有加锁
- `get_order_service()` 内部虽然使用了 `_order_service_lock` 确保全局单例唯一，但 **StrategyCoreService 层面的 `self._order_service` 赋值没有同步保护**
- 如果两个线程（如 on_tick 热路径和 execute_option_trading_cycle）同时首次调用，`self._order_service` 可能在 `get_order_service()` 返回前被读取为 None，导致两个调用都进入初始化路径

**影响场景**：
- 线程A（tick回调线程）调用 `execute_option_trading_cycle` → 进入 `_ensure_order_service()`
- 线程B（调度器线程）调用 `get_health_status()` → 也检查 `self._order_service`
- 两个线程同时读到 `self._order_service is None`，都触发 `get_order_service()` 调用
- 虽然全局单例有锁保护，但 StrategyCoreService 的 `self._order_service` 属性写入不是原子的

**同样的问题也出现在**：`_ensure_hft_engine()`（line 202），`_ensure_snapshot_collector()`（line 216）

### P1-01：StrategyEcosystem._bar_counter 和 _stats 的非原子自增

**文件**：`strategy_ecosystem.py:432, 636, 661, 847, 857-858, 926-928`  
**严重级别**：P1 — 可能竞态导致统计数据不一致

## 二、TOCTOU（检查后使用时差）（维度2）

### P0-02：execute_option_trading_cycle 风控检查与下单之间的 TOCTOU 窗口

**文件**：`strategy_core_service.py:274-346`  
**严重级别**：P0 — 确定竞态可能导致风控绕过后的错误交易

**执行序列**：
```
Time  Step                            Lock Scope
T1    readiness check (is_running)     _lock → read → release
T2    trading_lock acquire              _trading_lock hold
T3    t_type.select_otm_targets()       _trading_lock hold
T4    生态系统互斥检查                  _trading_lock hold
T5    风控检查 (risk_service.check)     _trading_lock hold  ← 通过！
T6    健康检查                          _trading_lock hold
T7    _ensure_order_service()           无锁！
T8    order_service.execute_by_ranking  _trading_lock hold
T9    trading_lock release
```

**问题描述**：
- 风控检查（line 298-312）与订单执行（line 324-328）之间，**虽然 `_trading_lock` 一直持有**，但风控检查的底层数据源（position_service.positions）可能已被其他线程修改
- `check_before_trade()` 检查持仓限额的方式是调用 `RiskService._check_position_limit()`，该函数会读取 `position_service.positions`，但 **`_trading_lock` 不是 `position_service.positions` 的保护锁**
- 风控通过的瞬间另一个线程（如 on_trade 成交回调）可能新开了一个持仓，导致限额检查时的状态已过时

**同样的问题也出现在**：`check_position_risk()`（line 412-425），其中 `_trading_lock.acquire(blocking=False)` 使用非阻塞模式——风控周期可能被**静默跳过**。

### P0-04：execute_option_trading_cycle 多步骤被 cancel_all_pending 交错插入

**文件**：`strategy_core_service.py:274-346` 和 `order_service.py:459-472`

**交错场景**：

```
T1: risk check passed
```

### P1-06：PositionService._add_position 中 positions 字典的多步非原子更新

**文件**：`position_service.py:645-718`  
**严重级别**：P1 — 字典按合约分片锁降低了风险，但同一合约内仍有原子性问题

```python
with self._get_instrument_lock(instrument_id):  # 获取合约级锁
    if instrument_id not in self.positions:
        self.positions[instrument_id] = {}       # 步骤1: 创建空字典
```

### P1-07：PositionService.on_trade 并发修改同一合约持仓的原子性风险

**文件**：`position_service.py:314-375`  
**严重级别**：P1 — 高频成交时同一合约可能被多线程同时修改


## 七、线程间通信安全（维度7）

### 总体评估：线程间通信安全基本良好

### P2-07：ThreadLifecycleManager.stop_all 中迭代修改字典

**文件**：`shared_utils.py:387-403`  
**严重级别**：P2 — 使用 `list(self._threads.items())` 避免迭代修改，但 join 期间其他线程可能注册新线程

```python
with self._lock:                       # 获取锁
    for name, thread in list(self._threads.items()):  # 快照迭代
        ...
        thread.join(timeout=...)       # 阻塞等待！
        # 锁仍持有，阻塞中其他线程无法注册/注销
```

分析：在持有 `_lock` 的情况下执行 `thread.join()`，导致在停止所有线程期间其他线程也无法注册新线程。这是**设计意图**（确保原子停止），但长时间 join 会阻塞其他操作。
