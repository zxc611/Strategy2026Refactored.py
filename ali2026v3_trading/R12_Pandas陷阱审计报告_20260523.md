## 三、索引对齐问题

### P0-01：`_enrich_bars`中`pd.concat`索引对齐风险 ⚠️ P0

**文件路径：** `C:\Users\xu\AppData\Roaming\InfiniTrader_SimulationX64\pyStrategy\demo\ali2026v3_trading\参数池\preprocess_ticks.py`  
**行号范围：** 634-643  
**问题描述：**
```python
def _enrich_bars(df: pd.DataFrame) -> pd.DataFrame:
    if df.empty:
        return df
    state = compute_option_state_vectorized(df)   # 返回新DataFrame，索引默认RangeIndex
    flow = compute_order_flow_vectorized(df)      # 返回新DataFrame，索引默认RangeIndex
    greeks = compute_greeks_vectorized(df)        # 返回新DataFrame，索引默认RangeIndex
    return pd.concat([df.reset_index(drop=True), state, flow, greeks], axis=1)
```
问题：
1. `df`使用`reset_index(drop=True)`重置索引，但`state`/`flow`/`greeks`三个DataFrame的索引是函数内部创建的新RangeIndex。如果`df`的原始索引不是从0开始的连续整数（如经过filter/slice后），`pd.concat`按索引对齐会导致**行错位**——state/flow/greeks的数据与df的OHLCV数据行不匹配。
2. 虽然各函数内部创建DataFrame时使用`np.zeros((n, ...))`或`np.empty(n)`（n=len(df)），索引自然从0开始，但这种正确性**依赖于实现细节而非契约**，未来修改任一函数都可能引入错位。

**全链路验证：**
- 定义端：`compute_option_state_vectorized` L109创建`pd.DataFrame(np.zeros((0, len(base_cols))), columns=base_cols)`，空数据帧也正确
- 消费端：`_process_tick_chunk` L653调用`_enrich_bars`，此时`df`已经过`_filter_bars`的mask+copy+reset_index，索引已从0开始
- **但**：如果上游调用链改变（如直接从外部传入经过slice的DataFrame），索引对齐就会失败

**严重等级：** P0  
**影响系统：** 测试系统（数据预处理流水线）  
**根因：** `pd.concat(axis=1)`依赖索引对齐，但没有显式保证所有参与concat的DataFrame索引一致。  
**建议：** 在concat前对所有DataFrame执行`reset_index(drop=True)`，或使用`ignore_index=True`+`axis=1`的替代方案（如`np.column_stack`+构造新DataFrame）。
