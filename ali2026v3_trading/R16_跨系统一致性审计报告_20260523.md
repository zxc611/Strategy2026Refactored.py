### 3.2 五对齐验证：(b) 算法对齐

#### 【R16-005】 滑点算法：回测 `_compute_dynamic_slippage_bps` vs 生产 `SmartOrderSplitter._estimate_slippage` — 核心算法完全不同 — P0

- **文件路径1（回测）**：`参数池/task_scheduler.py:1130-1158`
- **文件路径2（生产）**：`order_service.py:1432-1444`
- **问题描述**：
  - **回测滑点公式**：`max(base_slippage_bps, bid_ask_spread/price*10000*0.5) * expiry_multiplier`
    - 基于 bid-ask spread 模型
    - 考虑 spread_quality（数据质量）
    - 考虑到期日倍增（3天20x, 7天10x, 14天3x）
    - 使用全局 `SLIPPAGE_BPS`（默认3.0bps）
  - **生产滑点公式**：`avg_slip_ticks * tick_size / mid_price * 10000`
    - 基于订单簿深度和子订单拆分策略估算
    - 计算每个子订单的价格偏离 tick 数取平均
    - tick_size = best_ask - best_bid
    - 与 spread_quality、到期日无关
  - **两个公式产生不同的滑点成本估计**，在极端市场（到期日附近、spread 宽）时差异可能达到 20-60x
- **影响系统**：回测（可能低估/高估滑点导致回测结果不反映实盘表现）、生产（实际交易成本可能远超回测预期）
- **严重等级**：**P0** — 这是回测冠军实战崩盘的根本原因之一：回测的滑点模型（spread * 0.5）显著低于实盘订单簿深度级别的真实滑点

#### 【R16-006】 PnL 计算路径：回测内嵌 vs 生产委托 risk_service — P1

- **文件路径1（回测）**：`参数池/task_scheduler.py` — `_BacktestState` 内嵌 `equity`, `realized_pnl`, `commission` 等字段，直接在回测循环中更新
- **文件路径2（生产）**：`risk_service.py` — `record_trade_result()`, `compute_mode_position_size()` 等
- **问题描述**：
  - 回测 PnL = `(exit_price - entry_price) * lots * multiplier - commission_bps - slippage_bps` — 简化的线性公式
  - 生产 PnL 通过 `risk_service.py` 中的 `record_trade_result()` 记录，且涉及复杂的 Greeks 归因（`E7_PNL_RESIDUAL`）、`compute_mode_position_size` 等
  - 回测的 PnL 归因中没有 Greeks 分解、没有 order split 成本、没有 iceberg 拆分损耗
  - 生产系统通过 `mode_engine.py` 的 `PredictiveStateEngine` 进行方向化仓位调整，回测通过 `_compute_lots_with_risk_budget` 做独立仓位计算
- **影响系统**：回测（PnL 计算过于简化）、评判（基于回测 PnL 做评判 → 评判结论偏离真实）
- **严重等级**：P1 — 回测 PnL 精度低于实盘，导致回测高估收益

#### 【R16-007】 回测仓位计算 `_compute_lots_with_risk_budget` vs 生产 `compute_mode_position_size` — 不一致 — P0

- **文件路径1（回测）**：`参数池/task_scheduler.py:1049-1079`
- **文件路径2（生产）**：`risk_service.py:1167-1199`
- **问题描述**：
  - **回测公式**：`lots = min(lots_min, equity * risk / (price * sl_ratio))`，基于静态 risk 比率和 sl_ratio，含连亏缩减
  - **生产公式**：通过 `ModeEngine.compute_position_size(equity, entry_price, stop_price, win_rate, win_loss_ratio)` 加上 `PredictiveStateEngine` 的方向化乘数，还接收 TVF 六维因子（sortino/calmar/sharpe/ofi/cvd/smart_money_flow/delta/gamma/theta/vega）
  - 回测完全不使用 TVF 六维因子，也不使用 ModeEngine
- **影响系统**：回测、生产
- **严重等级**：**P0** — 仓位计算是交易的核心，两个系统使用完全不同的算法和输入，回测仓位与实盘仓位必然不一致

### 3.3 五对齐验证：(d) 流程对齐

#### 【R16-008】 回测 `_try_open` 中的生产风控调用被 try/except-pass 包裹 — P1

- **文件路径**：`参数池/task_scheduler.py:1434-1498`
- **问题描述**：
  - 回测 `_try_open` 调用 4 个生产 RiskService 方法，全部包裹在 `try/except Exception: pass` 中：
    1. `_rs.check_before_trade(_signal)` (1434-1444行) → 静默吞异常
    2. `_rs.check_regulatory_compliance(_position_data)` (1446-1455行) → 静默吞异常
    3. `_rs.check_capital_sufficiency(bt.equity, _required_margin, bt.open_positions)` (1457-1466行) → 静默吞异常
    4. `_rs.check_exchange_status()` (1469-1476行) → 静默吞异常
  - 如果 RiskService 模块加载失败（ImportError）或调用异常，回测会**静默跳过所有生产风控**继续开仓
  - 此外，3次重复 `from ali2026v3_trading.risk_service import get_risk_service` 导入同一模块
- **影响系统**：回测（绕过生产风控的快速路径）
- **严重等级**：P1 — 在回测环境 RiskService 不可用时，所有生产风控形同虚设，回测结果无风控约束

#### 【R16-009】 回测开仓流程中有生产没有的独立检查 — P2

- **文件路径**：`参数池/task_scheduler.py:1478-1498`
- **问题描述**：
  - 回测独有的检查，生产系统中不存在等价检查：
    1. **spread_quality 阻断**（1478-1489行）：`_spread_quality == 0` 时阻断开仓
    2. **绝对 EV 暂停检查**（1490-1498行）：通过 `shadow_strategy_engine.is_absolute_ev_paused()` 阻断
    3. **信号冷却**（1499-1504行）：`signal_cooldown_sec` 冷却检查
  - 生产系统通过 `RiskService.check_before_trade` 的 `_check_rate_limit` 实现信号冷却，但 spread_quality 和 EV 暂停检查在生产中没有等价物
- **影响系统**：回测（比生产多了一些保护 → 回测可能在更多场景下阻断交易，导致回测偏保守）
- **严重等级**：P2 — 回测独有保护导致回测偏保守，但这不是有害偏差（保守优于激进）

### 3.4 评判系统与回测系统代码路径一致性

#### 【R16-010】 评判系统经过 `adapt_backtest_result` 适配层，字段推导规则与回测直接输出存在偏差 — P1

- **文件路径**：`evaluation/cascade_judge.py:109-184`
- **问题描述**：
  - `adapt_backtest_result()` 函数将回测 dict 转换为 `BacktestMetrics`，存在以下推导偏差：
    1. **sortino_ratio**：当回测返回中无此字段时，使用 `min(sharpe * 1.2, sharpe + 0.3)` 估算 — 这是硬编码近似
    2. **peak_margin_used**：当回测返回中无此字段时，使用 `max_risk_ratio * 0.85` 估算 — 硬编码估算
    3. **max_flat_period_days**：当无 NAV 曲线时，默认填 999 — 最坏情况假设
  - 这些推导规则产生的值可能与实盘监控系统记录的值不同
- **影响系统**：评判（排序/诺/卡玛/保证金峰值的近似值引入误差）
- **严重等级**：P1 — 评判系统依赖的这些推导值如果系统性偏高，会导致本应被淘汰的策略通过评判

#### 【R16-011】 评判系统使用自己的门控阈值配置（cascade_threshold_grid.yaml），回测系统使用 P0_IRON_RULES — P2

- **文件路径1**：`evaluation/cascade_judge.py:223` — `self.thresholds` 从 `cascade_config.yaml` 读取
- **文件路径2**：`参数池/task_scheduler.py:705-724` — `P0_IRON_RULES` 硬编码
- **问题描述**：
  - 评判系统阈值来自外部 YAML 配置文件（动态可配）

  - 当前检查 `cascade_config.yaml` 中的阈值（`data_quality.min_trades: 30`）与 `P0_IRON_RULES.min_signal_count: 30` 一致 — **当前对齐但非强制**
- **影响系统**：评判（配置热更新不保证阈值与铁律同步）
- **严重等级**：P2 — 当前一致但缺乏强制同步机制

### 3.5 生产系统特化逻辑在回测中是否缺失

#### 【R16-012】 生产系统延迟/网络超时模拟在回测中仅在压力测试模式存在 — P0

- **文件路径1（回测压力测试）**：`参数池/task_scheduler.py:2997-3029` — `delay_skip_lambda` 用 Poisson 分布模拟微秒延迟跳 tick
- **文件路径2（回测主循环）**：主回测循环（`_worker_task`）**无延迟模拟**
- **问题描述**：
  - 回测主循环 `run_backtest` → `_worker_task` 直接遍历 bar_data，无任何网络延迟/时间衰减模拟
  - 延迟模拟仅存在于专门的 `_validate_latency_sensitivity` 压力测试函数中
  - 生产系统中，每笔交易都经过真实的网络 RTT、交易所撮合延迟，`ProductionQuantSystem.update_tick` 中记录 `time.perf_counter()` 但仅用于健康检查，不影响交易逻辑
  - 回测主循环 = 零延迟，生产 = 毫秒级延迟
  - 对 HFT 策略（S1）影响致命：回测假设信号到达零延迟，实盘 HFT 延迟 > 10ms 可能完全错过交易窗口
- **影响系统**：回测（大幅高估 HFT 策略表现）、评判（基于高估的回测结果评判，HFT 策略评分虚高）
- **严重等级**：**P0** — HFT 策略回测冠军实战崩盘的根本原因：回测无延迟 = 完美成交，实盘有延迟 = 滑成狗

#### 【R16-013】 生产系统的 SmartOrderSplitter/IcebergSplitter/TWAPSplitter 在回测中完全没有 — P0

- **文件路径1（生产）**：`order_service.py:1307-1489` — SmartOrderSplitter, IcebergOrderSplitter, TWAPSplitter 三个订单拆分器
- **文件路径2（回测）**：`参数池/task_scheduler.py` — 回测中无任何订单拆分逻辑，所有订单视为单笔全部成交
- **问题描述**：
  - 生产系统在开仓时通过 `SmartOrderSplitter.plan_order_split()` 将大单拆为子订单，产生实际的拆分滑点成本
  - 回测将所有交易视为单笔市价全量成交，无拆分损耗
  - 大单在回测中无冲击成本，在实盘中 IcebergOrderSplitter 会产生额外时间成本和滑点
  - TWAP 时间加权拆分在回测中完全不可见
- **影响系统**：回测（大单交易成本被严重低估）
- **严重等级**：**P0** — 回测中大资金仓位（S6做市/S5套利）的成交模型过于理想化

#### 【R16-014】 回测有换月成本建模，生产系统中换月成本由 exchange 真实发生 — P2

- **文件路径1（回测）**：`参数池/task_scheduler.py:2739-2774` — `_compute_rollover_cost` 建模 calendar basis + rollover slippage
- **文件路径2（生产）**：`risk_service.py:2058` — `record_rollover_cost()` 记录实际换月成本
- **问题描述**：
  - 回测的换月成本是估算值（calendar_basis + slippage * n_rollovers * 2）
  - 生产的换月成本是真实发生的，由交易所实际结算
  - 这种差异是合理的（回测必须建模），但需要注意参数对齐
- **影响系统**：回测（换月成本可能高估或低估）
- **严重等级**：P2 — 合理差异，需确保参数定期校准

#### 【R16-015】 生产系统的 `_check_consecutive_loss_protection` 在回测中无等价机制 — P1

- **文件路径1（生产）**：`risk_service.py:1086-1108` — 连续亏损暂停机制
- **文件路径2（回测）**：`参数池/task_scheduler.py:1071-1078` — `_compute_lots_with_risk_budget` 结尾有连亏减仓逻辑

### 3.6 三系统状态机一致性

#### 【R16-019】 评判系统 sortino/calmar/sharpe 硬阈值 vs 生产无等价阈值 — P2

### 4.3 架构级结论

1. **三系统本质上是三个独立实现**，而不是在共享核心引擎上运行：回测有自己完整的 PnL/仓位/滑点/风控逻辑，生产有另一套，评判对回测结果做后处理
2. **最大断裂点**是滑点+延迟+订单拆分：回测的完美成交假设（零延迟、简单滑点、无拆分）与实盘的复杂成交模型形成系统性差异
3. **评判系统的问题**：它基于回测结果做评判，但回测结果本身不可反映实盘表现 → 评判结论的系统性偏差
4. **文档滞后**：手册 V7.0 中 REASON_MULTIPLIERS 和 _resolve_time_stop 的代码示例与当前实际代码已不匹配
