## 维度1：原子性缺失

### P0-01: send_order() 异常路径未清理订单残留状态

- **文件**: `order_service.py`，行257–261
- **操作序列**:
  1. `send_order()` 生成order_id并设置状态为 `SUBMITTED`
  2. 将订单插入 `self._orders_by_id` 字典（即已"锁定资金"）
  3. 调用 `_execute_send_order()` 实际发送至交易平台
  4. 外层 try/except 捕获异常后仅累加 `failed_orders` 统计计数器，**未从 `_orders_by_id` 中移除该订单**
- **原子性缺口**: 订单在 Step 2 已占用字典条目（含 SUBMITTED 状态），Step 4 异常后未清理。`_check_self_trade()` 依赖 `_orders_by_id` 字典进行自成交检测，残留的 SUBMITTED 订单将被视为活跃订单，阻塞后续合法订单。
- **并发场景后果**: 若在极端行情下连续触发多个信号，前一个异常残留的订单会阻止后续订单通过自成交检查 → **资金被"影子锁定"，实际无持仓但系统认为有订单在执行中**。
- **风险等级**: **P0** — 原子性缺失导致自成交误判，阻塞合法交易。

### P1-01: send_order() 字典插入与平台发送不同步

- **文件**: `order_service.py`，行195–210 / 行220–245
- **操作序列**:
  1. 生成 order_id，插入 `_orders_by_id[order_id]`（状态 SUBMITTED）
  2. 调用平台API发送订单
  3. `_execute_send_order()` 成功则更新状态为 `WORKING`，失败则设为 `FAILED`
- **原子性缺口**: Step 1 到 Step 2 之间存在窗口期。若系统在此窗口期崩溃（进程OOM、断电），订单在 `_orders_by_id` 中永久为 SUBMITTED，但平台实际未收到。系统重启后无法区分"已发送但状态未知"与"从未发送"。
- **并发场景后果**: 重复下单 → 系统认为已有挂单，不会对新信号下单 → 实际无挂单 → **信号丢失，资金闲置**。
- **风险等级**: **P1** — 崩溃恢复后需手动清理残留订单，不会自动导致资金损失但会中断交易。

### P1-02: ModeEngine._rollback() 各组件回滚失败互不阻断

- **文件**: `mode_engine.py`，行970–1020
- **操作序列**:
  1. `switch_mode()` 按顺序对组件执行 `set_config()`（例如 PositionService → RiskService → OrderService）
  2. 若某组件失败，调用 `_rollback()` 对已成功传播的组件逐一撤销
  3. `_rollback()` 中每个组件的回滚失败仅 catch + log，继续执行下一个回滚
- **原子性缺口**: 回滚路径中，组件A回滚成功、组件B回滚失败 → 系统进入混合状态（A使用旧参数，B使用新参数）。没有整体"回滚成功/失败"的原子性判定。
- **并发场景后果**: 部分组件持有新参数、部分持有旧参数 → 风控阈值与下单参数不匹配 → **可能允许超标下单**。
- **风险等级**: **P1** — 部分回滚导致参数不匹配，需要人工介入修复。

## 维度2：操作顺序依赖（TOCTOU）

### P0-02: 风控检查通过后到实际下单之间的 TOCTOU 窗口

- **文件**: `risk_service.py` → `order_service.py` 调用链
- **操作序列**:
  1. `RiskService.check_before_trade()` 在 `_lock` 下完成全部检查（安全层、影子引擎、连亏保护、策略状态、信号有效性、频率限制、持仓上限、风险比率、希腊值、生命周期预期）
  2. 释放锁，返回 True
  3. 调用方（`strategy_core_service` 或 `position_service._trigger_close_position`）收到 True 后调用 `OrderService.send_order()`
  4. `send_order()` 再次执行 `_check_self_trade()` 内部检查
- **原子性缺口**: Step 1 的锁在 Step 2 释放，Step 3–4 期间风控状态可能已变化（另一线程触发连亏、另一合约触发总持仓超限等）。Step 4 的 `_check_self_trade()` 只检查自成交，不重新检查风控条件。
- **并发场景后果**:
  - T0: 风控检查通过（连亏计数=2，阈值=3）
  - T1: 另一合约触发亏损，连亏计数变为3
  - T2: 下单执行 → **越过了风控阈值，但订单已发出**
- **风险等级**: **P0** — 风控绕过导致超额风险暴露，可能造成资金损失。

### P1-03: signal_service 信号记录先于验证完成

- **文件**: `signal_service.py`，行265–320（`generate_signal` 方法）
- **操作序列**:
  1. 调用 `_build_signal()` 构建信号对象
  2. 调用 `_record_signal()` 将信号写入历史记录（`self._signal_history`）
  3. 调用 `_check_cooldown()` 进行冷却期验证
  4. 若冷却期内，返回 None
- **原子性缺口**: 信号在第2步已写入历史，第4步才判定无效。无效信号留存在 `_signal_history` 中，污染统计指标（信号计数、命中率）。
- **并发场景后果**: 
  - 信号历史中存在被标记为"已生成但被冷却期拦截"的信号，但调用方无法区分 → 统计数据偏大 → **影响决策质量**。
- **风险等级**: **P1** — 信号统计失真，影响策略评估。

### P2-01: _tick_dispatch_layer 运行状态检查非原子

- **文件**: `strategy_tick_handler.py`，行221–245
- **操作序列**:
  1. 读取 `_is_running`（无锁）
  2. 读取 `_is_paused`（无锁）
  3. 两者均通过后调用 `_process_tick(tick)`
- **原子性缺口**: Step 1/2 的读取与 Step 3 之间，`_is_running` 可能被设为 False（关闭/降级触发）。但 PIC 层面已有 Fuse 熔断保护，此处的漏洞仅影响"降级模式下多处理了一个 tick"。
- **并发场景后果**: 降级模式启动后可能额外处理 1–2 个 tick → **影响极小**（PIC旁路已拦截）。
- **风险等级**: **P2** — 实际风险由上游熔断机制覆盖，此处非关键路径。

## 维度3：事务回滚完整性

### P1-04: send_order() 异常回溯路径不完整

- **文件**: `order_service.py`，行257–261
- **问题详情**（与 P0-01 同源）:
  - `send_order()` 外层 try/except 捕获所有异常后：
    - ✅ 自增 `self.failed_orders` 统计
    - ❌ 未调用 `self._cancel_active_order(order_id)` 或类似清理
    - ❌ 未从 `self._orders_by_id` 中移除残留条目
    - ❌ 未恢复 `_order_count` 计数
  - 内层 `_execute_send_order()` 的 except 同样仅 mark FAILED，不做字典清理
- **回滚完整性评估**: try 中修改了 `_orders_by_id`、`_order_count`、`_orders_filled` 共3处状态，except 中仅恢复了 `failed_orders`，**回滚覆盖率为 0/3**（`failed_orders` 是正向累加，不算回滚）。
- **风险等级**: **P1** — 回滚路径缺失，系统残留垃圾状态。

### P1-05: storage_core cleanup() 统计查询与删除非原子

- **文件**: `storage_core.py`，行1630–1670（cleanup方法）
- **操作序列**:
  1. `SELECT COUNT(*) FROM ticks WHERE timestamp < cutoff` — 统计待清理行数
  2. `BEGIN TRANSACTION`
  3. `DELETE FROM ticks WHERE timestamp < cutoff` — 执行删除
  4. `COMMIT`
- **原子性缺口**: Step 1（SELECT）不在事务内。Step 1→Step 2 之间若有新数据插入（时间戳 < cutoff 的延迟到达数据），Step 1 的 COUNT 与 Step 3 实际删除的行数不匹配。统计日志报告（"已清理 N 行"）与实际删除行数有偏差。
- **并发场景后果**:
  - 极端情况：COUNT=10000 但 DELETE 了 10050 行（延迟插入的50条也被删） → 日志信息不准确，不影响数据完整性（DELETE 基于 WHERE 子句，事务保证一致性）。
- **风险等级**: **P1** — 日志统计失真但数据完整性未被破坏（SQL事务保证了 DELETE 的原子性）。

## 维度4：两阶段操作风险

### P0-03: 下单两阶段——字典登记与实际发送之间的窗口

- **文件**: `order_service.py`，行195–245
- **操作序列**:
  1. **Phase 1（锁仓）**: 生成order_id → 创建Order对象（状态SUBMITTED）→ 插入 `_orders_by_id` 字典
  2. **Phase 2（发送）**: 调用 `_execute_send_order()` → 调用平台API → 更新状态为 WORKING 或 FAILED
- **窗口期风险**:
  - Window size: Phase 1→Phase 2 之间包含平台API网络调用（100–500ms）
  - 窗口期内：订单在本地字典中已存在（SUBMITTED），但平台未收到
  - 窗口期崩溃：本地恢复后 SUBMITTED 订单无法与平台对账
- **并发场景后果**:
  - 极端行情下：Phase 1 完成 → 立即触发止盈信号 → `_check_self_trade()` 发现 SUBMITTED 订单 → 拒绝止盈下单 → **止盈延迟或错过**
  - 系统崩溃恢复后：SUBMITTED 订单无法自动与平台对账，需人工确认
- **对比行业最佳实践**: 应将 Phase 1 的"登记"操作放在 Phase 2 的平台确认后，或使用 WAL（预写日志）实现两阶段提交的恢复机制。
- **风险等级**: **P0** — 资金安全窗口期+崩溃恢复不确定性 → 可能导致资金损失。

### P1-06: ModeEngine.switch_mode() 组件传播与配置标记不同步

- **文件**: `mode_engine.py`，行700–970
- **操作序列**:
  1. **Phase 1（传播）**: 使用 `_propagation_lock`，依次对组件执行 `_apply_single_component_config()`，若成功追加到 `propagated` 列表
  2. **Phase 2（标记）**: 所有组件传播成功后，更新 `self._config` 为新的 ModeConfig
- **窗口期风险**:
  - 若 Phase 1 部分完成（已传播3个组件，还剩2个）时，系统收到新 tick：
    - 前3个组件使用新模式参数
    - 后2个组件仍使用旧模式参数
    - `self._config` 还是旧值（Phase 2 未执行）
    - **系统进入混合模式窗口，直到 Phase 1 完成或 _rollback() 执行**
- **并发场景后果**:
  - RiskService 使用新模式（更严格的风控），但 OrderService 还使用旧模式 → 风控拒绝但下单照常执行 → **风控失效**。
  - 若 Phase 1 中途异常触发 _rollback()，回滚过程同样存在窗口（P1-02）。
- **风险等级**: **P1** — 窗口期内存在参数不匹配，需手动修复。

## 维度5：数据库事务边界

### P1-07: storage_core cleanup() SELECT 与 DELETE 分离事务

- **文件**: `storage_core.py`，行1630–1670
- **问题**: 统计查询（SELECT COUNT(*)）在 DELETE 事务之外执行。虽然 DELETE 本身在事务内且正确 COMMIT/ROLLBACK，但统计报告与实际操作之间存在 TOCTOU 窗口。
- **后果**: 日志中"已清理 N 行"与实际删除行数可能不匹配。审计日志不可信。
- **风险等级**: **P1** — 数据完整性无影响，但运维可观测性受损。

### P2-02: 多表写入不在同一事务中

- **文件**: `strategy_tick_handler.py`，行715–732
- **操作序列**:
  1. `self.storage.process_tick(tick_item)` — 写入 ticks 表
  2. 同一 tick 后续可能触发 strategy_core 写入 orders 表、position_service 写入 positions 表
- **原子性缺口**: tick 持久化、订单生成、持仓更新发生在不同调用栈层级，使用不同事务。无法保证"一个 tick 触发的一整套操作"整体原子的持久化。
- **后果**: 极端情况下 ticks 表有记录但 orders 表无对应记录 → 对账不一致。
- **风险等级**: **P2** — 对账不一致可事后修复，不影响实时交易。

### P2-03: ds_data_writer 单次写入正确使用事务

- **文件**: `ds_data_writer.py`，行230–310
- **正面评价**: `batch_insert_ticks()` 和 `batch_insert_from_cache()` 均使用 `BEGIN TRANSACTION → INSERT → COMMIT` 模式，异常时执行 `ROLLBACK`。事务边界正确，数据一致性有保障。
- **风险等级**: **P2（正面记录）** — 无需修复，作为合规范例记录。

## 维度6：缓存与源数据同步顺序

### P2-04: config_params 参数缓存热更新：先失效缓存后重新加载的窗口

- **文件**: `config_params.py`，行245–283
- **操作序列**:
  1. `reset_param_cache()` 将 `_param_table_cache` 设为 None（失效缓存）
  2. 下次 `get_cached_params()` 调用时重新加载
- **原子性缺口**: Step 1 与 Step 2 之间，缓存为 None。若此时有代码路径调用 `get_cached_params()`，会触发重新加载。TTL 控制（300s）减少了并发读取冲突，但未消除。
- **并发场景后果**: 热重载期间首个 `get_cached_params()` 调用触发文件 I/O 加载，耗时约 50–200ms。对实时交易路径影响：若 tick handler 需要读参数，可能在 50–200ms 内使用旧缓存或等待加载完成 → 延迟微小但存在。
- **风险等级**: **P2** — 延迟可控，不影响正确性。

### P2-05: _dispatch_tick 缓存/持久化双写顺序分析

- **文件**: `strategy_tick_handler.py`，行646–660
- **操作序列**:
  1. RealTimeCache.update_tick() — 更新内存缓存（先写缓存）
  2. storage.process_tick() — 写入数据库（再写源数据）
- **评估**: 先写缓存后写 DB ← 语义上近似 Write-Through 模式（缓存先更新，DB 异步确认）。若 Step 2 失败，缓存与 DB 不一致（缓存有最新价格，DB 缺失该 tick）。但由于 tick 数据是追加型（非覆盖型），不一致不会导致错误决策（后续 tick 会覆盖缓存）。此顺序在追加型数据的场景下是可接受的。
- **风险等级**: **P2** — 追加型数据允许短暂不一致，无实质性风险。

## 维度7：启动与关闭操作顺序

### P1-08: ProductionQuantSystem.shutdown() 未先平仓

- **文件**: `ProductionQuantSystem.py`，行300–380
- **操作序列**:
  1. 设置 `_initialized = False`
  2. 停止 hot_config 观察线程
  3. 关闭持久化层
  4. 停止 EM 线程
  5. （无持仓检查，无平仓逻辑）
- **原子性缺口**: 关闭系统时不检查是否有未平仓合约。若用户计划外关闭（手动停止、进程被杀），未平仓的仓位将"遗弃"在交易所。恢复后系统可能不知道这些仓位（依赖于从交易所同步持仓的能力）。
- **并发场景后果**:
  - 场景A：系统持有 2 手多头 → 手动关闭系统 → 2 手裸仓留在市场 → 价格下跌 → **账户亏损且无止损保护**
  - 场景B：`SIGTERM` 信号触发 shutdown → 未平仓仓位未处理 → 重新启动后需要依赖交易所持仓同步
- **风险等级**: **P1** — 缺少优雅关闭时的平仓策略，可能导致不受保护的裸仓。

### P1-09: 策略初始化顺序——订阅先启动还是风控先就绪

- **文件**: `ProductionQuantSystem.py`，行100–250（初始化流程）
- **操作序列**:
  1. 创建 event_bus
  2. 初始化 signal_service
  3. 初始化 storage
  4. 初始化 strategy_tick_handler（含 subscription）
  5. 初始化 risk_service
  6. 启动 risk_service 监控
- **原子性缺口**: Step 4（subscription开始接收 tick）先于 Step 6（risk_service 监控启动）。若 Step 4 与 Step 6 之间有 tick 流入，风控尚未就绪，该 tick 可能绕过风控直接进入下单流程。
- **并发场景后果**:
  - 理论上，Step 4 到 Step 6 的时间窗口约 1–3 秒。若此窗口内有 tick 且触发信号 → **风控未就绪 → 下单未经过风控检查**。
  - 实际中，strategy_tick_handler 在 `_process_tick()` 中会调用 `risk_service.check_risk()`，若 risk_service 尚未初始化会抛异常 → 下单被中断（fail-safe），而非绕过。
- **风险等级**: **P1** — 初始化顺序存在时序漏洞，但有隐含的 fail-safe 保护。

### P2-06: 关闭时先停订阅还是先停内部处理的顺序

- **文件**: `ProductionQuantSystem.py`，行300–380
- **操作序列**:
  1. 设置 flags（`_initialized = False`）
  2. 停止持久化（storage close）
  3. 停止 EM 线程
  4. （无明确的"停止订阅"步骤，依赖父框架管理订阅生命周期）
- **分析**: 关闭时未明确"先停止订阅再停止内部处理"的顺序。`_initialized=False` 作为软开关被 `_tick_dispatch_layer` 检查，但订阅仍在推送 tick → tick 被 dispatch 但无法持久化 → 数据丢失。应在关闭持久化之前先停止订阅。
- **风险等级**: **P2** — 关闭期间少量 tick 未持久化，可接受的数据损失范围。

## 维度8：并发操作排序

### P1-10: Tick 处理流水线的 happens-before 顺序不完整

- **文件**: `strategy_tick_handler.py`，行680–732 → 行735+（策略 dispatch）
- **操作序列**（`_dispatch_tick` 内部）:
  1. Storage 持久化（shard buffer → process_tick）
  2. PositionService.on_tick() — 价格更新 + 止盈止损检测
  3. HFT 引擎 dispatch
  4. 信号过滤 dispatch
- **Happens-before 分析**:
  - Step 1 happens-before Step 2（在同一线程同一函数调用栈内） ✅
  - Step 2 happens-before Step 3 ✅
  - 但 Step 2 可能触发 `_trigger_close_position()` → 内部调用 `order_svc.send_order()` 在 ThreadPoolExecutor 中异步执行 → 与 Step 3/4 没有 happens-before 保证 → **下单与后续策略处理之间无顺序保证**
- **并发场景后果**:
  - T0: PositionService 检测止损触发 → 提交异步下单任务
  - T1: HFT 引擎 dispatch → 可能在止损订单执行之前又生成一个信号
  - T2: 信号过滤 dispatch → 生成新的开仓信号 → 下单 → **先止损卖出了，紧接着又开仓买入** → 异常操作序列
- **风险等级**: **P1** — 异步下单与同步策略 dispatch 之间无顺序保证，可能导致异常操作链。

### P2-07: 多策略回调对同一 tick 的执行顺序未定义

- **文件**: `strategy_tick_handler.py`，行750–800（策略 dispatch 循环）
- **操作序列**:
  1. 遍历 `self._strategies` 列表
  2. 依次调用 `strategy.on_tick(tick_data)`
- **分析**: 策略列表的顺序在初始化时确定，此后不变。但未显式定义"策略A 必须在策略B之前执行"的语义约束。目前通过列表插入顺序隐式控制 → 缺乏文档化保障 → 未来添加策略时可能破坏隐式顺序。
- **风险等级**: **P2** — 当前运行良好，但缺乏显式排序约束文档。
