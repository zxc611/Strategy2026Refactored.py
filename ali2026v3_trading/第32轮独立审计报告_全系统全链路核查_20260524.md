## 六、新发现的关键风险模式

### 6.1 "日志反噬"模式（日志可靠性维度独有）

```
每条tick→69条logging→每条log执行os.fsync()→磁盘IO阻塞→tick处理延迟→tick积压→错失交易机会
```

日志本应辅助交易决策，但同步磁盘IO的设计让**日志系统反噬了交易系统**。更严重的是，日志系统崩溃完全无感知——日志静默停止后，系统继续运行但已无可观测性。

### 6.2 "GC风暴"模式（性能反模式维度独有）

tick处理路径上每次分配新dict、每次风控new对象、每次信号生成大量临时对象。在高频tick下，GC暂停可达到毫秒级——刚好覆盖一个完整的tick间隔。结果：**系统在GC暂停期间丢失了所有交易机会**。

### 6.3 "个人依赖"模式（环境隐式依赖维度独有）

```
硬编码 C:\Users\xu\... → 其他用户不可用
中文目录"参数池"作为Python模块 → Linux部署import断裂
datetime.now()无时区 → 生产时间偏移8小时
```

系统隐式依赖开发者的个人环境（用户名、编码、操作系统）。这是"开发环境假阳性"的另一个表现——本地可用，但部署到任何其他环境都系统性失败。**测试/评判系统也嵌入了同样的个人路径，意味着这些辅助系统也无法在其他机器上运行。**

### 6.4 "非线性放大的因果链"（数据因果链维度独有）

一个NaN价格可以传播8个节点（缓存→DB→信号→订单→持仓→风控→CRM→HMM），一个错误持仓可污染后续30天所有计算。**输入端1个错误×链路长度N×时间尺度T = N×T个错误输出。** 这是一个典型的非线性放大效应。

更重要的是，**因果链形成有向循环图**（信号→持仓→风控→订单→持仓），使得单点错误可在环内无限循环放大，且没有任何隔离机制。
