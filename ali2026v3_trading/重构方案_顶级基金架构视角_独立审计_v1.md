## D.3 前序审计未识别的架构问题（完整代码阅读的独家发现）

### D.3.1 问题X1：StrategyCoreService的隐式状态爆炸（P0级）

**代码实证**：

```python
# strategy_core_service.py 第104-105行
class StrategyCoreService(_LifecycleMixin, _InstrumentHelperMixin,
                          HistoricalKlineMixin, TickHandlerMixin):
```

**完整阅读后的分析**：

StrategyCoreService表面只有**25个显式方法**，但通过4个Mixin隐式继承了：

| Mixin | 隐式引入的状态 | 隐式方法数（估算） |
|-------|-------------|------------------|
| _LifecycleMixin | _state, _is_running, _is_paused, _scheduler_manager, storage | ~15 |
| _InstrumentHelperMixin | 合约解析缓存 | ~8 |
| HistoricalKlineMixin | K线缓冲区, _historical_data | ~12 |
| TickHandlerMixin | tick队列, _tick_buffer | ~10 |

**总隐式状态**：25个显式方法 + ~45个隐式方法 = **~70个方法共享同一个self命名空间**

**前序审计盲区**：
- 一审到四审均只统计了StrategyCoreService的显式方法（25个），未统计Mixin引入的隐式方法
- 296处self._引用中，大量引用的是Mixin定义的属性（如self._state来自_LifecycleMixin，self.storage来自_LifecycleMixin的@property）
- **这意味着：296处self._中，约60%引用的是Mixin定义的状态，而非StrategyCoreService自身定义的状态**

**架构风险**：
1. **状态来源不透明**：阅读StrategyCoreService代码时，无法直接知道self._state是在哪里定义的（需要追溯到_LifecycleMixin）
2. **修改涟漪效应**：修改_LifecycleMixin的self._state可能影响StrategyCoreService + 所有其他继承_LifecycleMixin的类
3. **测试困难**：测试StrategyCoreService的transition_to方法需要初始化所有Mixin的状态

### D.3.2 问题X2：EventBus的同步阻塞风险（P1级）

**代码实证**：

```python
# event_bus.py 第96-99行
self._executor: ThreadPoolExecutor = ThreadPoolExecutor(
    max_workers=max_workers or self.DEFAULT_MAX_WORKERS,  # DEFAULT_MAX_WORKERS = 10
    thread_name_prefix='event_bus'
)
```

```python
# event_bus.py 第200-250行（publish方法核心逻辑）
def publish(self, event_type: str, event_data: Any, sync: bool = False) -> bool:
    # ...
    if sync:
        # 同步执行
        self._notify_subscribers(event_type, event_data)
    else:
        # 异步提交到线程池
        self._executor.submit(self._notify_subscribers, event_type, event_data)
```

**完整阅读后的分析**：

EventBus的默认行为是**异步发布**（sync=False），但存在以下风险：

1. **线程池耗尽**：10个worker线程，高频tick场景（如1000 ticks/秒）下，如果每个tick触发多个事件（TickEvent + KLineEvent + SignalEvent），线程池可能饱和
2. **背压机制有缺陷**：
   ```python
   # 第105-106行
   self._backpressure_limit: int = backpressure_limit or self.DEFAULT_BACKPRESSURE_LIMIT  # 5000
   self._pending_events: int = 0
   ```
   `_pending_events`计数器在异步模式下**不准确**——executor.submit()立即返回，任务可能还在队列中等待执行，但_pending_events已经增加
3. **同步调用路径未文档化**：strategy_core_service.py中多处直接调用event_bus.publish()，未明确指定sync参数，依赖默认行为

**前序审计盲区**：
- 一审到四审均将EventBus视为"已存在的基础设施"，未分析其性能瓶颈
- 三审提到"EventBus单点故障风险(R1)"，但未深入分析其同步/异步行为

## D.4 前序审计的架构误判修正

### D.4.1 误判M1："db_adapter是空壳"（四审）→ 修正为"db_adapter是薄封装但功能完整"

**四审结论**："db_adapter仅被引用3处，是几乎未使用的空壳"

**六审修正**：
- db_adapter有**17个函数**，覆盖连接、查询、数据管理等功能
- 被**5个文件**引用（非3个）
- **真正问题**：46处DuckDB引用（18 import + 28 API调用）散布在业务代码中，开发者选择绕过db_adapter直接import duckdb

**根因分析**：
```python
# 为什么开发者绕过db_adapter？
# db_adapter提供：connect(), execute(), fetchdf() —— 通用数据库操作
# 但业务代码需要：get_market_data(symbol, start, end), get_positions(), save_risk_state() —— 领域操作

# 开发者被迫在业务代码中直接import duckdb，因为db_adapter没有提供领域级接口
db_adapter.execute("SELECT * FROM market_data WHERE symbol = ?", [symbol])  # 太底层
# vs
duckdb.connect(db_path).execute("SELECT * FROM market_data WHERE symbol = ?", [symbol])  # 开发者实际写的
```

**修正结论**：db_adapter不是"空壳"，而是"**接口层级不匹配**"——它提供了底层数据库操作，但业务代码需要更高层的领域接口。

### D.4.2 误判M2："5个策略核心Mixin"（四审）→ 修正为"4个核心Mixin + UI是独立问题"

**四审结论**：Phase 1处理5个策略核心Mixin（Lifecycle/InstrumentHelper/HistoricalKline/TickHandler/UI）

**六审修正**：

```python
# strategy_core_service.py 第104-105行
class StrategyCoreService(_LifecycleMixin, _InstrumentHelperMixin,
                          HistoricalKlineMixin, TickHandlerMixin):
```

StrategyCoreService**只继承4个Mixin**，UIMixin不在继承列表中。

**UIMixin的实际位置**：
- 通过完整代码阅读发现，UIMixin是**独立类**，未被StrategyCoreService继承
- UIMixin可能属于quant_services.py或独立UI模块

**修正结论**：Phase 1的Mixin解耦范围应为**4个核心Mixin**（Lifecycle/InstrumentHelper/HistoricalKline/TickHandler），UIMixin是独立问题，不应混入Phase 1。

### D.4.3 误判M3："EventBus已存在1584行，基础设施已就绪"（三审/四审）→ 修正为"EventBus有功能缺口"

**前序审计**：将EventBus视为"已存在的基础设施"，建议"基于已有扩展"

**六审发现的功能缺口**：

| 功能 | EventBus当前状态 | 缺口 |
|------|----------------|------|
| 事件发布 | ✓ 同步/异步支持 | — |
| 事件订阅 | ✓ 支持优先级和过滤器 | — |
| 事件持久化 | ✓ 可选持久化 | 但默认关闭 |
| **事件重放** | △ 支持replay_last | 仅支持最近N个，非完整重放 |
| **版本向量** | ✗ 无 | 六审发现的新缺口 |
| **死信队列** | ✗ 无 | 六审发现的新缺口 |
| **发布确认** | ✗ 无 | 六审发现的新缺口 |
| **订阅者健康检查** | ✗ 无 | 六审发现的新缺口 |

**关键缺口分析**：
1. **无版本向量**：事件乱序时无法检测（如TickEvent时间戳 vs SystemEvent时间戳）
2. **无死信队列**：订阅者抛异常时事件丢失（虽然有异常隔离，但失败事件未存储）
3. **无发布确认**：异步publish()返回bool，但无法确认所有订阅者已处理

## D.8 六审最终建议

### 立即执行（本周）

1. **P0.1修正**：扫描46处DuckDB引用（18 import + 28 API调用）时，同时记录**每处调用的领域意图**（如"获取行情数据"、"保存持仓"），为db_adapter的领域级封装提供输入

4. **确认db_adapter的定位**：是"薄封装层"还是"领域数据访问层"？六审建议：保持db_adapter为薄封装，新增`data_access.py`作为领域层
5. **确认UIMixin的处理**：UIMixin不在StrategyCoreService继承链中，应作为独立UI模块处理，不混入Phase 1
