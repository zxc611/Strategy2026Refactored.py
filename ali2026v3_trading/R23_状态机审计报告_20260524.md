## 维度1: 策略生命周期状态机

### 1.1 状态机定义

**文件**: `strategy_lifecycle_mixin.py` (第31-39行) + `strategy_core_service.py` (第64-68行)

```
StrategyState Enum:
  INITIALIZING ──┬──→ RUNNING ──┬──→ PAUSED ──┬──→ RUNNING
                  │              │              ├──→ STOPPED
                  │              │              └──→ ERROR
                  │              ├──→ STOPPED
                  │              ├──→ ERROR
                  │              └──→ DEGRADED ──┬──→ RUNNING
                  │                               ├──→ STOPPED
                  │                               ├──→ ERROR
                  │                               └──→ DEGRADED_STOP ──→ STOPPED
                  │                                                      └──→ INITIALIZING
                  ├──→ ERROR ──→ INITIALIZING
                  │            └──→ STOPPED
                  └──→ DEGRADED
```

### 1.2 VALID_STATE_TRANSITIONS 定义

**文件**: `strategy_lifecycle_mixin.py` (第86-94行)

```python
VALID_STATE_TRANSITIONS = {
    INITIALIZING: [RUNNING, ERROR, DEGRADED],
    RUNNING: [PAUSED, STOPPED, ERROR, DEGRADED],
    PAUSED: [RUNNING, STOPPED, ERROR],
    DEGRADED: [RUNNING, STOPPED, ERROR, DEGRADED_STOP],
    ERROR: [INITIALIZING, STOPPED],
    DEGRADED_STOP: [STOPPED, INITIALIZING],
    STOPPED: [INITIALIZING],
}
```

### 1.3 缺陷清单

#### P0-1: DEGRADED→RUNNING 恢复时 `_is_running` 未同步恢复

- **位置**: `strategy_lifecycle_mixin.py:1246-1260` (resume方法) + `strategy_core_service.py:66`
- **发现**: `resume()` 方法中仅检查 `self._state != StrategyState.PAUSED`，将状态从 PAUSED 转到 RUNNING。但如果策略从 DEGRADED 通过 `transition_to(RUNNING)` 恢复，`_is_running` 标志未被设置回 True（`_is_running` 在 `on_stop` 时被设为 False，见第891行）
- **状态转移图缺失**: DEGRADED→RUNNING 转移存在但无配套 `_is_running=True` 的赋值
- **后果**: 策略状态显示为 RUNNING 但所有 tick 处理被 `_is_running` 检查拦截（`strategy_tick_handler.py:201-205`），导致实盘数据被静默丢弃
- **证明代码**:
  ```
  # strategy_tick_handler.py:200-205
  if not is_running:
      self._e2e_counters['dropped_not_running'] += 1
      return False  # ← _is_running=False 时所有Tick被丢弃
  ```
- **影响**: 降级恢复后策略名义上运行但实际不处理任何数据 → **静默停摆**

#### P1-1: PAUSED→DEGRADED 转移路径缺失

- **位置**: `strategy_lifecycle_mixin.py:89` (VALID_STATE_TRANSITIONS)
- **发现**: PAUSED 状态下合法的转移目标为 [RUNNING, STOPPED, ERROR]，不包含 DEGRADED。但如果策略暂停期间发生外部健康检查异常（如数据源断开），`get_health_status()` 返回 CRITICAL，`onStart()` 中会调用 `transition_to(StrategyState.DEGRADED)`（`strategy_core_service.py:989`），此转移被 `transition_to()` 拦截并返回 False
- **后果**: 暂停中的策略在环境恶化时无法进入降级模式，错误状态被静默吞掉
- **状态转移图缺失**:
  ```
  预期:  PAUSED ─→ DEGRADED  ← 缺失
  实际:  PAUSED ─→ DEGRADED 被 transition_to() 拒绝，状态卡在 PAUSED
  ```

#### P1-2: DEGRADED_STOP 状态无人触发

- **位置**: `strategy_lifecycle_mixin.py:37,91-92`
- **发现**: DEGRADED_STOP 状态定义在枚举中，且 VALID_STATE_TRANSITIONS 定义了 DEGRADED→DEGRADED_STOP→STOPPED 和 DEGRADED_STOP→INITIALIZING，但全局代码搜索未发现任何地方执行 `transition_to(StrategyState.DEGRADED_STOP)`
- **状态转移图**: DEGRADED_STOP 是孤岛状态，无法从任何代码路径到达
- **后果**: DEGRADED→STOPPED 直接跳跃，跳过了 DEGRADED_STOP 中间态设计。如果未来代码意图是在降级时"先安全关闭持仓再停止"，此中间态未被利用

#### P1-3: on_stop() 中 `_is_paused=True` 语义混淆

- **位置**: `strategy_lifecycle_mixin.py:891-892`
- **发现**: `on_stop()` 方法中同时设置 `self._is_running = False` 和 `self._is_paused = True`。但在 PAUSED 状态下 `_is_running` 仍为 True。这里 `_is_paused` 本意只是"数据流暂停"标记，但在 STOPPED 后设为 True 造成语义混淆
- **后果**: 停止后如果有代码检查 `_is_paused`，会错误认为策略"仅暂停而非已停止"

#### P2-1: `_destroyed` 标志无对应 StrategyState 枚举值

- **位置**: `strategy_core_service.py:68` + `strategy_lifecycle_mixin.py:1268-1278`
- **发现**: `_destroyed` 是独立的布尔标志，但 StrategyState 枚举中没有 DESTROYED 状态。`destroy()` 方法设置 `_destroyed=True` 但不修改 `_state`
- **后果**: 销毁后的策略 `_state` 仍保留原值（如 STOPPED 或 RUNNING），状态查询函数返回过时信息

#### P2-2: `_health_pause_new_open` 与 State 不同步

- **位置**: `strategy_core_service.py:70,577-581`
- **发现**: `_health_pause_new_open` 是独立于状态机的布尔标记，用于健康检查异常时暂停新开仓。但此标记不与任何 StrategyState 联动，形成了"影子状态"
- **后果**: 策略状态显示 RUNNING，但实际新开仓已被静默阻止，运维人员无法从状态字符串得知真实情况

## 维度2: 订单状态机

### 2.1 状态定义与使用

**文件**: `order_service.py`

订单状态在系统中以**分散字符串**形式存在，**无枚举定义**：

```
订单状态字段(order['status']):
  'SUBMITTED' (line 231)     — 创建并已提交
  'PENDING'   (line 357)     — 挂单中(用于self-trade检测)
  (隐式 PARTIAL_FILL)        — 通过 filled_volume < volume 推断 (line 532-534)
  'FILLED'    (line 536)     — 完全成交
  'ALL_FILLED' (line 536)    — 完全成交(别名)
  '全成'       (line 536)    — 完全成交(中文别名)
  'CANCELLED' (line 451)     — 已撤销
  'FAILED'    (line 312)     — 发送失败
```

### 2.2 状态转移图（系统预期 vs 实际）

```
预期转移:                         实际转移:
CREATE ─→ SUBMITTED ─→ FILLED     CREATE ─→ SUBMITTED ────→ FILLED
                   └→ CANCELLED                  ├──→ FAILED (再试失败)
                   └→ FAILED                     ├──→ PENDING (仅self-trade检测)
                                                  ├──→ FILLED/ALL_FILLED/全成 (CTP回调直接覆盖)
                                                  └──→ CANCELLED (无部分成交处理)
```

### 2.3 缺陷清单

#### P0-1: CTP回调状态直接覆盖本地状态，无转移合法性校验

- **位置**: `order_service.py:528`
- **发现**: `on_trade_update()` 方法中 `order['status'] = sts` 直接将CTP回调的状态字符串赋值给订单状态，不做任何合法性校验。例如，如果CTP回调 FILLED→CANCELLED（不可能的CTP状态），本地也会接受
- **后果**: CTP异常回调或网络错误导致的状态回跳（如 FILLED→SUBMITTED）会被静默接受，导致重复平仓、重复持仓计算
- **状态转移图缺失**:
  ```
  缺失: 任何状态下→任何状态下的合法性校验矩阵
  仅有: cancel_order() 中对 FILLED/CANCELLED/FAILED 做终态拒绝 (line 440)
  ```

#### P0-2: 部分成交无明确状态标识

- **位置**: `order_service.py:532-534`
- **发现**: 部分成交通过 `filled > 0 and filled < volume` 条件推断，但 `order['status']` 直接设置为 CTP 返回的状态。CTP 对部分成交的 status 返回值因版本而异，可能是 `PARTIAL_FILL`、保持原状态、或空字符串。系统没有统一的部分成交状态
- **后果**: 部分成交后如果CTP不返回明确的部分成交状态字符串，本地订单状态可能仍为 SUBMITTED，导致：
  1. 重复追单（chase reorder）对已部分成交的订单再次追价
  2. `cancel_order()` 不会阻止对部分成交订单的撤单请求
  3. 风控计算使用的 `filled_volume` 可能过期

#### P1-1: FAILED 状态订单无恢复/重试机制

- **位置**: `order_service.py:312,547-575`
- **发现**: `_execute_send_order()` 首次失败时设置 `status='FAILED'`（line 312），然后 `_retry_send_order()` 重试。但如果所有重试都失败（line 334），订单永远留在 `_orders_by_id` 中状态为 FAILED。`check_pending_orders()` 只检查 SUBMITTED/PENDING 状态，不检查 FAILED
- **后果**: FAILED 订单永久占用内存（直到 `check_pending_orders` 的 300 秒过期清理），且永不重试

#### P1-2: 状态字符串不一致 —— `FILLED` vs `ALL_FILLED` vs `全成`

- **位置**: `order_service.py:536`
- **发现**: CTP 成交回调判断 `sts in ('FILLED', 'ALL_FILLED', '全成')` 包含了三种不同的完全成交表示。`全成` 是中文CTP版本返回的状态，但系统中其他位置（如 `cancel_order` line 440）只检查 `FILLED` 而不检查 `ALL_FILLED` 和 `全成`
- **后果**: 如果CTP返回中文 `全成` 状态，`cancel_order()` 会错误地尝试对已成交订单撤单（line 440 检查 `order['status'] in ('FILLED', 'CANCELLED', 'FAILED')` 遗漏了 `全成`）

#### P1-3: PENDING 状态语义不明确

- **位置**: `order_service.py:357`
- **发现**: `PENDING` 状态仅出现在 `_check_self_trade_core()` 的自成交检测逻辑中（line 357-358），但 `send_order()` 创建订单时统一设为 `SUBMITTED`（line 231）。PENDING 的来源不明确——既不是本地创建也不是CTP回调的明确状态
- **后果**: 如果某些CTP版本返回 `PENDING` 作为已提交待确认状态，系统无法区分 "本地已提交" vs "交易所已确认"

#### P2-1: 订单超时处理后追单不重置状态

- **位置**: `order_service.py:567-575`
- **发现**: `check_pending_orders()` 中超时订单先调 `cancel_order()` 再 `_chase_reorder()`。但 `cancel_order()` 失败时（返回 False），`_chase_reorder()` 仍会执行。这导致可能出现同一合约同时存在两个方向的挂单（原始+追单）
- **后果**: 追单与原始订单并存，可能导致重复成交或自成交

## 维度3: 持仓状态机

### 3.1 状态定义

**文件**: `position_service.py`

持仓状态是**隐式的**，通过字段组合表达：

```
PositionRecord 状态标记:
  volume > 0  → OPEN (开仓/持仓中)
  volume = 0  → CLOSED (完全平仓)
  _closing=True → CLOSING (平仓中)
  _closing=False → 非平仓中

  方向: direction='long' | 'short'
```

### 3.2 状态转移图（实际）

```
开仓:  volume=0 → TType信号 → 创建PositionRecord(volume=N) → OPEN
平仓:  OPEN → _closing=True → on_trade(平仓成交) → volume递减
      volume>0 → PARTIAL_CLOSE (隐式)
      volume=0 → CLOSED (隐式)
异常:  无回滚机制
```

### 3.3 缺陷清单

#### P0-1: 平仓异常时无状态回滚

- **位置**: `position_service.py:314-375` (`on_trade`) + line 88 (`_closing`)
- **发现**: `_closing` 标记在平仓信号生成时被设为 True，但在 `on_trade()` 处理平仓成交时，不检查 `_closing` 标志。如果平仓下单失败（网络错误/资金不足），`_closing` 保持 True 且持仓未被平掉，导致持仓永久锁在 "关闭中" 状态
- **状态转移图缺失**:
  ```
  缺失转移: CLOSING(平仓失败) ─→ OPEN(回滚)
  缺失转移: CLOSING(超时无成交) ─→ OPEN(回滚)
  ```
- **后果**: 持仓被标记为 CLOSING 但永远无法平仓，也无法重新发起平仓（因为代码可能检查 `_closing` 跳过重复平仓）

#### P1-1: 已平仓持仓永久保留在 positions 字典中

- **位置**: `position_service.py:397-399` (stop_profit/stop_loss检查)
- **发现**: 当 `volume` 减至 0 时，`PositionRecord` 不从 `self.positions` 字典中移除。`check_all_positions()` 方法遍历所有记录包括 `volume=0` 的
- **后果**: 
  1. 内存泄漏：大量已平仓记录累积
  2. 误判：止盈止损检查可能对 volume=0 的记录产生除零错误
  3. 统计偏差：总持仓数量包含已平仓记录

#### P1-2: 多腿持仓缺少原子性保证

- **位置**: `position_service.py:314-375`
- **发现**: 对于期权策略（如Call-Put组合），`on_trade()` 逐笔处理每条成交，没有 "多腿持仓一致性检查"。如果两腿中一腿成交、另一腿失败，没有整体回滚或告警
- **后果**: 对冲持仓变成单边裸仓，风险敞口失控

#### P2-1: PositionRecord 缺少显式状态字段

- **位置**: `position_service.py:57-91` (`PositionRecord` dataclass)
- **发现**: PositionRecord 没有任何状态字段（如 `status` 或 `state`），状态完全通过 `volume` 值和 `_closing` 布尔标志隐式推断
- **后果**: 不可序列化状态、不利于审计日志、不利于恢复后状态重建

## 维度4: 模式引擎状态机

### 4.1 状态定义

系统中有两个层次的状态概念，但**未对齐**：

**层次A: ModeEngine 的 CapitalMode 枚举** (`mode_engine.py:23`):
```python
class CapitalMode(Enum):
    SHARPE_MAX = auto()
    PROFIT_RATIO = auto()
```

**层次B: ShadowStrategyEngine 的布尔标志** (`shadow_strategy_engine.py`):
```python
_degradation_active: bool = False    # Alpha衰减降级
_absolute_ev_pause: bool = False     # 绝对期望值暂停
```

**层次C: StrategyState.DEGRADED** (`strategy_lifecycle_mixin.py:35-37`):
```python
DEGRADED = "degraded"
DEGRADED_STOP = "degraded_stop"
```

### 4.2 缺陷清单

#### P0-1: 审计标准要求的 Normal→Shadow→Degraded→Emergency 四级模式不存在

- **位置**: 全局
- **发现**: 审计标准文档要求 Normal→Shadow→Degraded→Emergency 四级模式，但系统中 ModeEngine 只有 CapitalMode（SHARPE_MAX/PROFIT_RATIO），这与标准要求的四级模式完全不匹配
- **后果**: 
  1. 无 Shadow（影子）模式 — ShadowStrategyEngine 只有布尔标志而非独立模式
  2. 无 Emergency（紧急）模式 — 熔断仅表现为 `_absolute_ev_pause=True` 无系统级紧急协议
  3. 无法表达"影子观察但不交易"的半自动降级状态

#### P0-2: `is_circuit_breaker_shadow_mode()` 模式无退出条件

- **位置**: `risk_service.py:307-313` + `shadow_strategy_engine.py:691-707`
- **发现**: `safety_meta_layer.is_circuit_breaker_shadow_mode()` 返回 True 时，`risk_service` 阻止所有非平仓操作（"断路器影子模式观察期内仅允许平仓，禁止新开仓"）。但检查代码发现 `clear_degradation()` 和 `clear_absolute_ev_pause()` 存在却无自动触发机制——需要外部代码主动调用
- **后果**: 一旦进入断路器影子模式，除非运维人员手动调用 `clear_*`，系统永久锁定在只平仓状态

#### P1-1: `switch_mode()` 传播失败回滚不完整

- **位置**: `mode_engine.py:748-796`
- **发现**: `switch_mode()` 方法在传播失败时调用 `_rollback()`（line 771），但回滚仅检查注册表一致性，不恢复下游服务的旧配置状态。如果 `_apply_to_risk_service()` 成功但 `_apply_to_signal_service()` 失败，风控服务已经切换但信号服务仍是旧配置
- **后果**: 部分服务处于新模式、部分处于旧模式，形成不一致的并发处理

#### P1-2: 模式卡死检测缺失

- **位置**: 全局
- **发现**: 没有检测模式是否卡死的机制。例如：
  - `switch_mode()` 后如果因异常未完成，没有超时检测
  - `_degradation_active` 设为 True 后，没有定时检查是否应恢复的机制
- **后果**: 系统可能长期运行在错误的模式下

#### P2-1: ModeEngine 无当前模式可观测字段

- **位置**: `mode_engine.py:720-727`
- **发现**: ModeEngine 的属性 `scale_str` 和 `config` 可以推断当前模式，但没有明确的 `current_mode` 字符串字段。`get_health_status()` 也无法查询 ModeEngine 的当前模式
- **后果**: 运维诊断困难

#### P2-2: ShadowEngine 没有模式枚举

- **位置**: `shadow_strategy_engine.py`
- **发现**: ShadowStrategyEngine 的状态通过三个独立布尔标志表达（`_params_locked`, `_degradation_active`, `_absolute_ev_pause`），而非状态枚举。相互之间的优先级和互斥规则无文档约束
- **状态转移图缺失**:
  ```
  Normal ─→ (degradation_active=True) ─→ Degraded ─→ (absolute_ev_pause=True) ─→ Emergency
  Emergency ─→ (clear) ─→ Normal  ← 缺失
  Degraded ─→ (clear) ─→ Normal    ← 存在但无自动触发
  ```

## 维度5: 风控状态机

### 5.1 状态定义

**文件**: `risk_service.py`

```python
RiskCheckResult: PASS | BLOCK | WARNING | DEGRADED    (per-check, 无状态持久化)
RiskLevel: LOW | MEDIUM | HIGH | CRITICAL               (严重程度, 非状态)
```

### 5.2 缺陷清单

#### P0-1: 审计标准要求的 正常→预警→受限→熔断 四级无显式状态机

- **位置**: `risk_service.py:113-120` + 全局
- **发现**: RiskService 的 `check_before_trade()` 每次都返回一个 `RiskCheckResponse`（PASS/BLOCK/WARNING/DEGRADED），这是**无状态**的逐次检查。不存在持久的 "当前系统风控级别" 状态。系统没有实现：
  - 预警级别（WARNING）的持续状态——每次检查独立判断
  - 受限级别（RESTRICTED）——只有 BLOCK 阻断
  - 熔断级别（CIRCUIT_BREAKER）——依赖于 `safety_meta_layer.is_trading_paused()` 的布尔返回，而非正式风险状态
- **后果**: 风控决策是逐次独立的，无法实现 "连续N次预警后升级为受限" 或 "熔断后需要观察期才能解除" 的滞后逻辑

#### P1-1: 级别跳跃无检测

- **位置**: `risk_service.py:264-352` (`check_before_trade`)
- **发现**: 如果风险从 LOW 直接跳到 CRITICAL（如突然的大额亏损），系统不发出特殊告警。`check_before_trade()` 返回当前检查的结果，不比较历史趋势
- **后果**: 无法区分 "渐进恶化的风控事件" 和 "突发灾难事件"，后者可能需要立即全平而非渐进降级

#### P1-2: 熔断解除后无恢复验证

- **位置**: `shadow_strategy_engine.py:704-707` + `risk_service.py:650-656`
- **发现**: `clear_absolute_ev_pause()` 将 `_absolute_ev_pause` 设为 False 后立即恢复交易，没有 "冷却期" 或 "观察期" 验证。同样，safety_meta 的 `trading_paused` 解除也没有观察期
- **后果**: 熔断-恢复-再熔断的快速振荡（chattering）可能在几秒内发生多次

#### P1-3: 风控状态与策略生命周期状态脱钩

- **位置**: `strategy_core_service.py:274-346` (`execute_option_trading_cycle`) + `risk_service.py`
- **发现**: 策略生命周期中，`execute_option_trading_cycle()` 自行检查风控（line 297-313），但即使风控返回 BLOCK，策略状态仍保持 RUNNING。这意味着从外部看策略"正在运行"，但内部不产生任何交易
- **后果**: 运维面板显示策略RUNNING但实际无交易，导致误判"策略不工作"

#### P2-1: RiskLevel 与 RiskCheckResult 的映射不清晰

- **位置**: `risk_service.py:42-55`
- **发现**: RiskLevel 是 4 级（LOW/MEDIUM/HIGH/CRITICAL）但 RiskCheckResult 也是 4 级（PASS/BLOCK/WARNING/DEGRADED），两者语义重叠。LOW+BLOCK、CRITICAL+PASS 等组合理论上可能但语义矛盾
- **后果**: 代码审查时难以理解检查结果的含义

#### P2-2: 风控统计无时间序列

- **位置**: `risk_service.py:156-158`
- **发现**: `_stats` 字典只累计通过/警告/阻断的计数，不记录时间序列。无法进行 "最近N分钟内的风控拒绝率" 趋势分析
- **后果**: 无法自动检测风控的渐进收紧趋势

## 维度6: HMM隐状态转移

### 6.1 状态定义

**五态模型** (在多个文件中引用):
```
correct_rise: 正确预判+价格上涨
correct_fall: 正确预判+价格下跌
wrong_rise:   错误预判+价格上涨
wrong_fall:   错误预判+价格下跌
other:        不确定/低置信度
```

**聚合态** (两态模型，在 `task_scheduler.py:982-985` + `strategy_core_service.py:404-410`):
```python
correct_rise + correct_fall → "correct_trending"
wrong_rise + wrong_fall     → "incorrect_reversal"
other                       → "other"
```

### 6.2 缺陷清单

#### P1-1: HMM 转移矩阵无退化检测

- **位置**: `strategy_ecosystem.py:654-662` + `task_scheduler.py:6645-6661`
- **发现**: `StateTransitionAnalytics` (sta) 对象通过 `sta.get_transition_matrix()` 获取转移矩阵并在日志中输出，但不做任何退化检测。转移矩阵没有：
  - 均值漂移检测（转移概率是否偏离历史均值 > 3σ）
  - 对角线退化检测（自转移概率是否突然下降——表示状态不稳定）
  - 转移熵突然增大检测
- **后果**: HMM 模型失效（如过拟合到噪声）无法被自动检测

#### P1-2: 聚合态映射掩盖个体状态异常

- **位置**: `task_scheduler.py:982-985`
- **发现**: `correct_rise` 和 `correct_fall` 都被映射到 `correct_trending`，而 `wrong_rise` 和 `wrong_fall` 都被映射到 `incorrect_reversal`。这种聚合丢失了二分信息：
  - `correct_rise` 80% + `correct_fall` 5% → 聚合后仍是 `correct_trending`，但实际上是偏向单一方向
  - 如果 `wrong_rise` 占 45% 和 `wrong_fall` 占 5%，聚合后是 `incorrect_reversal`，但不对称性被抹平
- **后果**: 有方向偏向的市场状态被错误分类，导致主/反策略选择错误

#### P2-1: 状态驻留时间无异常检测

- **位置**: `risk_service.py:1482-1495` (`_compute_life_score`)
- **发现**: `_compute_life_score()` 使用 `estimator.get_life_expectancy(hmm_state)` 来评估状态寿命，但只用于风控评分，不检测异常驻留（如同一状态连续停留超过历史 P99 时间的 3 倍）
- **后果**: HMM 卡在某个状态无法切换时（可能表示数据质量问题），系统不发出告警

#### P2-2: `state_confirm_bars` 机制被移除但注释未清理

- **位置**: `task_scheduler.py:344,372,430,442,461,473,507`
- **发现**: 多行注释 `# NOTE: decision_interval_minutes 已从系统中移除（与HMM state_confirm_bars冲突）` 表明 `state_confirm_bars` 机制已被移除。这意味着状态切换不再有确认窗口——HMM 每输出一次新状态就立即被采用
- **后果**: HMM 在状态边界上的高频抖动直接传递到策略切换

## 维度7: 信号生命周期状态

### 7.1 状态定义

**文件**: `signal_service.py`

信号只有一个明确的状态值：
```python
signal['status'] = 'EMITTED'   # line 301 — 唯一状态
```

审计标准要求的生命周期（生成→过滤→评分→执行→确认→完成）在系统中**完全不存在**。

### 7.2 缺陷清单

#### P0-1: 信号生命周期严重不完整 —— 仅一个 EMITTED 状态

- **位置**: `signal_service.py:289-302` (`generate_signal`)
- **发现**: 信号从 `generate_signal()` 创建到返回，只有一个状态 `'EMITTED'`。审计标准要求的完整生命周期：
  ```
  标准要求:  GENERATED → FILTERED → SCORED → EXECUTED → CONFIRMED → COMPLETED
  系统实际:  (无状态) → (被过滤返回None) → (唯一状态: EMITTED) → (之后无跟踪)
  ```
- **缺失状态**:
  - GENERATED: 信号原始生成但尚未评分/执行
  - FILTERED: 信号被过滤但可记录原因
  - SCORED: 信号已评分待执行
  - EXECUTED: 信号已发送订单
  - CONFIRMED: 订单成交确认
  - COMPLETED: 平仓后信号完成
- **后果**: 
  1. 信号发出后无法追溯其执行结果
  2. 信号确认延迟无法检测（EMITTED → 实际成交的时延）
  3. 信号取消无法实现（无 CANCELLED 状态）
  4. 信号统计中 "已发信号数" = "EMITTED信号数"，无法区分过滤/执行/完成

#### P1-1: 被过滤的信号无痕迹

- **位置**: `signal_service.py:217-285`
- **发现**: 所有过滤条件（PLR过滤、ModeEngine过滤、冷却过滤、决策分数过滤、HFT过滤、自适应阈值过滤）在过滤时都 `return None`，不做任何记录。被过滤的信号仅在 `_stats['filtered_signals']` 计数中加一
- **后果**: 无法事后分析 "为什么这个合约没产生交易"——过滤原因没有关联到具体合约/时间

#### P1-2: 信号过期机制缺失

- **位置**: `signal_service.py:54-55`
- **发现**: `_signal_history` 有 maxlen=1000 限制，但这只是 FIFO 淘汰，不是基于时间的过期。一个 EMITTED 信号如果从未被执行（例如因风控阻断），会永远留在历史记录中直到被挤出
- **后果**: 信号历史中包含大量"僵尸信号"（已生成但永不会成交），干扰信号分析

#### P1-3: 信号取消/作废机制缺失

- **位置**: 全局
- **发现**: 一旦信号被 EMITTED（line 301），没有任何 API 可以取消或作废它。如果策略切换（correct_trending → incorrect_reversal），之前的 EMITTED 信号应该被取消（因为它们基于不同的市场假设），但系统做不到
- **后果**: 策略切换后旧信号继续被订单引擎处理，可能导致错误方向的交易

#### P2-1: 信号确认回调链路不完整

- **位置**: `signal_service.py:356-368` + `strategy_core_service.py:242-250`
- **发现**: 信号通过 EventBus 发布 `SignalEvent`（line 358-366），但 `on_order()` 回调（`strategy_core_service.py:242-250`）在收到订单更新时，不关联回原始信号。订单成交后无法确认 "哪个信号导致了哪笔成交"
- **后果**: 信号到订单到成交的端到端追踪链路断裂

## 维度8: 连接状态机

### 8.1 现状

**审计标准要求四态**: 断开→连接中→已连接→重连中

**系统中连接状态检测仅在两处出现**:
1. `strategy_core_service.py:437` — `get_health_status()` 中检查 `ds.realtime_cache` 是否存在来判断数据源状态
2. `ProductionQuantSystem.py:272-308` — `get_system_status()` 包含模块健康报告但不含连接状态

**系统中无显式连接状态机**。无连接断开/重连的回调处理。

### 8.2 缺陷清单

#### P0-1: 连接状态机完全缺失

- **位置**: 全局
- **发现**: 审计标准要求的 断开→连接中→已连接→重连中 状态机在系统中完全不存在。没有任何文件定义了连接状态枚举、连接状态转移逻辑或连接事件回调
- **后果**:
  1. 数据源断开时系统无感知（数据断流被当作 "市场无变动"）
  2. 重连后数据回放处理缺失 → 断连期间丢失的K线/Tick无法补回
  3. 连接抖动（频繁断开-重连）可能导致重复订单（如超时追单在断开时发出、重连后重复）
  4. 多个数据源（如 CTP + 行情源）的连接状态无法分别监控

#### P1-1: 数据健康检查粒度粗

- **位置**: `strategy_core_service.py:435-438`
- **发现**: `get_health_status()` 中数据源检查仅通过 `ds.realtime_cache` 是否存在来判断，如果缓存存在但最近N秒没有更新，仍返回 "OK"。没有：
  - 数据新鲜度检查（最后tick时间 vs 当前时间）
  - 多数据源分别检查
  - 数据流速率异常检测
- **后果**: 数据延迟不会被检测为问题

#### P1-2: 连接断开时无自动保护措施

- **位置**: 全局
- **发现**: 无连接断开回调（如 `on_disconnect`/`on_connection_lost`）。如果行情断开，交易模块可能继续基于过期数据发出信号
- **后果**: 数据断流期间的交易决策基于陈旧数据

#### P2-1: 连接统计信息不可用

- **位置**: 全局
- **发现**: 无连接持续时间、重连次数、最近断开时间、心跳延迟等统计
- **后果**: 无法分析连接稳定性

#### P2-2: 生产系统连接管理缺失

- **位置**: `ProductionQuantSystem.py:272-308`
- **发现**: `get_system_status()` 只返回模块就绪状态，不包含任何连接级别的状态。ProductionQuantSystem 的 `update_tick()` 方法直接处理 tick，如果数据源断开则静默无tick
- **后果**: 系统级状态监控缺少连接维度
