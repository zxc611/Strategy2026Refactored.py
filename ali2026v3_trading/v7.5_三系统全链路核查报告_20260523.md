## 一、P0级问题（6项）— 阻塞生产部署

### P0-1: CascadeJudge.from_config() 4/6调用点未传params — 评分权重覆盖失效

| 项目 | 内容 |
|------|------|
| **问题** | CascadeJudge.from_config()在6个调用点中，4个未传params参数 |
| **位置** | `strategy_ecosystem.py:1126` / `enhanced_phase_scan.py:316` / `optuna_multiobjective_search.py:311` / `mode_engine.py:876` |
| **现状** | 4处写`CascadeJudge.from_config()`而非`CascadeJudge.from_config(params=params)` |
| **后果** | 参数池Round4扫描的`scoring_*_weight`权重仅在task_scheduler 2处生效，其余4路径回退YAML默认值。**三系统参数池覆盖不对齐** |
| **V7.0违反** | 第二十七节"三系统共用，修改即同步更新"——三系统应同等支持参数池覆盖 |
| **修复方案** | 4处调用改为`CascadeJudge.from_config(params=params)`，其中params来源：strategy_ecosystem用self.params，enhanced_phase_scan/optuna用回测params，mode_engine用auto_select_mode已有的params参数 |

### P0-2: CascadeJudge.from_config() 全部6调用点未传capital_scale — scoring_small永不生效

| 项目 | 内容 |
|------|------|
| **问题** | 全部6个CascadeJudge.from_config()调用均未传capital_scale参数 |
| **位置** | 上述6处调用均无capital_scale |
| **现状** | `cascade_config.yaml`定义了`scoring_small`节（profit_ratio=0.50, sortino=0.25, calmar=0.15, sharpe=0.10），但从无调用方告知当前资金规模 |
| **后果** | 小资金模式下评判权重无法自动切换，`scoring_small`节为**死配置** |
| **V7.0违反** | CascadeJudge设计支持资金规模自适应(27.1节)，但集成端未启用 |
| **修复方案** | 各调用方根据实际资金规模传入capital_scale='small'/'medium'/'large' |

### P0-3: profitability_metrics所有调用方不传 — MCBV和BH始终降级跳过

| 项目 | 内容 |
|------|------|
| **问题** | on_backtest_finish的profitability_metrics参数，所有调用方均不传 |
| **位置** | `backtest_integration_hooks.py:292`；调用方`usage_example.py:62,118,232,296` |
| **现状** | 4个调用方均不传profitability_metrics（或传空dict），导致`_per_trade_returns=None`, `_p_values=None` |
| **后果** | (1) MCBV蒙特卡洛破产验证降级到i.i.d.假设的`validate()`而非`validate_block_bootstrap()`；(2) BH多重比较校正静默跳过。**两个P0级deep_validation检查形同虚设** |
| **V7.0违反** | 第8.3节"蒙特卡洛破产检验(>99%生存率)"——生存率验证无法以block_bootstrap模式执行 |
| **修复方案** | 回测框架在调用on_backtest_finish时，从回测结果中提取per_trade_returns和p_values，打包为profitability_metrics传入 |

### P0-4: SCORING_COEFFICIENTS 40个系数中36个硬编码 — 参数化覆盖率仅10%

| 项目 | 内容 |
|------|------|
| **问题** | strategy_judgment_engine.py的SCORING_COEFFICIENTS全部硬编码，无法外部覆盖 |
| **位置** | `strategy_judgment_engine.py:241-281` |
| **现状** | 40个评判系数全部硬编码为模块级字典，无from_config机制，无params覆盖入口 |
| **后果** | `win_rate_full_score_at`, `extreme_dd_norm`, `mcbv_simulations`, `bh_significance_level`等36个关键系数**无法通过参数池/配置文件调整** |
| **V7.0违反** | 第二十九节"评判系数层: SCORING_COEFFICIENTS 30+评分公式系数+阈值" + "所有验证阈值已全部参数化，无残留硬编码魔数"——**文档声称已参数化但实际36/40仍硬编码** |
| **修复方案** | (1) 创建`config/judgment_scoring_config.yaml`存储40个系数；(2) StrategyJudgmentEngine增加from_config(params=)类方法；(3) 优先从params读取，次选YAML，兜底硬编码 |
