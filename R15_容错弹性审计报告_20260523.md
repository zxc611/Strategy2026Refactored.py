## 一、优雅降级策略

### 1.1 降级状态定义存在但覆盖面不足

**问题 1.1.1** | **P1** | 生产系统 + 评判系统

- **文件**：`strategy_core_service.py`（第 274-346 行）
- **描述**：`StrategyState.DEGRADED` 仅在 `execute_option_trading_cycle()` 中检查（第 277-278 行），但 `check_position_risk()`（第 412-425 行）**未检查** DEGRADED 状态。导致即使系统已降级，持仓风控仍会继续执行——其依赖的数据源在降级时可能已经失效。
- **影响系统**：生产系统
- **手册对照**：手册 12.1 节要求健康检查 API 在组件状态非 OK 时暂停新开仓，但并未规范 DEGRADED 状态下是否需要跳过持仓检查。

```
strategy_core_service.py:274-278  → execute_option_trading_cycle 检查 DEGRADED ✅
strategy_core_service.py:412-425  → check_position_risk 未检查 DEGRADED ❌
```

**问题 1.1.2** | **P1** | 生产系统

- **文件**：`strategy_core_service.py`（第 427-592 行）
- **描述**：`get_health_status()` 汇聚了 8 个组件的状态，但降级判定逻辑有覆盖缺口：
  - `greeks_calculator` 导入失败仅标记 `"UNAVAILABLE"`（第 444 行），不触发降级
  - `risk_dashboard` 状态变量 `risk_dashboard_status` **未在文件中定义**（第 447 行直接引用），可能导致 `NameError`
  - 当 `data_feeds` = `"DEGRADED"` 且其他组件 = `"OK"` 时，最终 `health_level` = `"OK"`（因为 `"DEGRADED"` 不在 `any_critical` 的 4 个枚举中，也不等于 `"WARNING"`），但数据馈送已降级实际上应该触发 WARNING
- **影响系统**：生产系统

```
strategy_core_service.py:447  → risk_dashboard_status 未定义，直接引用
strategy_core_service.py:567  → "DEGRADED" 不在 any_critical 判定中
```

**问题 1.1.3** | **P0** | 生产系统

- **文件**：`strategy_core_service.py`（第 980-999 行）
- **描述**：`onStart()` 中的关键步骤失败时，策略切换为 `StrategyState.DEGRADED`（第 989 行），但**其后仍继续执行全部运行时流程**——包括注册定时任务（第 1020 行 `_schedule_storage_warmup()`）和绑定平台回调。这意味着 DEGRADED 状态的策略仍然会接收 tick 数据并触发 `onTick()`（第 1124-1142 行），该回调**不检查 DEGRADED 状态**。
- **风险**：如果 `strategy_core.start()` 失败导致降级，但 tick 回调仍会驱动整个交易流水线（包括信号生成、风控、下单），而核心组件可能未初始化完成，导致不可预知的行为和资金损失。
- **影响系统**：生产系统

```
strategy_core_service.py:980-989  → 关键步骤失败，标记 DEGRADED
strategy_core_service.py:1124-1142 → onTick 不检查 DEGRADED，继续全量处理
strategy_core_service.py:1020     → 定时任务继续注册
```

**问题 1.1.4** | **P2** | 生产系统

- **文件**：`risk_service.py`（第 1931-2019 行）
- **描述**：`check_tradeable()` 中 `scheduler_service_unavailable`（第 2019 行）导致 `fallback` 标记但最终 `reason = "scheduler_unavailable"`。这本应是降级逻辑，但实际上并未阻止交易——`tradeable` 仍然为 `True`（因为 `ImportError` 分支不修改 `tradeable`）。这违反了 fail-safe 原则：在市场状态未知时不应允许交易。

**问题 1.1.5** | **P1** | 评判系统

- **影响系统**：评判系统

**问题 1.1.6** | **P2** | 生产系统

- **影响系统**：生产系统

## 二、重试策略完备性

### 2.1 下单重试有指数退避但缺乏幂等保障

**问题 2.1.1** | **P0** | 生产系统

- **文件**：`order_service.py`（第 249-335 行）
- **描述**：`send_order()` → `_execute_send_order()` → `_retry_send_order()` 的下单链路具备指数退避重试（2^n 秒，最多 3 次），但**缺乏幂等机制**：
  - 第 249 行 `_execute_send_order(order, retry_count=0)` 可能实际上在平台侧已成交，但返回了异常或 `None`（例如网络超时）
  - 第 251 行 `_retry_send_order(order)` 会生成**新的** `order_id`（通过 `_generate_order_id()`），导致同一个逻辑订单被执行两次
  - 没有任何 `idempotency_key` 或 `client_order_id` 去重机制
- **风险**：平台下单 API 返回超时但实际已成交 → 策略重试 → 产生重复订单 → 双倍风险暴露 → 资金损失
- **影响系统**：生产系统

```
order_service.py:249  → _execute_send_order(order, retry_count=0)
order_service.py:251  → _retry_send_order(order) — 新 order_id，无幂等性
order_service.py:321-335 → 指数退避重试，但重复下单风险
```

**问题 2.1.2** | **P1** | 生产系统

- **文件**：`order_service.py`（第 608-634 行）
- **描述**：`_chase_reorder()` 追单重试（原订单超时/取消后的重新下单）最多 `_max_chase_retries` 次，追单时**没有检查原订单的实际成交状态**。如果原订单已部分成交（`PARTIAL_CANCEL`，第 540-542 行），追单将导致超额持仓。
- **影响系统**：生产系统

**问题 2.1.3** | **P1** | 生产系统

- **文件**：`order_service.py`（第 459-472 行）
- **描述**：`cancel_all_pending()` 逐个遍历并单独调用 `cancel_order()`，**单个撤单失败不会阻止后续撤单**，这本身是正确的。但**没有对撤单失败的重试机制**——撤单 API 调用失败后直接返回 `False`（`cancel_order` 第 448 行），不会重试。在断路器触发时，撤单失败意味着未成交订单可能仍在市场中，导致意外成交。
- **影响系统**：生产系统

**问题 2.1.4** | **P1** | 生产系统

- **文件**：`ds_db_connection.py`（第 84-102 行，通过 storage_core 间接调用）
- **描述**：DuckDB 连接池的 `_create_connection` 方法在连接失败时直接抛出异常。上层 `storage_core.py` 的 Phase2 初始化有 `max_retries`（第 87 行），但在运行时阶段（Phase3），如果连接失败（例如数据库文件被锁），没有重试机制——写操作直接失败进入 spill WAL（第 106-107 行），读操作直接抛出异常。
- **影响系统**：生产系统

**问题 2.1.5** | **P2** | 生产系统

- **文件**：`subscription_manager.py`（第 118-121 行，第 245-370 行）
- **描述**：订阅重试已有指数退避配置（`backoff_strategy`），但退避策略的具体实现未在审计范围内找到完整的退避曲线定义（指数退避的 base 和 max 值）。`retry_base_delay` 在初始化时传递（`storage_core.py` 第 170 行），但退避系数在 `SubscriptionManager` 内部仅为文档声称，需要确认实际计算逻辑是否与配置一致。
- **影响系统**：生产系统

## 三、熔断与隔离

### 3.1 断路器有框架但状态未持久化

**问题 3.1.1** | **P0** | 生产系统

- **文件**：`risk_service.py`（第 2115-2130 行）
- **描述**：断路器的所有状态变量**均为纯内存变量**：
  - `_circuit_breaker_triggered`（无显式初始化但通过 `_trading_paused_until` 隐式管理）
  - `_trading_paused_until`：truncated 暂停截止时间
  - `_circuit_breaker_shadow_mode`：影子观察模式标志
  - `_circuit_breaker_shadow_until`：影子模式截止时间
- **风险**：策略进程崩溃重启后，断路器状态完全丢失。如果崩溃是因为市场剧烈波动导致的连锁故障，重启后立即恢复全量交易将面临与崩溃前相同的风险条件，形成"崩溃→重启→再崩溃"的循环。
- **影响系统**：生产系统

```
risk_service.py:2116  → circuit_breaker_pause_sec 配置存在
risk_service.py:2119  → circuit_breaker_calm_period_sec 配置存在
risk_service.py:2030-2040 → is_circuit_breaker_shadow_mode() 纯内存
# 无任何 disk_persist / checkpoint 调用
```

**问题 3.1.2** | **P0** | 生产系统

- **文件**：`strategy_core_service.py`（第 274-346 行）、`strategy_core_service.py`（第 412-425 行）
- **描述**：交易锁 `_trading_lock`（`threading.RLock()`）同时保护 `execute_option_trading_cycle()` 和 `check_position_risk()`，两者共享同一把锁。如果 `execute_option_trading_cycle` 中的某笔下单调用阻塞（例如平台 API 无响应），将导致 `check_position_risk` 也无法获取锁，**持仓风控被完全阻塞**。
- **风险**：平台 API 卡住 → 持仓风控停滞 → 止损信号无法执行 → 亏损扩大 → 资金损失
- **影响系统**：生产系统

```
strategy_core_service.py:80   → _trading_lock = threading.RLock() （共享锁）
strategy_core_service.py:280  → execute_option_trading_cycle 获取锁
strategy_core_service.py:415  → check_position_risk 获取同一把锁
# 两操作共享同一锁：下单阻塞 → 风控阻塞
```

**问题 3.1.3** | **P1** | 生产系统

- **文件**：`strategy_core_service.py`（第 80 行）、`strategy_ecosystem.py`（整体架构）
- **描述**：六策略（S1-S6）共享同一个 `StrategyCoreService` 实例、同一把交易锁、同一定时调度器、同一个 `OrderService`。除 S5/S6 有互斥规则外，没有跨策略的资源隔离（如独立的线程池、独立的锁、独立的限额）。任何一个策略的异常不会自动隔离到其他五个策略。
- **影响系统**：生产系统

**问题 3.1.4** | **P1** | 生产系统

- **文件**：`risk_service.py`（第 2121-2130 行）
- **描述**：断路器触发时调用 `cancel_all_pending()`，但**没有禁用后续的 `execute_option_trading_cycle()`**。断路器期间，交易定时任务（30s 间隔）仍会触发，虽然 `execute_option_trading_cycle` 开头没有检查断路器状态——它只检查 `DEGRADED` 和健康检查（P-30 修复），但不检查 `SafetyMetaLayer` 的暂停状态。
- **影响系统**：生产系统

```
risk_service.py:2121-2130 → 断路器触发 → cancel_all_pending ✅
strategy_core_service.py:274-346 → execute_option_trading_cycle 不检查 SafetyMetaLayer.is_trading_paused() ❌
strategy_core_service.py:298-314 → 有 check_before_trade() 但风控异常时继续交易（第 313 行）
```

**问题 3.1.5** | **P2** | 生产系统

- **文件**：`strategy_core_service.py`（第 313-314 行）
- **描述**：风控检查异常时的策略是 `"继续交易"`（`logging.warning + 不 return`）。这是**显式的 fail-open 策略**（异常时放行而非阻断），在金融系统中应该是 fail-closed（异常时阻断）。
- **影响系统**：生产系统

## 四、部分失败处理

### 4.1 批量交易缺乏事务性保障

**问题 4.1.1** | **P0** | 生产系统

- **文件**：`strategy_core_service.py`（第 274-346 行）
- **描述**：`execute_option_trading_cycle()` 中批量下单（第 291-328 行）完全独立执行：`for t in targets: ... order_ids = self._order_service.execute_by_ranking(targets)`。如果 3 笔订单中第 1、第 2 笔成功、第 3 笔失败：
  - 没有回滚机制（已成交的订单不可逆）
  - 没有补偿交易（反向对冲）
  - 没有部分失败告警升级
  - 系统只是记录日志并继续
- **风险**：批量操作中部分成功部分失败可能导致预期外的风险暴露，例如：本来计划 3 笔订单对冲分散风险，但只成交了 2 笔，导致净暴露方向偏离预期。
- **影响系统**：生产系统

**问题 4.1.2** | **P0** | 生产系统

- **文件**：`order_service.py`（第 547-575 行）
- **描述**：`check_pending_orders()` 中超时订单的取消和追单操作（第 567-575 行）**不是原子性的**。在 `cancel_order(order_id)`（第 572 行）和 `_chase_reorder(current_order)`（第 575 行）之间，如果进程崩溃，订单可能已被取消但追单尚未发出，导致持仓缺口。
- **影响系统**：生产系统

**问题 4.1.3** | **P1** | 生产系统

- **文件**：`storage_core.py`（第 458-500 行）、`ds_data_writer.py`（第 637 行）
- **描述**：K 线和 Tick 的批量写入分批执行，部分批次成功部分失败时：
  - 队列未回滚（失败的批次数据丢失）
  - `drop_on_full` 配置允许丢弃数据（第 93 行），但没有丢弃数据的审计日志
  - 没有部分失败后的数据一致性校验
- **影响系统**：生产系统

**问题 4.1.4** | **P1** | 评判系统

- **文件**：`参数池/task_scheduler.py`（第 5650-5715 行）
- **描述**：参数网格搜索中的断点续传（checkpoint）方案仅在任务完成后保存到 JSON 文件。如果任务执行到一半崩溃（例如某个参数组合导致内存溢出），该参数组合的结果将丢失，且 checkpoint 中不会记录"正在执行中"的状态。
- **影响系统**：评判系统

**问题 4.1.5** | **P2** | 生产系统

- **文件**：`strategy_ecosystem.py`（第 462-496 行）
- **描述**：`_propagate_capital_scale_fallback()` 向 5 个下游服务（RiskService、SignalService、StateParamManager、BoxSpringStrategy 等）传播资金缩放配置。如果其中某个服务接收失败（异常被吞掉，仅 `logger.debug`），其他服务仍继续接收。这可能导致不同服务对资金规模的理解不一致。
- **影响系统**：生产系统

## 五、状态恢复能力

### 5.1 交易状态缺乏持久化快照

**问题 5.1.1** | **P0** | 生产系统

- **文件**：`order_service.py`（第 95-131 行，全局状态）
- **描述**：`OrderService` 的所有订单状态（`_orders_by_id`、`_pending_order_attempts`、`_virtual_positions`）均为纯内存数据结构。策略崩溃后：
  - 所有已提交但未成交的订单记录丢失
  - 无法判断哪些订单在平台侧实际已成交
  - 虚拟仓位的 PnL 历史丢失
  - 无法进行崩溃后的仓位对账
- **风险**：策略崩溃重启后，与平台实际持仓状态不一致，可能导致重复开仓或错误平仓。
- **影响系统**：生产系统

```
order_service.py:95-131  → 所有订单状态纯内存
# 无任何 on_disk 持久化（虽然有 _persist_trade_log 但仅用于日志审计）
```

**问题 5.1.2** | **P0** | 生产系统

- **文件**：`position_service.py`（全局状态）
- **描述**：持仓服务的位置数据同样是纯内存。`OrderService._persist_trade_log()` 写入交易日志（第 254 行），但该日志是追加式 CSV/JSONL，**不能用于崩溃恢复**（因为它记录的是"发生了什么"而非"当前状态是什么"）。崩溃后需要全量重建持仓状态，但重建过程依赖：
  - 平台 API 返回的当前持仓（可能不完整或有延迟）
  - 交易日志的回放（没有实现自动回放功能）
- **影响系统**：生产系统

**问题 5.1.3** | **P1** | 生产系统

- **文件**：`subscription_manager.py`（第 142-233 行）
- **描述**：订阅管理器具备 WAL（Write-Ahead Log）恢复机制（`recover_on_start`），这是良好的设计。但 WAL 恢复存在以下缺陷：
  - WAL 文件即使不启用（`enable_wal=False`），系统不会报错，静默丧失恢复能力
  - WAL 以 append-only JSONL 格式写入，没有 compaction 机制，长期运行后文件可能非常大
  - WAL 恢复时的去重仅依赖合约代码，如果同一个合约多次被订阅/取消/再订阅，WAL 中会有冗余记录
- **影响系统**：生产系统

**问题 5.1.4** | **P1** | 评判系统

**问题 5.1.5** | **P2** | 生产系统

- **描述**：`onStop()` 的重复执行检测（`_stop_executed` 标志，第 1037 行）是纯内存变量。如果 `onStop()` 因异常中途退出，`_stop_executed` 已设置为 `True`，下次调用（或 `onDestroy()` 第 1108 行）将跳过部分清理步骤。`onDestroy()` 有"onStop not executed"的回退路径（第 1113-1117 行），但 `_stop_executed = True` 后如果清理不完整，无法知道哪些资源已释放、哪些未释放。

## 六、超时与取消

### 6.1 多个阻塞操作缺乏超时控制

**问题 6.1.1** | **P0** | 生产系统

- **文件**：`order_service.py`（第 263-319 行）
- **描述**：`_execute_send_order()` 调用 `self._platform_insert_order(**filtered_params)`（第 299 行）**没有任何超时设置**。平台下单 API 是外部调用，如果网络延迟或平台侧处理缓慢，调用将无限阻塞。由于 `_trading_lock` 在此调用期间被持有（第 280 行），整个策略的交易线程被卡死。
- **风险**：平台 API 阻塞 → 持仓风控停滞 → 止损无法执行 → 资金损失
- **影响系统**：生产系统

```
order_service.py:299  → self._platform_insert_order(**filtered_params) — 无超时
strategy_core_service.py:280 → 调用期间持有 _trading_lock
strategy_core_service.py:415 → check_position_risk 等待同一把锁
```

**问题 6.1.2** | **P1** | 生产系统

- **文件**：`order_service.py`（第 443-448 行）
- **描述**：`cancel_order()` 调用 `self._platform_cancel_order(platform_id)`（第 445 行）**没有超时保护**。如果平台撤单 API 无响应，撤单操作永久阻塞。虽然它不持有共享锁（只在方法内使用 `_lock` 做局部操作），但阻塞的撤单线程无法被取消或通知外部。
- **影响系统**：生产系统

**问题 6.1.3** | **P1** | 生产系统

- **文件**：`query_service.py`（第 73-82 行）、`data_service.py`（全局 DuckDB 查询）
- **描述**：所有通过 DuckDB 的数据库查询操作**没有语句级别超时设置**。虽然 DuckDB 默认是内存/本地数据库查询，通常很快，但在大数据量扫描场景下（如历史 K 线加载）可能长时间运行。`get_data_service()` 返回的 DataService 对象没有 `query_timeout` 参数。
- **影响系统**：生产系统

**问题 6.1.4** | **P2** | 生产系统

- **文件**：`strategy_core_service.py`（第 726-794 行）
- **描述**：`_storage_warm_timer` 使用 `threading.Timer(delay_seconds, _warmup)`（第 794 行），但 Timer 本身没有超时取消机制。如果 warmup 逻辑在执行中阻塞，Timer 线程不会被主动中断，也没有 fallback。
- **影响系统**：生产系统

## 七、资源耗尽保护

### 7.1 队列有边界但背压机制不完整

**问题 7.1.1** | **P1** | 生产系统

- **文件**：`storage_core.py`（第 105-107 行）、`storage_core.py`（第 93 行）
- **描述**：Tick 分片队列 `queue.Queue(maxsize=_shard_cap)` 有容量限制，但入队操作（tick 写入路径）使用 `queue.Queue.put()` 而非 `put_nowait()`。如果队列满，生产者线程会**阻塞**等待，而不是丢弃数据或触发背压。由于生产者是平台 tick 回调线程，阻塞会导致所有后续 tick 堆积在平台层，可能造成更严重的连锁问题。
- **影响系统**：生产系统

**问题 7.1.2** | **P1** | 生产系统

- **文件**：`strategy_core_service.py`（第 274-346 行）
- **描述**：交易周期执行频率由 `StrategyScheduler` 的 30 秒间隔控制，但周期执行时长本身没有上限。如果某次 `execute_option_trading_cycle()` 因为平台 API 慢而执行了 60 秒，下一次定时触发（30 秒后）将检测到锁被占用而跳过（第 280-281 行），但**没有告警**，也没有跳帧计数。如果连续 N 次跳过，说明系统过载，但没有自动降频或告警机制。
- **影响系统**：生产系统

**问题 7.1.3** | **P2** | 生产系统

- **文件**：`ds_db_connection.py`（连接池管理）、`storage_core.py`（第 100-105 行）
- **描述**：DuckDB 连接池大小由配置决定，但运行时**没有动态调整**能力。如果写入线程因 DB 锁等待排队，连接池不会自动扩容。同样，如果空闲连接过多，不会自动缩容。
- **影响系统**：生产系统

**问题 7.1.4** | **P0** | 生产系统

- **文件**：全域搜索（无相关代码）
- **描述**：整个系统**没有任何内存警戒线或磁盘空间监控**。DuckDB 数据库文件持续增长（append-only），但：
  - 没有磁盘使用率检查
  - 没有数据库文件大小上限
  - 没有自动清理/归档机制
  - `maintenance_service.py` 有清理任务（第 254-267 行），但清理的是订阅数据去重和空表，不是数据库文件本身的 compact/vacuum
- **风险**：磁盘满 → 写入失败 → tick 和 K 线数据丢失 → 回测结果失真 → 信号失真 → 资金损失
- **影响系统**：生产系统

**问题 7.1.5** | **P2** | 生产系统

- **文件**：`width_cache.py`（第 1161-1182 行）、`ds_realtime_cache.py`（第 459 行）
- **描述**：缓存层有 `get_memory_stats()` 提供内存估算，但**没有主动的内存保护**（例如内存使用超过阈值时触发 LRU 淘汰或缓存收缩）。`width_cache` 的 `memory_estimate_kb` 仅为统计信息，不驱动任何保护行为。
- **影响系统**：生产系统
