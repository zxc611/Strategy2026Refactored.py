### 维度1：参数定义源一致性

#### R16-01 (P0) — correct_trending.max_risk_ratio 代码兜底严重偏离 YAML

| 项目 | 详情 |
|------|------|
| **文件路径** | `config_params.py` 第543行 vs `参数池/state_param_sets.yaml` 第6行 |
| **偏差点** | YAML值 = **0.8**, 代码兜底 = **0.3** (偏差 -62.5%) |
| **问题描述** | `get_default_state_param_sets()` 函数的代码兜底分支中 `correct_trending.max_risk_ratio` 硬编码为 `0.3`，但 `state_param_sets.yaml`（P0-18修复后）中正确定义为 `0.8`。当YAML文件因任何原因加载失败时（文件损坏、I/O错误、YAML解析异常），系统会静默回退到 `0.3`，将 correct_trending 状态的风控敞口从设计值的 80% 骤降至 30%，导致趋势行情中的交易规模被错误压缩 62.5%。 |
| **影响系统** | 生产系统、测试系统（两者均调用 `get_default_state_param_sets()`） |
| **偏离程度** | 极严重 — 趋势状态是利润核心贡献者，风险敞口误缩意味着该状态下所有交易的理论收益被压缩 62.5% |
| **触发条件** | YAML 文件读取失败（任何原因），包括但不限于：磁盘I/O瞬时故障、YAML解析器版本不兼容、文件被外部进程锁定 |
| **手册依据** | 手册 3.1.1 节明确 correct_trending.max_risk_ratio=0.8 |

### 维度2：参数加载链完整性

#### R16-03 (P1) — params_default.json 不存在，加载链缺少 JSON 覆盖层

| 项目 | 详情 |
|------|------|
| **涉及文件** | `config_params.py:348-359` (load_default_params_from_json), `config/params_default.json` (不存在) |
| **问题描述** | 系统设计的三层参数加载链为：**JSON覆盖层 → YAML参数池 → 代码DEFAULT_PARAM_TABLE兜底**。但 `config/params_default.json` 文件实际**不存在**。`load_default_params_from_json()` 函数在文件缺失时静默返回空字典，导致实际加载链简化为：**YAML → 代码兜底**，缺少了 JSON 覆盖层这一关键环节。手册设计的"部署工程师通过JSON做现场调参覆盖"能力完全丧失。 |
| **代码路径** | `get_cached_params()` (config_params.py:467-488) → 调用 `load_default_params_from_json()` → 返回 `{}` → 回退到 `DEFAULT_PARAM_TABLE` |
| **影响系统** | 生产系统 |
| **危害** | 现场运维人员无法通过修改JSON实现快速参数覆盖，所有调参必须修改YAML或代码；增加了误操作风险 |

**参数加载链的完整拓扑**:

```
get_cached_params()                             ← 全局参数入口
  ├── load_default_params_from_json()           ← 步骤1: JSON覆盖 (缺失!)
  │   └── 返回 {} (文件不存在)
  ├── DEFAULT_PARAM_TABLE.copy()                ← 步骤2: 代码兜底
  ├── _apply_resolved_overrides()               ← 步骤3: 环境变量/命令行覆盖
  └── _param_table_cache['state_param_sets']    ← 附加: 状态参数 (独立加载)
      └── get_default_state_param_sets()
          ├── 参数池/state_param_sets.yaml      ← 优先
          └── 代码兜底                           ← 回退

StateParamManager._load_param_sets()            ← 状态参数独立入口
  ├── 参数池/state_param_sets.yaml              ← 直接加载 (绕过缓存!)
  └── get_default_state_param_sets()            ← 回退

ParamsService._check_and_reload()               ← params_service 独立缓存
  └── 独立检索路径 (与上述完全解耦)

ModeConfig._make_mode_config()                  ← 模式配置入口
  └── config/mode_config.yaml OR 代码兜底
```

### 维度3：参数缓存一致性

#### R16-04 (P1) — 两套独立参数缓存，TTL 不一致且不互通

| 项目 | 详情 |
|------|------|
| **涉及文件** | `config_params.py:66-67` (CACHE_TTL=300), `params_service.py:61` (_check_interval=900) |
| **偏差点** | 全局缓存 TTL = **300秒** (5分钟), ParamsService 检查间隔 = **900秒** (15分钟) |
| **问题描述** | 系统存在两套完全独立的参数缓存系统：<br>1. **config_params 全局缓存**: `_param_table_cache` + `_cache_timestamp`，TTL=300秒，通过 `get_cached_params()` 访问<br>2. **ParamsService 实例缓存**: `self._params` dict + `self._last_check`，_check_interval=900秒，通过 `params_service.get()` 访问<br><br>两套缓存**互不通信**。同一个参数在两套缓存中可能因不同的刷新时机而产生不同的值。更严重的是：ParamsService 的 900 秒间隔是手册设计 300 秒的 **3倍**，意味着部分模块可能在长达 15 分钟内使用过期参数。 |
| **影响系统** | 所有通过 ParamsService 消费参数的模块（包括 shadow_strategy、strategy_tick_handler 等） |
| **手册依据** | 手册 6.3.2 节明确规定缓存 TTL=300 秒，但 ParamsService 未遵循 |

**缓存架构对比**:

| 特性 | config_params 全局缓存 | ParamsService |
|------|----------------------|---------------|
| TTL | 300秒 | 900秒 |
| 数据源 | DEFAULT_PARAM_TABLE + JSON + YAML | 独立参数文件 |
| 状态参数 | 内嵌(get_default_state_param_sets) | 未加载 |
| 更新方式 | update_cached_params() | self._check_and_reload() |
| 消费者 | risk_service, position_service 等 | shadow_strategy, tick_handler 等 |

### 维度4：运行时参数修改的传播

#### R16-05 (P1) — StateParamManager 绕过缓存直接读取 YAML

| 项目 | 详情 |
|------|------|
| **涉及文件** | `state_param_manager.py:80-130` (_load_param_sets), `config_params.py:510-570` (get_default_state_param_sets) |
| **问题描述** | `StateParamManager._load_param_sets()` 在每次请求时直接从 **YAML 文件**读取参数，完全绕过了 `get_cached_params()` 的缓存层。这意味着：<br>1. `update_cached_params()` 修改的是缓存中的参数，但 StateParamManager 看不到这些修改<br>2. `_load_param_sets()` 每次读文件，性能和缓存层的设计意图相矛盾<br>3. 运行时热更新参数对 StateParamManager 的消费者无效 |

**代码证据**:
- `state_param_manager.py`: `_load_param_sets()` 直接 `yaml.safe_load()` 打开 YAML 文件
- `config_params.py`: `get_cached_params()` 返回的缓存中包含 `state_param_sets` 键，但此键为一次性快照，不随 YAML 文件更新而自动刷新

| **影响系统** | 生产系统 (StateParamManager → position_service 开仓决策) |

#### R16-06 (P1) — 核心服务初始化时固化参数引用

| 项目 | 详情 |
|------|------|
| **涉及文件** | `risk_service.py:1048-1060`, `position_service.py:30-50` |
| **问题描述** | `risk_service` 和 `position_service` 在 `__init__()` 阶段通过 `get_cached_params()` 获取参数并存入 `self.params`。此后整个生命周期中使用 `self.params["key"]` 而非每次调用 `get_cached_params()`。<br>`update_cached_params()` 虽然可以更新全局缓存，但已初始化的 risk_service / position_service 实例中的 `self.params` 字典不会自动同步——它们是初始化时的**快照副本**，而非实时引用。 |

```
update_cached_params() 
  └── _param_table_cache 更新 ✓

      ├── position_service (self.params) → 旧值 ✗ (issue R16-06)
      └── StateParamManager → YAML直读 → YAML值 (issue R16-05)
```

| **影响系统** | 生产系统 (risk_service, position_service 初始化后不再感知参数变更) |

### 维度6：YAML参数池与代码默认值的优先级

#### R16-07 (P0) — tvf_enabled 三源严重不一致

| 项目 | 详情 |
|------|------|
| **涉及文件** | `参数池/parameter_attribute_matrix.yaml` tvf_enabled 条目, `参数池/tvf_params.yaml:2`, `mode_engine.py` CAPITAL_MODE_CONFIGS |
| **偏差点** | attribute_matrix = **true**, tvf_params.yaml = **false**, ModeConfig 默认 = **false** |
| **问题描述** | TVF（六维仓位调整因子）的启用开关在三个参数源中存在致命矛盾：<br>- `parameter_attribute_matrix.yaml` 将 `tvf_enabled` 的 `default` 设为 `true`<br>- `tvf_params.yaml` 将 `tvf_enabled` 设为 `false`<br>- `mode_engine.py` 中 `ModeConfig._make_mode_config()` 的 hardcoded 默认值也是 `false`<br><br>实际运行时，ModeConfig 从 tvf_params.yaml 读取 `tvf_enabled=false`，同时有代码兜底 false，所以 TVF 不会被启用。但 **parameter_attribute_matrix 作为"参数池权威属性矩阵"标记该功能为开启状态**，任何依赖矩阵做合规性检查的模块（评判系统）会认为 TVF 应当启用，与实际运行状态矛盾。 |
| **影响系统** | 生产系统（ModeConfig消费），评判系统（矩阵消费） |
| **手册依据** | 手册声明"TVF默认开启"，但实际 tvf_params.yaml 中为 false；手册 4.2.1 节 TVF 参数表未标注默认值矛盾 |

**TVF 参数加载优先级验证**:

```
mode_engine._make_mode_config()
  ├── tvf_params = _load_tvf_params()          ← 步骤1: 读 tvf_params.yaml
  │   └── 若 tvf_params.yaml 不存在 → 代码兜底 tvf_enabled=False
  ├── mode_config = CAPITAL_MODE_CONFIGS[mode]  ← 步骤2: 硬编码模式配置
  └── mode_config["tvf_enabled"] = tvf_params.get("tvf_enabled", False)  ← YAML优先 ✓
```
