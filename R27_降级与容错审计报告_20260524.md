## 一、熔断器模式（Circuit Breaker）

### 1.1 断路器核心机制（SafetyMetaLayer）

**代码位置**：`risk_service.py` 第1817-1949行（SafetyMetaLayer类）

**现状评估**：✅ 系统具备较完善的断路器框架

- **速率断路器**（`_check_circuit_breaker`，第1890行）：基于1分钟权益回撤与滚动3σ动态阈值比较（第1905-1922行）。阈值参数 `circuit_breaker_trigger_sigma` 从参数池读取（R3-P-05修复），替代硬编码。
- **冷静期**（`_circuit_breaker_calm_until`，第1826行）：首次触发后 `circuit_breaker_calm_period_sec`（默认600秒）内忽略二次触发，防止频繁"抽搐"（第1923-1929行）。
- **影子模式**（P1-7修复）：断路器触发后自动进入5分钟影子观察期（`_circuit_breaker_shadow_mode`，第1941-1942行），期间仅允许平仓。观察期结束后自动恢复正常交易（第2182-2192行）。
- **日回撤硬停止**（`_check_daily_drawdown`，第1951行）：双路径（固定5%阈值 + 5日均值乘数），触发后禁止所有交易直至人工确认（`confirm_daily_resume`，第2250行）。
- **连亏保护**（`_consecutive_loss_counts`，第164行）：按资金规模分级（small=5次, medium=7次, large=10次），有自动恢复超时（第167-171行，30/60/120分钟）。
- **跨策略断路器**（`CrossStrategyRiskGuard`）：ecosystem._check_cross_strategy_risk 中 BLOCK/CIRCUIT_BREAK 两级（第789-795行），异常时fail-safe阻断（第834-836行）。

### 1.2 发现与分级

#### P0-1.2.1 断路器状态无持久化——重启即丧失全部保护

- **证据**：
  ```
  risk_service.py:1824  → _trading_paused_until: float = 0.0      # 纯内存
  risk_service.py:1826  → _circuit_breaker_calm_until: float = 0.0 # 纯内存
  risk_service.py:1828  → _circuit_breaker_shadow_mode: bool       # 纯内存
  risk_service.py:1829  → _circuit_breaker_shadow_until: float     # 纯内存
  risk_service.py:1840  → _daily_hard_stop_triggered: bool         # 纯内存
  ```
  全局搜索 `disk_persist` `save_state` `checkpoint` `marshal` `pickle` 在 SafetyMetaLayer 中和断路器状态相关代码中**均未命中**。

- **失败链路**：市场剧烈波动 → 速率断路器触发 → 安全暂停交易 → 进程因相同波动造成的资源耗尽而崩溃 → 重启 → **所有断路器状态丢失** → 全量恢复交易 → 再次遭遇相同的极端市场条件 → 再次触发断路器或直接亏损。
- **最坏情况**：形成"崩溃→重启→再崩溃"循环，每次循环都暴露在相同的市场条件下。
- **对比**：参数池 `task_scheduler.py` 的回测断路器（`circuit_breaker_until`，第877行）同样为内存字段，但回测环境可接受；生产环境不可接受。
- **建议优先级**：最高。断路器状态必须在磁盘上有持久化表示（如JSON checkpoint），重启时恢复。

#### P0-1.2.2 断路器触发后策略周期仍继续执行交易

- **证据**：
  ```
  risk_service.py:1932  → _trading_paused_until = now + pause_duration  # 设置暂停
  risk_service.py:1949  → _cancel_pending_on_circuit_breaker()           # 取消待处理订单
  
  # 但是 execute_option_trading_cycle() 的入口检查中：
  strategy_core_service.py:277  → 检查 DEGRADED 状态
  strategy_core_service.py:298  → 调用 check_before_trade()
  # 没有在入口处检查 SafetyMetaLayer.is_trading_paused()
  ```

- **失败链路**：断路器触发 → 暂停标志设置 → 定时任务（30秒间隔）下次触发 `execute_option_trading_cycle()` → 入口未检查断路器暂停状态 → 进入 `check_before_trade()` → `_check_safety_meta_layer()` 拦截（第290行）→ 但是在此之前已经执行了部分代码且**持仓风控检查 `check_position_risk()` 不经过 `check_before_trade()` 的断路器检查**。
- **缓解因素**：`check_before_trade` 内部第290行会调用 `_check_safety_meta_layer`，其中第294-304行检查了 `is_trading_paused()`。但这依赖于每个调用点都正确地调用 `check_before_trade`。
- **建议**：在 `execute_option_trading_cycle()` 入口处显式检查断路器状态，尽早返回。

#### P1-1.2.3 断路器"半开"探测机制缺失

- **证据**：影子模式（第1941行）是纯时间驱动的（`shadow_until = now + pause + calm + 300`），而非行为驱动的探测。没有"半开"状态（允许少量试探性交易以验证市场条件是否恢复正常）。
- **现有设计**：冷静期结束 → 影子模式（只允许平仓）→ 5分钟后自动恢复全量交易。
- **缺陷**：如果造成断路器触发的市场条件仍然存在（但权益已经因为暂停而保持静止，`_equity_series` 没有被更新），5分钟后自动恢复可能立即再次触发断路器。
- **建议**：影子模式期间应允许有限数量的试探性开仓（例如单笔小额），只有在试探成功后才全量恢复。

#### P1-1.2.4 连亏保护的恢复逻辑简单但有隐患

- **证据**：`risk_service.py` 第1173-1198行。连亏暂停后仅基于固定超时（30/60/120分钟）自动恢复，**不检查恢复时的市场状态或盈亏比**。
- **失败链路**：连亏7笔 → 暂停 → 60分钟后自动恢复 → 但如果市场条件仍然不利（趋势延续），恢复后可能立即继续连亏。
- **建议**：恢复前增加条件检查（如最近N分钟权益是否稳定、波动率是否回归正常）。

#### P2-1.2.5 熔断器统计指标未被外部监控消费

- **证据**：`_stats` 字典（第1843-1850行）追踪了 `circuit_breaker_triggers`、`circuit_breaker_calm_rejects` 等指标，但这些指标仅存在于内存中。`HealthCheckAPI`（`health_check_api.py`，第67行）在 `ProductionQuantSystem.on_update_tick()`（第217行）更新组件状态，但**不消费 SafetyMetaLayer 的熔断统计**。
- **建议**：将熔断器统计接入 HealthCheckAPI 和诊断服务。

## 二、隔板模式（Bulkhead）

### 2.1 策略间资源隔离现状

**代码位置**：`strategy_ecosystem.py`（StrategyEcosystem，策略槽位定义第250-270行）、`strategy_core_service.py`（交易锁第80行）

**架构分析**：
- StrategyEcosystem 为6个策略定义了独立的 `StrategySlot` 对象（master/reverse/other/spring/arbitrage/market_making），各自有独立的 `position_count`、`win_loss_ratio`、`capital_allocation`。
- 但所有6个策略**共享同一个 `StrategyCoreService` 实例**（继承自 BaseStrategy），**同一把 `_trading_lock`**（`threading.RLock()`，第80行），**同一个 `OrderService` 单例**。
- 影子策略（ShadowStrategyEngine）有明确的**物理隔离**：独立 `paper_account`（第180行）、不经过 OrderService（第1098行 `uses_order_service=False`）、独立下单日志（第189行）。

---

### 2.2 发现与分级

#### P0-2.2.1 六策略共享执行线程——任一策略阻塞全部策略停摆

- **证据**：
  ```
  strategy_core_service.py:80   → _trading_lock = threading.RLock()  # 全局共享
  # 所有策略的交易决策都通过 execute_option_trading_cycle() 执行
  # 该方法获取 _trading_lock (第280行)
  # 如果某个策略的下单调用阻塞（如S2的共振订单调用平台API阻塞）
  # 则S1/S3/S4/S5/S6的后续交易全部排队等待
  ```
- **失败链路**：S2策略提交一笔下单价位较偏的订单 → 平台撮合引擎响应慢 → `_platform_insert_order` 阻塞30秒 → `_trading_lock` 在此期间被持有 → S1（HFT）的毫秒级交易窗口错过 → HFT策略信号失效 → 错失盈利机会或止损窗口。
- **对比**：影子策略在此方面做得更好——其订单完全不进入主OrderService，有独立的 `_shadow_order_log`。
- **建议**：按策略类型提供独立的下单通道或异步下单队列（至少HFT策略应有独立通道）。

#### P1-2.2.2 跨策略互斥规则有框架但执行不够彻底

- **证据**：`strategy_ecosystem.py` 第740-797行（`check_mutual_exclusion`方法）。
- ✅ 良好设计：master/reverse 间有同向持仓互斥；other状态禁止主/反策略新开仓；spring需检查其他策略的盈亏比；跨策略Delta敞口有 BLOCK/CIRCUIT_BREAK 两级阻断。
- ❌ 缺陷：互斥规则**仅在信号层面执行**。如果两笔订单几乎同时到达（不同 tick 回调），可能在锁的间隙绕过互斥检查（虽然 `_trading_lock` 提供了序列化，但信号生成在锁外）。
- **建议**：增加订单提交时的二次互斥检查。

#### P1-2.2.3 不同交易对的执行无隔离

- **证据**：所有交易对的订单通过同一个 `OrderService` 单例的 `send_order` 方法（第240行）执行，同一把 `_lock`（第90行 `threading.RLock()`）保护。一个交易对的下单失败/阻塞可能延迟另一个交易对的下单。
- **影响**：IF（沪深300）的下单延迟可能影响IC（中证500）的及时成交。
- **对比**：PositionService 按合约分片锁（`position_locks: Dict[str, threading.RLock]`，第242行）是良好的隔离设计。
- **建议**：OrderService 按交易对实现分片锁或独立下单队列。

#### P1-2.2.4 线程池/连接池无策略级隔离

- **证据**：
  - `PositionService._close_retry_executor`（第1028行）：全局单例 `ThreadPoolExecutor(max_workers=10)`，所有策略的平仓重试共享。
  - `ShadowStrategyEngine._log_writer_executor`（第243行）：`max_workers=1`，仅用于影子日志。
  - DuckDB连接池（`ds_db_connection.py`）：全局连接池，所有策略共享。
- **缺陷**：如果某策略触发大量平仓重试，占满线程池，其他策略的平仓重试将排队等待。
- **建议**：关键路径的线程池应按策略分组或使用优先级队列。

#### P2-2.2.5 影子策略隔离验证有但未自动化

- **证据**：`verify_shadow_isolation()` 方法（第1085-1100行）提供了完整的隔离验证（检查 `is_shadow_mode`、独立账户、不使用 OrderService 等），但此方法**仅在测试中手动调用**（`test_crack_validation.py` 第199行），未集成到启动检查或健康检查中。
- **建议**：在策略启动时自动运行隔离验证，失败则拒绝启动影子策略。

## 三、重试与退避（Retry & Backoff）

### 3.1 重试策略全景

| 模块 | 操作 | 重试次数 | 退避策略 | 幂等性 | 队列保护 |
|:---|:---|:---|:---|:---|:---|
| OrderService | 下单 `_retry_send_order` | 3次 | 指数 (2^n秒) | ❌ 无幂等 | ❌ |
| OrderService | 追单 `_chase_reorder` | 3次 (MAX_CHASE_RETRIES) | 线性（逐tick加价） | ❌ 无 | ❌ |
| PositionService | 平仓 `_schedule_close_retry` | 3次 | 指数 (0.1×2^n) | 部分（_closing标志） | 线程池(10) |
| SubscriptionManager | 订阅 `_retry_queue` | 3次 (max_retries) | 指数/线性/随机抖动 | WAL去重 | 队列(500) |
| StorageCore | Batch写入 | 3次 (_batch_retry_count) | 固定延迟(0.1-1.0s) | ❌ | 丢弃后WAL |
| ParamsService | 配置加载 | 3次 | 固定间隔(1.0s) | ✅ 配置文件 | ❌ |
| shared_utils | `retry_with_limit` | 可配置(3) | 指数(×2.0) | N/A（通用工具） | ❌ |

---

### 3.2 发现与分级

#### P0-3.2.1 下单重试缺幂等机制——重复成交风险

- **证据**（R15 2.1.1交叉验证 + 本报告深入分析）：
  ```
  order_service.py:249  → send_result = self._execute_send_order(order, retry_count=0)
  order_service.py:251  → send_result = self._retry_send_order(order)  # 生成新order_id
  
  order_service.py:321-335 → _retry_send_order:
      for retry in range(1, self._max_send_retries + 1):
          delay = 2 ** retry                              # 指数退避 ✅
          result = self._execute_send_order(order, retry_count=retry)
          # 每次调用 _execute_send_order 内部第275行调用 _generate_order_id()
          # 生成全新的 order_id — 没有 idempotency_key
  ```

- **失败链路**（详细）：
  1. 策略生成信号 → `send_order("IF2506", BUY, 1, 3800.0)` 
  2. `_generate_order_id()` → `order_id = "ORD-001"` → `_execute_send_order(order)`
  3. 平台接收订单 → 返回 HTTP 200 → 但网络在响应回传时中断
  4. `_platform_insert_order` 抛出 `ConnectionError`
  5. `_execute_send_order` 返回 `None`（第310行 `return None`）
  6. `_retry_send_order(order)` 触发 → `_generate_order_id()` → **`order_id = "ORD-002"`**
  7. 平台侧 ORD-001 **已经成交**（网络中断不影响平台撮合）
  8. ORD-002 也成交 → **2手IF2506多头，而非预期的1手** → 双倍风险暴露

- **资金损失量化**：IF2506 1手 ≈ 市值约114万（3800×300），双倍暴露 → 多出114万风险敞口。
- **所需修复**：增加 `client_order_id` / `idempotency_key` 字段，平台侧用此字段去重；或在重试前先查询平台订单状态。

#### P1-3.2.2 追单不检查原订单实际成交状态

- **证据**：
  ```
  order_service.py:547-575 → check_pending_orders():
      562: if elapsed > self._order_timeout_seconds:     # 超过5秒超时
      564:     timeout_orders.append((order_id, dict(order)))
      572: self.cancel_order(order_id)                   # 取消原订单
      573-575: if retry_count < self._max_chase_retries:
                self._chase_reorder(current_order, retry_count + 1)  # 追单
  ```
- **缺陷**：`cancel_order` 调用后**不等待**取消确认（异步特性），立即追单。如果原订单实际上已部分成交（`PARTIAL_CANCEL`，第540行），追单将产生超额持仓。
- **风险场景**：原订单2手，已成交1手 → 取消未能撤销剩余1手 → 追单再下2手 → 总持仓3手而非预期2手。
- **建议**：追单前验证原订单取消状态，追单量应扣除已成交量。

#### P1-3.2.3 撤单操作无重试机制

- **证据**：
  ```
  order_service.py:443-448 → cancel_order():
      445: self._platform_cancel_order(platform_id)  # 无重试
      447: return True
      448: return False  # 异常时直接返回False
  ```
- **影响**：断路器触发时（`_cancel_pending_on_circuit_breaker`，第2273行）调用 `cancel_all_pending`（第459行），如果撤单API瞬时不可用（如网络闪断），所有待撤订单保持活跃 → 后续市场波动可能导致意外成交。
- **建议**：撤单失败时至少重试1-2次，间隔100ms。

#### P1-3.2.4 存储写入重试溢出后的数据丢失

- **证据**：
  ```
  storage_core.py:437-442 → _batch_retry_count > 3:
      440: logging.critical("[DATA_LOSS][%s] batch重试超限，丢弃%d条数据", name, lost_count)
      441: batch.clear()
      442: self._batch_retry_count = 0
  ```
- **风险**：DuckDB写入连续失败4次 → 丢弃整批数据（最多5000条tick或50条K线）→ 回测数据不完整 → 信号失真。虽然有 `[DATA_LOSS]` 日志，但无告警回调。
- **缓解因素**：写入前数据已在队列中缓冲（`_tick_shard_queues` 容量约10000），溢出WAL spill机制（第125-128行）提供额外保护。

## 四、优雅降级路径（Graceful Degradation）

### 4.1 降级机制全景

## 五、级联故障防护（Cascading Failure Protection）

### 5.1 关键故障链分析

```
系统级故障传播链：

   平台下单API阻塞（无超时）
        ↓
   _platform_insert_order 永久阻塞
        ↓
   _trading_lock 被持有
        ↓
   ├─ check_position_risk 无法获取锁 → 持仓风控停滞 → 止损无法执行 → 亏损扩大
   ├─ execute_option_trading_cycle 下次触发 → 检测到锁被占用 → 静默跳过（无告警）
   └─ 所有其他策略的交易请求排队等待
        ↓
   系统级交易停摆 + 风控失效 = 灾难场景
```

---

### 5.2 发现与分级

#### P0-5.2.1 共享锁阻塞→级联风控失效（核心致命链路）

- **证据**（R15 3.1.2和6.1.1交叉验证，本报告深入追踪完整链路）：
  ```
  strategy_core_service.py:80  → _trading_lock = threading.RLock()  # 【共享锁】
  strategy_core_service.py:280 → with self._trading_lock:            # 【交易周期持有】
  strategy_core_service.py:415 → with self._trading_lock:            # 【持仓风控持有】
  # 两操作竞争同一把锁！
  
  order_service.py:299 → self._platform_insert_order(**filtered_params)  # 【无超时！】
  # 此调用在 _trading_lock 持有期间执行
  ```
- **失败链路（完整时序）**：
  1. T=0s: 交易定时器触发 `execute_option_trading_cycle()` → 获取 `_trading_lock`
  2. T=0.1s: 信号生成完毕，调用 `order_service.send_order()` → `_platform_insert_order()`
  3. T=0.2s: 平台API因网络/撮合延迟，进入阻塞状态（无超时）
  4. T=0.5s: `check_position_risk()` 定时器触发 → 尝试获取 `_trading_lock` → **阻塞等待**
  5. T=1s-∞: 持仓风控停滞——持仓时间止损、回撤止损、EOD平仓全部无法执行
  6. 所有策略的新交易请求也排队在 `_trading_lock` 上
- **最坏情况量化**：IF2506持仓浮亏从-2%扩大到-5%（触发硬止损阈值5%）但止损信号无法发出 → 最终亏损-8%才人工发现 → 3%超额亏损 × 合约面值114万 = 3.42万资金损失。
- **建议**：① `_platform_insert_order` 必须设置超时（如5秒）；② 持仓风控使用独立的锁，不与交易周期共享。

#### P0-5.2.2 下单/撤单API无超时保护

- **证据**（R15 6.1.1/6.1.2交叉验证）：
  ```
  order_service.py:299 → self._platform_insert_order(**filtered_params)  # 无超时
  order_service.py:445 → self._platform_cancel_order(platform_id)         # 无超时
  # 全局搜索 threading.Timer、signal.alarm、concurrent.futures.wait、async_timeout
  # 在 order_service.py 中均未命中
  ```
- **分析**：Python层面实现超时需要额外机制（`threading.Timer` + `_stop_event` 或 `concurrent.futures.ThreadPoolExecutor` + `future.result(timeout=...)`）。当前架构中 `_platform_insert_order` 是直接调用传入的函数对象，完全依赖调用者提供超时。
- **建议**：使用 `ThreadPoolExecutor` 包装平台API调用，通过 `future.result(timeout=N)` 实现超时。

#### P1-5.2.3 存储队列阻塞反向传播

- **证据**（R15 7.1.1交叉验证）：
  ```
  storage_core.py:105 → self._tick_shard_queues = [queue.Queue(maxsize=_shard_cap) for ...]
  # 入队使用 put()（阻塞）而非 put_nowait()
  # 生产者是平台tick回调线程
  # 如果写入慢于到达 → 队列满 → tick回调线程阻塞 → 后续tick堆积
  ```
- **失败链路**：DuckDB写入因磁盘高负载变慢 → 队列逐渐填满 → `put()` 阻塞调用线程 → 该线程是平台tick回调线程 → 新的tick数据无法被处理 → 所有依赖tick的策略（特别是HFT）失去数据源。
- **对比**：`subscription_manager.py` 的重试队列在满时使用 `dropped_count` 计数并 WAL 保护（第555-564行），不阻塞生产者。
- **建议**：使用 `put_nowait()` + 丢弃计数 + WAL保护，或将tick回调线程与写入线程解耦（使用无界队列或背压通知）。

#### P1-5.2.4 交易周期跳帧无告警无降频

- **证据**（R15 7.1.2交叉验证）：
  ```
  strategy_core_service.py:280-281 → 
      if not self._trading_lock.acquire(blocking=False):
          return  # 上次周期还在执行，静默跳过
  ```
- **缺陷**：没有跳帧计数，没有告警。如果连续10次被跳过（300秒），说明系统严重过载，但运维人员无从得知。
- **建议**：增加跳帧计数器，连续N次跳过后触发告警并自动降频。

#### P1-5.2.5 EventBus虽然隔离了callback异常但无压降机制

- **证据**：
  ```
  event_bus.py:428-437 → _invoke_callback:
      try:
          callback(event)
      except Exception as e:
          logging.error(...)  # 隔离异常 ✅
  ```
- ✅ 良好设计：单个订阅者异常不影响其他订阅者（R5-E-06修复）。
- ❌ 缺陷：没有慢回调检测。如果某个callback执行超过1秒（如复杂的数据库查询），EventBus没有告警也不会跳过该callback。
- **建议**：增加callback执行时间监控，超过阈值发出告警。

#### P2-5.2.6 无请求队列深度监控

- **证据**：`storage_core.py` 有 `_queue_stats`（第130-133行）追踪 `total_received`/`total_written`/`drops_count`，但没有暴露 `current_queue_depth` 到健康检查API。`OrderService` 没有请求队列概念（直接调用API）。

### 7.2 发现与分级

#### P0-7.2.1 下单API完全无超时——系统级阻塞根因

- **证据**（同R15 6.1.1，本报告级联故障链5.2.1中详细分析了影响）：
  ```
  order_service.py:299 → self._platform_insert_order(**filtered_params)
  # 没有任何 timeout 参数、threading.Timer、signal.alarm、future.result(timeout=...)
  ```
- **影响**：已在§5.2.1详细分析——是系统最致命的单点故障源。
- **建议**：将平台API调用包装在 `ThreadPoolExecutor` 中，通过 `future.result(timeout=5.0)` 实现5秒超时。

#### P0-7.2.2 撤单API无超时

- **证据**：`order_service.py:445` 无超时保护。
- **影响**：断路器触发时（最需要快速撤单的时刻），如果 `_platform_cancel_order` 阻塞，所有待撤订单无法撤销。
- **建议**：同下单API，添加5秒超时。

#### P1-7.2.3 DuckDB查询无语句级超时

- **证据**（R15 6.1.3交叉验证）：
  ```
  # DuckDB支持 statement_timeout 设置，但代码中未见使用
  # data_service.py 和 ds_db_connection.py 中无相关配置
  ```
- **风险场景**：大时间范围的历史数据查询（如加载3年5分钟K线），可能执行数十秒。
- **建议**：在 DuckDB 连接上设置 `SET statement_timeout = '30000'`（30秒）。

#### P1-7.2.4 长时间无tick的看门狗不完整

- **证据**：
  ```
  storage_core.py:1445 → _is_kline_missing() K线缺失检测
  # 针对K线有检测，但对tick流没有整体的"无数据"看门狗
  
  diagnosis_probe.py:440 → ContractWatchSummary 有合约维度tick超时
  # 但仅针对预设的14个监控合约，且有timeout限制
  ```
- **缺陷**：全局tick流的停滞（如平台行情推送断开）没有独立的看门狗检测。
- **缓解**：HealthCheckAPI（`health_check_api.py` 第96-100行）通过 `strategy_heartbeat_interval_seconds` 间接检测，但依赖系统调用链正常。
- **建议**：增加独立的 tick_heartbeat 看门狗线程，监控上次收到tick的时间差。

#### P1-7.2.5 策略heartbeat机制仅存在于配置中

- **证据**：
  ```
  config_service.py:108 → strategy_heartbeat_interval_seconds: int = 30
  config_service.py:109 → strategy_max_heartbeat_failures: int = 3
  # 这两个配置在 HealthCheckAPI 中被消费（health_check_api.py:96-100）
  # 但系统中没有主动的 heartbeat 发送机制
  ```
- **缺陷**：配置定义了heartbeat参数，但策略核心没有定期发送heartbeat。HealthCheckAPI 依赖 `update_last_signal_time()`（`ProductionQuantSystem.py:219`）的被动触发。
- **建议**：策略核心添加主动heartbeat发送线程，或确认被动方式足够。

#### P2-7.2.6 订单超时检测间隔偏大

- **证据**：
  ```
  order_service.py:78 → CLEANUP_INTERVAL_SECONDS = 300  # 5分钟才检查一次
  order_service.py:75 → DEFAULT_ORDER_TIMEOUT_SECONDS = 5.0  # 5秒超时
  ```
- **分析**：订单可能在5秒后已超时，但 `check_pending_orders()` 每300秒才运行一次。虽然超时后取消+追单可以纠正，但300秒的间隔意味着超时订单在市场中停留长达5分钟。
- **建议**：将清理间隔降低到30秒（与交易周期一致），或使订单超时检测事件驱动。

#### P2-7.2.7 scheduler_service的Job超时设计良好

- **证据**：
  ```
  scheduler_service.py:413-456 → _run_job_with_timeout:
      在独立线程执行 → thread.join(timeout=job.timeout) → 
      超时则设置 cancel_event + raise TimeoutError
  ```
- ✅ 良好设计：线程级超时 + 取消事件 + 异常传播。可作为其他超时实现的参考模式。

## 八、错误隔离（Error Isolation）

### 8.1 隔离机制全景

| 隔离点 | 机制 | 效果 | 文件:行 |
|:---|:---|:---|:---|
| EventBus订阅者 | try/except per callback | ✅ 单订阅者异常不影响其他 | event_bus.py:350,428 |
| 跨策略风险检查 | exception→fail-safe BLOCK | ✅ 异常时阻断交易 | ecosystem.py:834-836 |
| 状态切换递归 | 深度计数(max=3) | ✅ 防止无限递归 | ecosystem.py:506-508 |
| tick处理 | 共享_trading_lock | ⚠️ 可能阻塞 | strategy_core_service.py:280 |
| 订单处理 | 共享_order_service锁 | ⚠️ 可能阻塞 | order_service.py:90 |
| 存储写入 | 分片独立writer线程 | ✅ 单分片故障不影响其他 | storage_core.py:105-107 |
| 撤单循环 | 独立try/except per order | ✅ 单个失败继续 | order_service.py:459-472 |
| 模式传播 | 独立try/except per 下游 | ✅ 单下游失败不影响其他 | ecosystem.py:464-496 |

### 8.2 发现与分级

#### P0-8.2.1 tick处理失败可能级联到整个交易链

- **证据**：
  ```
  strategy_core_service.py:1124 → onTick(copy_data):
      # onTick 中的错误传播路径：
      # tick → HMM推断 → 信号生成 → 风控 → 下单
      # 如果 onTick 中的某个环节抛出未捕获异常：
      #   - onTick 本身没有全局 try/except
      #   - 异常向上传播到BaseStrategy的tick分发器
      #   - 后续tick的处理取决于框架的实现（未知行为）
  ```
- **分析**：虽然 `onTick` 内部分操作有 try/except（如HMM推断、信号生成），但**整体 onTick 没有全局异常处理**。如果某个未预见的异常类型逃逸，可能中断整个tick处理链。
- **对比**：`execute_option_trading_cycle` 有全局 try/except（第335-346行），但 `onTick` 没有。
- **建议**：在 `onTick` 的最外层添加全局 try/except，确保单个tick的异常不影响后续tick。

#### P1-8.2.2 EventBus异常隔离完善但无死信队列

- **证据**：
  ```
  event_bus.py:350 → # R5-E-06修复: 单个订阅者异常不影响其他订阅者
  event_bus.py:428-437 → try/except per callback ✅
  ```
- ✅ 良好设计：已实现 per-callback 异常隔离，包括完整堆栈记录（R5-E-07修复）。
- ❌ 缺陷：失败的callback事件被丢弃，没有死信队列(dead letter queue)用于后续重放或人工审查。
- **建议**：增加死信队列（有界，如maxlen=1000），记录失败的事件和异常。

#### P1-8.2.3 单订单失败影响其他订单的范围有限但存在竞争

- **证据**：
  ```
  order_service.py:291-328 → execute_option_trading_cycle 批量下单：
      for t in targets: ... send_order(...)
      # 每个订单独立调用 send_order，有独立的 try/except
      # 但共享 _trading_lock，一个订单阻塞会影响后续订单 ✅/❌
  ```
- **分析**：每个订单有独立异常处理，但如果第一笔订单的 `_platform_insert_order` 阻塞（无超时，见§7.2.1），后续订单全部排队等待。这不是"隔离"问题，而是"超时"问题的级联效应。
- **建议**：订单提交应改为异步模式，或至少设置超时。

#### P1-8.2.4 StorageCore分片写入隔离良好但存在数据丢失路径

- **证据**：
  ```
  storage_core.py:105 → _tick_shard_queues = [queue.Queue(maxsize=...) for _ in range(N)]
  # 每个分片独立队列 + 独立writer线程
  ```
- ✅ 良好设计：分片间的数据完全隔离，一个分片的写入异常不影响其他分片。
- ❌ 缺陷：writer线程异常退出后，没有自动重启机制（虽然 `_batch_retry_count` 提供了3次重试，但线程级崩溃需要手动检测）。

#### P2-8.2.5 策略异常不传播到全局事件循环（但有锁竞争隐患）

- **证据**：策略的核心逻辑在 `onTick` 和 `execute_option_trading_cycle` 中执行，两者都是框架回调。虽然框架本身可能捕获策略异常，但**锁竞争**（§5.2.1）意味着阻塞而非异常传播才是真正的隔离问题。
- **建议**：见§5.2.1的超时修复建议。

#### P2-8.2.6 策略间资金账户无隔离

- **证据**：
  ```
  # 所有策略共享同一个资金账户（通过 RiskService / PositionService）
  # 只有影子策略（ShadowStrategyEngine）有独立的paper_account
  ```
- **分析**：如果S1（HFT）出现异常大额亏损，直接消耗全局资金，S2-S6的可用资金同步减少。策略间没有独立的子账户或风险预算隔离。
- **对比**：StrategyEcosystem 的 `StrategySlot.capital_allocation` 提供了逻辑上的资金分配，但没有强制执行。
- **建议**：在 PositionService 层面按策略ID强制执行资金限额。
