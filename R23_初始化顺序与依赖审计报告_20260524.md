### 维度1：服务启动依赖图

#### D1-P0-01 | StrategyCoreService.__init__ 中直接实例化 SignalService（非单例），导致重复初始化风险
- **文件**：`strategy_core_service.py:112`
- **发现**：`self._signal_service = SignalService(event_bus=event_bus)` 直接构造，而系统中另有 `get_signal_service()` 单例工厂 (`signal_service.py:735`)。同一进程中可能存在**两个不同**的 SignalService 实例，一个挂载在 StrategyCoreService 上，另一个由其他模块通过 `get_signal_service()` 获取。
- **后果**：两个实例各自的 `_signal_history`、`_cooldown_times`、`_stats` 独立维护，策略生态系统的 PLR 过滤状态（通过 `get_signal_service().enable_plr_filter()`）不会反映到策略核心持有的实例上。
- **半初始化场景**：`get_signal_service()` 被策略生态系统调用后启用 PLR 过滤，但 `self._signal_service`（非单例实例）的 PLR 过滤仍为 false。
- **分级**：**P0**

#### D1-P1-01 | StrategyEcosystem.__init__ 隐式依赖 ModeEngine.create_engine 和 4 个下游服务
- **文件**：`strategy_ecosystem.py:453-495`
- **发现**：`_propagate_capital_scale()` 在 `__init__` 末尾调用，依赖 `ModeEngine.create_engine()` → 成功后 switch_mode → 失败后 fallback 四个服务 (`get_risk_service`, `get_signal_service`, `get_state_param_manager`, `get_box_spring_strategy`)。这些依赖在 `__init__` 中**未被显式声明**为参数。
- **后果**：若 ModeEngine 在 StrategyEcosystem 构造时不可用（YAML 文件缺失、tvf_params 损坏），fallback 路径中四个单例服务的初始化会级联发生。
- **分级**：**P1**

#### D1-P1-02 | risk_service.py 中 get_shadow_engine 函数名错误，SeverityScoreCalculator 的 alpha_decay 永远跳过
- **文件**：`risk_service.py:1390-1391` → 导入 `get_shadow_engine`
- **依赖关系**：`risk_service.py` → `shadow_strategy_engine.py`
- **发现**：`shadow_strategy_engine.py` 仅定义了 `get_shadow_strategy_engine`（行1297），**不存在** `get_shadow_engine` 函数。此 import 每次执行都会抛出 `ImportError`，但被 `try/except Exception` 捕获（行1394），导致 `alpha_decay` 评分从 `_alpha_ratio=None` 走到 `pass` 分支，永远使用空值。
- **后果**：11维风险评分中的 alpha_decay 维度的计算永久跳过，影子引擎 Alpha 衰减无法反馈到风险评分。
- **分级**：**P1**

#### D1-P1-03 | StrategyCoreService.__init__ 在 super().__init__ 返回前已启动子组件初始化
- **文件**：`strategy_core_service.py:83-105`
- **发现**：在 `__init__` 内部通过 `from ali2026v3_trading.strategy_scheduler import StrategyScheduler` 初始化调度器管理器、调用 `_init_historical_kline_mixin()` 和 `_init_tick_handler_mixin()`。这些调用发生在 `Strategy2026` 构造 `StrategyCoreService` 过程中，而此时 `self.strategy_id` 刚被赋值，`bind_platform_apis` 尚未调用，`_api_ready=False`。
- **后果**：调度器、历史K线 Mixin 和 Tick处理 Mixin 在 API 就绪前即被初始化。若 Mixin 中 `super().__init__()` 链有期望 `self.subscribe` 为 callable 的逻辑，将失败。
- **分级**：**P1**

#### D1-P2-01 | ServiceContainer 初始化在 onStart 末尾而非开始
- **文件**：`strategy_core_service.py:918-928`
- **发现**：`ServiceContainer.create_and_register_all_services()` 在 `Strategy2026.onStart` 第5步（core_start之后），而非第一步。此时 StrategyCoreService.start() 已完成，意味着 start() 内部用到的服务（EventBus、Config 等）不是通过 ServiceContainer 注册的。
- **后果**：ServiceContainer 的依赖层校验和依赖图（L0→L3）在核心服务已启动后才执行，失去"依赖未就绪则阻塞启动"的防护能力。
- **分级**：**P2**

#### D1-P2-02 | 日志初始化在多个层级重复调用
- **发现**：`ProductionQuantSystem.main:431` 调用 `logging.basicConfig()`，`StrategyCoreService.on_init:567-570` 强制修改 root_logger level，`Strategy2026` 的各方法中通过 `logging.info()` 输出。同一进程内多次 basicConfig 调用（仅首次生效），后续调用被忽略但不会报错，可能导致日志级别不一致的预期偏差。
- **分级**：**P2**

### 维度2：循环依赖检测

#### D2-P1-01 | strategy_ecosystem.py ↔ shadow_strategy_engine.py 隐式循环依赖（通过 mode_engine 中转）
- **文件**：`strategy_ecosystem.py:453` → `mode_engine.py:1077` → `strategy_ecosystem.py`
- **发现**：`StrategyEcosystem._propagate_capital_scale()` 调用 `ModeEngine.create_engine()` → `ModeEngine` 的 `switch_mode` 方法（行1077）调用了 `from ali2026v3_trading.strategy_ecosystem import get_strategy_ecosystem`。这意味着在 StrategyEcosystem 构造过程中（`_propagate_capital_scale` 在 `__init__` 末尾调用），ModeEngine 可能反过来获取还未完成初始化的 StrategyEcosystem。
- **后果**：`get_strategy_ecosystem()` 的全局变量 `_ecosystem` 在 `StrategyEcosystem.__init__` 执行完毕后才被赋值（行1401），因此 ModeEngine 调用 `get_strategy_ecosystem()` 时 `_ecosystem` 仍为 None，导致**新建第二个** StrategyEcosystem 实例。
- **分级**：**P1**

#### D2-P1-02 | position_service.py ↔ risk_service.py 双向懒导入
- **文件**：
  - `risk_service.py:91` → `from ali2026v3_trading.position_service import PositionLimitConfig`
  - `position_service.py:53-54` → `RiskService = None`（惰性模板导入）
- **发现**：`risk_service.py` 在模块顶层导入 `PositionLimitConfig`，而 `position_service.py` 通过 `RiskService = None` + 惰性 `from ... import RiskService` 避免顶层循环导入。但 `position_service.py:1317` `get_position_service()` 接受 `risk_service` 参数，构造时可传入 `RiskService` 实例，形成构造时的循环引用。
- **后果**：若第三方代码构造 `PositionService(risk_service=RiskService(...))`，且 RiskService 的 `position_manager` 参数回指同一个 PositionService，则形成对象级循环引用，导致 GC 无法回收。
- **分级**：**P1**

#### D2-P2-01 | main.py 的 `__getattr__` 延迟导入掩盖模块间循环依赖
- **文件**：`main.py:41-64`
- **发现**：通过 `__getattr__` 将 `Strategy2026`, `ServiceContainer`, `OrderService`, `RiskService` 等的导入延迟到首次属性访问。这打破了真正的循环导入链，但掩盖了模块之间底层存在循环引用的事实。
- **后果**：不影响功能但降低代码可维护性 — 新增模块时开发者不清楚底层依赖关系，可能引入新的循环依赖。
- **分级**：**P2**

### 维度3：半初始化风险

#### D3-P0-01 | strategy_core._api_ready 未在 __init__ 结束时强制检查即被使用
- **文件**：`strategy_core_service.py:122` → 初始化为 `False`
- **发现**：`_api_ready` 标记 API 是否就绪，仅在 `Strategy2026.onStart:856` 和 `Strategy2026.onInit:1007` 调用 `bind_platform_apis()` 后间接设为 True。但 `Strategy2026.__init__:713` 中构造 `StrategyCoreService` 时，`self.strategy_core` 已存在且 `subscribe=None`、`unsubscribe=None`、`get_instrument=None`。
- **后果**：在 `onStart` 调用 `strategy_core.start()` 前，若有任何外部代码触发 `strategy_core` 的方法（如健康检查、盘前数据查询），`self.subscribe` 为 None 会导致 `AttributeError('NoneType' object has no attribute ...)`。
- **分级**：**P0**

#### D3-P0-02 | get_health_status() 中 risk_dashboard_status 在赋值前被引用
- **文件**：`strategy_core_service.py:447-512`
- **发现**：第 447 行 `components["risk_dashboard"] = risk_dashboard_status` 使用 `risk_dashboard_status`，但该变量在第 498 行才首次赋值（`risk_dashboard_status = "UNAVAILABLE"`）。Python 不会在编译时报错，但运行时会触发 `UnboundLocalError`（因为第 447 行读取了一个后文才赋值的局部变量）。
- **后果**：调用 `get_health_status()` 时必定抛出 `UnboundLocalError`，整个健康检查崩溃。
- **分级**：**P0**

#### D3-P0-03 | StrategyCoreService.__init__ 在 Mixin 多层继承中部分属性未就绪即调用 Mixin 方法
- **文件**：`strategy_core_service.py:60-177`
- **发现**：`StrategyCoreService` 继承 `_LifecycleMixin, _InstrumentHelperMixin, HistoricalKlineMixin, TickHandlerMixin`，这些 Mixin 中 `__init__` 未显式调用 `super().__init__()`，且各自的初始化方法（`_init_historical_kline_mixin`, `_init_tick_handler_mixin`）在 `StrategyCoreService.__init__` 中被调用。但 Python MRO 下多个 Mixin 可能期望某些属性已由其他 Mixin 设置。
- **后果**：Mixin 之间的属性依赖缺乏显式检查，可能在特定 MRO 顺序下访问未初始化的属性导致 `AttributeError`。
- **分级**：**P0**

#### D3-P1-01 | StrategyEcosystem 在 __init__ 中设置 self._mode_engine=None，但 _propagate_capital_scale 中才赋值
- **文件**：`strategy_ecosystem.py:306`（`self._mode_engine = None`）→ 行 453 才调用 `ModeEngine.create_engine()`
- **发现**：在 `_propagate_capital_scale()` 之前（行 361-386 治理检查器初始化），`self._mode_engine` 为 None。虽然有 try/except 保护，但治理检查器若依赖 ModeEngine（如 E13ShadowStrategyCollusionDetector），将在 ModeEngine 未就绪时被初始化。
- **分级**：**P1**

#### D3-P1-02 | self._shadow_engine, self._strategy_ecosystem, self._state_param_manager 通过 getattr(self, '_xxx', None) 惰性访问
- **文件**：`strategy_core_service.py:375, 470, 481, 532, 596`
- **发现**：这些字段在 `__init__` 中初始化为 None，使用时通过 `getattr(self, '_shadow_engine', None)` + `if se is None: from xxx import get_xxx; se = get_xxx()` 模式惰性初始化。每次方法调用都重复检查+导入，且没有统一的"已初始化"标志。
- **后果**：惰性导入的开销在每条 tick 处理路径上重复发生（`_feed_shadow_engine` 每 tick 调用一次），构成高频路径上的隐式性能损耗。
- **分级**：**P1**

#### D3-P1-03 | execute_option_trading_cycle 中 BoxSpringStrategy 在交易周期内每次构造新实例
- **文件**：`strategy_core_service.py:332-336`
- **发现**：`bss = getattr(self, '_box_spring_strategy', None)` 检查实例是否存在，若为 None 则 `bss = BoxSpringStrategy()` 构造新实例。但 `get_box_spring_strategy()` 全局单例函数（`box_spring_strategy.py:169`）可能已经返回了另一个实例，导致同一 tick 处理路径中存在两个 BoxSpringStrategy 实例。
- **后果**：两个实例各自维护状态，`on_tick` 只更新其中之一的内部状态，另一个的状态永远不会被更新。
- **分级**：**P1**

#### D3-P1-04 | ProductionQuantSystem.__init__ 中 crm 在 update_tick 中延迟初始化
- **文件**：`ProductionQuantSystem.py:136`（`self._crm = None`）→ 行 187-189 惰性 import
- **发现**：`self._crm` 初始化为 None，在第一次 `update_tick` 时通过 `importlib.import_module` 加载。若 `cycle_resonance_module` 导入失败，每条 tick 都会尝试重新导入。
- **后果**：高频路径上的重复 importlib 调用，且失败时静默跳过（`logging.debug`），数据丢失不可追踪。
- **分级**：**P1**

#### D3-P2-01 | StrategyCoreService 的 initialize() 仅委派给 on_init()，两者语义可合并
- **文件**：`strategy_lifecycle_mixin.py:1202`
- **发现**：`def initialize(self, params=None) -> bool: return self.on_init(params=params)`，又多了一层代理调用。
- **分级**：**P2**

#### D3-P2-02 | StrategyScheduler.initialize() 在 StrategyCoreService.__init__ 中仅构造，未调用 initialize
- **文件**：`strategy_core_service.py:83-84`
- **发现**：`self._scheduler_manager = StrategyScheduler()` 仅构造函数，`StrategyScheduler.initialize()`（启动 APScheduler BackgroundScheduler）在 `_init_scheduler()` 中调用（`strategy_lifecycle_mixin.py:594` → `_init_scheduler()`）。若 `_init_scheduler` 调用时机晚于其他试图使用调度器的代码，将遇到 `self._scheduler` 为 None。
- **分级**：**P2**

### 维度4：单例初始化竞争

#### D4-P0-01 | StrategyEcosystem 的 get_strategy_ecosystem 在 __init__ 完成前可能被 ModeEngine 再次调用
- **文件**：`strategy_ecosystem.py:1396-1402`
- **发现**：`get_strategy_ecosystem()` 使用双重检查锁定模式创建 `StrategyEcosystem()`。但构造过程中（`_propagate_capital_scale` → `ModeEngine.create_engine` → `switch_mode` → `get_strategy_ecosystem()`），`_ecosystem` 仍为 None（因为赋值在 `__init__` 返回后），导致**构造第二个 StrategyEcosystem 实例**。
- **后果**：两个 StrategyEcosystem 实例共存，各自的策略槽位、资金路由、交易记录独立，A 实例的状态切换不反映到 B 实例，导致互斥规则（`check_mutual_exclusion`）在不同实例间无效。
- **分级**：**P0**

#### D4-P1-01 | get_risk_service 单例的 params 仅首次有效，后续调用忽略参数
- **文件**：`risk_service.py:2278-2288`
- **发现**：双重检查锁定后，仅 `_risk_service_instance is None` 时使用传入的 `params`（或空字典 `{}`）构造。后续调用传入不同的 `params` 会被静默忽略，返回首次构造的实例。
- **后果**：若 `execute_option_trading_cycle` 中 `rs = get_risk_service(None)` 与 `_propagate_capital_scale` 中的 `rs = get_risk_service()` 期望不同的 params 初始化，后者会被忽略。
- **分级**：**P1**

#### D4-P1-02 | ConfigService 使用 __new__ 单例但 __init__ 受 _init_lock 保护不完整
- **文件**：`config_service.py:246-270`
- **发现**：`__new__` 使用 `cls._init_lock` 保护双重检查，`__init__` 使用 `type(self)._init_lock` + `_initialized` 标志。但 `__init__` 内的 `__new__` 和 `__init__` 之间存在 **非原子间隙**：线程 A 在 `__new__` 创建实例后释放 `_init_lock`，线程 B 的 `__new__` 返回同一个实例，然后线程 A 和 B **同时**进入 `__init__`。虽然后续的 `_init_lock` + `_initialized` 检查防止双重初始化，但存在竞态窗口。
- **分级**：**P1**

#### D4-P2-01 | SingletonManager 注册方式与模块级单例并存，存在两套单例机制
- **文件**：`ProductionQuantSystem.py:136` → `_singleton_mgr.register()`，`risk_service.py:2260` → 模块级全局变量
- **发现**：`ProductionQuantSystem` 通过 `SingletonManager.get_instance().register()` 注册，而 `RiskService` 等通过模块级 `_risk_service_instance` 全局变量实现单例。两套单例机制并存，`shutdown_all()` 仅清理 `SingletonManager` 注册的实例。
- **分级**：**P2**

### 维度5：配置加载时序

#### D5-P1-01 | ConfigService.__init__ 加载 config.json 失败时 raise RuntimeError，阻止整个进程启动
- **文件**：`config_service.py:286-287`
- **发现**：`_load_config_file()` 失败时 `raise RuntimeError(f"Config load failed: {e}") from e`，没有降级路径。若 `config.json` 损坏或不存在，整个 ConfigService 构造失败，进而 `Strategy2026.__init__:710` 构造 `StrategyParams` 时触发 `ConfigService()` 失败，导致策略无法启动。
- **后果**：回测环境或 docker 部署时，若 `config/config.json` 不存在，系统完全无法启动。
- **分级**：**P1**

#### D5-P1-02 | 命令行参数覆盖仅在 ProductionQuantSystem.main 中实现，Strategy2026 启动路径未使用
- **文件**：`ProductionQuantSystem.py:361-395`（`_build_config_from_args`）
- **发现**：`ProductionQuantSystem.main` 支持 `--config`、`--symbols`、`--exchange` 等命令行参数覆盖，合并逻辑为：ConfigService 默认 → JSON 文件覆盖 → 命令行覆盖。但 `Strategy2026.onStart` 启动路径**不使用**此逻辑，而是通过 `_ensure_runtime_config_loaded()` 从 `get_cached_params()` + `self._runtime_config` 合并。
- **后果**：两条启动路径的配置优先级不一致。
- **分级**：**P1**

#### D5-P2-01 | get_cached_params 缓存 TTL 300秒，启动期间可能读到过期缓存
- **文件**：`config_params.py:22` → `CACHE_TTL = 300.0`
- **发现**：缓存有效期为 300 秒（5分钟），若两次启动间隔不足 5 分钟且参数表在外部修改，第二次启动仍读到旧缓存。
- **分级**：**P2**

#### D5-P2-02 | SignalService.__init__ 中 ConfigService 构造两次
- **文件**：`signal_service.py:102-113` + `signal_service.py:128-135`
- **发现**：两个独立的 try/except 块中各自调用 `ConfigService()`。虽然 ConfigService 是单例，第二次构造仅返回同一实例，但日志中的双重初始化信息可能令人困惑。
- **分级**：**P2**

### 维度6：数据源就绪检测

#### D6-P0-01 | on_start 中订阅启动前仅检查 self.subscribe 是否 callable，未检查 data_service 是否已连接
- **文件**：`strategy_lifecycle_mixin.py:747-752`
- **发现**：`_start_platform_subscribe_async` 仅检查 `callable(self.subscribe)` 即启动平台订阅，但未验证 DataService 的 DuckDB 连接是否存活、RealTimeCache 是否初始化成功。
- **后果**：若 DataService 初始化时 DuckDB 连接失败（只记录了 warning），订阅启动后 tick 到达时 `ds_realtime_cache.py` 的写入操作将失败。
- **分级**：**P0**

#### D6-P1-01 | DataService._initialize 中 RealTimeCache 初始化失败仅降级为 warning
- **文件**：`data_service.py:133-143`
- **发现**：`get_realtime_cache()` 失败时捕获异常并设为 `self.realtime_cache = None`、记录 warning，DataService 仍标记为 ready。下游代码（如 `ds_data_writer.py`）可能直接访问 `self.realtime_cache` 而不检查 None。
- **分级**：**P1**

#### D6-P1-02 | 回测模式 vs 实盘模式的订阅路径不统一
- **文件**：`strategy_lifecycle_mixin.py:739-797`
- **发现**：`subscription_manager.subscribe_all_instruments()` 调用了平台 API，但回测模式（`backtest_tick_mode=True`）下此 API 可能不适用。代码中未对 `backtest_tick_mode` 配置进行分支处理，统一走实盘订阅路径。
- **分级**：**P1**

#### D6-P2-01 | on_init 中的品种加载失败 raise RuntimeError，但不检查数据源连通性
- **文件**：`strategy_lifecycle_mixin.py:588`
- **发现**：`ensure_products_with_retry(ds)` 失败时 `raise RuntimeError`，但没有在调用前检查 DuckDB 文件是否存在/可读写。
- **分级**：**P2**

### 维度7：优雅启动降级

#### D7-P1-01 | onStart 中的 CRITICAL_STEPS 失败后仅转为 DEGRADED，不执行资源清理
- **文件**：`strategy_core_service.py:845-991`
- **发现**：`_CRITICAL_STEPS = ('config_load', 'api_bind', 'core_init', 'core_start')` 任一失败后执行 `transition_to(StrategyState.DEGRADED)`，但不会 rollback 已完成的步骤（如已打开的 DuckDB 连接、已启动的 APScheduler）。
- **后果**：`config_load` 成功但 `core_init` 失败时，DataService 已初始化、DuckDB 连接已打开，但策略标记为 DEGRADED。下次重启时可能遇到文件锁冲突。
- **分级**：**P1**

#### D7-P1-02 | StrategyEcosystem 治理检查器集成失败（ImportError 或 Exception）静默降级为 None
- **文件**：`strategy_ecosystem.py:359-386`
- **发现**：E7/E8/E9/E10/E11/E12/E13/WF6-10 共 8 个治理检查器的导入失败全部被同一个 try/except 捕获，任一个 ImportError 导致所有检查器均设为 None。失败原因只记录一条 warning。
- **后果**：如果 8 个检查器中 7 个成功仅 1 个失败，全部被降级。
- **分级**：**P1**

#### D7-P2-01 | 健康检查 API 和结构化日志在 ProductionQuantSystem 构造时降级为 None
- **文件**：`ProductionQuantSystem.py:55-62`
- **发现**：`HealthCheckAPI` 和 `StructuredJsonlLogger` 导入失败时降级为 None，后续 `update_tick` 中通过 `if self.health_check_api is not None` 保护。但降级原因的日志级别仅为 warning。
- **分级**：**P2**

### 维度8：热切换/重启初始化

#### D8-P0-01 | Strategy2026.onStop→onStart 热重启时 StrategyCoreService 的惰性引用可能指向旧单例
- **文件**：`strategy_core_service.py:1089-1098`
- **发现**：`internal_resume_strategy` 中从 STOPPED 状态恢复时调用 `self.onStart()`，但 `StrategyCoreService.__init__` 在 `Strategy2026.__init__` 中仅执行一次。`onStart→onStop→onStart` 周期中 `strategy_core` 内部属性（`_is_paused`, `_is_running`）被重置，但惰性初始化的单例引用（`_shadow_engine`, `_strategy_ecosystem`）**仍是旧实例**。
- **后果**：若 `ShadowStrategyEngine` 或 `StrategyEcosystem` 的单例在 `onStop-onStart` 之间被其他线程 reset，`strategy_core` 仍持有旧引用，导致状态不一致。
- **分级**：**P0**

#### D8-P1-01 | StrategyEcosystem 不提供 reset 方法，无法在参数池切换时重置状态
- **文件**：`strategy_ecosystem.py`（全文搜索无 `def reset`）
- **发现**：参数池切换时需要重置 StrategyEcosystem 的策略槽位状态、资金路由、EV 缓存，但该类未提供 reset 接口。目前只能通过重新构造全局单例来实现（需销毁旧实例）。
- **分级**：**P1**

#### D8-P1-02 | ShadowStrategyEngine 无交易日切换重置逻辑
- **文件**：`shadow_strategy_engine.py`（全文搜索无交易日切换逻辑）
- **发现**：`_alpha_history`, `_group_pnl_sum`, `_shadow_a_trades`, `_shadow_b_trades` 等队列在交易日切换时不重置，前日数据继续累积。
- **分级**：**P1**

#### D8-P2-01 | SignalService 无 reset 方法，仅内部 KalmanFilter1D 有 reset
- **文件**：`signal_service.py:777-783`
- **发现**：`KalmanFilter1D.reset()` 重置卡尔曼滤波器状态，但 `SignalService` 本身无 reset 方法（`_cooldown_times`, `_signal_history`, `_stats` 不重置）。
- **分级**：**P2**
