## 一、数据生产者-消费者图总览

### 1.1 核心数据流路径

```
平台Tick → Strategy2026.onTick()
  → StrategyCoreService.on_tick() [TickHandlerMixin]
    → _tick_probe_layer() → DiagnosisProbeManager (诊断数据)
    → _tick_dispatch_layer()
      → DataService.persist_tick() [DuckDB持久化]
      → RealTimeCache.update_tick() [内存缓存]
      → TTypeService.on_tick() [宽度/强度计算]
        → WidthStrengthCache [五态分布]
          → StateParamManager.update_state_from_width_cache() [三态路由]
      → PositionService.on_tick() [止盈止损检查]
        → _check_stop_profit() / _check_stop_loss()
          → OrderService.send_order(action='CLOSE') [平仓]
      → BoxDetector / BoxSpringStrategy.on_tick() [箱体/弹簧检测]

信号生成流:
  StateParamManager(三态) → StrategyCoreService.execute_option_trading_cycle()
    → TTypeService.select_otm_targets_by_volume() [选标的]
    → _check_ecosystem_exclusion() [互斥检查]
    → RiskService.check_before_trade() [风控检查]
    → OrderService.execute_by_ranking() [下单]
    → ShadowStrategyEngine.process_signal() [影子策略]

订单回报流:
  平台onOrder → StrategyCoreService.on_order()
    → OrderService.on_trade_update() [订单状态更新]

成交回报流:
  平台onTrade → StrategyCoreService.on_trade()
    → PositionService.on_trade() [持仓更新]
```

## 二、死端(Dead End)检测

### D-01 [P0] EventBus事件大量死端——6个事件类从未被发布或订阅

- **文件：** `event_bus.py` 行725-835
- **问题：** TickEvent/KLineEvent/OrderEvent/PositionEvent/RiskEvent/SystemEvent从未被publish()；SignalEvent虽被发布但无subscribe()；'signal.close'被发布但无subscribe()；SubscriptionCompletedEvent被发布但无subscribe()
- **影响：** 事件总线形同虚设，关键数据流依赖直接方法调用而非事件驱动
- **严重等级：** P0 | **影响系统：** 生产系统

### D-02 [P1] signal中call_pnl/put_pnl/underlying_pnl字段无消费者

- **文件：** `signal_service.py` 行303-341
- **问题：** P1-13修复为期权信号附加了call_pnl/put_pnl/underlying_pnl字段，但OrderService/PositionService/ShadowStrategyEngine等下游从未读取
- **严重等级：** P1 | **影响系统：** 生产系统

### D-03 [P1] 资金不足信号_insufficient_fund_signals无风控消费者

- **文件：** `order_service.py` 行124, 781-868
- **问题：** 资金不足信号仅存储不行动，不触发RiskService阻断或SafetyMetaLayer告警
- **严重等级：** P1 | **影响系统：** 生产系统

### D-04 [P2] PositionRecord._profit_history无限增长无外部消费者

- **文件：** `position_service.py` 行411-433
- **问题：** _profit_history无maxlen限制，长期运行可能OOM，且无外部读取接口
- **严重等级：** P2 | **影响系统：** 生产系统

### D-05 [P2] AdaptiveSignalThreshold的pnl始终为0.0

- **文件：** `signal_service.py` 行283, 904-908
- **问题：** record_signal()调用时pnl始终传入0.0，_recent_pnls全为0，自适应阈值调整失效
- **严重等级：** P2 | **影响系统：** 生产系统

### D-06 [P2] EventBus._event_history持久化数据无消费者

- **文件：** `event_bus.py` 行424-442, 474-506
- **问题：** 事件历史和持久化文件被写入但无业务逻辑消费
- **严重等级：** P2 | **影响系统：** 生产系统

## 三、孤儿消费者(Orphan Consumer)检测

### O-01 [P0] get_health_status()引用未定义变量risk_dashboard_status

- **文件：** `position_service.py` 行338-370

- **严重等级：** P1 | **影响系统：** 生产系统

### O-03 [P1] PositionRecord.current_price被消费但从未被设置

### O-04 [P1] AdaptiveSignalThreshold期望pnl数据但始终收到0.0

- **文件：** `signal_service.py` 行137-141, 283

- **严重等级：** P1 | **影响系统：** 生产系统

### O-05 [P2] violation_tracker/parameter_drift_detector无调用者

- **文件：** `evaluation/violation_tracker.py`, `evaluation/parameter_drift_detector.py`
- **问题：** 评判模块定义了track()/detect()方法但无代码调用

## 八、关键数据流路径完整性

### 路径: tick → signal → order → trade → position → pnl

| 节点 | 完整性 | 问题 |
|:---|:---|:---|
| tick | ✅ | 平台回调保证，分片缓冲+持久化 |
| tick→状态 | ⚠️ | state_confirm_bars确认窗口可能延迟状态切换 |
| 状态→signal | ⚠️ | 多层过滤可能静默丢弃信号 |
| signal→order | ❌ | signal_id不传递到订单，追溯链断裂 |
| order→trade | ✅ | 平台回调保证 |
| trade→position | ⚠️ | 平仓记录直接删除不保留 |
| position→pnl | ❌ | current_price从未更新，浮动盈亏计算失效 |

### P-01 [P0] signal→order数据流断裂——signal_id不传递到订单

- **文件：** `strategy_core_service.py` 行274-346, `order_service.py` 行401-431
- **问题：** execute_option_trading_cycle()调用execute_by_ranking(targets)，targets不含signal_id，order字典中signal_id始终为空
- **严重等级：** P0 | **影响系统：** 生产系统

### P-02 [P0] position→pnl数据流断裂——current_price从未更新

- **文件：** `position_service.py` 行377-436
- **问题：** on_tick()计算了profit_pct和profit_slope但从未写入record.current_price，PositionRecord无此字段。check_trailing_stop()永远返回None
- **严重等级：** P0 | **影响系统：** 生产系统

### P-03 [P1] order→position数据流断裂——open_reason传递不可靠

- **文件：** `position_service.py` 行798-810
- **问题：** _get_open_reason_from_order()依赖OrderService查询，订单被清理后open_reason丢失，止盈止损退化为默认倍数
- **严重等级：** P1 | **影响系统：** 生产系统

## 十二、优先修复建议

### P0级（数据流断裂导致关键数据丢失或重复）

1. **P-02**: 在PositionService.on_tick()中添加`record.current_price = price`，修复浮动止盈和时间止损
2. **P-01**: 在execute_option_trading_cycle()中将signal_id传递到targets和order字典
3. **O-01**: 修复get_health_status()中risk_dashboard_status的变量定义顺序
4. **D-01**: 评估EventBus事件驱动架构是否需要真正激活，或移除未使用的事件类
5. **L-01**: 为EventBus背压丢弃添加主动告警机制
6. **C-04**: 为_send_nack()添加递归深度限制

### P1级（数据流不完整导致功能偏差）

7. **O-03**: 在PositionRecord中添加current_price字段并在on_tick()中更新
8. **L-03**: 平仓记录归档而非删除，或发布PositionEvent通知下游
9. **F-01/F-02**: 为Signal和PositionRecord添加Schema验证和版本号
10. **D-03**: 将资金不足信号接入RiskService风控决策
11. **P-03**: 在trade回调中直接传递open_reason而非从OrderService反查

---

**审计结论：** 系统数据流图存在6个P0级断裂点，最严重的是position→pnl路径上的current_price从未更新（P-02）和signal→order路径上的signal_id丢失（P-01）。EventBus虽已实现但事件驱动架构实质未激活（D-01），关键数据流仍依赖直接方法调用。建议优先修复P0级问题，确保tick→signal→order→trade→position→pnl关键路径的数据完整性。
