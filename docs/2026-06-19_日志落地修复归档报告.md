## 一、问题概述

策略在 InfiniTrader 平台上运行时，`strategy.log` 日志文件无法正常创建和写入。该问题经历了**7轮以上**的诊断与修复尝试，是本次部署过程中最曲折的问题之一。最终根因是**初始化顺序错误**——日志初始化在品种加载之后，品种加载失败后日志初始化从未执行。

---

## 二、修复历程时间线

### 第1轮：发现日志不落地

**现象**：策略启动后，`strategy.log` 文件不存在或为空。

**初步判断**：日志路径配置问题。InfiniTrader 嵌入式 Python 环境中 `os.getcwd()` 返回 `C:\Users\xu\AppData\Roaming\InfiniTrader_SimulationX64\`，而非代码目录，导致相对路径 `logs/strategy.log` 解析到了错误位置。

**尝试修复**：修改日志路径为绝对路径。

**结果**：❌ 未解决。日志文件仍然不落地。

---

### 第2轮：路径解析修复

**现象**：`_resolve_log_file_path` 返回的路径在 InfiniTrader 进程环境中不可写。

**分析**：
- `os.path.abspath('logs/strategy.log')` 基于 `os.getcwd()` 解析
- InfiniTrader.exe 的 `os.getcwd()` 是 `%APPDATA%\InfiniTrader_SimulationX64\`
- 该目录下可能没有 `logs/` 子目录

**修复**：`_resolve_log_file_path` 增加回退逻辑——先尝试 `os.path.abspath()`，如果目标目录不存在，则回退到 `%APPDATA%\InfiniTrader_SimulationX64\logs\`。

**结果**：❌ 未解决。路径修复了，但日志仍然不写入。

---

### 第3轮：Handler 冲突排查

**现象**：`setup_logging()` 被调用后，日志仍不写入文件。

**分析**：
- `config_logging.py` 的 `setup_logging()` 会移除已有 handler 再重新添加
- `lifecycle_init.py` 的 `_init_logging()` 也会添加 handler
- 两者可能存在时序冲突：`setup_logging()` 移除了 `_init_logging()` 添加的 handler

**修复**：简化 `setup_logging()`，不再移除已有 handler，改为补充式添加（只添加缺失的 handler）。

**结果**：❌ 未解决。Handler 不冲突了，但日志仍然不落地。

---

### 第4轮：RotatingFileHandler 文件句柄问题

**现象**：InfiniTrader 进程重启后，`strategy.log` 停止写入。旧日志文件已达到 100MB 轮转阈值。

**分析**：
- `strategy.log` 38.68MB，`strategy.log.1/2/3` 各 100MB
- RotatingFileHandler 在进程重启时可能因文件句柄问题无法写入
- 日志在 17:20:36 停止，进程在 17:20:53 重启

**尝试修复**：手动重命名旧日志文件 `strategy.log → strategy.log.old`，强制创建新文件。

**结果**：❌ 临时缓解，但根因未解决。重启后日志仍可能再次停止。

---

### 第5轮：`__pycache__` 缓存问题

**现象**：修改了代码但日志行为未变。

**分析**：InfiniTrader 同一进程中重新初始化策略不会重新加载 `ali2026v3_trading` 模块。Python 使用缓存的 `.pyc` 文件。

**修复**：每次修改代码后，必须：
1. 清除所有 `__pycache__` 目录
2. 完全关闭 InfiniTrader.exe 再重新打开

**结果**：✅ 解决了"修改不生效"问题，但日志落地根因仍未解决。

---

### 第6轮：`concurrent_log_handler` 缺失

**现象**：RotatingFileHandler 在多进程/多线程场景下可能不安全。

**分析**：代码中有 `concurrent_log_handler` 的可选导入：
```python
try:
    from concurrent_log_handler import ConcurrentRotatingFileHandler
    _HAS_CONCURRENT_HANDLER = True
except ImportError:
    _HAS_CONCURRENT_HANDLER = False
```
但 `.venv` 中未安装该包，`_HAS_CONCURRENT_HANDLER = False`。

**判断**：这不是根因。RotatingFileHandler 在单进程场景下可以正常工作。

**结果**：❌ 非根因。

---

### 第7轮（最终）：初始化顺序 — 真正的根因

**现象**：策略启动后 `strategy.log` 完全不存在。

**关键发现**：

`on_init` 的执行顺序为：
```
Step1: 品种加载 (ensure_products_with_retry)
Step2: 合约加载+预注册 (load_and_preregister_instruments)
...
StepN: _init_logging  ← 日志初始化在最后
```

品种加载（Step1）因合约格式错误、year_month 不匹配等问题抛出 `RuntimeError`，`on_init` 的 try-except 捕获异常后直接返回 `False`。**`_init_logging` 从未执行**，因此日志系统从未初始化，日志文件从未创建。

**根因确认**：
1. `_init_logging` 在品种加载之后调用
2. 品种加载失败 → `on_init` 异常退出 → `_init_logging` 从未执行
3. 即使后续步骤失败，日志也应该能记录错误信息
4. 但由于日志系统未初始化，所有 `logging.error()` 只输出到 stderr（InfiniTrader 不显示）

**修复**：将 `_init_logging` 移到品种加载之前（Step0a），确保即使后续步骤失败，日志也能正常落地。

修改前（`lifecycle_init.py`）：
```python
def on_init(self, *args, **kwargs) -> bool:
    try:
        product_result = ensure_products_with_retry(ds)  # Step1: 品种加载
        # ... 其他步骤 ...
        p._init_logging(kwargs.get('params'))  # StepN: 日志初始化（太晚了！）
    except Exception as e:
        logging.error(f"Failed: {e}")  # 日志系统未初始化，此消息丢失
        return False
```

修改后：
```python
def on_init(self, *args, **kwargs) -> bool:
    try:
        p._init_logging(kwargs.get('params'))  # Step0a: 日志初始化（最先执行！）
        product_result = ensure_products_with_retry(ds)  # Step1: 品种加载
        # ... 其他步骤 ...
    except Exception as e:
        logging.error(f"Failed: {e}")  # 日志系统已初始化，消息正常落地
        return False
```

**结果**：✅ 彻底解决。`strategy.log` 成功创建并持续写入。

## 三、根因分析总结

### 因果链

```
品种加载失败
  → on_init 异常退出
    → _init_logging 从未执行
      → logging 系统未初始化
        → 所有 logging.error() 消息丢失
          → 无法从日志诊断问题
            → 形成诊断盲区，问题反复出现
```

### 核心教训

1. **日志初始化必须是第一个步骤**：在任何可能失败的初始化步骤之前，必须先确保日志系统可用。否则，后续步骤的失败信息无法被记录，形成诊断盲区。

2. **初始化顺序是隐式依赖**：`_init_logging` 放在品种加载之后，隐式假设品种加载一定成功。这种假设在复杂系统中不可靠。

3. **诊断代码不能替代根因修复**：在前几轮中，尝试了添加 `write_log()` 诊断代码、手动重命名日志文件等临时方案，但都没有解决根因。只有将 `_init_logging` 移到最前面，才能确保日志始终可用。

4. **`__pycache__` 缓存会掩盖修改**：在嵌入式 Python 环境中，修改代码后必须清除缓存并重启宿主进程，否则修改不会生效。这导致多轮修复"看似无效"，实际是缓存问题。

## 七、经验教训与最佳实践

### 1. 日志优先原则

> **任何可能失败的初始化步骤之前，必须先确保日志系统可用。**

这是嵌入式系统开发的基本原则。在 InfiniTrader 这样的宿主进程中，Python 策略的 `stderr` 不可见，日志文件是唯一的诊断手段。

### 2. 初始化顺序设计

初始化步骤的顺序应遵循：
1. **日志系统** — 最先初始化
2. **配置系统** — 读取配置
3. **数据系统** — 连接数据库
4. **业务系统** — 加载品种、合约等
5. **网络系统** — 订阅行情

### 3. 诊断盲区的危害

当日志系统不可用时，所有错误信息都丢失了。这导致：
- 无法判断品种加载为什么失败
- 无法判断配置文件哪里有问题
- 只能靠猜测和试错

### 4. `__pycache__` 在嵌入式环境中的陷阱

在 InfiniTrader 嵌入式 Python 环境中：
- 同一进程内重新初始化策略不会重新加载模块
- 必须完全关闭 InfiniTrader.exe 再重新打开
- 修改代码后必须清除所有 `__pycache__` 目录

### 5. 不要用诊断代码绕过根因

在前几轮中，尝试了添加 `write_log()` 诊断代码来输出信息。但这只是绕过了日志不落地的问题，没有解决根因。正确做法是找到日志不落地的根因并修复。

## 九、附录：关键代码片段

### A. `_resolve_log_file_path`（修复后）

```python
@staticmethod
def _resolve_log_file_path(params: Optional[Dict[str, Any]]) -> str:
    log_file = params.get('log_file', 'logs/strategy.log') if params else 'logs/strategy.log'
    if os.path.isabs(log_file):
        return log_file
    abs_from_cwd = os.path.abspath(log_file)
    log_dir = os.path.dirname(abs_from_cwd)
    if os.path.exists(os.path.join(log_dir, 'strategy.log')) or os.path.exists(os.path.join(log_dir, 'strategy.log.1')):
        return abs_from_cwd
    appdata = os.environ.get('APPDATA', '')
    if appdata:
        for candidate_name in ['InfiniTrader_SimulationX64', 'InfiniTrader']:
            candidate_dir = os.path.join(appdata, candidate_name, 'logs')
            if os.path.isdir(candidate_dir):
                return os.path.join(appdata, candidate_name, log_file)
    return abs_from_cwd
```
