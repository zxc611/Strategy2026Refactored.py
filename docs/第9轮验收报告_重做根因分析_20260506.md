## 五、真正根因：Python 代码中的关键缺陷

### 5.1 根因一：Preload 从 DB 拉取的是**快照价格**，不是实时流

```python
# strategy_lifecycle_mixin._preload_options_into_t_type (backup)
options_df = ds.query("""
    SELECT lp.instrument_id, lp.last_price, meta.option_type, meta.strike_price
    FROM latest_prices lp
    INNER JOIN latest_option_meta meta ON lp.instrument_id = meta.instrument_id
    WHERE lp.last_price > 0
""")
```

这个查询从 `latest_prices` 表取**静态快照**。这意味着：

- `latest_prices` 表的状态取决于 **上一次会话结束时的数据状态**
- 若上一次会话（14:36 会话）某期权从未收到 tick，则该期权的 latest_price = NULL → 本次 preload 不会加载该期权
- 若 `latest_prices` 有值，该值作为 initial_price 写入 WidthStrengthCache，但**这是一次性的**——后续的实时 price 更新必须通过 `on_option_tick` 实时推送

### 5.2 根因二：缺少"冷启动"数据预热机制

关键时序矛盾：

```
16:08:19  ParamsService: 期权合约加载完成 2752 个（从 DB 加载元数据）
16:08:28  InitOptions: No option prices found for preload（DB 中无价格数据）
16:08:33  Preload complete: Options: 0（预加载无期权价格）
16:08:43  Bulk subscription: 2785 instruments（平台订阅开始）
         ↓ ←【35秒窗口期：平台尚未开始推送数据，DB尚未被填充】
16:09:08  Preload complete: Options: 2752（预加载有了期权数据）
```

**触发选项写入缓存的真正原因是：平台订阅完成后，HistoricalLoader 或 RealTimeLoader 在 16:08:43 ~ 16:09:08 期间将历史/初始 tick 数据写入 DB 的 `option_ticks` / `latest_prices` 表，然后第三次 preload 从 DB 拉取到了这些数据。**

### 5.3 根因三：AL2605/CU2606 的合成价格来源

AL2605 期权价格 = 6050, 5950, 5850... (均匀间隔100)。这些不是市场 tick，而是某种**合成计算**的结果。

计算逻辑最可能在 `_preload_options_into_t_type` 或 `_compute_synthetic_option_prices` 中：

```python
# 合成价格 = underlying_future_price - strike_price（对深度 ITM CALL）
# 或当 latest_prices 中无数据时，用标的物价格减去行权价作为近似
```

CU2606 同样有合成价格（3842, 2660, 1818...），而 CU2605 全部 @0.00 或仅 `CU2605C104000 @16.00`。

**这意味着：对于 CU/AU 的某些合约月，预加载能找到标的价格但找不到期权价格，于是计算了合成近似价格。但这些合成价格是一次性的——没有后续的实时 tick 来更新它们。**

### 5.4 根因总结：三条链断裂

```
期望的数据流：
  Platform → SubscriptionManager → _on_tick_impl → t_type_service.on_option_tick
    → WidthStrengthCache：每次 tick 更新价格、计算方向、更新状态

实际的数据流（16:07 会话）：
  ① DB 预加载：从 latest_prices 取快照 → 只有部分合约有数据
     ├─ AL2606 ✅（有 DB 快照价格，且后续收到实时 tick）
     ├─ AL2605 ❌（仅有合成价格，无实时 tick 更新）
     ├─ AU2605/06 ❌（有 DB 快照价格，但无实时 tick 更新）
     ├─ CU2606 ❌（仅有合成价格，无实时 tick 更新）
     └─ 其他 ❌（DB 中无价格 @0.00）
  
  ② 平台订阅后：实时 tick 仅对 AL2606 和 CFFEX 期权持续推送
     ├─ AL2606: 后续收到 tick → 价格变动 → 退出 'other' → 72 非other
     └─ CU/AU/ZN: 未收到后续 tick → 价格不变 → 永远 'other'
```

> **核心问题：系统依靠两种数据输入——DB 快照（静态）和实时 tick（动态）。预加载通道负责快照，_on_tick_impl 通道负责实时。当前代码缺少"冷启动→热更新"的过渡机制。当某个期权的 DB 快照为空或为合成价格时，只能等待实时 tick 来修正；若实时 tick 也未到达，则永久停留在 'other'。**

## 九、结论

**自洽根因：**

期权从 'other' 状态迁移到非 other 状态的必要条件是**至少两次不同价格的 tick 更新**。在 16:07 会话中：

- ✅ AL2606 的期权在 16:08:43 订阅后，持续收到平台的实时 tick 更新 → 72 个期权获得两次以上不同价格 → 成功进入非 other 状态
- ❌ AU2605/2606 的期权虽然注册了真实市场价，但在整个 2.5 小时会话中**再无第二次价格变动** → 始终停在首次 tick 产生的 'other' 状态
- ❌ AL2605/CU2606 的期权获取的是**合成价格**（标的价-行权价），用于填充无数据的空白，同样无后续实时修正
- ❌ 其他远月合约和 ZN 的期权本身没有 DB 快照价格，平台也未推送 tick → @0.00 → 'other'

**Python 代码层面的缺陷不是 tick 路由或处理逻辑的 bug（该逻辑对全品种一致且正确），而是：**

1. **DB 预加载通道产出的是"一次性快照"**，写入后不再更新，缺少与实时 tick 通道的衔接机制
2. **_preload_from_db 中的 column 名称 bug（`row['last_price']` KeyError）导致第一轮预加载失败**，错失在订阅前完成完整数据加载的窗口期
3. **合成价格填充逻辑（`_preload_options_into_t_type`）在 DB 无价格时生成近似值**，但这些近似值仅写入一次且无后续刷新
4. **缺少"订阅后价格不更新"的检测与告警**，导致 0% 非 other 的局面持续 2.5 小时无人发现

**CU/AU/ZN 的失败不是因为代码对它们有差异化处理，而是因为代码对所有品种一视同仁地"脆弱"——依赖外部 DB 快照和被动 tick 接收，缺少主动验证和重试机制。AL2606 恰好是唯一一个在外部数据管道中"流通"的合约月，所以它成功了；其他合约月数据管道不通，代码也无能为力。**
