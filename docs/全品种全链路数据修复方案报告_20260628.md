### P0-2：_recalc替换_status_counts导致92.7%期权丢失

**根因定位**: `width_cache_state_mixin.py:806`

```python
# 行806：整体替换，snapshot外的期权计数被丢失
self._status_counts[future_internal_id] = defaultdict(...)
```

**根因分析**：
1. `_recalc`阶段1快照只包含`_options_by_future_type[future_id]`中的期权
2. 如果期权的`_option_info`被TTL淘汰（R-73），`opt_info=None`，行734跳过该期权
3. 如果期权的`_options_by_future_type`被期货TTL淘汰删除（行146），快照为空
4. 行806整体替换后，不在snapshot中的期权计数全部丢失
5. **正证据**：6月28日日志显示total从15898降到1189，损耗92.7%
6. **反证据排除**：不是register_option注册失败（日志显示注册15898/15898成功）

**修复方案**：将行806的"整体替换"改为"逐期权增量更新"

**具体修改**（`width_cache_state_mixin.py`）：

阶段3（行793-807）改为：
```python
# 阶段3：锁内增量更新 - 逐期权更新_status_counts，不替换
with self._lock:
    if future_internal_id in self._sort_buckets:
        self._sort_buckets[future_internal_id].clear()
    
    for item in sort_bucket_updates:
        self._current_status[item['iid']] = item['status']
        self._sync_flag[item['iid']] = item['is_sync']
        self._do_update_sort_bucket(
            item['iid'], item['opt_info'],
            item['future_internal_id'], item['month'], item['opt_type']
        )
        # 增量更新：用_set_current_status替代整体替换
        self._set_current_status(item['iid'], item['opt_info'], item['status'])
    
    # _sync_otm_count也改为增量更新
    # 先清零该期货的sync_otm_count，再从sort_bucket_updates重建
    self._sync_otm_count.pop(future_internal_id, None)
    for item in sort_bucket_updates:
        if item['is_sync']:
            self._sync_otm_count[future_internal_id][item['month']][item['opt_type']] += 1
```

**修复成功标准**：
1. `_recalc`前后，`sum(_status_counts[future_id][month][opt_type].values())`不变
2. `_recalc`后，不在snapshot中的期权计数不丢失
3. 日志验证：recalc后total不下降

**验证方式**：
- 单元测试：创建3个期权，recalc后断言total=3
- 日志断言：grep "recalc" 日志，确认前后total一致

**潜在隐患**：
- `_set_current_status`内部已有old_status减1+new_status加1逻辑，但old_status可能不在bucket中（行309 `if old_status in bucket`）。修复：行309改为无条件减1（`bucket[old_status] = max(0, bucket.get(old_status, 0) - 1)`），确保守恒
- _sync_otm_count清零后重建，如果sort_bucket_updates中有遗漏，计数不准。缓解：_sync_otm_count的清零重建逻辑与sort_bucket_updates一致

**彻底性检查**：
- ✅ 根因修复：不再整体替换，改为增量更新
- ✅ 无绕过：所有recalc路径都走增量更新
- ⚠️ 新隐患：_set_current_status中old_status不在bucket时需处理（见上方修复）
- ✅ 无冗余：删除行806-807的替换代码，改为调用已有方法_set_current_status

4. 但该期权仍在接收tick！`on_option_tick`在行597-612发现`_option_info`为None，直接return
5. **正证据**：`_CACHE_TTL_SECONDS=86400`（`width_cache_types.py:130`），策略运行24小时后所有初始化时注册的期权timestamp过期
6. **反证据排除**：不是容量淘汰导致（MAX_OPTION_CACHE_SIZE=20000 > 16288）

**修复方案**：on_option_tick成功处理后更新timestamp

**具体修改**（`width_cache_state_mixin.py`）：

行691（`self._enforce_cache_size_limit()`之前）添加：
```python
self._option_info_timestamps[internal_id] = time.time()
```

**修复成功标准**：
1. on_option_tick处理后，`_option_info_timestamps[internal_id]`被更新为当前时间
2. 活跃期权不会被TTL淘汰（timestamp始终新鲜）
3. 不活跃期权（24小时无tick）仍会被TTL淘汰

**验证方式**：
- 单元测试：调用on_option_tick后断言`_option_info_timestamps[iid] > before`
- 日志验证：运行24小时后grep "CacheEvict"确认无活跃期权被淘汰

**潜在隐患**：
- 每次tick多1次dict写入，开销极小（1次赋值），可忽略
- 如果期权永远收到tick，永远不会被TTL淘汰。这是正确行为（活跃期权不应被淘汰）

**彻底性检查**：
- ✅ 根因修复：timestamp更新使活跃期权不被TTL淘汰
- ✅ 无绕过：每次tick都更新timestamp
- ✅ 无新隐患：开销极小，行为正确
- ✅ 无冗余：仅1行代码

### P0-5：on_future_tick价格为0未拦截

3. **正证据**：`_is_out_of_the_money`在`width_cache_context_mixin.py:252-257`，当underlying_price=0时逻辑错误
4. **反证据排除**：subscription_service.py的on_tick入口已过滤last_price<=0（行1384-1386），但on_future_instrument_tick可能从其他路径调用

**修复方案**：on_future_tick入口拦截price<=0

**具体修改**（`width_cache_state_mixin.py`）：

行507（`with self._lock:`之前）添加：
```python
if price <= 0:
    if not hasattr(self, '_future_zero_price_count'):
        self._future_zero_price_count = 0
    self._future_zero_price_count += 1
    if self._future_zero_price_count <= 5:
        logging.warning("[FutureZeroPrice] on_future_tick price<=0: fid=%s price=%s", future_internal_id, price)
    return
```

**修复成功标准**：
1. price<=0的期货tick被拦截，不写入`_future_prices_by_month`
2. price<=0的期货tick不触发`_recalc`
3. 拦截时输出warning日志（前5次）

**验证方式**：
- 单元测试：调用on_future_tick(fid, 0)后断言`_future_prices_by_month`不变
- 日志断言：grep "FutureZeroPrice"确认拦截日志

**潜在隐患**：
- 首个tick price=0时，期货永远不初始化。但price=0意味着无有效报价，不应初始化是正确的
- register_future中initial_price=0时仍会注册（行293），但不会设置`_future_initialized=True`（由on_future_tick设置）

**彻底性检查**：
- ✅ 根因修复：price<=0不写入，避免虚值判断全面错误
- ✅ 无绕过：on_future_tick和on_future_instrument_tick都经过此检查
- ⚠️ 新隐患：register_future中initial_price=0时仍写入_future_prices_by_month（行387），需同步修复
- ✅ 无冗余：仅3行代码+计数器
