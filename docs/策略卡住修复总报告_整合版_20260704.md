## 二、根因分析

### 2.1 卡住主因：DuckDB跨线程竞争

```
TTS-Deferred线程调用get_latest_price(_oid) → DuckDB查询latest_prices视图
→ 与_bulk_subscribe_thread并发访问同一单连接(无锁, _SYNC_EXEC=True) → 策略挂起
```

### 2.2 共因矩阵

| 共因 | 现象 | 影响 | 修复编号 |
|------|------|------|----------|
| 非交易时段平仓洪泛 | 23:36 CZCE已收盘，所有平仓被拒绝result=-1 | 日志洪泛+重试耗尽 | Fix 4, Fix 29-31 |
| 盘口DBL_MAX误报 | bid=1.79e308通过>0检查 | 盘口交叉异常洪泛247条 | Fix 3 |
| 平仓价格负数 | FG608P900 price=-0.5000 | 跳过下单+死循环重试 | Fix 2 |
| DuckDB跨线程无保护 | owner=9180 current=19192 | C API原生崩溃风险 | Fix 9 |
| strategy_id类型错误 | int in str → TypeError | 2次启动完全失败 | Fix 10 |
| 幽灵持仓恢复 | 561个历史持仓平台不存在 | 321条/轮ERROR+WARNING洪泛 | Fix 29-31 |
| 启动时持仓立即平仓 | 146个恢复持仓elapsed远超hold_time | 全量平仓洪泛 | Fix 9d |

### 2.3 重要澄清

- `register_option_contract`（t_type_service.py:327）调用`WidthStrengthCache.register_option`是Python内存缓存操作，**不调用C++ StrategyLib.dll API**，不是DLL崩溃并发源
- `get_latest_price(_oid)`是经`ds_query_cache.py`查DuckDB的`latest_prices`视图，**才是卡住根因**
- 22:41 DLL崩溃已由"双线程join"修复（FIX-20260703-REGRESSION）解决；23:36卡住是**独立问题**

## 十、已知限制与未来优化

### 10.1 已知限制

**限制1：收盘后CANNOT_CLOSE标记**
- 现状：Fix 30(恢复跳过) + Fix 31(耗尽删除)已修复幽灵持仓问题；Fix 34收盘重试保护已实现；但BoundedRetry对返回None仍不重试
- 影响：第二天开盘后不会自动重试平仓，需人工介入
- 临时缓解：运维人员应在交易日开盘前检查CANNOT_CLOSE标记

**限制2：合约配置文件加载3次重试均失败**
- 现象：2次ERROR（13:20和13:27）
- 影响：可能是网络/文件系统问题，非代码缺陷
- 建议：观察重启后是否复现

### 10.2 未来优化建议

1. **收盘重试机制**：引入定时重试，等待开盘后自动重试（替代当前延后重试）
2. **DuckDB连接池**：从单连接模式升级为连接池模式，彻底解决跨线程问题
3. **DBL_MAX哨兵值统一过滤**：在PythonGO库的C++绑定层统一过滤
4. **运行时段控制**：增加启动时市场状态预检查，若所有交易所均CLOSED则延迟启动到下一个开盘时段
5. **审计密钥管理**：所有敏感密钥（HMAC/SSL/API token）统一从环境变量读取，禁止默认值

### 14.3 发现的问题（3项）

#### Fix 56: _onStart_step_core_init阻塞导致策略停留INITIALIZING（P0）

**根因**: `strategy_2026.py:357-359` 的 `_onStart_step_core_init` 方法在检测到 `state=INITIALIZING` 时调用 `self.strategy_core.initialize(self.config)`。该方法委托到 `lifecycle_init.py:on_init()`，后者执行 `with p._lock:`。由于on_init已完成(`_initialized=True`)，理论上应立即返回。但8:57 LIVE日志证据显示：
- L7271 (08:58:55.082) 输出state后，**13秒日志空白**
- "正在调用 strategy_core.initialize()..." 日志**从未出现**
- "FIX-20260704-STATE-RUNNING" 日志**从未出现**
- 策略从08:58:55到09:11:19（13分钟）一直停留INITIALIZING状态

**分析**: `initialize()`调用被`p._lock`阻塞（可能被analytics warmup或recovery线程持有）。由于C++原生阻塞无法被Python try/except捕获，Fix 32的后续逻辑（设置_is_running=True）从未执行。

**修复** (FIX-20260706-INIT-BLOCK):
```python
# 修改前 (strategy_2026.py:357-359):
if _state_is(current_state, StrategyState.INITIALIZING):
    logging.info("[Strategy2026] 正在调用 strategy_core.initialize()...")
    init_result = self.strategy_core.initialize(self.config)

# 修改后:
if _state_is(current_state, StrategyState.INITIALIZING):
    _already_initialized = getattr(self.strategy_core, '_initialized', False)
    if _already_initialized:
        logging.info("[Strategy2026] strategy_core已初始化(_initialized=True)，跳过二次initialize()避免锁争用")
    else:
        logging.info("[Strategy2026] 正在调用 strategy_core.initialize()...")
        init_result = self.strategy_core.initialize(self.config)
```

### 15.6 关键教训

1. **`__getattr__`委托机制的陷阱**: `StrategyCoreService.__getattr__`通过`_PHASE2_GETATTR_MAP`将属性访问委托到`_lifecycle_mgr`。大部分属性（`_state`、`_is_running`）能成功返回，但`_initialized`却阻塞。Fix 56假设`getattr`会正常返回False，未考虑`__getattr__`委托阻塞的可能。Fix 59用`object.__getattribute__`绕过`__getattr__`，从根本上消除委托机制的不确定性。
2. **Fix 56的盲点**: Fix 56通过检查`_initialized`避免二次`initialize()`，但`getattr(self.strategy_core, '_initialized', False)`本身也会触发`__getattr__`委托，导致与`initialize()`相同的阻塞。Fix 59b将`getattr`替换为`object.__getattribute__`直接访问，彻底消除阻塞源。
3. **`object.__getattribute__` vs `getattr`**: `object.__getattribute__`是Python属性查找的底层机制，绕过自定义`__getattr__`，直接访问对象`__dict__`。适用于已知属性存在但`__getattr__`可能阻塞的场景。
4. **运行时验证的递进性**: Fix 32（假设initialize()正常返回）→ Fix 56（检查_initialized跳过initialize()）→ Fix 59（绕过__getattr__直接访问_initialized）。每一轮修复都暴露了前一轮的盲点，只有通过LIVE实盘运行才能发现。

### 18.7 关键教训

**教训一：属性访问委托机制的陷阱**
`StrategyCoreService`的自定义属性访问（`__getattr__`）将部分属性请求委托给生命周期管理器（`_lifecycle_mgr`），但当管理器尚未完成初始化时，委托过程可能阻塞。解决方案是使用`object.__getattribute__`绕过自定义访问，直接读取实例的内部字典（`__dict__`）。此类委托机制在设计时必须考虑属性访问的阻塞性风险。

**教训二：代码同步的完整性**
模拟环境（SIM）的修复必须同步到实盘环境（LIVE），否则实盘仍会报旧缺陷。本次`health_monitor.py`的遗漏同步导致124次索引越界错误。建议每次修复后立即执行同步验证，确认所有受影响文件的MD5值一致。

**教训三：幂等操作的日志级别**
创建表（`CREATE TABLE IF NOT EXISTS`）和创建视图（`CREATE OR REPLACE VIEW`）是幂等操作——多次执行结果相同。此类操作的"表已存在/已创建"日志应使用调试级别（`DEBUG`），仅在首次创建或失败时使用信息级别（`INFO`）或警告级别（`WARNING`）。否则，在线程局部连接模式下，每个新连接都会产生冗余日志。

**教训四：数据库连接的生命周期管理**
线程局部连接模式下，连接可能被DuckDB引擎回收。查询前应检查连接有效性，或在异常后自动重建连接。当前的查询缓存（`ds_query_cache`）已捕获此类异常，但可能导致诊断数据不完整。
