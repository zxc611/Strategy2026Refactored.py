## 一、BUG-1胖手指防护修复修正

### 1.1 问题描述

前次修复中，开仓价fallback场景使用`ref_price=-1.0`标记跳过胖手指防护，这**不符合文档7.2.1节的原始修复意图**。

文档原文："`OrderContext`新增`ref_price`字段，胖手指防护**优先使用**`ctx.ref_price`"

### 1.2 正确方案

开仓价fallback时，应传**开仓价本身**作为`ref_price`，让胖手指防护正常工作：
- `ref_price = open_price`
- `price = open_price ± tick_size`
- 偏差 = `tick_size / open_price` ≈ 1-2%，远小于20%阈值
- 胖手指防护正常通过，不会误判

### 1.3 修改文件

| 文件 | 修改内容 |
|------|----------|
| `position/position_command_service.py:1027-1029` | 移除`if _used_open_price_fallback: _close_ref_price = -1.0`，直接用`base`（开仓价）作为ref_price |
| `position/position_command_service.py:1188-1190` | 重试路径同样移除`-1.0`覆盖 |
| `order/order_executor.py:244-268` | 移除`ref_price<0`时跳过胖手指防护的分支，增加ref_price日志输出 |

### 1.4 修复前后对比

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 开仓价fallback | ref_price=-1.0，跳过胖手指防护 | ref_price=open_price，胖手指防护正常工作 |
| 正常市价 | ref_price=0.0，从市价获取 | 不变 |
| 偏差计算 | 跳过，无保护 | tick_size/open_price ≈ 1-2% < 20%阈值，正常通过 |

## 二、五态分类全OTHER_SCALP根因修复

### 2.1 问题描述

20:21重启验证发现所有交易的`open_reason`均为`OTHER_SCALP`，对应市场状态`other`。

### 2.2 根因分析

**链路**：`_classify_status()` → `_status_counts` → `update_state_from_width_cache()` → `get_current_state()` → `resolve_open_reason()`

**4个叠加根因**：

| 编号 | 根因 | 严重程度 | 说明 |
|------|------|----------|------|
| R1 | `non_other_ratio_threshold=0.65`过高 | P0 | 期权注册时全为`other`，占比>35%就永远无法切换到非other状态 |
| R2 | 状态确认参数过于保守 | P0 | `confirm_bars=5`+`check_interval=60s`+`hold=600s`，状态切换需5分钟以上 |
| R3 | `update_state_from_width_cache`调用频率极低 | P0 | 仅在`output_periodic_summary`中调用（每180秒） |
| R4 | `resolve_open_reason`中spm为None时直接兜底 | P1 | `_state_param_manager`未设置时state='unknown'→OTHER_SCALP |

**正反馈死循环**：`other`占比高 → 状态保持`other` → open_reason=OTHER_SCALP → 无有效分类信号 → `other`占比持续高

### 2.3 修改文件

| 文件 | 修改内容 |
|------|----------|
| `config/state_param.py:782-790` | `confirm_bars`: 5→2, `check_interval`: 60→15s, `non_other_ratio_threshold`: 0.65→0.40, `min_state_hold_seconds`: 600→120s |
| `strategy/strategy_config_layer.py:145-165` | `resolve_open_reason`增加spm fallback获取 + 主动调用`update_state_from_width_cache` |
| `lifecycle/lifecycle_callbacks.py:37-41` | `bind_width_cache`添加成功/失败诊断日志 |

### 2.4 修复前后对比

| 参数 | 修复前 | 修复后 | 效果 |
|------|--------|--------|------|
| `non_other_ratio_threshold` | 0.65 | 0.40 | 非other占比≥40%即可切换（原需≥65%） |
| `state_confirm_bars` | 5 | 2 | 确认次数从5次降至2次 |
| `state_check_interval_sec` | 60 | 15 | 检查间隔从60秒降至15秒 |
| `min_state_hold_seconds` | 600 | 120 | 最小持有期从10分钟降至2分钟 |
| `resolve_open_reason` | spm=None→OTHER_SCALP | fallback获取spm + 主动更新状态 | 消除兜底路径 |

### 2.5 预期效果

- 五态分类能正确从`other`切换到`correct_trending`/`incorrect_reversal`等状态
- `open_reason`将出现CORRECT_RESONANCE/INCORRECT_REVERSAL等多样性
- 品种数量应从5个增加（不同策略组有不同的止盈止损参数，开仓门槛降低）
- 状态切换响应时间从>5分钟降至<30秒

## 三、品种数量偏少（间接效果）

品种只有5个（FG:74, CF:62, cu:31, MA:25, CJ:4）的根因与五态分类问题直接相关：

1. **OTHER_SCALP的止盈止损过于保守**：tp=1.1, sl=0.30，而CORRECT_RESONANCE的tp=1.5, sl=0.50
2. **box策略组（OTHER_SCALP）的开仓条件更严格**：信号筛选门槛高
3. 修复五态分类后，更多品种将获得CORRECT_RESONANCE/INCORRECT_REVERSAL等状态，开仓条件更宽松
