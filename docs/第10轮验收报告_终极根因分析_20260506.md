## 四、代码层排查：Python路径完全自洽证明

### 4.1 订阅路径（strategy_lifecycle_mixin.py）

```python

for i, inst in enumerate(instrument_ids, 1):
    subscribe_fn(inst)
```

日志证据：
```

[Subscribe] Done: ok=2785, fail=0
```

**所有2785个合约（含全部CU/AU/ZN期权）订阅成功，无一失败。订阅顺序：CFFEX期货→CFFEX期权→SHFE期货→SHFE期权(按al→cu→au→zn)。**

### 4.2 Tick入口路径（strategy_tick_handler.py）

```python
# _tick_probe_layer → _tick_check_layer → _tick_dispatch_layer → _process_tick
sub_mgr.on_tick(instrument_id, last_price, volume)
```

**无任何产品/交易所相关的条件分支。所有tick均一视同仁。**

### 4.3 Tick路由路径（subscription_manager._on_tick_impl）

```python
# 三层路由：
# Layer 1: Metadata路由 (ps.get_instrument_meta_by_id)
# Layer 2: 自动注册 (enable_auto_registration)
# Layer 3: 降级路由 (SubscriptionManager.is_option)
tts.on_option_tick(normalized_id, price, volume)
```

**无任何产品相关的条件分支。所有SHFE商品期权的on_option_tick调用路径完全相同。**

### 4.4 期权状态分类路径（width_cache.on_option_tick → _classify_status）

`_classify_status` 返回 'other' 的条件（按优先级）：

| 条件 | 代码行 | 描述 |
|------|-------|------|
| (A) | future_id is None | 未来合约未关联 |
| (B) | current_price ≤ 0 or prev_price ≤ 0 | 价格为0或负 |
| (C) | not initialized | 期货未初始化（_future_initialized=False） |
| (D) | option_direction is None | 价格未变化，方向无法确定 |

所有产品的期货均已初始化（18:41诊断显示"期货方向"字段对全部future_id均有值，证明_future_initialized=True）。因此条件(C)对所有产品均不触发。

**关键瓶颈是条件(D)**：如果current_price == prev_price，direction为None，返回'other'。

### 4.5 首次tick的关键行为差异

`register_option` 在初始化时写入 `_option_price[internal_id]` 和 `_option_prev_price[internal_id]`（通过`get_latest_price`）。这意味着：

- **第一个真实tick到来时**：`prev_price` = DB最新价（寄存器时写入），`price` = 实时行情价
- **如果DB价 ≠ 实时价** → 方向确定 → 可分类 ✅
- **如果DB价 = 实时价** → 方向None → 'other' ❌
- **如果没有tick到来** → `_option_price`始终等于`_option_prev_price` → `_update_option_direction`只在`on_option_tick`中调用 → 永远'other' ❌

## 五、最终根因结论

### 5.1 三个层次的真实根因

**第一层（直接原因）**：平台对SHFE商品期权的tick推送极不完整

- CFFEX指数期权：全月份、全合约获得连续tick推送
- SHFE AL2606期权：获得tick推送（2.5小时内推动61%合约脱离'other'）
- SHFE CU/AU/ZN/AL其他月份期权：**零tick推送**（2.5小时后仍全部'other'）

这不是Python代码的bug，而是平台数据源在不同产品上表现不一致。Python代码在整个链路中没有对任何产品做区隔处理，所有产品通过完全相同的代码路径。

**用户所述"CU/AU比AL更活跃成交量更大"——若指期货成交量，不能直接推导出期权tick推送的完整性。期权的流动性远低于对应期货，且不同产品的期权做市商活跃度差异极大。AL2606恰是当时主力合约月份，可能是唯一具备活跃期权交易的SHFE品种。**

### 5.2 Python代码的五个缺陷（并非根因，但加剧了问题）

| 序号 | 缺陷 | 位置 | 影响 |
|------|------|------|------|
| **缺陷1** | `_preload_from_db`使用错误列名`row['last_price']` | t_type_service.py L390(运行版本) | 后台预加载线程对全部2752个期权均抛出KeyError，导致依赖此路径的选项得不到初始价格。虽然后续init路径补偿注册了，但浪费了首次预加载窗口。 |
| **缺陷2** | `register_option`调用`get_latest_price`写入`_option_price`和`_option_prev_price` | width_cache.py L650-656 | 这创建了"伪首次tick"——使得真实首次tick变成"第二次tick"。假如DB存储的latest_price与实际行情价相同，则方向始终为None，陷入永久'other'。 |
| **缺陷3** | `on_option_tick`入口处无任何日志 | width_cache.py L754 | 无法从日志直接确认tick是否到达某个期权。排查时必须依赖间接观测点（PROBE_RAW、ContractWatch、诊断）。 |
| **缺陷4** | 无"价格僵死"检测机制 | width_cache.py 整体 | 如果某个期权的 `_option_price[internal_id] == _option_prev_price[internal_id]` 持续时间超过阈值（如60秒），系统应发出警告，而不是静默保持'other'。 |
| **缺陷5** | `get_latest_price`对部分合约返回合成价格 | app/core/database | AL2605和CU2606期权的"最新价"是 `underlying_price - strike`，不是真实的交易所行情。这导致DB价与实际tick价差异人为放大——表面上看这"有助于"分类（因为价格差异大更容易判定方向），实际上产生的分类方向可能不准确。 |

## 六、为什么"CFFEX指数期权成功"和"SHFE AL2606成功"能自洽地共存

### 6.1 成功路径的物理过程

```
平台C++数据源
  │
  ├── CFFEX (中金所) tick数据源
  │     └── 指数期权(HO/IO/MO) tick 大量推送 → ContractWatch检测到 → on_option_tick → 全部分类
  │
  └── SHFE (上期所) tick数据源
        ├── 期货(铜/金/铝/锌) tick 正常推送 → on_future_tick → 期货方向建立
        ├── 商品期权 al2606  tick 有推送(主力月) → on_option_tick → 61%分类
        ├── 商品期权 al2605  tick 无推送 → 永远'other'
        ├── 商品期权 cu*     tick 无推送 → 永远'other'
        ├── 商品期权 au*     tick 无推送 → 永远'other'
        └── 商品期权 zn*     tick 无推送 → 永远'other'
```

### 6.2 自洽性说明

**CFFEX vs SHFE**：两个交易所数据源独立。CFFEX的数据源对期权tick推送完整，SHFE的数据源对期权tick推送极不完整。这不是Python代码能控制的差异。

**AL2606 vs CU/AU/ZN**：都通过同一个SHFE数据源。AL2606获得推送，其他没有。原因不在于Python代码（代码路径完全相同），而在于：
1. AL2606期权在该交易日可能有实际交易发生（流动性足够触发tick生成）
2. CU/AU/ZN期权在该交易日无交易（成交量极低或为零），平台因此无tick生成
3. 或者平台SHFE期权数据源的tick推送仅覆盖部分品种

## 八、结论声明

1. **Python代码中不存在产品相关的分支逻辑**，所有CU/AU/AL/ZN期权经过的代码路径100%一致。
2. **tick数据未到达的观测是客观事实**：PROBE_RAW_INSTRUMENT_ID零条SHFE商品期权、ContractWatch零tick（30秒采样）、18:41诊断所有CU/AU/ZN期权零分类——三重独立证据收敛。
3. **AL2606确实是唯一收到tick的SHFE商品期权月份**，但这不是代码差异造成，而是该月份恰好有实际交易活动（潜在因：AL2606是主力合约月，其期权有做市商报价）。
4. **Python代码有多个防御性缺陷**——如果修复缺陷4（僵死检测）和增加缺陷3（入口日志），未来类似问题可以第一时间发现，而不必等到2.5小时后看诊断输出。
