## 一、总体验证结论

本轮核查结论是: 原报告总体可信度高，核心风险判断基本成立，尤其是以下几类问题具有明确源码证据支撑:

1. 裸 except 与静默吞异常确实存在，且集中出现在 data_service.py、ui_service.py、risk_service.py、strategy_historical.py、width_cache.py 等关键路径。
2. 封装穿透属实，典型如 position_service.py 直接修改 risk_service 的私有状态，query_service.py 直接访问 storage._params_service。
3. 大量延迟导入与全局单例属实，依赖关系隐式化问题明显。
4. 原报告对 CRITICAL 模块的风险排序基本可靠，data_service.py、ui_service.py、strategy_lifecycle_mixin.py 的高风险判断有足够证据支持。

但原报告并非所有统计值都精确，至少存在两类应修正的事实项:

1. 裸 except 统计写为 3 处，但本轮精确搜索到 4 处。
2. HIGH 模块数量写为 7，但其自身列出的 HIGH 模块名单实际为 9 个。

因此，本核查报告的结论是:

- 原报告的方向、风险优先级、主要技术债判断可信。
- 原报告中的个别硬统计与汇总表述需要修正后再作为正式验收材料使用。

## 二、原报告需修正的事实项

### 2.1 裸 except 数量低估

本轮对 ali2026v3_trading 下 Python 文件执行精确搜索，命中 4 处裸 except:

1. data_service.py:2443
2. ui_service.py:449
3. ui_service.py:462
4. ui_service.py:492

结论: 原报告第 7.1 节写成 3 处，不准确，应修正为至少 4 处。

### 2.2 HIGH 模块数量与名单不一致

原报告“风险等级分布”中 HIGH 数量写为 7，但其列出的模块为:

1. diagnosis_service.py
2. config_service.py
3. width_cache.py
4. subscription_manager.py
5. storage_query.py
6. position_service.py
7. strategy_core_service.py
8. strategy_tick_handler.py
9. t_type_service.py

结论: 名单实际为 9 个，汇总数字需修正。

### 2.3 静默异常总量不宜写成精确值

原报告中对 except Exception: pass 写成 30+ 处，这个方向判断大概率成立，但若要作为正式统计，应区分:

1. 一行写法: except Exception: pass
2. 两行写法: except Exception: 换行后 pass
3. except Exception 后仅 warning/debug 的情况

本轮已确认 data_service.py 单文件就存在 6 处一行写法，因此“问题存在且不轻”是成立的；但若要给出全仓精确数字，建议单独重跑统计脚本后再落正式版。

## 三、全局问题核查

### 3.1 静默异常问题

已核实。

直接证据:

1. data_service.py:630-643 连续 6 处一行 except Exception: pass。
2. data_service.py:2443 为裸 except。
3. ui_service.py:449、462、492 为裸 except。
4. risk_service.py:347 的异常分支直接 pass，可能导致限额检查被跳过。
5. strategy_historical.py:692 为 except Exception 后 pass。
6. width_cache.py:276、290、1364 存在 except Exception 后静默返回或跳过。

判断: 原报告关于“交易关键路径存在静默失败风险”的核心结论成立。

### 3.2 全局单例泛滥

已核实。

直接证据:

1. params_service.py 使用模块级 _params_service_instance 与 _params_service_lock。
2. __init__.py 使用 _instrument_data_manager_instance 维护全局 InstrumentDataManager。
3. 原报告点名的多个 service 文件都采用 get_xxx_service() 风格的全局入口。

判断: 原报告关于单例分布广、缺少统一生命周期治理的判断成立。

### 3.3 延迟导入与隐式依赖

已核实。

直接证据:

1. diagnosis_service.py:910 在运行路径中导入 config_service.get_cached_params。
2. query_service.py:503、772、797、1951 运行时导入 params_service。
3. strategy_lifecycle_mixin.py 的订阅重试与初始化逻辑包含多处延迟绑定和延迟调用。

判断: 原报告关于“IDE 与静态分析不易看清真实依赖图”的结论成立。

### 3.4 封装穿透

已核实。

直接证据:

1. position_service.py:315-321 直接进入 self._risk_service._lock，并写入 self._risk_service._position_limits。
2. query_service.py:296、355-356、409、420、1038 直接访问 self._storage._params_service。
3. storage_core.py:227、235、504 直接访问 self._params_service._instrument_meta_by_id。

判断: 原报告关于迪米特法则破坏的结论成立，且问题并非个例。

### 3.5 硬编码路径与密钥

已核实。

直接证据:

1. config_service.py:798-800 默认路径硬编码为 D:\\ali2026_data\\ticks.duckdb 与 D:\\ali2026_data\\ticks.parquet。
2. config_service.py:1368-1371 出现默认 API key / secret 字符串。

判断: 原报告关于硬编码问题的判断成立。
