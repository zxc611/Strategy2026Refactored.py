# ali2026v3_trading 容器化部署方案

**日期**: 2026-05-09
**版本**: v1.1（增加IP保护+Cython编译方案）
**项目**: ali2026v3_trading 量化交易系统
**模块总数**: 42个（含__init__.py）

---

## 1. 设计目标

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **修复便利** | 策略逻辑修改无需重建Docker镜像，volume mount热更新.py文件即可 |
| **数据安全** | 数据管道+存储层在容器外，策略容器崩溃不丢数据、不断管道 |
| **故障隔离** | 策略容器崩溃自动重启，不影响基础服务；基础服务独立滚动更新 |
| **变更频率分层** | 高频修改模块放容器内，稳定基础设施放容器外 |
| **IP保护** | 核心算法模块Cython编译为.so，防止源码泄露；非算法模块保留.py可热更新 |

### 1.2 Docker对策略的保护作用

```
策略容器崩溃场景：
  容器外：DuckDB数据完好 + AsyncWriter管道继续运行 + 注册信息保留
  容器内：策略进程重启 → 热加载最新.py → 自动重新订阅 → 恢复交易

基础服务升级场景：
  只读镜像滚动更新 → 新Pod启动 → 健康检查通过 → 旧Pod终止
  策略容器不受影响（通过event_bus解耦通信）
```

## 4. 关键边界决策说明

### 4.1 shared_utils.py → 容器内

**反直觉**：扇入35，看似应放容器外。

**理由**：
- 无状态无IO，纯函数工具集
- 8轮修复曾修改（删.upper()），放容器外需重建镜像
- 所有调用者都是容器内模块（35个扇入中29个是容器内模块）
- 放容器内：修改→策略重启即生效；放容器外：修改→重建infra镜像+滚动更新

### 4.2 query_service.py → 容器内

**争议**：包含_KlineAggregator（数据管道组件），也包含register_instrument（注册逻辑）。

**理由**：
- 主要职责是CQRS Query层——为策略提供查询接口
- 扇入5全部来自策略侧（strategy_lifecycle_mixin等）
- _KlineAggregator是策略侧K线合成，非存储管道
- register_instrument被策略初始化调用，属于策略组装

### 4.3 subscription_manager.py → 容器外

**反直觉**：9轮修复频繁修改。

**理由**：
- 扇入29，全局注册中心
- 15个模块依赖它（含Layer2的storage_core、ds_data_writer等）
- 容器崩溃时：放容器内=注册信息丢失需重新加载2752条期权；放容器外=注册信息保留
- 修复频率虽高，但每次修复后趋于稳定（8轮已删除3条旧注册路径，架构已收敛）

### 4.4 diagnosis_service.py → 容器内

**反直觉**：扇入28，看似应放容器外。

**理由**：
- 扇入28全部是import诊断函数的调用点，不是被依赖的基础服务
- 诊断是"观察者"角色，崩溃不影响交易主链路
- 调优高频修改（阈值、宽限期、监控合约），热更新价值高

### 5.2 基础服务Dockerfile

```dockerfile
# docker/infra/Dockerfile
FROM python:3.11-slim

WORKDIR /app

# 依赖（Layer 2自己的依赖）
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 基础服务代码（只读，需重建镜像才能更新）
COPY code/ /app/ali2026v3_trading/

# 数据目录（Layer 1挂载）
VOLUME ["/data", "/app/config", "/app/logs"]

ENV PYTHONPATH=/app
ENV DB_PATH=/data/ticks.duckdb

# 健康检查
HEALTHCHECK --interval=30s --timeout=5s \
    CMD python -c "import duckdb; duckdb.connect('${DB_PATH}', read_only=True).close()" || exit 1

CMD ["python", "-m", "ali2026v3_trading.infra_server"]
```

### 5.3 策略Dockerfile

```dockerfile
# docker/strategy/Dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 策略代码不COPY进镜像，通过volume mount热更新
# 镜像仅包含运行时依赖+entrypoint

ENV PYTHONPATH=/app
ENV STRATEGY_CODE_PATH=/app/strategy

COPY entrypoint.sh .
RUN chmod +x entrypoint.sh

# 健康检查：策略进程存活+最近tick时间
HEALTHCHECK --interval=10s --timeout=3s \
    CMD python -c "import os; assert os.path.exists('/app/strategy/main.py')" || exit 1

ENTRYPOINT ["./entrypoint.sh"]
```

### 5.4 entrypoint.sh

```bash
#!/bin/bash
# 策略启动脚本：等待基础服务就绪后启动策略

echo "[Strategy] Waiting for infra service..."
while ! python -c "import socket; s=socket.socket(); s.connect(('infra', 8520)); s.close()" 2>/dev/null; do
    sleep 1
done

echo "[Strategy] Infra service ready. Starting strategy..."
cd /app/strategy
exec python -m ali2026v3_trading.main "$@"
```

## 6. 热更新操作手册

### 6.1 策略逻辑热更新（Layer 3，零停机）

```bash
# 步骤1：修改宿主机上的.py文件
vim /opt/ali2026/strategy/signal_service.py

# 步骤2：策略重启（容器内进程重启，不重建镜像）
kubectl rollout restart deployment/ali2026-strategy

kubectl logs -f deployment/ali2026-strategy --tail=50
```

```bash
# 步骤1：修改infra代码
vim docker/infra/code/storage_core.py

docker push registry.cn-shanghai.aliyuncs.com/ali2026/infra:v1.2.1

kubectl set image deployment/ali2026-infra infra=registry.cn-shanghai.aliyuncs.com/ali2026/infra:v1.2.1
```

## 8. 容器间通信机制

### 8.1 通信方式

```
策略容器 ──→ 基础服务容器
  │              │
  │  1. HTTP/gRPC: 策略调用基础服务API
  │     - process_tick: tick写入+K线聚合
  │     - register_instrument: 合约注册
  │     - query_kline: K线查询
  │     - query_tick: tick查询
  │
  │  2. Event Bus (跨容器): 发布/订阅解耦
  │     - 策略→infra: KLineCompleted, SubscriptionCompleted
  │     - infra→策略: MarketDataUpdate, InstrumentRegistered
  │
  │  3. 共享DuckDB (只读): 策略直接读DB
  │     - 避免查询走网络，降低延迟
  │     - 写操作必须通过infra服务（保证单写一致性）
```

### 8.2 接口定义（需从直接函数调用迁移为服务接口）

当前状态：策略直接import infra模块调用函数。容器化后需改为远程调用。

| 当前调用 | 迁移为 | 优先级 |
|---------|--------|--------|
| `self.storage.process_tick(tick)` | `infra_client.process_tick(tick)` | P0 |
| `self.storage.register_instrument(id)` | `infra_client.register_instrument(id)` | P0 |
| `self.storage.query_kline(id, period)` | `infra_client.query_kline(id, period)` 或 直接读DuckDB | P1 |
| `SubscriptionManager.is_option(id)` | `infra_client.is_option(id)` | P1 |
| `get_params_service().get_meta(id)` | `infra_client.get_instrument_meta(id)` | P1 |

**注意**：此迁移是后续工作，当前方案先以volume mount+同进程运行的方式部署，逐步拆分为跨容器通信。

### 11.4 编译脚本

#### setup.py

```python
# cython_build/setup.py
from setuptools import setup, Extension
from Cython.Build import cythonize
import numpy as np

# 需要编译为.so的核心算法模块
CYTHON_MODULES = [
    'greeks_calculator',
    'width_cache',
    't_type_service',
    'signal_service',
    'position_service',
    'risk_service',
]

extensions = [
    Extension(
        f"ali2026v3_trading.{mod}",
        sources=[f"ali2026v3_trading/{mod}.py"],
        include_dirs=[np.get_include()] if mod == 'greeks_calculator' else [],
        define_macros=[('NPY_NO_DEPRECATED_API', 'NPY_1_7_API_VERSION')]
        if mod == 'greeks_calculator' else [],
    )
    for mod in CYTHON_MODULES
]

setup(
    name="ali2026v3_trading_compiled",
    ext_modules=cythonize(
        extensions,
        compiler_directives={
            'language_level': '3',
            'boundscheck': False,      # 生产环境关闭边界检查（性能）
            'wraparound': False,       # 关闭负索引支持（性能）
            'cdivision': True,         # C除法（无Python零除检查）
            'embedsignature': True,    # 嵌入函数签名到docstring
        },
        annotate=True,                 # 生成HTML性能分析报告
    ),
)
```

#### compile.sh

```bash
#!/bin/bash
# cython_build/compile.sh — 一键编译核心算法模块
set -e

SCRIPT_DIR="$(cd "$(dirname "$0")" && pwd)"
PROJECT_ROOT="$(dirname "$SCRIPT_DIR")"

cd "$PROJECT_ROOT"

find . -name "*.c" -path "*/ali2026v3_trading/*" -delete

# 步骤3：验证编译产物
```

```
  t_type_service.so      → C机器码，五态分类逻辑保护
  signal_service.so      → C机器码，信号生成逻辑保护
  position_service.so    → C机器码，持仓计算逻辑保护
  risk_service.so        → C机器码，风控模型保护

Layer 3 .py模块（21个）：
  strategy_lifecycle_mixin.py  → 明文可读，但不含核心算法
  shared_utils.py              → 明文可读，仅工具函数
  ...其余均为框架/IO/配置逻辑，非核心IP
```

#### 保护评估

| 维度 | 评估 |
|------|------|
| 核心算法保护率 | 6/27 = 22%模块，但覆盖100%交易决策IP |
| 源码泄露风险 | .so模块：极低（C机器码）；.py模块：中（框架逻辑可读，但非核心算法） |
| 逆向工程成本 | .so需IDA Pro/Ghidra反编译→伪C代码→人工理解算法，成本极高 |
| 热更新影响 | .so模块更新需重建镜像；.py模块热更新不受影响 |

#### 与纯.py部署对比

| 方案 | IP保护 | 热更新比例 | 性能 |
|------|--------|-----------|------|
| 全.py部署 | 无（源码完全暴露） | 100% | 基准 |
| Cython混合部署 | 核心算法保护 | 21/27=78% | 算法模块2-10x加速 |
| 全.so部署 | 完全保护 | 0% | 全模块加速 |

**Cython混合部署是最优平衡**：核心IP保护+78%热更新能力+算法性能提升。

### 11.7 热更新与.so更新的操作差异

```bash
# 场景1：修改.py模块（如strategy_lifecycle_mixin.py）
# → 直接热更新，零停机
vim /opt/ali2026/strategy/strategy_lifecycle_mixin.py
kubectl rollout restart deployment/ali2026-strategy

# 场景2：修改.so模块源码（如greeks_calculator.py）
# → 需重新编译+重建镜像
vim ali2026v3_trading/greeks_calculator.py
cd cython_build && bash compile.sh              # 重新编译
docker build -t ali2026-strategy:v1.2.1 ./docker/strategy
docker push registry/ali2026/strategy:v1.2.1
kubectl set image deployment/ali2026-strategy strategy=registry/ali2026/strategy:v1.2.1
```

### 11.8 Cython编译注意事项

| 注意项 | 说明 | 处理 |
|--------|------|------|
| Python动态特性 | `getattr`/`hasattr`/`__dict__`在Cython中受限 | 编译模块避免使用动态属性访问，改用直接属性引用 |
| 延迟导入 | 函数内`from xxx import yyy`在.so中正常工作 | 无需修改，Cython支持 |
| 类型注解 | Python type hints在Cython中变为C类型声明 | 可选优化：为热点函数添加`cdef`类型声明 |
| 调试困难 | .so中异常堆栈为C帧 | 开发环境用.py，生产环境用.so；Docker多阶段构建天然支持 |
| numpy依赖 | greeks_calculator使用math，未用numpy | 如需加速可引入numpy+memoryview，但当前math已足够 |
| 跨模块继承 | Mixin继承链在.so中正常工作 | Cython支持Python class继承 |
