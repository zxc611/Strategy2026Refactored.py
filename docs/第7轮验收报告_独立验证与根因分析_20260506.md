### 2.2 核心谜题: `_option_info` 如何从 0 → 2752 而所有路径计数为 0?

经过全面代码排查，**`_option_info` 只在 `width_cache.register_option()` L669 处写入**，调用链只有4条:

1. **Path A** `_preload_from_db` (t_type_service.py L423) → BROKEN 代码 KeyError 在 L390，注册前已失败
2. **Path B** `_preload_options_into_t_type` (strategy_lifecycle_mixin.py L499) → "No option prices found" → 0 rows
3. **Path C** `_auto_register_options_to_ttype` (subscription_manager.py L1152) → 2752 failures (ValueError)
4. **Path C 调用 `register_option_contract`** → 同上

所有路径自己报告的注册数均为 0。但 `_option_info` 在两次 `get_cache_stats()` 调用之间从 0 增长到 2752。

### 2.3 唯一自洽的解释

经过对日志的深度交叉比对，确认日志中**没有任何 `[WidthStrengthCache] register_option called:` 或 `_option_info[...]` 条目**（这些 log 在 `register_option()` L632/L680 发出）。这意味着:

**2752条 `_option_info` 条目并非在本 session 中通过 `register_option()` 添加，而是通过以下机制之一进入:**

#### 机制A（最可能）: 平台状态恢复/反序列化

InfiniTrader 平台在策略重启时可能对 WidthStrengthCache 执行 pickle/反序列化，直接恢复 `_option_info` 字典的内部状态，**不经过 `register_option()` 函数调用**，因此不产生 log。若此机制存在:

- 第1次预加载 (16:08:33) 时反序列化尚未完成 → Options: 0
- 第2次预加载 (16:09:08) 时反序列化已完成 → Options: 2752
- 35秒的延迟是平台异步恢复的耗时

#### 机制B: 模块级单例的旧数据残留

`get_t_type_service()` (t_type_service.py L633-639) 使用模块级单例:
```python
_t_type_service_instance: Optional['TTypeService'] = None
```

若 Python 进程在策略重启时未被完全终止（"热重启"），旧的 `_t_type_service_instance` 及其 `_width_cache._option_info` 可能残留。

但第1次预加载显示 Options: 0，说明单例可能被显式重置或 WidthStrengthCache 被重新构造。35秒后恢复到2752，可能是:

- **延迟的注册回调**：当 ParamsService 完成初始化后，触发了针对已订阅期权的批量注册逻辑（可能在下层/框架层，未在此三个 Python 路径中体现）
- **多个 WidthStrengthCache 实例**：`get_cache_stats()` 在 L450 读取的是 `self._width_cache._option_info`。若 `self._width_cache` 在第1次和第2次预加载之间被替换或重新指向（例如通过热修复重新导入了模块），则新实例已有2752条数据

### 2.4 真正路径总结

AL/指数期权数据能成功流转的核心链路是:

```
ticks_raw (历史数据)
  → latest_prices VIEW (实时聚合)
    → Path A/B/C 任意一条成功后
      → width_cache.register_option() → _option_info 注册
        → on_option_tick() 收到 tick
          → _update_option_direction() 计算方向
            → _classify_status() 归类 5态 (非other)
```

**AL独有的优势**: `ticks_raw` 中有 14,091 条历史数据（2026年4月采集期），`latest_prices` VIEW 因此保留了 AL 期权的价格快照。注册时 `initial_price > 0`，首个 tick 即可确定方向。

## 三、AL成功而CU/AU/ZN失败的根因分析（自洽版）

### 3.1 前提条件确认

| 条件 | AL | CU | ZN | AU |
|------|:--:|:--:|:--:|:--:|
| 同一交易所 (SHFE) | ✅ | ✅ | ✅ | ✅ |
| 同一数据源 | ✅ | ✅ | ✅ | ✅ |
| 同一代码路径 | ✅ | ✅ | ✅ | ✅ |
| 同一月份 (2605/2606) | ✅ | ✅ | ✅ | ✅ |
| 在订阅列表 `subscription_options_fixed.txt` | ✅ | ✅ | ✅ | ✅ |
| 在 `option_instruments` 表 (is_active=true) | 522 | 432 | 234 | 1,074 |

### 3.2 差异点: 历史 tick 数据 (`ticks_raw`)

| 品种 | `ticks_raw` 期权 tick 数 | 涉及合约数 | 数据时间范围 |
|------|------------------------:|----------:|-------------|
| **AL** | **14,091** | 90 | 2026-04-20 ~ 2026-04-23 |
| AU | 32 | 4 | (稀疏) |
| **CU** | **0** | 0 | N/A |
| **ZN** | **0** | 0 | N/A |

`latest_prices` 数据与此一致:
- AL: 95 个期权有 `last_price > 0`
- AU: 9 个
- CU: 8 个
- ZN: 3 个

### 3.3 "小条件"揭示: `ticks_raw` 历史数据的存在性

这是 AL 满足而 CU/ZN/AU 不满足的关键条件。根因链路如下:

```
                     P1 重构引入 SQL 别名 bug (KeyError: 'last_price')
                                    |
                    ┌───────────────┴───────────────┐
                    |                               |
              重构前的时间窗口                  重构后的时间窗口
         (AL 期权正活跃交易)              (CU/ZN/AU 期权活跃期)
                    |                               |
            ticks_raw 正常采集               ticks_raw 采集失败
          AL: 14,091 条历史数据           CU: 0, ZN: 0 (被 on_option_tick
                                              return 掉，从未入库)
                    |                               |
          latest_prices > 0                 latest_prices = NULL
                    |                               |
         register_option 时有价格         register_option 时价格=0
         option_price 正确设置            option_price 被 pop(不设置)
                    |                               |
         首个 tick 即可计算方向            首个 tick: option_direction=None
         状态从 'other' 变为非other         _classify_status → 'other'
                                          需第2个不同价格的 tick 才能翻状态
```

**这个解释是自洽的**:

1. AL 数据在 **P1 重构之前**（2026年4月）已经被正常捕获到 `ticks_raw`
2. P1 重构引入 `row['last_price']` KeyError bug → 所有 SHFE 期权的路径A预加载全部失败 → `_option_info` 为空 → `on_option_tick` 在 L770-773 检查 `_option_info` 为空 → 直接 return → tick 被丢弃
3. CU/ZN/AU 期权的活跃期恰好在 **P1 重构之后**，它们的 tick 数据从未成功入库
4. 因此形成了一个**自我强化的数据黑洞**: 
   - 历史 session: bug → tick 不入库 → `latest_prices` = NULL
   - 当前 session: `latest_prices` = NULL → register 时无价格 → 需更多 tick 才能翻状态 → 依然容易卡在 'other'

5. 即使 bug 已被修复（当前代码 L390 已是 `row['option_last_price']`），CU/ZN/AU 缺少历史数据的事实已成定局。他们需要在新 session 中**持续 2+ 个不同价格的 tick**才能累积足够的 `last_distinct_price` 进行方向判断。

### 3.4 与"同一数据源"的一致性

"同一数据源"（交易所行情）指**实时行情订阅**，而非 `ticks_raw` 历史表。二者的区别:

| | 实时行情 | `ticks_raw` 历史表 |
|--|---------|-------------------|
| AL | ✅ 有数据 | ✅ 14,091条 (4月积累) |
| CU | ✅ 有数据 | ❌ 0条 (bug期间未入库) |
| ZN | ✅ 有数据 | ❌ 0条 (bug期间未入库) |
| AU | ✅ 有数据 | ❌ 32条 (仅4个合约) |

实时行情对所有 SHFE 品种均正常推送，但 `width_cache` 处理这些行情时依赖 `_option_info` 中的历史价格 (`_option_price`) 来判断方向。没有历史价格的期权需要等待至少 **2个不同价格的 tick** 才能摆脱 'other' 状态。

### 3.5 当前状态预期

修复已应用到磁盘文件（L390 → `row['option_last_price']`），下一次策略**冷重启**后:

- Path A 会成功注册全部 2752 个期权
- AL/CFFEX 等指数期权因为 `latest_prices` 中有价格，会立即进入非 other 状态
- CU/ZN/AU 因为 `latest_prices` 中价格仍可能为 NULL，仍会在首个 tick 时为 'other'
- 但随着 session 运行，CU/ZN/AU 的 tick 会被正常处理 → 逐合约积累 2+ tick → 逐步脱离 'other' 状态
- 同时 CU/ZN/AU 的 tick 会被写入 `ticks_raw` → `latest_prices` 逐渐有数据 → 后续 session 受益

**完全恢复需要时间 + 足够的行情量。**
