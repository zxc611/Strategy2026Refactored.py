## 二、调整方案（上中下三策）

### m2-01 上策：最小侵入式Dry-Run拦截层（推荐）

**核心思路**: 在 `_platform_insert_order` 调用前插入一个拦截层，将实际下单替换为日志记录，同时保留订单全生命周期管理。

| 编号 | 调整项 | 文件 | 具体操作 | 风险 |
|------|--------|------|----------|------|
| A-01 | 新增 `dry_run_mode` 参数 | `config/params.yaml` | 在engineering层新增 `dry_run_mode: default: true, type: bool` | 极低 |
| A-02 | OrderExecutor拦截 | `order/order_executor.py:586` | `_execute_platform_insert` 中检查dry_run_mode，若为True则跳过`_platform_insert_order`调用，改为记录日志+模拟返回成功 | 低 |
| A-03 | 取消下单拦截 | `order/order_executor.py:1402` | `svc._platform_cancel_order(platform_id)` 调用前检查dry_run_mode，若为True则跳过并记录日志 | 低 |
| A-04 | 关闭评判引擎 | `strategy_ecosystem/services.py:853-885` | CascadeJudge调用前检查dry_run_mode，若为True则跳过评判，直接放行 | 低 |
| A-05 | 关闭CyclicDependencyGuard | `order_executor.py:215-224` | dry_run_mode下跳过循环依赖检查 | 中（需确认无真循环） |
| A-06 | 保留数据链路不变 | - | tick_dispatch/subscription_service/data_service不做任何修改 | 无 |
| A-07 | 保留信号生成不变 | - | signal_service/signal_filter_chain不做修改 | 无 |
| A-08 | 持仓链路需配套模拟回调 | `strategy_business_layer.py:509-575,136-160` | ⚠️ update_position_from_trade仅由on_order/on_trade平台回调触发；dry_run拦截_platform_insert_order后回调链断裂，必须配套模拟成交回报（详见m3-06），否则持仓/PnL链路无法验证 | 中 |

**优势**: 改动≤5个文件，新增1个参数，不破坏任何现有逻辑，dry_run_mode=False即可恢复实盘。
**劣势**: 需要新增代码逻辑。

### m2-02 中策：平台层替换法

**核心思路**: 不修改策略代码，而是在 `lifecycle_bind.py` 绑定平台API时，将 `insert_order` 替换为一个空操作函数。

| 编号 | 调整项 | 文件 | 具体操作 | 风险 |
|------|--------|------|----------|------|
| B-01 | 替换insert_order | `lifecycle/lifecycle_bind.py:55` | 将 `_platform_insert_order` 替换为返回0的lambda | 中 |
| B-02 | 替换cancel_order | `lifecycle/lifecycle_bind.py:56` | 同理替换 | 中 |
| B-03 | 关闭评判 | 同A-04 | - | 低 |

**优势**: 改动更少（2-3个文件）。
**劣势**: (1) 平台回调on_order/on_trade不会触发，持仓/PnL链路无法验证；(2) 无法验证订单全生命周期；(3) 返回值格式可能与平台不一致导致后续逻辑异常；(4) lambda替换会破坏`_platform_insert_order.__self__`属性，使`order_executor.py:610-611`的`_trading_flag`取值为None，R37-TRADING-RECOVERY恢复路径与trading状态日志路径异常。

### m2-03 下策：环境变量硬阻断

**核心思路**: 设置环境变量或修改平台配置，使CTP连接处于"只读"模式。

| 编号 | 调整项 | 文件 | 具体操作 | 风险 |
|------|--------|------|----------|------|
| C-01 | 平台trading=False | 外部配置 | 设置策略宿主对象的trading=False | 高 |
| C-02 | 关闭评判 | 同A-04 | - | 低 |

**优势**: 零代码修改。
**劣势**: (1) ⚠️ 经核查 `order_executor.py:610-633` 的 R37-TRADING-RECOVERY 逻辑：`trading=False` 时 CLOSE 订单会被强制 `setattr(_trading_flag, 'trading', True)` 并实际下单重试，**仍会穿透到 CTP**，与"零下单"目标直接冲突；只有 OPEN 订单尊重风控决策不恢复；(2) 数据链路可能受影响（部分平台只读模式下不推送完整tick）；(3) 无法验证交易链路完整性，违背"跑通两链路"之目标。

### m3-06 成交回报模拟方案（关键设计，解决A-08链路断裂）

**背景**: dry_run 拦截 `_platform_insert_order` 后，平台永远收不到订单 → `on_order`/`on_trade` 回调永不触发 → `strategy_business_layer.py:155` 的 `_on_order_filled_trigger_position_update` 永不执行 → `update_position_from_trade` 永不调用 → 持仓/PnL/WAL 链路全部断裂。本方案通过 OrderExecutor 主动构造虚拟回报注入业务层，闭合订单生命周期。

**注入点**: `order_executor.py` 的 `_simulate_dry_run_callbacks(ctx)` 方法（m3-02 中已调用）

**实现伪代码**:
```python
def _simulate_dry_run_callbacks(self, ctx):
    """dry_run下构造虚拟on_order/on_trade回报，闭合订单生命周期"""
    import threading, time
    _delay = getattr(self, '_dry_run_callback_delay_sec', 0.05)  # 默认50ms模拟网络往返

    def _fire():
        time.sleep(_delay)
        _platform_id = ctx.order.get('platform_order_id', f"DRY_{ctx.order_id}")
        # 1. 构造虚拟 on_order 回报（全部成交）
        _virtual_order = {
            'order_id': _platform_id,
            'instrument_id': ctx.instrument_id,
            'direction': ctx.direction,
            'offset': ctx.action,
            'volume': ctx.volume,
            'price': ctx.price,
            'status': '全部成交',  # 触发 _is_filled 分支
            'memo': ctx.order_id,  # FIX-R34-MEMO: memo携带内部ID
        }
        # 2. 构造虚拟 on_trade 回报
        _virtual_trade = {
            'trade_id': f"DRY_TRADE_{ctx.order_id}",
            'order_id': _platform_id,
            'instrument_id': ctx.instrument_id,
            'direction': ctx.direction,
            'offset': ctx.action,
            'volume': ctx.volume,
            'price': ctx.price,
            'memo': ctx.order_id,
        }
        try:
            _provider = getattr(self, '_provider_ref', None)
            _bl = getattr(_provider, '_business_layer', None) or _provider
            if hasattr(_bl, 'on_order'):
                _bl.on_order(_virtual_order)
            if hasattr(_bl, 'on_trade'):
                _bl.on_trade(_virtual_trade)
            logging.info("[DRY-RUN] 虚拟回调已注入: order_id=%s trade_id=%s",
                         ctx.order_id, _virtual_trade['trade_id'])
        except (ValueError, KeyError, TypeError, RuntimeError, AttributeError) as e:
            logging.warning("[DRY-RUN] 虚拟回调注入失败(非阻断): %s", e)

    threading.Thread(target=_fire, name='DryRunCallback', daemon=True).start()
```

**关键设计决策**:
- **异步注入**: 使用 daemon 线程延迟注入，模拟真实平台异步回调时序，避免同步调用导致回调链中再次触发下单形成递归
- **memo字段携带内部ID**: 复用现有 FIX-R34-MEMO 映射机制，确保 `_build_mapping_from_memo` 正常工作
- **status='全部成交'**: 触发 `_is_filled` 分支，使 `_on_order_filled_trigger_position_update` 被调用
- **延迟参数化**: `_dry_run_callback_delay_sec` 默认 50ms，可在 params.yaml 调整
- **失败容忍**: 虚拟回调注入失败仅记录 warning，不阻断 dry_run 主流程

**新增参数**:
```yaml
dry_run_callback_delay_sec:
  default: 0.05
  type: float
  range: [0.0, 5.0]
  layer: engineering
  editable: runtime
  description: "dry_run模式下虚拟回调注入延迟(秒)，模拟平台网络往返"
  source: design
  evidence: "50ms≈CTP本地回报典型延迟"
```

**验证收益**: 通过此方案可验证的链路从原本的 4 项扩展到 8 项：
| 链路 | 验证项 | 原方案 | 本方案 |
|------|--------|--------|--------|
| 订单生命周期 | SUBMITTED→DRY_RUN_ACCEPTED | ✅ | ✅ |
| 订单生命周期 | DRY_RUN_ACCEPTED→FILLED | ❌ | ✅ |
| 持仓服务 | update_position_from_trade | ❌ | ✅ |
| PnL计算 | on_trade→PnL更新 | ❌ | ✅ |
| WAL审计 | trade WAL记录 | ❌ | ✅ |

| 风控dedup | _processed_trade_ids | ❌ | ✅ |

```python
def _assert_dry_run_state():
    """启动时验证dry_run模式配置正确加载"""
    try:
        from ali2026v3_trading.config.config_service import get_cached_params
        _params = get_cached_params()
        _dry_run = _params.get('engineering', {}).get('dry_run_mode', False)
        _stress = _params.get('engineering', {}).get('stress_test_mode', False)
        # 1. 互斥性校验
        assert not (_dry_run and _stress), \
            "[STARTUP-ASSERT] dry_run_mode 与 stress_test_mode 不可同时开启"
        # 2. dry_run模式横幅
        if _dry_run:
            logging.warning("=" * 60)
            logging.warning("[DRY-RUN] 策略以模拟模式启动，不会实际下单！")
            logging.warning("[DRY-RUN] 订单将被拦截并记录，虚拟回调将注入")
            logging.warning("[DRY-RUN] 切回实盘：将 params.yaml:dry_run_mode 设为 false 并重启")
            logging.warning("=" * 60)
        return True
    except AssertionError as e:
        logging.error("[STARTUP-ASSERT] 配置校验失败: %s", e)
        return False
    except Exception as e:
        logging.warning("[STARTUP-ASSERT] dry_run状态读取失败(默认按实盘处理): %s", e)
        return False
```

## 九、结论

**推荐采用上策（A方案）**，核心改动为：

1. **新增2个参数**: `dry_run_mode`与`dry_run_callback_delay_sec`（engineering层，runtime可改）
2. **修改3个文件**: `order_executor.py`（下单拦截+cancel拦截+虚拟回调注入）、`strategy_ecosystem/services.py`（评判跳过）、`params.yaml`（参数定义+互斥规则）
3. **新增1条互斥规则**: `[dry_run_mode, stress_test_mode]`（见m4-04）
4. **新增启动assertion**: `_assert_dry_run_state`（见m3-07）
5. **调整7个参数**: 见m4-02清单（lots_min已移至m4-03不调整）
6. **保持不变**: 数据链路全部模块、信号链路全部模块、风控服务、订单生命周期状态机
7. **核心创新**: m3-06虚拟成交回报注入方案，闭合订单生命周期至FILLED，使持仓/PnL/WAL/平台ID映射等8项链路可验证

此方案最小侵入、可逆、不破坏现有逻辑：
- `dry_run_mode=false`即可完全恢复实盘行为
- 虚拟回调失败仅记录warning，不阻断主流程
- 启动assertion确保配置互斥性正确
