# 第7轮验收报告：CU/ZN/AU期权'other'状态根因修复

**日期**: 2026-05-05

**严重等级**: P0 — 导致2752个期权5态判定全部失败

## 1. 问题描述

CU/ZN/AU期权在`t_type_service`中始终为'other'状态（5态判定失败），AL/CFFEX期权正常。

## 2. 根因分析

### 2.1 主要根因：`_query_latest_options()` SQL别名回归（已修复）

| 项目 | 详情 |
|------|------|
| **文件** | `t_type_service.py` |
| **原因** | P1 SRP拆分重构时，L279 SQL将`lp.last_price`改为`lp.last_price AS option_last_price`（加别名），但L390循环代码仍用`row['last_price']`访问 |
| **后果** | KeyError → 路径A（`_preload_from_db`）2752个期权全部预加载失败 → `_option_info`为空 → `on_option_tick`中`info=None` → return → 期权5态始终为'other' |
| **对比** | git原始代码(`c48dec3`)中SQL无别名+`row['last_price']`匹配，是正确的 |
| **影响范围** | SHFE全部期权（CU/ZN/AU/AL等）+ DCE期权均受影响；CFFEX期权因另有注册路径不受此特定KeyError影响 |

### 2.2 机制性根因：`_normalize_instrument_id`大写化（确认非本次修改，但影响tick路由）

| 项目 | 详情 |
|------|------|
| **文件** | `shared_utils.py` L81, `width_cache.py` L768 |
| **机制** | `on_option_tick`将instrument_id大写化（`al2605C18900` → `AL2605C18900`），然后`_resolve_internal_id`查找 |
| **影响** | 若期权未通过`register_option`注册（即路径A失败时），大写key查不到ParamsService中的小写key → tick被丢弃 |
| **CFFEX不受影响原因** | CFFEX期权ID本身大写（`IO2605-C-3950`），`_normalize`不改变 |
| **当前状态** | 路径A修复后，`register_option`注册时也会大写化key存入本地缓存，后续tick大写化后能匹配 → **间接解决** |

### 2.3 辅助根因（待重启后验证是否仍需处理）

| 路径 | 状态 | 原因 |
|------|------|------|
| **路径B** (`strategy_lifecycle_mixin._preload_options_into_t_type`) | 失败 | SQL窗口函数查询`ticks_raw`大表可能超时，`ds.query(raise_on_error=False)`返回None |
| **路径C** (`subscription_manager._auto_register_options_to_ttype`) | failed=2752 | 具体异常类型未确定（debug级日志不输出） |

期权5态'other' bug 根因分析（完整链路）
观测现象：CU/ZN/AU期权在t_type_service中始终为'other'状态（5态判定失败），AL/CFFEX期权正常。

三条期权注册路径：

路径A: t_type_service._preload_from_db() — 后台线程，从option_instruments表+latest_prices查
路径B: strategy_lifecycle_mixin._preload_options_into_t_type() — 同步调用，从latest_prices + ticks_raw查
路径C: subscription_manager._auto_register_options_to_ttype() — 订阅后调用，从params_service metadata查
根因1（已修复，需重启生效）：t_type_service.py L390，P1重构SQL封装时，_query_latest_options() SQL将lp.last_price改为lp.last_price AS option_last_price（加别名），但循环代码仍用row['last_price']访问 → KeyError → 路径A 2752个期权全部预加载失败 → _option_info为空 → on_option_tick中L782 info=None → return → 期权5态始终为'other'。

验证：git原始代码(c48dec3)中SQL无别名+row['last_price']匹配，是正确的。P1重构引入回归bug。当前代码L390已修正为row['option_last_price']，L279 SQL别名也是option_last_price，匹配。

根因2（_normalize_instrument_id大写化问题）：width_cache.on_option_tick L768将instrument_id通过_normalize_instrument_id大写化（al2605C18900 → AL2605C18900），然后_resolve_internal_id查找：

先查本地缓存_instrument_id_to_internal_id（如果通过register_option注册，key是大写 → 能找到）
后查ParamsService（key是小写al2605C18900从DB加载 → 大写查不到 → None）
结果：如果期权未通过register_option注册，tick被丢弃。路径A修复后register_option先注册大写key，后续tick能匹配。
CFFEX不受影响原因：CFFEX期权ID本身就是大写（IO2605-C-3950），_normalize_instrument_id不改变，始终能找到。

SHFE全受影响原因：SHFE期权ID小写（al2605C18900），大写化后ParamsService小写key查不到。但路径A修复后register_option注册时也会大写化key，后续tick大写化后能匹配本地缓存。

路径B失败原因：SQL查询涉及ticks_raw大表的窗口函数（ROW_NUMBER() OVER PARTITION BY），ds.query(raise_on_error=False)可能超时返回None → options_df无iterrows属性 → 循环不执行 → options_loaded=0。

路径C失败原因：日志显示Auto-registered 0 options to TType, failed=2752，具体异常类型未知（debug级日志不输出）。待重启验证路径A修复后是否仍需要路径C。

附带发现1：DB中存在两个表：futures_instrument（单数，0条记录，ds_schema_manager.py:173创建的空表）和futures_instruments（复数，33条记录，实际数据表）。option_instruments.underlying_future_id引用futures_instruments（复数）的internal_id。空表是历史遗留问题。

附带发现2：_get_option_underlying_product()对SHFE期权小写输入（如'cu'）返回原值'cu'（因为option_products字典key是大写"CU"查不到），但对CFFEX（'IO'→'IF'）正常。由于parse_option返回小写product，ensure_products_with_retry中underlying_future_key = (小写product, month)，而future_ids也是小写product（来自parse_future），所以预验证通过。这不是当前bug的原因，但是隐患。

自洽性问题：OptionStatus诊断（18:41）显示AL/CFFEX期权有正确的5态（非零价格），但3条注册路径全部失败，_option_info理论上应为空。这个矛盾尚未完全解释。可能的原因包括：旧进程的残留数据、或者某条路径部分成功的时序问题。重启后路径A修复应消除此矛盾。
