## 二、逐维度详细评判

### 2.1 参数分类严谨性

| | 方案甲 | 方案乙 | 方案丙 |
|---|--------|--------|--------|
| **分层体系** | A/B/C/D 四层：基础运行/扩展/注入变量/条件能力 | 12维P1-P12（时间驱动/数据窗口/排序算法...） | 因子候选池（连续/离散/对数） |
| **核心原则** | "只把代码中被真实读取的参数纳入主池"——区分 config key vs function param vs injection variable | 覆盖所有关键决策分支，但未区分参数"层级" | 接受连续+离散混合，但无层级意识 |
| **评判** | ✅ 这是最关键的架构洞见。`ranking_mode`和`target_top_n`不是config key而是injection variable——这个区分避免了实验变量误写成平台配置 | ⚠️ 将`lots_min`、`top3_rows`等不同层级的参数平铺在同一维度矩阵中——排列组合会混入不可比的组合 | ⚠️ 贝叶斯/LHS采样器对所有参数一视同仁 |

**结论**: 方案甲在这个维度上建立了**不可替代的基线**——方案乙和丙都应以此为准绳重新整理参数层级。

| | 方案甲 | 方案乙 | 方案丙 |

### 2.6 方案甲的独特贡献：治理框架（不可替代）⭐

**E1-E6剔除编码体系**（方案甲 §14.1）:
```
E1: 代码无稳定读取点
E2: DOE主效应不稳定，方向反复
E3: Walk-forward 6窗入选次数 < 2
E4: 收益改善小，但回撤/拒单率/拥塞明显恶化
E5: 最优点高度尖峰化，相邻档位无法形成稳定区间
E6: 只能在测试注入层存在
```

**4类生产治理边界**（方案甲 §14.3）:
```
生产固定值: 工程稳定性确定，只巡检不调优
生产可调值: 白名单区间内微调
研究保留值: 实验池保留，默认不上线
测试注入值: 只允许回测/仿真脚本使用，不进统一配置表
```

**Walk-forward验证**（方案甲 §13）:
- 20训练/5验证/5步长/6窗滚动
- 伪优解淘汰5条规则
- 参数区间升级4条规则
- 参数漂移检测（漂移>1档→退回研究池）

**方案乙和丙完全没有这些。** 没有治理框架的参数池，找到最优参数后不知道能不能上线、上线后会不会漂移、什么条件下应回退。

**为什么这个贡献是颠覆性的**:

- **方案甲**指出了"不能把不同层级参数混入笛卡尔积"，但没有给出"哪两个参数不能同时取极端值"的具体规则——约束体系正是这个规则的**精确形式化**。
- **方案乙**的L18正交表假设"所有组合合法"，但C1-C19显示大量组合是**逻辑上无效的**——如C5约束下cooldown=60s+window=30s的组合根本不应该进入实验矩阵。
- **方案丙**的Optuna采样器对参数空间一无所知——它可能在illegal区域浪费大量trial（如suggest出tp_ratio=0.5+sl_ratio=0.8→C15违反）。

#### 2.7.2 L1-L4 决策分层逻辑（平行于方案甲的A/B/C/D分类）

方案丁提出了**第三套正交分类法**——不同于方案甲的"代码可达性分类"和方案乙的"参数维度分类"：

| 层级 | 含义 | 核心决策 | 关键参数 | 对绩效的阶 |
|------|------|----------|----------|-----------|
| L1 信号生成层 | 五态分类 + 宽度 + 选标 | **"买什么"** | `option_width_min_threshold`, `month_count`, `lots分配`, `sync_tolerance` | **一阶** ⭐⭐ |
| L2 信号过滤层 | 冷却 + 限频 + 风控 | **"买不买"** | `signal_cooldown_sec`, `max_signals_per_window`, `rate_limit_window_sec` | **一阶** (生存) ⭐⭐ |
| L3 持仓管理层 | 止盈止损 + 追单 | **"何时卖"** | `close_take_profit_ratio`, `close_stop_loss_ratio`, `max_hold_days`, `chase_attempts` | **一阶** (盈亏比) ⭐⭐ |
| L4 基础设施层 | 订阅 + K线 + Greeks | **"数据质量"** | `history_minutes`, `batch_size`, `risk_free_rate`, `greeks_update_threshold` | **二阶** ⭐ |

#### 2.7.3 层级间正交性的数学证明

方案丁提供了**DAG（有向无环图）证明**：在19条约束中，**没有任何一条跨层约束**：

```
约束归属:
  L1内部约束: C4, C7, C8, C14       → 4条
  L2内部约束: C3, C5, C6, C12, C16, C18 → 6条
  L3内部约束: C1, C2, C10, C11, C15  → 5条
  L4内部约束: C9, C13, C17, C19      → 4条
  
跨层约束: 0条 ← 这是DAG的关键
```

**推论**: 任何L1参数合法值 + 任何L2参数合法值 + 任何L3参数合法值 = 仍为合法组合。这就是"组间正交"的数学基础——**层间排列组合不会产生非法组合**。

#### 2.7.5 方案丁在"排列组合够不够"问题上的独特视角

方案丁 §4.4 给出了一组量化结论：

> "86组覆盖了主效应+部分交互效应，排列组合**远远不够**覆盖3^39的完整空间。但这是正确的——完整覆盖既不可能也不必要。"

并证明：
```
L1层用L9正交表12组：因子数d=4, 12≥2×4=8 ✓ (可估计主效应)
L2层用Taguchi 12组：因子数d=4, 12≥2×4=8 ✓
L3层用L12正交12组：因子数d=4, 12≥2×4=8 ✓
L4层用L12正交12组：因子数d=4, 12≥2×4=8 ✓
```

——这是基于**分层降维后的N≥2d规则**证明组合数足够，比方案甲的直觉判断或方案乙的L18全覆盖更精确。

## 二B、方案丁对前三方案的"补丁清单"

### 补丁1: 约束体系（对方案甲）

方案甲定义了E1-E6剔除编码——但那是**运行后淘汰**。方案丁的C1-C19约束定义了**运行前拦截**。两者配合：

```
参数生成 → [C1-C19约束检查] → 非法→拒绝 → 跳过这组
                                 ↓ 合法
                              → 进入方案甲的DOE矩阵
                              → 运行后 → E1-E6淘汰规则
```

### 补丁2: 决策分层（对方案乙）

方案乙的P1-P12维度分类是按**参数功能**分的（时间驱动/数据窗口/排序算法...），方案丁的L1-L4是按**决策层次**分的（买什么/买不买/何时卖/数据精度）。两种分类正交——方案乙的P4(`option_width_min_threshold`)和方案丁的L1层完全一致，但方案丁首次将"对绩效的一阶/二阶影响"纳入了分类维度。

### 补丁3: 约束检查守卫（对方案丙）

方案丙的Optuna脚本在`define_search_space()`后直接生成参数JSON并写入——中间没有任何验证步骤。应插入方案丁的`validate_params()`守卫：

```python
# 在方案丙的 objective() 函数中
def objective(trial: optuna.Trial) -> float:
    search_params = define_search_space(trial)
    
    # ← 插入方案丁的约束检查
    if not validate_constraints(search_params):
        raise optuna.TrialPruned()  # 跳过非法trial
    
    ...
```

## 四、关键争议点深度分析

### 4.1 等权 vs 量加权排序的实现

| | 方案甲 | 方案乙 |
|---|--------|--------|
| **等权实现方式** | `ranking_mode = equal_weight_round_robin`——测试层注入，同产品/月份/方向候选轮转分配，不看成交量 | 调整`lots_min=1, min_option_width=1`代表等权，`lots_min=5, min_option_width=3`代表量加权 |
| **问题** | 需要代码拦截SortBucket的排序逻辑，方案甲未给出具体拦截代码 | 调整`lots_min`和`min_option_width`**只是改变成交量过滤阈值**，核心排序逻辑仍然是量加权。方案乙的"等权"只是降低了成交量过滤门槛 |

**评判**: 方案甲在这一维度上定义更准确（提出了真正的等权概念），但缺少实现代码。方案乙的处理过于粗糙（误将"降低成交量门槛"等同于"等权排序"）。应采纳方案甲的定义，结合方案丙的测试层注入代码框架来实现。

### 4.2 `top3_rows` / `target_top_n` 的层级归属

| | 方案甲 | 方案乙 |
|---|--------|--------|
| **归属** | C层（测试注入变量）——是`select_trading_targets(top_n=...)`的函数参数，不是`DEFAULT_PARAM_TABLE`的键 | P3维度（排序算法参数）——视为主参数池中的可调参数 |
| **理由** | 代码搜索结果显示它仅存在于函数调用层，不能与`option_width_min_threshold`同样视为配置表键 | （未明确给出理由） |

**评判**: 方案甲的判断**正确**。将函数级参数误写为配置参数会导致两个后果：
1. 批量参数矩阵中含不可比的组合（某些组合的top_n根本不会生效）
2. 如果将来有人将top_n加入生产配置表而忘记同步修改代码读取路径——参数变但逻辑不变

方案乙应立即修正：将`top3_rows`从核心参数池降级到C层标注，并在测试执行代码中添加对应的Harness Injection逻辑。

### 4.3 Greeks计算器的参数归属

方案甲明确指出：GreeksCalculator未被主策略代码链路主动调用。因此希腊参数（`price_change_pct`, `iv_change_pct`, `risk_free_rate`）应归入D层（条件能力池）——默认不启用，只有明确将Tick接入GreeksCalculator时才纳入测试池。

方案乙将`price_change_pct`和`iv_change_pct`列为P12维度（希腊计算频率），与其他核心参数并列。这在架构上是错误的——如果GreeksCalculator未被主链调用，调整`price_change_pct`不会对策略收益产生任何影响。

**评判**: 方案甲正确 ✅。方案乙应立即将P12整体移入D层，并在参数池执行计划中标注"条件启用：需先验证GreeksCalculator是否已接入主Tick处理链路"。

| | 方案甲 §11: 五层DOE | 方案丙: 三方案 |

| **第二层** | 筛选型正交池（PBD筛8→3） | RSM第一阶段(PBD-12) ✅ |
