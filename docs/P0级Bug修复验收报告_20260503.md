## 一、P0-1: _pending_on_stop_data崩溃不可靠 → WAL持久化

### 问题根因

`storage.py` L88的`_pending_on_stop_data`为纯内存list，进程crash（kill -9/OOM/断电）时数据全部丢失。优化方案第9.1节第7条明确要求"改为spill，用spill替代_pending_on_stop_data兜底角色"。

### 修复方案

在`_pending_on_stop_data`基础上增加**JSONL格式的WAL文件**持久化：

1. **启动恢复**: `__init__`中调用`_restore_spill_wal()`，从`_spill_wal.jsonl`恢复上次的spill数据
2. **溢出持久化**: `queue.Full`异常处理器在append到内存后同步写WAL
3. **drain持久化**: `_stop_async_writer`中`_drain_queue`同步写WAL
4. **回放清理**: `_try_replay_pending_data`在remaining为空时调用`_spill_wal_clear()`
5. **WAL文件**: 位于DuckDB同目录下`_spill_wal.jsonl`，每行一条JSON记录

### 修改文件与行号

| 文件 | 修改位置 | 修改内容 |
|------|---------|---------|
| storage.py | L89-91 | 新增_spill_wal_path、_spill_wal_lock、_restore_spill_wal()调用 |
| storage.py | L484-486 | queue.Full处理器增加_spill_wal_append(task) |
| storage.py | L511-512 | _try_replay_pending_data增加_spill_wal_clear() |
| storage.py | L304-305 | _drain_queue增加_spill_wal_append(task) |
| storage.py | L649-699 | 新增_restore_spill_wal()、_spill_wal_append()、_spill_wal_clear()三个方法 |

- **WAL格式**: JSONL（每行一条JSON），支持部分读取，损坏行跳过

## 二、P0-2: pause无多通道drain能力 → pause时flush+drain

### 问题根因

`strategy_core_service.py` L1811-1831的`pause()`方法仅设置`_is_paused=True`标志位，不执行任何drain操作。优化方案第9.4节第3条和第15.3节第6条明确要求"pause/drain支持Tick/K线/replay分级排空"。

### 修复方案

pause时按顺序执行两步drain：

1. **flush shard buffer**: 调用`tick_handler._flush_tick_buffer()`，将第一层分片缓冲中的数据刷入storage
2. **drain所有队列**: 调用`storage.drain_all_queues(timeout_per_queue=2.0)`，逐shard逐queue排空并立即写入DuckDB

新增`storage.drain_all_queues()`方法：
- 遍历16个tick shard queue并逐个drain
- drain kline_queue
- drain maintenance_queue
- 每个queue有独立超时(默认5秒，pause调用时传2秒)
- 返回Dict[str, int]包含per-queue drain计数

### 修改文件与行号

| 文件 | 修改位置 | 修改内容 |
|------|---------|---------|
| storage.py | L701-741 | 新增drain_all_queues()方法 |
| strategy_core_service.py | L1811-1851 | pause()增加flush shard buffer + drain queues逻辑 |

### 验收结果

| # | 验证项 | 结果 |
|---|--------|------|
| 1 | pause()中调用tick_handler._flush_tick_buffer | PASS |
| 2 | pause()中调用storage.drain_all_queues | PASS |
| 3 | drain_all_queues遍历16个shard+kline+maintenance | PASS |
| 4 | drain在try/except中包装，pause失败仍返回True | PASS |

### 彻底性保证

- **pause时数据不滞留**: shard buffer和所有队列均被排空
- **drain超时保护**: 每个queue有独立超时，不会无限等待
- **异常安全**: flush和drain均try/except包裹，失败只记录warning，pause仍成功
- **stop兼容**: stop时的drain逻辑(_stop_async_writer)不受影响

## 三、P0-3: diagnosis仅消费汇总backlog → 消费per-shard字段

### 问题根因

`diagnosis_service.py` L1988-2009的ResourceOwnership诊断段调用`storage.get_queue_stats()`后仅读取`current_queue_size`（汇总值），完全忽略per-shard字段。优化方案第9.4节第2条要求"统一stats接口输出per-shard/per-writer"，第11.2节第5条要求"每个Tick shard的queue depth、enqueue rate、drain rate、batch latency"。

### 修复方案

扩展ResourceOwnership诊断段的消费逻辑：

1. **读取分通道填充率**: tick_fill_rate、kline_fill_rate、maintenance_fill_rate
2. **读取spill/replay统计**: spill_count、replay_count、pending_on_stop_data_size
3. **输出per-shard backlog**: 遍历tick_shard_*_size键，输出每个shard的深度和填充率
4. **输出spill/replay/pending统计**: 当spill或pending非零时输出

### 修改文件与行号

| 文件 | 修改位置 | 修改内容 |
|------|---------|---------|
| diagnosis_service.py | L1990-2033 | ResourceOwnership段扩展：读取per-shard字段+输出shard backlog+输出spill/replay统计 |

### 验收结果

|---|--------|------|

| 3 | 输出per-shard backlog（迭代tick_shard_*_size） | PASS |
| 4 | 输出spill/replay/pending统计 | PASS |
| 5 | 原仅读current_queue_size的代码已扩展替换 | PASS |

### 彻底性保证

- **shard级瓶颈可见**: 每个shard的深度和填充率均输出到日志
- **spill/replay可追踪**: 溢出和回放次数均输出
- **pending可观测**: _pending_on_stop_data的大小可从日志中获取
- **兼容性**: 原有汇总输出仍保留(current_queue_size)，新增内容为补充而非替换
