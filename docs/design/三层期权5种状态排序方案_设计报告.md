## 0 批判式吸收摘要

### 0.1 对第十四章《顶级基金视角》的批判吸收

| 维度 | 合理内核（保留） | 缺陷（拒绝） | 本方案的处理 |
|------|----------------|-------------|-------------|
| 设计哲学 | 品种异质性需要差异化处理 | 与现有 Tier 体系错位、自适应衰减与 coverage 功能重叠 | 保留"品种差异化"方向，用数据驱动替代主观权重 |

## 1 总体架构

### 1.1 三层递进结构

```
┌─────────────────────────────────────────────────────────────────────┐
│                     第三层：跨品种全域综合排序                          │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │  全域 score = 品种内排名分(60%) + 品种间共振分(40%)             │  │
│  │  输出：全市场 Top N 期权 |> 动态风险预算分配                     │  │
│  │  [改进] 高频品种(IF/IH/IC/IM)使用5秒缓存减少计算延迟             │  │
│  │  [改进] 波动率突变检测：ATR 5min增长>50%时立即重算预算           │  │
│  └───────────────────────────────────────────────────────────────┘  │
│                                    ↑                                 │
├─────────────────────────────────────────────────────────────────────┤
│                     第二层：联动品种综合排序                           │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │  联动组 score = 组内各品种 Layer1 排序合并 + 组内共振否决        │  │
│  │  联动组定义：基于历史五态分布 JS 距离聚类 + 领域常识验证          │  │
│  │  [改进] JS聚类结果须与交易所/产业链常识一致，否则保留历史分组     │  │
│  │  输出：联动组内 Top N 期权                                      │  │
│  └───────────────────────────────────────────────────────────────┘  │
│                                    ↑                                 │
├─────────────────────────────────────────────────────────────────────┤
│                     第一层：单品种月份层                               │
│  ┌───────────────────────────────────────────────────────────────┐  │
│  │  品种内排序：当月+下月+3个季月 = 5个月份维度                      │  │
│  │  [改进] 前置硬过滤：Delta∈[0.15,0.85] 且 Gamma<0.08(近月)       │  │
│  │  [改进] 字典序排序：Tier > freshness > liquidity（去除greeks权重）│  │
│  │  输出：品种内 Top N 期权                                        │  │
│  └───────────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 设计原则

1. **全序优先**：三层排序之间保持全序关系（每个层级输出的排序结果可直接比较），不引入偏序，不破坏下游接口契约
2. **渐进可部署**：Layer 1 可单独上线运行，Layer 2/3 依次叠加，每层均有独立验证
3. **数据驱动**：所有权重参数均可通过第十七章研究基础设施校准，无"专家拍脑袋"参数
4. **与现有框架兼容**：不改变 Tier 分级体系的经济学含义，在其上层做叠加而非替换
5. **非对称衰减**：错误状态（WR/WF）衰减速度快于正确状态（CR/CF），但不改变 score 的正负号

### 1.3 与现有排序键的关系

现有排序键：`(tier升序, correct_up_pct降序, wilson降序)`

本方案在此之上扩展为：

```
Layer 1 排序键: (tier升序, freshness降序, liquidity降序)
               [改进] 去除greeks_quality_score维度，前置硬过滤替代
Layer 2 排序键: (tier升序, combined_score降序, resonance_flag降序)
Layer 3 排序键: (global_score降序, tier升序)
```

**关键设计决策**：扩展而不颠覆现有全序结构。下游代码（`TTypeService` → `StrategyBusinessLayer`）的 `取 Top N` 逻辑无需修改。

## 2 第一层：单品种月份层

### 2.1 目标

对**单一期货品种**下的所有期权，按**当月 + 下月 + 3 个季月（共 5 个月份维度）**进行 5 态排序，选出该品种内的最优期权。

### 2.2 输入与输出

```
输入：
  - future_internal_id → product 映射
  - _status_counts[fid][month][opt_type] → 五态统计
  - _option_info → 期权元数据（行权价、月份、Delta、Gamma）
  - month_mapping → 品种月份配置

输出：
  - product_top_list: [(option_id, tier, score, month, delta_bucket), ...]
  - 每个品种输出 Top N（N 由动态风险预算决定，默认 N=3）
```

### 2.3 前置硬过滤与字典序排序

**⚠️ [改进：源自附录B B.2.2、附录C C.2.3]**

原方案使用加权和评分公式包含 greeks_quality_score（权重0.05），经两方评判后重构为：

#### 2.3.1 前置硬过滤（Pre-requisite Hard Filter）

在进入排序前，对所有候选期权执行 Greeks 硬过滤：

```python
def _greeks_hard_filter(self, delta: float, gamma: float, month_type: str) -> bool:
    """
    Greeks 前置硬过滤。
    不满足条件的期权直接踢出候选列表，不参与排序。
    [改进] 替代原方案中权重仅0.05的greeks_quality_score
    """
    # Delta 适中：[0.15, 0.85]
    if abs(delta) < 0.15 or abs(delta) > 0.85:
        return False
    
    # 近月 Gamma 约束：Gamma >= 0.08 的近月期权状态翻转过快
    if month_type == 'front_month' and gamma >= 0.08:
        return False
    
    return True
```

**设计要点**：
- Delta ∈ [0.15, 0.85]：排除深度虚值（噪声主导）和深度实值（流动性差）
- Gamma < 0.08（近月）：近月高Gamma期权状态翻转速度远超衰减半衰期设计预期
- 硬过滤替代加权评分：消除"0.05权重给了做风控的安全幻觉"（附录B B.2.2）
- 过滤率监控：若过滤率 > 30%，触发告警（候选池质量可能下降）

#### 2.3.2 字典序排序（替代加权和）

过滤后的候选期权使用字典序排序：

```
排序键: (tier升序, freshness降序, liquidity降序)

- Tier 1 > Tier 2 > Tier 3（Tier 4 已过滤，绝对优先级）
- 同 Tier 内：freshness 更高者优先（状态更新、更可靠）
- 同 Tier + 同级 freshness：liquidity 更高者优先
```

| 排序维度 | 说明 | 数据来源 |
|---------|------|---------|
| tier | Tier 1/2/3 分级（Tier 4 已在前置过滤中跳过） | `determine_tier()` 现有逻辑 |
| freshness | 状态时效性：基于非对称衰减的正确有效状态占比 | 见 2.3.3 |
| liquidity | 成交量归一化（滚动 30 分钟均值） | tick volume 数据 |

**字典序排序的优势**：

| 维度 | 原方案（加权和） | 改进后（字典序） |
|------|----------------|-----------------|
| 物理含义 | 复合分数 "0.715" 无法解释 | 排序键每维含义清晰 |
| 权重校准 | 需要 Optuna 校准 4 个权重 | 无需权重校准 |
| 交易员理解 | "为什么 A > B？" 无法回答 | "因为 A 是 Tier 1, B 是 Tier 2" |
| 参数数量 | 4 个权重参数 | 0 个（硬过滤仅需 3 个阈值） |

#### 2.3.3 非对称时效性评分（freshness）

替代第十四章的 `decay_half_life` 和 `state_weights`，采用**非对称计数衰减**：

```python
def compute_freshness_score(counts: dict, last_update: dict, now: float) -> float:
    """
    非对称衰减：错误状态衰减更快，正确状态保留更久。
    输入: _status_counts[fid][month][opt_type] 的各状态计数和时间戳
    输出: [0, 1] 区间的时效性评分
    """
    config = {
        'CR': {'weight': 1.0, 'half_life': 600},   # 正确上涨，10分钟衰减
        'CF': {'weight': 1.0, 'half_life': 600},   # 正确下跌，10分钟衰减
        'WR': {'weight': 0.3, 'half_life': 120},   # 错误上涨，2分钟衰减
        'WF': {'weight': 0.3, 'half_life': 120},   # 错误下跌，2分钟衰减
        'Other': {'weight': 0.0, 'half_life': 60},  # 噪声，1分钟衰减
    }
    
    total_effective = 0
    for status, cfg in config.items():
        count = counts.get(status, 0)
        ts = last_update.get(status, now)
        age = now - ts
        decay_factor = 0.5 ** (age / cfg['half_life'])
        effective_count = count * decay_factor * cfg['weight']
        total_effective += effective_count
    
    return min(1.0, total_effective / 100)  # 归一化
```

**设计要点**：
- 不改变 `correct_up_pct`、`noise_ratio`、`Tier` 的定义（与 15.3.2 一致）
- 不做"双重衰减"（与 15.2.1 一致），衰减仅作用于 freshness_score，不影响 Tier 分级
- 可在配置文件中选择关闭衰减（`half_life` 设为极大值 86400s），具备完整回滚能力
- 与第十六章 16.2.5 的非对称衰减建议完全一致

### 2.4 月份维度处理

| 月份类型 | 默认权重 | 说明 |
|---------|---------|------|
| 当月 (front_month) | 0.35 | 流动性最好，主力月份 |
| 下月 (next_month) | 0.25 | 次优流动性 |
| 季月 1 (quarter_1) | 0.20 | 第一个季月 |
| 季月 2 (quarter_2) | 0.12 | 第二个季月 |
| 季月 3 (quarter_3) | 0.08 | 第三个季月，流动性最差 |

**设计要点**：
- 与现有 `MONTH_WEIGHTS_5 = (0.35, 0.25, 0.20, 0.12, 0.08)` 完全一致
- 月份权重在 `correct_up_pct` 计算中已体现，在 Layer 1 中不再重复施加
- 对不足 5 个有效月份的品种，剩余权重按比例重新分配（与现有 `_get_scoring_months` 一致）

### 2.5 Layer 1 排序算法

```python
def rank_single_product(
    self, 
    future_internal_id: int,
    top_n: int = 3
) -> List[Dict]:
    """
    单品种月份层排序。
    [改进] 前置硬过滤 + 字典序排序（替代原加权和评分）
    """
    # Step 1: 获取品种信息和月份配置
    product = self._future_id_to_product(future_internal_id)
    months = self._get_scoring_months(product)  # 复用现有方法
    
    # Step 2: 遍历所有月份、CALL/PUT、行权价，执行前置硬过滤
    candidates = []
    for month in months:
        for opt_type in ['CALL', 'PUT']:
            # 获取该月份+类型的五态统计
            counts = self._status_counts.get(future_internal_id, {}).get(month, {}).get(opt_type, {})
            if not counts:
                continue
            
            # 获取该期权的平均 Greeks
            avg_delta = counts.get('avg_delta', 0.5)
            avg_gamma = counts.get('avg_gamma', 0.01)
            month_type = self._classify_month_type(month)
            
            # [改进] 前置硬过滤：Delta/Gamma 不达标直接跳过
            if not self._greeks_hard_filter(avg_delta, avg_gamma, month_type):
                continue
            
            # 计算 Tier（复用现有逻辑）
            tier = self.determine_tier(counts, product=product)
            if tier >= 4:  # Tier 4 过滤
                continue
            
            # [改进] 字典序排序维度计算（无需加权和）
            freshness = compute_freshness_score(
                counts, 
                self._status_last_update.get(future_internal_id, {}).get(month, {})
            )
            
            liquidity = min(1.0, self._get_rolling_volume(
                future_internal_id, month, opt_type) / self.MIN_REF_VOLUME)
            
            candidates.append({
                'option_id': f"{future_internal_id}_{month}_{opt_type}",
                'tier': tier,
                'freshness': freshness,
                'liquidity': liquidity,
                'month': month,
                'product': product,
                'counts': counts,
            })
    
    # [改进] 字典序排序：Tier > freshness > liquidity
    candidates.sort(key=lambda x: (x['tier'], -x['freshness'], -x['liquidity']))
    
    return candidates[:top_n]
```

## 3 第二层：联动品种综合期权排序层

### 3.1 目标

在 Layer 1 单品种排序的基础上，将**具有联动关系的品种**（如同一板块、高度相关品种）合并排序，捕捉板块内部共振效应。

### 3.2 联动品种分组

#### 3.2.1 分组方法

使用**实测五态分布相似度**进行数据驱动分组，并引入领域常识约束：

```python
def compute_product_groups(feature_df: pd.DataFrame, n_groups: int = 6) -> Dict[str, int]:
    """
    输入: product_feature_daily 中的 pct_cr, pct_wr, pct_cf, pct_wf, pct_other
    方法: 计算产品间的 JS 距离 → 层次聚类 → 领域常识验证 → 输出分组
    输出: {product: group_id}
    
    [改进] JS聚类结果须与领域常识（交易所、产业链）交叉验证：
    - 若 JS 聚类将 cu（沪铜）与 IF（沪深300）归为一组 → 违反交易所常识 → 拒绝
    - 若 JS 聚类将 cu, al, zn 归为一组（有色金属）→ 与产业链一致 → 采纳
    """
```

### 3.3 联动共振否决机制

**⚠️ [改进：源自两方评判综合]**

原方案使用"共振增强加分"机制（0.20 × resonance），经评判后发现边际贡献率低。改进为**共振否决机制**：

```python
def rank_related_product_group(
    self,
    group_id: int,
    products: List[str],
    top_n: int = 5
) -> List[Dict]:
    """
    联动品种综合排序核心逻辑。
    [改进] 共振否决机制替代原共振增强加分
    """
    # Step 1: 获取组内各品种的 Layer 1 排序结果
    group_candidates = []
    for product in products:
        layer1_results = self._get_product_rank_cache(product)
        group_candidates.extend(layer1_results)
    
    if not group_candidates:
        return []
    
    # Step 2: 计算组内基准正确率
    total_cr = sum(c['counts'].get('CR', 0) for c in group_candidates)
    total_cf = sum(c['counts'].get('CF', 0) for c in group_candidates)
    total_all = sum(c['counts'].get('Total', 0) for c in group_candidates)
    group_accuracy = (total_cr + total_cf) / total_all if total_all > 0 else 0
    
    # Step 3: [改进] 共振否决：品种正确率低于组平均80%则否决
    filtered = []
    for c in group_candidates:
        product_accuracy = (
            c['counts'].get('CR', 0) + c['counts'].get('CF', 0)
        ) / max(c['counts'].get('Total', 1), 1)
        
        resonance = product_accuracy / group_accuracy if group_accuracy > 0 else 1.0
        
        # [改进] 否决机制替代加分机制
        if resonance < 0.8:  # 品种正确率低于组平均的80%
            continue  # 共振否决
        
        c['resonance'] = resonance
        c['group_accuracy'] = group_accuracy
        c['product_accuracy'] = product_accuracy
        filtered.append(c)
    
    # Step 4: 按 Tier + Layer 1 字典序排序（继承 Layer 1 的排序键）
    filtered.sort(key=lambda x: (x['tier'], -x['freshness'], -x['liquidity']))
    
    return filtered[:top_n]
```

**共振否决 vs 共振加分对比**：

| 维度 | 原方案（加分） | 改进后（否决） |
|------|--------------|---------------|
| 机制 | 从 0.30 增强到 0.33（无效增强） | 直接否决弱品种（保护资金） |
| 对弱品种影响 | 增强后仍然弱 | 直接排除 |
| 参数数量 | 需校准 resonance_weight=0.20 | 只需 1 个否决阈值（0.8） |
| 与字典序兼容性 | 需加权和 | 天然兼容（前置过滤） |

## 4 第三层：跨品种全域综合排序层

### 4.1 目标

在 Layer 1（品种内精排）和 Layer 2（联动组共振否决）的基础上，进行**全市场跨品种综合排序**，输出最终交易目标列表，并执行**动态风险预算分配**。

### 4.2 全域评分

```python
def rank_global(
    self,
    top_n: int = 10,
    max_per_group: int = 4
) -> List[Dict]:
    """
    跨品种全域综合排序。
    """
    # Step 1: 获取所有联动组的 Layer 2 排序结果
    global_candidates = []
    for group_id, group_info in self.product_groups.items():
        group_results = self._get_group_rank_cache(group_id)
        
        for candidate in group_results:
            # 全域评分 = 组内评分(60%) + 跨组基准(40%)
            # 跨组基准：用市场整体正确率校正组间不可比性
            candidate['global_score'] = (
                0.60 * self._compute_relative_rank(candidate, group_results) +
                0.40 * candidate['group_accuracy']
            )
            candidate['group_id'] = group_id
        
        global_candidates.extend(group_results)
    
    if not global_candidates:
        return []
    
    # Step 2: 按 global_score 排序
    global_candidates.sort(key=lambda x: -x['global_score'])
    
    # Step 3: [改进] 波动率突变检测（见4.3）
    self._check_volatility_spike()
    
    # Step 4: 动态风险预算分配
    allocated = self._dynamic_budget_allocation(
        global_candidates, 
        top_n=top_n,
        max_per_group=max_per_group
    )
    
    return allocated
```

### 4.3 动态风险预算分配与波动率突变检测

替代第十四章 14.4 的 `budget = {cat: 3 for cat in product_config}` 固定预算：

```python
def _dynamic_budget_allocation(
    self,
    candidates: List[Dict],
    top_n: int,
    max_per_group: int
) -> List[Dict]:
    """
    动态风险预算分配核心逻辑。
    
    原则：
    1. 高波动联动组分配更少标的数量（但更多保证金）
    2. 低波动联动组分配更多标的数量（分散化）
    3. 同一联动组内最多 max_per_group 个
    4. 总输出 top_n 个
    """
    # 获取各组波动率（基于30分钟ATR）
    group_vol = {
        gid: self._get_group_realized_vol(gid, window_minutes=30)
        for gid in self.product_groups.keys()
    }
    
    # 波动率反比分配
    dynamic_limits = {}
    for gid, vol in group_vol.items():
        if vol > 0.30:
            dynamic_limits[gid] = max(2, max_per_group - 1)
        elif vol > 0.20:
            dynamic_limits[gid] = max_per_group
        else:
            dynamic_limits[gid] = max_per_group + 1
    
    # 按分配限制筛选
    result = []
    group_count = defaultdict(int)
    for c in candidates:
        gid = c['group_id']
        if group_count[gid] < dynamic_limits.get(gid, max_per_group):
            result.append(c)
            group_count[gid] += 1
        if len(result) >= top_n:
            break
    
    return result
```

**⚠️ [改进：源自附录B B.2.5] 波动率突变检测**

```python
def _check_volatility_spike(self) -> None:
    """
    [改进] 波动率突变检测。
    当 ATR 在 5 分钟内增长超过 50% 时，立即触发风险预算重算。
    解决原方案 30 分钟 ATR 窗口在高波动事件时的滞后问题。
    """
    for gid in self.product_groups.keys():
        atr_5min_ago = self._get_group_realized_vol(gid, window_minutes=5, offset_minutes=5)
        atr_now = self._get_group_realized_vol(gid, window_minutes=5)
        
        if atr_5min_ago > 0 and atr_now / atr_5min_ago > 1.50:
            # ATR 5分钟增长 > 50%，触发重算
            logging.warning(
                f"VOLATILITY_SPIKE: Group {gid} ATR surged "
                f"{atr_5min_ago:.4f}→{atr_now:.4f} in 5min"
            )
            self._force_recalculate_budget(gid)
```

## 5 三层联动机制

### 5.1 数据流

```
每个 tick
    ↓
```

### 5.2 缓存策略

**⚠️ [改进：源自附录B B.2.4、附录C C.2.5]**

| 层级 | 缓存对象 | 生命周期 | 刷新条件 |
|------|---------|---------|---------|
| Layer 1 | `product_rank_cache[product]` | 30 秒 | 超时或 `_status_counts` 变化超过 10% |
| Layer 2 | `group_rank_cache[group_id]` | 15 秒 | 超时或组内任一品种缓存刷新 |
| Layer 3 | 不缓存（常规品种） | 实时 | 每次 `select_otm_targets_by_volume` 调用 |
| Layer 3 (高频) | [改进] 高频品种缓存 | **5 秒** | 仅对 IF/IH/IC/IM 启用 |

**高频品种 5 秒缓存说明**：
- 适用品种：IF、IH、IC、IM（金融期货，每秒调用数十次）
- 缓存内容：Layer 3 全域评分结果 + 动态预算分配
- 失效条件：5 秒超时 或 任一高频品种的 Layer 1 缓存刷新
- 背景：高频场景下 Layer 3 实时计算可能导致排序结果滞后于市场变化

### 5.3 降级策略

| 故障场景 | 降级到 | 降级延迟 |
|---------|--------|---------|
| Layer 3 计算超时 | 直接使用 Layer 2 结果 | < 100ms |
| Layer 2 计算超时 | 直接使用 Layer 1 结果 | < 100ms |
| Layer 1 计算超时 | 使用 Tier 排序（现有逻辑） | 无延迟 |
| 某联动组数据不足 | 该组退化为单个品种的 Layer 1 | 即时 |

```
└── 0.6 决策门禁（0.5 天）
    ├── 若测试 A 通过（p ≤ 0.05）且测试 B 通过（p ≤ 0.05）→ 完整三层方案
```

### 8.3 极端市场熔断机制

**⚠️ [改进：源自附录B B.2.6、附录C C.2.2]**

原方案缺少极端市场状态下的熔断机制，经两方评判后新增：

```python
class MarketCircuitBreaker:
    """
    极端市场熔断机制。
    
    触发条件（任一满足）：
    1. market_accuracy < 0.20（全市场趋势判断正确率极低）
    2. 全市场波动率 > 历史 99 百分位数，持续 5 分钟
    
    触发后动作：
    - 进入"防御模式"：仓位减半
    - 仅保留 Tier 1 品种交易
    - Tier 2/3 品种暂停交易
    """
    
    def __init__(self, history_volatility: np.ndarray):
        self.vol_99th = np.percentile(history_volatility, 99)
        self.defense_mode = False
        self.consecutive_minutes = 0
    
    def check(self, market_accuracy: float, current_volatility: float) -> str:
        """
        检查是否需要触发熔断。
        返回: 'normal' | 'defense' | 'shutdown'
        """
        trigger_vol = current_volatility > self.vol_99th
        trigger_acc = market_accuracy < 0.20
        
        if trigger_vol or trigger_acc:
            self.consecutive_minutes += 1
        else:
            self.consecutive_minutes = 0
            self.defense_mode = False
            return 'normal'
        
        if self.consecutive_minutes >= 5:
            self.defense_mode = True
            return 'defense'
        
        return 'normal'
    
    def get_position_multiplier(self) -> float:
        """防御模式下仓位减半"""
        return 0.5 if self.defense_mode else 1.0
    
    def get_tier_filter(self) -> int:
        """防御模式下仅保留 Tier 1"""
        return 1 if self.defense_mode else 0  # 0 = 不限制
```

**熔断机制参数**：

| 参数 | 值 | 说明 |
|------|----|------|
| market_accuracy 阈值 | < 0.20 | 全市场正确率极低 |
| 波动率阈值 | 历史 99 百分位数 | 极端波动事件 |
| 持续分钟数 | 5 分钟 | 防止瞬时误触发 |
| 防御模式仓位 | 50% | 减半 |
| 防御模式 Tier | 仅 Tier 1 | 最高置信度品种 |

### C.2 方案的关键短板与修复对照

以下列出原始评估中发现的关键短板及本方案对应的修复措施：

#### C.2.1 基础假设验证缺失（致命风险 → 已修复）

**原始问题**：`correct_up_pct 越高 → 次日收益越高` 这一核心假设尚未验证。

**修复措施**（见 6.3）：
- 升级为"前置门禁"（Pre-requisite Gate），所有编码开始前完成
- 测试 A 不通过 → 终止整个项目
- 测试 A 通过但测试 B 不通过 → 跳过 Layer 2 共振设计
- Phase 0 周期从 3 天调整为 2-3 周（含详细数据清洗子任务）

#### C.2.2 极端市场熔断机制缺失（高风险 → 已修复）

**原始问题**：缺少黑天鹅事件处理机制。

**修复措施**（见 8.3）：
- 新增 `MarketCircuitBreaker` 类
- 触发条件：`market_accuracy < 0.20` 或波动率 > 历史 99 百分位数，持续 5 分钟
- 防御模式：仓位减半（50%），仅保留 Tier 1 品种

#### C.2.3 Greeks 权重过低（中高风险 → 已修复）

**原始问题**：原方案 Greeks 质量评分权重仅 0.05。

**修复措施**（见 2.3.1）：
- 从加权评分改为**前置硬过滤**
- Delta ∉ [0.15, 0.85] → 直接排除
- 近月 Gamma ≥ 0.08 → 直接排除
- Greeks 不再参与评分维度，节省一个维度

#### C.2.4 联动组聚类稳定性（中风险 → 已修复）

**原始问题**：JS 距离聚类在不同市场状态下可能频繁重组。

**修复措施**（见 3.2.1）：
- JS 聚类结果必须通过**领域常识验证**（交易所约束 + 产业链约束）
- 仅当聚类结果与领域常识一致时才采纳
- 不一致时保留历史分组

#### C.2.5 Layer 3 高频性能瓶颈（中风险 → 已修复）

**原始问题**：Layer 3 不缓存，高频场景每秒调用数十次。

**修复措施**（见 5.2）：
- 为高频品种（IF/IH/IC/IM）设置 **5 秒缓存**
- 缓存失效条件：超时或 Layer 1 缓存刷新

#### C.2.6 波动率滞后（中风险 → 已修复）

**原始问题**：30 分钟 ATR 窗口在高波动事件时滞后。

**修复措施**（见 4.3）：
- 新增**波动率突变检测**：ATR 在 5 分钟内增长 > 50%
- 触发后立即重算风险预算
