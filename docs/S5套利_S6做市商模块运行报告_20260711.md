## 一、设计目标

S5 套利与 S6 做市商模块以**实盘模拟监控模式**运行，达成以下目标：

1. **信号生成与快照保存**：发出开仓/平仓信号并落盘为 JSON 快照
2. **不参与资金分配**：`route_capital` 与初始化阶段均设 `capital_allocation=0.0`
3. **不实际下单**：`handle_arbitrage_signal` 改为调用 `ArbitrageMonitor`，不再调用 `OrderService.send_order`
4. **为方向选择提供参考**：监控套利价差收敛与做市 inventory skew，辅助其他策略判断方向
5. **硬约束链路**：开仓/平仓信号经过 `_filter_by_s5_s6_monitor` 硬约束过滤，S5/S6方向冲突时**绝对拒绝**（不能违背）

**3层诊断设计**:
- **层1 (HFT信号源)**: 检查 `on_arbitrage_signal` 是否被回调 + TTL(60s) + confidence(≥0.6)
- **层2 (主捕获)**: 检查 `hft_consumed=True` 信号是否被主交易周期捕获 + confidence(≥0.8)
- **层3 (STANDBY阻断)**: 检查 STANDBY 状态是否正确阻断信号流转

### 2.2 S6 做市商监控模块 — `market_making_monitor.py`

**文件**: `ali2026v3_trading/strategy/monitor/market_making_monitor.py`

**类**: `MarketMakingMonitor`

**类常量**:
| 常量 | 值 | 说明 |
|------|-----|------|
| `PARTICIPATES_CAPITAL_ALLOCATION` | `False` | **不参与资金分配** |
| `SENDS_REAL_ORDERS` | `False` | **不发送真实订单** |

**核心方法**:
| 方法 | 签名 | 返回 | 说明 |
|------|------|------|------|
| `__init__` | `(instrument_id="", snapshot_dir=..., ...)` | - | 初始化做市参数 |
| `compute_theoretical_price` | `(mid_price)` | `float` | 理论价 = mid + skew × inventory_factor |
| `compute_bid_ask_quotes` | `(mid_price, volatility_bps=None)` | `(bid, ask, half_spread)` | 双边报价 |
| `update_inventory` | `(delta, fill_price=None)` | `float` | 更新模拟持仓 |
| `get_inventory_skew` | `()` | `float` | 持仓偏斜系数 ∈ [-1, 1] |
| `detect_adverse_selection` | `()` | `bool` | 毒单检测 |
| `compute_metrics` | `()` | `Dict` | 做市指标(命中率/价差捕获/周转率/毒单率/夏普) |
| `generate_simulated_signal` | `(tick_data=None)` | `MarketMakingSignal` | 生成模拟做市信号(不下单) |
| `save_snapshot` | `(signal=None)` | `str` | 保存JSON快照，断言文件存在 |
| `on_tick` | `(tick_data)` | `None` | tick回调 |
| `get_monitoring_report` | `()` | `Dict` | 完整监控报告 |

**理论价公式**:
```
theo_price = mid_price + inventory_skew × inventory_factor
inventory_factor = inventory_factor_bps × mid_price / 10000
```
- 多头持仓(skew>0) → 理论价下移（更愿意卖）
- 空头持仓(skew<0) → 理论价上移（更愿意买）

**双边报价公式**:
```
half_spread = max(base_spread, base_spread × (vol/vol_target) / competition)
bid = theo_price - half_spread
ask = theo_price + half_spread
```

## 五、S5/S6 硬约束链路 — 与7模块做相同处理

### 5.1 设计原理

S5套利和S6做市商模块的作用与订单流、希腊字母、夏普、三角评判、盈亏比、延迟、资金大小七模块做相同处理，但具有**硬约束**优先级：

| 模块类型 | 消费方式 | 优先级 | 冲突时行为 |
|---------|---------|--------|-----------|
| 订单流/希腊字母/夏普/三角评判/盈亏比/延迟/资金大小 | 决策评分维度 | 评分因子 | 降低决策分数 |
| 盈亏比(PLR) | 专用过滤 `_filter_by_plr` | 硦约束 | 信号被拒绝 |
| **S5套利** | 专用过滤 `_filter_by_s5_s6_monitor` | **硬约束** | **方向冲突绝对拒绝** |
| **S6做市商** | 专用过滤 `_filter_by_s5_s6_monitor` | **硬约束** | **方向冲突绝对拒绝** |

| # | 冗余项 | 代码证据 | 分析 | 建议 |
|---|--------|---------|------|------|
| 1 | **S6模块整体与现有策略框架冗余** | S6原`MARKET_MAKING`开仓理由在`_REASON_STRATEGY_MAP`中映射到`market_making` strategy_group，但S6已被改为监控模式（capital=0, 不下单） | S6的`_STRATEGY_HOLD_OVERRIDES['market_making']=15.0`、`_TWO_STAGE_STRATEGY_OVERRIDES['market_making']`、`_STRATEGY_HARD_STOP_OVERRIDES['market_making']`、`REASON_MULTIPLIERS['MARKET_MAKING']`全部是死参数——S6不会产生持仓，这些参数永远不会被消费 | 保留但不维护，标注为"监控模式遗留" |
| 2 | **MarketMakingMonitor与ArbitrageMonitor功能重叠** | 两者都有`on_tick()`、`generate_simulated_signal()`、`save_snapshot()`、`get_monitoring_report()`、单例模式 | 模板代码重复，但职责不同（套利vs做市），合并会降低可读性 | 保留分离，但可提取公共基类`_BaseMonitor` |
| 3 | **S6快照与S5快照格式不统一** | S5快照顶层是`signal`+`snapshot_time_iso`+`strategy_meta`；S6快照顶层是`module`+`version`+`participates_capital_allocation`+`sends_real_orders`+`signal` | 两套快照格式不一致，后续统一分析时需要两套解析逻辑 | 建议统一快照schema |
| 4 | **_simulate_quote_hit在监控模式下无意义** | L857-903：模拟报价命中并更新inventory/pnl | 监控模式下S6不下单，`_simulate_quote_hit`的"模拟成交"是**对模拟的模拟**——二次模拟。实际inventory由`update_inventory()`外部调用控制，`_simulate_quote_hit`产生的inventory变动与外部调用冲突 | 建议在监控模式下禁用`_simulate_quote_hit`，或将其改为纯统计（不修改inventory） |
| 5 | **S6硬约束与现有风控冗余** | `_filter_by_s5_s6_monitor`在`_FILTER_CHAIN`中位于`_filter_by_plr`之后、`_filter_by_decision_score`之前 | S6的preferred_side基于inventory skew，而inventory是**模拟**的（非真实持仓），基于模拟数据的硬约束可能产生误拒。与`_filter_by_decision_score`中的S5/S6维度评分功能重叠——同一数据既做硬约束又做评分因子 | 建议降级为评分因子（与7模块一致），而非硬约束。或至少将TTL从60秒缩短到10秒以减少过期信号误拒 |

| 维度 | 评分 | 说明 |
|------|------|------|
| 代码质量 | A | 线程安全、参数校验、防御性编程、文档完整 |
| 设计合理性 | B | 理论价/双边报价/inventory/毒单检测专业，但preferred_side阈值不一致、on_tick自动IO是性能隐患 |

### C.3 四个核心缺陷的深度分析

#### C.3.1 缺陷1：`_compute_sharpe`样本为累计PnL值（严重度：中）

**代码证据**（`market_making_monitor.py`）:

```python
# L880: 模拟命中时追加累计PnL
self._realized_pnl += captured_bps * signal.mid_price / 10000.0
self._pnl_samples.append(self._realized_pnl)  # ← 追加的是累计值，不是增量

# L940-943: 计算累计值的一阶差分
samples = list(self._pnl_samples)
prev = samples[0]
diffs = []
for v in samples[1:]:
    diffs.append(v - prev)  # ← 累计值的差分 = 增量变化
    prev = v
```

**分析**: `_pnl_samples`存储的是`self._realized_pnl`（累计已实现盈亏），而`_compute_sharpe`计算的是累计值的一阶差分。这意味着：
- 如果每笔交易都是+1.0利润，累计值序列为[1.0, 2.0, 3.0, 4.0...]，差分序列为[1.0, 1.0, 1.0...]——sharpe=0（标准差0）
- 如果累计值序列为[1.0, 3.0, 6.0, 10.0...]，差分序列为[1.0, 2.0, 3.0, 4.0...]——sharpe>0

**结论**: 这不是一个bug（差分计算在数学上是正确的），但sharpe的含义变成了"累计PnL加速率"而非"每笔交易PnL的夏普比率"。在监控模式下，这个指标的经济含义不清晰，但**不影响安全性**（仅用于监控报告）。

#### C.3.2 缺陷2：preferred_side阈值不一致（严重度：中）

**代码证据**:

| 位置 | 阈值 | 代码 |
|------|------|------|
| `generate_simulated_signal()` L580 | 0.2 | `if skew > 0.2: preferred_side = "sell"` |
| `get_preferred_side()` L967 | 0.15 | `if skew > 0.15: return 'sell'` |

**分析**: 当skew在[0.15, 0.20]区间时：
- `generate_simulated_signal()`生成的信号中`preferred_side = "neutral"`（不触发0.2）
- `get_preferred_side()`返回`"sell"`（触发0.15）
- 硬约束链路(`_filter_by_s5_s6_monitor`)调用`get_preferred_side()`，会拒绝SELL方向信号
- 但S6自己在快照中记录的是"neutral"

**结论**: 这是一个**真实的逻辑缺陷**。信号生成和硬约束消费使用不同的阈值，在skew∈[0.15, 0.20]区间会产生方向冲突。应统一阈值，建议统一为0.15。

#### C.3.3 缺陷3：`_simulate_quote_hit`逻辑循环（严重度：中）

**代码证据**:

```python
# generate_simulated_signal() L555-622: 生成信号（基于_last_mid_price等）
# L622: self._simulate_quote_hit(signal)  ← 生成信号后立即模拟命中

# _simulate_quote_hit() L862: market_price = self._last_last_price  ← 用当前价格判定命中
# L867: if market_price <= signal.bid_price:  ← 判定"市价穿越报价"
```

**分析**: `_simulate_quote_hit`在信号生成后立即执行，使用的是信号生成时的价格（`_last_last_price`）。该价格就是生成bid/ask的基础——用同一个价格判定"市价穿越报价"在逻辑上形成循环：
- 信号生成：`mid_price=3500` → `bid=3500.56`
- 模拟命中：`market_price=3500` → 判定`3500 <= 3500.56` → **命中**

**但是**，市场价3500就是在信号生成前已存在的价格，而非信号发出后市场的新方向。如果`_last_last_price`在下一个tick更新为3501，则`3501 >= 3500.56` → 命中（这次是合理的）。

**结论**: 这是一个**时序错误**。在`generate_simulated_signal`内部调用`_simulate_quote_hit`时，使用的价格是信号生成时的价格，而非信号发出后的价格。在监控模式下，这会导致第一次信号生成时出现"假命中"（总是命中bid或ask之一，取决于价格在当前tick的位置）。应改为在`on_tick`中调用`_simulate_quote_hit`（使用新tick的价格），而非在信号生成时。

#### C.3.4 缺陷4：on_tick每tick自动生成信号+快照（严重度：高）

**代码证据**（`on_tick()` L696-717）:

```python
def on_tick(self, tick_data):
    with self._lock:
        self._ingest_tick(tick_data)
    # 自动生成并保存信号（实盘模拟节奏）
    signal = self.generate_simulated_signal()  # ← 每tick一次
    self.save_snapshot(signal)                 # ← 每tick一次IO写入
```

**分析**:
- 典型期权品种tick频率：1-10 tick/秒
- 每次`generate_simulated_signal()`内部调用`compute_metrics()`（含`_compute_sharpe`的list转换）和`detect_adverse_selection()`（遍历fill_events）
- 每次`save_snapshot()`产生一次JSON文件写入（约1000字节）
- 1秒10 tick = 10次计算 + 10次IO = 10KB/秒/品种

**这在多品种同时运行时是不可接受的。** 如果10个品种同时运行，每秒100次IO写入，磁盘很快成为瓶颈。

**结论**: 这是S6模块**最严重的性能隐患**。应改为节流模式（如每5秒或每10个tick生成一次快照），或改为纯内存模式（仅在`get_monitoring_report()`时落盘）。

### C.4 功能冗余深度分析

#### C.4.1 冗余1：S6的`_STRATEGY_HOLD_OVERRIDES`等死参数

**代码证据**:

S6改为监控模式后（`capital_allocation=0.0`, `SENDS_REAL_ORDERS=False`），以下参数全部成为死代码：

| `_TWO_STAGE_STRATEGY_OVERRIDES['market_making']` | `strategy_ecosystem/_core.py` | 配置存在 | 两阶段override永不触发 |
| `_STRATEGY_HARD_STOP_OVERRIDES['market_making']` | `strategy_ecosystem/_core.py` | 配置存在 | 硬止损override永不触发 |
| `REASON_MULTIPLIERS['MARKET_MAKING']` | `infra/commission_utils.py` | tp_mult=1.0等 | 虽然修复了tp_mult=1.0，但S6不开仓，该乘子永不消费 |
| `_REASON_STRATEGY_MAP['MARKET_MAKING']` | `strategy_ecosystem/_core.py` | → `market_making` | 映射存在但永不触发 |

**结论**: 这些参数是**监控模式遗留**，不产生错误但增加代码维护负担。**评级**: 功能冗余C+（低风险但应清理）。

#### C.4.2 冗余2：`_simulate_quote_hit`在监控模式下是"二次模拟"

**代码证据**: 监控模式下S6不下单，`_simulate_quote_hit`模拟的"成交"是**对模拟信号的模拟成交**——二次模拟。

**分析**:
- 真实的做市商：发出报价 → 市场价穿越报价 → 成交 → 更新inventory
- S6监控模式：生成模拟信号 → 用信号生成时的价格判定"假命中" → 更新模拟inventory → 生成下一个模拟信号
- 但监控模式下，inventory应该由外部`update_inventory()`调用控制（反映真实持仓），而非由`_simulate_quote_hit`自动修改

**结论**: `_simulate_quote_hit`在监控模式下产生的inventory变动与外部调用冲突。建议在监控模式下禁用`_simulate_quote_hit`，或将其改为纯统计（不修改inventory）。**评级**: 功能冗余B（中等风险，会产生误导性inventory数据）。

#### C.4.3 冗余3：S5/S6快照格式不统一

**对比**:

| 维度 | S5快照格式 | S6快照格式 |
|------|-----------|-----------|
| 顶层字段 | `signal` + `snapshot_time_iso` + `monitor_stats` + `strategy_meta` | `module` + `version` + `participates_capital_allocation` + `sends_real_orders` + `signal` |
| 策略标识 | `strategy_meta.strategy_id` | `module` |
| 资金分配标识 | `strategy_meta.capital_allocation` | `participates_capital_allocation` |
| 下单标识 | `strategy_meta.simulated` | `sends_real_orders` |

**结论**: 两套快照格式使用不同的字段名表达相同语义。后续统一分析时需要两套解析逻辑。**评级**: 功能冗余B-（增加维护成本但不影响功能）。

#### C.4.4 冗余4：S6硬约束与评分因子双重消费

**代码证据**:
- `_filter_by_s5_s6_monitor`：硬约束，方向冲突绝对拒绝
- `_collect_decision_dimensions`：同时收集S6维度作为评分因子

**分析**: 同一数据（S6的preferred_side）被消费两次——一次是硬约束（方向冲突拒绝），一次是评分因子（影响decision_score）。如果硬约束已经拒绝，评分因子永远不会被消费；如果硬约束放行，评分因子才生效。这种双重消费在逻辑上是冗余的：**硬约束已经覆盖了评分因子能表达的全部信息**。

**结论**: 建议将S6从硬约束降级为评分因子（与订单流、希腊字母等7模块一致），删除`_filter_by_s5_s6_monitor`中的S6硬约束部分。如果保留硬约束，则不需要在`_collect_decision_dimensions`中重复收集S6维度。**评级**: 功能冗余B（逻辑重复，增加维护复杂度）。

#### C.4.5 冗余5：`MarketMakingMonitor`与`ArbitrageMonitor`的模板代码重复

**代码证据**: 两者共享以下模式：
- `on_tick()` 结构相同
- `generate_simulated_signal()` + `save_snapshot()` 模式相同
- `get_monitoring_report()` 结构相同
- 单例模式（`get_instance()`）相同
- 线程安全锁（`_lock`）相同

**结论**: 可提取`_BaseMonitor`基类减少重复代码。**评级**: 功能冗余C+（代码重复但不影响功能，提取基类是优化而非修复）。

### F.2 六项优化详情

#### F.2.1 P0: on_tick节流改造（最严重性能隐患）

**问题**: `on_tick()` 每个tick自动调用 `generate_simulated_signal()` + `save_snapshot()`，高频场景下每秒10次IO写入。

**修复**: 添加双维度节流机制
- `signal_interval_ticks=10`（tick计数维度，每10个tick生成一次）
- `snapshot_interval_sec=5.0`（时间维度，每5秒生成一次）
- 两个条件满足其一即触发（OR逻辑）
- 节流计数器在信号生成后重置

**代码变更** (`market_making_monitor.py`):
```python
# __init__新增参数
signal_interval_ticks: int = _DEFAULT_SIGNAL_INTERVAL_TICKS,  # 10
snapshot_interval_sec: float = _DEFAULT_SNAPSHOT_INTERVAL_SEC,  # 5.0

# on_tick节流判断
_ticks_ok = self._tick_count >= self.signal_interval_ticks
_time_ok = (_now - self._last_signal_time) >= self.snapshot_interval_sec
if not (_ticks_ok or _time_ok):
    return  # 节流：不生成信号
```

**效果**: IO频率从10次/秒降至~1次/5秒（降低98%），多品种同时运行不再有磁盘瓶颈。

#### F.2.2 P1: preferred_side阈值统一（方向冲突修复）

**问题**: `generate_simulated_signal()` 用0.2，`get_preferred_side()` 用0.15，skew∈[0.15,0.20]区间信号生成说"neutral"但硬约束说"sell"。

**修复**: 统一为0.15

**代码变更** (`market_making_monitor.py` `generate_simulated_signal()`):
```python

# 新: if skew > 0.15: preferred_side = "sell"  (与get_preferred_side一致)
```

**效果**: 信号生成与硬约束消费使用相同阈值，消除方向冲突误判。

#### F.2.3 P2: _simulate_quote_hit时序错误修复

**问题**: `_simulate_quote_hit()` 在 `generate_simulated_signal()` 内部调用，使用信号生成时的价格判定命中——用同一个价格生成报价又判定穿越，逻辑循环。

**修复**: 移至 `on_tick()` 中调用，使用NEW tick价格检查上一信号是否被穿越

**代码变更** (`market_making_monitor.py`):
```python
# generate_simulated_signal()中移除:
# self._simulate_quote_hit(signal)  ← 旧: 用生成信号时的价格

# on_tick()中新增:
with self._lock:
    self._ingest_tick(tick_data)  # 摄入NEW tick价格
    if self._last_signal is not None:
        self._simulate_quote_hit(self._last_signal)  ← 新: 用NEW tick价格检查上一信号
```

**效果**: 命中判定基于信号发出后的新价格，时序正确，消除假命中。

#### F.2.4 P2: _compute_sharpe样本污染修复

**问题**: `_pnl_samples` 存储累计 `_realized_pnl`（累计值），`_compute_sharpe` 计算累计值的一阶差分——sharpe含义为"累计PnL加速率"而非"每笔交易PnL夏普比率"。

**修复**: 存储增量PnL，直接计算mean/std

**代码变更** (`market_making_monitor.py`):
```python
# _simulate_quote_hit()中:
# 旧: self._pnl_samples.append(self._realized_pnl)  ← 累计值
# 新: incremental_pnl = captured_bps * signal.mid_price / 10000.0
#     self._pnl_samples.append(incremental_pnl)  ← 增量值

# _compute_sharpe()中:
# 旧: 先转list，再计算一阶差分diffs，再mean/std
# 新: 直接用samples计算mean/std（无需差分）
```

**效果**: sharpe经济含义清晰——"每笔交易PnL的夏普比率"而非"累计PnL加速率"。

#### F.2.5 P2: 单例改造为多合约registry

**问题**: `get_instance()` 全局单例，所有合约共享同一个inventory/skew/preferred_side，期权多合约场景无法区分。
