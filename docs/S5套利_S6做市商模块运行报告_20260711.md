## 一、设计目标

S5 套利与 S6 做市商模块以**实盘模拟监控模式**运行，达成以下目标：

1. **信号生成与快照保存**：发出开仓/平仓信号并落盘为 JSON 快照
2. **不参与资金分配**：`route_capital` 与初始化阶段均设 `capital_allocation=0.0`
3. **不实际下单**：`handle_arbitrage_signal` 改为调用 `ArbitrageMonitor`，不再调用 `OrderService.send_order`
4. **为方向选择提供参考**：监控套利价差收敛与做市 inventory skew，辅助其他策略判断方向
5. **硬约束链路**：开仓/平仓信号经过 `_filter_by_s5_s6_monitor` 硬约束过滤，S5/S6方向冲突时**绝对拒绝**（不能违背）

## 五、S5/S6 硬约束链路 — 与7模块做相同处理

### 5.1 设计原理

S5套利和S6做市商模块的作用与订单流、希腊字母、夏普、三角评判、盈亏比、延迟、资金大小七模块做相同处理，但具有**硬约束**优先级：

| 模块类型 | 消费方式 | 优先级 | 冲突时行为 |
|---------|---------|--------|-----------|
| 订单流/希腊字母/夏普/三角评判/盈亏比/延迟/资金大小 | 决策评分维度 | 评分因子 | 降低决策分数 |
| 盈亏比(PLR) | 专用过滤 `_filter_by_plr` | 硦约束 | 信号被拒绝 |
| **S5套利** | 专用过滤 `_filter_by_s5_s6_monitor` | **硬约束** | **方向冲突绝对拒绝** |
| **S6做市商** | 专用过滤 `_filter_by_s5_s6_monitor` | **硬约束** | **方向冲突绝对拒绝** |

### B.4 综合评价

| 维度 | 评分 | 说明 |
|------|------|------|

### C.3 四个核心缺陷的深度分析

#### C.3.1 缺陷1：`_compute_sharpe`样本为累计PnL值（严重度：中）

**代码证据**（`market_making_monitor.py`）:

```python
# L880: 模拟命中时追加累计PnL
self._realized_pnl += captured_bps * signal.mid_price / 10000.0
self._pnl_samples.append(self._realized_pnl)  # ← 追加的是累计值，不是增量

# L940-943: 计算累计值的一阶差分
samples = list(self._pnl_samples)
prev = samples[0]
diffs = []
for v in samples[1:]:
    diffs.append(v - prev)  # ← 累计值的差分 = 增量变化
    prev = v
```

**分析**: `_pnl_samples`存储的是`self._realized_pnl`（累计已实现盈亏），而`_compute_sharpe`计算的是累计值的一阶差分。这意味着：
- 如果每笔交易都是+1.0利润，累计值序列为[1.0, 2.0, 3.0, 4.0...]，差分序列为[1.0, 1.0, 1.0...]——sharpe=0（标准差0）
- 如果累计值序列为[1.0, 3.0, 6.0, 10.0...]，差分序列为[1.0, 2.0, 3.0, 4.0...]——sharpe>0

**结论**: 这不是一个bug（差分计算在数学上是正确的），但sharpe的含义变成了"累计PnL加速率"而非"每笔交易PnL的夏普比率"。在监控模式下，这个指标的经济含义不清晰，但**不影响安全性**（仅用于监控报告）。

#### C.3.2 缺陷2：preferred_side阈值不一致（严重度：中）

**代码证据**:

| 位置 | 阈值 | 代码 |
|------|------|------|
| `generate_simulated_signal()` L580 | 0.2 | `if skew > 0.2: preferred_side = "sell"` |
| `get_preferred_side()` L967 | 0.15 | `if skew > 0.15: return 'sell'` |

**分析**: 当skew在[0.15, 0.20]区间时：
- `generate_simulated_signal()`生成的信号中`preferred_side = "neutral"`（不触发0.2）
- `get_preferred_side()`返回`"sell"`（触发0.15）
- 硬约束链路(`_filter_by_s5_s6_monitor`)调用`get_preferred_side()`，会拒绝SELL方向信号
- 但S6自己在快照中记录的是"neutral"

**结论**: 这是一个**真实的逻辑缺陷**。信号生成和硬约束消费使用不同的阈值，在skew∈[0.15, 0.20]区间会产生方向冲突。应统一阈值，建议统一为0.15。

#### C.3.3 缺陷3：`_simulate_quote_hit`逻辑循环（严重度：中）

**代码证据**:

```python
# generate_simulated_signal() L555-622: 生成信号（基于_last_mid_price等）
# L622: self._simulate_quote_hit(signal)  ← 生成信号后立即模拟命中

# _simulate_quote_hit() L862: market_price = self._last_last_price  ← 用当前价格判定命中
# L867: if market_price <= signal.bid_price:  ← 判定"市价穿越报价"
```

**分析**: `_simulate_quote_hit`在信号生成后立即执行，使用的是信号生成时的价格（`_last_last_price`）。该价格就是生成bid/ask的基础——用同一个价格判定"市价穿越报价"在逻辑上形成循环：
- 信号生成：`mid_price=3500` → `bid=3500.56`
- 模拟命中：`market_price=3500` → 判定`3500 <= 3500.56` → **命中**

**但是**，市场价3500就是在信号生成前已存在的价格，而非信号发出后市场的新方向。如果`_last_last_price`在下一个tick更新为3501，则`3501 >= 3500.56` → 命中（这次是合理的）。

**结论**: 这是一个**时序错误**。在`generate_simulated_signal`内部调用`_simulate_quote_hit`时，使用的价格是信号生成时的价格，而非信号发出后的价格。在监控模式下，这会导致第一次信号生成时出现"假命中"（总是命中bid或ask之一，取决于价格在当前tick的位置）。应改为在`on_tick`中调用`_simulate_quote_hit`（使用新tick的价格），而非在信号生成时。

#### C.3.4 缺陷4：on_tick每tick自动生成信号+快照（严重度：高）

**代码证据**（`on_tick()` L696-717）:

```python
def on_tick(self, tick_data):
    with self._lock:
        self._ingest_tick(tick_data)
    # 自动生成并保存信号（实盘模拟节奏）
    signal = self.generate_simulated_signal()  # ← 每tick一次
    self.save_snapshot(signal)                 # ← 每tick一次IO写入
```

**分析**:
- 典型期权品种tick频率：1-10 tick/秒
- 每次`generate_simulated_signal()`内部调用`compute_metrics()`（含`_compute_sharpe`的list转换）和`detect_adverse_selection()`（遍历fill_events）
- 每次`save_snapshot()`产生一次JSON文件写入（约1000字节）
- 1秒10 tick = 10次计算 + 10次IO = 10KB/秒/品种

**这在多品种同时运行时是不可接受的。** 如果10个品种同时运行，每秒100次IO写入，磁盘很快成为瓶颈。

**结论**: 这是S6模块**最严重的性能隐患**。应改为节流模式（如每5秒或每10个tick生成一次快照），或改为纯内存模式（仅在`get_monitoring_report()`时落盘）。

#### C.4.2 冗余2：`_simulate_quote_hit`在监控模式下是"二次模拟"

**代码证据**: 监控模式下S6不下单，`_simulate_quote_hit`模拟的"成交"是**对模拟信号的模拟成交**——二次模拟。

**分析**:
- 真实的做市商：发出报价 → 市场价穿越报价 → 成交 → 更新inventory
- S6监控模式：生成模拟信号 → 用信号生成时的价格判定"假命中" → 更新模拟inventory → 生成下一个模拟信号
- 但监控模式下，inventory应该由外部`update_inventory()`调用控制（反映真实持仓），而非由`_simulate_quote_hit`自动修改

**结论**: `_simulate_quote_hit`在监控模式下产生的inventory变动与外部调用冲突。建议在监控模式下禁用`_simulate_quote_hit`，或将其改为纯统计（不修改inventory）。**评级**: 功能冗余B（中等风险，会产生误导性inventory数据）。

#### C.4.3 冗余3：S5/S6快照格式不统一

**对比**:

#### C.4.4 冗余4：S6硬约束与评分因子双重消费

**代码证据**:
- `_filter_by_s5_s6_monitor`：硬约束，方向冲突绝对拒绝
- `_collect_decision_dimensions`：同时收集S6维度作为评分因子

**分析**: 同一数据（S6的preferred_side）被消费两次——一次是硬约束（方向冲突拒绝），一次是评分因子（影响decision_score）。如果硬约束已经拒绝，评分因子永远不会被消费；如果硬约束放行，评分因子才生效。这种双重消费在逻辑上是冗余的：**硬约束已经覆盖了评分因子能表达的全部信息**。

**结论**: 建议将S6从硬约束降级为评分因子（与订单流、希腊字母等7模块一致），删除`_filter_by_s5_s6_monitor`中的S6硬约束部分。如果保留硬约束，则不需要在`_collect_decision_dimensions`中重复收集S6维度。**评级**: 功能冗余B（逻辑重复，增加维护复杂度）。

#### C.4.5 冗余5：`MarketMakingMonitor`与`ArbitrageMonitor`的模板代码重复

**代码证据**: 两者共享以下模式：
- `on_tick()` 结构相同
- `generate_simulated_signal()` + `save_snapshot()` 模式相同
- `get_monitoring_report()` 结构相同
- 单例模式（`get_instance()`）相同
- 线程安全锁（`_lock`）相同

**结论**: 可提取`_BaseMonitor`基类减少重复代码。**评级**: 功能冗余C+（代码重复但不影响功能，提取基类是优化而非修复）。

### F.2 六项优化详情

#### F.2.1 P0: on_tick节流改造（最严重性能隐患）

**问题**: `on_tick()` 每个tick自动调用 `generate_simulated_signal()` + `save_snapshot()`，高频场景下每秒10次IO写入。

**修复**: 添加双维度节流机制
- `signal_interval_ticks=10`（tick计数维度，每10个tick生成一次）
- `snapshot_interval_sec=5.0`（时间维度，每5秒生成一次）
- 两个条件满足其一即触发（OR逻辑）
- 节流计数器在信号生成后重置

**代码变更** (`market_making_monitor.py`):
```python
# __init__新增参数
signal_interval_ticks: int = _DEFAULT_SIGNAL_INTERVAL_TICKS,  # 10
snapshot_interval_sec: float = _DEFAULT_SNAPSHOT_INTERVAL_SEC,  # 5.0

# on_tick节流判断
_ticks_ok = self._tick_count >= self.signal_interval_ticks
_time_ok = (_now - self._last_signal_time) >= self.snapshot_interval_sec
if not (_ticks_ok or _time_ok):
    return  # 节流：不生成信号
```

**效果**: IO频率从10次/秒降至~1次/5秒（降低98%），多品种同时运行不再有磁盘瓶颈。

#### F.2.2 P1: preferred_side阈值统一（方向冲突修复）

**问题**: `generate_simulated_signal()` 用0.2，`get_preferred_side()` 用0.15，skew∈[0.15,0.20]区间信号生成说"neutral"但硬约束说"sell"。

#### G.3.1 完整 Avellaneda-Stoikov 框架

**Reservation Price（理论价）**:
```
r(t) = s(t) - q * γ * σ² * (T-t)
```
- `s(t)`: 中间价
- `q`: inventory - target_inventory（目标偏离）
- `γ`: 风险厌恶参数（默认 0.1）
- `σ`: 波动率（价格单位）
- `T-t`: 剩余时间窗口（默认 300 秒）

**最优半价差**:
```
δ* = γ*σ²*(T-t)/2 + ln(1+γ/κ)/γ
```
- 第一项: 风险分量（inventory 惩罚）
- 第二项: 毒性分量（订单到达率）
- `κ`: 订单到达率敏感度（默认 1.5）

**不对称报价**:
```
base_half = max(δ*, min_half)
skew = get_inventory_skew()
asym_factor = skew * 0.5
bid_half = base_half * (1.0 + asym_factor)  # 多仓时 bid 更宽
ask_half = base_half * (1.0 - asym_factor)  # 多仓时 ask 更窄
bid = reservation - bid_half
ask = reservation + ask_half
```
- 多仓时: bid 更宽（更不想买），ask 更窄（更想卖）→ 促进减仓
- 空仓时: bid 更窄（更想买），ask 更宽（更不想卖）→ 促进回补

### G补遗.1 背景与问题识别

附录G 完成的 V1 重写虽然引入了协整/A-S 框架等顶级基金数学模型，但**信息源仍受限**：

- 旧版 S5 `_ingest_tick` 仅提取 `last_price / bid_price / ask_price` 三个字段
- 旧版 S6 `_ingest_tick` 仅提取 `last_price / bid_price / ask_price` 三个字段
- 系统可用 tick 字段（`TICK_FIELD_NAMES`）实际包含：`instrument_id, exchange, last_price, volume, datetime, open_interest, turnover, bid_price1, ask_price1, bid_volume1, ask_volume1` 共 11 个字段
- 遗漏字段：`bid_volume1`（买盘挂单量）、`ask_volume1`（卖盘挂单量）、`volume`（累计成交量）、`turnover`（累计成交额）、`open_interest`（持仓量）、`direction`（真实成交方向）

**与顶级基金差距**：Citadel/Two Sigma/Jane Street 等顶级机构在做市/套利中显式建模订单簿不平衡、VWAP 偏离、持仓量趋势、真实成交流方向，而非仅依赖中间价。仅用三态价格信息会导致：
1. VPIN 毒单检测只能用价格推断方向（噪声大），无法使用 CTP 提供的真实 `direction` 字段
2. 订单簿不平衡（买盘/卖盘厚度差异）这一关键做市 skew 因子被忽略
3. VWAP 偏离无法计算，无法识别算法交易冲击
4. 持仓量趋势（资金流入/流出）这一重要的趋势确认信号缺失

### G补遗.2 多源数据采集架构

采用**位掩码（bitmask）+ 多入口方法**架构，不限于单一 tick 管道：

| 数据源 | bit 位 | S6 入口方法 | S5 入口方法 | 用途 |
|--------|--------|-------------|-------------|------|
| Tick（含全字段） | 0x01 | `on_tick(tick)` | `on_tick(tick)` / `on_price_update(..., tick_data=tick)` | 提取 bid_volume1/ask_volume1/volume/turnover/open_interest/direction |
| 多级订单簿 | 0x02 | `on_order_book(levels)` | `on_order_book(instrument_id, levels)` | 多级深度订单簿不平衡 |
| 单笔成交流 | 0x04 | `on_trade(trade)` | `on_trade(instrument_id, trade)` | 真实成交方向 + VPIN |
| 持仓量 | 0x08 | `on_open_interest(oi_data)` | `on_open_interest(instrument_id, oi, prev)` | OI 趋势（rising/falling/stable） |
| K线 | 0x10 | （预留） | （预留） | 多周期残差融合（后续扩展） |

### G补遗.3 S6 做市商多源升级明细

**1. `__init__` 新增状态字段**：
- `_last_bid_volume1`, `_last_ask_volume1` — 盘口挂单量
- `_order_book_imbalance` — `(bid_vol - ask_vol)/(bid_vol + ask_vol)` ∈ [-1, 1]
- `_multi_level_book: Deque[Dict]` — 多级订单簿深度（maxlen=10）
- `_last_open_interest`, `_prev_open_interest`, `_oi_trend` — 持仓量趋势
- `_vwap`, `_cum_turnover`, `_cum_volume` — VWAP 计算（利用 CTP 累计值）

**3. VPIN 真实方向优先**：`on_tick` 中 VPIN 计算时优先使用 `_real_trade_direction`，回退到 `_infer_trade_direction`（价格推断）：
```python
direction = self._real_trade_direction
if direction == 0:
    direction = self._infer_trade_direction(tick_data)
```

**4. 订单簿不平衡影响报价**：`compute_asymmetric_quotes` 中 OBI 作为额外 skew 因子：

ob_imbalance = self._order_book_imbalance

买盘厚（OBI>0）→ bid_half 更大（买盘挂单多，买价更安全可挂宽），ask_half 更小（促成交卸货）
