## 一、排查背景

策略在 InfiniTrader 平台上运行时出现以下问题：
1. 08:17:55 重启平台加载策略，08:18:23 显示"策略初始化完毕"，但停止在初始化状态不动
2. 08:27:10 用户点击执行按钮，状态变为"执行中"后又停止不动
3. 08:47:17 再次关闭平台重启后加载策略，策略启动但tick数据落盘失败（temp_ticks表不存在）
4. 历史K线加载被跳过（auto_load_history=False）

排查发现根因包括：平台回调参数值不合规、数据库连接池耗尽降级后缺表、历史K线加载参数读取不一致等。

### 3.3 运行时问题修复

#### 3.3.1 temp_ticks表不存在（P0）

**现象**: 连接池耗尽降级到内存DB后，`batch_insert_ticks` 中 `conn.register('temp_ticks', arrow_table)` 后 `INSERT INTO ticks_raw SELECT FROM temp_ticks` 报 `Catalog Error: Table with name temp_ticks does not exist!`

**根因**: 共享内存连接上并发操作，`conn.register` 注册的临时表可能被其他线程的 `unregister` 清除。

**修复**: `ds_data_writer.py:278` — 在 `conn.register('temp_ticks', arrow_table)` 前先 `conn.unregister('temp_ticks')` 确保清理旧注册。

**日志证据**: 08:48:54.825 起连续161次 `temp_ticks does not exist` 错误。

#### 3.3.2 auto_load_history=False导致历史K线不加载（P0）

**现象**: `[HKL] auto_load_history=False, skip`，但 `lifecycle_callbacks.py` 中已检查 `auto_load=True`。

**根因**: `_start_historical_kline_load`（strategy_historical.py:610）重复检查 `auto_load_history`，而 `self.params` 与 `lifecycle_callbacks.py` 中的 `params` 对象不同，导致读取结果不一致。

**修复**: 删除 `_start_historical_kline_load` 中的 `auto_load_history` 重复检查，因为调用方 `lifecycle_callbacks.on_start()` 已检查。

**日志证据**: 08:48:54.034 `[HKL] auto_load_history=False, skip`

#### 7.2.2 未真正修复的项（关键发现）

**A. F-08 未完全落地 — position_command_service.py 仍存在驼峰候选传入**

- **文件**: `position/position_command_service.py:72-73`
- **代码**:
  ```python
  inst_id = self._ps._get_platform_attr(trade, "instrument_id", "InstrumentID", default="")
  exch = self._ps._get_platform_attr(trade, "exchange", "ExchangeID", default="")
  ```
- **问题**: 虽然 `_get_platform_attr` 方法内部已委托到 `shared_utils.get_tick_field`（TICK_FIELD_NAMES中已无驼峰候选），但**调用方仍显式传入了不合规的驼峰候选名 `InstrumentID` 和 `ExchangeID`**。这与修复目标"删除驼峰候选"相违背。一旦 `_get_platform_attr` 内部异常回退到原始逻辑（第341行`for attr in attr_names`），这些驼峰名仍会被尝试访问，构成潜在合规风险。
- **规范要求**: 对平台对象的属性访问应直接使用合规的snake_case名，不应将驼峰名作为候选传入。

**B. F-30 未完全落地 — strategy_historical.py 仍存在 `timestamp` 回退**

- **文件**: `strategy/strategy_historical.py:132`
- **代码**:
  ```python
  ts = storage._to_timestamp(getattr(kline, 'datetime', getattr(kline, 'timestamp', time.time())))
  ```
- **问题**: `kline.timestamp` 回退仍然存在。根据平台规范（`kline.py:6-83`），KLineData只有 `datetime` 属性，没有 `timestamp` 属性。此处 `getattr(kline, 'timestamp', ...)` 是对不存在属性的回退访问，属于不合规行为。
- **规范要求**: 应直接使用 `kline.datetime`，删除 `timestamp` 回退。

**C. TICK_FIELD_NAMES 中 `ts` 候选的合规性说明**

- **文件**: `infra/shared_utils.py:1199` / `strategy/tick_processing_service.py:126`
- **代码**: `'datetime': ['datetime', 'update_time', 'ts']`
- **说明**: `ts` 不是 PythonGO TickData 的规范属性（TickData只有 `datetime` 和 `update_time`）。若 `ts` 是为了兼容内部存储或回测数据的设计，应在文档中明确说明；若用于访问平台 TickData，则属于不合规候选。
