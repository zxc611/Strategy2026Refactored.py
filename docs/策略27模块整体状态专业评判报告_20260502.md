## 七、config_service.py 重构路线图（专项）

### 现状

| 指标 | 值 |
|------|-----|
| 行数 | ~2036 |
| 核心问题 | `DEFAULT_PARAM_TABLE` 约 100+ 参数硬编码在 Python dict 中，修改参数 = 修改代码 |
| 副作用 | 无参数校验层、无版本管理、无环境切换（dev/test/prod 共享同一代码） |
| 上游消费 | strategy_core_service, params_service, t_type_service, order_service, position_service, risk_service 等 10+ 模块 |

### 分三步重构

#### 阶段 1：参数外化（不改变面，风险最低）

**目标**: 将 `DEFAULT_PARAM_TABLE` 从代码中移至外部 JSON 文件，代码仅读取。

**操作**:
1. 将 `config_service.py` 中 `DEFAULT_PARAM_TABLE` dict 的全部内容导出为 `config/params_default.json`
2. 在 `config_service.py` 中 `load_default_params()` 改为从 JSON 文件读取 + 内存缓存
3. 废弃模块内的 `DEFAULT_PARAM_TABLE` 字面量（保留 30 天作为回退，然后删除）

**JSON 结构示例**:
```json
{
  "top3_rows": {
    "default": 10,
    "type": "int",
    "min": 3,
    "max": 50,
    "layer": "L3",
    "status": "待确认"
  },
  "close_max_hold_days": {
    "default": 5,
    "type": "int",
    "min": 1,
    "max": 30,
    "layer": "L3",
    "status": "待确认"
  }
}
```

**验收**: 所有现有功能不变，启动时从 JSON 加载参数，`DEFAULT_PARAM_TABLE` 仍可工作作为回退。

#### 阶段 2：参数校验层（生产安全）

**目标**: 引入 `validate_params()` 函数，在策略启动时校验所有参数。

**操作**:
1. 基于 JSON schema 中 `type`/`min`/`max` 字段生成校验规则
2. `validate_params()` 在策略 `__init__` 阶段调用
3. 非法参数 → 阻断启动 + 错误日志（绝不静默）
4. 添加 `params_env.json` 支持环境切换（dev/test/prod 三套参数值）

**验收**: 提交一个 `top3_rows: -1` 的 JSON，策略拒绝启动。

#### 阶段 3：模块拆解（可维护性）

**目标**: 2036 行打散为 3 个子模块。

**操作**:
1. `config_loader.py`（~200行）：JSON 加载 + 缓存 + 环境切换
2. `config_validator.py`（~200行）：参数校验 + 约束检查
3. `config_service.py`（~600行）：重构后仅保留 `DEFAULT_PARAM_TABLE`（已外化后是加载/查询逻辑）+ 交易所映射 + 兼容层
4. 删除外化后变成冗余的旧 dict 字面量

**验收**: `import ali2026v3_trading.config_service` 的现有消费者无需修改 import 路径（`config_service.py` 作为 Facade 重导出子模块的符号）。

### 风险矩阵

| 阶段 | 破坏性变更 | 需要并行运行的旧代码 | 回退难度 |
|------|-----------|---------------------|---------|
| 阶段1 | 无 | 是（30天并行期） | 一行 rollback |
| 阶段2 | 启动时新增检查 | 否 | 删除 validate 调用即可 |
| 阶段3 | 可能（如果消费者 import 了内部名称） | 否 | git revert |
