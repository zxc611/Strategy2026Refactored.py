## 二、核心发现

### 2.1 生产策略只需1个配置文件

生产策略运行时的配置入口链路:

```
params.yaml (2,170行, 唯一配置源)
  → config_loader.load_default_params_from_json()
    → config_params.get_cached_params() (带TTL缓存)
      → ParamsService (参数校验/类型转换/环境覆盖)
```

**结论**: 生产策略确实只需 `params.yaml` 一个配置文件。其余18个Python模块中:
- 12个是**配置加载/校验/缓存基础设施**（生产必需）
- 3个是**参数管理/迁移/灰度**（应归入参数池）
- 1个是**UI控制面板**（应归入测试系统）
- 1个是**历史Facade残留**（应拆分清理）
- 1个是**包标识**（保留）

### 2.2 config/ 与 param_pool/ 的功能重叠

| 重叠维度 | config/ | param_pool/ | 重叠程度 | 关系性质 |
|----------|---------|-------------|----------|----------|
| 参数默认值 | get_cached_params() | PARAM_DEFAULTS/PARAM_DEFAULTS_HFT | **中** | 运行时 vs 回测场景分化 |
| 属性矩阵 | AttributeMatrixService(消费者) | parameter_attribute_matrix.yaml(生产者) | **低** | 消费-生产 |
| PLR/瀑布阈值 | load_plr_thresholds()(消费者) | plr_thresholds.yaml(生产者) | **低** | 消费-生产(跨目录穿透) |
| TVF参数 | tvf_param_loader(消费者) | tvf_params.yaml(生产者) | **低** | 消费-生产 |
| 参数版本管理 | config_version_tracker | 无 | **无** | 独占 |
| 回测/优化/验证 | 无 | backtest/optimization/validation | **无** | 独占 |

## 三、优化方案

### 方案上: 归属校正 — 将18个模块中的4个归位 (减2,519+1,558=4,077行)

**操作**:
1. `_params_canary_env.py`(447行) + `_params_migration.py`(1,016行) + `config_version_tracker.py`(95行) → 移入 `param_pool/` 根目录
   - 理由: 灰度发布/参数迁移/版本追踪是参数池管理功能，非配置加载基础设施
   - param_pool/已有`_params_canary_env`/`_params_migration`的re-export，移入后消除跨包导入
2. `ui_service.py`(2,519行) → 移入 `tests/` 或新建 `ui/` 子包
   - 理由: UI控制面板与配置中心无关，生产无头运行不需要
3. `config_service.py` 中删除与子模块重复的类定义，仅保留re-export
   - 预估可削减 ~800行 重复代码

**效果**: config/ 从 17,131行 → ~12,454行 (减27%)

### 方案中: config_service.py 瘦身 + 参数池统一入口 (减~1,200行)

**操作**:
1. `config_service.py`(1,835行) 拆分:
   - 保留: `ConfigService`类 + `get_config_service()` + re-export (~400行)
   - 删除: 所有与子模块重复的类定义(FlushHandler/AuditLogger/TradingConfig等) (~1,200行)
   - 调用方通过子模块直接导入，config_service.py仅做兼容re-export
2. param_pool/ 统一参数默认值入口:
   - `_param_defaults.py`(946行) + `_leaf_param_defaults.py`(704行) + `_param_grids.py`(583行) → 合并为 `param_defaults.py`(2,233行，超限需再拆)
   - 或保持3文件但建立 `param_pool/defaults.py` 门面模块
3. ~~消除config/→param_pool/的跨目录YAML读取~~ ✅ 已完成:
   - `config_loader.py` 中 `load_plr_thresholds()` 改为调用 `param_pool.load_plr_thresholds_data()` API
   - `param_pool/__init__.py` 新增 `load_plr_thresholds_data()` 公共API

**效果**: ~~config_service.py 从1,835行→~400行; 消除跨目录边界穿透~~ ✅ 已完成: config_service.py 从1,835行→1,153行; 跨目录边界穿透已修复

### 方案下: 配置中心重构为三层架构 (结构性优化)

**目标架构**:

```
config/                          # 配置中心 (目标: ~8,000行)
├── __init__.py                  # 包标识
├── _constants.py                # 零依赖常量
├── core/                        # 核心层: 运行时配置加载
│   ├── config_loader.py         # 配置加载/热重载
│   ├── config_params.py         # 参数表管理/缓存
│   ├── config_exchange.py       # 交易所映射
│   └── config_logging.py        # 日志初始化
├── params/                      # 参数层: 参数服务
│   ├── params_service.py        # ParamsService Facade
│   ├── _params_core.py          # ParamsService核心
│   ├── _params_instrument_cache.py  # 合约缓存
│   └── _params_attribute_matrix.py  # 属性矩阵校验
├── dataclasses.py               # 配置数据类
├── state_param.py               # 状态参数管理器
├── final_three_layer_config.py  # 三层排序常量
└── tvf_param_loader.py          # TVF参数加载
```

**操作**:
1. config_service.py 彻底退化为纯re-export(~100行)，所有逻辑已在子模块
2. 新建 `config/core/` 和 `config/params/` 子包，按职责分层
3. ui_service.py 移出; canary/migration/version_tracker 移入param_pool
4. param_pool/ 建立 `ParamPoolService` 统一入口，替代config/中的参数管理功能
5. 消除所有跨目录YAML直接读取，改为API调用

**效果**: config/ 从17,131行→~8,000行; 清晰三层架构; 消除边界穿透
