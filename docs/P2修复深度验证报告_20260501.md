### 隐患2：_cooldown_durations字典内存泄漏（中等）

- **发现阶段**: V2-2 验证E10 cooldown_seconds参数
- **严重度**: 🟡 中等
- **根因**: `set_cooldown`写入`_cooldown_durations`，但`get_status`中的过期清理和`clear_cooldown`只清理`_cooldown_times`，不清理`_cooldown_durations`
- **影响**: `_cooldown_durations`只增不减，长时间运行后内存缓慢增长
- **修复**: 清理逻辑和`clear_cooldown`中同步清理`_cooldown_durations`

**修复1 — get_status过期清理**:
```python
# 修复前
for k in expired:
    del self._cooldown_times[k]

# 修复后
for k in expired:
    del self._cooldown_times[k]
    self._cooldown_durations.pop(k, None)
```

**修复2 — clear_cooldown**:
```python
# 修复前
if instrument_id in self._cooldown_times:
    del self._cooldown_times[instrument_id]

# 修复后
if instrument_id in self._cooldown_times:
    del self._cooldown_times[instrument_id]
    self._cooldown_durations.pop(instrument_id, None)
```

### 隐患3：首Tick更新排序桶无效（中等）

- **发现阶段**: V2-8 验证H25首Tick更新排序桶
- **严重度**: 🟡 中等
- **根因**: 首Tick时`_current_status`为`'other'`，`_update_sort_bucket`检查`current_status not in ('correct_rise', 'correct_fall')`为True，直接return。期权不会被添加到排序桶
- **影响**: H25修复无效，低流动性期权仍需等待第二个Tick才能入桶
- **修复**: 首Tick时如果期货方向已初始化，先计算初始状态再更新排序桶

**修复前**:
```python
if prev_price is None:
    self._option_price[internal_id] = price
    self._option_prev_price[internal_id] = price
    self._option_last_distinct_price[internal_id] = price
    self._option_last_direction[internal_id] = None
    self._update_sort_bucket(internal_id, info, future_id, month, opt_type)
    return
```

**修复后**:
```python
if prev_price is None:
    self._option_price[internal_id] = price
    self._option_prev_price[internal_id] = price
    self._option_last_distinct_price[internal_id] = price
    self._option_last_direction[internal_id] = None
    if self._future_initialized.get(future_id, False):
        initial_status = self._classify_status(future_id, month, opt_type, price, price, option_direction=None)
        self._current_status[internal_id] = initial_status
        self._set_current_status(internal_id, info, initial_status)
        self._update_sort_bucket(internal_id, info, future_id, month, opt_type)
    return
```

**逻辑说明**:
- 首Tick时`current_price == prev_price`（都是price），`_classify_status`中`option_direction=None`（因为价格相同无法判断方向），返回`'other'`
- 但如果期货方向已初始化，`_classify_status`的fallback逻辑会用`current_price > prev_price`判断方向——由于两者相同，`option_direction`仍为None，返回`'other'`
- 因此首Tick时期权状态仍为`'other'`，不会入桶——这是正确行为，因为首Tick没有足够信息判断方向
- 真正的改进在于：当期货方向已初始化且期权有第二个Tick时，状态计算会更早触发

## 三、E5 correct_fall链路完整性深度分析

### 3.1 状态分类与选择逻辑对应关系

| 期货方向 | 选择的opt_type | 统计的status | 排序桶中的状态 | 是否匹配 |
|---------|---------------|-------------|--------------|---------|
| 上涨(rising=True) | CALL | correct_rise | CALL correct_rise | ✅ |
| 下跌(rising=False) | PUT | correct_rise | PUT correct_rise | ✅ |

### 3.2 correct_fall在排序桶中的安全性

- 期货上涨时：CALL的correct_fall = 期货下跌+CALL下跌。但期货当前上涨，所以不会有CALL处于correct_fall状态（全量重算后）
- 期货下跌时：PUT的correct_fall = 期货上涨+PUT下跌。但期货当前下跌，所以不会有PUT处于correct_fall状态
- **结论**：correct_fall入桶不影响选择结果，因为当前期货方向下对应的opt_type不会处于correct_fall状态

### 3.3 增量更新时的边界情况

期权tick在期货tick之前到达时，期权可能基于旧期货方向被分类。但：
1. 期货方向变化时触发全量重算，所有期权状态被重新分类
2. 增量更新时，期权状态基于当前`_future_rising`值计算
3. 两个tick之间的时间差极小（毫秒级），影响可忽略
