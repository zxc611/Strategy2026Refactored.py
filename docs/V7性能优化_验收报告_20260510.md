## 二、逐项核查

### 2.1 卡点6.1/7.1: _status_counts无锁读取 → 快照接口

**问题**: StateParamManager.update_state_from_width_cache()通过`getattr(self._width_cache_ref, '_status_counts', {})`直接读取width_cache的内部字典，遍历期间若width_cache写线程修改字典大小，触发`RuntimeError: dictionary changed size during iteration`。

**方案**:
1. `WidthStrengthCache.get_status_counts_snapshot()`: 在`self._lock`内深拷贝`_status_counts`，返回独立快照
2. `StateParamManager.update_state_from_width_cache()`: 优先调用`get_status_counts_snapshot()`，降级兼容无快照接口的旧版width_cache

**修改文件**:
- `width_cache.py`: +10行（第966-975行）
- `state_param_manager.py`: +4行（第120-124行）

**验证**: 快照在锁内生成，调用方在快照上遍历无并发风险 ✅

### 2.2 卡点3.1: detect_box()重复list()转换

**问题**: detect_box()第289-291行做了3次`list(deque)`转换，但_compute_adx_simplified和_find_support_resistance又对传入的deque再次`list()`转换，共6次O(N)转换。

**方案**: detect_box()一次性转为list，直接传list给两个静态方法，方法内部直接用`[-i]`索引。

**修改文件**:
- `box_detector.py`: _compute_adx_simplified去掉5处`list()`包装，_find_support_resistance去掉2处`list()`包装

**验证**: 输入list支持`[-i]`索引和`sorted()`，功能完全等价 ✅

### 2.3 卡点3.2: classify_extreme_state嵌套锁

**问题**: classify_extreme_state持self._lock后调用check_iv_filter/check_order_flow_exhaustion，二者也持self._lock。RLock不致死锁但有重入开销（每次acquire需检查owner线程）。

**方案**: 将check_iv_filter的IV百分位判断和check_order_flow_exhaustion的流量衰竭判断逻辑直接内联到classify_extreme_state中。

**修改文件**:
- `box_detector.py`: 第407-416行，内联IV过滤+订单流衰竭逻辑

**验证**: 逻辑完全等价，统计计数器更新位置一致 ✅

### 2.4 卡点3.3: _compute_iv_percentile O(N) → O(logN)

**问题**: 每次调用遍历全部IV历史（maxlen=1000）计算小于current_iv的个数，O(N)线性扫描。

**方案**:
1. 新增`_iv_sorted: List[float]`，始终有序
2. `update_iv()`中增量维护：deque满时移除最旧元素（bisect_left定位+pop），insort插入新IV
3. `_compute_iv_percentile()`使用bisect_left二分查找，O(logN)

**修改文件**:
- `box_detector.py`: +import bisect_left/insort, +_iv_sorted字段, update_iv增量维护, _compute_iv_percentile改用二分

**性能对比**:
| 场景 | 优化前 | 优化后 |
|------|--------|--------|
| IV百分位计算(1000条) | O(1000)遍历 | O(10)二分查找 |
| update_iv(正常) | O(1) append | O(logN) insort |
| update_iv(满deque) | O(1) append | O(logN) 移除+insort |

**验证**: bisect_left返回的count_below等价于`sum(1 for iv in iv_list if iv < current_iv)` ✅

### 2.5 卡点2.3: check_absolute_ev_bottomline O(N) → O(1)

**问题**: 每次调用对3个策略各遍历全部交易记录求EV，O(N)×3。

**方案**: StrategySlot新增`_closed_pnl_sum`和`_closed_count`，在`record_strategy_pnl()`平仓时增量更新。check_absolute_ev_bottomline直接用`slot._closed_pnl_sum / slot._closed_count`计算EV，O(1)。

**修改文件**:
- `strategy_ecosystem.py`: StrategySlot+2字段, record_strategy_pnl+2行增量更新, check_absolute_ev_bottomline改用增量EV, get_stats改用slot.expected_value

**验证**: 增量值与遍历计算结果数学等价（sum/count = 累加sum/累加count）✅

### 2.6 卡点2.4/4.2: get_health_status EV缓存

**问题**: get_health_status每次调用都执行check_absolute_ev_bottomline，高频调用时浪费CPU。

**方案**: 新增`_ev_cache`/`_ev_cache_ts`/`_ev_cache_ttl=5.0`，5秒内复用上次EV检查结果。

**修改文件**:
- `strategy_ecosystem.py`: +3字段, get_health_status增加TTL缓存逻辑

**验证**: TTL内返回缓存值，超TTL重新计算；EV变化在5秒内可能延迟反映（可接受）✅

### 2.7 卡点5.3: tick热路径锁分离

**问题**: `_tick_dispatch_layer`第231-246行每个tick持`self._lock`更新统计计数器，与主业务逻辑共享同一把大锁，造成不必要的竞争。

**方案**: 新增`_tick_stats_lock`独立小锁，仅保护tick统计相关字段，与`self._lock`（主业务锁）分离。

**修改文件**:
- `strategy_tick_handler.py`: +1字段_tick_stats_lock, 统计更新改用_tick_stats_lock

**验证**: tick统计字段（total_ticks/tick_count/tick_by_type等）仅在此处写，用独立锁保护正确 ✅

### 2.8 卡点6.3: width_cache全局锁标注

**问题**: width_cache使用单个RLock保护所有状态，高并发下成为瓶颈。但分片锁改造需重构整个类的锁粒度，改动面太大。

**方案**: 加注释标注V8专项重构，本期不改。

**修改文件**:
- `width_cache.py`: self._lock行加注释
