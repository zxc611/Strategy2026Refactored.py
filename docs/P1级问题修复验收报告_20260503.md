| 项目 | 内容 |
|------|------|

### 修改文件与行号

| 文件 | 修改位置 | 修改内容 |

| storage.py | L222 | `_get_shard_index_for_task`：删除depth/option_snapshot/signal的死代码分片逻辑 |

### 修复原理

**变更前路由**:
```
depth/snapshot → TickShard队列(16个) → TickWriter(6个) → DuckDB(持_tick写锁)
```

**变更后路由**:
```
depth/snapshot → Maintenance队列(1个) → MaintenanceWriter(1个) → DuckDB(持_maintenance写锁)
```

**核心收益**:
1. depth/snapshot不再与tick竞争同一个shard队列和TickWriter线程
2. 使用独立的`_db_maintenance_write_lock`，不与tick写锁互斥
3. tick热路径的延迟不再受depth直接SQL拼接影响

## 三、隐患审计

### H1: Maintenance队列容量(WARN)

- 当前容量50000，约62.5秒缓冲(按800条/秒估算)
- 正常负载下足够，DuckDB卡顿时可能不足
- 队列满后走spill逻辑(WAL持久化)，不会丢失
- **建议**: 高负载场景下可考虑增至100000

### H2: MaintenanceWriter单线程瓶颈(WARN)

- 1个MaintenanceWriter承载depth+snapshot+signal+underlying+kv
- batch_tasks=100足够攒批次，但单线程是瓶颈风险
- **建议**: 监控maintenance_fill_rate，必要时增至2线程

### H3: tick与depth写入顺序不再严格FIFO(WARN)

- 变更前：同一合约tick和depth在同一个shard FIFO队列
- 变更后：tick走TickShard，depth走Maintenance，不同队列/Writer
- 对基于timestamp查询的下游**无影响**
- **需确认**: 下游无"tick必先于depth写入"的隐含假设
