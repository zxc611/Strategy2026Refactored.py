## 一、修复清单

### FIX-01: lifecycle_init.py缩进错误
**问题**: `_force_log_flush()`函数缩进错误，导致IndentationError  
**根因**: 函数被错误缩进为嵌套函数  
**修复**: 将`_force_log_flush()`缩进修正为模块级函数  
**文件**: `lifecycle/lifecycle_init.py` (两个目录)  
**验证**: ✅ PASS — 无IndentationError

### FIX-02: strategy_id类型错误
**问题**: `strategy_id in name`触发TypeError: 'int' object is not iterable  
**根因**: strategy_id为int，但用于`in`操作符检查字符串包含  
**修复**: 恢复strategy_id为int类型；将`strategy_id in name`改为`str(strategy_id) in name`  
**文件**: `strategy/strategy_2026.py`, `alth_monitor.py` / `infra/health_monitor.py`  
**验证**: ✅ PASS — 无TypeError

### FIX-03: DuckDB预检误判
**问题**: DuckDB预检失败，回退到内存模式，导致所有表不存在  
**根因**: PythonGO嵌入式环境将sys.stdout重定向到平台日志系统，导致subprocess.run捕获不到print('OK')输出  
**修复**: 当returncode=0且stderr无错误时，视为预检通过  
**文件**: `data/ds_db_connection.py`  
**验证**: ✅ PASS — DuckDB文件模式正常工作

### FIX-04: instruments_registry表不存在
**问题**: 查询instruments_registry表触发Catalog Error  
**根因**: 表可能不存在于某些环境  
**修复**: 查询前检查表存在性，不存在时回退到futures_instruments+option_instruments  
**文件**: `alth_monitor.py` / `infra/health_monitor.py`  
**验证**: ✅ PASS — 无Catalog Error

### FIX-05: klines_raw表不存在
**问题**: 查询klines_raw表触发Catalog Error  
**根因**: ensure_config_coverage_for_today在独立连接上查询klines_raw，但表可能不存在  
**修复**: 所有SQL查询前检查表存在性，INSERT加try/except保护  
**文件**: `data/ds_data_writer.py`  
**验证**: ✅ PASS — 无Catalog Error

### FIX-06: StrategyState UnboundLocalError
**问题**: `UnboundLocalError: cannot access local variable 'StrategyState'`  
**根因**: 函数内部有条件导入，导致Python将StrategyState视为局部变量  
**修复**: 删除局部导入，使用文件顶部已有导入  
**文件**: `strategy/strategy_2026.py`  
**验证**: ✅ PASS — 无UnboundLocalError

### FIX-07: AUDIT_HMAC_KEY日志级别
**问题**: `[SECURITY] AUDIT_HMAC_KEY使用默认密钥！`日志级别为CRITICAL，刷屏  
**根因**: 测试环境使用默认密钥，触发安全警告  
**修复**: 将日志级别从CRITICAL降为WARNING，增加1小时限流  
**文件**: `config/config_params.py`  
**验证**: ✅ PASS — 无CRITICAL级别刷屏

### FIX-08: DuckDB连接关闭后不重建
**问题**: `Query failed: Connection Error: Connection already closed!` 每30秒3次  
**根因**: 连接关闭后未标记`_unhealthy=True`，`_get_connection`返回同一个已关闭的连接  
**修复**: 
1. `ds_query_cache.py` query(): 连接检查失败时标记`_unhealthy=True`并重建连接；查询异常时也标记
2. `ds_data_writer.py` refresh_chain_coverage_audit(): 失败时标记`conn._unhealthy = True`
**文件**: `data/ds_query_cache.py`, `data/ds_data_writer.py`  
**验证**: ✅ PASS — 连接关闭后自动重建（但20:03重启后发现线程局部连接未清理，见FIX-09）

### FIX-09: 线程局部连接_unhealthy后未清理_THREAD_LOCAL_CONNS
**问题**: 20:03重启后，`Connection already closed!` 仍然每30秒出现  
**根因**: `_destroy_connection`销毁连接后，没有从`_THREAD_LOCAL_CONNS`字典中移除该连接。  
  当同一线程下次调用`_get_connection`时，第479行`_THREAD_LOCAL_CONNS.get(tid)`返回已关闭的旧连接  
  （虽然`_unhealthy=True`会跳过，但进入`_SINGLE_CONN`分支后，`_SINGLE_CONN`也可能已被标记不健康，  
  重建后又被其他线程的线程局部连接拿走，形成"重建-关闭-重建"死循环）  
**修复**: 
1. `ds_db_connection.py` `_destroy_connection()`: 销毁连接时从`_THREAD_LOCAL_CONNS`中移除已销毁连接
2. `ds_db_connection.py` `_return_connection()`: 新增线程局部连接归还逻辑，不健康时触发销毁重建
**文件**: `data/ds_db_connection.py`  
**验证**: ⏳ 待重启验证（代码已部署，需重启策略生效）
