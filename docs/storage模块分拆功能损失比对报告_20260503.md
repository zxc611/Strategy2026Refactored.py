## 三、7项逻辑差异详细分析

### 差异1: `register_instrument` — 🔴 并发安全退化 (P0)

**原始逻辑** (锁外快查 + 锁内double-check):
```python
# 1. 锁外快查 params_service (性能优化)
existing_meta = self._params_service.get_instrument_meta_by_id(...)
if existing_meta:
    return existing_meta.get('internal_id')

# 2. 锁外快查 DB
info = self._get_instrument_info(...)
if info:
    return self._get_info_internal_id(info)

# 3. 锁内double-check + 注册
with self._lock:
    existing_meta = self._params_service.get_instrument_meta_by_id(...)
    if existing_meta:
        return existing_meta.get('internal_id')
    # ... 实际注册逻辑
```

**分拆逻辑** (无锁):
```python
# 1. 无锁查询 params_service + DB
existing_info = self._params_service.get_instrument_meta_by_id(...)
if existing_info is None:
    existing_info = self._get_instrument_info(...)
if existing_info:
    return self._get_info_internal_id(existing_info)

# 2. 无锁直接注册 (no self._lock!)
# ... 直接进入注册逻辑
```

**影响**: 并发调用`register_instrument`可能产生重复ID、重复INSERT、数据不一致。  
**严重等级**: **P0 — 必须修复**  
**修复方案**: 在分拆版`register_instrument`中恢复`with self._lock:`包裹注册段。

### 差异2: `__exit__` — 🔴 模块级单例丢失 (P0)

**原始**: `__exit__`方法体之后包含模块级单例：
```python
_instrument_data_manager_instance: Optional[InstrumentDataManager] = None
_instrument_data_manager_lock = threading.Lock()

def get_instrument_data_manager() -> InstrumentDataManager:
    global _instrument_data_manager_instance
    with _instrument_data_manager_lock:
        if _instrument_data_manager_instance is None:
            ...
```

**分拆**: 此单例函数和变量**完全缺失**。

**影响**: `config_service.py`、`diagnosis_service.py`、`strategy_lifecycle_mixin.py` 均通过 `from ali2026v3_trading.storage import get_instrument_data_manager` 引用此函数。  
**严重等级**: **P0 — 如果启用facade会导致ImportError**  
**修复方案**: 在facade或`storage_core.py`中补回`get_instrument_data_manager`函数及全局变量。

---

### 差异3: `save_external_kline` + `_normalize_tick_fields` — 🟡 类属性位置错乱 (P1)

**原始**: `_KLINE_PROVIDER_PROBES` 定义在`save_external_kline`方法体末尾，`@staticmethod`装饰器紧随其后。

**分拆**: `_KLINE_PROVIDER_PROBES` 被移到`_normalize_tick_fields`方法体末尾，装饰器也错位：
- `save_external_kline` 末尾从 `_KLINE_PROVIDER_PROBES + @staticmethod` 变为 `@requires_phase(InitPhase.READY)`
- `_normalize_tick_fields` 末尾从 `@requires_phase(InitPhase.READY)` 变为 `_KLINE_PROVIDER_PROBES + @staticmethod`

**影响**: `_KLINE_PROVIDER_PROBES`类属性定义位置错误——它在原始中属于`save_external_kline`方法定义之后的类属性，分拆后归入了`_normalize_tick_fields`之后。由于Python类属性是类级别的，位置不影响功能，但**语义归属混乱**。  
**严重等级**: **P1 — 功能无损但语义错误，影响可维护性**

---

### 差异4: `_is_ext_kline_missing` — 🟡 装饰器phase降级 (P1)

**原始**: `@requires_phase(InitPhase.READY)`  
**分拆**: `@requires_phase(InitPhase.EXTERNAL_SERVICES)`

**影响**: phase守卫从READY(8)降为EXTERNAL_SERVICES(3)，允许更早阶段调用此方法。  
**严重等级**: **P1 — 降级可能允许不安全调用，但此方法仅读`_last_ext_kline`字典，风险可控**

---

### 差异5: `_resolve_kline_provider` — 🟡 类引用变更 (P1)

**原始**: `InstrumentDataManager._KLINE_PROVIDER_PROBES`  
**分拆**: `_StorageQueryMixin._KLINE_PROVIDER_PROBES`

**影响**: 分拆后类名变了，引用路径随之变化。MRO保证最终解析到同一属性，**功能无损**。  
**严重等级**: **P1 — 功能无损，仅适配分拆**

---

### 差异6: `register_instrument` exchange参数处理 — 🟡 语义缩窄 (P2)

**原始**: `if str(exchange or 'AUTO').strip() == 'AUTO':`  
**分拆**: `if exchange == "AUTO":`

**影响**: 原始版容错：`None`→`'AUTO'`、`'  AUTO  '`→`'AUTO'`、`'auto'`(不触发，因为大小写敏感)。分拆版严格相等，`None`或带空格值将跳过推断。  
**严重等级**: **P2 — 缩窄了容错范围，但调用方传参规范应为`"AUTO"`**

---

### 差异7: `register_instrument` 锁外快查移除 — 🟡 性能退化 (P2)

原始版有锁外快查(params_service + DB)避免不必要的锁竞争。分拆版合并为单一无锁查询。  
**影响**: 在高并发注册场景下，原始版锁外快查可跳过90%+的锁获取。分拆版无锁但也无double-check保护。  
**严重等级**: **P2 — 性能退化（与差异1同源，修复差异1时一并恢复锁外快查）**
