## 一、架构设计评审 (C+, 6.0/10)

### 1.1 架构亮点

- **Facade + 委托模式**: 通过`__getattr__`实现服务委托，`OrderService`作为Facade将订单处理委托给6+子服务（`OrderExecutor`、`OrderStateManager`、`OrderRiskGuard`等），成功将千行`send_order`方法拆解为三阶段流水线（前置校验→平台下单→后置持久化）。
- **三阶段订单流水线**: `_pre_send_checks` → `_execute_platform_insert` → `_post_send_persist`，职责清晰，可独立测试。
- **桥接解耦**: `RiskPositionBridge` / `PositionBridge` 协议接口成功打破risk ↔ position的循环依赖。
- **模块化拆分**: 原超大文件拆分为多个职责单一的模块（如`data_service.py`拆分为5个Mixin+RealTimeCache），大部分模块控制在300-800行。
- **生命周期状态机**: 完整状态 RUNNING/PAUSED/DEGRADED/PARALLEL_RUNNING，四元状态同步（`_state`/`_is_paused`/`_is_running`/`_is_trading`）。
- **Tick处理管道**: 500ms处理阈值+2s/10s窗口监控+降级模式（不丢弃tick），达到基金级标准。

### 1.2 架构严重缺陷

#### 缺陷1: 双副本代码不一致（P0）

[`strategy_2026.py`](file:///c:/Users/xu/AppData/Roaming/InfiniTrader_QhZijintianfengPythonX64/pyStrategy/demo/ali2026v3_trading/strategy_2026.py) 和 [`strategy/strategy_2026.py`](file:///c:/Users/xu/AppData/Roaming/InfiniTrader_QhZijintianfengPythonX64/pyStrategy/demo/ali2026v3_trading/strategy/strategy_2026.py) 存在两个副本，且`strategy/`子目录版本包含关键修复（`FIX-20260706-STATE-FORCE`、`FIX-20260707-PAUSE-SAFE`），根目录版本缺少这些修复。同样的双副本问题存在于`strategy_monitoring_layer.py`、`position_check_service.py`、`strategy_config_layer.py`、`lifecycle_callbacks.py`等至少6对文件。

**影响**: 代码修改可能只在其中一个副本生效，导致修复"时好时坏"。这是Fix 18-20声称已修复但实际无效的根因之一。

**建议**: 统一为一个源文件，另一个改为`from strategy.xxx import *`的re-export。

#### 缺陷2: `__getattr__`委托链脆弱

多个核心类使用`__getattr__`将方法调用多层委托给内部服务对象：`StrategyCoreService` → `Strategy2026` → `BoxSpringStrategy` → `BoxSpringDetectorService` / `BoxSpringExecutorService`。这种多层委托在调试时极度困难，且容易导致无限递归（已在`strategy/strategy_2026.py`中发现需要`object.__getattribute__`绕过的问题）。

**建议**: 使用显式组合替代`__getattr__`委托，或至少限制委托层数≤2层。

#### 缺陷3: 桩函数/未完成功能

[`box_spring_executor.py`](file:///c:/Users/xu/AppData/Roaming/InfiniTrader_QhZijintianfengPythonX64/pyStrategy/demo/ali2026v3_trading/box_spring_executor.py) 中：

## 二、风控安全评审 (B+, 7.5/10)

### 2.1 风控亮点

- **5级风控等级**: LOW → MEDIUM → HIGH → RESTRICTED → CRITICAL，RESTRICTED中间状态防止直接熔断。
- **断路器机制**: 连续失败≥5次自动熔断，60秒自动恢复半开，跨日保留需人工`confirm_daily_resume()`。
- **胖手指防护**: 价格偏离3%/20%/50%分级拦截，按`action`类型分级。
- **4重dry_run拦截**: `svc._dry_run_mode` → `_provider_ref._dry_run_active` → `platform_insert_order.__self__._dry_run_active` → `get_cached_params().get('dry_run_mode')`，任一为True即拦截。
- **fail-safe设计**: 所有风控异常默认返回阻断结果，宁可误杀不可漏过。
- **自成交禁止期**: 30分钟禁止期，CLOSE订单豁免，拆单原子性检测。
- **安全元数据层**: `SafetyMetaLayer`实现策略级风控实例隔离，包含`on_equity_update()`、`_peak_equity`峰值追踪、`is_hard_stop_triggered()`、跨策略风险守卫。

### 2.2 风控缺陷

#### 缺陷1: dry_run拦截反复出现逻辑漏洞（P0）

**历史**:
- V1: `_dry_run_mode=False is not None` → 跳过所有回退 → 122次订单穿透到平台
- V2: 修复为4级回退，`if getattr(svc, '_dry_run_mode', False)` 替代 `is None`
- **但**: `init_order_service_attrs()`中`get_params_service().get_bool('dry_run_mode', False)`可能返回False（ParamsService未加载dry_run_mode键），导致V2的四重回退中检查1失败。虽然检查2-4可兜底，但依赖`lifecycle_callbacks`正确设置`_dry_run_active`。

**建议**: 在`init_order_service_attrs`中直接读取配置文件而非依赖ParamsService。

#### 缺陷2: hard_stop跨日保留需人工恢复

7/5触发后7/6、7/7、7/8持续保留，策略连续多日无法开仓。顶级基金要求自动恢复机制（如次日开盘自动reset）或至少告警通知。20:46会话中563次风控阻断（`hard_stop_triggered`）均为跨日保留，而非真实风险事件，过度阻断掩盖真正的风险告警。

#### 缺陷3: 风险参数偏离基金标准

| 参数 | 本系统值 | 基金基准 | 评估 |
|------|---------|---------|------|
| max_risk_ratio | 0.8 | 0.02-0.05 | 过高（单边80%风险占比） |
| daily_loss_hard_stop_pct | 0.10 | 0.03-0.05 | 偏高 |
| max_open_positions | 3 | 5-20 | 偏少（集中度风险） |
| circuit_breaker_pause_sec | 60 | 300+ | 偏短 |

#### 缺陷4: 影子策略隔离未验证

`_shadow_strategy_signal.py`中的`verify_shadow_isolation()`仅在测试中调用，生产环境无验证。

### 3.1 核心发现：策略胜率0%

7/8 22:06重启后46笔交易（37笔开仓+46笔平仓，含9笔遗留持仓平仓），**0笔盈利，胜率0%**。

| 盈亏区间 | 笔数 | 占比 |
|---------|------|------|

| 持平 (=0) | 4 | 11% |

## 四、代码质量评审 (C, 5.0/10)

### 4.1 代码质量亮点

- **模块化拆分**: 从超大文件拆分为300-800行的职责单一模块。

## 十四、评级总结

**ali2026v3_trading 量化交易系统**在架构设计和风控安全方面展现了较高的工程水平：Facade+委托模式、三阶段订单流水线、5级风控+fail-safe设计、桥接解耦等均属优秀实践。DuckDB数据管道在16,288合约规模下稳定运行。

然而，**策略有效性是致命短板**。V1报告的止盈/止损/时间止损三元错配分析揭示了策略亏损的数学必然性——`tp=1.8`（需涨80%）+ `sl=0.30`（需跌30%）+ 实际持仓3-5分钟（CRM+PLR链式缩短），导致时间止损必然垄断所有退出，止盈止损形同虚设。37笔交易0%胜率不是偶然，是**策略设计缺陷的必然结果**。

代码质量方面，双副本文件不一致、桩函数未实现、硬编码配置、`__getattr__`委托链脆弱等问题需要在生产使用前解决。运维可观测性方面，信号/交易计数为0运行2天未被发现，说明监控告警体系缺失。

**总体评级C+**，不建议直接用于实盘交易。完成P0整改项后（三元组重新标定、统一副本、修复信号链路、建立监控告警），可升级至B级，进一步完成后升级至A-级。
