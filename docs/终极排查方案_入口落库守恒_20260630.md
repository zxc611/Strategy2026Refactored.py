## 二、未被20轮排查覆盖的损耗路径（3维度6项）

### 2.1 维度A：正则/标准化不一致（3项）

#### A-01 [P0] `cache_instrument_info` 不做normalize——键不匹配导致tick路由全量失败

**文件**: `_params_instrument_cache.py:127`

**问题**: `cache_instrument_info` 写入 `_instrument_id_to_internal_id` 时，**直接使用传入的 `instrument_id` 原始值作为键**，不调用 `normalize_instrument_id` 标准化。

**损耗链**:
```
load_caches_from_db → cache_instrument_info(instrument_id='SHFE.au2501')
  → _instrument_id_to_internal_id['SHFE.au2501'] = 42
    → on_tick → _strip_exchange_prefix('SHFE.au2501') → 'au2501'
      → get_internal_id('au2501') → _instrument_id_to_internal_id.get('au2501') → None
        → get_instrument_meta_by_id返回None → tick被静默丢弃
```

**验证方法**: 在 `get_internal_id` 入口添加诊断日志，记录查找失败的 instrument_id 及其在 `_instrument_id_to_internal_id` 中的近似匹配（如 `au2501` vs `SHFE.au2501`）。

**影响评估**: 如果 `load_caches_from_db` 从数据库读取的 `instrument_id` 带交易所前缀（如 `SHFE.au2501`），则**所有tick路由全部失败**，这可以解释"入口仅~600"的现象——因为只有那些 instrument_id 恰好不带前缀的合约才能被路由。

**修复方案**: 在 `cache_instrument_info` 入口添加 `instrument_id = normalize_instrument_id(instrument_id)`。

---

#### A-02 [P1] 两套normalize实现并存——on_tick用正则、register_option用分隔符

**文件**: `subscription_service.py:1310` vs `width_cache_state_mixin.py:468`

**问题**: `_on_tick_impl` 使用 `_strip_exchange_prefix`（正则 `r'([A-Za-z]+\d+.*?)$'`），而 `register_option`/`on_option_tick` 使用 `_normalize_instrument_id`（委托 `shared_utils.normalize_instrument_id`，基于 `.`/`|`/`:` 分隔符分割）。

**当前一致性**: 对标准格式（`SHFE.au2501`、`DCE.m2607`）两者结果一致。

**不一致场景**: 
- `platform|m2607` → 正则匹配 `m2607` ✅ / 分隔符分割 `m2607` ✅
- `SHFE:au2501` → 正则匹配 `au2501` ✅ / 分隔符分割 `au2501` ✅
- `XAUUSD`（纯字母无数字）→ 正则不匹配返回 `XAUUSD` / 分隔符无变化返回 `XAUUSD` ✅

**风险**: 两套逻辑独立维护，未来修改一处时遗漏另一处，导致标准化结果分歧。

**修复方案**: 统一为 `normalize_instrument_id`（分隔符方式更健壮、更可预测），删除 `_strip_exchange_prefix` 正则方式。

---

#### A-03 [P1] `parse_option` 不支持 Call/Put 全称格式

**文件**: `instrument_parser.py:132`

**问题**: 正则3 `r'^([A-Za-z]+)(\d{3,4})-?([CP])-?(\d+(?:\.\d+)?)$'` 中 `[CP]` 只匹配单字母。CFFEX若使用 `Call`/`Put` 全称格式，三个正则均不匹配，`is_option` 返回 `False`，合约被错误归入期货。

**影响**: `classify_instruments` 初始化分类出错，但 `on_tick` 路由依赖 ParamsService metadata 的 `type` 字段而非正则，tick数据不丢失。**P1级**。

### 2.2 维度B：键唯一性（2项）

#### B-01 [P1] WidthCache与ParamsService双层缓存不一致窗口

**文件**: `width_cache_types.py:161-180` vs `_params_instrument_cache.py`

**问题**: WidthCacheStateService 维护独立的 `_instrument_id_to_internal_id` 字典，与 ParamsService 的同名字典是**不同实例**。`_resolve_internal_id` 先查本地缓存，再查 ParamsService 回退。

**不一致场景**: 
- WidthCache 本地缓存因 TTL 淘汰某映射 → 回退查 ParamsService → **安全**
- ParamsService 被 `clear_instrument_cache` 重建 → WidthCache 本地缓存未同步 → 回退查 ParamsService → **安全**
- WidthCache 本地缓存有映射A→42，ParamsService 有映射A→43（数据库更新后） → **不一致但极罕见**

**结论**: 双层缓存设计有短暂不一致窗口，但回退机制确保最终一致性。**不会导致永久性查找失败**。P2级。

---

#### B-02 [P2] `_option_info` 覆盖写入——不同instrument_id映射同一internal_id

**文件**: `width_cache_state_mixin.py:513`

**问题**: `_option_info[internal_id]` 使用 `internal_id` 为键。如果两个不同 `instrument_id` 映射到同一 `internal_id`，后者覆盖前者。

**防护**: `cache_instrument_info`（`_params_instrument_cache.py:152-170`）有冲突检测——同一 `internal_id` 对应不同 `instrument_id` 时抛 `RuntimeError`。因此**此路径在正常流程下不会触发**。

**但**: 如果 WidthCache 的 `_resolve_internal_id` 和 ParamsService 的 `get_internal_id` 对同一 `instrument_id` 返回不同的 `internal_id`（因缓存不一致），则可能产生覆盖。**极罕见，P2级**。

### 2.3 维度C：平台推送验证（1项）

#### C-01 [P0] tick去重注释与代码矛盾——1秒vs5秒

**文件**: `subscription_service.py:1290`（注释）vs `1295`（代码）

**问题**: 注释写"5秒内相同三元组才丢弃"，但代码实际去重窗口为 **1.0秒**（`(_now - _last_dedup_ts) < 1.0`）。

**影响**: 
- 盘口静止时（价格和成交量不变），每秒仅第一个tick通过
- 模拟环境tick频率低，1秒窗口可能丢弃有效tick
- 但1秒窗口比5秒窗口**更宽松**，不应是损耗主因

**修复方案**: 修正注释为1秒，或根据实际需求调整窗口。

## 三、顶级基金式排查方案：全链路守恒验证

### 3.1 设计原则

顶级基金量化系统的数据守恒验证遵循以下原则：

1. **入口即审计**：每个数据入口点必须有计数器，记录接收数量
2. **环节即校验**：每个处理环节必须有输入/输出计数对比
3. **损耗即告警**：任何环节的损耗率>0.1%必须产生告警
4. **归因即证据**：每个损耗点必须有日志证据，杜绝推测性归因
5. **守恒即断言**：最终断言入口=落库，不满足则阻断启动

### 3.2 全链路守恒审计方案

#### 阶段三：instrument_id标准化一致性验证（核心排查项）

**验证脚本**（在策略启动后执行）：

```python
def verify_instrument_id_consistency():
    """验证所有合约的instrument_id在所有normalize路径下结果一致"""
    from ali2026v3_trading.config._params_instrument_cache import get_params_service
    from ali2026v3_trading.infra.instrument_parser import strip_exchange_prefix
    from ali2026v3_trading.infra.shared_utils import normalize_instrument_id
    
    ps = get_params_service()
    all_ids = list(ps._instrument_id_to_internal_id.keys())
    
    inconsistencies = []
    for inst_id in all_ids:
        result_regex = strip_exchange_prefix(inst_id)
        result_split = normalize_instrument_id(inst_id)
        if result_regex != result_split:
            inconsistencies.append({
                'original': inst_id,
                'regex_result': result_regex,
                'split_result': result_split,
            })
    
    # 反向验证：平台推送格式 → 注册格式
    for inst_id in all_ids:
        # 模拟平台推送带交易所前缀
        for prefix in ['SHFE.', 'DCE.', 'CZCE.', 'INE.', 'CFFEX.', 'GFEX.']:
            platform_id = prefix + inst_id
            normalized = strip_exchange_prefix(platform_id)
            if normalized != inst_id:
                inconsistencies.append({
                    'original': inst_id,
                    'platform_push': platform_id,
                    'normalized': normalized,
                    'expected': inst_id,
                })
    
    return inconsistencies
```
