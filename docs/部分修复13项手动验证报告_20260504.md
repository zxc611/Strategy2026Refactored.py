## 发现并当场修复的隐患

### 隐患A: _get_cached_params_fn无锁保护（#1相关）

**发现**: `_get_cached_params_fn`是模块级全局变量，多线程并发首次调用时存在TOCTOU竞态
**风险等级**: LOW（Python import锁+GIL保证最终一致性，但不够严谨）
**修复**: 加双重检查锁定(DCL)模式——新增`_get_cached_params_lock = threading.Lock()`，外层快速路径`if _get_cached_params_fn is None:`，内层`with _get_cached_params_lock:` + 二次检查
**验证**: ✅ 语法通过，首次import仅触发一次，后续零锁开销

### 隐患B: strategy_tick_handler.py遗漏2处except:pass（#9相关）

**发现**: L100 `except Exception: pass`（_route_shard_index）和 L490 `except Exception: return`（_save_tick_snapshot）
**风险等级**: MEDIUM（调试时无法发现失败）
**修复**: 
- L100: `except Exception as _route_err: logging.debug(f"[_route_shard_index] Fallback route: {_route_err}")`
- L490: `except Exception as _snap_err: logging.debug(f"[_save_tick_snapshot] Path resolution failed: {_snap_err}"); return`
**验证**: ✅ 语法通过，grep确认strategy_tick_handler.py中`except.*:pass`零匹配

### 隐患C: t_type_service预加载重试死循环（#11相关）

**发现**: `_preload_failed=True`后finally重置`_preload_started=False`，下次on_future_tick将无限重试新线程
**风险等级**: HIGH（DB持续不可用时产生无限线程）
**修复**: 在`_ensure_preload_started`中加`_preload_retry_count`限制，最大3次重试，超限后log error并return。提供`_preload_retry_count=0`手动重置方式。
**验证**: ✅ 语法通过，3次失败后停止重试，可手动重置

### 隐患D: scheduler_service cancel_event无法中断已运行的func()（#13相关）

**发现**: Python无线程取消机制，cancel_event仅在func()调用前检查
**风险等级**: MEDIUM（语言固有限制，非代码缺陷）
**修复**: 添加注释标注此限制：`若job.func()已进入执行，cancel_event无法中断（Python线程模型限制）`，daemon=True兜底
**验证**: ✅ 语义正确，docstring说明了限制
