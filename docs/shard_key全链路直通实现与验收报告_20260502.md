## 一、需求背景

### 1.1 用户原始需求

> 1、请实现在品种ID创建时一次性算好并增加品种ID哈希值路由属性
> 2、删除没有品种ID哈希值（通道路由）的旧数据表和表中数据(模拟数据没价值)，让策略重新创建新表
> 3、验证功能完整性

### 1.2 设计理念

> 它把逻辑编译进了数据，是"全链路直通"的自然延伸。
> 更专业又更简单的最终方案是：存储分片键（shard_key）而非最终通道号，进而在所有需要通道绑定的地方，直接用 品种.shard_key % 通道总数 来得到通道。如果您的通道总数始终保持为 2 的幂，甚至可以把分片键直接设计为位掩码结果，连取模都用位操作完成。

### 1.3 核心目标

| 目标 | 描述 |
|------|------|
| 逻辑编译进数据 | 品种注册时一次性计算 shard_key 并持久化，运行时零哈希开销 |
| 全链路一致性 | 创建→存储→加载→路由→回退，所有路径 shard_key 结果相同 |
| 跨重启确定性 | 重启后从 DB 加载的 shard_key 与注册时计算的完全一致 |
| 位掩码优化 | SHARD_COUNT=16（2^4），`shard_key & 0x0F` 等价于 `shard_key % 16` |

# 修改前（一致性哈希环 + 二分查找）
def route(self, instrument_id: str) -> int:
    product_code = extract_product_code(instrument_id)  # 大写
    ...
    h = self._deterministic_hash(product_code)
    lo, hi = 0, len(self._ring) - 1
    while lo < hi:           # 二分查找虚拟节点
        mid = (lo + hi) // 2
        if self._ring[mid][0] < h:
            lo = mid + 1
        else:
            hi = mid
    if lo >= len(self._ring):
        lo = 0
    shard_idx = self._ring[lo][1]
    ...

# 修改后（简单取模 + 小写统一）
def route(self, instrument_id: str) -> int:
    _pc_raw = extract_product_code(instrument_id)
    product_code = _pc_raw.lower() if _pc_raw else ''   # 统一小写
    if not product_code:
        product_code = 'unknown'
    ...
    h = self._deterministic_hash(product_code)
    shard_idx = h % self._shard_count                    # 简单取模
    ...
```

# 修改前
def _route_shard_index(self, instrument_id: str) -> int:
    return self._shard_router.route(instrument_id)  # 运行时哈希

# 修改后
def _route_shard_index(self, instrument_id: str) -> int:
    _SHARD_MASK = self._TICK_SHARD_COUNT - 1
    try:
        ps = get_params_service()
        info = ps.get_instrument_meta_by_id(instrument_id)
        if info:
            shard_key = info.get('shard_key', -1)
            if isinstance(shard_key, int) and shard_key >= 0:
                return shard_key & _SHARD_MASK       # 预计算快速路径
    except Exception:
        pass
    _pc_raw = extract_product_code(instrument_id)    # 回退：简单取模
    pc = _pc_raw.lower() if _pc_raw else ''
    return ShardRouter._deterministic_hash(pc) & _SHARD_MASK if pc else 0
```

```python
def _backfill_shard_key(self, conn: duckdb.DuckDBPyConnection):
    from ali2026v3_trading.shared_utils import ShardRouter
    backfilled = 0
    for table_name in ('futures_instruments', 'option_instruments'):
        rows = conn.execute(
            f"SELECT internal_id, product FROM {table_name} WHERE shard_key IS NULL"
        ).fetchall()
        for row in rows:
            product_code = row[1].lower()
            shard_key = ShardRouter._deterministic_hash(product_code)
            conn.execute(
                f"UPDATE {table_name} SET product_code = ?, shard_key = ? WHERE internal_id = ?",
                [product_code, shard_key, row[0]],
            )
            backfilled += 1
```

## 三、审计发现的 BUG 及修复

### 3.1 BUG 清单

| BUG ID | 严重度 | 根因 | 影响 | 修复方案 |
|--------|--------|------|------|----------|
| BUG-3 | **严重** | `_make_instrument_info` 用大写 product_code（`extract_product_code` 返回大写），INSERT 用小写 `.lower()`，md5 大小写敏感 | 重启后 shard_key 漂移，同一合约路由到不同 shard | 全链路统一 `.lower()` |
| BUG-2 | **中等** | 主路径用 `shard_key & mask`（简单取模），回退路径用 `ShardRouter.route()`（一致性哈希环），两种算法结果不同 | 回退时路由到错误 shard | 全部统一为 `_deterministic_hash & mask` |
| BUG-Ring | **严重** | 删除 `_build_consistent_hash_ring` 后，`reconfigure()` 仍引用 `self._ring` | 调用 reconfigure 时 `AttributeError` 崩溃 | 改为 `h % self._shard_count` |
| BUG-4 | **低** | `_get_instrument_info`/`_get_info_by_id` SELECT 缺少 product_code/shard_key 列 | 缓存未命中时重建的 info 与 DB 值不一致 | 扩展 SELECT + 优先使用 DB 值 |
| BUG-Dup | **中等** | `extract_product_code(inst_id).lower() if extract_product_code(inst_id) else ''` 调用两次 | 格式异常时两次调用可能返回不同结果 | 提取为 `_pc_raw` 变量，单次调用 |

### 3.2 BUG 修复时间线

```
1. 实现 shard_key 预计算 → 发现 BUG-3（大小写不一致）
2. 修复 BUG-3 → 发现 BUG-2（路由算法不一致）
3. 修复 BUG-2 → 发现 BUG-Ring（reconfigure 崩溃）
4. 修复 BUG-Ring → 发现 BUG-4（SELECT 缺列）
5. 修复 BUG-4 → 发现 BUG-Dup（重复调用）
6. 修复 BUG-Dup → 全部实证通过
```

## 六、关键设计决策

### 6.1 为什么选择简单取模而非一致性哈希环

| 维度 | 一致性哈希环 | 简单取模 |
|------|-------------|----------|
| 路由一致性 | 与 `shard_key & mask` 不一致 | 完全一致 |
| 计算复杂度 | O(log N) 二分查找 | O(1) 取模/位运算 |
| 代码复杂度 | 需维护虚拟节点环 | 零额外数据结构 |
| 伸缩性 | 通道变更时最小迁移 | 通道变更时全量重分布 |
| 适用场景 | 通道数频繁变更 | 通道数固定为 2^n |

**决策**：当前架构 SHARD_COUNT=16 固定为 2 的幂，且用户明确要求"直接用 shard_key % 通道总数"，简单取模是最优选择。若未来需要动态伸缩通道数，可通过 `reconfigure()` 重新计算所有绑定。

### 6.2 为什么存储 shard_key 而非通道号

| 维度 | 存储通道号 | 存储 shard_key |
|------|-----------|---------------|
| 通道伸缩 | 必须全量更新 DB | 只需改 SHARD_COUNT，重新取模 |
| 跨系统一致性 | 通道号含义依赖配置 | shard_key 是纯数学值，无歧义 |
| 位掩码优化 | 通道号无法位运算 | `shard_key & mask` 直接得到通道 |

### 6.3 为什么统一 product_code 为小写

`_deterministic_hash` 使用 md5，大小写敏感：`hash("IF") ≠ hash("if")`。若全链路大小写不统一，同一品种在不同路径计算出不同 shard_key，导致跨重启路由漂移。统一为小写是最简单、最可靠的方案。
