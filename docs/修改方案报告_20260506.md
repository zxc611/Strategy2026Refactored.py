## 一、问题原因地图（Root Cause Map）

### P0: 下单失败 — `unexpected keyword argument 'direction'`

```
execute_option_trading_cycle (定时任务, 每30秒)
  → strategy_core_service.py: execute_option_trading_cycle()
    → order_service.py:178: execute_by_ranking(targets, direction='BUY', action='OPEN')
      → order_service.py:195-201: send_order(instrument_id, volume, price, direction=target_direction, action=target_action)
        → order_service.py:149-156: self._platform_insert_order(exchange, instrument_id, volume, price, direction, action)
          → BaseStrategy.send_order() ← 平台API, 不接受direction参数
            ❌ TypeError: got an unexpected keyword argument 'direction'
```

**根因**: `OrderService.send_order()` L149-156 调用平台API时传递了 `direction` 和 `action` 参数，但平台的 `BaseStrategy.send_order()` 不接受这些参数。平台API的参数签名与OrderService的假设不匹配。

**影响文件**: `order_service.py` L107-176 (send_order方法), L178-206 (execute_by_ranking方法)

---

### P1: 合约元数据未找到 — `Contract metadata not found for: XXX`

```
subscription_manager.py:1127: _on_tick_impl(normalized_id, last_price, volume)
  → L1152: ps.get_instrument_meta_by_id(normalized_id)
    → params_service.py:263: get_internal_id(instrument_id)
      → params_service.py:236: self._instrument_id_to_internal_id.get(instrument_id)
        → 返回 None ← 合约不在缓存中
  → L1186: if not routed:
    → L1201: logger.warning("Contract metadata not found for: %s", normalized_id)
```

**根因**: `_instrument_id_to_internal_id` 缓存中不包含这些合约。数据库中**存在**这些合约（如 `HO2605-P-2900` internal_id=58），但 `ParamsService` 的缓存加载可能因为时序问题（`clear_instrument_cache` 在 `load_caches_from_db` 之前被调用）导致部分合约未被加载。

**影响文件**: `params_service.py` L606-746 (load_caches_from_db), L340-361 (clear_instrument_cache)

---

### P2: 订阅操作失败 — `Operation failed: '2605'`

```
subscription_manager.py:1160: tts.on_option_tick(normalized_id, float(last_price), volume)
  → t_type_service.py:215: self._width_cache.on_option_tick(instrument_id, price, volume)
    → width_cache.py:810: self._set_current_status(internal_id, info, initial_status)
      → width_cache.py:516: bucket = self._status_counts[future_id][month][opt_type]
        ❌ KeyError: '2605' ← _status_counts[future_id] 是普通dict, 不含month键
```

**根因**: `_recalc_all_state_for_underlying` L957 将 `temp_status_counts` (defaultdict) 转换为 `dict(temp_status_counts)`，嵌套的 defaultdict 也变成了普通 dict。之后 `_set_current_status` L516 访问 `self._status_counts[future_id][month][opt_type]` 时，如果 `month` 不在普通 dict 中，就会抛出 `KeyError('2605')`。

**影响文件**: `width_cache.py` L516 (_set_current_status), L957 (_recalc_all_state_for_underlying)

---

**根因**: 策略停止时，后台线程和定时任务没有被正确清理。

### P4: 诊断误报 — "无Tick数据(实时链路异常)"

**影响文件**: `diagnosis_service.py`

## 三、详细修改方案

### 修改1: P0 下单失败 — 适配平台API参数

**模块文件**: `c:\Users\xu\AppData\Roaming\InfiniTrader_SimulationX64\pyStrategy\demo\ali2026v3_trading\order_service.py`

**问题行号**: L144-156

**问题描述**: `send_order()` L149-156 调用 `self._platform_insert_order()` 时传递了 `direction` 和 `action` 参数，但平台API `BaseStrategy.send_order()` 不接受这些参数。

**修改操作**: 
1. 在 `bind_platform_apis()` 中检测平台API的参数签名
2. 在 `send_order()` 中根据平台API签名动态构建参数
3. 使用 `inspect.signature()` 检测平台API接受的参数

**成功修复标准**:
- 下单不再报 `unexpected keyword argument 'direction'` 错误
- 日志中出现 `[OrderService] 平台下单成功` 而非 `平台下单失败`

**验证方法**: 运行策略，检查日志中是否出现下单成功记录

**彻底性保证**: 通过参数签名检测，自动适配任何平台API变体

**预防隐患**: 在 `bind_platform_apis()` 中增加参数签名日志，便于后续排查

### 修改2: P2 订阅操作失败 — _status_counts defaultdict丢失

**模块文件**: `c:\Users\xu\AppData\Roaming\InfiniTrader_SimulationX64\pyStrategy\demo\ali2026v3_trading\width_cache.py`

**问题行号**: L516, L957

**问题描述**: `_recalc_all_state_for_underlying` L957 将 `temp_status_counts` (defaultdict) 转换为 `dict(temp_status_counts)`，嵌套的 defaultdict 变成普通 dict。之后 `_set_current_status` L516 访问不存在的 month 时抛出 KeyError。

**修改操作**:
1. L516: 在 `_set_current_status()` 中添加 defaultdict 兜底，确保访问不存在的键时自动创建
2. L957: 保持 `_status_counts[future_internal_id]` 的 defaultdict 嵌套结构，不转换为普通 dict

**成功修复标准**:
- 日志中不再出现 `Operation failed: '2605'` 错误
- 所有期权tick都能正确路由到 `on_option_tick`

**验证方法**: 运行策略，检查日志中是否还有 `Operation failed` 错误

**彻底性保证**: 修改 `_status_counts` 的赋值逻辑，确保嵌套 defaultdict 结构不被破坏

**预防隐患**: 在 `_set_current_status` 中增加防御性检查
