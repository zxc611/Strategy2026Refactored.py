## 4. 核心发现：`RealTimeCache` 的 tick 丢失机制

### 4.1 发现过程

日志行10454-11012 揭示了策略重启时的关键时序：

| 时间 | 事件 | 日志行 |
|------|------|--------|
| 16:08:19 | `params_service未就绪，tick暂存` | 10454 |
| 16:08:19 | pending_count 增长：1→2→3→5→10→20→50→100→200→500... | 10454-10464 |
| 16:08:24 | `params_service已接入，回放暂存tick (pending_count=798)` | 11011 |
| 16:08:24 | **`暂存tick回放完成 (replayed=0, dropped=0)`** | **11012** |

### 4.2 影响

- **798 个 tick** 在策略重启窗口（16:08:19-16:08:24）到达
- **全部 798 个 tick 回放数为 0**——静默丢弃
- 这些 tick 中包含 SHFE 期权的第一批数据，如果这些是第一波价格更新，丢失后意味着某些期权永远得不到初始价格

### 4.3 这不能解释 AL vs ZN 差异

此机制影响**所有**期权品种，AL 和 ZN 的 tick 都可能在此窗口被丢弃。因此不能解释为什么 AL2606 后续有 tick 而 ZN 没有。

## 6. AL vs ZN 差异的自洽根因

### 6.1 排除的假说

| 假说 | 排除原因 |
|------|---------|
| 订阅格式差异 | PROBE_SUB 显示格式完全一致：`al2606C18900` vs `zn2605C19200` |
| ID映射失败 | 两者都在 `_option_info` 中有条目（2752条），证明 internal_id 映射成功 |
| 代码路径差异 | 同属SHFE、同走Path A+C，KeyError相同，无差异化代码 |
| RealTimeCache丢弃 | 影响所有品种，非AL独有 |

### 6.2 确立的根因

**根因是平台数据源对SHFE不同品种的期权tick数据推送存在差异。**

证据链：

1. **订阅层面**：2785个合约全部订阅成功（`Bulk subscription completed: success=2785, failed=0`），含AL+ZN+CU+AU全部期权
2. **注册层面**：2752个期权全部在 `_option_info` 中有条目
3. **价格层面**：
   - CFFEX（HO/IO/MO）：tick数据持续到达，价格实时更新，状态正常 → **数据源正常推送**
   - AL2606：tick数据持续到达（14:36 16个 → 18:41 72个非other），价格实时更新 → **SHFE针对AL2606月份有tick推送**
   - AL2605/2607/2608：有合成价格（6050→5950递减），全other → tick未达或仅单次更新，不足以产生方向分类
   - ZN/CU/AU：价格@0.00，全other → **tick从未到达**

4. **ContractWatch佐证**（14:35，前会话）：
   - CFFEX 期权获 tick：HO2605-C-2800 ticks=11、IO2605-C-4200 ticks=14、MO2605-C-7000 ticks=25
   - SHFE 期权无 tick：cu2605C90000 NO_TICK（wait 21.8s）、au2605C1000 NO_TICK、al2605C24000 NO_TICK、zn2605C26000 NO_TICK

### 6.3 自洽解释

```
┌──────────────────────────────────────────────────┐
│              平台 C++ 层数据源/回放引擎              │
│                                                    │
│  CFFEX 期权 tick ──────────────► 正常推送 ✅        │
│  SHFE AL tick  ────────────────► 部分月份推送 ⚠️    │
│  SHFE ZN tick  ────────────────► 不推送 ❌           │
│  SHFE CU tick  ────────────────► 不推送 ❌           │
│  SHFE AU tick  ────────────────► 不推送 ❌           │
│                                                    │
├──────────────────────────────────────────────────┤
│            Python 策略层                            │
│                                                    │
│  订阅模块: 2785全订阅成功                           │
│  注册模块: 2752全注册成功（_option_info有条目）      │
│  tick路由: on_option_tick → width_cache分类         │
│                                                    │
│  分类结果取决于实际tick数据是否到达:                  │
│    • 有tick → 价格更新 → 方向分类 → 非other ✅       │
│    • 无tick → 价格不变 → 无法分类 → other @0.00 ❌   │
│    • 极少tick → 单次更新 → 仍无法分类 → other       │
└──────────────────────────────────────────────────┘
```

**Python策略层的代码不存在对AL/CU/ZN/AU的差异化处理**。所有SHFE期权走相同的订阅、注册、tick路由路径。差异来自上游——平台的数据引擎对不同品种推送了不同量的tick数据。

`RealTimeCache` 的 798 个 tick 全丢弃（replayed=0）加剧了这个问题：如果在重启窗口丢失了某品种的唯一一次tick推送，该品种将永远没有价格更新。

## 7. 验收报告内容核查

原报告 `第7轮验收报告_期权other状态bug修复_20260505.md` 的核心结论：

| 验收点 | 原始声称 | 实际验证 |
|--------|---------|---------|
| `_option_info` = 2752 | ✅ 全注册 | 确认（16:09:08诊断） |
| 非other状态可选 | 声称AL/CFFEX有 | **仅CFFEX + AL2606 确认** |
| AL全品种工作 | 声称 | **不准确：仅AL2606工作，AL2605全other** |
| CU/AU/ZN全other | 声称 | 确认 |
| 根因解释 | Bug修复 | **根因在上游数据推送，非代码bug** |

### 关键修正

1. **"AL/CFFEX has non-other status"**：实际仅 **CFFEX 全部 + AL2606** 有非other状态。AL2605/2607/2608 全other（虽有合成价格但无tick驱动的方向分类）。

2. **根本原因不在Python策略代码**：三条路径的代码均正确（除 Path A 的 KeyError 因运行版本问题），期权注册成功，tick路由正常。差异来源于上游tick数据推送量的不同。

3. **`RealTimeCache`** 798 tick 全丢弃加剧了数据缺失。
