## 二、逐项详细分析

### R-01: dry_run_mode配置不一致（主因）

**现象**: 策略启动后所有订单被`[DRY-RUN]`拦截，52次下单全部失败

**证据链**:
1. 日志 `2026-07-01 16:32:16.259` - `[DRY-RUN] 策略以模拟模式启动，不会实际下单！`
2. 日志 `2026-07-01 16:32:19.541` - `[DRY-RUN] 订单拦截(仅记录)` × 52次
3. `params.yaml:787` - `dry_run_mode: default: false` ← 正确
4. `params_default.json:38` - `dry_run_mode: default: true` ← **错误！覆盖了yaml**

**参数加载链路**:
```
params_default.json (default:true) 
  → load_default_params_from_json() 
  → get_cached_params() 
  → 返回 dry_run_mode=True
```

**根因**: 实盘模拟跑通调整报告实施时，修改了`params.yaml`新增`dry_run_mode`（default:false），但同步生成的`params_default.json`中写入了`default:true`。由于参数加载优先级JSON > YAML内联DEFAULT_PARAM_TABLE，导致运行时`dry_run_mode=True`。

**修复**: `params_default.json:38` - `"dry_run_mode": {"default": true` → `"default": false`

**附带修复**: `params_default.json`中`max_open_positions`从5恢复为3（与yaml对齐）

### R-04a: lifecycle登记失败触发DEGRADED（18:35重启后新问题）

**现象**: 18:35重启后，全部16288个合约lifecycle登记失败，策略进入DEGRADED状态，后续95001+个tick决策被跳过

**证据链**:
1. 日志 `2026-07-01 18:44:19.895` - `[Subscribe] Done: registered=0, failed=16288, total=16288, elapsed=4.8s`
2. 日志 `2026-07-01 18:44:19.896` - `[Subscribe] 全部lifecycle登记失败，策略进入DEGRADED状态`
3. 日志 `2026-07-01 18:44:15.096` - `[SubscribeTickGapAudit][CRITICAL] tick覆盖率<10%! local_subscription_calls=16288 received=0 gap=100%`
4. 日志 `2026-07-01 18:38:33.306` - `[PositionService] R15-P0-RES-06: 从position_state.jsonl恢复146条持仓(完整快照), CLOSE回放248条`
5. 日志 `2026-07-01 18:38:33.308` - `[PositionService] 时间止损触发: instrument=CF609C17000 elapsed=2258.2min` × 81次

**关键分析**:

`_platform_subscribe_worker`（`lifecycle_bind.py:286-310`）中的逻辑缺陷：

1. **lifecycle登记只是内部记录**：`LifecyclePlatform.subscribe_instrument(inst)`只是将instrument_id append到列表，不调用任何平台API。真正的平台订阅已在数据库登记阶段（`subscribe_all_instruments` → `DataService.subscribe` → `_subscribe`包装器 → 平台API）完成。

2. **DEGRADED触发条件过于激进**：`if failed > 0 and failed == total: p.transition_to(StrategyState.DEGRADED)` — 即使lifecycle登记全部失败，也不应让策略进入DEGRADED状态，因为：
   - lifecycle登记不影响交易功能
   - 平台订阅已在数据库登记阶段完成（日志确认：`Bulk subscription completed: success=16288, failed=0`）
   - DEGRADED状态会导致策略跳过所有tick决策（`strategy_2026.py:661-662`），这是致命的

3. **DEGRADED连锁影响**：
   - tick覆盖率0% → DEGRADED状态跳过所有tick决策
   - 无开仓 → DEGRADED状态跳过所有交易决策
   - 恢复持仓平仓错单 → DEGRADED状态下订单处理异常

**修复**: 移除lifecycle登记失败时的DEGRADED触发逻辑：
```python
# 修复前
if failed > 0 and failed == total:
    logging.error("[Subscribe] 全部lifecycle登记失败，策略进入DEGRADED状态")
    p.transition_to(StrategyState.DEGRADED)

# 修复后
if failed > 0 and failed == total:
    logging.error("[Subscribe] 全部lifecycle登记失败(failed=%d)，但lifecycle登记仅为内部记录，不触发DEGRADED(平台订阅已在数据库登记阶段完成)", failed)
    logging.error("[Subscribe] R-04 ROOT CAUSE: p._lifecycle_platform属性访问或subscribe_instrument调用异常，详见上方R-04 DIAG日志")
```

### R-04b: lifecycle登记失败根因（待诊断）

**现象**: `p._lifecycle_platform.subscribe_instrument(inst)`在16288次调用中全部抛异常

**分析**:
- `LifecyclePlatform.subscribe_instrument`只是`self._subscribed_instruments.append(instrument_id)`，不可能抛异常
- 因此异常必然来自`p._lifecycle_platform`属性访问本身
- `p`是`StrategyCoreService`实例，`p._lifecycle_platform`通过`__getattr__`委托到`_lifecycle_svc._lifecycle_platform`
- 委托链：`StrategyCoreService.__getattr__` → `object.__getattribute__(self, '_lifecycle_svc')` → `getattr(_lifecycle_svc, '_lifecycle_platform')` → `LifecycleService._lifecycle_platform` property → `LifecyclePlatform(provider=self._provider)`

**可能根因方向**:
1. `__getattr__`委托链中某环节抛`AttributeError`（被`_platform_subscribe_worker`的except块捕获）
2. `_lifecycle_svc`未正确初始化（但日志显示策略正常启动，排除此可能）
3. `_lifecycle_platform` property中`LifecyclePlatform(provider=self._provider)`构造失败

**已添加诊断**:
1. `_start_platform_subscribe_async`中添加`p._lifecycle_platform`可访问性检查
2. `_platform_subscribe_worker`中分离`p._lifecycle_platform`属性访问和`subscribe_instrument`调用的异常捕获
3. 首次异常时输出完整traceback、`p`的类型、`p.__dict__`的keys等诊断信息
4. 异常日志级别从debug提升为warning/error

**下次重启后可通过日志`[Subscribe] R-04 DIAG`确认根因**

## 六、预防措施

1. **参数一致性检查**: 修改`params.yaml`后必须同步更新`params_default.json`，增加CI校验
2. **dry_run_mode启动横幅**: 已有m3-07机制，但需增加JSON/YAML一致性校验
3. **虚拟回调路径测试**: 增加`_simulate_dry_run_callbacks`的单元测试，覆盖3条路径
4. **幂等key构造统一**: 抽取为公共方法`build_idempotent_key()`，消除散落构造
5. **DEGRADED触发条件审查**: 非关键功能失败不应触发DEGRADED，需区分"致命错误"和"非致命错误"
6. **lifecycle登记与平台订阅解耦**: lifecycle登记仅为内部记录，其失败不应影响策略运行状态
