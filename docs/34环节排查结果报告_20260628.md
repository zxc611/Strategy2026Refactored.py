## 五、tick_return=597 与 simulated_coverage=15691 详解

### 5.1 什么是tick_return（真实tick）

**tick_return = 平台实际推送了tick数据并成功回调到系统的合约数。**

这些合约在交易时段（09:30~14:58）收到了真实的行情推送：

| 属性 | 值 |
|------|-----|
| 合约数 | 600（355期货 + 245期权） |
| 总行数 | 1,514,829行 |
| 平均成交量 | 97,151.8 |
| 时间范围 | 09:30:05 ~ 14:58:06 |
| last_price | 有真实价格（如8501.6） |
| volume | 有真实成交量（如118449） |
| sync_status | NULL（无标记） |

**样本：**
```
inst=IC2609  price=8501.6  vol=118449  ts=2026-06-28 13:54:37
inst=IC2607  price=8670.6  vol=43128   ts=2026-06-28 13:55:05
inst=PF608   price=6898.0  vol=195139  ts=2026-06-28 13:55:15
```

### 5.2 什么是simulated_coverage（模拟覆盖行）

**simulated_coverage = 系统为"当天未收到真实tick"的合约自动生成的占位行，确保ticks_raw表覆盖配置全集。**

这些合约在当天没有收到平台tick推送，系统在初始化阶段（`ensure_config_coverage_for_today()`）为其插入一行占位数据：

| 属性 | 值 |
|------|-----|
| 合约数 | 15,691（35期货 + 15,656期权） |
| 总行数 | 15,691行（每合约仅1行） |
| 平均成交量 | 0.0 |
| 时间范围 | 00:00:00（固定占位时间戳） |
| last_price | NULL（无价格） |
| volume | 0（无成交量） |
| sync_status | 'simulated_coverage'（明确标记） |

**样本：**
```
inst=MA607P2800  price=NULL  vol=0  ts=2026-06-28 00:00:00  sync=simulated_coverage
inst=MA608C2325  price=NULL  vol=0  ts=2026-06-28 00:00:00  sync=simulated_coverage
inst=MA609P2250  price=NULL  vol=0  ts=2026-06-28 00:00:00  sync=simulated_coverage
```

### 5.3 两者的本质区别

| 维度 | tick_return（真实tick） | simulated_coverage（模拟覆盖） |
|------|------------------------|-------------------------------|
| 数据来源 | 平台行情推送 | 系统自动生成 |
| 价格数据 | 有真实价格 | NULL（无价格） |
| 成交量 | 有真实成交量 | 0 |
| 时间戳 | 真实推送时刻 | 00:00:00（占位） |
| 行数 | 每合约数千~数万行 | 每合约仅1行 |
| 用途 | 驱动策略交易决策 | 确保DB覆盖配置全集，供审计/查询 |
| 可否用于交易 | ✅ 可以 | ❌ 不可以（无价格数据） |

### 5.4 simulated_coverage的生成逻辑

代码位置：`ds_data_writer.py:616` `ensure_config_coverage_for_today()`

```sql
-- 对instruments_registry中每个合约，
-- 如果ticks_raw中没有该合约当天的数据，
-- 插入一行sync_status='simulated_coverage'的占位行
INSERT INTO ticks_raw (..., sync_status, ...)
SELECT ..., 'simulated_coverage', 'simulated_coverage'
FROM instruments_registry r
WHERE NOT EXISTS (
    SELECT 1 FROM ticks_raw t
    WHERE t.instrument_id = r.instrument_id AND t.date = ?
)
ON CONFLICT DO NOTHING
```

**调用时机：** `product_initializer.py:528`，在Init-Step2阶段（18:45:58之前）。

**关键特性：**
- ON CONFLICT DO NOTHING → 如果该合约已有真实tick行，不会重复插入
- 只补齐缺失的合约 → 真实tick到达后，simulated_coverage行仍然存在但不影响
- 明确标记sync_status → 下游查询可区分真实数据和占位数据

### 5.5 为什么tick_return只有597/16288

**原因：模拟环境在非交易时段（18:44重启后）只推送了部分合约的tick。**

具体分布：
- 390个期货中355个收到真实tick（91%）
- 15,898个期权中245个收到真实tick（1.5%）
- 8个品种（ZC,PM,HO,JR,WH,MO,RI,IO）完全没收到tick

**这不是代码bug**，而是模拟环境的行为特征。在真实交易时段（09:30~15:00），所有订阅的合约都会收到tick推送。

### 5.6 chain_coverage_audit中的关系

```
config_count = 16288          ← 配置全集
subscription_confirmed_count = 16288  ← 订阅成功数
tick_return_count = 597       ← 真实tick回调合约数
ticks_raw_count = 16288       ← ticks_raw总覆盖合约数
simulated_coverage_tick_count = 15691 ← 模拟覆盖合约数
active_tick_count = 597       ← 活跃tick合约数

等式: tick_return(597) + simulated_coverage(15691) = ticks_raw(16288) = config(16288)
```
