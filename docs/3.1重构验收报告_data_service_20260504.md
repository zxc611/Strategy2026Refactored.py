## 一、修复项清单与逐项核查

### 1. [P0] 消除裸except (原L2443)

| 项目 | 详情 |
|------|------|
| **问题** | `close()`方法中裸`except:`吞掉所有异常，连接关闭失败无感知 |
| **原代码** | `except: pass` |
| **修复代码** | `except Exception as e: logger.warning(f"Failed to close thread-local connection: {e}")` |
| **功能影响** | 无。原行为是静默忽略，现改为记录warning日志后同样继续执行，不改变控制流 |
| **验收结果** | **PASS** - 异常不再被吞掉，日志可追溯，控制流不变 |

### 2. [P0] 消除6处`except Exception: pass` (原L630/633/635/639/641/643)

| 项目 | 详情 |
|------|------|
| **问题** | 6处DuckDB配置SET语句异常被`except Exception: pass`静默吞掉 |
| **涉及配置** | `enable_dictionary_compression`, `preserve_insertion_order`, `zstd_compression_level`, `enable_progress_bar`, `enable_http_metadata_cache`, `catalog_error_max_scans` |
| **修复方式** | 全部改为`except Exception as e: logger.warning(f"Failed to set {config_name}: {e}")` |
| **功能影响** | 无。配置设置失败时原行为是静默跳过，现改为记录warning后同样继续，不改变控制流 |
| **验收结果** | **PASS** - 6处全部修复，异常有日志可追溯，DuckDB配置行为不变 |

### 3. [P0] `_all_connections`加锁保护

| 项目 | 详情 |
|------|------|
| **问题** | `_all_connections`列表无锁保护，多线程并发append/remove存在竞态条件 |
| **修复方式** | (1) 新增`self._all_connections_lock = threading.RLock()` (2) 所有append/remove/遍历/clear操作均在`with self._all_connections_lock`保护下执行 |
| **修改点** | `_get_connection()`: 2处append + 1处remove加锁; `close_all()`: 遍历+clear改为锁内快照后锁外关闭 |
| **功能影响** | 无。锁只增加并发安全性，不改变单线程行为。使用RLock保证可重入，与DataService其他RLock风格一致 |
| **验收结果** | **PASS** - 并发安全，无死锁风险(RLock可重入)，close_all先快照再关闭避免持锁时间过长 |

### 4. [P1] 消除硬编码路径 (原L50/60)

| 项目 | 详情 |
|------|------|
| **问题** | 硬编码`D:\ali2026_data\ticks.duckdb`和`D:\ali2026_data\ticks.parquet`，跨环境不可移植 |
| **修复方式** | 新增`_get_default_data_dir()`函数，优先级: (1)`ALI2026_DATA_DIR`环境变量 (2)`APPDATA/InfiniTrader_SimulationX64/ali2026_data` (3)用户主目录fallback |
| **原路径** | `D:\ali2026_data\ticks.duckdb` → `os.path.join(_get_default_data_dir(), 'ticks.duckdb')` |
| **功能影响** | 兼容。当`ALI2026_DATA_DIR=D:\ali2026_data`时行为完全等价；无环境变量时使用APPDATA目录，更符合Windows规范 |
| **验收结果** | **PASS** - 设置`ALI2026_DATA_DIR=D:\ali2026_data`可还原原行为，新默认路径合理 |

### 5. [P0] `_get_data_paths_config`异常处理改进 (原L39)

| 项目 | 详情 |
|------|------|
| **问题** | `except Exception: return None`吞掉所有异常，包括配置错误、属性错误等 |
| **修复方式** | 拆分为: (1)`except ImportError` → debug日志(模块未安装属正常情况) (2)`except Exception as e` → warning日志+返回None(异常可追溯) |
| **功能影响** | 无。返回值不变(None)，但异常现在有日志记录，便于排查配置问题 |
| **验收结果** | **PASS** - 返回值语义不变，异常分级处理(ImportError用debug,其他用warning) |
