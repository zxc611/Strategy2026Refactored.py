### 2.1 架构对比

```

│              当前 task_scheduler.py 回测链路                    │
│                                                                 │
│  minute_data ──→ run_backtest() ──→ _check_state_transition()  │
│                      │                    (简化版五态判定)      │
│                      ├──→ _check_safety()                      │
│                      │      (简化版熔断/日内止损)              │
│                      ├──→ _try_open()                          │
│                      │      (硬编码方向+TP/SL)                 │
│                      └──→ _check_positions()                   │

└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐

│                                                                 │
│  minute_data ──→ StateParamManager.update_state_from_width_cache│
│       │              (真实五态诊断+确认机制)                     │

│       └──→ PositionService._get_tp_sl_ratios_by_reason()        │
│              (YAML优先→硬编码兜底，6种差异化TP/SL)             │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 差距明细

| 功能模块 | 当前 `run_backtest` | 真实 V7 策略 | 差距等级 |
|----------|---------------------|-------------|----------|
| **状态诊断** | `_check_state_transition()` 简化版，仅用 `correct_*_pct` 求和对比阈值 | `StateParamManager.update_state_from_width_cache()` 完整五态分类+确认bars+最小持有时长 | ⚠️ 中 |
| **开仓理由映射** | 硬编码 `_STATE_REASON_MAP`：3种映射 | `_resolve_open_reason()` 支持 4 种状态→6 种理由编码 | ⚠️ 中 |
| **TP/SL 差异化** | `_resolve_tp_sl()` 所有理由相同 TP/SL | `_get_tp_sl_ratios_by_reason()` YAML优先→硬编码兜底，6组差异化值 | 🔴 高 |
| **信号生成** | 简单 `strength > 0.3` 阈值 + `imbalance` 方向 | `t_type_service.select_otm_targets_by_volume()` 多目标排序 | 🔴 高 |
| **希腊字母风控** | 无 | `risk_service._check_greeks_limits()` delta/gamma/vega/theta 限额 | 🔴 高 |
| **熔断机制** | 简化版单笔亏损后暂停 | `SafetyMetaLayer` 多层熔断+日内累计止损+冷静期 | ⚠️ 中 |
| **弹簧策略** | 无 | `BoxSpringStrategy.detect_spring()` 波动率脉冲四条件 | 🔴 高 |
| **影子策略** | 无 | `ShadowStrategyEngine` alpha衰减监控 | 🟡 低(回测可不集成) |

**结论**：当前 `run_backtest` 是一套**功能性的骨架回测**，能跑通参数扫描流程并产出夏普/回撤/收益指标，但它**没有使用真实的策略逻辑**。需要将3个核心决策点接入真实模块：**状态路由 → 信号生成 → 差异化止盈止损**。

## 三、共振策略真实逻辑部署方案

### 3.1 整体策略

**核心原则：不修改真实策略模块代码，仅在 `task_scheduler.py` 中导入并调用其接口。**

选择性地接入以下模块即可覆盖回测场景的关键决策逻辑：

```
导入模块                    用途
─────────────────────────────────────────
state_param_manager    → 状态诊断 + 参数路由
position_service       → TP/SL 差异化（静态方法可用）
risk_service           → 希腊字母限额 + 熔断风控
box_spring_strategy    → 弹簧策略信号检测
```

**不需要（也不应该）在回测中初始化的模块**：
- `SignalService`（通过 EventBus 发布事件的实盘信号服务，回测中信号由数据直接驱动）
- `OrderService`（下单到交易所，回测中不需要）
- `StrategyCoreService`（完整的 Facade 生命周期管理器，回测中不需要）
- `ShadowStrategyEngine`（影子策略监控，回测中非必要）

### 3.2 状态诊断接入方案

**现状**（[task_scheduler.py:L142-L175](file:///C:/Users/xu/AppData/Roaming/InfiniTrader_SimulationX64/pyStrategy/demo/ali2026v3_trading/task_scheduler.py#L142-L175)）：

```python
def _check_state_transition(bt, bar, params):
    non_other = bar.correct_rise_pct + bar.correct_fall_pct + bar.wrong_rise_pct + bar.wrong_fall_pct
    threshold = params.get("non_other_ratio_threshold", 0.4)
    if non_other < threshold:
        candidate = "other"
    else:
        correct = bar.correct_rise_pct + bar.correct_fall_pct
        incorrect = bar.wrong_rise_pct + bar.wrong_fall_pct
        candidate = "correct_trending" if correct >= incorrect else "incorrect_reversal"
    # 确认bars机制
```

**接入方案**：复用 `StateParamManager` 的五态聚合逻辑（从 `_STATE_MAP` 映射到三态），但将数据源从 `width_cache` 替换为 `minute_data` 的列值。有两种方案：

**方案A（推荐）：保持当前简化版，但使用真实的参数路由值**

`StateParamManager` 需要实盘 `width_cache` 对象才能工作（调用 `get_status_counts_snapshot()`），但 backtest 中没有这个对象。因此**不直接实例化 StateParamManager**，而是：

1. 保持当前的 `_check_state_transition()` 状态诊断逻辑
2. 新增一个函数 `_resolve_params_from_state()`，模拟 `StateParamManager.get_params()` 的行为，从 `state_param_sets.yaml` 加载对应状态的参数
3. 使用 `position_service._get_tp_sl_ratios_by_reason()` 获取真正的差异化 TP/SL

**方案B（更彻底但复杂度高）**：

重新实现一个 `_BacktestStateParamAdapter`，它实现了与 `StateParamManager` 相同的五态→三态聚合逻辑，但直接从 `minute_data` 列读取数据。这样可以保证回测中的状态诊断逻辑与实盘完全一致。

**推荐方案A**，因为：
- 五态列（`correct_rise_pct` 等）已经在预处理阶段计算好了，语义等价
- 接入复杂度低，改动范围可控
- 核心差异（TP/SL 差异化）可以通过调用 `_get_tp_sl_ratios_by_reason()` 解决

### 3.3 差异化 TP/SL 接入方案

**现状**（[task_scheduler.py:L136-L139](file:///C:/Users/xu/AppData/Roaming/InfiniTrader_SimulationX64/pyStrategy/demo/ali2026v3_trading/task_scheduler.py#L136-L139)）：

```python
def _resolve_tp_sl(params, open_reason):
    tp = params.get("close_take_profit_ratio", 1.5)
    sl = params.get("close_stop_loss_ratio", 0.5)
    return (tp, sl)
```

**接入方案**：直接调用 `PositionService._get_tp_sl_ratios_by_reason()` 静态方法。

该方法已经有完整的三层优先级路由：
- P3: `state_param_sets.yaml` 中的 reason 级参数（如 `CORRECT_RESONANCE`, `BOX_SPRING` 等）
- P2: 硬编码 `_HARDCODED_REASON_DEFAULTS`（6组差异化值）
- P1: `_FALLBACK_TP_SL` 兜底 (1.5, 0.50)

```python
# 改造后的 _resolve_tp_sl()
def _resolve_tp_sl(params, open_reason):
    from ali2026v3_trading.position_service import PositionService
    return PositionService._get_tp_sl_ratios_by_reason.__func__(None, open_reason)
```

**注意事项**：
- 方法内部会调用 `get_state_param_manager()` 读取 YAML（P3 优先级）
- `state_param_sets.yaml` 需要在回测工作目录可访问
- 如果 YAML 不可用，自动降级到硬编码值（不会崩溃）

### 3.4 弹簧策略接入方案

**现状**：`run_backtest` 完全不支持弹簧策略。

**接入方案**：

1. 新增 `_try_spring_open()` 函数，调用 `BoxSpringStrategy` 的核心检测逻辑
2. 弹簧信号需要 per-bar 的数据：`iv`（隐含波动率）、`strike_price`、`expire_date`、`option_type`、`close`
3. 当 `bt.current_state == "other"` 时，额外执行弹簧信号检测

简化接入的最小集：
```python
def _try_spring_open(bt, bar, params):
    if bt.current_state != "other":
        return
    # 检测弹簧条件
    iv = bar.get("iv", 0)
    if iv <= 0:
        return
    # ... 四条件检查（箱体触摸、IV百分位、距到期天数、权利金占比）
```

**更推荐的方案**：由于 `BoxSpringStrategy` 需要维护内部状态（`_iv_history`, `_box_ranges`），建议在 `_BacktestState` 中增加弹簧策略的状态字段，逐 bar 更新后调用 `detect_spring()`。

### 3.5 希腊字母风控接入方案

**现状**：回测中没有希腊字母风控检查。

**接入方案**（可选，建议放在 `_check_safety()` 中）：

```python
def _check_greeks_limits(bt, bar, params):
    delta = bar.get("delta", 0)
    gamma = bar.get("gamma", 0)
    vega = bar.get("vega", 0)
    # 希腊字母限额检查
    if abs(delta) > 0.8:   # delta 暴露过大
        return False
    if abs(gamma) > 0.05:  # gamma 风险过高
        return False
    return True
```

**优先级**：希腊字母限额的阈值应来自 `state_param_sets.yaml` 或 `config_params.py`，而不是硬编码。

### 4.2 问题分析

当前 Round1/Round2 分离策略存在以下问题：

1. **Round1 只扫 6 个参数**（972 组合），其余 10 个参数固定为默认值
2. **Round1 的 Top-K 只固定了 Round1 的 6 个核心参数**，传给 Round2 时 `fixed_core` 仅包含 `PARAM_GRID_ROUND1` 的 key
3. **Round2 虽然是"精扫"**，但 `spring_*` / `capital_route_*` / `shadow_*` 等子系统的搜索范围只有 2 个值，粒度较粗
4. **缺少中级扫描层**：Round1(粗)→Round2(精) 之间跳跃太大

### 4.3 改进方案：三层分层网格

```
Round1 (粗扫): 核心交易参数，宽范围
    ↓ Top-20
Round2 (中扫): 安全+限频参数 + Round1固定核心
    ↓ Top-10
Round3 (精扫): 子系统参数 + Round1+Round2固定核心
```

## 六、风险与注意事项

### 6.1 多进程环境兼容性

`task_scheduler.py` 使用 `ProcessPoolExecutor` + 全局变量共享数据（[L435-L437](file:///C:/Users/xu/AppData/Roaming/InfiniTrader_SimulationX64/pyStrategy/demo/ali2026v3_trading/task_scheduler.py#L435-L437)）。这意味着：

1. 每个子进程中必须能独立 `import` 真实策略模块
2. `state_param_sets.yaml` 必须在正确的相对路径中
3. `duckdb` 连接必须各自独立（`read_only=True`）

### 6.2 路径依赖

如果从 `task_scheduler.py` 的路径 `import` 真实模块，需要确保：
- `ali2026v3_trading` 包可被导入
- `state_param_sets.yaml` 的路径解析正确（当前使用 `__file__` 相对路径，在子进程中可能失效）

### 6.3 性能影响

接入真实模块导入会增加子进程启动时间（~1-2 秒），但不会影响每个 bar 循环的执行效率（核心调用都是 Python 方法调用）。TP/SL 差异化和参数路由主要影响的是数值计算，不增加 IO 开销。
