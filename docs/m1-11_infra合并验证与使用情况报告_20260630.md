### 7.3 关键发现：import引用 vs 运行时活跃度差异

| 模块 | 代码引用次数 | 6/28日志 | 6/29日志 | 差异说明 |
|---|---|---|---|---|
| `shared_utils` | 302 | 0 | 0 | 最广泛引用的工具库，但作为底层函数库不产生日志 |
| `_helpers` | 154 | 0 | 0 | 合并后最广泛引用，含get_logger/DiskSpaceMonitor等，但6/28-29无磁盘告警 |
| `subscription_service` | 56 | 6753 | 1535 | 引用数中等，但运行时最活跃（持续行情订阅） |
| `health_monitor` | 68 | 3059 | 309 | 引用数中等，运行时高活跃（周期性诊断） |
| `service_container` | 17 | 300 | 176 | 引用数低，但每次启动都产生大量日志 |
| `resilience` | 92 | 3 | 2 | 引用数高，但6/28-29仅3/2次安全事件输出 |
| `serialization_utils` | 102 | 0 | 0 | 引用数高，纯工具函数不产生日志 |
| `security` | 66 | 0 | 0 | 引用数高，6/28-29无安全事件触发 |
| `commission_utils` | 27 | 0 | 0 | 费用计算，被调用但不产生日志 |
| `exceptions` | 8 | 0 | 0 | 6/28-29无L0/L1级异常触发 |

**结论**: 
- **代码引用次数**衡量的是模块的"静态依赖广度"——多少文件import了它
- **日志命中次数**衡量的是模块的"运行时活跃度"——实际执行时产生了多少可观测输出
- 两者差异巨大：`shared_utils`引用302次但日志0命中（纯工具函数），`subscription_service`引用56次但6/28日志6753次（持续运行的核心服务）
- 21/29个infra模块在6月28-29日均为静默状态，说明大部分infra模块属于"被动工具"或"事件驱动"类型，只在特定条件下才产生运行时痕迹
- 两天活跃模块完全一致（8个），活跃度排序也一致，说明运行时模式稳定
