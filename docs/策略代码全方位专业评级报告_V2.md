## 一、架构设计评审 (B+, 7.0/10)

### 1.1 架构亮点

- **Facade + 委托模式**: 通过`__getattr__`实现服务委托，`OrderService`作为Facade将订单处理委托给6+子服务（`OrderExecutor`、`OrderStateManager`、`OrderRiskGuard`等），成功将千行`send_order`方法拆解为三阶段流水线（前置校验→平台下单→后置持久化）。
- **三阶段订单流水线**: `_pre_send_checks` → `_execute_platform_insert` → `_post_send_persist`，职责清晰，可独立测试。
- **桥接解耦**: `RiskPositionBridge` / `PositionBridge` 协议接口成功打破risk ↔ position的循环依赖。
- **模块化拆分**: 原超大文件拆分为多个职责单一的模块（如`data_service.py`拆分为5个Mixin+RealTimeCache），大部分模块控制在300-800行。

### 1.2 架构严重缺陷

#### 缺陷1: 双副本代码不一致（P0）

[`strategy_2026.py`](file:///c:/Users/xu/AppData/Roaming/InfiniTrader_QhZijintianfengPythonX64/pyStrategy/demo/ali2026v3_trading/strategy_2026.py) 和 [`strategy/strategy_2026.py`](file:///c:/Users/xu/AppData/Roaming/InfiniTrader_QhZijintianfengPythonX64/pyStrategy/demo/ali2026v3_trading/strategy/strategy_2026.py) 存在两个副本，且`strategy/`子目录版本包含关键修复（`FIX-20260706-STATE-FORCE`、`FIX-20260707-PAUSE-SAFE`），根目录版本缺少这些修复。同样的双副本问题存在于`strategy_monitoring_layer.py`、`position_check_service.py`、`strategy_config_layer.py`、`lifecycle_callbacks.py`等文件。

**影响**: 代码修改可能只在其中一个副本生效，导致修复"时好时坏"。这是Fix 18-20声称已修复但实际无效的根因之一。

**建议**: 统一为一个源文件，另一个改为`from strategy.xxx import *`的re-export。

#### 缺陷2: `__getattr__`委托链脆弱

多个核心类使用`__getattr__`将方法调用委托给内部服务对象（`StrategyCoreService` → `Strategy2026` → `BoxSpringStrategy` → `BoxSpringDetectorService` / `BoxSpringExecutorService`）。这种多层委托在调试时极度困难，且容易导致无限递归（已在`strategy/strategy_2026.py`中发现需要`object.__getattribute__`绕过的问题）。

**建议**: 使用显式组合替代`__getattr__`委托，或至少限制委托层数≤2层。

#### 缺陷3: 桩函数/未完成功能

[`box_spring_executor.py`](file:///c:/Users/xu/AppData/Roaming/InfiniTrader_QhZijintianfengPythonX64/pyStrategy/demo/ali2026v3_trading/box_spring_executor.py) 中：
- `_compute_hedge_ratio` 返回固定 1.0
- `_check_cross_strategy_risk` 返回固定 True
- `_record_spring_trade` 返回固定空字典
- `_find_straddle_pair` 无实际配对查找逻辑

这些桩函数在日志中产生大量"对冲比率非零"的INFO日志，但实际未执行任何风险管理。

## 二、风控安全评审 (A-, 8.0/10)

### 2.1 风控亮点

- **5级风控等级**: LOW → MEDIUM → HIGH → RESTRICTED → CRITICAL，RESTRICTED中间状态防止直接熔断。
- **断路器机制**: 连续失败≥5次自动熔断，60秒自动恢复半开，跨日保留需人工`confirm_daily_resume()`。
- **胖手指防护**: 价格偏离3%/20%/50%分级拦截，按`action`类型分级。
- **4重dry_run拦截**: `svc._dry_run_mode` → `_provider_ref._dry_run_active` → `platform_insert_order.__self__._dry_run_active` → `get_cached_params().get('dry_run_mode')`，任一为True即拦截。
- **fail-safe设计**: 所有风控异常默认返回阻断结果，宁可误杀不可漏过。
- **自成交禁止期**: 30分钟禁止期，CLOSE订单豁免，拆单原子性检测。

### 2.2 风控缺陷

#### 缺陷1: dry_run拦截反复出现逻辑漏洞（P0）

**历史**:
- V1: `_dry_run_mode=False is not None` → 跳过所有回退 → 122次订单穿透到平台
- V2: 修复为4级回退，`if getattr(svc, '_dry_run_mode', False)` 替代 `is None`
- **但**: `init_order_service_attrs()`中`get_params_service().get_bool('dry_run_mode', False)`可能返回False（ParamsService未加载dry_run_mode键），导致V2的四重回退中检查1失败。虽然检查2-4可兜底，但依赖`lifecycle_callbacks`正确设置`_dry_run_active`。

**建议**: 在`init_order_service_attrs`中直接读取配置文件而非依赖ParamsService，或在ParamsService中强制加载dry_run_mode。

#### 缺陷2: 92次"风控阻断"但0次真实风险

报告显示Session 2中563次风控阻断（`hard_stop_triggered`），但实际是因为`hard_stop`跨日保留，而非真实风险事件。这种"过度阻断"会掩盖真正的风险告警。

#### 缺陷3: 影子策略隔离未验证

`_shadow_strategy_signal.py`中的`verify_shadow_isolation()`声称验证影子策略与主策略完全隔离，但该函数仅在测试中调用，生产环境无验证。

## 三、策略有效性评审 (D+, 3.5/10)

### 3.1 核心发现：策略胜率0%

7/8 22:06重启后46笔交易（37笔开仓+46笔平仓，含9笔遗留持仓平仓），**0笔盈利，胜率0%**。所有亏损均为系统性买卖价差成本。

#### 亏损根因分析

```
开仓(BUY期权) → 5分钟时间止损 → 强制平仓(SELL) → 支付买卖价差(~1点)
                                    ↓
                    止盈/止损/浮动止盈从未触发（持仓时间过短）
```

**核心问题**: 5分钟时间止损（`_check_time_stop`）设置为`max_hold_minutes=5`，91.3%的平仓在5分钟内发生。期权5分钟内价格波动通常<1个点，无法覆盖买卖价差（约1点），策略等同于"每5分钟稳定支付1点价差成本"。

**详细数据**:

| 盈亏区间 | 笔数 | 占比 |
|---------|------|------|
| 盈利 (>0) | 0 | 0% |
| 持平 (=0) | 10 | 27% |
| 亏损 (-1.0) | 33 | 89% |
| 亏损 (-1.2~-1.6) | 4 | 11% |
