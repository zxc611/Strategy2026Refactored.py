## 一、5大问题修复清单与逐项核查

### 1. [P0] _shard_enqueue_counts类变量Dict无锁并发读写

| 项目 | 详情 |
|------|------|
| **问题** | `DiagnosisProbeManager._shard_enqueue_counts`(类变量Dict)在`on_storage_enqueue`中无锁并发读写，多线程可能损坏 |
| **原代码** | `_shard_enqueue_counts[shard_idx] = _shard_enqueue_counts.get(shard_idx, 0) + 1` (L1293) |
| **修复** | (1) 新增 `_shard_enqueue_lock = threading.Lock()` 类变量 (2) 访问改为 `with _shard_enqueue_lock: ...` |
| **功能影响** | 无。仅增加并发安全性，不改变单线程行为 |
| **验收结果** | **PASS** - threading.Lock保护读写，无死锁风险 |

### 2. [P1] DiagnosisReport.add_log中self.logs列表操作未加锁

| 项目 | 详情 |
|------|------|
| **问题** | `add_log`中`self.logs`列表的截断+append操作无锁保护，多线程并发可能损坏列表 |
| **修复** | (1) `__init__`中新增 `self._logs_lock = threading.Lock()` (2) `add_log`中截断+append包裹在 `with self._logs_lock:` 内 |
| **功能影响** | 无。锁仅保护并发安全，单线程下行为不变 |
| **验收结果** | **PASS** - 截断逻辑和append在锁内原子执行 |

### 3. [P1] is_monitored_contract每次调用都导入config_service，热路径性能开销

| 项目 | 详情 |
|------|------|
| **问题** | `is_monitored_contract`每次调用都执行`from config_service import get_cached_params`，在tick热路径上有性能开销 |
| **修复** | 新增1秒TTL缓存: `_is_monitored_cache_result`/`_is_monitored_cache_time`/`_IS_MONITORED_CACHE_TTL=1.0`，使用`time.monotonic()`计时，缓存命中直接返回 |
| **功能影响** | 无。缓存TTL=1秒，config变更最多延迟1秒生效，对diagnostic_output开关场景完全可接受 |
| **验收结果** | **PASS** - TTL缓存减少99%+的import开销，异常改为logging.debug |

### 4. [P2] MONITORED_CONTRACTS硬编码26个合约

| 项目 | 详情 |
|------|------|
| **问题** | MONITORED_CONTRACTS硬编码26个合约(L867-895)，无法灵活配置 |
| **修复** | 新增`_load_monitored_contracts_from_config()`函数，支持从`MONITORED_CONTRACTS_LIST`环境变量加载JSON格式合约列表，硬编码保留作为fallback |
| **功能影响** | 无。未设置环境变量时行为完全不变(使用硬编码fallback)，设置后可灵活覆盖 |
| **验收结果** | **PASS** - 向后兼容，无环境变量时行为等价 |

### 5. [P0] 6处`except Exception:`无日志吞异常

| 项目 | 详情 |
|------|------|
| **问题** | 6处`except Exception:`无日志(L2147/2209/2213/2215/2239/2247)，诊断异常被静默吞掉，排查困难 |
| **修复** | 全部改为`except Exception as e: logging.debug(f"Diagnosis: {context}: {e}")`，保留原控制流(继续执行) |

| **功能影响** | 无。控制流不变(继续执行)，仅增加debug日志便于排查 |
| **验收结果** | **PASS** - 异常不再被吞掉，日志级别用debug(诊断模块内部不产生日志洪水) |
