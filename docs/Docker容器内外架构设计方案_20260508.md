# Docker容器内外架构设计方案

**版本**: v1.0  
**日期**: 2026-05-08  
**包名**: ali2026v3_trading (42个模块)  

---

## 一、设计目标

| 目标 | 说明 |
|------|------|
| 🔐 **知识产权保护** | 核心策略算法（T型图排序、宽度计算、信号模型）封装在容器内，不可逆向 |
| 🔧 **运维便利性** | 配置调参、数据存储、诊断监控在容器外，无需重新构建镜像即可调整 |
| ⚡ **低延迟通信** | Tick热路径延迟控制在毫秒级，不影响策略实时性 |
| 🔌 **松耦合接口** | 容器内外通过明确定义的API契约通信，可独立升级部署 |

## 二、当前架构通信模式分析

### 2.1 现有通信模式（重构前）

```
┌──────────────────────────────────────────────────────────┐
│              当前：单一进程内直接导入                         │
│                                                            │
│  main.py ──→ strategy_core_service.py                      │
│                  │ MRO继承                                 │
│                  ├── strategy_lifecycle_mixin.py            │
│                  ├── strategy_instrument_mixin.py           │
│                  ├── strategy_historical.py                 │
│                  └── strategy_tick_handler.py               │
│                      │                                     │
│                      ├── t_type_service.py                  │
│                      │     └── width_cache.py               │
│                      ├── position_service.py                │
│                      │     └── risk_service.py              │
│                      ├── order_service.py                   │
│                      │     └── event_bus.py                 │
│                      └── signal_service.py                  │
│                                                            │
│  所有通信：from ali2026v3_trading.xxx import XXX             │
│  事件总线：EventBus.publish/subscribe (同进程异步)            │
│  服务容器：ServiceContainer (同进程IoC)                      │
│  单例模式：get_xxx_service() (模块级全局变量)                 │
└──────────────────────────────────────────────────────────┘
```

### 2.2 现有通信方式的局限性

| 方式 | 当前用途 | 跨容器障碍 |
|------|---------|-----------|
| `from ... import` | 模块间直接引用 | 跨进程无法直接导入 |
| `EventBus` | 异步事件分发 | 线程池在同一进程内 |
| `ServiceContainer` | IoC服务注册 | 内存中的字典，不能跨进程 |
| `get_xxx_service()` | 单例工厂 | 全局变量在进程间不共享 |
| 平台API绑定 (`self.subscribe`, `self.get_kline`) | 运行时注入 | 回调函数不能跨进程传递 |
| `threading.Lock` / `threading.RLock` | 线程同步 | 不能跨进程加锁 |

### 2.3 关键数据流识别

| 数据流 | 路径 | 频率 | 延迟要求 |
|--------|------|------|---------|
| **Tick入口** | 平台 → on_tick() → _process_tick() → t_type → width_cache | 数千/秒 | <1ms |
| **订单出口** | width_cache → OrderService → 平台 send_order() | 数十/秒 | <10ms |
| **配置读取** | config_params → strategy_core | 启动时大量/运行时低频 | 秒级 |
| **历史K线** | platform.get_kline → strategy_historical | 启动时/定时 | 秒级 |
| **事件通知** | EventBus.publish → subscribers | 低频 | 百毫秒 |
| **持仓同步** | platform.on_trade → position_service | 数十/秒 | <100ms |
| **诊断输出** | diagnosis_probe → 日志 | 持续 | 无 |
| **数据持久化** | RealTimeCache → DuckDB | 数千/秒 | 异步 |

### 5.2 共享内存Tick格式（热路径专用）

```python
"""
共享内存Tick通道格式定义
──────────────────────────────
格式: 固定长度二进制帧，无序列化开销
"""

class TickFrame:
    """76字节二进制帧"""
    
    MAGIC: int = 0x5449434B  # "TICK" ASCII
    SIZE: int = 76
    
    # 字段偏移量定义
    FIELDS = {
        'magic':      (0,  'I'),   # uint32, 魔数校验
        'length':     (4,  'I'),   # uint32, 帧长度=76
        'timestamp':  (8,  'Q'),   # uint64, Unix ms
        'instrument': (16, '24s'), # char[24], 合约ID (如 'IF2605.CFFEX')
        'price':      (40, 'd'),   # float64, 最新价
        'volume':     (48, 'Q'),   # uint64, 累积成交量
        'open_int':   (56, 'd'),   # float64, 持仓量
        'exchange':   (64, '8s'),  # char[8], 交易所 (如 'CFFEX')
        'flags':      (72, 'I'),   # uint32, 位标志
    }
    
    # flags 位定义
    FLAG_IS_OPTION = 0x01      # bit0: 期权/期货
    FLAG_IS_FIRST   = 0x02     # bit1: 首个tick
    FLAG_HAS_BIDASK = 0x04     # bit2: 含买卖报价（扩展帧）
```

## 六、关键数据流设计

### 6.1 Tick热路径（容器外 → 容器内，<100μs）

```
容器外                                   容器内
───────────────────────────────────────────────────────────
平台 on_tick(tick)
    │
    ▼
tick_handler.extract_fields()
    │ 提取: inst_id, price, vol, oi, exchange
    │
    ▼
SharedMemory.Publisher.write(frame)
    │ 固定76字节，无序列化，无锁RingBuffer
    │
    ▼
    ═══════════ 共享内存通道 ═══════════
    │
    ▼
SharedMemory.Consumer.read(frame)
    │
    ▼
strategy_tick_handler.on_tick(frame)
    │ 重建tick对象
    │
    ▼
strategy_tick_handler._process_tick()
    │
    ├── width_cache.update()       ← 宽度计算
    ├── t_type_service.ingest()    ← T型图更新
    ├── position_service.check()   ← 持仓检查
    ├── risk_service.evaluate()    ← 风控评估
    └── signal_service.maybe_emit()← 信号判断

关键设计:
- 共享内存 RingBuffer 大小: 1MB (可容纳 ~13,000 tick帧)
- 无锁 CAS 读写，单生产者(容器外) 单消费者(容器内)
- 溢出策略: 丢弃最旧数据 + 计数
```

### 6.2 订单提交路径（容器内 → 容器外，<5ms）

```
容器内                                   容器外
───────────────────────────────────────────────────────────
strategy_tick_handler/width_cache
产生交易信号
    │
    ▼
```

## 九、安全加固措施

| 层级 | 措施 | 说明 |
|------|------|------|
| **文件系统** | `read_only: true` | 容器内不可写入，防止篡改 |
| **系统权限** | `cap_drop: [ALL]` | 移除所有Linux capabilities |
| **提权防护** | `no-new-privileges: true` | 禁止setuid等提权操作 |
| **网络隔离** | `internal: true` | 容器间网络不暴露外部 |
| **源码保护** | 编译.pyc + 删除.py | 只保留字节码，增加逆向难度 |
| **敏感参数** | 环境变量注入，不写代码 | API key等通过环境变量传入 |
| **审计日志** | 全部gRPC调用记录 | trace_id贯穿全链路 |
| **资源限制** | cgroup CPU/Memory限制 | 防止策略容器资源耗尽影响平台 |

## 十、风险矩阵与缓解

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| **共享内存通道故障** | 🔴高 | 失败自动回退到gRPC Tick通道（降级模式），延迟略增 |
| **gRPC连接中断** | 🟡中 | 自动重连（指数退避），本地缓存最后配置 |
| **容器冷启动延迟** | 🟡中 | 预热机制：启动即加载历史K线到共享内存 |
| **通信序列化开销** | 🟢低 | 共享内存无序列化；gRPC用Unix Socket零拷贝 |
| **容器外修改影响策略** | 🟡中 | 配置变更需版本号校验，不兼容版本拒绝启动 |
| **DuckDB文件损坏** | 🟡中 | WAL模式自动恢复，定期备份（容器外独立） |
| **日志丢失（容器内）** | 🟢低 | Unix Socket写入容器外日志文件，容器内不落盘 |
| **平台API变更** | 🟡中 | 容器外适配层隔离，容器内只依赖Protobuf契约 |
