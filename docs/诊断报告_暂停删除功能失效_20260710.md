# 暂停/恢复/删除功能失效 — 完整诊断与修复报告

**日期**: 2026-07-10 ~ 2026-07-11  
**版本**: v4.0（新增运行时功能验证）  
**范围**: 暂停(pause)/恢复(resume)/删除(destroy)功能全链路诊断与修复  
**关键事实**: 暂停/删除功能曾经正常工作，在V3组合模式重构后失效

---

## 一、核心根因

### 1.1 根因一：t_type_bootstrap.py 方法委托缺失

`t_type_bootstrap.py` 是平台实际加载的策略入口类，使用组合模式委托给 `Strategy2026` 实例。但**只委托了6个方法**（on_init/on_start/on_stop/onTick/onOrder/onTrade），**遗漏了 pause/resume/destroy 等生命周期方法**。

**平台调用链路（修复前）**：
```
C++ 宿主 → 用户点击"暂停" → 调用 t_type_bootstrap.pause() [不存在]
  → BaseStrategy.pause() [空操作]
  → LifecycleCallbacks.pause() 从未被调用
  → 四元状态从未同步到 PAUSED
  → 策略实际仍在运行
```

**演变历史**：

| 版本 | 变更 | 暂停/删除状态 |
|------|------|---------------|
| V0 | `__new__`代理到Strategy2026 | ✅ 正常 |
| V1 | Strategy2026定义移入t_type_bootstrap（TEST_LEVEL渐进式） | ⚠️ 部分正常 |
| V2 | 简化为直接import | ❌ DLL崩溃 |
| V3 | 组合模式，延迟创建实例 | ❌ **遗漏pause/resume/destroy委托** |

### 1.2 根因二：pause()空壳 — 不停止运行中的工作

`LifecycleCallbacks.pause()` 只设状态变量（`_is_paused=True`），但不停止任何运行中的工作：

| 层级 | 问题 | 影响 |
|------|------|------|
| L1 | pause()不调用pause_scheduler() | APScheduler定时任务继续执行 |
| L2 | `_cancel_all_timers`委托链断裂 | Timer无法取消 |
| L3 | `_state_checker`未绑定 | `_can_run_jobs()`永远返回True |
| L4 | onTick不检查`_is_paused` | 策略决策继续执行 |

### 1.3 根因三：暂停指令被异步线程覆盖

| 覆盖路径 | 文件:行号 | 机制 |
|----------|-----------|------|
| subscribe-retry线程 | `lifecycle_callbacks.py:381-402` | DEGRADED→RUNNING恢复时竞态覆盖 |
| exit_parallel_running() | `lifecycle_parallel.py:195` | 退出并行运行期时无条件转RUNNING |
| transition_to(RUNNING)无保护 | `lifecycle_service.py:201` | 任何路径都可覆盖_is_paused |

## 三、核查发现的BUG及修复

### 3.1 BUG: historical_data_manager.py:327 `mgr_self._provider` 不存在

**严重度**: P0（暂停检查永远不生效）

**问题**: `mgr_self` 是 `HistoricalDataManager` 实例，其 `__init__` 仅初始化 `_state_store` 和 `_callback_group`，**不存在 `_provider` 属性**。`getattr(mgr_self._provider, '_is_paused', False)` 会抛出 `AttributeError`，导致重试线程崩溃，`_is_paused` 检查永远不会生效。

**修复**: 改为 `mgr_self._state_store.get('_is_paused', False)` 和 `mgr_self._state_store.get('_destroyed', False)`，与同文件L219-224的 `stop_check` lambda 写法一致。

### 3.2 封装性: TickProcessingService 缺少 on_stop() 方法

**严重度**: P2（功能可工作但封装不完整）

**问题**: `_flush_stop_requested` 标志由 `lifecycle_callbacks.py` 直接操作内部属性设置，`TickProcessingService` 无自身 `on_stop()` 方法。

**修复**: 
- 在 `TickProcessingService` 中添加 `on_stop()` 方法，封装 `_flush_stop_requested = True` 设置 + `_flush_fallback_thread.join()`
- `lifecycle_callbacks.py` 改为调用 `_tick_svc.on_stop()` 而非直接操作内部属性

## 四、PAUSE-GUARD 机制详解

### 4.1 设计

```
resume()                          transition_to(RUNNING)
   │                                      │
   ├─ _resume_in_progress = True ────────►├─ 检查 _is_paused?
   │                                      │    ├─ No → 正常转换
   │                                      │    └─ Yes → 检查 _resume_in_progress?
   │                                      │         ├─ True → 允许转换
   │                                      │         └─ False → 拒绝，返回False
   ├─ transition_to(RUNNING)              │
   ├─ _resume_in_progress = False ◄───────┘
```

### 4.2 运行时功能验证结果

| 测试项 | 结果 | 说明 |
|--------|------|------|
| 暂停时`transition_to(RUNNING)`被拒绝 | ✅ PASS | PAUSE-GUARD正确拦截 |
| 暂停时`_state`仍为PAUSED | ✅ PASS | 状态未被覆盖 |
| 暂停时`transition_to(PARALLEL_RUNNING)`被拒绝 | ✅ PASS | 并行运行也被拦截 |
| resume路径`transition_to(RUNNING)`通过 | ✅ PASS | `_resume_in_progress=True`放行 |
| resume路径后`_state=RUNNING` | ✅ PASS | 状态正确转换 |
| 暂停时`transition_to(STOPPED)`不被拒绝 | ✅ PASS | 非running目标不拦截 |

### 4.3 防护覆盖的攻击面

| 攻击路径 | 防护层 |
|----------|--------|
| subscribe-retry线程竞态 | L1: `_is_paused`前置检查 → L2: `transition_to`后二次检查 → L3: PAUSE-GUARD兜底 |
| exit_parallel_running()覆盖 | L1: `_is_paused`前置检查 → L2: PAUSE-GUARD兜底 |
| 未来新增的`transition_to(RUNNING)`调用 | L3: PAUSE-GUARD兜底（零代码改动即可防护） |

## 六、线程安全分析

### 6.1 resume()中的锁保护

- `_resume_in_progress`的设置/清除在`p._lock`内完成
- `transition_to()`在锁内调用，PAUSE-GUARD检查与状态变更原子执行
- 异步线程(subscribe-retry)在锁外调用`transition_to`，PAUSE-GUARD仍可正确拦截

### 6.2 竞态窗口

subscribe-retry的竞态窗口已通过**三层防护**消除：
1. 前置`_is_paused`检查（L385）
2. `transition_to`后二次检查（L392-398）
3. `transition_to`内部PAUSE-GUARD（L204-213，兜底）
