# 暂停/恢复/删除功能失效 — 完整诊断与修复报告

**日期**: 2026-07-10 ~ 2026-07-13
**版本**: v7.0（FIX-20260713-PLATFORM-STATE-AUTO: 平台状态自动同步 + 暂停/删除最终修复确认 + 7策略模拟下单统计）
**范围**: 暂停(pause)/恢复(resume)/删除(destroy)功能全链路诊断与修复
**关键事实**: 暂停/删除功能曾经正常工作，在V3组合模式重构后失效。v6.0找到真正根因（auto_start调用路径错误）并彻底修复暂停/删除。v7.0解决平台状态自动同步（无需用户点击执行按钮）并确认7策略模拟下单统计。

## 一、核心根因

### 1.1 根因一：t_type_bootstrap.py 方法委托缺失

`t_type_bootstrap.py` 是平台实际加载的策略入口类，使用组合模式委托给 `Strategy2026` 实例。但**只委托了6个方法**（on_init/on_start/on_stop/onTick/onOrder/onTrade），**遗漏了 pause/resume/destroy 等生命周期方法**。

**平台调用链路（修复前）**：
```
C++ 宿主 → 用户点击"暂停" → 调用 t_type_bootstrap.pause() [不存在]
  → BaseStrategy.pause() [空操作]
  → LifecycleCallbacks.pause() 从未被调用
  → 四元状态从未同步到 PAUSED
  → 策略实际仍在运行
```

**演变历史**：

| 版本 | 变更 | 暂停/删除状态 |
|------|------|---------------|
| V0 | `__new__`代理到Strategy2026 | ✅ 正常 |
| V1 | Strategy2026定义移入t_type_bootstrap（TEST_LEVEL渐进式） | ⚠️ 部分正常 |
| V2 | 简化为直接import | ❌ DLL崩溃 |
| V3 | 组合模式，延迟创建实例 | ❌ **遗漏pause/resume/destroy委托** |

### 1.2 根因二：pause()空壳 — 不停止运行中的工作

`LifecycleCallbacks.pause()` 只设状态变量（`_is_paused=True`），但不停止任何运行中的工作：

| 层级 | 问题 | 影响 |
|------|------|------|
| L1 | pause()不调用pause_scheduler() | APScheduler定时任务继续执行 |
| L2 | `_cancel_all_timers`委托链断裂 | Timer无法取消 |
| L3 | `_state_checker`未绑定 | `_can_run_jobs()`永远返回True |
| L4 | onTick不检查`_is_paused` | 策略决策继续执行 |

### 1.3 根因三：暂停指令被异步线程覆盖

| 覆盖路径 | 文件:行号 | 机制 |
|----------|-----------|------|
| subscribe-retry线程 | `lifecycle_callbacks.py:381-402` | DEGRADED→RUNNING恢复时竞态覆盖 |
| exit_parallel_running() | `lifecycle_parallel.py:195` | 退出并行运行期时无条件转RUNNING |
| transition_to(RUNNING)无保护 | `lifecycle_service.py:201` | 任何路径都可覆盖_is_paused |

## 三、核查发现的BUG及修复

### 3.1 BUG: historical_data_manager.py:327 `mgr_self._provider` 不存在

**严重度**: P0（暂停检查永远不生效）

**问题**: `mgr_self` 是 `HistoricalDataManager` 实例，其 `__init__` 仅初始化 `_state_store` 和 `_callback_group`，**不存在 `_provider` 属性**。`getattr(mgr_self._provider, '_is_paused', False)` 会抛出 `AttributeError`，导致重试线程崩溃，`_is_paused` 检查永远不会生效。

**修复**: 改为 `mgr_self._state_store.get('_is_paused', False)` 和 `mgr_self._state_store.get('_destroyed', False)`，与同文件L219-224的 `stop_check` lambda 写法一致。

### 3.2 封装性: TickProcessingService 缺少 on_stop() 方法

**严重度**: P2（功能可工作但封装不完整）

**问题**: `_flush_stop_requested` 标志由 `lifecycle_callbacks.py` 直接操作内部属性设置，`TickProcessingService` 无自身 `on_stop()` 方法。

**修复**: 
- 在 `TickProcessingService` 中添加 `on_stop()` 方法，封装 `_flush_stop_requested = True` 设置 + `_flush_fallback_thread.join()`
- `lifecycle_callbacks.py` 改为调用 `_tick_svc.on_stop()` 而非直接操作内部属性

## 四、PAUSE-GUARD 机制详解

### 4.1 设计

```
resume()                          transition_to(RUNNING)
   │                                      │
   ├─ _resume_in_progress = True ────────►├─ 检查 _is_paused?
   │                                      │    ├─ No → 正常转换
   │                                      │    └─ Yes → 检查 _resume_in_progress?
   │                                      │         ├─ True → 允许转换
   │                                      │         └─ False → 拒绝，返回False
   ├─ transition_to(RUNNING)              │
   ├─ _resume_in_progress = False ◄───────┘
```

### 4.2 运行时功能验证结果

| 测试项 | 结果 | 说明 |
|--------|------|------|
| 暂停时`transition_to(RUNNING)`被拒绝 | ✅ PASS | PAUSE-GUARD正确拦截 |
| 暂停时`_state`仍为PAUSED | ✅ PASS | 状态未被覆盖 |
| 暂停时`transition_to(PARALLEL_RUNNING)`被拒绝 | ✅ PASS | 并行运行也被拦截 |
| resume路径`transition_to(RUNNING)`通过 | ✅ PASS | `_resume_in_progress=True`放行 |
| resume路径后`_state=RUNNING` | ✅ PASS | 状态正确转换 |
| 暂停时`transition_to(STOPPED)`不被拒绝 | ✅ PASS | 非running目标不拦截 |

### 4.3 防护覆盖的攻击面

| 攻击路径 | 防护层 |
|----------|--------|
| subscribe-retry线程竞态 | L1: `_is_paused`前置检查 → L2: `transition_to`后二次检查 → L3: PAUSE-GUARD兜底 |
| exit_parallel_running()覆盖 | L1: `_is_paused`前置检查 → L2: PAUSE-GUARD兜底 |
| 未来新增的`transition_to(RUNNING)`调用 | L3: PAUSE-GUARD兜底（零代码改动即可防护） |

## 六、线程安全分析

### 6.1 resume()中的锁保护

- `_resume_in_progress`的设置/清除在`p._lock`内完成
- `transition_to()`在锁内调用，PAUSE-GUARD检查与状态变更原子执行
- 异步线程(subscribe-retry)在锁外调用`transition_to`，PAUSE-GUARD仍可正确拦截

### 6.2 竞态窗口

subscribe-retry的竞态窗口已通过**三层防护**消除：
1. 前置`_is_paused`检查（L385）
2. `transition_to`后二次检查（L392-398）
3. `transition_to`内部PAUSE-GUARD（L204-213，兜底）

## 十二、v6.0 修复 — auto_start调用路径错误 + 平台状态同步（2026-07-13）

**背景**: 用户确认暂停/删除功能修复成功，但发现平台状态仍显示"初始化"，需手动点击"执行"按钮才变为"运行中"。经深入排查发现3个此前未覆盖的根因。

### 12.1 根因四：auto_start调用路径错误（P0 — 暂停/删除失效的真正根因）

**文件**: `t_type_bootstrap.py` + `strategy_2026.py`

**根因**: `Strategy2026.on_init()` 中 auto_start 调用 `self.on_start()`，但 `self` 是 **Strategy2026 实例**，调用的是 `Strategy2026.on_start()`，而非 `t_type_bootstrap.on_start()`。

```
修复前的调用链:
C++ → t_type_bootstrap.on_init()
  → _real_strategy.on_init()  (Strategy2026.on_init)
    → auto_start=True → self.on_start()  ← self是Strategy2026!
      → Strategy2026.on_start()  ← 错误! 应调用t_type_bootstrap.on_start()
      → BaseStrategy.on_start() → sub_market_data(strategy_obj=Strategy2026)
    ← t_type_bootstrap.on_start() 从未被调用
    ← t_type_bootstrap.trading 保持 False
  ← C++通过t_type_bootstrap.trading判断状态 → 认为策略未运行
  ← C++拒绝调用on_stop/on_destroy → 暂停/删除无效
```

**之前代码中的同步尝试也有bug**:
```python
# strategy_2026.py 旧代码 (line 626)
_ttb.__dict__['trading'] = True  # 修改的是模块对象属性, 不是实例属性!
```
即使`_ttb_inst`是正确的实例引用，`_ttb.__dict__['trading'] = True`也不会修改`_ttb_inst.trading`。

**修复**: auto_start逻辑从`Strategy2026.on_init()`移到`t_type_bootstrap.on_init()`

```
修复后的调用链:
C++ → t_type_bootstrap.on_init()
  → 强制同步strategy_id到_real_strategy
  → 注入_outer_ref = self
  → _real_strategy.on_init()  (Strategy2026.on_init, 不调auto_start)
  → auto_start=True → self.on_start()  ← self是t_type_bootstrap!
    → t_type_bootstrap.on_start()
      → real.on_start()  (Strategy2026.on_start)
      → self.trading = True  ← C++现在能感知策略已运行
    ← t_type_bootstrap.trading = True
  ← C++通过t_type_bootstrap.trading判断状态 → 认为策略已运行
  ← C++可以调用on_stop/on_destroy → 暂停/删除有效
```

### 12.2 根因五：strategy_id同步条件永远不满足（P0）

**文件**: `t_type_bootstrap.py` on_init()

**根因**: strategy_id同步条件 `getattr(_real, 'strategy_id', 0) == 0` 永远不满足

```python
# Strategy2026.__init__():
_sid = kwargs.get('strategy_id')  # kwargs中不含strategy_id → None
if isinstance(_sid, int):
    self.strategy_id = _sid
elif isinstance(_sid, str) and _sid.isdigit():
    self.strategy_id = int(_sid)
else:
    self.strategy_id = int(time.time())  # ← 默认值, 非零!

# t_type_bootstrap.on_init() 旧代码:
if _real is not None and getattr(_real, 'strategy_id', 0) == 0:  # ← 永远False!
    _real.strategy_id = _my_sid  # 永远不执行
```

**影响**:
- `update_status_bar()` 使用错误的strategy_id → C++无法匹配策略实例
- `sub_market_data()` 传递错误的strategy_obj → 订阅关联到错误策略
- C++无法正确管理策略生命周期 → 暂停/删除失败

**修复**: 移除 `== 0` 条件，改为强制同步

```python
# 修复后:
if _real is not None:
    _real.strategy_id = _my_sid  # 无条件强制同步
```

### 12.3 根因六：平台状态不同步（P1 — UI显示"初始化"而非"运行中"）

**文件**: `t_type_bootstrap.py` on_start()

**根因**: auto_start是Python内部调用on_start()，C++平台不知道。C++维护自己的状态机（Created→Initialized→Running→Stopped），只有C++自己调用on_start()（用户点击"执行"按钮）才会转换为Running。

**INFINIGO API分析**:
- `pauseStrategy(strategy_id)` — 暂停策略，效果同客户端点击暂停
- **没有** `startStrategy` 或 `resumeStrategy` 函数
- `updateState(strategy_id, data)` — 仅更新状态栏显示值，不改变策略运行状态

**修复**: 在`t_type_bootstrap.on_start()`中尝试3级通知机制

```python
# 1. 优先尝试 INFINIGO.startStrategy()
# 2. 次选尝试 INFINIGO.resumeStrategy()
# 3. Fallback: 通过 updateState 通知平台刷新UI
try:
    import INFINIGO
    if hasattr(INFINIGO, 'startStrategy'):
        INFINIGO.startStrategy(StrategyID=self.strategy_id)
    elif hasattr(INFINIGO, 'resumeStrategy'):
        INFINIGO.resumeStrategy(StrategyID=self.strategy_id)
    else:
        from pythongo import infini
        infini.update_state(
            strategy_id=self.strategy_id,
            data={"trading": "True", "inited": "True", "state": "Running"}
        )
except Exception:
    pass
```

**注意**: 若INFINIGO无startStrategy函数，平台状态可能仍需用户点击"执行"按钮。此修复确保了所有可能的通知路径都被尝试。

### 12.4 onTick自愈机制修复

**根因**: onTick自愈机制调用 `self.on_start()`，self是Strategy2026实例，同样不会触发t_type_bootstrap.on_start()。

## 十三、v7.0 修复 — 平台状态自动同步（无需用户点击）（2026-07-13）

**背景**: 用户确认暂停/删除功能修复成功，但反馈"策略实际上已经自动进入运行状态，但平台状态仍然是初始化状态，用户点击后状态变化为运行状态（也应当自动而无需用户点击）"。经分析发现根因是时机问题，v6.0的延迟调用方案仍然无法让C++在on_init()返回时读取到正确的`trading=True`。

### 13.1 根因七：C++状态读取时机错误（P1 — 平台状态显示"初始化"而非"运行中"，需手动点击）

**根因链分析**:
```
修复前v6.0流程:
C++ → t_type_bootstrap.on_init()
  → 检查auto_start_after_init=True
  → 创建Timer延迟0.05s → on_init()返回 → C++读取t_type_bootstrap.trading
  ← 此时trading仍然是False (on_start()还没执行)
  ← C++认为策略未运行 → 保持INITIALIZED状态 → UI显示"初始化"
  → 0.05s后Timer触发 → self.on_start() → 设置trading=True
  ← C++已经完成状态读取，不会自动刷新 → UI仍然显示"初始化"
  ← 用户需要手动点击"执行" → C++重新读取trading=True → 转换为RUNNING
```

**关键观察**:
- C++平台在`on_init()`返回后**立即同步读取**`t_type_bootstrap.inited`和`t_type_bootstrap.trading`属性
- 读取结果决定了UI显示状态，之后C++不会自动刷新状态，除非用户手动点击"执行"
- v6.0延迟方案导致读取时`trading`仍然是False，所以UI错误显示

### 13.2 修复方案：预设置trading=True + 延迟业务逻辑

**修复**: 在`on_init()`主线程**同步预设置**`self.inited=True`和`self.trading=True`，让C++读取到正确值，然后延迟`on_start()`业务逻辑执行。

```python
# 修复后:
if _auto_start:
    # 1. 主线程同步预设置，让C++读取到正确值
    self.inited = True
    self.trading = True
    # 2. 业务逻辑延迟0.05s执行，给C++时间完成状态转换
    logging.info("[t_type_bootstrap] auto_start_after_init=True, 预设置trading=True → 延迟0.05s执行on_start()业务逻辑")
    import threading
    def _delayed_start():
        try:
            self.on_start()  # 执行业务逻辑 + 3级平台通知兜底
            logging.info("[t_type_bootstrap] auto_start延迟业务逻辑完成, trading=%s", self.trading)
        except Exception as _as_err:
            logging.error("[t_type_bootstrap] auto_start延迟业务逻辑失败: %s", _as_err, exc_info=True)
    _timer = threading.Timer(0.05, _delayed_start)
    _timer.daemon = True
    _timer.start()
```

**为什么这样修复**:
1. `on_init()`返回前，C++还没读取trading属性 → 此时预设置`trading=True`是安全的
2. `on_init()`返回后，C++立即读取 → 读到`trading=True` → 正确转换为RUNNING状态 → UI显示"运行中"
3. `on_start()`业务逻辑（订阅行情、启动定时任务等）仍然延迟执行，不影响C++状态转换
4. 保留原有3级INFINIGO API通知机制作为兜底

**关键教训**: C++平台状态同步依赖`on_init()`返回时读取实例属性，必须在`on_init()`返回前设置好正确的`inited`和`trading`值，否则即使后续Python修改了属性，C++也不会自动刷新状态。

**关键教训**: 暂停/删除失效的真正根因不是方法委托缺失（v1.0-v5.0的修复方向），而是**auto_start调用路径错误**导致`t_type_bootstrap.trading`保持False，C++平台认为策略未运行。v1.0-v5.0的修复虽然补全了方法委托，但C++根本不会调用这些方法（因为认为策略未运行），所以修复无效。v6.0+通过修正auto_start调用路径，让C++正确感知策略已运行，才真正解决了暂停/删除失效问题。
