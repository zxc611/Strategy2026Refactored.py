## 二、重复文件产生原因

### 2.1 历史演进

项目经历了从 **扁平结构 → 包结构** 的重构：

```
阶段1（早期）: ali2026v3_trading/strategy_2026.py     ← 所有文件在根目录
阶段2（重构）: ali2026v3_trading/strategy/strategy_2026.py  ← 按功能划分子包
```

重构时将文件从根目录移入子目录（strategy/、position/、config/、lifecycle/），但**未删除根目录的旧文件**，导致双份。

### 2.2 修复时只更新子目录版

后续所有代码修改（07-01至07-09的30+次修复）都针对子目录版本，根目录版本停留在06-30或更早。这导致：

- 根目录版 = **旧版快照**（最后一次同步是06-30）
- 子目录版 = **活跃开发版**（持续更新至今）

### 2.3 严重风险：修改可能遗漏

当开发者看到两个同名文件时，容易混淆应该修改哪个。本次会话中就发生了此问题——
`strategy_2026.py` 的 FIX-20260709-PQS-INIT 被同时应用到根目录和子目录两个版本，
但部分修复（如 lifecycle_service.py）只修改了子目录版，根目录版仍为旧代码。
