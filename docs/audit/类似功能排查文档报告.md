## 三、高风险类似点详细分析（Top 5）

### S-04 WAL 预写日志服务（代码级逐行重复）

**涉及文件：**
- `order/order_wal_state_service.py` — `OrderWALStateService`
- `order/order_persistence.py` — `OrderPersistenceService`

**重复证据：**
- `_wal_write` vs `wal_write`：两者均构造 `{'order_id':..., 'state':..., 'instrument_id':..., 'direction':..., 'volume':..., 'price':..., 'timestamp': time.time()}`，均使用 `_tmp_path + os.replace` 原子写入模式。
- `_wal_read` vs `wal_read`：均使用 `os.path.exists` + `json.load(f)`。
- `_wal_delete` vs `wal_delete`：均使用 `os.path.exists` + `os.remove`。

**根因：**
`order_persistence.py` 的 docstring 明确说明“从 order_service.py 拆分，委托到 order_wal_state_service.py”，但实际仍保留了完整独立的 WAL 实现，未真正完成委托。

**影响：**
- 同一订单状态变更可能触发两次 WAL 写入（如果上层同时调用两个服务）。
- 孤儿订单恢复逻辑分散在两处，恢复行为可能不一致。

### S-05 RiskService 内联委托（接口+逻辑重复）

**涉及文件：**
- `risk/risk_service.py` — `RiskService`
- `risk/risk_check_service.py` — `RiskCheckService`
- `risk/risk_check_engine.py` — `RiskCheckEngine`

**重复证据：**
`RiskService` 中以下方法均为纯一行委托：
- `check_before_trade` → `self._check_service.check_before_trade(signal)`
- `_check_safety_meta_layer` → `self._check_service._check_safety_meta_layer(signal)`
- `_check_operational_risks` / `_check_market_risks` / `_check_counterparty_risks` / `_check_regulatory_risks`
- `_detect_abnormal_trading_step` / `_record_and_publish` / `_check_strategy_status` / `_check_rate_limit`
- `_check_position_limit` / `_check_risk_ratio` / `_check_risk_consistency` / `_check_single_trade_risk`
- `_check_sharpe_iron_rule` / `_check_e7_residual_block` / `_check_capital_sufficiency_in_trade`
- `_check_spread_degradation` / `_check_governance_violations` / `_check_greeks_limits`
- `_check_consecutive_loss_protection` / `_check_life_expectancy` / `_check_invariant_runtime`

累计约 20+ 个方法、80-100 行委托代码。

**根因：**
重构过程中将 `RiskCheckService` 从 `RiskService` 拆分，但 `RiskService` 保留了全部旧接口以维持向后兼容，未彻底转为 Facade 薄壳。

**影响：**
- 任何风控规则变更需同时修改 `RiskCheckEngine`、`RiskCheckService`、`RiskService` 三处接口签名。
- 新开发者容易误在 `RiskService` 中直接实现逻辑，破坏分层。

### S-03 滑点计算（逻辑+接口重复）

**涉及文件：**
- `infra/shared_utils.py` — `compute_slippage_bps`
- `order/order_risk_guard.py` — `estimate_slippage`
- `param_pool/backtest_loop_slippage.py` — `_compute_dynamic_slippage_bps`
- `param_pool/backtest_runner_utils.py` / `backtest_loop_core.py` / `backtest_position_manager.py` / `backtest_runner_base.py` / `backtest_state.py`

**重复证据：**
- 生产侧 `compute_slippage_bps`：处理 `base_slippage_bps`、`spread_bps * quality_scale * liquidity_mult`、`instrument_mult`、`expiry_mult`。
- 回测侧 `_compute_dynamic_slippage_bps`：同样处理 `base_slippage_bps`、`spread_bps * _quality_scale * _liquidity_tier`、`expiry_mult`，并额外加入 `_premium_bps`。
- `order_risk_guard.py` 的 `estimate_slippage` 只是对 `compute_slippage_bps` 的薄封装。

**根因：**
生产与回测的滑点模型独立演进，回测侧为保真度需求增加了更细粒度的参数（到期日倍增、级联滑点、流动性分层），但未反向同步到生产侧统一模型。

**影响：**
- 生产与回测的滑点估算结果存在系统性偏差，可能导致回测通过的参数在生产上表现不佳。
- 同一滑点公式在 6 个文件中重复定义，维护成本高。

### S-19 SafetyMeta 安全元层（代码级重复，内部）

**涉及文件：**
- `risk/risk_circuit_breaker.py` — 遗留巨型 `SafetyMetaLayer`
- `risk/safety_meta_layer.py` — 精简版 `SafetyMetaLayer`
- `risk/safety_meta_circuit.py` / `safety_meta_equity.py` / `safety_meta_position.py` / `safety_meta_audit.py`

**重复证据：**
- `risk_circuit_breaker.py` 中的 `SafetyMetaLayer` 是重构前的完整实现（含熔断+回撤+硬时间停止+合规+资本充足性），约 500+ 行。
- 拆分后的 4 个子文件提取了上述功能，但 `risk_circuit_breaker.py` 中的旧实现仍被保留（PAT-01 已确认是 Facade 委托，但代码实体仍在）。
- `RiskService` 内联了所有 SafetyMeta 子服务方法，形成 1:1 重复。

**根因：**
重构拆分后，旧文件未彻底删除或改为纯 re-export，而是保留了完整类定义作为“Facade”。

**影响：**
- 旧 `SafetyMetaLayer` 中的逻辑若被误调用，可能与拆分后的新实现产生冲突。
- 文件体积虚增，误导代码阅读者。

### S-08 Greeks 聚合（逻辑重复）

**涉及文件：**
- `position/position_greeks.py` — `aggregate_greeks_exposure`
- `governance/greeks_calculator.py` — `GreeksCalculator.get_portfolio_greeks`
- `risk/risk_service.py` — `RiskDashboardService.aggregate_greeks`

**重复证据：**
- 三者均执行“遍历持仓 → 获取单标 Greeks → 加权累加”的相同逻辑。
- 数据来源不同：`position_greeks.py` 使用估计值；`greeks_calculator.py` 使用 Black-Scholes 计算值；`risk_service.py` 使用 `pos.greeks` 属性。
- 输出结构相同：均返回 `{'delta':..., 'gamma':..., 'theta':..., 'vega':...}` 字典。

**根因：**
持仓、治理、风控三个子系统各自需要 Greeks 聚合，但未建立统一的 `GreeksAggregationService`。

**影响：**
- 同一持仓在不同模块中看到的 Greeks 总量可能不一致（估计误差 vs 精确计算 vs 缓存值）。
- 新增 Greeks 维度（如 rho、vomma）需修改 3 处。

## 四、中风险类似点摘要（6 项）

### S-02 MaxDD 最大回撤计算
- `param_pool/backtest_metrics.py` 使用 numpy 向量化实现。
- `strategy/shadow_strategy_pnl_metrics.py` 使用纯 Python 循环实现。
- **建议：** 提取统一 `max_drawdown` 工具函数到 `infra/shared_utils.py`，向量化版作为默认实现，循环版作为无 numpy 依赖的 fallback。

### S-06 Cooldown 信号冷却期
- `signal/cooldown_manager.py` 提供通用线程安全组件。
- `param_pool/backtest_position_manager.py` 内联了相同判断逻辑。
- **建议：** 回测模块接入通用 `CooldownManager`，或提取 `is_in_cooldown(elapsed, threshold)` 纯函数到共享工具。

### S-10 Config 参数读取
- `config/config_params.py` 提供 `DEFAULT_PARAM_TABLE` 和 `get_cached_params`。
- `risk/risk_config_provider.py` 独立实现了 `_get_signal_cooldown`、`_get_max_risk_ratio` 等相同参数的读取和缓存。
- **建议：** `RiskConfigProvider` 委托到 `ConfigService` 或 `get_cached_params`，消除独立缓存链。

### S-11 TickAggregator
- M1 生产侧：`quant_platform.py` / `tick_aggregator_service.py` 驱动实时 Bar 生成。
- M2 回测侧：`param_pool/tick_aggregator.py` / `preprocess_ticks.py` 用于回测数据预处理。
- **建议：** 提取核心聚合算法（时间窗口/成交量窗口、OHLCV 计算）到 `infra/shared_utils.py`，生产与回测分别包装 I/O 适配层。

### S-12 PositionManager 持仓管理
- M1：`position/position_service.py` + `position_command_service.py` + `position_check_service.py`。
- M2：`param_pool/backtest_position_manager.py`。
- **建议：** 回测持仓管理器可复用 M1 的 `PositionCommandService` 和 `PositionCheckService` 的逻辑，通过接口适配器注入回测专用数据访问层。

### S-15 Commission 佣金计算
- M2：`param_pool/backtest_commission.py` 提供按品种、交易所、maker/taker 的精细化费率。
- M1：`risk/risk_compute_service.py` 的 `compute_simplified_pnl` 使用 `commission_per_lot` 简化扣除。
- **建议：** 生产侧引入 `backtest_commission.py` 的费率模型，或提取统一 `CommissionCalculator`。

## 七、结论

1. **本报告共识别 22 个功能类似点**，其中：
   - **高风险（代码级重复）：3 项**（S-04 WAL、S-05 RiskService、S-19 SafetyMeta）
   - **显著风险（逻辑+接口重复）：1 项**（S-03 滑点计算）
   - **中风险（逻辑重复）：10 项**
   - **低风险/伪重复：5 项**
   - **无法验证：1 项**（S-01，目标文件缺失）

2. **已消重审计处理但未彻底的项：** PAT-02（RiskService 内联委托）需要二次审计。

3. **建议优先处理顺序：** S-04 → S-05 → S-19 → S-03 → S-08 → S-10 → S-11 → S-12 → S-02 → S-06 → S-15。
