### 维度1：装饰器叠加顺序与副作用

#### 1.1 [P1] `@property` 中执行惰性初始化（含import + DB连接）

- **文件：** `strategy_lifecycle_mixin.py`
- **行号：** 72-84
- **代码片段：**
```python
@property
def storage(self):
    if self._storage is None:
        with self._storage_lock:
            if self._storage is None:
                try:
                    from ali2026v3_trading import get_instrument_data_manager
                    self._storage = get_instrument_data_manager()
                except Exception as e:
                    logging.error(f"[Storage] Init failed: {e}")
                    raise RuntimeError(f"Storage init failed: {e}")
    return self._storage
```
- **问题描述：** `@property` 约定为轻量级属性访问（无可观察副作用）。此property执行`import`和数据库连接创建，可能引发意外行为：异常上下文中访问property导致二次异常、调试器/IDE变量检查触发重量级连接、序列化/日志记录无意创建DB连接。
- **严重等级：** P1 — property中执行DB连接初始化是反模式
- **影响系统：** 生产系统（`StrategyCoreService` via `_LifecycleMixin`）

#### 1.2 [P1] `@contextmanager` 中 monkey-patch `pd.DataFrame.__getattr__` 线程不安全

- **文件：** `参数池/L1参数量化/v7_meta_audit_v2.py`
- **行号：** 520-570
- **问题描述：** `_data_monitor()` 上下文管理器在 `__enter__` 阶段 monkey-patch `pd.DataFrame.__getitem__` 和 `pd.DataFrame.__getattr__`。代码自身在529-534行已警告多线程不安全，但仅记录日志警告，未硬性阻止多线程执行。此外 `pd.DataFrame.__getattr__` 在部分pandas版本（Cython实现）中替换可能静默失效。
- **严重等级：** P1 — monkey-patch未做硬性阻止，pandas版本兼容性不确定
- **影响系统：** 测试/评判系统（`v7_meta_audit_v2.py`）

#### 1.3 [P2] `@staticmethod` 叠加 `@contextmanager` — 装饰器顺序审查（已确认安全）

- **文件：** `width_cache.py`
- **行号：** 16-19
```python
@staticmethod
@contextmanager
def startup_step(name):
    yield
```
- **问题描述：** 审查确认装饰器叠加顺序正确。`@contextmanager`（底层）先将generator函数转为`_GeneratorContextManager`，`@staticmethod`（上层）再包装为静态方法描述符。`cls.startup_step('test')` 返回正确的上下文管理器。**无问题，仅记录审查结论。**
- **严重等级：** P2 — 建议添加注释说明顺序意图
- **影响系统：** 评判系统

### 维度3：`__slots__` 使用

#### 3.1 [P2] `@dataclass` 类可考虑使用 `__slots__` 优化内存

- **文件：** 多个文件（约25个@dataclass类）
- **问题描述：** 系统中大量 `@dataclass` 类（`RiskDimensionScores`、`LifeExpectancySnapshot`、`CyclePredictionSnapshot`、`StrategyStateSnapshot`、`MarketSnapshot`、`HookConfig`、`SortEntry` 等）均未设置 `__slots__`。高频交易场景中每个tick创建多个快照对象，使用`__slots__`可显著减少内存分配。Python 3.10+ 支持 `@dataclass(slots=True)`。
- **严重等级：** P2 — 性能优化建议
- **影响系统：** 三系统均涉及

#### 3.2 `__slots__` 使用审查 — 全部正确

对19处 `__slots__` 声明进行了逐项审查：
- 所有 `__slots__` 类均未继承其他类（或仅继承 `object`），无继承冲突
- `SurvivalAnalyzer._MAX_OBSERVATIONS = 5000`（类变量）与 `__slots__` 正确共存
- `StrategyParams` 使用 `__slots__ = ('_data',)` 配合自定义 `__getattr__`/`__setattr__`，设计合理
- 无 `__dict__` 与 `__slots__` 共存导致的混淆
- 无 `__weakref__` 相关冲突

### 维度7：`__init_subclass__` 和元类

#### 7.1 [P2] `exec()` 受限执行环境 — AST过滤不完整

- **文件：** `参数池/L1参数量化/v7_meta_audit_v2.py`
- **行号：** 1623-1637
- **问题描述：** `RestrictedExecLoader.safe_exec()` 使用 `ast.parse` + walk 过滤 import 语句，仅允许特定模块。但以下绕过路径未被阻止：`__import__('os')`（绕过AST检查的内置函数调用）、`getattr(__builtins__, '__import__')('os')`。由于传入的 `__builtins__` 是受限的，实际风险较低。生产系统不使用此模块。
- **严重等级：** P2 — 安全加固建议，目前风险低
- **影响系统：** 测试系统

#### 7.2 [P2] 模块级 `__getattr__` 惰性导入 — 与 `__all__` 一致性问题

- **文件：** `main.py`
- **行号：** 24-51
- **问题描述：** 模块级 `__getattr__` 实现惰性导入，依赖 `_LAZY_IMPORTS` 字典和 `__all__` 列表。若在 `__all__` 添加新导出但忘记在 `_LAZY_IMPORTS` 添加映射，访问符号将静默抛出 `AttributeError`。两者之间无自动一致性校验。
- **严重等级：** P2 — 手动维护不一致会导致运行时错误
- **影响系统：** 生产系统入口（`main.py`）

## 补充发现

### S1. [P1] `__slots__` 类中 `@property` 读取与 `_lock` 不一致

- **文件：** `quant_platform.py`
- **行号：** 196、219-221（`AtomicSystemState`）
- **问题描述：** `capture()` 方法通过 `self._lock` 保护写入。但 `@property version` 和 `@property age_ms` 直接读取 `self._version` / `self._last_update_ms` 不获取锁。在CPython中int/float的简单读取在GIL保护下是原子的，不会数据损坏。但 `age_ms` 可能因竞态条件返回不完全一致的时间差，且违反了类自身建立的加锁契约。
- **严重等级：** P1（实际影响极小，GIL保护+简单类型）
- **影响系统：** 生产系统

### S2. [P2] 模块级 `__getattr__` 写入 `globals()`

- **文件：** `__init__.py`
- **行号：** 76-85
- **问题描述：** `__getattr__` 惰性导入后执行 `globals()[name] = value`。PEP 562标准模式，但模块全局命名空间在首次属性访问后被修改，对依赖模块重载的场景（如`importlib.reload`）可能引起意外。
- **严重等级：** P2 — PEP 562标准模式，影响极微
- **影响系统：** 生产系统入口
