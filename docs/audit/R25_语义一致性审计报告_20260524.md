## P0 — 致命级（3项）

### P0-01: `RiskService._check_greeks_limits` 使用不存在的 `self._params` → `AttributeError`

- **文件**: `risk_service.py`
- **行号**: 1055
- **声明内容**: `safe_get_float(self._params, "theta_abs_threshold", 0.5)`
- **实际状态**: `RiskService.__init__`（行138）定义的是 `self.params`，**没有** `self._params` 属性。`self._params` 仅存在于 `SafetyMetaLayer` 类（行1818），那是另一个独立的类。
- **不一致类型**: 函数名与实际行为不一致（_check_greeks_limits 在某些信号/仓位组合下会调用此路径），且为 **逻辑矛盾** — 引用了不存在的属性。
- **后果**: 当 `RiskService._check_greeks_limits` 执行到 theta 软约束检查分支（即持仓非空、Greeks计算器可用、无硬约束违规时），此行代码将抛出 `AttributeError: 'RiskService' object has no attribute '_params'`，导致整个风控检查崩溃。
- **修复建议**: 将 `self._params` 改为 `self.params`。

### P0-02: `strategy_core_service.py` — `risk_dashboard_status` 变量先使用后赋值 → `UnboundLocalError`

- **文件**: `strategy_core_service.py`
- **行号**: 447（使用）vs 498（首次赋值）
- **声明内容**: `components["risk_dashboard"] = risk_dashboard_status`（行447）
- **实际状态**: `risk_dashboard_status` 首次被赋值在行498（`risk_dashboard_status = "UNAVAILABLE"`），距离使用处51行。中间无其他赋值路径。
- **不一致类型**: 逻辑矛盾 — 变量使用顺序与定义顺序颠倒。
- **后果**: 每次调用 `get_component_status()` 或包含此逻辑的监控方法时，第一行（447）就会抛出 `UnboundLocalError`，导致整个组件状态收集失败。该函数的其他19个组件状态（行430-469之间）已全部正常填充，唯独 `risk_dashboard` 插入点异常提前。
- **修复建议**: 将行446-447移到行517之后，或将 `risk_dashboard_status = "UNAVAILABLE"` 提前到行446之前。

### P0-03: `RiskDashboardService` 歧义缓存在多个 `self._params` 访问间可能导致线程竞态

- **文件**: `risk_service.py`
- **行号**: 2374（类定义）处 `RiskDashboardService` 类
- **声明内容**: 代码中多处使用 `safe_get_float(self._params, ...)`（行2268-2319）来获取断路器参数
- **实际状态**: `RiskDashboardService.__init__` 接收的参数对象可能在大类 `StrategyParams` 的不同生命周期中（初始化→JSON加载→YAML覆盖→热更新）被替换，但 `self._params` 是类实例初始化时的快照引用，不会自动跟随全局参数变更。
- **不一致类型**: 变量名与实际内容不一致 — `self._params` 名为参数对象，实际是初始化时刻的静态引用，非参数服务的实时视图。
- **后果**: 如果参数在运行时通过热更新（R4-D-10）或 YAML 重新加载变更，`RiskDashboardService` 仍使用旧的参数值，导致断路器阈值、止损时间等关键保护参数与 `RiskService` 其他部分不同步。
- **修复建议**: 改为每次使用时通过 `self._params_ref()` 回调获取最新参数，或通过全局 `get_default_params()` 动态获取。

## P1 — 高危级（8项）

### P1-01: `DEFAULT_PARAM_TABLE.close_stop_loss_ratio=0.5` 与手册V7规范 `0.3` 偏差67%

- **文件**: `config_params.py` 行198 vs 手册 Step1参数默认值表
- **声明内容**: `config_params.py:198`: `"close_stop_loss_ratio": 0.5`
- **手册标准**: V7手册中止损比例标准为 `0.3`（30%），且在 `task_scheduler.py:339` 中的回测网格起始值也是 `OPTION_STOP_LOSS_GRID = [0.25, 0.3, ...]`
- **不一致类型**: 审计标准术语与代码术语不一致（默认值偏差）
- **后果**: 当 `config_params.json` 加载失败或不存在时，系统回退到代码硬编码默认值 0.5，意味着实际止损带宽扩大67%（亏损50%才止损而非30%）。这是"配置漂移"的典型场景，已在R24审计中标记为P0-03。
- **已在上轮审计标记**: R24-P0-03

- **后果**: 取决于加载路径，止损带宽可能在5%到60%之间大幅波动，导致生产行为不可预测。

- **不一致类型**: 命名风格不一致 + 逻辑矛盾（P0-01的实现层面表现）
- **后果**: 极易诱发类似 P0-01 的 bug。

### P1-06: `_signal_list` 变量名暗示列表但实际类型未统一为队列结构

- **涉及文件**: `signal_service.py`, `risk_service.py`, `strategy_tick_handler.py`
- **声明内容**: `risk_service.py:149`: `self._signal_times: Dict[str, deque] = {}`
- **实际状态**: `_signal_times` 使用 `deque` 而非 `list`，但 `signal_service.py` 中的信号存储使用普通 `list`（`.append()` + 切片截断）。
- **不一致类型**: 变量名与实际内容不一致 — 多处信号容器命名含 `list` 但实际存储结构不同（一部分是 `deque`，一部分是 `list`），`deque` 具有固定 `maxlen` 自动截断的特性，`list` 无此特性。
- **后果**: 在 `list` 版本中如忘记手动截断，信号存储会无限增长导致内存泄漏。

## P2 — 改进级（6项）

### P2-01: `state_param_sets` vs `TP_SL_REASON_DEFAULTS` — 同一概念用不同命名

- **涉及文件**: `config_params.py:677` vs `position_service.py` 内部
- **声明内容**: `config_params.py` 使用 `get_default_state_param_sets()` 返回三个状态（`correct_trending`, `incorrect_reversal`, `other`）的参数默认值
- **实际状态**: `position_service.py` 使用 `TP_SL_REASON_DEFAULTS` 做类似的事情但命名完全不同
- **不一致类型**: 命名风格不一致 — 同一领域概念用了两个不同的函数/字典名。

---

### P2-02: 函数命名前缀风格不一致 — `_check_` vs `_validate_` 语义边界模糊

- **涉及文件**: 多文件（`risk_service.py`, `storage_query.py`, `config_params.py`, `params_service.py`）
- **声明内容**: 
  - `_check_` 前缀: 30个函数（`risk_service.py` 的 `_check_safety_meta_layer`, `_check_greeks_limits` 等）
  - `_validate_` 前缀: 11个函数（`storage_query.py` 的 `_validate_tick`, `params_service.py` 的 `_validate_param_against_matrix` 等）
- **实际状态**: `_check_*` 普遍返回 `RiskCheckResponse`（含 pass/block/warn），而 `_validate_*` 普遍返回 `bool`。但部分 `_check_*` 函数也存在副作用（写缓存、发日志），与 `_validate_*`（纯校验）的隐含语义冲突。
- **不一致类型**: 函数名与实际行为不一致 — `_check_greeks_limits` 不仅检查还更新仪表盘缓存（有副作用）。
- **后果**: 调用方无法从函数名判断是否有副作用，可能在不恰当的上下文调用导致缓存脏数据。

- **文件**: 手册 0.1 节 vs 代码全局搜索

- **不一致类型**: 审计标准术语与代码术语不一致 — 手册定义的全局概念在代码中无单一准确性字段。
- **后果**: 按手册查找 `overall_accuracy` 时会找不到对应代码位置。

### P2-04: `_snapshot` 命名参差不齐 — 有的拷贝有的引用

- **文件**: `params_service.py:1255`, `params_service.py:1453`
- **声明内容**: 两处都使用 `params_snapshot = dict(self._params)` 获取参数快照
- **实际状态**: `dict()` 浅拷贝 — 如果 `self._params` 的值是可变对象（如列表、字典），则快照中的对应值仍然是原对象的引用，不是真正的独立副本。好在当前参数值均为不可变类型（数字、字符串），暂不触发问题。
- **不一致类型**: 变量名与实际内容不一致 — 名为"快照"但只是浅拷贝。
- **后果**: 未来若引入可变类型参数值（如 `allowed_states = ['trending', 'other']`），浅拷贝的快照与原始数据共享可变子对象，违背快照的语义约定。

### P2-06: `shadow_strategy_engine.py` 的 `_feed_shadow_engine` 与 `strategy_core_service.py` 中的调用函数签名不匹配风险

- **涉及文件**: `strategy_core_service.py`（调用方）vs `shadow_strategy_engine.py`（实现方）

- **不一致类型**: 命名风格不一致 — 两端对同一语义数据使用不同的字段名约定，缺少中间适配层。
- **后果**: 若任一端修改了字段名约定，另一端将静默接收/忽略数据（非报错，极难发现）。
