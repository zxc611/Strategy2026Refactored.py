## 三、生产系统现状（代码实证）

### 3.1 实证确认已修复的历史债务

**（1）交易前安全检查三函数**

文档多次声称 `check_regulatory_compliance()`、`check_capital_sufficiency()`、`check_exchange_status()` "定义完整但零调用"。代码实证**反驳该声明**：

- **`backtest_position_manager.py:97-133`**: `_try_open()` 中按序调用三者，均有 `fail-safe` 阻断逻辑：
  ```python
  _compliance = _rs.check_regulatory_compliance(_position_data)
  if not _compliance.get('compliant', True): return False
  _capital = _rs.check_capital_sufficiency(bt.equity, _required_margin, _open_positions)
  if not _capital.get('sufficient', False): return False
  _exchange_status = _rs.check_exchange_status()
  if _exchange_status.get('status') != 'OPEN': return False
  ```
- **`position_command_service.py:183-210`**: `_add_position()` 中调用 `check_capital_sufficiency()`，阻断逻辑为 `logging.critical` + `return`。
- **`backtest_runner_utils.py:242-280`** 和 **`backtest_loop_core.py:292`**: `_try_open()` 通过 `backtest_position_manager.try_open_risk_checks()` 间接调用三者。

**结论**: 三个交易前安全检查函数**已被集成到回测核心循环和生产持仓服务中**，调用链完整。

**（2）PredictiveStateEngine**

文档声称 "PredictiveStateEngine 零调用"。代码实证**反驳该声明**：

- **`governance/mode_engine_propagation.py:50`**: `pse = PredictiveStateEngine.get_instance()` 在组件传播时被激活。
- **`risk/risk_compute_service.py:265,383`**: 两处调用 `PredictiveStateEngine.get_instance()` 用于希腊字母/风险计算。

**结论**: `PredictiveStateEngine` 在风控计算和模式引擎传播路径中被实际调用，非死代码。

**（3）影子策略API污染**

文档声称 "内部模块被外部直接import" 导致API污染。代码实证**部分反驳**:

- `shadow_strategy_core.py`、`shadow_strategy_signal.py`、`shadow_strategy_pnl.py` 的**非备份/非测试**调用者仅有：
  - `strategy/shadow_strategy_facade.py`（合法Facade）
  - `strategy/_shadow_services.py`（合法内部服务）
  - `tests/test_mixin_elim_rounds4_7.py`（测试文件）
- **没有证据表明 `strategy` 包外部的生产代码直接导入这些模块**。

**结论**: API污染问题在**当前版本中已被控制**，未发现跨包违规引用。但内部模块命名未统一使用下划线前缀（如 `_shadow_strategy_core.py`），边界保护依赖开发者自律而非工具强制。

### 3.2 实证确认仍存在的问题

**（1）RiskDashboardService — 真·死代码**

- 定义于 `risk/risk_service.py:643`，实现完整（Greeks缓存、PnL归因、压力测试）。
- **当前生产代码中无外部调用**。历史上曾在 `_backup_r22_20260603_210608/strategy_core_service.py:1692` 中被调用，但该调用在重构后已消失。
- `risk_service.py` 将其列入 `__all__`，但没有任何模块 `get_risk_dashboard_service()`。

**结论**: 这是文档中少数**被代码证实的真死代码**案例。

**（2）shadow_strategy_facade.py 继承而非组合**

文档建议的PA-03方案要求 "Facade纯组合（非继承）"。实际代码：

```python
class ShadowStrategyEngine(ShadowStrategyCoreService):  # 继承，非组合
    def __init__(self, **kwargs):
        super().__init__(**kwargs)
        self._signal_service = ShadowStrategySignalService()
        self._pnl_service = ShadowStrategyPnLService(self)
```

- `Signal` 和 `PnL` 确实通过 `__getattr__` 委托，但 `Core` 仍通过继承暴露。
- 基类 `ShadowStrategyCoreService` 的公共方法（如 `_load_absolute_ev_floor`）被Facade隐式继承，存在内部签名泄漏风险。

**（3）三源分裂（YAML / Code / 手册）**

`shadow_strategy_core.py:90-118` 的 `_load_absolute_ev_floor()` 同时从 `parameter_attribute_matrix.yaml` 和类常量 `ABSOLUTE_EV_FLOOR = -0.5` 加载：

```python
try:
    _matrix_path = os.path.join(..., 'param_pool', 'parameter_attribute_matrix.yaml')
    ...
    _ev_floor = _data.get('shadow_absolute_ev_floor', {}).get('default', None)
except Exception:
    self._absolute_ev_floor: float = self.ABSOLUTE_EV_FLOOR
```

- 文档声称的 YAML=0.8 / Code=0.3 / 手册=0.3 分裂问题**在代码中确实存在**，且 `_load_absolute_ev_floor` 的 fallback 路径可能使实际运行值与YAML声明值不一致。

**（4）风控委托链过深**

`risk_service.py:393-403` 中，三个检查方法均完全委托给 `get_safety_meta_layer()`：

```python
def check_regulatory_compliance(self, position_data=None):
    return get_safety_meta_layer().check_regulatory_compliance(position_data)
```

- 表面看是合理的Facade模式，但 `get_safety_meta_layer()` 的实现未知。
- 如果 `safety_meta_layer` 内部又存在多层委托，可能形成**调用链深度 > 5** 的性能隐患（tick路径）。

## 四、测试系统现状（代码实证）

### 4.1 实证确认已集成的验证器

**MultiPeriodCrossValidator / SurvivalBiasTest / ParameterProximityTracker / CheckpointManager / MultiParameterTracer**

文档多次声称 "全部零调用"。代码实证**部分反驳**：

- **`param_pool/phase_scan_cli.py:77-163`**: 5个验证器全部被实例化调用，代码结构为：
  ```python
  try:
      from ... import MultiPeriodCrossValidator
      mpcv = MultiPeriodCrossValidator()
      logger.info("[T-03] MultiPeriodCrossValidator集成完成")
  except Exception as _t03_err:
      logger.warning("[T-03] MultiPeriodCrossValidator集成失败(非致命): %s", _t03_err)
  ```
- **`param_pool/checks_statistical.py:131-184`**: `MultiPeriodCrossValidator` 和 `SurvivalBiasTest` 被集成到统计检验门控中，有 `all_passed = False` 硬阻断逻辑。

**结论**: 验证器**并非全部零调用**，在 CLI 入口和统计检验门控中已集成。

### 4.2 实证确认仍存在的问题

**（1）核心扫描器未集成验证器**

- **`param_pool/phase_scan_core.py`**（核心扫描器）中 **无任何验证器调用**。
- `enhanced_phase_scan.py` 仅为 re-export 门面，也未调用验证器。

**结论**: 验证器集成停留在 CLI 层和统计门控层，**核心参数扫描器（PhaseScan）独立运行时无过拟合/幸存者偏差检验**。

**（2）回测主引擎未直接调用风控检查**

- `backtest_runner_base.py` 中**无任何** `check_regulatory_compliance` / `check_capital_sufficiency` / `check_exchange_status` 调用。
- 风控检查仅通过 `backtest_position_manager.py` 间接集成。

**结论**: 回测主引擎与风控检查之间存在**间接耦合**，如果 `backtest_position_manager` 被绕过或替换，风控检查将失效。

**（3）task_scheduler.py 无风控集成**

- `param_pool/task_scheduler.py` 中无风控检查调用。
- 多进程调度时，风控依赖子进程中的 `backtest_position_manager`，存在跨进程一致性风险。

## 五、评判系统现状（代码实证）

### 5.1 实证确认已修复的历史债务

**（1）governance_engine checker 集成**

文档声称 "governance_engine 所有 checker 均未被 strategy_judgment_engine.judge() 调用"。代码实证**反驳该声明**：

- **`strategy_judgment/_judgment_services.py:409-458`**: 7个checker被显式实例化和调用：
  - `E12ReverseStrategyPseudoIndependenceDetector`
  - `E13ShadowStrategyCollusionDetector`
  - `WF6ToWF10EliminationChecker`
  - `E8E9E10EliminationChecker`
  - `E7UnexplainedReturnChecker`
  - `E11QuantitativeSourceChecker`

- **`strategy_judgment/judgment_scoring_helpers.py:91-99`** 和 **`judgment_deep_validation.py:387`**: 均导入 `get_governance_engine()` 并执行检查。

**结论**: governance checker **已被集成到评判深度验证链路中**。

**（2）evaluation 组件跨系统调用**

- `CascadeJudge` 被调用点：**4个系统**（`strategy_judgment/`、`governance/`、`param_pool/`、`strategy/strategy_ecosystem/`）
- `ChicoryEvictionPolicy`、`MarqueeThreshold`、`ActivityWeightedScorer`、`StateEDensityDecayTracker` 被调用点：`strategy_judgment/judgment_scoring_helpers.py`

**结论**: evaluation 组件**已被多个系统消费**，非评判系统独占。

### 5.2 实证确认仍存在的问题

**（1）DeepValidationSuite 输入为空字典**

```python
# strategy_judgment/_judgment_services.py:213-214
dvs = DeepValidationSuite()
dvs_result = dvs.run_full_validation({})  # 传入空字典
```

- `run_full_validation({})` 接收空参数字典，**实际验证效果存疑**。
- 这可能是文档中提到的 "修复不完整" 的实证——代码调用了函数，但输入参数未正确传递。

**（2）Governance checker 输入为 stub**

```python
# _judgment_services.py:425-434
_e12_result = _e12.detect([], [])  # 传入空列表
_e13_result = _e13.detect({}, {}, [], [])  # 传入空集合
_e7_result = _e7.check({"total_pnl": 0.0})  # 硬编码 0.0
```

- checker 被调用了，但输入为**空集合或硬编码值**，实际检测能力有限。
- 这属于 "**调用链贯通但数据流断裂**" 的典型模式。

**（3）CascadeJudge 跨系统导入路径不一致**

- `strategy_judgment/` 使用 `from ali2026v3_trading.evaluation.cascade_judge import CascadeJudge`
- `strategy/strategy_ecosystem/_trading_ev.py` 和 `param_pool/backtest_state.py` 使用 `from evaluation.cascade_judge import CascadeJudge`（**缺少 `ali2026v3_trading` 包前缀**）

**结论**: 存在**相对导入与绝对导入混用**，在特定PYTHONPATH环境下可能引发 `ImportError`。

### 9.2 升A三阶段路径总览

