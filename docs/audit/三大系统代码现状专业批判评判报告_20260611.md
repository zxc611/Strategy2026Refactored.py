## 二、事实级错误纠正（P0/P1分级）

### P1-01: 报告A声称 backtest_loop_core.py 调用三安全检查函数 —— 不成立

**报告A原文**（第52行）:

**实证结果**: `backtest_loop_core.py` 中**零直接调用**三个安全检查函数。grep搜索结果为空。

**批判**: 报告A混淆了"间接可能经过"与"确认调用"。`backtest_loop_core.py` 调用 `try_open_risk_checks()`，后者内部调用三安全检查——这是**间接调用链**，报告A声称的行号292不存在对应调用。报告A应明确标注"间接调用"而非暗示直接调用。

**严重度**: P1（事实错误，但不影响"调用链已贯通"的结论本身）

## 三、方法论偏差批判

### 偏差1: "调用=集成"的认知等价谬误（两份报告共有，P0级）

**核心问题**: 两份报告将"函数出现在调用链中"等同于"功能已正确集成"，这是**工程评判中最危险的认知偏差**。

**实证反例**:

| 报告声称"已集成" | 实际代码证据 | 真实状态 |
|:---|:---|:---|
| DeepValidationSuite已集成 | `dvs.run_full_validation({})` 传入空字典 | **调用链贯通，数据流断裂** |
| E12 checker已集成 | `_e12.detect([], [])` 传入空列表 | **调用链贯通，数据流断裂** |
| E13 checker已集成 | `_e13.detect({}, {}, [], [])` 传入空集合 | **调用链贯通，数据流断裂** |
| E7 checker已集成 | `_e7.check({"total_pnl": 0.0})` 硬编码 | **调用链贯通，数据流断裂** |
| E11 checker已集成 | `_e11.check({})` 传入空字典 | **调用链贯通，数据流断裂** |
| WF6-WF10 checker已集成 | `_wf_chk.check_all([])` 传入空列表 | **调用链贯通，数据流断裂** |

**6个checker全部传入空数据/stub**——这在任何严肃的工程审计中应判定为"**未集成**"而非"已集成"。两份报告将此评为"已修复"或"已集成"，是对"集成"定义的严重降格。

**报告A的纠偏尝试**: 报告A在第5.2节识别了"DeepValidationSuite输入为空字典"和"Governance checker输入为stub"，将其定性为"调用链贯通但数据流断裂"。**这是报告A最有价值的发现**，但报告A未将此发现充分传导到总评——仍给出B+评级，未对"6个checker全部stub"做足够扣分。

**报告B的不足**: 报告B在4.3节仅以"注意"方式提及"验证但不拒绝"模式，未识别出checker输入为空这一更严重的问题。

**修正建议**: 
- "函数被调用"应标注为"**调用链已贯通**"
- "函数被调用且传入有效业务数据"才应标注为"**功能已集成**"
- 当前6个checker的状态应为"**调用链贯通，数据流断裂，功能未集成**"

### 偏差2: "软集成"风险量化不足（P1级）

**报告A的贡献**: 识别出"软集成epidemic"为P0级系统性风险，这是两份报告中**最深刻的工程洞察**。

**但量化不足**: 
- 实测 `strategy_judgment/` 中 `except Exception` 有 **102处**
- 实测 `param_pool/` 中 `except Exception` 有 **250处**
- 合计 **352处** `except Exception` 软集成点
- `judgment_scoring_helpers.py` 单文件有 **15个try/16个except**

报告A仅定性描述为"epidemic"，未给出352这一精确数字。报告B未做任何量化。

**352处 `except Exception` 意味着**: 系统中有352个点位的异常被静默吞掉，任何一个点位的功能失效都不会被感知。这不是"软集成"问题，这是**系统性静默失效隐患**。

### 偏差3: 评级传导逻辑断裂（P1级）

**问题**: 两份报告在分项评级中识别了严重问题，但未将问题严重度传导到总评。

**报告A的传导断裂**:
- 识别"软集成epidemic"为P0 → 但总评仍给B+
- 识别"6个checker全部stub"为P0 → 但总评仍给B+
- 识别"三源分裂"为P0 → 但总评仍给B+

**报告B的传导断裂**:
- 识别"验证但不拒绝"模式 → 但M3仍给B级
- 识别CascadeJudge架构不一致 → 但M3仅扣至B而非C+

**修正评级逻辑**: 
- 6个P0级checker stub → M3评判系统应从B降至**C+**（评判引擎的核心checker无真实输入，评判结果不可信）
- 352处软集成 → M1生产系统应从B+降至**B**（静默失效风险使系统可靠性打折）
- CascadeJudge架构不一致 → M3应进一步降至**C**（架构与手册对齐是评判系统的基础要求）

### 偏差4: "文档过时"归因过于简单（P2级）

**两份报告的共同叙事**: "五份文档描述的是5月代码状态，6月重构后已过时"

**批判**: 此归因忽略了两个可能性：
1. **文档审计本身可能存在偏差**——52裂缝报告和132轮审计报告是否真的基于代码实证，还是也存在"文档声称→文档声称"的传递偏差？
2. **重构可能引入新问题**——目录重组、Facade引入、Mixin消除等重构本身是否引入了新的技术债务？两份报告对此分析不足。

```python
class ShadowStrategyEngine:
    def __init__(self, **kwargs):
        self._core_service = ShadowStrategyCoreService(**kwargs)
        self._signal_service = ShadowStrategySignalService()
        self._signal_service._facade = self
        self._pnl_service = ShadowStrategyPnLService(self._core_service)

    def __getattr__(self, name):
        for _svc in (self._signal_service, self._pnl_service):
            if hasattr(_svc, name):
                return getattr(_svc, name)
        if hasattr(self._core_service, name):
            return getattr(self._core_service, name)
        raise AttributeError(f"'{type(self).__name__}' has no attribute '{name}'")

    # 显式委托Core公共方法（避免__getattr__性能开销在热路径）
    def validate_dependencies(self): return self._core_service.validate_dependencies()
    def shutdown(self, wait=True, timeout=10.0): return self._core_service.shutdown(wait, timeout)
    def get_shadow_a_params(self): return self._core_service.get_shadow_a_params()
    def get_shadow_b_params(self): return self._core_service.get_shadow_b_params()
    def get_params_snapshot(self): return self._core_service.get_params_snapshot()
    def relock_params(self, **kw): return self._core_service.relock_params(**kw)
    def are_params_locked(self): return self._core_service.are_params_locked()
    def set_snapshot_collector(self, collector): return self._core_service.set_snapshot_collector(collector)
```

#### R3-1: 异常分层策略制定与实施

**策略定义**:

| 异常层级 | 捕获范围 | 失效处理 | 适用场景 |
|:---|:---|:---|:---|
| **L0 致命** | `except (ImportError, AttributeError, TypeError)` | `raise RuntimeError` 硬阻断 | 核心门控（CascadeJudge、风控检查、交易执行） |
| **L1 严重** | `except (ValueError, KeyError, RuntimeError)` | 记录error + 标记结果无效 | 评判组件、验证器、checker |
| **L2 可降级** | `except Exception` | 记录warning + fallback默认值 | 辅助指标、非关键评分、监控采集 |
| **L3 可忽略** | `except Exception` | 记录debug + 跳过 | 日志、遥测、非功能性采集 |

**实施优先级**:
1. `strategy_judgment/` 102处 → 按上述分层改造（预估L0: 8处, L1: 35处, L2: 45处, L3: 14处）
2. `param_pool/` 250处 → 同上（预估L0: 15处, L1: 80处, L2: 120处, L3: 35处）
3. 每改一处必须添加对应断言测试

#### R6-2: CascadeJudge gate方法与judge()逻辑同步

**修改文件**: `evaluation/cascade_judge.py:860-879`

**修改方案**: 将 `gate_l2_ml_model` 拆分为3个sub-gate（与judge()一致），将 `gate_l3_shadow_stealth` 扩展为3个约束（与judge()一致）

```python
def gate_l2_ml_model(self, metrics: BacktestMetrics) -> List[GateReport]:
    """L2 ML模型 ← 三角验证+统计检验(安全壳) — 与judge()第二关逻辑同步"""
    gates = []
    gates.append(self._check_gate(
        name="L2-索提诺验证", value=metrics.sortino_ratio,
        threshold=self.thresholds["sortino"], hard_block=True,
        reason_template="索提诺{value:.2f} < 阈值{threshold}，下行风险失控",
    ))
    gates.append(self._check_gate(
        name="L2-卡玛验证", value=metrics.calmar_ratio,
        threshold=self.thresholds["calmar"], hard_block=True,
        reason_template="卡玛{value:.2f} < 阈值{threshold}，回撤路径不可承受",
    ))
    gates.append(self._check_gate(
        name="L2-夏普验证", value=metrics.sharpe_ratio,
        threshold=self.thresholds["sharpe"], hard_block=False,
        reason_template="夏普{value:.2f} < 阈值{threshold}，整体风险收益比偏低",
    ))
    return gates

def gate_l3_shadow_stealth(self, metrics: BacktestMetrics) -> List[GateReport]:
    """L3影子隐形 ← 小资金约束+影子EV(驾驶座) — 与judge()第三关逻辑同步"""
    gates = []
    gates.append(self._check_constraint(
        name="L3-连续亏损约束", value=metrics.max_consecutive_losses,
        threshold=self.constraints["consecutive_losses"], direction="max",
        reason_template="连续亏损{value}次 > 阈值{threshold}次",
    ))
    gates.append(self._check_constraint(
        name="L3-横盘天数约束", value=metrics.max_flat_period_days,
        threshold=self.constraints["flat_days"], direction="max",
        reason_template="横盘{value}天 > 阈值{threshold}天",
    ))
    gates.append(self._check_constraint(
        name="L3-保证金约束", value=metrics.peak_margin_used,
        threshold=self.constraints["margin_ratio"], direction="max",
        reason_template="保证金峰值{value:.1%} > 阈值{threshold:.1%}",
    ))
    return gates
```
