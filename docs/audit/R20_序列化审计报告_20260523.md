## 维度1：JSON序列化正确性

### 问题1.1（P0）：`default=str` 导致 datetime 类型信息不可逆丢失

**涉及文件**：多个文件，共14处使用 `default=str` 模式  
**代表性位置**：

| 文件 | 行号 | 写法 |
|:---|:---|:---|
| `order_service.py` | 754 | `json.dumps(entry, ensure_ascii=False, default=str)` |
| `quant_services.py` | 85 | `json.dumps(value, default=str, ensure_ascii=False)` |
| `ui_service.py` | 642, 705, 1008 | `json.dumps(..., default=str)` |
| `ProductionQuantSystem.py` | 456, 461 | `json.dumps(status, ..., default=str)` |
| `参数池/optuna_multiobjective_search.py` | 818, 844 | `json.dumps(..., default=str)` |
| `策略评判/strategy_judgment_engine.py` | 125 | `json.dumps(self.to_dict(), ..., default=str)` |
| `参数池/enhanced_phase_scan.py` | 871, 877, 886, 891 | `json.dump(..., default=str)` |
| `策略评判/backtest_integration_hooks.py` | 447 | `json.dump(..., default=str)` |

**问题描述**：
- `default=str` 对所有非JSON原生类型统一调用 `str()`，包括 `datetime`、`Decimal`、`numpy` 标量等
- datetime 调用 `str()` 输出 `"2026-05-24 10:30:00"`（Python repr格式），而非ISO 8601标准格式。这种格式：
  - 无法在读取时自动还原为 datetime 对象（`datetime.fromisoformat` 无法识别空格分隔的格式）
  - 丢失时区信息
  - 各平台不一致（Windows vs Linux 的 `str()` 可能有微妙差异）
- numpy float 调用 `str()` 输出如 `"3.141592653589793"`，精度可恢复但格式不可控
- 这些数据被读出（`json.loads`）后，所有 datetime 和特殊类型都变为纯字符串，**需要调用方手动重新解析，但没有任何元数据标记原始类型**

**影响的系统**：生产系统、测试系统、评判系统  
**关键影响路径**：
- `order_service.py:754`：交易日志中的时间戳无法还原为 datetime，影响交易审计
- `quant_services.py:85`：持久化的配置值中的 datetime 无法还原
- `ui_service.py:1008`：状态快照在 queue 中传输后无法作为 datetime 还原

### 问题1.2（P1）：`storage_core.py` 的 `_json_default` 回退到 `str()` 导致类型丢失

**文件**：`storage_query.py`，行号 308–311  
```python
@staticmethod
def _json_default(value: Any) -> Any:
    if isinstance(value, datetime):
        return value.isoformat()
    return str(value)
```

**问题描述**：
- 该方法只处理了 `datetime` 类型（转为 ISO 格式），其他非JSON原生类型（如 `numpy.int64`、`numpy.float64`、`Decimal`、`date`、`timedelta` 等）全部通过 `str()` 降级
- `storage_core.py` 的 `_save_signal_impl`（行1249）、`_save_underlying_snapshot_impl`（行1267）、`save`（行1375）均使用此 `_json_default`
- 信号数据中可能存在 `numpy` 标量（从 pandas DataFrame 计算而来），这些值在被序列化后永久变为字符串
- `_json_default` 处理 `datetime` 使用 `isoformat()`，但读取端（`storage_query.py:1009` 的 `json.loads`）不做任何类型还原，datetime 读回后是 ISO 字符串而非 datetime 对象

**影响的系统**：生产系统（信号/快照持久化链路）

### 问题1.3（P1）：`event_bus.py` 事件持久化截断并损坏JSON结构

**文件**：`event_bus.py`，行号 458–467  
```python
event_data = {
    'type': event_type,
    'timestamp': timestamp,
    'data': str(event)[:500]  # 危险：简单截断500字符
}
```

**问题描述**：
- `str(event)[:500]` 对事件对象做Python repr后直接截取前500字符
- 截断后的字符串不再有任何结构化信息，无法解析回原始事件对象
- 如果事件对象包含换行或特殊字符，`str()` 可能产生非法的JSON值（如未转义的引号），导致整个JSON失效
- 此截断是单向转换，完全不可逆——持久化等于信息销毁

**影响的系统**：生产系统（事件追溯）

### 问题1.4（P1）：`storage_core.py` WAL序列化使用 `default=str` 但预检用 `json.dumps` 做可序列化测试

**文件**：`storage_core.py`，行号 758–773

```python
    args_data = task[1] if len(task) > 1 else None

        wal_entry.append(args_data)
    except (TypeError, ValueError):
        wal_entry.append(str(args_data))   # 失败则直接用str()

```

**问题描述**：
- 预检使用 `json.dumps(args_data, default=str)` 测试是否可序列化，但预检成功仅表示 `str()` 可以转换，不代表参数能被正确还原
- 实际的WAL写入（行780）又再次使用 `json.dumps(wal_entry, ..., default=str)` 序列化整个entry，形成双层 `default=str` 转换
- WAL恢复（行747）使用 `json.loads` 读回，所有复杂类型已永久变为字符串

**影响的系统**：生产系统（WAL恢复链路）

### 问题1.5（P1）：`quant_services.py` 的 `_json_serializer` 处理 numpy 但最终回退到 `str()`

**文件**：`quant_services.py`，行号 163–171  
```python
@staticmethod
def _json_serializer(obj: Any) -> Any:
    if isinstance(obj, np.ndarray):
        return obj.tolist()
    if isinstance(obj, (np.integer,)):
        return int(obj)
    if isinstance(obj, (np.floating,)):
        return float(obj)
    return str(obj)  # 兜底：所有其他类型→str()
```

**问题描述**：
- numpy 处理较为完善（ndarray→list, integer→int, floating→float）
- 但 `np.bool_`、`np.datetime64`、`np.timedelta64` 等 numpy 标量类型**未覆盖**，会落入 `str()` 分支
- `datetime.date`、`datetime.timedelta`、`Decimal` 等 Python 标准库类型也未覆盖
- 保存（行85）和快照保存（行132）均使用此 serializer，读取端仅做 `json.loads`，不做类型还原

**影响的系统**：生产系统（配置持久化）

### 问题1.6（P1）：`json.dumps` 未处理 NaN/Infinity，可能触发 `ValueError`

**文件**：`strategy_tick_handler.py`，行号 614  
```python
f.write(_json.dumps(snapshot, ensure_ascii=False))
```

**问题描述**：
- tick snapshot 数据包含 `bid_price`、`ask_price`、`strike_price` 等数值字段
- 如果数据源产生 `NaN` 或 `Infinity` 值（如行权价计算失败），`json.dumps` 默认配置下会抛出 `ValueError`（因为JSON标准不支持 NaN/Infinity）
- 该处未设置 `default` handler，也未设置 `allow_nan=True`（Python 3.11+默认 `allow_nan=True` 但仍会输出 `NaN`/`Infinity` 字符串，导致非标准JSON）
- 虽然 `task_scheduler.py:1087-1094` 有 NaN 防护代码，但并不覆盖所有进入 `json.dumps` 的数据路径

**影响的系统**：生产系统（tick快照写入链）

### 问题1.7（P2）：`json.dumps` 输出 key 排序不统一

**文件**：多处（`config_service.py:437`, `params_service.py:909`, `quant_services.py:132`, `signal_service.py:662` 等）

**问题描述**：
- 大部分 `json.dumps`/`json.dump` 调用未使用 `sort_keys=True`
- 仅有 `shadow_strategy_engine.py:259` 和 `v7_meta_audit_v2.py:1317` 正确使用了 `sort_keys=True`
- key 顺序不稳定会导致：
  - 相同逻辑内容产生不同的JSON字符串，影响 hash 校验（如参数锁定校验）
  - diff 工具无法有效对比配置文件差异
  - git 提交产生无意义的 diff noise

**影响的系统**：三系统均有影响

### 问题1.8（P2）：`params_service.py` 环境变量参数解析有歧义

**文件**：`params_service.py`，行号 952–953  
```python
try:
    parsed = json.loads(value)
    self._params[param_key] = parsed
except json.JSONDecodeError as e:
    self._params[param_key] = value
```

**问题描述**：
- 尝试将环境变量值作为JSON解析，如果失败则当作原始字符串
- 这导致歧义：纯数字字符串 `"3.0"` 会被成功解析为 `float(3.0)`，而 `"S1"` 会被当作字符串 `"S1"`

### 问题1.9（P2）：`config_service.py` 的 `to_dict` 未处理 dataclass 中的非JSON类型

## 维度2：YAML序列化安全

### 问题2.1（P0）：`test_design_suite.py` 使用 `yaml.dump()` 而非 `yaml.safe_dump()`

**文件**：`参数池/test_design_suite.py`，行号 746, 752  
```python
yaml.dump(config, f, allow_unicode=True, default_flow_style=False)
yaml.dump(self._history, f, allow_unicode=True)
```

**问题描述**：
- `yaml.dump()` 是 <mark>不安全版本</mark>，会输出 Python 对象标签（如 `!!python/object:...`）
- 这些标签在下次 `yaml.safe_load()` 时无法解析，导致加载失败
- 但该文件的读取端（行724）使用的是 `yaml.safe_load()`，**读取和写入使用的API不匹配**——如果 `dump` 产生了 Python 标签，`safe_load` 将无法加载
- 当前可能因为 config 内容不包含自定义 Python 对象而侥幸运行，但这是定时炸弹

**影响的系统**：测试系统（配置版本控制）

### 问题2.2（P2）：所有 YAML 加载正确使用 `yaml.safe_load()`，但缺少布尔值歧义处理

**文件**：13处 `yaml.safe_load()` 调用点（`config_params.py:527`, `tvf_param_loader.py:92`, `state_param_manager.py:117`, `strategy_judgment_engine.py:508`, `enhanced_phase_scan.py:285`, `optuna_multiobjective_search.py:640`, `shadow_strategy_engine.py:300`, `cascade_judge.py:277`, `test_design_suite.py:724`, `sensitivity_analysis.py:58`）

**问题描述**：
- 全链路一致使用 `yaml.safe_load()` ✅ 这是正确做法
- 但 YAML 1.1 规范将 `yes/no/on/off/y/n/ON/OFF/Y/N` 均解析为布尔值
- 例如：如果 `state_param_sets.yaml` 中包含 `option_type: no`（表示无期权类型），YAML会将其解析为 `False` 而非 `"no"`
- 当前 YAML 配置文件 (`parameter_attribute_matrix.yaml`, `state_param_sets.yaml`, `tvf_params.yaml`, `plr_thresholds.yaml`, `cascade_config.yaml`, `cascade_threshold_grid.yaml`, `judgment_scoring_config.yaml`) 中均为数值型配置，暂未触发此问题，但缺少防护

### 问题2.3（P2）：`yaml.dump` 输出未设置 `sort_keys=True`

**文件**：`参数池/test_design_suite.py`，行号 746, 752

**问题描述**：
- `yaml.dump` 输出的 key 顺序依赖于 Python 字典插入顺序（3.7+有序），但不保证逻辑排序
- 版本控制工具会产生不必要的 diff
- 与 JSON 维度同样的风险

## 维度3：Parquet/Arrow序列化

### 问题3.1（P0）：`preprocess_ticks.py` Parquet 路径SQL注入风险

**文件**：`参数池/preprocess_ticks.py`，行号 796–798  
```python
con.execute(f"""
    INSERT INTO minute_data
    SELECT * FROM read_parquet('{parquet_path}')
""")
```

**问题描述**：
- `parquet_path` 直接通过 f-string 嵌入 SQL 字符串 `read_parquet('{parquet_path}')`
- 虽然 `parquet_path` 在此处来自 `tmp_dir` + 受控的 `symbol` 拼接，但如果 `symbol` 来自外部输入（命令行参数），攻击者可构造 `symbol` 为 `test'; DROP TABLE minute_data; --`
- 对比 `ds_schema_manager.py:46-48` 提供了 `_validate_parquet_path` 进行路径校验（使用正则 `^[a-zA-Z0-9_./\\:\-]+$`），但 `preprocess_ticks.py` 未使用此校验
- `duckdb_tick_storage.py:148` 有同样的问题：`FROM read_parquet('{parquet_path}')`

**影响的系统**：测试系统（数据预处理链）

### 问题3.2（P1）：Parquet round-trip 后 datetime 时区丢失

**文件**：`ds_schema_manager.py`，行号 72–75；`preprocess_ticks.py`，行号 785

**问题描述**：
- `ds_schema_manager.py` 从 Parquet 加载时：`CAST(timestamp AS DATE)` 将 datetime 截断为日期，丢失时分秒和时区
- `preprocess_ticks.py:785` 写入 Parquet 时使用 `index=False, compression="zstd"`，未指定 `partition_cols` 或 `schema` 参数，由 pandas 自动推断
- 如果源数据 timestamp 是 timezone-naive，读取后的 DuckDB 列也是 timezone-naive；如果源数据有时区，可能导致时区信息丢失
- pandas `to_parquet` 默认使用 PyArrow 引擎，datetime 被保存为 `timestamp[ns]`（无时区），时区信息会被丢弃

### 问题3.3（P1）：Parquet schema evolution 未处理

**文件**：`ds_schema_manager.py`，行号 70–87；`ds_data_writer.py`，行号 546–599

**问题描述**：
- `_load_or_create_table` 使用 `read_parquet(..., union_by_name=false)` 加载数据
- `union_by_name=false` 意味着列按位置匹配而非列名匹配——如果新版本 Parquet 文件的列顺序与旧表不同，**数据会错位写入错误的列**
- `incremental_load` 使用参数化查询 `read_parquet(?)`，但插入时使用 `batch_insert_ticks(dict_list, use_arrow=True)`，如果新 Parquet 多了新列，旧表缺少对应列，插入会失败
- 没有 schema 迁移机制来处理列的新增/删除/重命名

**影响的系统**：生产系统（数据加载链）

### 问题3.4（P1）：`ds_schema_manager.py` 的 `_validate_parquet_path` 校验仅为局部使用

**文件**：`ds_schema_manager.py`，行号 46–49

**问题描述**：
- `_validate_parquet_path` 是正确的安全实践 ✅
- 但该函数仅在 `_load_or_create_table`（行70）中调用
- `ds_realtime_cache.py` 的 `_resolve_parquet_path` 未使用此校验
- `ds_data_writer.py:584` 的 `read_parquet(?)` 使用参数化查询（`?` 占位符），这是安全的 ✅
- 但两处 SQL f-string 注入问题（问题3.1）未使用此校验

**影响的系统**：生产系统（Realtime Cache 路径）

### 问题3.5（P2）：`market_snapshot_collector.py` 导出 Parquet 未指定 schema

**文件**：`策略评判/market_snapshot_collector.py`，行号 951–961  
```python
df.to_parquet(file_path, index=False)
```

**问题描述**：
- 未指定 `schema` 参数，数据类型由 pandas 自动推断
- 如果 DataFrame 列类型在不同运行中不一致（如某次运行某列全为 None 导致类型为 object），重新读取时可能得到与预期不同的 dtype
- 应使用显式 PyArrow schema 确保一致性

### 问题3.6（P2）：Parquet 路径竞态条件

**文件**：`参数池/preprocess_ticks.py`，行号 783–785  
```python
tmp_path = os.path.join(tmp_dir, f"preprocess_{symbol}_{os.getpid()}.parquet")
result.to_parquet(tmp_path, index=False, compression="zstd")
```

**问题描述**：
- 使用 `os.getpid()` 作为临时文件名的一部分以避免并发冲突 ✅
- 但 `to_parquet` 写入非原子操作——如果写入过程中进程崩溃，会留下不完整的 Parquet 文件
- 后续 `_merge_temp_file`（行791）直接读取该文件，可能因文件不完整而失败或加载错误数据
- 应使用 `tempfile.NamedTemporaryFile` 写入后原子 `os.replace` 到目标位置

### 问题4.3（P1）：WAL 格式无版本标记，读取端无兼容策略

**文件**：`subscription_manager.py`，行号 287–339（恢复）；行号 476（写入）  
**文件**：`storage_core.py`，行号 740–817（SpillWAL）

**问题描述**：
- WAL 的每条记录是纯 JSON 行，未在记录中添加 `format_version` 字段
- 如果 WAL 记录结构变化（如新增字段），旧恢复代码读取新WAL记录会：

## 维度5：配置文件序列化

### 问题5.1（P1）：`config_service.py` `save_to_file` 无 round-trip 保障

**文件**：`config_service.py`，行号 433–443  
```python
def save_to_file(self, config_path: str) -> bool:
    with open(path, 'w', encoding='utf-8') as f:
        json.dump(self.to_dict(), f, indent=2)
```

**问题描述**：
- `json.dump` 无 `default` handler — 如果任何 dataclass 字段是 `Path` 对象（实际路径属性可能使用 `pathlib.Path`），会直接抛出 `TypeError`
- 无 `ensure_ascii=False` — Unicode 路径会被转义
- 无 `sort_keys=True` — 字典 key 顺序不可预测，导致配置文件每次保存都可能产生不同内容
- 保存与加载不对等：加载使用 `_update_from_dict`（行289），通过 `hasattr+setattr` 直接设置属性，不进行类型检查。例如 JSON 中 `"log_level": 10` 会被接受，但代码期望的是字符串 `"DEBUG"`

**影响的系统**：生产系统

### 问题5.2（P1）：`config_service.py` `_load_config_file` 从 JSON 还原时不做类型验证

**文件**：`config_service.py`，行号 272–287, 289–310

**问题描述**：
- `_update_from_dict` 使用 `hasattr+setattr` 模式直接设置属性值
- JSON 中的数字 3 会被赋值给期望 bool 类型的字段（如 `enable_auto_stop_loss`），Python 中 `bool(3) == True`，不会报错但语义不同
- JSON 中的字符串异常的配置值（如 `"max_position_limit": "100"`）会被接受为字符串而非整数，后续使用该值时可能产生 `TypeError`
- 没有 JSON Schema 验证层

### 问题5.3（P2）：`params_service.py` `save_params` 不做 round-trip 验证

**文件**：`params_service.py`，行号 895–921

**问题描述**：
- `save_params` 写入后直接更新内存缓存（`self._param_cache[path] = params`）
- 不执行 `save → load → compare` 验证循环
- 如果磁盘写入部分成功（如磁盘满导致截断），内存缓存与服务文件中数据不一致

### 问题5.4（P2）：YAML round-trip 行为不一致

**文件**：`参数池/test_design_suite.py`，行号 724, 746

**问题描述**：
- 读取时使用 `yaml.safe_load()` → 解析 YAML 为 Python dict
- 写入时使用 `yaml.dump()` → 可能产生 Python 标签
- Round-trip：`yaml.safe_load(yaml.dump(data))` 可能无法还原原始数据（如果 dump 产生了 `!!python/...` 标签，safe_load 会拒绝或报错）

## 维度6：数据帧序列化

### 问题6.1（P1）：`query_service.py` CSV 导出丢失索引

**文件**：`query_service.py`，行号 1206, 1253  
```python

```

**问题描述**：
- K线和Tick数据导出为CSV时使用 `index=False`，丢失原始行索引
- 如果原始数据按时间戳索引，丢失后无法按时间高效查询
- 对Tick数据尤其重要——tick 的时间戳索引是唯一稳定标识
- 建议使用 `index=True, index_label='timestamp'` 保留索引

### 问题6.2（P2）：`market_snapshot_collector.py` 导出 Parquet 未保留 DataFrame 索引

**文件**：`策略评判/market_snapshot_collector.py`，行号 961  
```python
df.to_parquet(file_path, index=False)
```

**问题描述**：
- 使用 `index=False` 但 `to_flat_dict()` 生成的数据中可能包含需要作为索引的快照本地UUID之类的标识
- Parquet 本身支持索引/排序键（Row Group statistics），不使用索引意味着后续查询效率低

### 问题6.3（P2）：`preprocess_ticks.py` Parquet 写入未指定 schema

**文件**：`参数池/preprocess_ticks.py`，行号 785  
```python
result.to_parquet(tmp_path, index=False, compression="zstd")
```

**问题描述**：
- 未显式指定 schema，列类型由 pandas 推断
- 如果 tick 数据中某个列在某些 chunk 中为 `None`（如 `volume=0` 导致数值型被推断为 object），schema 在不同文件间不一致
- DuckDB 读取时使用 `union_by_name=true`（`_merge_temp_file` 未设置此参数，使用默认值），列类型不一致可能导致读取失败

### 问题6.4（P2）：`external_validation_pipeline.py` CSV 导出无编码指定

**文件**：`参数池/L1参数量化/external_validation_pipeline.py`，行号 285  
```python

```

**问题描述**：

### 问题7.2（P1）：`order_service.py` 交易日志 datetime 格式差异
