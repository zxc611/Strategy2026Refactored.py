## 维度1：循环引用与GC

### 1.1 【P2】系统无 `__del__` 定义 —— 正面确认

- **文件**: 所有 72 个 .py 文件
- **行号**: 全局搜索 `__del__` 返回 0 结果
- **描述**: 全系统没有任何类定义 `__del__` 方法。在 Python 中，`__del__` 会阻止 GC 回收包含循环引用的不可达对象（Python 3.4+ 虽有改进，但 `__del__` 仍会阻碍分代 GC）。系统主动避免了这一陷阱。
- **影响**: 正面。循环引用仍可被分代 GC 正常回收。
- **影响的系统**: 生产/测试/评判

### 1.2 【P1】ServiceContainer → 服务 → ServiceContainer 潜在环

- **文件**: `service_container.py` 行号 ~35-120
- **描述**: `ServiceContainer.__init__()` 创建 `self._data_service`、`self._governance_engine` 等内部服务实例，同时将自己作为 `container` 引用传递给这些子服务（如 StrategyEcosystem 构造函数接收 `container`）。如果子服务保持对 container 的强引用，则形成双向引用环。
- **风险**: 若 `ServiceContainer` 的 `shutdown()` 未被调用，该环阻止引用计数降为 0，依赖分代 GC 回收，延迟可能在内存紧张的 tick 环境中导致短时浪费。
- **影响的系统**: 生产

### 1.3 【P1】StrategyEcosystem ↔ BoxSpringStrategy ↔ GovernanceEngine 三角环

- **文件**: `strategy_ecosystem.py` 行号 ~250-420, `box_spring_strategy.py` 行号 ~93-170, `governance_engine.py` 行号 ~535-560
- **描述**: `StrategyEcosystem` 持有 `self._box_detector` (BoxSpringStrategy 实例)，`BoxSpringStrategy` 通过全局单例 `get_governance_engine()` 引用 GovernanceEngine，`GovernanceEngine` 又可通过 `get_ecosystem()` 反向引用 StrategyEcosystem。加上各服务通过 `get_box_spring()` 等多路径交叉引用，形成复杂的多节点循环引用网。
- **风险**: 三方循环引用。系统重启/初始化阶段若旧实例未被显式清理，可能导致旧对象驻留。
- **影响的系统**: 生产

### 1.4 【P2】ShadowStrategyEngine 持有 SnapshotCollector 扩展引用

- **文件**: `shadow_strategy_engine.py` 行号 ~1298
- **描述**: `ShadowStrategyEngine` 创建 `self._snapshot_collector`，`snapshot_collector` 通过全局 `_shadow_engine` 变量持有反向引用。若 system 在 Restart 时调用 `shutdown()` 先置 None 再创建新实例，则不存在环问题；但若忘记清理全局变量而直接创建新对象，旧对象因全局变量的强引用无法回收。
- **影响的系统**: 生产

### 1.5 【P2】EventBus._subscribers 强引用导致订阅者无法自动释放

- **文件**: `event_bus.py` 行号 ~50-65
- **描述**: `EventBus` 的 `_subscribers` 字典以 `Dict[str, List[Callable]]` 形式存储，其中 Callable 通常是定义了 `__call__` 的对象方法（bound method），这导致对订阅者对象的强引用。当订阅者（如 Strategy、Service）生命周期结束后，其 bound method 仍被 EventBus 持有，形成引用链：`EventBus → bound method → subscriber instance`。
- **风险**: 在长期运行中，若订阅者被替换而不调用 `unsubscribe()`，旧对象无法被回收。
- **影响的系统**: 生产

## 维度2：弱引用使用

### 2.1 【正面】TickHandlerMixin 使用 weakref.ref 避免 atexit 泄漏

- **文件**: `strategy_tick_handler.py` 行号 ~87-91
- **描述**: `TickHandlerMixin` 在注册 `atexit` 回调时，使用 `weakref.ref(self)` 避免 atexit 强引用导致对象在进程退出前无法释放。
- **影响**: 正面。这是生产级别的正确做法。
- **影响的系统**: 生产

### 2.2 【正面】SingletonManager 用 weakref.ref 管理注册表

- **文件**: `quant_services.py` 行号 370, 383
- **描述**: `SingletonManager.register()` 使用 `weakref.ref(instance)` 存储服务实例，`shutdown()` 时通过 `ref()` 获取对象。当引用失效时自动跳过。
- **影响**: 正面。SingletonManager 本身不会阻止被管理对象的回收。
- **影响的系统**: 生产

### 2.3 【正面】ProductionQuantSystem 使用 weakref.ref 管理全局引用

- **文件**: `ProductionQuantSystem.py` 行号 ~322
- **描述**: `set_reference()` 使用 `weakref.ref(self)` 而非直接赋值全局变量，避免模块级全局变量阻止实例回收。
- **影响**: 正面。设计优秀的全局实例管理模式。
- **影响的系统**: 生产

### 2.4 【正面】strategy_lifecycle_mixin 弱引用重试回调

- **文件**: `strategy_lifecycle_mixin.py` 行号 ~754-755
- **描述**: 重试回调 `_schedule_retry_checker_init` 中使用 `weakref.ref(self)` 确保重试时对象若已销毁则自动跳过。
- **影响**: 正面。
- **影响的系统**: 生产

### 2.5 【P1】EventBus 订阅者使用强引用 —— 缺少弱引用机制

- **文件**: `event_bus.py` 行号 ~64
- **描述**: `_subscribers: Dict[str, List[Callable]]` 使用强引用。对比 `TickHandlerMixin`、`SingletonManager` 等已经正确使用 weakref 的模块，EventBus 缺少弱引用保护是系统中最显著的弱引用缺失点。每次 `subscribe()` 调用的 bound method 会阻止订阅者实例回收。
- **风险**: 若一个策略被停止/替换但未调用 `unsubscribe()`，它持有的 DataFrame、tick 缓存、信号队列等全部无法释放。策略品种多个策略可导致内存持续膨胀。
- **建议**: `_subscribers` 改为 `Dict[str, List[weakref.WeakMethod]]`（Python 3.4+），`emit()` 时过滤已死的订阅者。
- **影响的系统**: 生产

### 2.6 【P1】get_global_event_bus() 全局变量直接赋值无弱引用

- **文件**: `event_bus.py` 行号 861, 870
- **描述**: `_global_event_bus` 模块级变量直接赋值为 `EventBus()` 实例而非 `weakref.ref`，与 `ProductionQuantSystem` 的 weakref 模式不一致。
- **影响的系统**: 生产

### 2.7 【P2】25+ 全局单例函数均使用直接赋值

- **文件**: 见下方维度 6.1 的完整清单
- **描述**: 所有 `get_xxx()` 全局单例函数均使用 `global _instance; _instance = Xxx()` 模式，无一处使用 weakref。虽然这些单例预期生命周期与进程相同，但在可选的 Restart/Reinitialize 场景下，旧实例无法立即被回收。
- **影响的系统**: 生产

## 维度3：对象池与复用

### 3.1 【正面】deque maxlen 复用模式已普遍采用

- **文件**: 
  - `event_bus.py` 行号 ~25: `_event_history: deque(maxlen=10000)`
  - `shadow_strategy_engine.py` 行号 ~25: 6 个 `deque(maxlen=10000)` 存储 trades
  - `strategy_ecosystem.py` 行号 ~25: 6 个 `deque(maxlen=10000)` 存储 trades
  - `ds_realtime_cache.py` 行号 ~25: `deque(maxlen=500)` 存储 ticks
- **描述**: deques 使用 maxlen 限定上限，超出部分自动丢弃释放，实现了滑动窗口式的对象复用。在 tick 驱动的系统中，这是高效的策略。
- **影响**: 正面。
- **影响的系统**: 生产

### 3.2 【P1】copy.deepcopy 被大量高频调用

- **文件/行号**:
  - `config_params.py` 行号 ~240-245: `get_cached_params()` 每次调用都 `copy.deepcopy()`，该函数被 strategy_ecosystem 高频调用
  - `shadow_strategy_engine.py` 行号 ~1298: 参数快照 `copy.deepcopy()`
  - `risk_service.py` 行号 ~2250-2282: 风控计算中 `copy.deepcopy()` 持仓列表
- **描述**: `copy.deepcopy()` 会创建全新的对象体系，每次调用消耗内存分配和 GC 追踪时间。参数配置表可能包含嵌套 dict/list，深拷贝开销不可忽视。
- **风险**: 在 tick 高频场景下（每秒数百 tick），`deepcopy` 可导致显著的内存碎片和分配压力。
- **建议**: 对不可变参数使用 `copy.copy()`（浅拷贝足够）；或使用 `frozendict` 等不可变结构替代 deepcopy。
- **影响的系统**: 生产

### 3.3 【P2】DuckDB 每 tick 临时表无对象池

- **文件**: `data_service.py` 查询逻辑（多行）
- **描述**: DuckDB SQL 查询每 tick 都在内存中创建临时结果集（Arrow/C 层的对象）。Python 侧接收为 `duckdb.DuckDBPyRelation`，查询后不显式关闭但依赖 Python GC。高频 tick 场景下，短命临时对象产生大量内存碎片和 GC 压力。
- **影响的系统**: 生产

### 3.4 【P2】dict/list 高频创建未复用

- **文件**: `strategy_tick_handler.py`、`event_bus.py` 等
- **描述**: Tick 处理循环中每次创建 `{"tick_time": ..., "price": ...}` 字典、"tick_update" 事件的 list 参数等小对象。这些小对象在 CPython 中生命周期极短（创建-使用-丢弃），但高频场景（每秒数百次）会导致内存分配器和 GC 的踩踏效应。
- **影响的系统**: 生产

## 维度4：大对象与内存碎片

### 4.1 【P1】RealTimeCache FULL_CAPTURE 模式下 tick_sequences 无上限

- **文件**: `ds_realtime_cache.py` 行号 ~25-40
- **描述**: `RealTimeCache._tick_sequences` 是一个 `Dict[str, deque]` 结构。在 `FULL_CAPTURE` 模式下，`maxlen` 可能设置为较大的值（如缓存时长 * tick频率），或未设置上限。长时间运行下，每个品种的 tick 序列不断增长。
- **风险**: 同时监控 50+ 品种，每品种每秒 2 tick，运行 8 小时后单品种可达 57600 条记录。50 品种 × 57600 条 × 200 字节 ≈ 576MB。
- **影响的系统**: 生产

### 4.2 【P1】ShadowStrategyEngine 权益曲线无上限截断

- **文件**: `shadow_strategy_engine.py` 行号 ~20-30（_equity_curve deque 声明）
- **描述**: 6 个 trade deques 有 `maxlen=10000` 上限，但权益曲线 (`_equity_curve` 相关字段) 的存储限制需要确认。若权益曲线存储 points 远多于 trades 数量（每次 tick 更新一次 vs 每次交易更新一次），则需要独立上限。
- **风险**: 权益曲线点数是交易数的 10-100 倍（每 tick 记录），无上限则持续增长。
- **影响的系统**: 生产

### 4.3 【P1】EventBus.get_event_history() 每次创建整个历史副本

- **文件**: `event_bus.py` 行号 ~60-80
- **描述**: `get_event_history(event_type)` 调用 `copy.deepcopy(list(self._event_history))` 或遍历过滤后返回副本。若历史 > 10000 条且嵌套 dict 数据，每次创建列表副本消耗大量内存。
- **风险**: 若诊断/监控模块定时拉取事件历史，每日产生数百次完整副本，内存碎片严重。
- **影响的系统**: 生产

### 4.4 【P2】DuckDB max_memory = 75% 总 RAM 可能过激进

- **文件**: `ds_db_connection.py` 行号 73-76
- **描述**: `_get_duckdb_memory_limit()` 配置 DuckDB `max_memory` 为总 RAM 的 75%。DuckDB 作为进程内嵌入式数据库，会直接分配此内存作为缓存。在同一台机器上运行多个进程（如评判系统 + 生产系统）时，内存争抢严重。
- **影响的系统**: 生产/评判

### 4.5 【P2】NumPy 数组/DuckDB 查询结果无显式 del

- **文件**: `data_service.py`、`storage_core.py` 等多处
- **描述**: DuckDB 查询产生 `DuckDBPyRelation` 和 `DataFrame` 对象，赋值给临时变量后立即使用。使用后变量超出作用域但依赖引用计数回收，在高频 tick 场景下可能累积大量待回收对象。无 `del` 显式释放。
- **影响的系统**: 生产

## 维度5：闭包与延迟绑定

### 5.1 【P2】task_scheduler.py Lambda 捕获循环变量

- **文件**: `参数池/task_scheduler.py` 行号 ~769-778
- **描述**: 
  ```python
  for name in score_names:
      self._score_fns[name] = lambda s=Strategy, n=name: self._compute_score_v2(...)
  ```
  使用了 `n=name` 的默认参数技巧正确绑定，但 `Strategy` 未作为默认参数传入，依赖外部作用域。
- **影响的系统**: 评判

### 5.2 【P1】EventBus subscribe 闭包包装函数

- **文件**: `event_bus.py` 行号 ~78-79
- **描述**: `subscribe()` 内部 `filter_func` 包装了 `callback`，形成闭包：
  ```python
  wrapped = lambda event, cb=callback, filt=filter_func: (
      cb(event) if filt is None or filt(event) else None
  )
  ```
  该闭包捕获 `callback`（通常是 bound method），与维度 2.5 形成联动——即使 unsubscribe 移除了 wrapped，bound method 仍被该闭包的 `__closure__` 保持，但 EventBus 的列表移除已解除引用。
- **影响的系统**: 生产

### 5.3 【P2】get_global_event_bus() Lambda 默认参数小问题

- **文件**: `event_bus.py` 行号 ~856
- **描述**: `get_global_event_bus()` 中 `lambda: EventBus()` 作为延迟初始化，挂接在 `_global_event_bus` 上。注：后续赋值 `_global_event_bus = bus` 后旧 lambda 被替换，无泄漏。
- **影响的系统**: 生产

- **影响的系统**: 生产

### 6.4 【P2】width_cache.py 四级缓存永久增长

- **文件**: `width_cache.py`
- **描述**: 箱体宽度缓存使用多级缓存策略（tick/近端/中期/远端），各级缓存通过字典存储。每个品种-周期的缓存条目在运行时不断增加。股票品种数量 × 周期数量 × 缓存层级数量可导致条目膨胀。
- **影响的系统**: 生产

### 6.5 【P2】shared_utils.py ShardRouter 模块级单例无清理

- **文件**: `shared_utils.py` 行号 334
- **描述**: `_GLOBAL_SHARD_ROUTER` 作为分片路由器，存储品种→分片的映射表。品种数量增加时映射表增长，但从未压缩或清理。
- **影响的系统**: 生产

## 维度8：引用计数与gc交互

### 8.1 【正面】无 gc.disable() 调用

- **文件**: 全部 72 个 .py 文件
- **描述**: 全局搜索 `gc.disable` 返回 0 结果。系统没有在任何位置禁用垃圾回收，分代 GC 全程正常运行。
- **影响**: 正面。
- **影响的系统**: 生产/测试/评判

### 8.2 【正面】gc.collect() 仅在两处明确调用

- **文件/行号**:
  - `quant_services.py` 行号 ~420: `SingletonManager.shutdown_all()` 中调用 `gc.collect()`，确保所有 singleton 关闭后立即触发 GC
  - `ProductionQuantSystem.py` 行号 ~347: `shutdown_quant_system()` 中调用 `gc.collect()` 两次，确保系统完全清理后回收所有残留对象
- **描述**: 这两处 `gc.collect()` 调用都在系统 shutdown 流程中，用于加速循环引用对象的回收。不在热点路径上调用，不发生性能降级。
- **影响**: 正面。
- **影响的系统**: 生产

### 8.3 【P2】Python 默认 gc 阈值对 tick 高频场景可能偏低

- **文件**: 全局
- **描述**: Python 默认 GC 阈值 `(700, 10, 10)` 意味着第 0 代每 700 个分配触发一次收集。在 tick 高频场景下（每秒数百 tick × 每次 tick 分配多个对象），第 0 代 GC 收集频率可能达到每秒多次。每次分代 GC 会暂停所有线程执行 `tp_traverse` 遍历。
- **风险**: 高频率 GC 可能在市场剧烈波动（tick 最密集时）引入延迟抖动。
- **建议**: 考虑在 ProductionQuantSystem 初始化时适当调高 GC 阈值，例如 `gc.set_threshold(2000, 20, 20)`。
- **影响的系统**: 生产

### 8.4 【P1】SingletonManager.shutdown() 无 gc 收集

- **文件**: `quant_services.py` 行号 ~435
- **描述**: 对比 `SingletonManager.shutdown_all()`（有 `gc.collect()`），单个 `shutdown()` 方法没有调用 `gc.collect()`。若单例服务之间形成循环引用，单个 shutdown 后对象可能不能立即释放。
- **影响的系统**: 生产

## 结论

ali2026v3_trading 交易系统的内存模型整体设计**良好**，无 P0 级严重缺陷。系统的亮点包括：

1. **零 `__del__` 调用** —— 主动避免了 Python 内存管理的最大陷阱
2. **deque maxlen 普遍应用** —— 关键数据结构有明确的上限保护
3. **weakref 已多处使用** —— atexit、SingletonManager、ProductionQuantSystem 均正确使用弱引用
4. **正确的 gc.collect() 位置** —— 仅在 shutdown 流程调用，不在热路径
5. **文件 I/O 全部使用 with** —— 资源安全释放

主要风险集中在：
- **26 个全局单例永久存活**（最大的系统性风险）
- **EventBus 订阅者缺少弱引用保护**（最直接的泄漏点）
- **评判系统 _TRAIN_DATA/_TEST_DATA 无清理**（最大的 OOM 风险）
- **copy.deepcopy 高频调用**（最大的性能消耗点）

建议按上述优先级清单逐步整改，先解决 6.1、2.5、4.1、6.6 四个最高优先级问题。
