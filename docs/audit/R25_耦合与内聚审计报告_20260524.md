### 1.3 循环import链

#### P0级循环依赖链

**CYCLE-1: 参数池内部循环**
```
参数池/advanced_validation.py ←→ 参数池/task_scheduler.py
   └─ advanced_validation 懒加载 task_scheduler 的 8 个 validate_* 函数
   └─ task_scheduler 懒加载 advanced_validation 的 WalkForwardValidator
```
- **风险**: 任何一方修改函数签名，另一方编译期不报错，运行时才暴露
- **紧密度**: 高（advanced_validation 通过7个独立懒加载依赖 task_scheduler）

**CYCLE-2: 策略生态→风控→信号→策略生态**
```
strategy_ecosystem → risk_service(get_risk_service) → signal_service(get_signal_service) → strategy_ecosystem(间接)
```
- 具体链路: strategy_ecosystem._init_services() 懒加载 risk_service → risk_service.reset_risk_pause() 可能触发信号 → 信号操作可能更新 ecosystem 状态
- **风险**: 三方懒加载互相等待可能导致死锁

**CYCLE-3: 策略评判↔参数池分析**
```
策略评判/strategy_judgment_engine ←→ 参数池/task_scheduler
   └─ judgment_engine 懒加载 task_scheduler 的 validate_doomed_tests
   └─ task_scheduler 懒加载 策略评判/market_snapshot_collector
```

**CYCLE-4: 参数池增强扫描↔优化搜索**
```
参数池/enhanced_phase_scan ← 参数池/optuna_multiobjective_search
   └─ enhanced_phase_scan 懒加载 optuna 的 enrich_backtest_result 和 run_enhanced_optuna_optimization
   └─ optuna 懒加载 advanced_validation 的 WalkForwardValidator
```

**CYCLE-5: position_service ↔ t_type_service**
```
position_service.py → t_type_service.py(get_t_type_service)
t_type_service 可能反向引用 position_service 的类型定义
```

### 1.4 延迟import（`import inside function`）掩盖的耦合

统计：全项目共 **100+处** 函数内import。分布如下：

| 模块 | 函数内import数 | 掩盖的耦合 |
|------|---------------|-----------|
| **参数池/task_scheduler.py** | 25+ | 与risk_service, shadow_strategy_engine, mode_engine, state_param_manager, governance_engine, quant_core, 策略评判/* 等耦合 |
| **risk_service.py** | 12 | 与shadow_strategy_engine, greeks_calculator, data_service, governance_engine, state_param_manager, mode_engine, bayesian_shrinkage_life_estimator, position_service, order_service, scheduler_service, 参数池/task_scheduler 耦合 |
| **strategy_ecosystem.py** | 15 | 与config_params, config_service, governance_engine, mode_engine, risk_service, signal_service, state_param_manager, box_spring_strategy, position_service, 策略评判/strategy_judgment_engine, performance_monitor 耦合 |
| **策略评判/strategy_judgment_engine.py** | 12 | 与evaluation/*, 参数池/sensitivity_analysis, statistical_validation, advanced_validation, l2_optimizer, test_design_suite, task_scheduler 耦合 |
| **signal_service.py** | 7 | 与event_bus, config_params, config_service, health_check_api, mode_engine, greeks_calculator, risk_service, market_snapshot_collector 耦合 |
| **参数池/advanced_validation.py** | 9 | 与task_scheduler(8个validate_*), L1参数量化/v7_meta_audit_v2, governance_engine 耦合 |

**分析结论**: 延迟import本质上是一种"技术债务应对手段"，将本应在模块顶层声明的依赖推迟到运行时。虽然避免了Python的import循环错误，但代价是：
1. IDE/jedi无法自动补全
2. 编译器无法做静态检查
3. 依赖关系隐藏在业务逻辑中
4. 测试时mock困难

## 审计维度2: 跨模块直接属性访问

### 2.1 通过`getattr`绕过类型检查的私有属性访问

#### P0级发现

**FINDING-A1: risk_service大量通过getattr访问position_manager内部属性**
- 位置: `risk_service.py` 行476, 615, 976, 1022, 1097, 1775
- 模式: `getattr(self.position_manager, "positions", {})`、`getattr(self.position_manager, "_daily_pnl_history", None)`
- 耦合类型: 直接内部属性访问
- **风险**: position_manager内部数据结构变更（如`positions`重命名为`_position_registry`）将导致risk_service运行时崩溃，编译期无任何提示

**FINDING-A2: strategy_ecosystem通过getattr访问BoxSpringStrategy内部状态**
- 位置: `strategy_ecosystem.py` 行491
- 模式: `hasattr(bss, 'set_capital_scale') and callable(getattr(bss, 'set_capital_scale'))`
- 耦合类型: Duck typing + 属性探测
- **风险**: 依赖于BoxSpringStrategy的未文档化接口，接口变更无编译期保护

**FINDING-A3: governance_engine访问checker私有反馈通道**
- 位置: `governance_engine.py` 行486
- 模式: `channel = getattr(checker, "_feedback_channel", [])`
- 耦合类型: 直接访问私有属性（`_feedback_channel`）
- **风险**: checker内部实现变更直接破坏governance_engine功能

### 2.2 绕过公共API的访问模式

| 访问模式 | 出现频率 | 示例 |
|----------|---------|------|
| `getattr(obj, "_private_attr", default)` | 极高 | `risk_service.py` 的 `_daily_pnl_history` |
| `getattr(obj, "public_attr", default)` | 高 | `risk_service.py` 的 `positions` 字典 |
| `hasattr() + getattr()` 组合 | 中 | `strategy_ecosystem.py` 的 `set_capital_scale` |
| 直接 `obj._private_attr` 赋值 | 低（仅限自身） | `strategy_ecosystem.py:494` `bss._capital_scale = scale_str` |

**分析**: 最严重的问题是`risk_service.py`通过`getattr`访问`position_manager`的`positions`字典和`_daily_pnl_history`列表。这些属于position_service的内部数据结构，没有通过任何公共API暴露，直接依赖破坏了封装。

## 审计维度3: 接口隔离违反

### 3.1 import * 污染命名空间
**✅ 未发现 `import *` 使用** — 这是良好的实践。

### 3.2 接口过宽（胖接口）

**FINDING-B1: config_params.py导出过多**
- 导出: `DEFAULT_PARAM_TABLE`(~300行), `get_cached_params()`, `invalidate_param_cache()`, `load_param_table_from_file()`, `validate_params()`, `merge_env_overrides()`, `get_default_params_path()` 等
- 影响: 6个依赖方各自只使用其中1-2个功能，但每次import都会加载整个模块（包括300行的DEFAULT_PARAM_TABLE）

**FINDING-B2: strategy_ecosystem仅导入CapitalScale枚举却链入整个评判引擎**
- 位置: `strategy_ecosystem.py:43`
- 模式: `from ali2026v3_trading.策略评判.strategy_judgment_engine import CapitalScale`
- 影响: 虽然只导入1个枚举类，但strategy_judgment_engine.py的顶层import会触发其子模块import链：
  ```
  strategy_judgment_engine → strategy_behavior_diagnosis → resonance_turning_point_marker → market_snapshot_collector
  ```
  以及通过TYPE_CHECKING导入CycleResonanceOutput

**FINDING-B3: 参数池内模块互相导入但仅用少量函数**
- `参数池/advanced_validation.py` 从 `参数池/task_scheduler.py` 导入8个独立validate_*函数 → 支持了task_scheduler的全部import链
- `参数池/optuna_multiobjective_search.py` 从 `参数池/advanced_validation.py` 仅导入 `WalkForwardValidator` 和 `MultipleComparisonCorrector`

### 3.3 延迟import的接口碎片化

**FINDING-B4: task_scheduler多次重复import相同模块**
- `参数池/task_scheduler.py` 中对 `risk_service.get_risk_service` 的import出现4次（行1513, 1524, 1535, 1546），每次都在不同函数内重复import
- 应抽取为模块级条件导入

**关键发现**: 形成了一个**以EventBus为根、RiskService和StrategyEcosystem为核心的6节点耦合网**。每个节点都是全局单例，无法在测试中替换为mock。

### 5.3 单例测试困难

测试代码中出现的 `_reset_global_singletons()` 函数证实了测试困难：
- `test_spring_order.py`：每个测试方法前都调用 `_reset_global_singletons()`
- `test_resonance_order.py`：同样模式
- `test_hft_order.py`：同样模式
- `test_box_order.py`：同样模式

22个单例中仅有3个提供了reset方法，其余19个一旦创建就无法在进程内重置。

## 审计维度6: 隐式耦合

### 6.1 通过全局变量/模块级变量传递状态

**FINDING-C1: Module-Level可变缓存**
| 变量 | 定义位置 | 类型 | 访问范围 |
|------|---------|------|---------|
| `_param_table_cache` | config_params.py | `Dict[str, Any]` | 全局(通过get_cached_params) |
| `_DB_FILE_CACHE` | data_service.py | `str` | data_service及其子模块 |
| `_PARQUET_PATH_CACHE` | data_service.py | `str` | data_service及其子模块 |
| `_GLOBAL_SHARD_ROUTER` | shared_utils.py | `ShardRouter` | 全局(通过get_shard_router) |
| `_ui_global_root` | ui_service.py | `Any` | ui_service内部 |
| `_ui_global_running` | ui_service.py | `bool` | ui_service内部 |

**风险**: 这些缓存变量没有TTL管理（除`_param_table_cache`有300s TTL外），无法在运行时主动失效。

**FINDING-C2: 模块级logger实例**
几乎所有模块都定义 `logger = logging.getLogger(__name__)`，这些logger通过根logger配置耦合在一起。修改根logger配置会影响所有模块。

### 6.2 通过EventBus进行的隐式通信

EventBus作为解耦工具但同时创造隐式耦合：
- **发布者不关心订阅者**: 信号模块发布 `SignalEvent`，不知道哪些模块订阅
- **时序依赖**: 启动顺序必须保证订阅者注册先于事件发布
- **调试困难**: 无法从调用栈追踪跨EventBus的事件传播路径

**实际隐式通信链路**:
```
signal_service → EventBus.publish("SignalEvent") 
  → [risk_service.on_signal, order_service.on_signal, strategy_core.on_signal, ...]
```
这些订阅关系不在任何模块的import中体现，只能通过搜索 `subscribe` 调用找到。

### 6.3 通过文件系统/DuckDB进行的隐式通信

| 通信方式 | 写入方 | 读取方 | 风险 |
|---------|--------|--------|------|
| DuckDB存储 | ds_data_writer | data_service, query_service, storage_core | Schema版本不同步 |
| JSONL WAL日志 | subscription_manager | 重启恢复逻辑 | WAL格式变更需协调双方 |
| 参数锁文件 | Walk-forward | strategy_ecosystem | 文件存在性作为参数冻结信号 |

### 6.4 时序/初始化顺序耦合

**ServiceContainer._initialization_order** 强制定义了初始化顺序：
```
event_bus(L0) → config(L0) → params(L0) → storage(L1) → market_data(L1)
  → analytics(L2) → risk(L2) → order(L2) → signal(L2) → t_type(L2)
  → diagnosis(L2) → ui(L2) → strategy_core(L3)
```

任何模块在初始化顺序中插队或跳过将导致运行时错误。此顺序由 `ServiceContainer._dependencies` 字典硬编码，未通过依赖注入框架管理。

### 8.1 系统分层定义
