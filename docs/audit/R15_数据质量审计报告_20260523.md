### 维度1：Tick数据质量校验

#### 1.1 问题清单

##### P0-Q1: bid/ask价差校验完全缺失 —— 无效tick静默通过
- **文件**: `storage_query.py`, `strategy_tick_handler.py`, `subscription_manager.py`
- **行号范围**: 
  - `storage_query.py`: 162-187 (`_validate_tick`)
  - `strategy_tick_handler.py`: 282-310 (`_process_tick`)
  - `subscription_manager.py`: `on_tick` 回调链
- **问题描述**: `_validate_tick()`仅校验dict类型、instrument_id/price/timestamp存在性、负volume，**完全不检查bid/ask价格的合理性**。系统提取bid_price和ask_price（strategy_tick_handler.py:305-306）但仅用于OrderFlowBridge喂入，不校验 bid < ask。当交易所推送延迟/错误数据（bid >= ask）时，tick被静默纳入Bar合成，导致市场微观结构数据失真。手册V7.1第22.2节P0-2明确指出"Tick含NaN→聚合NaN→动态滑点退化为静态1.0bps"即为同类问题。
- **严重等级**: P0
- **影响系统**: 生产系统（live trading实时tick）、测试系统（preprocess_ticks批处理）

##### P0-Q2: price=0被_validate_tick允许通过 —— 潜在零价格Bar合成
- **文件**: `storage_query.py`
- **行号范围**: 180-182
```python
if not isinstance(last_price, (int, float)) or last_price < 0:
    logging.warning("save_tick 价格无效（结算tick可能为0）：%s", last_price)
    return True  # ⚠️ price=0 被允许通过！
```
- **问题描述**: 代码明确注释"结算tick可能为0"，price=0不拒绝而返回True。但在实际交易时段内，price=0的tick传入会产生Open=0/High=0/Low=0/Close=0的Bar。虽然`_filter_bars`（preprocess_ticks.py:613-633）会在批处理阶段过滤op/cl/hi/lo>0的Bar，但**实时路径没有等效保护**。`TickAggregator.update_tick`（quant_platform.py:97-98）拒绝price<=0，但该代码路径是否始终被调用存疑——`_KlineAggregator.update()`（query_service.py:165-209）无price<=0检查。
- **严重等级**: P0
- **影响系统**: 生产系统（实时Bar合成）

##### P1-Q3: tick价格跳空无检测 —— 异常波动静默纳入
- **文件**: `strategy_tick_handler.py`, `storage_core.py`
- **行号范围**: 全tick处理链路
- **问题描述**: 系统没有实现tick级别的价格跳空检测（如连续tick价格波动超过N个tick_size）。交易所偶发错误数据（如价格瞬间跳变至极端值）会直接进入K线合成，污染OHLC统计。preprocess_ticks.py的`validate_circuit_breaker_halts`（934行）是回测/评判系统的验证工具，**未纳入实时处理链路**。
- **严重等级**: P1
- **影响系统**: 生产系统（实时Bar合成）

##### P1-Q4: 成交量累加模式未校验 —— CTP累计量vs逐笔量混淆风险
- **文件**: `quant_platform.py`, `storage_core.py`
- **行号范围**:
  - `quant_platform.py`: 95-107 (`TickAggregator.update_tick`)
  - `storage_core.py`: 1008 (`process_tick`中agg.update调用)
- **问题描述**: `TickAggregator`支持volume_mode='delta'（CTP累计量转逐笔），但`_KlineAggregator.update()`直接使用传入的volume值累加（query_service.py:204: `self.volume += volume`）。两套系统对volume的语义假设不同：一套是delta模式、另一套是直接累加。如果上游推送的是CTP累计成交量而非逐笔成交量，`_KlineAggregator`会重复累加导致volume被严重放大。
- **严重等级**: P1
- **影响系统**: 生产系统（实时Bar合成）

##### P2-Q5: 无tick数据质量标记 —— 无法追溯数据质量时间线
- **文件**: `ds_realtime_cache.py`, `strategy_tick_handler.py`
- **行号范围**: 全tick采集路径
- **问题描述**: tick数据没有质量标记字段（如is_synthetic、quality_score、source_vendor）。无法区分正常tick、结算tick、回放tick、补传tick，也无法事后审计数据质量。手册要求`_spread_quality`和`_option_metadata_quality`标记（见P0-2/P0-4），但**仅存在于preprocess_ticks.py批处理**，未在实时tick层面打标。
- **严重等级**: P2
- **影响系统**: 三系统

##### P2-Q6: RealTimeCache的tick_entry缺少bid/ask有效性校验
- **文件**: `ds_realtime_cache.py`
- **行号范围**: 194-198
```python
tick_entry = {
    'internal_id': internal_id, 'price': price, 'timestamp': timestamp,
    'volume': volume, 'bid_price': bid_price, 'ask_price': ask_price,
    'option_type': option_type, 'strike_price': strike_price,
}
```
- **问题描述**: `update_tick()`接收bid_price和ask_price但不做任何校验就存入缓存。get_spread()（238-244行）检查`bid > 0 and ask > 0`后返回差值，但无`bid < ask`校验。这意味着无效价差可能被下游消费。
- **严重等级**: P2
- **影响系统**: 生产系统（RealTimeCache状态）

### 维度2：Bar构建正确性

#### 2.1 问题清单

##### P1-Q7: 两套并行K线合成实现 —— 潜在Bar不一致
- **文件**: `query_service.py`, `quant_platform.py`, `ds_realtime_cache.py`
- **行号范围**:
  - `query_service.py`: 118-264 (`_KlineAggregator`)
  - `quant_platform.py`: 68-191 (`TickAggregator`)
  - `ds_realtime_cache.py`: 385-436 (`get_kline_from_memory`)
- **问题描述**: 系统存在**三套**各自独立的从tick合成K线的实现：
  1. `_KlineAggregator`（query_service.py）：简单OHLCV累加，无跨日处理
  2. `TickAggregator`（quant_platform.py）：含volume delta模式、夜盘跨天trade_date、VWAP计算
  3. `get_kline_from_memory`（ds_realtime_cache.py）：基于elapsed秒数判断Bar边界

  三套实现在以下方面不一致：
  - **跨日处理**: `_KlineAggregator`用`// period_seconds * period_seconds`（UTC时间戳自然跨天），`TickAggregator`用ExchangeTime标准化+夜盘时段处理
  - **volume语义**: `_KlineAggregator`直接累加，`TickAggregator`支持delta模式
  - **VWAP**: 仅`TickAggregator`实现
  - **tick_count**: 仅`TickAggregator`和`get_kline_from_memory`跟踪

  主链路`process_tick`（storage_core.py:1000-1019）使用`_KlineAggregator`，但系统初始化时可能混用`TickAggregator`。
- **严重等级**: P1
- **影响系统**: 生产系统（实时Bar合成一致性）

##### P1-Q8: _KlineAggregator缺少Bar元数据 —— 无tick_count/trade_date
- **文件**: `query_service.py`
- **行号范围**: 211-225 (`_get_current_kline`)
```python
return {
    'instrument_id': self.instrument_id,
    'period': self.period,
    'ts': self.kline_start_time,
    'open': self.open_price, 'high': self.high_price,
    'low': self.low_price, 'close': self.close_price,
    'volume': self.volume, 'amount': self.amount,
    'open_interest': self.open_interest,
}
```
- **问题描述**: 合成的K线不包含`tick_count`和`trade_date`。`tick_count`是Bar质量的重要度量（低tick_count的Bar基于少量样本，可靠性差）。`trade_date`在夜盘跨天场景下可能错误。对比`TickAggregator`（quant_platform.py:125-132）包含trade_date和tick_count。
- **严重等级**: P1
- **影响系统**: 生产系统（实时K线质量不可度量）

##### P2-Q9: K线时间戳语义不明确 —— close_time vs open_time
- **文件**: `query_service.py`, `ds_data_writer.py`
- **行号范围**:
  - `query_service.py`: 216-217 (`_get_current_kline`中ts设为kline_start_time)
  - `ds_data_writer.py`: 468-484 (`batch_insert_klines`)
- **问题描述**: `_KlineAggregator._get_current_kline()`返回`'ts': self.kline_start_time`（Bar开始时间），`batch_insert_klines`写入`timestamp`字段。但存储层未记录这是open_time还是close_time语义。手册也未明确规定。查询时可能混淆close_time（某些数据源）和open_time（本系统），导致时间对齐错误。
- **严重等级**: P2
- **影响系统**: 三系统（时间戳语义一致性）

### 维度3：缺失数据处理

#### 3.1 问题清单

##### P1-Q10: 无tick断流实时检测 —— 数据缺口静默导致错误Bar
- **文件**: `strategy_tick_handler.py`, `storage_core.py`, `subscription_manager.py`
- **行号范围**: 全tick处理链路
- **问题描述**: 系统没有实时检测tick断流机制。若某合约超过阈值时间（如10秒）无新tick，系统不会告警、不会标记、不会回补。`SubscriptionManager`有统计性`tick_missing`（subscription_manager.py:1295-1314），但这是启动阶段的订阅覆盖率，不是运行时断流检测。断流会导致：
  1. 当前Bar的OHLCV基于不完整数据
  2. 无数据的分钟被跳过（无0-volume占位Bar），时间序列不连续
  3. 后续重连时的tick可能被错误归入当前Bar
- **严重等级**: P1
- **影响系统**: 生产系统（实时数据完整性）

##### P1-Q11: 无缺失数据回补策略 —— 断流后数据永久丢失
- **文件**: `strategy_historical.py`, `storage_query.py`
- **行号范围**:
  - `strategy_historical.py`: 29-200 (`load_historical_klines_with_stop`)
  - `storage_query.py`: 1428 (`_is_ext_kline_missing`)
- **问题描述**: 历史K线加载在策略启动时执行一次（`_check_and_start_historical_load_on_tick`）。运行时断流导致的数据缺口**不会被回补**。`_is_ext_kline_missing`仅用于决定是否需要从交易所补拉K线，但无主动回补触发机制。断流10分钟后恢复，中间10分钟的tick永久丢失——Bar合成将出现空洞。
- **严重等级**: P1
- **影响系统**: 生产系统（数据完整性不可恢复）

### 维度4：数据去重与乱序处理

#### 4.1 问题清单

##### P0-Q12: 无tick去重机制 —— 重复tick静默双重消费
- **文件**: `storage_core.py`, `strategy_tick_handler.py`
- **行号范围**: 全tick处理链路（on_tick → _process_tick → process_tick）
- **问题描述**: 整个实时tick处理链路**没有任何去重逻辑**。无序列号追踪、无时间戳+instrument_id唯一性约束、无DuckDB UNIQUE索引防止重复写入。交易所偶发推送重复tick（如连接重连后的快照重推），会导致：
  1. 同一tick被Bar合成累加两次（volume/amount翻倍）
  2. DuckDB INSERT OR REPLACE基于timestamp的conflict resolution（如write_tick_to_table,line 1109-1127）无法区分同一timestamp的多个合法tick
  3. 重复tick触发重复策略信号
- **严重等级**: P0
- **影响系统**: 生产系统（重复数据导致交易决策错误）

##### P0-Q13: 无tick乱序处理 —— 迟到数据误分类
- **文件**: `query_service.py`, `storage_core.py`
- **行号范围**:
  - `query_service.py`: 173-209 (`_KlineAggregator._update_impl`)
  - `storage_core.py`: 953-1019 (`process_tick`)
- **问题描述**: `_KlineAggregator._update_impl()`按照"tick到达顺序"而非"tick时间戳顺序"处理。若网络延迟导致tick乱序到达（timestamp=t1先到达，timestamp=t0后到达），后到达的tick会被错误归入当前Bar `[kline_start_time, kline_start_time+period_seconds)`或错误触发新Bar（如果`t0 >= kline_start_time+period_seconds`则创建新Bar，但实际上t0应属于上一根Bar）。

  `validate_out_of_order_ticks`（preprocess_ticks.py:973-1005）验证了回测数据集的乱序程度（"夏普下降>30%则需增加tick缓存+去重+排序模块"），但**该验证未集成到实时链路**。
  
  审计还发现`audit_edge_cases_v71.py:387-397`有一个关于"Tick时间戳乱序"的检查，判定逻辑是"使用交易所时间戳排序"即返回True表示已处理，但**该检查是代码存在性检查**，未验证运行时实际排序行为。
- **严重等级**: P0
- **影响系统**: 生产系统（实时tick乱序导致Bar归属错误）

##### P0-Q14: P0-3分钟边界安全分块未激活 —— 数据完整性破坏
- **文件**: `参数池/preprocess_ticks.py`
- **行号范围**: 648-696, 手册第1609-1626行
- **问题描述**: 手册V7.1 P0-3明确标记："`process_symbol()`行数切割未调用已实现的`_minute_safe_chunk_indices()`"。`_minute_safe_chunk_indices`（663-696行）和`check_minute_boundary_integrity`（699行起）已实现但未被调用。批处理按行数切割时可能将同一分钟的tick拆分到两个Chunk，导致：
  1. 聚合后的分钟Bar在该分钟被两个Chunk各自生成一次（重复Bar）
  2. 两个Chunk生成的同一分钟Bar的OHLCV值不同（各自基于部分tick）
  3. 手册要求验证标准："重复分钟Bar=0，状态切换次数差异<5%"
- **严重等级**: P0
- **影响系统**: 测试系统+评判系统（Bar数据完整性破坏）

### 维度5：数据损坏检测

#### 5.1 问题清单

##### P1-Q15: 无DuckDB完整性校验 —— 数据损坏静默传播
- **文件**: `ds_data_writer.py`, `ds_db_connection.py`, `quant_services.py`
- **行号范围**: 整个DuckDB交互层
- **问题描述**: 系统对DuckDB/WAL文件**从未执行`PRAGMA integrity_check`**。现有操作仅限于：
  - `FORCE CHECKPOINT`（ds_data_writer.py:312）：截断WAL但无完整性验证
  - `PRAGMA journal_mode=WAL`（quant_services.py:69）：设置日志模式
  - `PRAGMA synchronous=NORMAL`（quant_services.py:70）：设置同步级别
  
  **完全的遗漏**：
  1. 启动时无`PRAGMA quick_check`或`integrity_check`
  2. 定期运行时不执行完整性检查
  3. WAL截断前不验证数据完整性
  4. DuckDB文件损坏会导致静默数据丢失或查询返回损坏结果而非抛错
  
  仅`duckdb_tick_storage.py:359`（L1参数量化独立模块）执行了CHECKPOINT，但仍无integrity_check。
- **严重等级**: P1
- **影响系统**: 三系统（DuckDB为共享存储，损坏影响全部功能）

##### P1-Q16: WAL文件无校验和 —— 损坏记录静默注入
- **文件**: `subscription_manager.py`
- **行号范围**: 195-348（WAL管理全链路）
- **问题描述**: SubscriptionManager的JSONL WAL文件（`_wal_file`）采用逐行追加模式，无每行校验和或完整性标记。测试`test_wal_corrupted_entry_isolation`验证了单行损坏后跳过恢复，但这只是JSON解析层面的错误处理。如果：
  1. WAL文件部分写入（进程崩溃中间状态）
  2. 磁盘静默位翻转
  3. 文件截断

  系统无法检测这些损坏，会静默跳过损坏行（`json.JSONDecodeError`捕获）但不报告损坏程度和范围。无法区分'该行JSON格式错误但仍为有效状态'和'WAL文件底层损坏'。
- **严重等级**: P1
- **影响系统**: 生产系统（订阅状态恢复完整性）

##### P2-Q17: 数据回读路径无有效性检查 —— 损坏数据可能产生错误计算结果
- **文件**: `data_service.py`, `storage_query.py`, `ds_query_cache.py`
- **行号范围**: 所有DuckDB查询方法
- **问题描述**: 从DuckDB读取数据后直接使用，不检查：
  1. 返回的DataFrame行数是否与预期一致（SQL执行成功但返回空数据集）
  2. 数值字段是否在合理范围（如price>0、volume>=0、high>=low）
  3. 时间戳是否连续/单调递增
  4. JOIN结果是否丢失行（LEFT JOIN后的NaN行数）

  `_filter_bars`（preprocess_ticks.py:609-633）在批处理阶段执行OHLC合理性过滤，但实时查询路径`query_kline`（storage_query.py:817-831）返回原始数据，没有任何生效检查。
- **严重等级**: P2
- **影响系统**: 三系统（查询结果可靠性）

### 维度6：数据一致性

#### 6.1 问题清单

##### P2-Q18: 数据类型回读未验证 —— DuckDB→Pandas类型映射风险
- **文件**: `ds_data_writer.py`, `storage_query.py`, `data_service.py`
- **行号范围**: 
  - `ds_data_writer.py`: 452-501 (`batch_insert_klines`)
  - `storage_query.py`: 817-831 (`query_kline`)
- **问题描述**: 写入路径使用参数化查询传递Python float/int，DuckDB自动映射为DOUBLE/BIGINT。但读取路径`fetchdf()`返回的pandas DataFrame中的dtype未经验证。可能的风险：
  1. 存储的DOUBLE被DuckDB以FLOAT精度返回（Compiler版本差异）
  2. BIGINT溢出到float64导致精度损失（>2^53的volume值）
  3. 时间戳类型在DuckDB→Pandas转换中时区丢失
  4. NULL值在Pandas中变为NaN但下游假设为0

  当前代码无任何类型断言或验证。
- **严重等级**: P2
- **影响系统**: 三系统（数据类型一致性）

##### P2-Q19: 多表JOIN后完整性未验证 —— 静默行丢失
- **文件**: `query_service.py`, `storage_query.py`
- **行号范围**: 所有涉及JOIN的查询
- **问题描述**: K线查询需要通过`internal_id` JOIN `instrument_meta`获取`instrument_id`。如果`instrument_meta`中缺少对应行（预注册失败、表损坏），JOIN结果静默减少行数。系统没有：
  1. 对比JOIN前后的行数
  2. 检测anti-join（某表有某表无）
  3. LEFT JOIN后NaN列的数量统计
- **严重等级**: P2
- **影响系统**: 三系统（关联查询完整性）

### 维度7：历史数据回放保真度

#### 7.1 问题清单

##### P1-Q20: HFT回放引擎未实现 —— 分钟级回测不能替代tick级回测
- **文件**: `参数池/task_scheduler.py`, `参数池/preprocess_ticks.py`
- **行号范围**:
  - `task_scheduler.py`: 4547-4589 (`_interpolate_ticks_in_bar`)
  - `task_scheduler.py`: 4769-4830 (`run_backtest_hft_tick_fidelity`)
- **问题描述**: 手册V7.1第22.1节P3-1明确标注："HFT回放引擎实现 | P3 | 16h | S1策略不可信 | 分钟级vs tick级Sharpe差异>20%"。当前实现：
  1. `_interpolate_ticks_in_bar`用线性插值在Bar内生成虚拟tick（均匀成交量分配、线性价格路径）
  2. `run_backtest_hft_tick_fidelity`使用这些虚拟tick运行S1策略
  3. 手册明确：`hft_replay_only`参数"**仅在HFT回放引擎中测试**，分钟级回测结果标记为DEGRADED"

  线性插值无法模拟：
  - 真实市场的bid-ask bounce（买卖价差间的价格跳跃）
  - 成交量聚集模式（开盘/收盘放量）
  - 价格跳跃和买卖压力不对称
  - 微观结构噪声
  
  这导致S1高频策略的回测Sharpe可能系统性高估。
- **严重等级**: P1
- **影响系统**: 测试系统+评判系统（S1策略评估失真）

##### P2-Q21: 回测数据集的tick密度无验证 —— 虚拟tick数量与实际市场差异
- **文件**: `参数池/task_scheduler.py`
- **行号范围**: 4547-4550
```python
def _interpolate_ticks_in_bar(bar, n_ticks: int = 5):
```
- **问题描述**: 固定`n_ticks=5`意味着每根分钟Bar最多5个虚拟tick（~12秒/tick），但活跃合约实际tick频率可达每秒数十笔。这种固定插值导致：

  2. 成交量被均匀分配（`vol / n_ticks`），但实盘中成交量非均匀

## 三、审计总结

### 3.1 关键发现

1. **P0级缺陷集中于实时tick处理链路**：5个P0问题中有4个位于生产系统的实时tick处理路径（无bid/ask校验、price=0允许、无去重、无乱序处理）。这些缺陷意味着**当前实时Bar合成基于未经充分校验的tick数据**。

2. **手册已知问题仍有未修复项**：手册V7.1第22.1节标记的P0级阻塞项中，P0-3（分钟边界安全分块）虽已实现但**未被调用**，属于"代码在但链路未接通"的典型模式。

3. **两套并行Bar合成实现**：`_KlineAggregator`和`TickAggregator`同时存在且语义不一致，意味着同一条tick流经不同代码路径可能产生不同的Bar。

4. **DuckDB完整性保障近乎空白**：无`PRAGMA integrity_check`、无WAL校验和、无回读验证，数据库损坏将静默传播到交易决策。

5. **HFT回放保真度手册已自知**：手册明确标记P3-1"HFT回放引擎实现"为未完成，且自知"分钟级vs tick级Sharpe差异>20%"。

### 3.2 严重等级分布

| 等级 | 数量 | 典型场景 |
|:---|:---|:---|
| P0 | 5 | 无tick去重/乱序/价差校验导致交易决策错误 |
| P1 | 9 | 断流无检测、Bar元数据缺失、HFT回放失真 |
| P2 | 9 | 无溯源、无完整性度量、规范性问题 |

### 3.3 优先修复建议（Top 5）

1. **P0-Q12/P0-Q13**: 在`process_tick`入口增加tick去重+排序缓存（基于timestamp+instrument_id+sequence_number）
2. **P0-Q14**: 激活`_minute_safe_chunk_indices()`调用，修复分钟边界破坏
3. **P0-Q1**: 在`_validate_tick`和`_process_tick`中增加bid<ask校验
4. **P0-Q2**: `_validate_tick`中增加price>0的强制检查（结算tick特殊处理应有明确标记）
5. **P1-Q15**: 系统启动时执行`PRAGMA integrity_check`，定期（每小时）重复

### 3.4 与手册对照总结

手册明确关注的数据质量问题：
- P0-2（bid_ask_spread NaN）→ 代码已有初步修复但实时路径校验缺失
- P0-3（分钟边界安全分块）→ 代码已实现但未调用
- P0-4（strike_price NaN）→ 代码已有修复但缺少质量标记传递

手册关注但本审计发现超出范围的：
- tick去重/乱序 → 手册未提及但本审计发现为P0级缺陷
- DuckDB完整性校验 → 手册未提及但本审计发现为P1级缺陷
- 数据生命周期溯源 → 手册未提及但为生产化必备能力
