## 2. 名称遮蔽（Name Shadowing）

### 2.1 P2-01：format 参数遮蔽内置函数 format()

- **文件**：`performance_monitor.py`
- **行号**：120
- **代码**：`def get_stats(cls, format: str = 'dict', **kwargs) -> Any:`
- **描述**：函数参数 `format` 覆盖了 Python 内置函数 `format()`。在该方法体内，如果代码尝试调用 `format(value, spec)` 会出现 `TypeError: 'str' object is not callable`。
- **严重等级**：**P2**（规范改进建议）
- **影响系统**：生产系统（性能监控模块）
- **当前风险**：经核查，`get_stats()` 方法体中未使用 `format()` 内置函数（第120-180行主要做字符串拼接和字典操作），当前不会触发运行时错误。但若后续维护时在该方法内添加 `format()` 调用，将静默失败。
- **建议修复**：将参数重命名为 `output_format` 或 `fmt`。

### 2.2 P2-02：self.type 实例变量名与内置 type() 同名

- **文件**：`event_bus.py`
- **行号**：718
- **代码**：`self.type = event_type or type(self).__name__`
- **描述**：`BaseEvent` 类的实例变量名 `type` 与 Python 内置 `type()` 同名。在类方法内，`self.type` 访问实例属性，`type(obj)` 调用仍然正常工作（Python 正确区分），但代码可读性降低，IDE 可能发出警告。
- **严重等级**：**P2**（规范改进建议）
- **影响系统**：生产系统（事件总线）
- **当前风险**：无实际运行时风险。`self.type` 和 `type()` 在 Python 语法中不会冲突，因为一个必须通过 `self.` 访问。
- **建议修复**：考虑重命名为 `self.event_type` 或 `self.kind`，但非强制。

### 2.3 伪阳性排除说明

以下模式经核查**不构成**名称遮蔽：
- `type=str` / `type=int` 在 `argparse.add_argument()` 调用中：这是关键字参数传递，不影响命名空间
- `type(self).__name__`：使用内置 `type()` 函数的正确方式
- `isinstance(val, type)`：将 `type` 作为元类判断，是内置 `type` 的使用
- `target_type: type = float`：类型注解中使用 `type`，不会遮蔽

### 3.2 P1-03：position_service.py 中 _cross_strategy_risk_guard 无锁保护

- **文件**：`position_service.py`
- **行号**：1603-1609
- **代码**：
  ```python
  _cross_strategy_risk_guard: Optional[CrossStrategyRiskGuard] = None

  def get_cross_strategy_risk_guard() -> CrossStrategyRiskGuard:
      global _cross_strategy_risk_guard
      if _cross_strategy_risk_guard is None:
          _cross_strategy_risk_guard = CrossStrategyRiskGuard()
      return _cross_strategy_risk_guard
  ```
- **描述**：与同文件的 `_global_position_service`（第1325-1334行，有 `_position_service_lock` 保护）不同，`_cross_strategy_risk_guard` 没有任何线程锁保护。在高并发 tick 驱动环境下，两个线程可能同时检测到 `None`，各自创建实例，导致全局对象不一致。
- **严重等级**：**P1**（导致难以追踪的 bug）
- **影响系统**：生产系统（跨策略风险守卫）
- **建议修复**：添加 `threading.Lock()` 和双检锁模式。

### 3.3 P1-04：__init__.py __getattr__ 中动态修改 globals() 不可重入

- **文件**：`__init__.py`
- **行号**：84
- **代码**：`globals()[name] = value`
- **描述**：延迟导入机制中，`__getattr__` 在成功导入后将符号直接注入模块全局命名空间。这在正常使用中安全，但如果多个线程同时首次访问不同符号，`globals()` 字典可能被并发修改（Python 3.x 中 `dict.__setitem__` 是原子的，但 `globals()` 调用与后续修改之间存在竞态窗口）。
- **严重等级**：**P1**（导致难以追踪的 bug）
- **影响系统**：生产系统（包入口模块）
- **当前风险**：Python 的 GIL 在 CPython 中提供了一定程度的保护，且 `globals().__setitem__` 对单个键的操作为原子性。但严格来说这不是线程安全的设计。若未来在 PyPy 或无 GIL 的 Python 3.13+ 上运行，可能暴露竞态条件。
- **建议修复**：使用 `threading.Lock()` 保护 `globals()[name] = value` 操作。

- **描述**：这些验证脚本在模块级定义了可变的 `dict`/`list`。这些脚本是单线程执行的独立工具脚本，实际不会出现并发问题，但作为代码规范仍值得注意。
- **严重等级**：**P2**（规范改进建议）
- **影响系统**：验证/测试系统
- **当前风险**：极低。这些脚本作为 `__main__` 运行，且不涉及模块重载。

## 6. 模块导入顺序与循环导入

### 6.1 P0-07：strategy_judgment_engine.py 导入不存在的模块

- **文件**：`策略评判\strategy_judgment_engine.py`
- **行号**：722
- **代码**：
  ```python
  try:
      from ali2026v3_trading.策略评判.config import SURVIVED_THRESHOLD
      if overall < SURVIVED_THRESHOLD:
          conditions.append(f"[E-08] 综合评分{overall:.4f}低于survived阈值{SURVIVED_THRESHOLD}")
      logging.info("[E-08] survived判定阈值检查完成: threshold=%.4f", SURVIVED_THRESHOLD)
  except Exception as _e08_err:
      warnings.append(f"[E-08] survived判定阈值集成失败(非致命): {_e08_err}")
  ```
- **描述**：导入的目标模块 `ali2026v3_trading.策略评判.config` **不存在**。`策略评判/` 目录下没有 `config.py`、`config/__init__.py` 或任何 `config` 子模块。此导入**每次都会失败**（被 `try/except` 捕获为 `ModuleNotFoundError`），导致 E-08 survived 判定阈值检查**从不上线**。
- **严重等级**：**P0**（运行时静默错误——评判系统的核心功能（survived 阈值）从未生效）
- **影响系统**：评判系统
- **当前风险**：该代码在 try/except 中被吞没，不抛出异常，但 E-08 判定维度完全虚设。所有策略的 survived 判定都绕过了阈值检查。
- **建议修复**：明确以下三选一：
  1. 创建 `策略评判/config.py` 并定义 `SURVIVED_THRESHOLD`（如果在 V7.0 手册 §7.4 节有定义）
  2. 从实际定义 SURVIVED_THRESHOLD 的模块导入
  3. 在 `strategy_judgment_engine.py` 中内联定义该阈值

### 6.2 P1-08：evaluation/__init__.py 中 try/except ImportError 静默吞没缺失依赖

- **文件**：`evaluation\__init__.py`
- **行号**：3-31（整文件）
- **代码**：
  ```python
  from .cascade_judge import CascadeJudge, CascadeReport, GateReport, GateResult, BacktestMetrics, adapt_backtest_result

  try:
      from .parameter_drift_detector import ParameterDriftDetector
  except ImportError:
      ParameterDriftDetector = None

  try:
      from .chicory_eviction import ChicoryEvictionPolicy
  except ImportError:
      ChicoryEvictionPolicy = None
  # ... (共7个同样的模式)
  ```
- **描述**：6个子模块使用 `try/except ImportError` 并静默设置为 `None`。这是一个反模式：如果子模块实际存在但因**内部导入错误**（非模块不存在）而失败，错误会被静默吞没，导致 `AttributeError: 'NoneType' object has no attribute 'xxx'` 在使用时暴露，而非在导入时暴露。
- **严重等级**：**P1**（难以追踪的 bug）
- **影响系统**：评判系统
- **当前风险**：目前子模块文件都实际存在，所以 `ImportError` 都不会触发。但若子模块内部有损坏的导入（如依赖不存在的第三方库），错误会被静默转为 `None`，后续使用时出现诡异的 `NoneType` 错误。
- **建议修复**：区分 `ModuleNotFoundError`（文件不存在，可容错）和 `ImportError`（内部错误，应抛出）。或者移除 try/except，因为所有子模块都实际存在。

### 6.3 P1-09：大量延迟导入掩盖依赖关系

- **涉及文件**：
  - `参数池/task_scheduler.py`（16处延迟导入）
  - `mode_engine.py`（13处延迟导入）
  - `策略评判/strategy_judgment_engine.py`（12处延迟导入）
  - `参数池/enhanced_phase_scan.py`（12处延迟导入）
  - `strategy_core_service.py`（5处延迟导入）
  - 合计：约**50+处**函数体内的 `from ali2026v3_trading.xxx import ...`
- **描述**：大量使用函数体内导入来打破循环依赖。虽然这在技术上可行，但：
  - 使模块依赖关系图不可通过静态分析获得
  - 若模块实际缺失，错误延迟到运行时（功能调用时才报错）
  - 增加首次调用的延迟开销
- **严重等级**：**P1**（导致难以追踪的 bug）
- **影响系统**：三系统
- **当前风险**：中。这是已知技术债务，但当前运行稳定。主要风险在于：某些延迟导入的目标模块若被删除/重命名，错误不会在启动时暴露，而是在特定功能被使用时才暴露。
- **建议修复**：长期来看应重构导入结构减少循环依赖，短期可考虑添加启动时可选的全量导入验证。

## 9. 架构层面问题

### 9.1 A-01：全局单例模式的系统性影响

代码库中约30+模块使用模块级单例 `get_xxx()` 模式。这种模式的设计影响了命名空间和模块状态管理：

- **优势**：服务生命周期明确，支持 `reset_xxx()` 用于测试
- **劣势**：
  - 模块间隐式耦合：`get_xxx()` 内部可能触发其他 `get_yyy()` 调用，形成隐式初始化链
  - 模块重载(reload)问题：`importlib.reload(module)` 后模块级变量会重置，但其他模块持有的旧引用不会更新
  - 测试隔离需额外操作：`conftest.py` 中的 `_reset_all_singletons()` 需要准确知道所有模块的 singleton 变量名

**当前状态**：整体设计合理但脆弱。`conftest.py` 的重置逻辑需要随新增模块同步更新。

### 9.2 A-02：延迟导入模式的命名空间效应

`__init__.py` 和 `main.py` 使用 `__getattr__` 实现延迟导入，将符号动态注入 `globals()`：

```python
# __init__.py:84
globals()[name] = value
```

此行为的命名空间效应：
- `dir(ali2026v3_trading)` 返回 `__dir__()` 定义的固定列表（第88-89行），无论符号是否已被首次访问
- 首次访问后，符号出现在 `globals()` 中，后续访问为 O(1)
- 如果字符串字面量导入路径中存在拼写错误（如 `_EXPORTS` 中的模块路径错误），错误延迟到首次使用

**当前状态**：机制工作正常，但依赖 `_EXPORTS` 和 `_LAZY_IMPORTS` 字典的维护质量。

## 11. 结论

本轮专项审计发现了**1个P0问题、2个P1问题、11个P2规范建议**。

### 关键发现：

1. **P0：E-08 survived 判定阈值从未上线** —— `strategy_judgment_engine.py:722` 导入了不存在的模块，导致评判系统中 survived 阈值检查永久失效。这个错误被 try/except 吞没，无任何日志告警。

2. **全局单例模式整体合格但有缺口** —— 30+模块的 singleton 模式大多数有双检锁保护，但 `_cross_strategy_risk_guard` 缺少锁保护，`__init__.py` 中 `globals()` 修改也存在轻微线程安全隐患。

3. **代码库在关键维度上表现优秀**：
   - 零通配符导入
   - 零可变默认参数
   - 零闭包陷阱
   - 零 sys.modules 破坏性操作
   - 无实际的内置函数名称遮蔽导致的运行时崩溃

4. **延迟导入是系统性的技术债务** —— 50+处的函数体内延迟导入，虽然目前运行稳定，但增加了运行时故障风险和维护难度。
