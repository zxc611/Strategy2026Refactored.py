## 二、系统网络架构总览

### 2.1 架构定性

**关键发现**: ali2026v3_trading 是一个 **Python 策略插件**，运行在 InfiniTrader 仿真/实盘平台容器内。**所有实际的网络 I/O（与交易所的 WebSocket 连接、REST API 调用、SSL/TLS 握手、数据压缩）均由平台底层（C++/Go 层）处理**，不在本 Python 策略代码的审计范围内。

Python 策略层通过以下接口与平台通信：

| 通信方向 | 接口 | 文件位置 |
|---------|------|---------|
| 数据接收 | `on_tick(tick)`, `on_order(order)`, `on_trade(trade)` 回调 | strategy_core_service.py:1124-1178 |
| 市场订阅 | `strategy_obj.sub_market_data(exchange, instrument_id)` | strategy_lifecycle_mixin.py:182-188 |
| 取消订阅 | `strategy_obj.unsub_market_data(exchange, instrument_id)` | strategy_lifecycle_mixin.py:194-196 |
| 下单 | `strategy_obj.insert_order(**params)` | order_service.py:299 |
| 撤单 | `strategy_obj.cancel_order(order_ref)` | order_service.py:139 |
| K线获取 | `market_center.get_kline_data(...)` | strategy_lifecycle_mixin.py:208-209 |
| 行情中心 | `pythongo.core.MarketCenter()` | strategy_lifecycle_mixin.py:281-283 |

### 2.2 代码证据

对所有 .py 文件执行了以下精准搜索，**零命中**：
- `import requests` / `import websocket` / `import aiohttp` / `import httpx` / `import urllib` / `import socket` / `import ssl` / `import asyncio`
- `.get(` / `.post(` / `.put(` / `.delete(` 作为 HTTP 动词（所有 `.get()` 均为 Python dict 操作）

**结论**: Python 策略层无直接网络套接字操作，无对外 HTTP/WebSocket 客户端代码。本审计聚焦于 Python 层对平台网络服务的间接依赖、错误处理、超时/重试机制和数据完整性保障。

### 3.5 SSL/TLS 安全

#### 3.5.1 现状

**代码证据**: 全量搜索 `ssl`, `tls`, `SSL`, `TLS`, `certificate`, `verify`, `CERT`，**零匹配**（Python 策略层）。

**结论**: SSL/TLS 完全由 InfiniTrader 平台底层处理。Python 层不参与证书验证、TLS 版本协商、中间人防护。此维度的安全性取决于平台的实现质量。

| # | 文件 | 行号 | 描述 | 等级 | 影响系统 |
|---|------|------|------|------|---------|
| R21-TLS-01 | order_service.py | 39-58 | `PlatformAuthenticator` 管理平台认证 Token，有效期 3600 秒（1小时）。这是**业务层认证**而非传输层安全。Token 存储在内存中（`self._token`），进程重启时丢失但会自动重新获取。设计合理。 | ✅ 无问题 | 生产 |
| R21-TLS-02 | order_service.py | 39-58 | `PlatformAuthenticator` 的 Token 无加密存储、无安全擦除（进程退出时 token 残留在内存中）。在仿真环境风险可接受，但在实盘环境中建议增加安全擦除。 | **P2** | 生产 |

**SSL/TLS 维度审计结论**: 此维度在 Python 策略层无代码可审。建议在平台层面单独进行 TLS 安全审计（验证证书链、TLS 1.2+、禁用弱密码套件）。

## 五、对受影响系统的详细影响分析

### 5.1 生产系统 (ProductionQuantSystem)

**核心链路**: `onStart → bind_platform_apis → initialize → subscribe_all → on_tick → execute_option_trading_cycle → send_order → on_order`

| 风险点 | 描述 |
|-------|------|
| 下单无超时(R21-API-01/TO-01/TO-02) | 如果平台下单 API 挂起，策略执行线程永久阻塞，所有后续交易周期停止触发 |
| Tick 无序列号(R21-MSG-01) | 平台重连后可能收到乱序/重复 tick，策略在"错误时刻"做决策导致错误交易 |
| API 断线无感知(R21-MON-03) | 平台连接断开后策略继续运行，可能出现"假运行"——在过时数据上持续产生信号 |

**生产中风险**: **中等偏高**（P1 问题均有明确的生产影响路径）

### 5.2 测试系统

**核心链路**: 与生产相同但 `test_mode=True`。所有配置参数和超时逻辑与生产共享。

**测试中风险**: **低**（测试环境通常有更短的时间窗口，超时和连接问题影响较小）

### 5.3 评判系统

**核心链路**: `parameter_pool_adapter → enhanced_phase_scan → delay_time_sharpe_3d → backtest_integration_hooks`

评判系统使用本地历史数据回测，**不依赖实时网络连接**。相关代码完全通过本地数据文件（Parquet/DuckDB）运行。除 API 兼容性映射外，不受网络维度问题影响。

**评判中风险**: **极低**（离线回测，不产生网络依赖）

## 六、架构层面总结

### 6.1 强项

1. **分层隔离清晰**: 网络 I/O 完全由平台层处理，Python 策略层专注业务逻辑，符合良好架构实践
2. **API 兼容性成熟**: R21-COMPAT 维度的 7 个检查项全部为 ✅ 无问题，多字段名映射、签名检测、dict/object 双模式读取设计极其稳健
3. **订阅管理完善**: SubscriptionManager 集成重试队列、指数退避、WAL 持久化、诊断探针——是目前代码库中网络相关质量最高的模块
4. **降级设计合理**: DEGRADED 状态机 + 健康检查双重保护，API 就绪检查 + 交易跳过护城河

### 6.2 核心弱点

1. **"下单无超时"是最大的单点风险**: `order_timeout_seconds` 定义了 5 秒但从未消费。一个阻塞的 `_platform_insert_order()` 调用可以导致整个策略线程永久挂起
2. **"无序列号"导致数据完整性缺口**: 依赖平台保证 tick 顺序，无独立的验证机制。这在金融交易系统中是基础要求（FIX 协议强制 `MsgSeqNum`）
3. **"无连接心跳"导致被动感知**: 平台断开后 Python 策略无法主动感知，只能在下一次 `execute_option_trading_cycle` 中通过交易失败间接发现

### 6.3 改进优先级建议

| 优先级 | 改进项 | 预计工时 |
|--------|--------|---------|
| 1st | 为 `_execute_send_order` 添加 `threading.Timer` 超时熔断（R21-API-01） | 2h |
| 2nd | 在 `on_tick` 中增加 tick 时间戳延迟监控（R21-MON-01） | 1h |
| 3rd | 激活 SubscriptionManager 告警回调（R21-MON-05） | 0.5h |
| 4th | 考虑向平台团队请求 WebSocket 断线回调接口（R21-MON-03） | 沟通 + 集成 |
| 5th | P2 问题按需修复 | 按需 |
