### 1.2 总体结论

系统在可追溯性和审计能力方面存在**重大缺口**。核心问题为：**signal_id 在信号→订单环节完全断裂**，导致无法将任意一笔成交追溯到其触发信号和决策上下文。日志以非结构化纯文本为主，审计事件与调试日志混合输出，无不可否认性机制。建议优先修复 P0 项，特别是 signal_id 链路的贯通。

### 2.2 链路断裂总结

```
Tick ──(✅ log)──→ Signal ──(signal_id=SIG_xxx, ✅ JSONL)──→
    │
    └── 🔴 断裂点 ──→ TradeCycle(execute_option_trading_cycle)
                        targets 不含 signal_id
                              │
                              ▼
                        Order(send_order)
                        order dict 不含 signal_id
                              │
                              ▼
                        TradeLog(_persist_trade_log)
                        signal_id = ""  ← 🔴 永远为空
```

**结论**: 一笔交易从信号到成交，signal_id 在 `execute_option_trading_cycle()` 入口处即被丢弃，此后所有环节都无法追溯到触发信号。外部系统（数据库/日志）无法重建 signal→order→execution 链路。

## 3. 维度1: 决策审计轨迹完整性

> **标准**: 每笔交易能否追溯到：哪个信号触发→哪个策略生成→哪个参数版本→哪个风控通过→哪个时间点的哪个tick。链路中任一环节日志缺失即为断裂。

### 3.1 审计矩阵

| 链路环节 | 可追溯性 | 日志存在 | 关联性 | 评级 |
|:---|:---|:---|:---|:---|
| Tick 来源 | ❌ 无 tick_id | ✅ | ❌ 无法关联 | P2 |
| 信号生成 | ✅ signal_id | ✅ | ✅ 自包含 | 合格 |
| 策略归属 | ⚠️ open_reason | ⚠️ | ⚠️ 间接关联 | P1 |
| 参数版本 | ❌ 无 param_version | ❌ | ❌ | P0 |
| 风控通过/拒绝 | ⚠️ 有 reason | ✅ | ❌ 无 signal_id | P1 |
| 订单执行 | ✅ order_id | ✅ | ❌ signal_id 断裂 | P0 |
| 持仓变更 | ⚠️ position_id | ✅ | ❌ 无 signal_id | P1 |

### 3.2 具体发现

**A. 参数版本不可追溯 [P0]**
- `execute_option_trading_cycle()` 不记录当前使用的参数集版本号。
- `state_param_manager.py` 虽然管理状态参数，但参数切换事件未关联到具体交易。
- 若一笔交易亏损，无法确定当时使用的具体参数值。
- **文件**: `strategy_core_service.py:274` — `execute_option_trading_cycle()` 缺少 param_version 记录。

**B. tick→signal 无直接追溯 [P1]**
- `emit_signal()` 生成的信号不包含触发 tick 的时间戳或引用。
- tick 通过 `strategy_tick_handler.py::_dispatch_tick()` 分发后，信号生成通过策略内部状态驱动，链路是非直接的。
- **文件**: `signal_service.py:308-319` — signal dict 无 tick_ref / bar_index 字段。

**C. 风控决策可追溯性不完整 [P1]**
- `check_before_trade()` 阻断时输出 `logging.warning`，但不记录结构化审计事件。
- `_record_result()` 仅更新内存计数器，无持久化。
- **文件**: `risk_service.py:344-354` — `_record_result()` 仅增加计数器。

## 4. 维度2: 信号→订单→执行全链路可追溯

> **标准**: signal_id 是否贯穿全链路、order_id 是否关联回 signal_id、execution_id 是否关联回 order_id、链条是否可被外部系统（数据库/日志）重建。

### 4.1 关键发现

**A. signal_id 断裂（核心缺陷）[P0 🔴]**

| 位置 | 文件:行号 | 问题 |
|:---|:---|:---|
| 信号生成 | `signal_service.py:311` | `signal_id = f"SIG_{uuid.uuid4().hex[:12]}"` ✅ |
| 信号→交易周期 | `strategy_core_service.py:287` | `targets = t_type.select_otm_targets_by_volume()` — **不含 signal_id** ❌ |
| 交易周期→订单 | `strategy_core_service.py:326` | `self._order_service.execute_by_ranking(targets)` — targets 无 signal_id ❌ |
| 订单生成 | `order_service.py:401-431` | `execute_by_ranking()` 遍历 targets 但不提取 signal_id ❌ |
| 订单字典 | `order_service.py:222-235` | order dict **不含 signal_id 字段** ❌ |
| 订单日志 | `order_service.py:733` | `"signal_id": order.get('signal_id', '')` — 永远为空 ❌ |

**影响**: 从交易日志无法知道一笔订单由哪个信号触发，无法重建 signal→order 关联。

**B. 无 execution_id 概念 [P0]**
- 系统没有独立的 `execution_id` 概念来区分"下单"和"成交"。
- `order_id` 承担双重角色，但成交确认后的状态更新没有独立的 execution 记录。
- 平台返回的 `platform_order_id` 映射到 `order_id`，但未持久化到结构化日志。

**C. BoxSpring 策略使用独立 ID 命名空间 [P1]**
- `box_spring_strategy.py:480`: `signal_id = f"SPRING_{instrument_id}_{int(now*1000)}"` — 使用 `SPRING_` 前缀
- SignalService 使用 `SIG_` 前缀 — 两套 ID 命名空间不兼容
- OrderService 的 `send_order()` 通过 `box_spring_strategy.py:867` 接收 `signal_id` 传给 order dict，但这是在 BoxSpring 内部调用路径中，非标准路径
- **文件**: `box_spring_strategy.py:480` vs `signal_service.py:311`

**D. 跨日志文件无关联键 [P0]**
- `signals.jsonl` (StructuredJsonlLogger) — 记录 signal_id
- `orders.jsonl` (StructuredJsonlLogger) — 记录 order_id，无 signal_id
- `trade_log` (OrderService._persist_trade_log) — signal_id 为空
- 三套日志文件无法通过 ID 关联重建全链路

### 4.2 最小可追溯链路（建议目标）

```
signal_id ──→ order.signal_id ──→ trade_log.signal_id
   │                                    │
   └── signals.jsonl                    └── trade_log.jsonl
                                          │
                                    order_id ──→ orders.jsonl
```

**当前状态**: 这条链路在每一步都断裂。

| 策略状态 | ✅ L250 | ✅ (RiskCheckResponse) | N/A | 合格 |
| 信号无效 | ✅ L253 | ✅ | N/A | 合格 |

### 5.2 信号过滤日志

| 过滤维度 | 日志存在 | 记录过滤原因 | 信号 ID | 评级 |
|:---|:---|:---|:---|:---|
| 冷却期 | ✅ L262 | ✅ | ❌ (过滤在 signal_id 生成之前) | P1 |

## 6. 维度4: 不可否认性（Non-Repudiation）

> **标准**: 交易决策是否有时间戳签名、日志是否可防篡改、关键事件是否有序列号防跳号。

### 6.1 审计矩阵

| 要求 | 实现状态 | 详情 |
|:---|:---|:---|
| 时间戳签名 | ❌ 无 | 使用 `datetime.now().isoformat()`，无 HMAC/数字签名 |
| 日志防篡改 | ❌ 无 | JSONL 文件可任意修改，无 hash chain |
| 序列号防跳号 | ❌ 无 | 无 event sequence number |
| 事件完整性 | ❌ 无 | 无 Merkle tree / hash chain |
| 身份绑定 | ❌ 无 | 策略决策无法绑定到具体策略实例身份 |
| 写入确认 | ⚠️ 部分 | `order_service.py:752-754` — trade_log 写入但无 fsync 确认 |

### 6.2 具体发现

**A. 无防篡改机制 [P0]**
- 所有日志文件（signals.jsonl, orders.jsonl, trade_log）均为普通 JSONL 文件，可被任意修改。
- 无 hash chain，无法检测日志被篡改或删除。
- 监管审查场景下，日志完整性无法自证。

**B. 无事件序列号 [P0]**
- 系统不维护全局事件序列号，无法检测日志跳号/丢失。
- 例如：signal_id 为随机 UUID，两个信号之间无顺序保证。

**C. 时间戳无签名 [P1]**
- 所有时间戳使用 `datetime.now()` 本地时间，未使用 UTC+NTP 同步时间。
- 无 HMAC 签名保护时间戳不被事后修改。

**D. 影子引擎的 hash 使用仅限于配置完整性 [P2]**
- `shadow_strategy_engine.py:260`: `hashlib.md5()` 仅用于配置内容 hash，非审计用途。
- `config_params.py:300`: MD5 仅用于配置变更检测，非审计用途。

## 12. 整改路线图

### 第一阶段：修复核心链路（P0，建议 1-2 周）

1. **贯通 signal_id 链路** (P0-1, P0-2)
   - `execute_option_trading_cycle()` 中将 signal_id 注入 targets
   - `send_order()` 的 order dict 中添加 signal_id 字段
   - `_persist_trade_log()` 正确读取 signal_id
   - 新增 `execution_id` 字段区分下单与成交

2. **建立交易事件数据库表** (P0-5)
   - 创建 `trade_events` 表：event_id, signal_id, order_id, execution_id, event_type, timestamp, payload
   - 实现 `storage_query.reconstruct_daily_trades()` 查询接口

3. **分离审计日志通道** (P0-6, P0-7)
   - 创建 `audit_logger` (独立 logging.Logger, 文件 handler)
   - 交易决策、风控阻断、信号发射统一写入 audit_logger
   - 配置日志保留策略（如 7 年）

### 第二阶段：增强不可否认性与事后分析（P0+P1，建议 2-4 周）

4. **引入事件序列号与 hash chain** (P0-4)
   - 全局事件序列号（monotonic counter + 持久化）
   - 每个审计事件哈希链（prev_event_hash → current_event_hash）
   - 添加 HMAC 时间戳签名

5. **系统状态快照** (P0-8, P0-9)
   - 定时（如每分钟）将持仓/订单/信号/参数状态快照写入审计日志
   - signal_history 持久化到数据库或 JSONL

6. **完善风控审计** (P1-1, P1-2)
   - `_record_result()` 写入结构化审计日志
   - 每次风控检查记录：threshold vs actual 值
   - 被过滤信号生成 `filtered_signal_id` 并记录过滤维度

### 第三阶段：结构化日志与交易重建（P1+P2，建议 2-4 周）

7. **统一结构化日志格式** (P2-1, P2-2, P2-5)
   - 所有审计事件使用统一的 JSON Schema
   - 统一 event_type 枚举
   - 迁移核心 logging.info 到结构化日志

8. **性能指标时序记录** (P1-7)
   - 决策延迟：tick_arrival → signal_emission
   - 订单延迟：send_order → filled
   - 信号命中率：emitted / (emitted + filtered)

9. **盈亏独立验证支持** (P1-6)
   - 记录每笔持仓的 mark-to-market 序列
   - 提供 Greeks 归因（delta_pnl, gamma_pnl, vega_pnl, theta_pnl）
