## 维度1: 函数签名不一致

### 【R13-SIG-01】`on_tick` 方法签名跨模块严重不一致
- **文件**: `box_spring_strategy.py:1374` / `position_service.py:377` / `strategy_tick_handler.py:145` / `subscription_manager.py:1102` / `ds_query_cache.py:24`
- **行号**: 1374, 377, 145, 1102, 24
- **问题描述**: 同名方法`on_tick`在5个不同类中参数签名完全不同：
  - `BoxSpringStrategy.on_tick(self, instrument_id: str, price: float, high: float = 0.0, low: float = 0.0, volume: int = 0, timestamp: Optional[datetime] = None)` — 6个参数
  - `PositionService.on_tick(self, tick: Any)` — 1个参数(tick对象)
  - `TickHandlerMixin.on_tick(self, tick: Any)` — 1个参数(tick对象)
  - `SubscriptionManager.on_tick(self, instrument_id: str, last_price: float, volume: float = 0)` — 3个参数
  - `QueryCacheMixin.on_tick(self, symbol: str, price: float, timestamp: datetime)` — 3个参数
  - 调用方`StrategyCoreService.execute_option_trading_cycle()`以`bss.on_tick(inst_id, t.get('price', 0.0))`调用BoxSpringStrategy，仅传2个参数，缺少high/low/volume/timestamp
- **严重等级**: **P0**
- **影响系统**: 生产系统
- **风险**: BoxSpringStrategy.on_tick被调用时high/low均为0，导致`update_box`永远不会被触发（条件`if high > 0 and low > 0`永远不满足），箱体检测在实盘中完全失效

## 维度2: 参数类型不匹配

### 【R13-TYP-01】`RiskService` 单例工厂首次/后续调用参数不一致
- **文件**: `risk_service.py:2167` / `strategy_core_service.py:299`
- **行号**: 2167, 299
- **问题描述**: `get_risk_service(params, position_manager, strategy)`是单例工厂，首次调用传入的参数会创建实例，但后续调用传入不同参数时会被静默忽略（单例已创建）。`StrategyCoreService.execute_option_trading_cycle()`中调用`get_risk_service(None)`，首次调用若由其他模块先触发（如`StrategyEcosystem._propagate_capital_scale_fallback`），则params可能为None，导致RiskService.params=None。后续调用传入真实params时被忽略
- **严重等级**: **P0**
- **影响系统**: 生产系统
- **风险**: RiskService.params为None时，`_check_safety_meta_layer`中`get_safety_meta_layer(self.params)`可能传入None，导致安全元层初始化异常，风控检查失效

### 【R13-TYP-02】`check_before_trade` 信号dict的key不一致
- **文件**: `risk_service.py:217` / `strategy_core_service.py:301-308`
- **行号**: risk_service.py:217, strategy_core_service.py:301-308
- **问题描述**: `RiskService.check_before_trade(signal)`期望signal dict包含`symbol`、`is_valid`、`action`、`account_id`、`amount`等key。但调用方`StrategyCoreService.execute_option_trading_cycle()`构造的signal为：
  ```python
  {'symbol': t.get('instrument_id', ''), 'direction': ..., 'price': ..., 'volume': ..., 'is_valid': True}
  ```
  缺少`action`、`account_id`、`amount`等key。`_check_safety_meta_layer`中`signal.get("action", "")`永远返回空字符串，导致断路器平仓豁免逻辑永远不触发（`action == "CLOSE"`永远为False）
- **严重等级**: **P0**
- **影响系统**: 生产系统
- **风险**: 断路器暂停期间，平仓信号无法通过风控检查（手册P0-裂缝25修复的平仓豁免逻辑失效），可能导致应平仓的头寸被阻断

## 维度3: 返回值语义不一致

### 【R13-RET-01】`send_order` 返回值语义模糊：成功返回order_id，失败返回None
- **文件**: `order_service.py:185-261`
- **行号**: 185-261
- **问题描述**: `send_order`在以下情况均返回None：(1)限流 (2)重复订单 (3)自成交检测 (4)跨策略风控阻断 (5)平台下单异常 (6)重试全部失败。调用方无法区分"被风控阻断"和"网络异常"，可能导致错误的重试策略或告警。在实盘中，自成交阻断和风控阻断是正常的业务逻辑，不应与异常混为一谈
- **严重等级**: **P0**
- **影响系统**: 生产系统
- **风险**: 调用方无法区分阻断原因，可能导致：(1)对风控阻断进行不必要的重试，浪费限流配额；(2)对网络异常不进行告警，错过关键故障窗口

### 【R13-RET-02】`ProfitLossRatioCalculator.update_from_trades` 返回`float('inf')`

## 维度5: 跨模块接口契约

### 【R13-CRS-01】`StrategyCoreService._feed_shadow_engine` 调用`process_signal`缺少`strategy_group`参数
- **文件**: `strategy_core_service.py:372-392` / `shadow_strategy_engine.py:880`
- **行号**: strategy_core_service.py:383-390, shadow_strategy_engine.py:880
- **问题描述**: `_feed_shadow_engine`调用`se.process_signal(market_state=..., instrument_id=..., signal_direction=..., price=..., quantity=..., signal_strength=...)`，但`process_signal`内部调用`process_shadow_a_signal`时传入`strategy_group="s2_resonance"`（默认值）。实际上不同交易目标可能属于不同策略组（s1_hft/s2_resonance/s3_box等），但调用方未传递strategy_group信息，导致所有信号都被归入s2_resonance组
- **严重等级**: **P0**
- **影响系统**: 生产系统
- **风险**: 影子策略Alpha计算仅基于s2_resonance组，无法反映s1_hft/s3_box/s4_spring等策略组的真实表现，Alpha指标失真可能导致降级判断错误

### 【R13-CRS-02】`StrategyCoreService._check_ecosystem_exclusion` 假设了Ecosystem内部状态映射
- **文件**: `strategy_core_service.py:348-370`
- **行号**: 354-359
- **问题描述**: `_check_ecosystem_exclusion`内部硬编码了`state_to_strategy = {'correct_trending': 'master', 'incorrect_reversal': 'reverse', 'other': 'other'}`，与`StrategyEcosystem.switch_active_strategy`中的映射完全相同。但此映射是Ecosystem的内部实现细节，若Ecosystem修改映射（如新增spring策略映射），此处不会同步更新
- **严重等级**: **P0**
- **影响系统**: 生产系统
- **风险**: 若Ecosystem新增策略类型映射但此处未同步，互斥检查会使用错误的strategy_id，可能导致本应被互斥规则阻断的交易通过检查

## 维度6: 构造函数与初始化契约

### 【R13-INIT-01】`StrategyCoreService` Mixin初始化依赖隐式调用顺序
- **文件**: `strategy_core_service.py:60-177`
- **行号**: 60-177
- **问题描述**: `StrategyCoreService(_LifecycleMixin, _InstrumentHelperMixin, HistoricalKlineMixin, TickHandlerMixin)`的`__init__`中手动调用`self._init_historical_kline_mixin()`和`self._init_tick_handler_mixin()`，但未调用`_LifecycleMixin`和`_InstrumentHelperMixin`的初始化方法。这两个Mixin没有定义`__init__`，所以不会出问题，但若未来添加了初始化逻辑，则会被遗漏。此外，`_LifecycleMixin`的`storage`属性是惰性property，但`_storage`在StrategyCoreService.__init__中初始化为None，Mixin的property依赖此属性
- **严重等级**: **P0**
- **影响系统**: 生产系统
- **风险**: Mixin的初始化完全依赖StrategyCoreService.__init__中手动设置属性（如`self._storage = None`、`self._state = StrategyState.INITIALIZING`等），若初始化顺序变更或属性名修改，Mixin方法会AttributeError

### 【R13-INIT-02】`DataService` 多Mixin组合未调用super().__init__
- **文件**: `data_service.py:84-104`
- **行号**: 84-104
- **问题描述**: `DataService.__init__`未调用`super().__init__()`，意味着5个Mixin的`__init__`方法都不会被调用。DataService直接在自身__init__中设置Mixin需要的属性（如`self._query_cache`、`self._lock`等）。若Mixin的__init__中有关键初始化逻辑，则会被跳过
- **严重等级**: P1
- **影响系统**: 生产系统

### 【R13-INIT-03】`OrderService` 继承BaseService但未传递event_bus到后续使用
- **文件**: `order_service.py:81-82`
- **行号**: 81-82
- **问题描述**: `OrderService(BaseService).__init__`调用`super().__init__(event_bus)`将event_bus存入`self.event_bus`，但OrderService内部从未使用`self.event_bus`发布事件。所有订单状态变更（提交、成交、失败、取消）都未通过EventBus通知其他组件，违反了事件驱动架构的设计意图
- **严重等级**: P1
- **影响系统**: 生产系统

### 【R13-INIT-04】`BoxSpringStrategy.__init__` 参数依赖外部dict结构
- **文件**: `box_spring_strategy.py:200-267`
- **行号**: 200-267

### 【R13-INIT-05】`ShadowStrategyEngine` 初始化依赖YAML文件但无降级保障

## 维度7: 异常契约

### 【R13-EXC-01】`_LifecycleMixin.storage` property 抛出RuntimeError但调用方未捕获
- **文件**: `strategy_lifecycle_mixin.py:73-84`
- **行号**: 73-84
- **问题描述**: `storage` property在初始化失败时`raise RuntimeError(f"Storage init failed: {e}")`，但此property的调用方（如TickHandlerMixin中的tick持久化逻辑）通常在try/except中调用，异常会被吞掉。但若在策略启动阶段调用storage（如`on_start`中），RuntimeError会导致策略启动失败且无优雅降级
- **严重等级**: **P0**
- **影响系统**: 生产系统
- **风险**: Storage初始化失败时，策略进入ERROR状态，但未触发安全元层的交易暂停，可能导致在无数据服务的情况下继续交易

### 【R13-EXC-02】`EventBus.subscribe` 在shutdown后抛RuntimeError
- **文件**: `event_bus.py:142-143`
- **行号**: 142-143
- **问题描述**: `if self._shutdown: raise RuntimeError("EventBus已关闭")`，但调用方（如策略启动时注册事件回调）通常不捕获RuntimeError。若EventBus在策略启动前被shutdown（如测试环境），会导致策略初始化失败
- **严重等级**: P1
- **影响系统**: 生产系统/测试系统

### 【R13-EXC-03】`RiskService.check_before_trade` 异常时返回block但未区分异常类型
- **文件**: `risk_service.py:265-267`
- **行号**: 265-267
- **问题描述**: `check_before_trade`的顶层except捕获所有异常并返回`block_result("check_error", ...)`。这是fail-safe设计，但未区分异常类型——网络超时、参数缺失、代码bug等不同异常都返回相同的block结果，调用方无法根据异常类型做差异化处理
- **严重等级**: P1
- **影响系统**: 生产系统

### 【R13-EXC-04】`StrategyCoreService._update_position_from_trade` 吞掉所有异常
- **文件**: `strategy_core_service.py:258-272`
- **行号**: 258-272
- **问题描述**: `_update_position_from_trade`外层try/except仅记录日志`logging.error(...)`，不抛出异常也不返回错误。若持仓更新失败（如PositionService异常），策略不会感知到错误，可能继续基于错误的持仓数据做交易决策
- **严重等级**: P2
- **影响系统**: 生产系统

### 【R13-EXC-05】`StrategyJudgmentEngine.judge` 未校验输入参数完整性
- **文件**: `策略评判/strategy_judgment_engine.py:522-540`
- **行号**: 522-540
- **问题描述**: `judge()`方法接受13个可选参数（均为Optional[Dict]），但未校验必填参数是否为None。当关键参数如`profitability_metrics`为None时，对应维度评分会降级为0分，但不会抛出异常或返回INSUFFICIENT_EVIDENCE。调用方可能不知道评判结果基于不完整数据
- **严重等级**: P2
- **影响系统**: 评判系统
