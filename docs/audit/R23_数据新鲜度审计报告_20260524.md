# R23 数据新鲜度与陈旧数据风险审计报告

**审计日期**: 2026-05-24  
**审计对象**: ali2026v3_trading 交易系统  
**审计标准**: 参数池统一执行方案与使用手册_V7.0_工程落地版_20260510.md  
**审计范围**: 全模块数据缓存、快照、查询、指标计算、回测对齐  

---

## 执行摘要

- **总发现数**: 32项
- **P0（直接导致错误交易决策）**: 6项
- **P1（导致指标计算偏差）**: 15项
- **P2（性能优化不足）**: 11项

**核心风险**:
1. 交易系统价格缓存（config_params.py CACHE_TTL=300s）与实时tick缓存（ds_realtime_cache.py 无TTL）之间没有一致性保障，在极端行情中可能导致5分钟陈旧价格参与开仓决策
2. 期权估值链路（risk_service.py _option_info_cache）无任何过期/新鲜度检查，Greeks计算可能使用数分钟前甚至更早的价格
3. 配置热加载完全依赖手动触发，参数池切换后旧参数可能残留长达300s
4. 信号系统仅有冷却管理，缺乏信号"有效期"概念——signal_max_age参数声明但未实际执行

## 审计维度一：缓存数据过期

### 发现1.1 — config_params.py 价格/参数缓存TTL过长 (P0)
- **文件**: `config_params.py` (CACHE_TTL=300, PARAMS_JSON_CACHE_TTL=300)
- **数据来源**: 参数表JSON文件 + 参数池数据库表
- **缓存策略**: 内存dict + TTL=300s + _last_refresh手动触发
- **最大陈旧时间**: **300秒**（默认）— 对于价格参数（13类核心价格快照）而言过长
- **陈旧时后果**: 极端波动行情中，strategy_core获取到5分钟前的价格参数快照，产生的交易决策基于过时价格，可能导致错误的进场/出场，尤其对短线策略（持仓时间<5min）影响致命
- **证据**: config_params.py定义了`CACHE_TTL = int(os.environ.get('CACHE_TTL', '300'))`，所有13类价格参数（`_FLOAT_COLUMNS_BY_TABLE`）均共用一个300s TTL

### 发现1.2 — ds_realtime_cache.py 价格缓存无TTL机制 (P0)
- **文件**: `ds_realtime_cache.py` 第164-215行 (update_tick), 第217-226行 (get_latest_price)
- **数据来源**: 实时tick流
- **缓存策略**: `_latest_ticks` dict（无TTL），仅在`update_tick`时覆盖；`_tick_sequences` deque（无过期驱逐）
- **最大陈旧时间**: **无限**（直到下一个tick覆盖为止）
- **陈旧时后果**: 若某合约长时间无新tick（如停牌、流动性枯竭），`get_latest_price()` 返回的价格可能是数分钟甚至更久之前的，risk_service的持仓估值、strategy_core的决策逻辑均依赖此接口
- **证据**: `get_latest_price()` 方法仅从 `_latest_ticks` 取值，无任何时间戳检查、无TTL驱逐逻辑

### 发现1.3 — risk_service.py _option_info_cache无TTL (P0)
- **文件**: `risk_service.py` 第692行 (`calc._option_info_cache.get(inst_id, {})`)
- **数据来源**: 期权合约元信息（strike_price, underlying_future_id, future_product, 等）
- **缓存策略**: `_option_info_cache`（dict，无TTL），仅在期权Greeks计算器初始化时填充
- **最大陈旧时间**: **无限**（可能在启动后永不更新）
- **陈旧时后果**: 期权strike_price或underlying_mapping变化后（如主力合约切换），Greeks计算使用错误的strike/underlying，导致delta/gamma/vega全部偏斜，进而导致stress test输出错误的跳空损失估算，影响L1安全层断路器
- **证据**: `risk_service.py:692` 直接读取 `calc._option_info_cache`，无任何过期检查

### 发现1.4 — risk_service.py _strategy_status_cache无TTL (P1)
- **文件**: `risk_service.py` 第366-370行
- **数据来源**: strategy_status（策略状态快照）
- **缓存策略**: 元组 `(ts, cached)`，由 `update_strategy_status()` 写入，`get_strategy_status()` 读取时无过期检查
- **最大陈旧时间**: **无限**（仅被显式更新覆盖）
- **陈旧时后果**: 若strategy_core状态变更（如暂停、恢复）后未触发风险服务更新，风控决策使用旧状态导致误判

### 发现1.5 — width_cache.py _option_info无时间戳/过期标记 (P1)
- **文件**: `width_cache.py` 第177行 (`_option_info: Dict[int, Dict]`)
- **数据来源**: 期权宽跨式状态分类结果
- **缓存策略**: 内存dict，无TTL，无`last_update`时间戳
- **最大陈旧时间**: **无限**
- **陈旧时后果**: StateParamManager读取五态统计快照时可能使用过时的期权状态分类，影响参数池状态评估的准确性

### 发现1.6 — data_service.py _query_cache TTL机制正常但策略层未感知 (P1)
- **文件**: `data_service.py` 第68行 (`QUERY_CACHE_TTL=60`), `ds_query_cache.py` 第84-107行
- **数据来源**: DuckDB查询结果
- **缓存策略**: OrderedDict + MD5 key + 60s TTL + on_tick时按symbol失效
- **最大陈旧时间**: **60秒**（设计合理），但无版本号/序列号标记
- **陈旧时后果**: 主从切换或数据重加载后，缓存可能持续返回旧查询结果60秒而调用方未感知

## 审计维度二：延迟数据混入

### 发现2.1 — ds_realtime_cache.py 启动期tick堆积风险 (P1)
- **文件**: `ds_realtime_cache.py` 第164-179行
- **机制**: 当`_params_service is None`时，所有tick被暂存到`_pending_ticks`，上限`MAX_PENDING_TICKS`个后丢弃
- **风险**: 若params_service初始化延迟（如数据库连接慢），最多`MAX_PENDING_TICKS`个tick会被推迟回放，回放时所有tick时间戳可能已显著晚于实际市场时间
- **后果**: 回放tick在width_cache和indicator计算中被视为"新tick"，但实际上市场已演进

### 发现2.2 — strategy_core_service.py 历史数据加载期间降级路径 (P1)
- **文件**: `strategy_core_service.py` 第357-360行
- **机制**: `_historical_load_in_progress` 为 True 时，tick通过 `_dispatch_tick_degraded` 处理
- **风险**: 降级路径与非降级路径使用不同处理逻辑，降级期间产生的指标计算可能混入历史数据与实时数据的混合结果
- **后果**: 策略在历史数据加载完成前后的决策信号可能不一致

### 发现2.3 — subscription_manager.py 订阅重连回放无时间戳保护 (P1)
- **文件**: `subscription_manager.py` 第668-693行
- **机制**: `_cleanup_expired_events` 基于 `max_event_age_seconds` 清理过期事件，但重连回放的数据没有"tick年龄"校验
- **风险**: 网络中断恢复后，底层连接通道可能推送积压的旧tick，subscription_manager接收后将其作为当前tick分发
- **后果**: 旧tick混入新tick流，width_cache的状态更新可能短暂震荡

### 发现2.4 — L2数据盘口快照无新鲜度标记 (P2)
- **文件**: `ds_realtime_cache.py` 第238-244行 (`get_spread`)
- **风险**: `get_spread()` 直接从 `_latest_ticks` 提取bid_price/ask_price，如果bid/ask更新频率低于last_price（常见于低流动性期权），spread快照可能比last_price更陈旧
- **后果**: 回测/实盘中使用的spread可能是几秒甚至几十秒前的

### 发现3.3 — strategy_tick_handler.py 无kline_max_age强制校验 (P0)

### 发现3.4 — risk_service.py Greeks计算使用`ds.realtime_cache.get_latest_price` (P1)

- **陈旧时后果**: 若underlying期货tick延迟（如主力合约切换窗口期），delta/gamma使用过时标的价格，stress test跳空模拟输出错误PnL估算

### 发现3.5 — risk_service.py IV（隐含波动率）取值无新鲜度检查 (P1)

## 审计维度四：数据库查询结果时效

### 发现4.1 — ds_query_cache.py 查询缓存无version/etag机制 (P1)
- **文件**: `ds_query_cache.py` 第84-107行
- **机制**: MD5(sql+params)作为缓存key，60s TTL
- **风险**: 同一SQL在60s内多次执行返回相同缓存，即使底层DuckDB表已更新（除非对应symbol有新tick触发缓存失效）
- **后果**: 跨模块读取同一结果集时，若A模块已写入新数据但B模块的查询缓存在TTL内，B获取到旧数据

### 发现4.2 — DuckDB主从无延迟监控 (P2)
- **文件**: `data_service.py`
- **当前状态**: 使用单DuckDB实例，不存在主从架构
- **风险**: 若将来引入读写分离架构，需关注主从延迟
- **评估**: 当前为单实例，此项无直接风险，标记为**架构预留**

### 发现4.3 — get_latest_price的双路径返回不一致时间点 (P2)
- **文件**: `ds_query_cache.py` 第109-123行
- **机制**: 优先从RealTimeCache获取，fallback到DuckDB `latest_prices` 表
- **风险**: RealTimeCache的价格与DuckDB持久化价格更新时间差可达秒级，调用方两种返回路径的价格可能不是同一时间点
- **后果**: 某些symbol走cache路径、某些走DB路径时，价格数据一致性差

## 审计维度五：跨模块数据一致时间点

### 发现5.1 — strategy_core_service 持仓视图 vs position_service 持仓快照 (P1)
- **strategy_core_service**: 通过 `on_tick` → `_dispatch_tick` → `_process_tick` 链条更新内部持仓视图
- **position_service**: 通过 `position_service.get_positions()` 从数据库读取持仓快照
- **时间差**: strategy_core的内存持仓更新是tick驱动的（毫秒级），position_service的持仓快照可能滞后数秒（取决于持久化频率）
- **风险**: 同一时刻两个模块看到不同的持仓状态
- **证据**: `position_service.py` 内部持仓从数据库加载，而 strategy_core 通过事件驱动更新

### 发现5.2 — risk_service 敞口 vs order_service 挂单 (P1)
- **risk_service**: 通过 `_compute_stress_test` / `_compute_pnl_attribution` 计算敞口，使用`_positions_dict`
- **order_service**: 管理挂单队列
- **风险**: risk_service计算敞口时不包括已挂出但未成交的订单，敞口统计偏小；尤其在快速市场中，大量挂单堆叠时敞口显著低估
- **时间差**: 挂单存在直到成交/撤单之间的整个窗口期

### 发现5.3 — width_cache 五态快照与 StateParamManager 读取时差 (P2)
- **文件**: `width_cache.py` 第1044-1053行 (`get_status_counts_snapshot`)
- **机制**: StateParamManager通过深拷贝读取快照，但width_cache在两次快照之间的五态变更（尤其是高频tick）未被捕获
- **时间差**: 两次快照读取之间的所有中间态变更

### 发现5.4 — 多模块共用 `ds.realtime_cache` 的一致性窗口 (P2)
- **涉及模块**: strategy_core, risk_service, width_cache, order_flow_bridge, 等
- **风险**: 多个模块在同一tick窗口内读取 `_latest_ticks`，但底层dict被shard_lock保护，不同模块可能读取到不同版本的价格（两次`update_tick`之间）
- **后果**: 极端小tick间隔下，A模块读到的价格可能与B模块不同

## 审计维度六：信号时效性

### 发现6.1 — signal_max_age参数未实际执行失效剔除 (P0)
- **文件**: `signal_service.py`（全文搜索 `signal_max_age`）
- **声明的参数**: `kline_max_age` 在config_params/task_scheduler中声明
- **实际情况**: signal_service.py中**无任何基于信号时间的过期剔除逻辑**；信号生成后即永久存在于 `_signal_history`（最多1000条）
- **最大陈旧时间**: 理论上无限（仅在deque满时被LRU淘汰）
- **陈旧时后果**: 数分钟前生成的信号（当时价格已过时）仍参与决策分数计算、模式引擎过滤、回测模拟

### 发现6.2 — 信号冷却与信号过期的混淆 (P1)
- **文件**: `signal_service.py` 第55行 (DEFAULT_COOLDOWN_SECONDS=60), 第548-556行 (cleanup)
- **机制**: `_cooldown_times` 仅用于防止同一(instrument, signal_type)的重复信号，cleanup每300s清理过期的冷却条目
- **问题**: 冷却管理的是"信号重复频率"，而非"信号是否仍有效"——这是两个正交维度
- **风险**: 市场快速反转时，冷却期内价格可能已显著偏离信号生成时价格，但系统不检查

### 发现6.3 — 信号从生成到执行的端到端延迟无监控 (P1)
- **文件**: `signal_service.py` 第289-352行
- **机制**: `generate_signal()` → `_emit_signal()` → EventBus发布 → order_service接收执行
- **风险**: 若链条中任一环节阻塞（如EventBus满、order_service忙），信号到执行的延迟可能超过有效窗口
- **证据**: 无端到端延迟监控

### 发现6.4 — ModeEngine信号过滤使用 days_to_expiry（合约到期日）而非信号有效期 (P2)
- **文件**: `signal_service.py` 第225-236行
- **机制**: `ModeEngine.filter_signal_by_mode(signal_type, signal_strength, days_to_expiry=...)`
- **风险**: `days_to_expiry` 是期权合约剩余天数（通常>10天），与信号自身有效期（秒级）完全无关
- **后果**: 模式引擎考虑了错误的时间维度

## 审计维度七：配置文件热加载滞后

### 发现7.1 — config_params.py 300s TTL导致参数池切换延迟 (P0)
- **文件**: `config_params.py` (CACHE_TTL=300, PARAMS_JSON_CACHE_TTL=300)
- **机制**: `_last_refresh` 与 `_cache_age()` 对比，超过TTL才重新从文件/数据库加载
- **最大陈旧时间**: **300秒**（参数池切换后，调用方可能在5分钟内仍使用旧参数）
- **陈旧时后果**: 参数池切换（如从default切换到high_vol）后，strategy_core继续使用旧参数运行5分钟，所有仓位分配、风险阈值、信号过滤阈值都基于旧参数池
- **证据**: config_params.py中的`_cache_age()`方法对比`_last_param_table_update`和`_last_params_json_update`与`time.time()`

### 发现7.2 — config_service.py reload()仅手动触发 (P1)
- **文件**: `config_service.py` 第356-363行
- **机制**: `reload()` 检查 `_config_file_path.stat().st_mtime > _last_modified_time`
- **问题**: 无自动轮询/文件监控/hot-reload线程，完全依赖外部调用
- **最大陈旧时间**: **无限**（直到有人调用reload()或重启服务）
- **陈旧时后果**: 配置文件（config.json）修改后策略可能永不感知

### 发现7.3 — params_service.py force_refresh()手动触发无效化 (P1)
- **文件**: `params_service.py` 第861-892行
- **机制**: `force_refresh(path)` 手动强制刷新特定参数或全部参数
- **问题**: 无自动检测参数文件变更的机制，依赖运维手动操作
- **最大陈旧时间**: **无限**（无定时自动刷新）

### 发现7.4 — signal_service 启动时一次性读配置，运行时不刷新 (P1)
- **文件**: `signal_service.py` 第93-113行
- **机制**: `__init__` 时从 `ConfigService` 和 `get_cached_params` 读取冷却时间和清理间隔
- **问题**: 运行期间若配置变更，信号服务不会自动重新读取
- **最大陈旧时间**: **运行期内永不更新**

### 发现7.5 — 各模块独立初始化 config_params，缺乏统一失效总线 (P1)
- **涉及文件**: signal_service.py:95, strategy_core_service.py, risk_service.py, params_service.py
- **机制**: 各模块在 `__init__` 时独立调用 `get_cached_params()` 缓存子集
- **问题**: 参数变更后，各模块持有的本地缓存副本不会同步失效，需各自重新调用
- **证据**: signal_service.py 第93-98行缓存了`default_order_flow_consistency`，运行时不可更新
