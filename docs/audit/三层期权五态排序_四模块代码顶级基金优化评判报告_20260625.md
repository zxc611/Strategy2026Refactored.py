## 二、layer1_month_sort.py 优化评判

### 2.1 P0级问题（生产阻塞）

#### P0-01: GreeksHardFilter.filter_batch 缺少空值防御

**位置**: `layer1_month_sort.py:70-85`

**问题**: `filter_batch` 方法未对 `options` 列表中的元素做空值/缺失键防御。当 `o.get('delta', 0.0)` 返回 `None`（上游数据污染），`abs(delta)` 将抛出 `TypeError`。

**机构级标准**: 所有外部数据入口必须做类型校验和空值防御，使用 `guard` 模式。

**优化方案**:
```python
def filter(self, delta: float, gamma: float, theta: float,
           option_price: float, month_type: str) -> bool:
    # 类型防御
    try:
        delta = float(delta) if delta is not None else 0.0
        gamma = float(gamma) if gamma is not None else 0.0
        theta = float(theta) if theta is not None else 0.0
        option_price = float(option_price) if option_price is not None else 0.0
    except (TypeError, ValueError):
        return False
    # ... 原有逻辑
```

#### P0-02: compute_metrics 中 correct_up_pct 计算与方案文档不一致

**位置**: `layer1_month_sort.py:239-285`

**问题**: 方案文档3.4节定义 `correct_up_pct_1 = Σ_m [ w_m × (CR_m_eff + CF_m_eff) ] / Σ_m [ w_m × Total_m_eff ]`，但代码实现中 `correct_up_pct` 使用 `cr / rise`（仅CR占看涨比例），而非 `(CR+CF)/Total`。这是**语义不一致**：方案文档的"正确率"包含看涨CR和看跌CF，代码只计算了CR占看涨的比例。

**影响**: Tier分级和后续排序均基于此指标，偏差将传导至全链路。

**优化方案**: 严格按方案文档重写：
```python
correct_up_pct += w * (cr + cf) / max(total, 1e-9)
```

#### P0-03: AsymmetricDecay 与 FreshnessCalculator 衰减逻辑重复且不一致

**位置**: `layer1_month_sort.py:88-153`

**问题**: 
- `AsymmetricDecay.decay_counts` 使用 `cfg.ASYMMETRIC_DECAY` 配置（按行为簇差异化半衰期）
- `FreshnessCalculator.compute` 使用硬编码的 `FRESHNESS_CONFIG`（固定半衰期600/120/60）
- 两者对同一组五态数据做衰减，但使用不同参数，导致 `freshness` 与 `decayed` 计数语义不统一

**机构级标准**: 单一数据源原则（Single Source of Truth），同一衰减逻辑只能有一套参数。

**优化方案**: `FreshnessCalculator` 应接收 `AsymmetricDecay` 实例或共享同一配置源：
```python
class FreshnessCalculator:
    def __init__(self, decay: AsymmetricDecay):
        self._decay = decay
    
    def compute(self, counts, last_update, now):
        decayed = self._decay.decay_counts(counts, last_update, now)
        # 使用 decayed 结果 + 权重配置计算 freshness
```

#### P1-03: sort_weighted 直接修改输入 candidates

**位置**: `layer1_month_sort.py:316-327`

**问题**: `sort_weighted` 方法通过 `c['score_1'] = ...` 直接修改输入字典，违反不可变数据原则。若同一 candidates 列表被多次排序（如A/B测试），会导致数据污染。

**优化方案**: 使用浅拷贝或返回新列表：
```python
def sort_weighted(self, candidates):
    scored = []
    for c in candidates:
        copy = dict(c)
        copy['score_1'] = (...)
        scored.append(copy)
    return sorted(scored, key=lambda x: -x.get('score_1', 0.0))
```

#### P1-04: emit_open_signal 缺少信号去重和限频

**位置**: `layer1_month_sort.py:445-465`

**问题**: 每次调用都无条件发出信号，无品种级去重（同品种短时间内可能发出多个信号）、无信号限频（高频场景下信号洪泛）。

**优化方案**: 添加信号去重窗口（如60秒内同品种仅发一次信号）。

#### P0-04: rank_cluster 中否决模式的 counts 构造逻辑错误

**位置**: `layer2_cluster_sort.py:274-283`

**问题**: 否决模式下，为每个品种构造 `counts` 字典时：
```python
counts = {
    'CR': r.get('total_cr_eff', r.get('correct_up_pct_1', 0) * 100),
    'CF': r.get('total_cf_eff', 0),
    'WR': r.get('noise_ratio_1', 0) * 50,
```

```python
product_accuracy = r.get('correct_up_pct_1', 0.0)
group_accuracy = sum(lr.get('correct_up_pct_1', 0.0) for lr in layer1_results) / len(layer1_results)
resonance = product_accuracy / group_accuracy if group_accuracy > 0 else 1.0
```

## 四、layer3_global_sort.py 优化评判

### 4.1 P0级问题

#### P0-06: MarketCircuitBreaker 缺少状态重置保护和持久化

**位置**: `layer3_global_sort.py:26-81`

**问题**:
- `consecutive_minutes` 在进程重启后归零，可能错过已持续的熔断状态
- `defense_mode` 无持久化，重启后默认为 `False`，极端行情下可能误开仓
- `check` 方法中 `consecutive_minutes` 按调用次数递增，但方法名暗示按分钟——若调用频率非1次/分钟，逻辑错误

**机构级标准**: 熔断状态必须持久化至数据库，重启后恢复。时间维度必须基于实际时间戳而非调用次数。

**优化方案**:
```python
def check(self, market_accuracy, current_volatility, now: float = None) -> str:
    if now is None:
        now = time.time()
    trigger = (market_accuracy < self.acc_floor or current_volatility > self.vol_99th)
    if trigger:
        if self._first_trigger_ts is None:
            self._first_trigger_ts = now
        elapsed_min = (now - self._first_trigger_ts) / 60.0
        if elapsed_min >= cfg.CIRCUIT_BREAKER_CONSECUTIVE_MIN:
            self.defense_mode = True
            return 'defense'
    else:
        self._first_trigger_ts = None
        self.defense_mode = False
    return 'normal'
```

#### P0-07: compute_market_state 缺少 market_bias 计算

**位置**: `layer3_global_sort.py:214-246`

**问题**: 方案5.1节定义两个市场基准指标：
- `market_accuracy = Σ(CR_eff + CF_eff) / Σ(Total_eff)` ✅ 已实现
- `market_bias = Σ(CR_eff + WR_eff) - Σ(CF_eff + WF_eff)` ❌ 未实现

`market_bias` 用于判断市场方向（看涨/看跌），直接影响 `market_state` 标签（`strong_bull` vs `strong_bear`）。当前代码中 `market_state` 始终为 `strong_bull`/`bull`，无法区分多空方向。

**优化方案**:
```python
market_bias = (total_cr + total_wr) - (total_cf + total_wf)
if market_accuracy >= self.p80:
    market_state = 'strong_bull' if market_bias > 0 else 'strong_bear'
```

### 4.2 P1级问题

#### P1-08: DynamicBudgetAllocator.allocate 簇预算未归一化

**位置**: `layer3_global_sort.py:124-168`

**问题**: 簇预算计算 `cluster_budgets[cluster_id] = min(max_single_cluster, (1/ATR) / total_inv_atr)`，当多个簇的 `(1/ATR) / total_inv_atr` 之和被 `max_single_cluster` 截断后，总预算可能不等于1.0，导致品种级预算分配不完整。

#### P1-10: _compute_suggested_lots 手数计算过于粗糙

**位置**: `layer3_global_sort.py:358-377`

**问题**: 
- Tier 1 固定2手、Tier 2/3 固定1手，未考虑账户规模、品种ATR、波动率
- 未考虑 `position_limit_pct` 与 `position_cap` 的联合约束
- 机构级手数计算应基于Kelly公式或风险预算（如单笔最大亏损不超过权益的1%）

**优化方案**: 基于ATP（Average True Range Percent）和风险预算计算：
```python
def _compute_suggested_lots(self, candidate, position_cap, account_equity, atr_pct):
    max_loss_per_lot = account_equity * 0.01  # 单笔最大亏损1%
    lots_by_risk = max_loss_per_lot / (atr_pct * contract_multiplier)
    lots_by_budget = account_equity * position_cap * candidate['position_limit_pct']
    return max(0, int(min(lots_by_risk, lots_by_budget)))
```

#### P1-11: 历史分位数默认值缺乏统计依据

**位置**: `layer3_global_sort.py:203-212`

**问题**: 当 `history_market_accuracy` 为空时，默认 `p10=0.30, p20=0.40, p50=0.50, p80=0.65`。这些值缺乏统计依据，且与实际市场分布可能严重偏离。

**优化方案**: 无历史数据时应进入"保守模式"（默认p50=0.40），并触发告警要求尽快补充历史数据。

## 五、three_decision_source_router.py 优化评判

### 5.1 P0级问题

#### P0-08: _check_dm_test_pass 为占位实现，生产不可用

**位置**: `three_decision_source_router.py:197-212`

**问题**: 方案6.3节明确要求"运行2周后用Diebold-Mariano检验对比，p<0.05且Calmar提升≥0.1才推广"。但 `_check_dm_test_pass` 仅检查 `calmar_ratio差值 >= 0.1`，未实现真正的DM检验。这导致：
- A/B测试结论不可靠，可能误选决策源
- 不满足方案验收标准

**优化方案**: 集成 `statsmodels` 的DM检验：
```python
def _check_dm_test_pass(self) -> bool:

    # 获取实验组和对照组的PnL序列
    # 计算损失差异序列 d_t = L_experiment - L_control
    # DM统计量 = mean(d) / (sqrt(var(d) * (1 + 2*sum(acf(d, k)))) / sqrt(T))
    # 与标准正态临界值比较
```

#### P0-09: route 方法缺少信号去重和冲突解决

**位置**: `three_decision_source_router.py:121-156`

**问题**: 
- 三个决策源可能对同一品种发出不同信号（如Layer1说Tier2开仓，Layer3说Tier1开仓），路由器仅返回选中源的信号，未处理跨源冲突
- 无品种级信号去重：同一品种可能在不同层被多次选中
- 无信号时间戳对齐：三层信号可能在不同时刻发出

**机构级标准**: 多决策源路由必须实现：
1. 品种级信号去重（同品种仅保留最优信号）
2. 跨源冲突解决策略（如"最保守原则"取最高Tier）
3. 信号时间窗口对齐

**优化方案**:
```python
def route(self, layer1_signals, layer2_signals, layer3_signals):
    # 1. 收集所有信号
    all_signals = self._collect_all_signals(...)
    
    # 2. 品种级去重与冲突解决
    deduplicated = self._deduplicate_by_product(all_signals)
    
    # 3. 返回选中决策源的信号
    return self._filter_by_source(deduplicated, selected)
```

### 5.2 P1级问题

```python
if self.ab_test_enabled:
    return {
        'experiment': {'signals': best_source_signals, 'alloc': 0.20},
        'control': {'signals': layer3_signals, 'alloc': 0.80},

```

#### P1-13: SignalRecord 缺少关键字段

**位置**: `three_decision_source_router.py:46-53`

**问题**: `SignalRecord` 仅有 `decision_source, product_id, tier, score, timestamp, pnl`，缺少：
- `signal_type`（开仓/平仓/调仓）
- `direction`（多/空）
- `suggested_lots`（建议手数）
- `market_state`（市场状态）
- `risk_flags`（风控标记）

这些字段对后续量化研究和PnL归因至关重要。

#### P1-14: _select_best_source 默认降级到 layer3 逻辑不合理

**位置**: `three_decision_source_router.py:158-189`

**问题**: 无回测结果时默认使用 `layer3`，但 `layer3` 是最复杂的层，依赖 `layer2` 输出。若 `layer2` 数据不足，`layer3` 结果可能比 `layer1` 更差。应默认使用最稳定的层（`layer1`），而非最复杂的层。

**优化方案**: 默认降级到 `layer1`（最稳定、最少依赖），而非 `layer3`。

#### P1-15: 缺少信号历史持久化

**位置**: `three_decision_source_router.py:79-84`

**问题**: 信号历史使用 `deque(maxlen=10000)` 内存存储，进程重启后丢失。量化研究需要长期信号历史进行回测和归因分析。

**优化方案**: 信号记录持久化至DuckDB：
```python
def collect_signal(self, layer, signal):
    # 内存缓存 + 异步写入DuckDB
    self._signal_history[layer].append(record)
    self._persist_to_duckdb(record)
```

## 六、跨模块系统性问题

### 6.1 数据不可变原则系统性违反

**涉及模块**: layer1_month_sort.py, layer2_cluster_sort.py, layer3_global_sort.py

**问题**: 三个模块均存在直接修改输入字典的问题（P1-03, P1-05, P1-09），这是系统性设计缺陷。在机构级量化系统中，数据流应遵循不可变原则（Immutable Data Pipeline），每层输出为新对象，不污染输入。

**优化方案**: 引入 `frozen_dict` 或 `dataclass(frozen=True)` 作为数据载体：
```python
@dataclass(frozen=True)
class Layer1Result:
    product_id: str
    tier: int
    score_1: float
    freshness: float
    # ...
```

### 6.2 缺少统一时钟源

**涉及模块**: 全部四个模块

**问题**: 各模块独立使用 `time.time()`，在分布式或高频场景下，不同模块的时间戳可能不一致，导致衰减计算、信号时序混乱。

**优化方案**: 由上层提供统一时钟源（如 `ClockService`），所有模块使用同一时间戳。

### 6.3 缺少DuckDB持久化层

**涉及模块**: 全部四个模块

**问题**: 用户偏好使用DuckDB存储（PREFERENCE_4），但四个模块均无数据库写入逻辑。所有中间结果、信号记录、聚类模型、熔断状态均仅在内存中。

**优化方案**: 添加 `ThreeLayerSortRepository` 持久化层：
```python
class ThreeLayerSortRepository:
    def __init__(self, db_path: str):
        self._conn = duckdb.connect(db_path)
        self._init_tables()
    
    def save_layer1_result(self, result: Dict): ...
    def save_signal(self, signal: Dict): ...
    def load_cluster_model(self, version: int): ...
    def save_circuit_breaker_state(self, state: Dict): ...
```

### 6.4 缺少信号唯一标识

**涉及模块**: three_decision_source_router.py

**问题**: 用户偏好语义化命名（PREFERENCE_3），但信号无唯一ID。信号应包含：策略名+开仓信号状态快照+平仓方式等属性信息。

**优化方案**:
```python
signal_id = f"TLS_{decision_source}_{product_id}_T{tier}_{int(timestamp)}_{direction}"
```

## 九、结论与建议

### 9.1 核心判断

四模块代码在**功能覆盖面**上与落地方案高度对齐，核心算法逻辑完整。但从顶级基金生产标准看，存在以下系统性短板：

1. **数据一致性**: correct_up_pct计算、衰减参数双源、market_bias缺失——3处与方案文档不一致
2. **生产健壮性**: 空值防御、异常隔离、幂等性、持久化——均缺失
3. **可观测性**: 无结构化日志、无指标埋点、无审计追踪——运维盲区
4. **路由器可靠性**: DM检验占位、信号去重缺失、A/B测试不完整——决策质量不可控

### 9.2 建议执行顺序

1. **立即修复P0-02（correct_up_pct不一致）**——影响全链路指标
2. **立即修复P0-07（market_bias缺失）**——影响市场方向判断
3. **1周内修复P0-01/P0-03/P0-04**——数据质量基础
4. **2周内修复P0-06/P0-08/P0-09**——生产安全基础
5. **4周内完成P1级全部修复**——质量提升
6. **8周内完成P2级演进**——机构化升级
