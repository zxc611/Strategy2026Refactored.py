## 2. 审计维度与方法论

### 审计 8 维度

| 维度 | 描述 | 风险指标 |
|------|------|---------|
| D1 | 裸except吞异常 | `except: pass` / `except Exception: pass` 未重抛 |
| D2 | 异常上下文丢失 | `raise` 新异常未用 `from` 保留链 |
| D3 | 异常转为正常返回值 | catch 后返回 None/空列表/零值 |
| D4 | finally 异常覆盖 | finally 中异常覆盖原始异常 |
| D5 | 跨层级传播断裂 | 底层异常被中间层吞没 |
| D6 | 异常处理不一致 | 同类错误不同模块处理方式不同 |
| D7 | 资源泄漏伴随异常 | 异常路径未清理资源 |
| D8 | 日志误报/漏报 | 异常降级为 warning/debug、关键异常未记录 |

### 分级标准

- **P0（致命）**：静默异常 → 交易决策错误，可能导致资金损失
- **P1（严重）**：数据完整性风险、功能退化，影响策略正确性
- **P2（改进）**：可观测性缺失、代码健壮性不足，不直接影响当前功能

## 3. P0 问题

### P0-1：risk_service.py:240 — 影子引擎检查静默吞异常

**位置**：`risk_service.py` 第 240-241 行  
**代码**：
```python
try:
    from ali2026v3_trading.shadow_strategy_engine import get_shadow_strategy_engine
    _sse = get_shadow_strategy_engine()
    if _sse is not None:
        if _sse.is_absolute_ev_paused():
            return RiskCheckResponse.block_result(...)
        if _sse.is_degradation_active():
            logging.warning(...)
except Exception:
    pass
```

**风险分析**：

- 此块位于 `check_before_trade()` 主风控入口中（第 217 行）
- 如果 `get_shadow_strategy_engine()` 因为模块加载异常、运行时错误等原因失败，异常被完全静默吞没
- **传播链路**：`check_before_trade()` → 调用方（`execute_option_trading_cycle` / `SignalService`）→ 继续交易
- 后果：影子引擎 EV 暂停/降级状态未检测到，可能允许在应阻断的情况下开仓

**影响等级**：**P0 致命**  
**建议修复**：至少改为 `logging.error` + 返回 `block_result("shadow_engine_error")` 进行 fail-safe 阻断

### P0-4：risk_service.py:1394-1395 — Alpha 衰减获取静默吞异常

### P0-5：risk_service.py:1040-1042 — 希腊字母检查异常静默放行

**位置**：`risk_service.py` 第 1037-1042 行  
**代码**：
```python
except ImportError:
    logging.debug("[RiskService._check_greeks_limits] GreeksCalculator not available, skip")
except Exception as e:
    logging.warning("[RiskService._check_greeks_limits] Check error: %s", e)

return RiskCheckResponse.pass_result()  # ⚠️ 异常后仍返回 PASS
```

**风险分析**：

- `_check_greeks_limits()` 是希腊字母硬约束检查，位于 `check_before_trade()` 主风控链中
- `ImportError` → PASS（合理：GreeksCalculator 未安装时跳过）
- **`Exception` → PASS（不合理）**：如果是运行时异常（如计算溢出、数据损坏），应阻断而非放行
- 该函数在 `check_before_trade()` 的第 272 行被调用，异常后返回 PASS，信号继续处理

**传播链路**：`check_before_trade()` → `_check_greeks_limits()` → 异常 → `pass_result()` → 风控绕过

**影响等级**：**P0 致命**  
**建议修复**：区分 `ImportError`（预期跳过）和 `Exception`（应 fail-safe 阻断）

### P0-6：strategy_core_service.py:313-314 / 368-370 — 风控/互斥检查异常降级为 warning 继续

**位置**：`strategy_core_service.py` 第 313-314 行、368-370 行  
**代码**：
```python
# (1) execute_option_trading_cycle: 风控检查异常时继续交易
except Exception as e:
    logging.warning("[OptionTrading] 风控检查异常: %s, 继续交易", e)

# (2) _check_ecosystem_exclusion: 互斥检查异常时阻断交易
except Exception as e:
    logging.warning("[StrategyCoreService._check_ecosystem_exclusion] Error: %s, 阻断交易以保护安全", e)
    return False
```

**风险分析**：

- **(1) 不一致**：风控检查异常时"继续交易"（第 314 行），而互斥检查异常时"阻断交易"（第 370 行）——两者同为保护性检查，处理方式相反
- **(1) 危险**：`check_before_trade()` 本身已有 fail-safe 阻断逻辑（第 285-287 行），但**此处的 try/except 在 check_before_trade() 外部**，捕获的是调用包装代码的异常，而非风控本身的异常。然而，如果 `get_risk_service()` 返回 None 后的调用出错，异常被吞没且交易继续

**传播链路**：`execute_option_trading_cycle()` → 风控异常 → `logging.warning` + "继续交易" → 绕过风控

**影响等级**：**P0 致命**  
**建议修复**：风控检查异常时统一应"阻断交易"（fail-safe），与互斥检查保持一致

### P1-15：query_service.py — `resolve_instrument_exchange` 异常重抛不正确

**位置**：`query_service.py` 第 395-404 行  
**代码**：
```python

    logging.warning(...)
    return 'UNKNOWN'
```
