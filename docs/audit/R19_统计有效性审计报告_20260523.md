### P0-02: CounterfactualValidator 使用 simple shuffle 而非 block bootstrap — 破坏时间依赖结构

| 字段 | 内容 |
|:---|:---|
| **文件路径** | `参数池/statistical_validation.py` |
| **行号范围** | 64-67 |
| **手册引用** | 2.4节: "保持期货涨跌不变，将期权涨跌标签在行权价和月份间随机打乱" |
| **严重等级** | 🔴 P0 |
| **影响系统** | 测试系统 |
| **问题描述** | 第65行 `shuffled_returns = np.random.permutation(option_returns)` 使用的是 simple shuffle（完全随机排列），这破坏了时间序列的局部依赖结构（波动率聚类、自相关、趋势持续性）。手册描述的打乱是针对"行权价和月份间"的结构化随机化，而非全局shuffle。`MonteCarloBankruptcyValidator.validate_block_bootstrap()`（行261-345）已经正确实现了 block bootstrap，但 `CounterfactualValidator` 没有复用它。金融时间序列的block bootstrap（block_size≥5）是标准方法，simple shuffle会导致虚无分布标准差被低估，使真实策略更容易通过检验，产生**虚假显著性**。 |
| **后果** | 过拟合策略的虚假设分布被人为收窄，p值系统性偏低，假阳性率失控。 |

## 四、统计显著性检验

### P0-07: run_backtest 的 Sharpe 年化系数错误 — `np.sqrt(252*240)` 失真

| 字段 | 内容 |
|:---|:---|
| **文件路径** | `参数池/task_scheduler.py` |
| **行号范围** | 1941 |
| **手册引用** | 未直接提及，属基础统计规范 |
| **严重等级** | 🔴 P0 |
| **影响系统** | 生产系统、测试系统、评判系统 |
| **问题描述** | 第1941行 `sharpe = np.sqrt(252 * 240) * mean_r / std_r` 使用的年化系数 `sqrt(252 * 240)`。该公式的逻辑是：每分钟1个Bar × 每天240分钟 × 252个交易日 = 60480个bar/年。然而实际数据中：1) 并非每个交易日都有240分钟数据（夜盘+日盘合计约345分钟，但数据密度各异）；2) 收益率 `mean_r` 是逐bar收益率，乘以 `sqrt(252*240)` 做年化仅在收益率i.i.d.且bar间隔恒定的前提下成立；3) 期权标的有非交易时段（午休、收盘后），bar密度随时间变化。正确的做法应该是：先计算日收益率，再按252天年化，即 `np.sqrt(252) * mean_daily_return / std_daily_return`。此错误会系统性高估Sharpe比率（`sqrt(240)≈15.5`倍放大），导致回测夏普虚高约15倍以上。 |
| **后果** | 所有基于 `run_backtest()` 的夏普报告严重失真。**这是回测冠军实战崩盘的核心技术原因之一**——回测显示Sharpe=3.0的策略，实际日度Sharpe可能只有0.2。 |

### P1-06: 缺少收益正态性检验 — t检验假设不满足风险

| 字段 | 内容 |
|:---|:---|
| **文件路径** | `参数池/task_scheduler.py` (run_backtest), `参数池/statistical_validation.py` |
| **行号范围** | task_scheduler: 1843-1975 |
| **手册引用** | 未直接提及 |
| **严重等级** | 🟡 P1 |
| **影响系统** | 测试系统、评判系统 |
| **问题描述** | `compute_alpha_confidence_interval()` 使用 `z = stats.norm.ppf()` 或 `stats.t.ppf()` 计算CI，这些方法假设收益率服从正态或t分布。期权策略收益高度非正态（正偏态=多发小额亏损偶发大额盈利，或负偏态=稳定盈利偶发巨亏）。未检验偏度(skewness)和峰度(kurtosis)，可能使CI宽度系统性地被低估。应在计算CI前执行Jarque-Bera检验或至少报告偏度/峰度。 |
| **建议** | 在 `compute_alpha_confidence_interval()` 开头加入偏度/峰度检查，若 |skewness| > 1 或 kurtosis > 5，警告用户使用 Bootstrap CI 替代参数CI。 |

### P1-07: 信号数≥30 作为统计显著性最低要求过于宽松

| 字段 | 内容 |
|:---|:---|

## 十、审计结论

### 10.1 总体判定

**🔴 高风险 — 回测系统存在重大统计严谨性缺口**。虽然代码库中已实现了手册要求的大部分验证组件（反事实验证、蒙特卡洛破产检验、walk-forward检验、CI计算、多次比较校正等），但关键组件的集成链路断裂：验证组件与其调用方（主回测流水线）之间存在"已实现但未调用"的间隙。

此外，`run_backtest()` 中 Sharpe 年化系数 `sqrt(252*240)` 存在根本性错误，导致所有策略的Sharpe报告被系统性高估约15倍——这是此类系统的"定时炸弹"级缺陷。

### 10.2 建议修复优先级

| 优先级 | 修复项 | 预估工时 |
|:---|:---|:---|
| **最高** | 修复 `run_backtest()` 的 Sharpe 年化系数（P0-07） | 0.5h |
| **最高** | 将 `CounterfactualValidator` 集成到 `_run_final_checks()`（P0-01） | 1h |
| **高** | 重构 `WalkForwardValidator` 为真实 expanding-window 穿行（P0-06） | 4h |
| **高** | 将 `compute_alpha_confidence_interval()` 集成到 `run_backtest()` 返回值（P0-10） | 1h |
| **高** | 强制 `L2Optimizer` 要求独立数据集，否则报错而非降级（P0-11） | 0.5h |
| **中** | 修复 Optuna BH 校正的 p_value 输入源头（P0-09） | 2h |
| **中** | `CounterfactualValidator` 改用 block bootstrap（P0-02） | 2h |
| **中** | 修复 `L2Optimizer.cross_fold_overlap` 恒为0（P0-04） | 3h |

### 10.3 对已有回测结果的影响评估

由于P0-07（Sharpe年化错误），建议对**所有**已有的回测结果做如下修正：

```
修正后Sharpe = 报告Sharpe / sqrt(240) ≈ 报告Sharpe / 15.49
```

示例：报告Sharpe=3.0 → 修正后日度Sharpe≈0.19

此修正后，多数"回测冠军"策略的实际风险调整后收益将退化至不可交易水平，印证了审计主题"回测冠军实战崩盘"的根因。
