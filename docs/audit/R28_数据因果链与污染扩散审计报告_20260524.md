## 2. 维度1: 单个错误值的传播距离

### 2.1 P0-C001: tick入口NaN/Inf价格传播到全链路

**污染源头**: `strategy_tick_handler.py:286` — `_get_tick_field(tick, 'last_price', 0.0)`

**关键证据**:
- 第286行: `last_price = self._get_tick_field(tick, 'last_price', 0.0)` — 无NaN/Inf检查
- 第301-310行: `last_price` 直接喂入 `OrderFlowBridge.on_tick_feed()` — 无有效性检查
- 第655-659行: `ds.realtime_cache.update_tick(price=last_price)` — NaN写入缓存
- 第669-679行: tick_data['last_price'] = last_price 直接存入DuckDB — NaN持久化
- 第740-744行: `pos_svc.on_tick(tick)` — NaN传播到PositionService

**传播路径** (节点数: **8**):

```
tick.last_price (NaN)
  → ① _process_tick (strategy_tick_handler.py:286)  [无校验]
  → ② OrderFlowBridge.on_tick_feed (strategy_tick_handler.py:307)  [无校验]
  → ③ RealTimeCache.update_tick (strategy_tick_handler.py:655)  [NaN写入缓存]
  → ④ tick_data['last_price'] (strategy_tick_handler.py:674)  [NaN准备持久化]
  → ⑤ storage.process_tick (strategy_tick_handler.py:719)  [NaN持久化到DuckDB]
  → ⑥ PositionService.on_tick (strategy_tick_handler.py:742)  [止盈止损使用NaN]
  → ⑦ HFT引擎._dispatch_tick (strategy_tick_handler.py:837-881)  [多个信号生成使用NaN]
  → ⑧ SignalService.generate_signal (strategy_tick_handler.py:849-881)  [信号价格=NaN]
```

**受影响节点数**: 8个连续节点（从tick入口到信号生成、订单生成、持仓更新、风控决策）

**P0评级理由**: 一个NaN价格可以在单个tick周期内传播到**所有**下游子系统（缓存层、持久化层、信号层、订单层、持仓层、风控层），且**无任何中间节点进行NaN/Inf检查**。

### 2.2 P0-C002: 错误风控决策阻断所有后续交易

**污染源头**: `risk_service.py` — 风控阈值配置错误或计算错误

**传播路径**:

```
RiskService.check_before_trade() (错误阻断)
  → ① OrderService.send_order() 被拒绝 (risk_service.py:269)
  → ② SignalService.generate_signal() 生成了但无法执行
  → ③ 错失交易机会 → 盈亏计算偏差
  → ④ PositionService 持仓不变 → 但预期持仓与实际持仓不一致
  → ⑤ 后续风控计算基于错误持仓 → 偏差放大
```

**影响范围**: 一个错误的风控决策（假阳性阻断）不仅阻止当前交易，还：
- 影响后续所有风控检查（因为`_signal_times`和`_consecutive_loss_counts`状态会因缺失交易而偏移）
- **跨策略影响**: OrderService中的`_check_cross_strategy_risk()` (order_service.py:337) 基于全局持仓检查，错误阻断一个策略会使所有策略的净敞口计算偏差

**P0评级理由**: 风控阻断影响所有策略的所有后续交易，且无恢复机制区分"真实风险"与"错误风控决策"。

### 2.3 P1-C003: 一个错误信号触发多个下游操作

**污染源头**: `signal_service.py:185` — `generate_signal()` 中信号类型未受白名单校验

**传播路径**:

```
错误signal_type（如"OPEN_LONG"拼写为"OPEN_LONGG"）
  → ① SignalService内部去重失败（信号类型不匹配）
  → ② OrderService.send_order() 接收错误指令
  → ③ _execute_send_order() 平台API调用失败
  → ④ _retry_send_order() 重试3次仍然失败
  → ⑤ 错误反馈到_stats计数器（failed_orders递增但不精准）
```

**影响范围**: 这个链路的后果相对局限于当前订单，但**重试机制**（指数退避3次）会延迟其他正常订单的下单（因为OrderService使用同一把`_lock`）。

**P1评级理由**: 错误信号的影响局限在订单执行层，但通过共享锁可扩散到其他订单。

## 3. 维度2: 信号链的因果依赖

### 3.1 P0-C004: 信号A→下单→持仓变化→信号B（循环因果链）

**完整因果链**:

```
信号A (generate_signal)
  → OrderService.send_order (order_service.py:185)
    → PositionService.update_position (strategy_core_service.py:254)
      → 持仓变化 (position_service.py)
        → RiskService.check_before_trade (risk_service.py)  [基于新持仓检查]
          → 信号B的通过/阻断决策
            → 信号B如果通过 → OrderService.send_order → 持仓再次变化 → 信号C...
```

**循环依赖关键节点**:

| 步骤 | 文件 | 行号 | 依赖关系 |
|------|------|------|---------|
| S1 | signal_service.py | 185 | 生成信号A |
| S2 | order_service.py | 185-261 | send_order → _check_cross_strategy_risk |
| S3 | order_service.py | 337 | _check_cross_strategy_risk → get_cross_strategy_risk_guard |
| S4 | position_service.py | get_cross_strategy_risk_guard | 读取全局持仓 |
| S5 | risk_service.py | 523-545 | _check_risk_ratio → _check_risk_consistency |
| S6 | strategy_core_service.py | 254 | _update_position_from_trade → 持仓更新 |

**循环依赖问题**:
- `OrderService._check_cross_strategy_risk()` (order_service.py:337) 调用 `get_cross_strategy_risk_guard()` (来自position_service)，而position_service又依赖order_service下单（`_trigger_close_position` → `send_order`），形成**潜在的死锁循环**
- 具体路径: `order_service.send_order` → `_check_cross_strategy_risk` → `position_service.get_cross_strategy_risk_guard` → 如果触发了`trigger_close_position` → 又调用`order_service.send_order` → **死锁风险**

**P0评级理由**: 这是一个**有向循环图（DCG）而非有向无环图（DAG）**，信号→持仓→风控→订单形成了闭环，在特定条件下（锁顺序不一致）可导致**完整死锁**。

### 3.2 P1-C005: 信号过期后，下游基于过期信号假设的决策

**代码证据**:
- `signal_service.py`: `cooldown_seconds` 参数控制信号冷却时间（默认60秒）
- `strategy_tick_handler.py:1039`: HFT套利信号使用`cooldown_seconds=5`
- `strategy_tick_handler.py:1077`: HFT状态转换信号使用`cooldown_seconds=3`

**问题描述**: 信号过期（超过冷却时间）后被清理（`_cleanup_expired_cooldowns`），但以下情况可能发生：

```
T=0: 信号A生成 (signal_type=OPEN_LONG, price=3800)
T=5: 信号A冷却过期
T=6: 另一个模块调用generate_signal，依赖"最近的市场方向"信息
      此时信号A已被删除，无法判断"刚刚有开多信号"
T=7: 信号B (OPEN_SHORT) 生成，与信号A方向相反
      系统无法知道"5秒前刚有过开多信号"
```

**问题**: 信号历史仅保留冷却时间信息，**不保留信号方向/强度的时序记忆**。依赖"最近信号方向"的决策（如HFT增强引擎的`resonance_strength`计算）在信号过期后失去上下文。

**P1评级理由**: 影响HFT增强引擎的部分决策质量，但不影响核心交易逻辑。

### 3.3 P1-C006: 信号评分依赖其他信号的评分结果

**代码证据**:
- `RiskService`中11维度综合评分 `composite_risk_score`
- `risk_service.py:876`: `ranges = (recent['high'] - recent['low']) / recent['close'].replace(0, np.nan)` — 依赖bar数据计算波动率
- 波动率计算结果影响`composite_risk_score`

**因果链**:

```
bar1.close → 波动率计算 → composite_risk_score (11维综合)
  → 风控决策（阻断/放行）
    → 信号A是否执行
      → 持仓变化
        → Greeks变化（Delta/Gamma/Vega）
          → 风险评分变化（Greeks维度的子评分）
            → 信号B的风控决策
```

**问题**: 不同的信号执行顺序会导致不同的`composite_risk_score`，从而产生**状态路径依赖性**——先执行信号A再执行信号B的结果，与先B再A的结果不同。

**P1评级理由**: 这是所有交易系统的固有特性，但系统未对这种路径依赖性做量化分析或对冲。

## 4. 维度3: 错误数据的"通货膨胀"

### 4.1 P0-C007: 一次价格错误导致N次错误tick聚合

**传染路径**:

```
单个错误last_price (NaN/Inf/0)
  → ① RealTimeCache.update_tick → 缓存覆盖 (strategy_tick_handler.py:655-659)
  → ② DuckDB持久化 (storage.process_tick)
  → ③ K线合成（tick聚合为1分钟/5分钟K线）— 感染聚合后的OHLC
  → ④ 指标计算（基于K线的MA/ATR/IV/Bollinger等）— 感染所有指标
  → ⑤ 信号触发（基于指标的条件判断）— 感染信号
  → ⑥ 订单生成（基于信号的价格/方向）— 感染交易
  → ⑦ 持仓更新（基于成交价格）— 感染盈亏
  → ⑧ 风控检查（基于持仓的盈亏/敞口）— 感染风控
```

**指数级放大**:

| 阶段 | 受污染数据 | 放大倍数 |
|------|-----------|---------|
| 1个错误tick | 1个tick记录 | 1× |
| K线聚合（60个tick/分钟K） | 1根分钟K线 | 60× |
| 指标计算（20期MA） | 20根K线的MA值 | 1200× |
| 信号触发（基于MA交叉） | 所有后续信号的MA条件 | 持续放大 |
| 所有持仓的浮动盈亏 | 每个持仓的盈亏计算 | 持仓数×（持续） |

**具体代码证据**:
- `strategy_tick_handler.py:286`: `last_price = self._get_tick_field(tick, 'last_price', 0.0)` — **零价格也是错误**（交易价格为0在真实市场中不可能）
- `strategy_tick_handler.py:655`: `ds.realtime_cache.update_tick(price=last_price)` — 零价格覆盖缓存
- `strategy_tick_handler.py:719`: `self.storage.process_tick(tick_item)` — 零价格/NaN持久化

**对比R24审计发现**: R24报告已确认`last_price`可能为`float('nan')`、`float('inf')`、负值、0，`float('nan') <= 0` 检查会返回False（NaN比较总是False），所以简单的`if last_price <= 0: return`无法捕获NaN。

**P0评级理由**: 单个错误tick可通过K线聚合放大为影响所有技术指标的系统性污染，且无任何中间层做价格有效性验证。

## 5. 维度4: 计算结果的时序依赖性

### 5.1 P0-C010: `a = f(x); b = g(a); c = h(b)` 链式依赖

**代码证据**: `strategy_tick_handler.py:_dispatch_tick` 中的时序调用链

```python
# strategy_tick_handler.py:646-881
# 顺序A: RealTimeCache更新
ds.realtime_cache.update_tick(price=last_price)          # 步骤1: 缓存更新

# 顺序B: 持久化
self.storage.process_tick(tick_data)                     # 步骤2: DuckDB写入

# 顺序C: PositionService止盈止损
pos_svc.on_tick(tick)                                    # 步骤3: 基于步骤1的缓存

# 顺序D: HFT引擎
hft.on_tick_enhanced(price=last_price)                  # 步骤4: 基于步骤1/3
  → pursuit_entry → send_order → update_position        # 步骤5: 基于步骤4
```

**严格时序依赖**:
- 步骤3依赖步骤1完成（PositionService读取RealTimeCache中的最新价格做止盈止损判断）
- 步骤4依赖步骤1和步骤3完成（HFT引擎需要RealTimeCache中的tick数据和持仓状态）
- 步骤5依赖步骤4完成（HFT的pursuit_entry需要先判断再下单）

**违反后果**: 如果并发执行顺序被违反（如多线程下步骤3在步骤1之前执行），PositionService将使用**过时的缓存价格**做止盈止损判断，导致：
- 止盈未触发（价格已达到止盈位但读取的是旧价格）
- 止损未触发（类似）
- 或虚假触发（旧价格触发不应触发的条件）

**P0评级理由**: 步骤1-5之间的严格时序依赖在并发环境下没有强制执行保障，虽依赖GIL但GIL不保证Python级别的执行顺序。

### 5.2 P0-C011: 先更新持仓再计算风控 vs 先计算风控再更新持仓

**代码证据**:

```python
# strategy_core_service.py:254 (on_trade回调中)
self._update_position_from_trade(trade_data)    # 步骤A: 先更新持仓
  → PositionService.update_position()           # 持仓变化

# 而RiskService.check_before_trade() (risk_service.py) 在send_order内部调用
# order_service.py:213:
risk_level = self._check_cross_strategy_risk()  # 基于当前持仓检查风控
```

**问题**: 交易执行的回调中先更新持仓（步骤A），但下次交易前的风控检查基于**已更新的持仓**。这本身是合理的。但问题是：

1. **时序窗口**: 从持仓更新到风控检查之间的时间窗口内，如果另一个tick触发信号，可能使用**刚刚更新的持仓**做风控检查，导致连锁反应
2. **中间状态暴露**: `PositionService`的`_add_position`和`_reduce_position`之间（如果有中间状态），可能被另一个`on_tick`读到部分更新的持仓

**P0评级理由**: 时序窗口虽小但存在，多个`on_tick`并发时（通过不同分片线程），持仓中间状态可能被读取。

---

### 5.3 P1-C012: cache/flush时序不当导致数据丢失

**代码证据**:
```python
# strategy_tick_handler.py:696-732
with shard_lock:
    self._shard_buffers[shard_idx].clear()           # 清空buffer
    self._last_buffer_flush_time = time.time()
    ticks_to_flush = None                            # 竞态条件: buffer已清空但未flush

# 如果在clear()之后、flush之前发生异常，ticks永久丢失
for tick_item in ticks_to_flush:
    self.storage.process_tick(tick_item)              # 逐条flush
    if 异常:                                          # 某条失败
        failed_ticks.append(tick_item)
```

**问题**: buffer先clear再逐条flush，如果第N条失败，前N-1条已丢失（buffer已清空），第N+1条及后续被回写buffer。**已成功flush的数据不可回滚**。

**P1评级理由**: 数据丢失范围限制在单个flush批次内（最多50条tick），且异常tick会被回写到buffer重试。

## 6. 维度5: 缓存不一致的级联效应

### 6.1 P0-C013: RealTimeCache过时价格级联效应

**代码证据**:

**P0评级理由**: 缓存不一致通过状态路由机制放大为影响**所有交易参数**的系统性偏差。

## 7. 维度6: 跨系统因果断裂

### 7.1 P0-C016: 回测参数→生产参数因果链

**完整路径**:

```
task_scheduler.py 参数网格扫描 (972组合×16参数 = 15552次回测)
  → CascadeJudge.judge() 三层瀑布式评判
    → 参数池选择（Top-K）
      → 生产系统load_default_params_from_json()
        → get_cached_params() → resolve_config_with_priority()
          → 所有下游模块使用选中的参数
```

**因果断裂风险**:

| 环节 | 风险 | 证据 |
|------|------|------|
| 回测→评判 | 回测数据质量影响评判结果 | task_scheduler.py:1343 `valid_mask = ~(np.isnan(vals) | np.isnan(future_returns))` — 有NaN过滤 |
| 评判→参数选择 | 评判打分错误影响参数排名 | cascade_judge.py:496 `v is None or (math.isnan(v) or math.isinf(v))` — 有无效值检查 |
| 参数选择→生产 | 参数兼容性问题 | config_params.py:401-410 `validate_params()` — 有类型/范围校验 |
| 生产→交易 | 参数错误影响所有交易 | 所有模块使用get_cached_params()读取参数 |

**关键风险点**: `cascade_judge.py:589`:
```python
if math.isnan(final) or math.isinf(final):
    return 0.5  # 默认评分0.5
```
当最终评分为NaN/Inf时返回0.5（中等评分），但**0.5可能使参数通过评判进入选择**，但实际上该系统不应该使用"未知"评分的参数。

**P0评级理由**: 回测→参数选择→生产使用是一个完整的因果链，任一环节的误差都会放大到生产系统的所有交易。特别是NaN回退到0.5可能导致**不应该被选中的参数被选中**。

### 7.2 P1-C017: CascadeJudge评判维度缺失/错误

**代码证据**:
```python
# cascade_judge.py:496
_missing_dims = [k for k, v in _dim_values.items() 
                 if v is None or (isinstance(v, float) and (math.isnan(v) or math.isinf(v)))]
```

**问题**: 当评判维度值缺失时，系统检查哪些维度缺失。但有**缺失维度的容错策略**：缺失的维度是否用默认值替代？还是整个评判失败？

**因果断裂**: 如果某个关键维度（如`sortino_ratio`）缺失但仍有默认值通过评判，则回测结果无法真正反映策略质量，导致**劣质参数被选中进入生产**。

**P1评级理由**: 缺失维度的容错策略不透明，可能导致"虚假通过"的评判结果。

### 8.2 P0-C019: 给定错误交易无法追溯到源头

**问题**: 如果发现一笔错误交易（如价格异常、方向错误），**无法确定**是哪个tick触发的、哪个信号导致的、通过哪个路径执行的。

**缺失的追踪链路**:

```
错误交易
  ← OrderService.send_order()  — 有order_id但无signal_id映射
  ← SignalService.generate_signal() — 有信号记录但无tick_id映射
  ← TickHandlerMixin._dispatch_tick() — 有tick但不唯一标识
  ← 原始tick数据 — 在DuckDB中但无关联到交易
```

**具体问题**:
1. `send_order`（order_service.py:185）生成的订单ID与`generate_signal`（signal_service.py:185）生成的信号ID之间**无关联字段**
2. 每个tick没有**全局唯一ID**，无法将交易映射到触发它的tick
3. `_signal_history`（deque，最大1000条）不持久化，服务重启后信号历史丢失
4. `_stats`计数器不区分"被哪个信号/合约触发"，仅统计总数

**P0评级理由**: 完全无法从错误输出追溯到原始输入，这是系统韧性的致命缺陷。

### 8.3 P1-C020: 无法区分"原因"和"症状"

**现有机制**:
- `_e2e_counters`（strategy_core_service.py:126-134）记录了6个阶段的计数，可以区分：
  - `dropped_not_running` = 系统未运行 → 原因
  - `dropped_paused` = 系统暂停 → 原因
  - `persisted_count` > `enqueued_count` = 持久化滞后 → 症状
- `_tick_drop_count`（strategy_tick_handler.py:389）记录因异常被丢弃的tick数 → 症状

**不足**: 当`errors_count`增加时，无法区分是：
- **原因**: 平台API返回错误（如订阅失败）
- **症状**: 因为之前的数据错误导致后续计算异常

例如：某模块抛出`ZeroDivisionError`，可能是由于之前的价格为0（原因），但系统只记录"当前模块异常"（症状）。

**P1评级理由**: 错误计数聚合在一起，缺乏错误分类体系（I/O错误 vs 数据错误 vs 逻辑错误）。

## 9. 维度8: 污染清理机制

### 9.1 P0-C022: 检测到数据污染后无级联清理机制

**代码搜索**: 全仓库搜索 `rollback|cleanup|clean_up|purge|reset.*state|revert`

**发现的清理机制**:

| 清理机制 | 位置 | 作用域 | 是否级联 |
|---------|------|--------|---------|
| `reset_param_cache()` | config_params.py:262 | 仅参数缓存 | 否（仅清理参数缓存本身） |
| `_emergency_cleanup()` | storage_core.py:215 | 仅存储层DB连接 | 否 |
| `_cleanup_legacy_tables()` | ds_schema_manager.py:305 | 仅历史表 | 否 |
| `_cleanup_dead_threads()` | scheduler_service.py:463 | 仅线程 | 否 |
| `_cleanup_closed_positions()` | strategy_tick_handler.py:1330 | 仅已关闭持仓 | 否（仅内存清理） |
| `_rollback()` | mode_engine.py:1054 | 模式切换回滚 | **局部是**（配置+scale） |
| `_reset_daily()` | task_scheduler.py:1937 | 回测日重置 | **局部是**（回测状态） |

**关键缺失**: **没有任何机制可以级联清理一个受污染的数据链条**。

**问题场景**:
```
检测到tick.last_price = NaN
  → 需要清理：
    ① RealTimeCache中的对应条目
    ② DuckDB中已写入的tick记录
    ③ 基于该tick合成的K线
    ④ 基于该K线计算的指标
    ⑤ 基于该指标触发的信号
    ⑥ 基于该信号执行的交易（回滚）
    ⑦ 基于该交易更新的持仓
    ⑧ 基于该持仓调整的风控状态
  → 现实：没有任何清理机制，污染数据永久存在于系统中
```

**P0评级理由**: 系统完全没有级联清理的概念。检测到的数据污染仅停留在日志层面，没有任何执行层面的清理动作。

---

### 9.2 P0-C023: 错误交易后无法回滚所有后续影响

**设计缺陷**: 交易系统设计的单向性（先开仓→再平仓），使得错误开仓后无法"撤销"，只能通过反向交易"对冲"。

**但以下影响无法通过对冲消除**:
1. 开仓时的滑点成本（真实损失）
2. 开仓期间的浮动盈亏波动（影响风控状态）
3. Greeks变化（影响组合风控）
4. 连亏计数器增加（影响策略暂停/恢复）
5. 日间回撤计算变化（可能触发日回撤止损）
6. 仓位限额占用（影响其他策略的开仓能力）
7. 信号去重集合中的记录（影响后续信号生成）

**P0评级理由**: 错误交易的后续影响不可逆，且影响范围远超交易本身。

## 12. 关键结论与建议

### 12.1 最严重的因果断裂（Top 5）

1. **[P0-C001]** Tick价格无任何有效性检查 → 全链路污染 → **建议**: 在`_get_tick_field`后立即添加`math.isfinite(last_price)`检查，拒绝NaN/Inf/负值/零值
2. **[P0-C019]** 错误交易完全无法追溯 → **建议**: 添加`tick_id → signal_id → order_id`的全链路追踪ID
3. **[P0-C022/C023]** 检测到污染后无级联清理/回滚机制 → **建议**: 设计并实现`ContaminationManager`级联清理框架
4. **[P0-C004]** 信号→持仓→风控→订单形成有向循环图 → **建议**: 重构为严格的DAG，或统一锁顺序
5. **[P0-C007/C009]** 单点错误通过K线聚合/持仓存续期指数级放大 → **建议**: 添加价格有效性检查+持仓级联回滚
