## 一、环境差异化配置

### 1.1 缺乏环境级配置文件隔离机制

**问题编号: R14-CFG-01-001**

- **文件路径:** `config_params.py`，行 26-215
- **行号范围:** 26–215（`DEFAULT_PARAM_TABLE` 定义区域）
- **问题描述:** 三套环境（生产系统、测试系统、评判系统）共用同一个 `DEFAULT_PARAM_TABLE` 字典（180+ 参数）。虽然存在 `test_mode: False`（第 75 行）、`backtest_tick_mode: False`（第 39 行）、`debug_mode: False`（第 196 行）、`stress_test_mode: False`（第 197 行）等环境标记字段，但这些字段仅在同一个字典内以布尔值存在，**不存在环境级别的配置切换机制**——没有 `config/production.yaml`、`config/testing.yaml`、`config/judgment.yaml` 之类的独立环境配置文件。
- **严重等级:** **P0**
- **影响系统:** 生产系统、测试系统、评判系统（三系统均受影响）
- **风险分析:** 
  - 修改任何一个"环境无关"的配置项时，如果忘记手动将 `test_mode` 切回 `False`，测试模式的配置（如缩小资金规模、放松风控限制）会直接渗入生产系统
  - 手册中明确提到生产/测试/评判为三个独立系统（如第21.3节四策略参数网格），但没有规定如何通过环境隔离来防止配置泄露
  - `_build_config_from_args()`（`ProductionQuantSystem.py:361-395`）虽然接受 `--config` 参数加载外部JSON配置，但没有环境选择器（如 `--env production`），依赖人工传入正确文件

### 1.2 test_mode / backtest_tick_mode 仅作为布尔值硬编码

**问题编号: R14-CFG-01-002**

- **文件路径:** `config_params.py`
- **行号范围:** 第 39 行（`backtest_tick_mode: False`）、第 75 行（`test_mode: False`）、第 196 行（`debug_mode: False`）、第 197 行（`stress_test_mode: False`）
- **问题描述:** 环境标记字段是硬编码布尔值而不是从环境配置文件加载。`Strategy2026.__init__`（`strategy_core_service.py:646-708`）在初始化时也仅读取运行时 `kwargs` 字典中的配置，没有环境感知逻辑。如果在运行时重新加载参数表（TTL=300秒），不会根据当前运行环境自动应用正确的 `test_mode` 值。
- **严重等级:** **P1**
- **影响系统:** 生产系统、测试系统
- **风险分析:** 生产系统从 `params_default.json` 加载配置时，如果该文件中的 `test_mode` 被意外设置为 `True`，缓存生效期间（300秒）全部交易将在测试模式下运行。

### 1.3 环境切换缺乏安全防护

**问题编号: R14-CFG-01-003**

- **文件路径:** `ProductionQuantSystem.py`，行 354–395；`config_service.py`，行 300–344
- **行号范围:** `_build_config_from_args()` 函数 + `_apply_environment_overrides()` 方法
- **问题描述:** 虽然 `ConfigService._apply_environment_overrides()`（`config_service.py:325-344`）支持通过环境变量覆盖部分运行时配置（`TRADING_DB_PATH`、`LOG_LEVEL`、`OUTPUT_MODE`、`ENABLE_PERFORMANCE_MONITORING`、`DUCKDB_FILE`、`TICK_DATA_PATH`、`INTRADAY_FLUSH_WINDOWS`），但**不覆盖**关键交易参数——如 `close_take_profit_ratio`、`max_risk_ratio`、资金管理相关参数。环境变量仅覆盖基础设施级配置，不覆盖金融决策参数。
- **严重等级:** **P1**
- **影响系统:** 生产系统
- **风险分析:** 跨环境运行时无法通过环境变量区分交易策略参数（如止盈止损倍数），所有环境共用同一套策略参数。例如，测试环境可能使用更宽松的 `close_stop_loss_ratio` 来测试极端场景，但这一配置同样会被生产环境继承。

### 1.4 评判系统无独立配置入口

**问题编号: R14-CFG-01-004**

- **文件路径:** `策略评判/usage_example.py`、`参数池/task_scheduler.py`、`参数池/L1参数量化/v7_meta_audit_v2.py`
- **问题描述:** 评判系统的三个独立入口（回测任务调度器 `task_scheduler.py:6416`、审计器 `v7_meta_audit_v2.py:1700`、示例入口 `usage_example.py:367`）均依赖 `from ali2026v3_trading.config_params import DEFAULT_PARAM_TABLE` 模块，与生产系统共用同一个参数表。评判系统的回测参数与生产系统的实盘参数未隔离，一段在评判系统中修改的回测配置可能被生产系统缓存复用（因为 `get_cached_params()` 全局缓存 TTL=300秒）。
- **严重等级:** **P1**
- **影响系统:** 评判系统、生产系统
- **风险分析:** 评判系统回测使用的波动率、滑点等假设参数可能与生产系统参数在同一个缓存中被交叉污染。

## 二、部署脚本完整性

### 2.1 完全缺失部署脚本

**问题编号: R14-CFG-02-001**

- **文件路径:** 项目根目录 `ali2026v3_trading/`
- **问题描述:** 经过全量扫描（`*.sh`, `*.bat`, `*.ps1`, `*.cmd` 均搜索），**未发现任何部署脚本、启动脚本、停止脚本或进程管理配置**。系统唯一可运行入口是：
  - `ProductionQuantSystem.py` 的 `main()` 函数（行 398–497），通过 `python -m ali2026v3_trading.ProductionQuantSystem` 调用
  - `task_scheduler.py`（行 6416）、`预处理/Tick预处理` (`preprocess_ticks.py:930`) 等独立 `__main__` 入口
- **严重等级:** **P0**
- **影响系统:** 生产系统、测试系统、评判系统
- **风险分析:**
  - 生产部署完全依赖人工命令行调用，无标准化启动流程
  - 没有依赖检查（如 Python 版本、pip 包安装、DuckDB/SQLite 可用性）
  - 无法自动化重启或故障恢复

### 2.2 无进程守护（daemon/auto-restart）机制

**问题编号: R14-CFG-02-002**

- **文件路径:** 全量扫描无 `supervisor`/`systemd`/`pm2`/`nssm` 相关文件或配置
- **问题描述:** `ProductionQuantSystem.main()`（行 398–497）的主循环仅是 `while True: time.sleep(1.0)`，没有任何进程守护。如果进程因 Python 异常、OOM Killer、平台 API 调用异常而崩溃，没有自动重启机制。该函数捕获了 `KeyboardInterrupt`（行 491）和通用 `Exception`（行 493），但通用异常捕获后仅记录日志并 `return 1`，不会重新拉起。
- **严重等级:** **P0**
- **影响系统:** 生产系统
- **风险分析:** 生产系统在交易时段（日盘/夜盘）进程崩溃后，如果没有人工立即干预，将导致挂单无法平仓、持仓过夜等资金损失风险。手册中未提及进程管理要求，这是工程落地的严重缺失。

### 2.3 有基本优雅关机但缺乏超时控制

**问题编号: R14-CFG-02-003**

- **文件路径:** 
  - `ProductionQuantSystem.py`，行 340–347（`shutdown()` 全局函数）、行 494–497（`finally` 块）
  - `strategy_tick_handler.py`，行 87–92（`atexit` 刷新注册）
- **问题描述:** `shutdown_production_quant_system()` 函数（行 340–347）调用 `pqs.shutdown()` + `SingletonManager.shutdown_all()` + `gc.collect()`，但没有超时控制——如果 `shutdown()` 内部某个服务长时间阻塞（如数据库写入卡住），程序将无法退出。`strategy_tick_handler.py:87-92` 使用 `atexit` + `weakref` 注册了 Tick 缓冲刷新回调，这是一个好的实践，但同样没有超时控制。
- **严重等级:** **P1**
- **影响系统:** 生产系统
- **风险分析:** 若数据库写入操作在网络分区或磁盘满时无限等待，优雅关机将退化为无限挂起。应添加 `shutdown_timeout` 参数（建议 30 秒）和强制 `os._exit()` 回退。

### 2.4 无依赖检查

**问题编号: R14-CFG-02-004**

- **文件路径:** `ProductionQuantSystem.py` 行 361–377
- **问题描述:** `_build_config_from_args()` 中尝试 `from ali2026v3_trading.config_service import ConfigService`（行 367）并外包裹 `try/except`——这是**唯一的隐式依赖检查**。但没有显式检查以下关键依赖的存在性：
  - `numpy`（`quant_services.py:23` 无条件 `import numpy as np`）
  - `yaml`（多个文件使用，如 `config_service.py`）
  - `duckdb`（`config_service.py:202` 使用环境变量）
  - `numba`（可选，`quant_services.py:30` 有 `HAS_NUMBA` 检测）
  - Python版本（要求 3.8+ 用于 `from __future__ import annotations`）
- **严重等级:** **P1**
- **影响系统:** 全部三系统
- **风险分析:** 缺少 numpy 或 yaml 等核心依赖时，`ImportError` 在深层代码抛出，错误信息不友好且难以排查。

## 三、配置漂移检测

### 3.1 同一参数多处定义且默认值不一致（close_take_profit_ratio）

**问题编号: R14-CFG-03-001**

- **文件路径与行号:**
  1. `config_params.py:129` — `"close_take_profit_ratio": 1.5`（全局兜底默认值，注释注明"被 get_default_state_param_sets() 中各状态特定值覆盖"）
  2. `config_params.py:541` — `get_default_state_param_sets()` 的 `correct_trending`: `'close_take_profit_ratio': 2.5`
  3. `config_params.py:550` — `get_default_state_param_sets()` 的 `incorrect_reversal`: `'close_take_profit_ratio': 1.3`
  4. `config_params.py:559` — `get_default_state_param_sets()` 的 `other`: `'close_take_profit_ratio': 1.1`
  5. `参数池/state_param_sets.yaml` — 从YAML加载的状态参数（若存在）
  6. `参数池/plr_thresholds.yaml:4` — `small.target_plr: 2.5`, `plr_thresholds.yaml:20 medium.target_plr: 2.0`, `plr_thresholds.yaml:36 large.target_plr: 1.5`（PLR维度阈值，语义不同但可能被误用）
  7. `参数池/cascade_config.yaml` — 三级联配置中的 `close_take_profit_ratio`（如存在）
- **问题描述:** `close_take_profit_ratio` 在 **至少 5 处不同位置**定义，且值范围从 1.1 到 2.5 不等。虽然这些值来自不同上下文（不同状态、不同资金规模），但如果存在代码路径错误使用全局默认值 1.5 而非状态特定值 2.5（correct_trending），将导致 `correct_trending` 状态下的止盈过早触发，盈利减少约 40%。
- **严重等级:** **P0**
- **影响系统:** 生产系统、评判系统
- **风险分析:** 代码依赖运行时查询状态参数管理器来选择正确的值，但如果某条代码路径绕过管理器直接访问 `DEFAULT_PARAM_TABLE['close_take_profit_ratio']`，就会错误使用全局值 1.5。这属于**配置意图扩散**（一个参数被赋予多个含义），且无编译时或运行时的跨源一致性校验。

### 3.2 同一参数多处定义且默认值不一致（close_stop_loss_ratio、signal_cooldown_sec）

**问题编号: R14-CFG-03-002**

- **文件路径:**
  1. `config_params.py:198` — `"close_stop_loss_ratio": 0.5`（全局兜底）
  2. `config_params.py:542` — `correct_trending.close_stop_loss_ratio: 0.4`
  3. `config_params.py:551` — `incorrect_reversal.close_stop_loss_ratio: 0.6`
  4. `config_params.py:560` — `other.close_stop_loss_ratio: 0.8`
  5. `config_params.py:110` — `"signal_cooldown_sec": 0.0`（全局）
  6. `config_params.py:540` — `correct_trending.signal_cooldown_sec: 15`
  7. `config_params.py:549` — `incorrect_reversal.signal_cooldown_sec: 120`
  8. `config_params.py:558` — `other.signal_cooldown_sec: 300`
- **问题描述:** 同 R14-CFG-03-001 模式——两个关键风控参数在全局表与状态参数集中存在不同默认值。特别是 `signal_cooldown_sec` 全局默认 `0.0`（无限冷却）与状态特定值（15秒到300秒）差异极大。若代码路径访问全局默认值 0.0 而非状态特定值，会导致冷却机制完全失效，可能在同一 tick 上重复下单。
- **严重等级:** **P0**
- **影响系统:** 生产系统
- **风险分析:** 冷却机制失效是高频重复下单的直接原因，可导致当日风控限额瞬间耗尽或交易所处罚。

### 3.3 参数池 YAML 与代码硬编码默认值关系不明确

**问题编号: R14-CFG-03-003**

- **文件路径:** 
  - `config_params.py:510-565` — `get_default_state_param_sets()`
  - `参数池/state_param_sets.yaml`
  - `参数池/plr_thresholds.yaml`
  - `参数池/tvf_params.yaml`
  - `参数池/parameter_attribute_matrix.yaml`
- **问题描述:** `get_default_state_param_sets()`（`config_params.py:510-565`）优先从 `state_param_sets.yaml` 加载，失败时回退到硬编码值。但是以下 YAML 文件没有找到对应的加载路径：
  - `plr_thresholds.yaml` — 搜索代码未发现直接的 `load_plr_thresholds()` 函数，可能仅被 `参数池/task_scheduler.py` 或内部模块加载
  - `parameter_attribute_matrix.yaml` — 未搜索到对应的加载代码
  - `tvf_params.yaml` — 被 `tvf_param_loader.py` 加载
- **严重等级:** **P2**
- **影响系统:** 全部三系统
- **风险分析:** 缺乏一个统一的"配置注册表"文档来说明每个 YAML 文件的加载路径、加载时机、与代码默认值的优先级关系。运维人员在 YAML 中修改参数后，可能不确定该参数是否实际生效（因为代码中可能还有硬编码回退）。

### 3.4 热重载后配置未持久化到主参数表

**问题编号: R14-CFG-03-004**

- **文件路径:** 
  - `quant_services.py:185-306` — `HotConfigManager`
  - `config_params.py:224-242` — `get_cached_params()`
- **问题描述:** `HotConfigManager.update_config()`（`quant_services.py:291-306`）确实将变更写入了 `hot_config.json` 磁盘文件。但是，`get_cached_params()`（`config_params.py:224-242`）仅从 `params_default.json`（通过 `load_default_params_from_json()`）加载，与 `hot_config.json` 无交集。这意味着：
  - 通过 `HotConfigManager` 修改的运行时参数在内存中生效
  - 但 `params_default.json` 不会被热重载更新
  - 系统重启后，`HotConfigManager` 的变更会丢失（因为它从 `hot_config.json` 加载，而 `get_cached_params()` 从 `params_default.json` 加载）
- **严重等级:** **P1**
- **影响系统:** 生产系统
- **风险分析:** 运维人员通过热配置调整风控参数后在当前交易时段内有效，但次日重启后风控参数回到旧值。如果运维人员没有同时手动更新 `params_default.json`，这构成**配置漂移**（运行时配置 vs 持久化配置不一致）。

## 四、启动依赖链

### 4.1 启动依赖顺序合理但缺乏显式健康检查门控

**问题编号: R14-CFG-04-001**

- **文件路径:** 
  - `strategy_core_service.py:646-708` — `Strategy2026.__init__()`
  - `ProductionQuantSystem.py:398-497` — `main()`
  - `width_cache.py` — 宽度缓存启动
- **问题描述:** `Strategy2026.__init__()` 初始化顺序为：
  1. `StrategyCoreService` 创建（行 688）
  2. `t_type_bootstrap` 注册（行 691–696）
  3. `market_time_service` 获取（行 703–704）
  4. `get_cached_params()` 延迟加载（行 726–732，通过 `_ensure_runtime_config_loaded()` 在首次使用时触发）
  
  这个顺序基本合理。但 `ProductionQuantSystem.main()`（行 438–449）在 `initialize()` 后直接进入主循环 `while True:`（行 473），没有调用健康检查接口确认所有子系统就绪。`health_check_api.py` 定义了完整的健康检查 API 但未被 `main()` 调用。
- **严重等级:** **P1**
- **影响系统:** 生产系统
- **风险分析:** 如果某个关键服务（如 DataService、MarketTimeService）在静默中初始化失败（不抛异常但返回空/无效对象），系统会在主循环中运行一段时间后才因数据缺失而报错，此时可能已错过交易信号或产生因数据缺失导致的错误交易决策。

### 4.2 初始化失败时降级策略不完整

**问题编号: R14-CFG-04-002**

- **文件路径:**
  - `strategy_core_service.py:691-696` — `t_type_bootstrap` 注册失败
  - `ProductionQuantSystem.py:375-376` — `ConfigService` 加载失败
  - `config_params.py:817-824` — 期权参数合并失败
  - `params_service.py:228-248` — 配置初始化失败
- **问题描述:** 多处代码使用 `try/except` + `logging.warning/logging.debug` 进行降级处理：
  - `t_type_bootstrap` 注册失败 → 仅 warning，不中断初始化（`strategy_core_service.py:696`）
  - `ConfigService` 加载失败 → 仅 debug（`ProductionQuantSystem.py:376`）
  - 期权参数合并失败 → warning（`config_params.py:817`）
  - `params_service.py:228-248` — 配置初始化失败时**不抛异常**，静默继续
  
  这些降级策略的问题是：**不存在统一的降级级别判定**。例如"缺少 ConfigService 导致无法获取交易所信息"可能是致命的（P0级），但在同一 `try/except` 块中只产生一条 DEBUG 日志。
- **严重等级:** **P1**
- **影响系统:** 生产系统

## 六、运行时配置变更

### 6.1 热重载体系结构分析

**问题编号: R14-CFG-06-001**

- **文件路径:**
  - `quant_services.py:185-306` — `HotConfigManager`（热配置管理器）
  - `config_params.py:224-242` — `get_cached_params()`（参数缓存 TTL=300秒）
  - `params_service.py:923-962` — `load_from_env()`（环境变量加载）
- **问题描述:** 系统存在**三套独立的配置加载/重载机制**，它们互不通信：
  1. **HotConfigManager**（`quant_services.py:185`）：基于文件修改时间（mtime）的轮询（`poll_interval=5.0秒`），管理 `hot_config.json`，支持 Schema 验证（`_CONFIG_SCHEMA`）和回调通知（`_notify_changes`）。**仅管理 8 个技术参数**（adx_period, pca_window, hmm_n_states, vol_lookback, coint_window, bar_interval_sec, bar_vol_threshold, snapshot_interval_ms）。
  2. **get_cached_params() TTL缓存**（`config_params.py:224`）：300 秒（5分钟）TTL 缓存，从 `params_default.json` 重新加载。覆盖全套 180+ 参数。
  3. **ParamsService.load_from_env()**（`params_service.py:923`）：一次性从环境变量加载，无热重载机制。
  
  这三套机制独立运行，没有统一的重载协调器。修改 `hot_config.json` 中的 `bar_interval_sec` 会通过 `HotConfigManager` 回调立即生效，但 `get_cached_params()` 缓存中的 `bar_interval_sec` 要等最多 300 秒才会被 TTL 过期重新加载覆盖。
- **严重等级:** **P1**
- **影响系统:** 生产系统
- **风险分析:** 同一参数同时存在于 `hot_config.json` 和 `params_default.json` 时，两套重载机制会产生不一致的时间窗口。HotConfig 回调修改了内存中的 `ProductionQuantSystem` 内部状态，但缓存参数表中的值仍是旧的，可能导致不同子系统（QuantSystem vs StrategyEcosystem）使用不同的 `bar_interval_sec` 值。

### 6.2 参数热重载缺乏结构化审计日志

**问题编号: R14-CFG-06-002**

- **文件路径:** `quant_services.py:238-249` — `_notify_changes()`
- **问题描述:** `HotConfigManager._notify_changes()` 在检测到配置变更时仅调用回调函数并记录回调错误，**没有结构化的审计日志**来记录：
  - 谁（哪个用户/进程）修改了哪个参数
  - 从什么旧值修改为什么新值
  - 修改的时间戳
  - 修改的原因
  
  当前变更通知中使用的日志是 `rate_limit_log()`（去重限流日志），这意味着如果同一个参数在同一分钟内被修改多次，只有第一次变更会被记录。
- **严重等级:** **P1**
- **影响系统:** 生产系统
- **风险分析:**
  - 金融合规审计要求所有参数变更可追溯，缺失审计日志违反了最低合规要求
  - 出现交易异常时无法回溯是谁何时修改了关键参数
  - `rate_limit_log` 可能吞掉高频变更记录，导致审计链断裂

### 6.3 参数热生效 vs 冷启动分类无文档

**问题编号: R14-CFG-06-003**

- **文件路径:** 全量代码无 `HOT_RELOADABLE_PARAMS` / `COLD_START_PARAMS` 常量或文档
- **问题描述:** 手册中未明确说明每个参数的重载类型（热生效 vs 需重启）。代码层面：
  - `HotConfigManager` 的 `_CONFIG_SCHEMA` 定义了 8 个热重载参数，但这 8 个参数是否真的支持热重载取决于回调函数的实现
  - 缓存参数表（`get_cached_params()`）TTL 过期后重新加载，但哪些参数在重载后需要触发级联更新（如交易所参数变更→重建合约缓存）没有明确约定
  - `_rebuild_instrument_cache()`（`params_service.py:416-430`）可以在合约缓存变化时重建，但没有被热重载回调触发
- **严重等级:** **P2**
- **影响系统:** 全部三系统
- **风险分析:** 运维人员修改一个"直觉上应为热生效"的参数（如 `option_width_min_threshold`），但该参数需要 `_rebuild_instrument_cache()` 重建缓存才能生效——这在热重载回调中没有触发逻辑。

### 6.4 配置一致性校验缺失

**问题编号: R14-CFG-06-004**

- **文件路径:** 全量搜索无 `config_constraint` / `config_validate` 语义结果（除 `_CONFIG_SCHEMA` 的类型校验外）
- **问题描述:** `HotConfigManager._validate_config()`（`quant_services.py:205-218`）仅校验类型（`int`/`str`），不校验参数间的合理性约束。例如：
  - `close_take_profit_ratio` 必须大于 `close_stop_loss_ratio`（否则止盈在止损之内）
  - `max_signals_per_window` 必须大于 `lots_min`（否则信号窗口容纳不下最小手数）
  - `hmm_n_states` 必须大于 2（否则 HMM 无法工作）
  - `option_width_min_threshold` 不能为负数
- **严重等级:** **P1**
- **影响系统:** 生产系统
- **风险分析:** 运维人员可能无意中将 `close_stop_loss_ratio` 设置为大于 `close_take_profit_ratio` 的值（如止损 2.5，止盈 1.0），导致每笔交易都被立刻止损平仓——这在金融上是致命的配置错误，但当前系统不会检测。

## 七、灾难恢复配置

### 7.1 存在 Fallback 到硬编码默认值的机制

**问题编号: R14-CFG-07-001**

- **文件路径:**
  - `config_params.py:224-242` — `get_cached_params()` JSON 加载失败 → 回退到 `DEFAULT_PARAM_TABLE`
  - `config_params.py:510-565` — `get_default_state_param_sets()` YAML 加载失败 → 回退到硬编码值
  - `config_params.py:456-481` — `load_option_params_from_file()` 多个路径查找期权参数 → 最终回退到空字典
- **问题描述:** 三层 Fallback 设计是好的防御性编程实践：

  2. YAML 文件 → 代码硬编码
  3. 生成文件多路径 → 空字典
  但硬编码默认值的安全性是关键问题——见下一项。
- **严重等级:** P2（正面评价）
- **影响系统:** 全部三系统

### 7.2 Fail-Safe 默认值安全性评估

**问题编号: R14-CFG-07-002**

- **文件路径:** `config_params.py:26-215` — `DEFAULT_PARAM_TABLE` 完整表
- **问题描述:** 对核心风控参数的 fail-safe 默认值逐一评估：
  | 参数 | 默认值 | Fail-Safe? | 说明 |
  |:---|:---|:---|:---|
  | `close_take_profit_ratio` | 1.5 | ✅ 偏保守 | 止盈1.5倍—偏低但不会导致亏损 |
  | `close_stop_loss_ratio` | 0.5 | ⚠️ 偏激进 | 50%止损在方向错误时可造成大损失 |
  | `max_risk_ratio` | 0.3 | ✅ 合理 | 单笔30%风险—偏大但非致命 |
  | `max_signals_per_window` | 10 | ⚠️ | 窗口内10个信号—正常但无上限 |
  | `lots_min` | 5 | ⚠️ | 最小5手—偏大，资金不足可能跳过信号 |
  | `max_hold_minutes` | 120 | ✅ 合理 | 2小时最大持仓 |
  | `signal_cooldown_sec` | 0.0 | ❌ 危险 | 0秒冷却→无冷却→重复下单风险 |
  | `option_width_min_threshold` | 4.0 | ✅ 保守 | 宽度阈值偏高，过滤多于入场 |
  | `circuit_breaker_pause_sec` | 180.0 | ✅ 保守 | 断路器暂停3分钟 |
  | `daily_drawdown_multiplier` | 2.0 | ⚠️ | 日回撤2倍—可能导致过早熔断 |
  
  综合评估：大部分默认值偏向保守（有利于防止激进交易），但 `signal_cooldown_sec: 0.0` 是一个**危险的 Fail-Safe 默认值**——如果状态参数管理器初始化失败，冷却机制完全失效。
- **严重等级:** **P1**
- **影响系统:** 生产系统
- **风险分析:** 在灾难恢复（JSON 文件损坏）场景下，`signal_cooldown_sec: 0.0` 作为默认值将导致无冷却限制，可能在单一 tick 事件中重复触发多个信号。

### 7.3 无显式备份配置或恢复流程

**问题编号: R14-CFG-07-003**

- **文件路径:** 全量搜索 `backup`/`restore`/`snapshot` 关键词
- **问题描述:** 
  - `LightweightPersistence`（`quant_services.py:44`）有 JSON 快照功能（`snapshot_interval_ms`），但这是**量化状态/运行状态**的快照，不是配置文件的备份
  - `HotConfigManager.update_config()` 直接覆盖 `hot_config.json`，不保留旧版本
  - 没有任何代码或脚本实现配置文件备份（如 `params_default.json.bak`、`cascade_config.yaml.bak`）
  - 手册中无"配置备份"、"灾难恢复"等章节
- **严重等级:** **P1**
- **影响系统:** 生产系统
- **风险分析:** 配置文件被误修改或文件系统损坏后，无法从备份恢复。只能回退到代码中的硬编码默认值——这已经在 7.2 节中评估为部分不安全。

### 7.4 热配置管理器有回调失败记录但无恢复机制

**问题编号: R14-CFG-07-004**

- **文件路径:** `quant_services.py:238-263` — `_notify_changes()` + `get_all()`
- **问题描述:** `HotConfigManager` 通过 `_callback_failures` 字典（`quant_services.py:201`）记录了回调失败的原因（行 246：`self._callback_failures[key] = str(e)`），且在 `get_all()` 输出中暴露（行 261-263）。这是一个好的可观测性设计。但是，**没有重试机制**——回调失败后不会自动重新调用，也没有告警升级（如连续3次失败后发出P0级告警）。
- **严重等级:** **P2**
- **影响系统:** 生产系统
- **风险分析:** 如果 `bar_interval_sec` 的热重载回调失败（例如因为 TickAggregator 未初始化），新配置会写入 `hot_config.json` 磁盘，但永不生效——形成"静默配置黑洞"。
