## 一、资金计算逻辑漏洞

### V1-001 [P0] 保证金释放未纳入可用资金计算
- **文件**: `risk_service.py` 行1934-1961
- **描述**: `check_capital_sufficiency()` 计算可用保证金时使用 `equity - required_margin`（行1956），仅扣除新开仓所需保证金，未扣除已有持仓的占用保证金。实际可用资金 = equity - 已占用保证金 - 新增保证金，当前计算会高估可用资金，导致超额开仓。
- **手册对照**: 手册5.1节要求"生产值取安全通过值×0.7~0.8"，但代码未实现已有持仓保证金扣除，安全系数形同虚设。
- **全链路**: `check_capital_sufficiency()` → `equity - required_margin` → 缺少 `existing_margin_used` 变量 → 可用资金虚高
- **影响系统**: 生产系统

### V1-002 [P0] 权益计算使用开仓价而非市价
- **文件**: `position_service.py` 行1083-1084
- **描述**: `check_all_positions()` 中权益计算使用 `rec.open_price` 而非当前市场价格，导致：1）浮盈持仓权益被低估；2）浮亏持仓权益被高估；3）熔断器/日回撤硬停止的触发阈值基于错误权益值，可能延迟触发或误触发。
- **手册对照**: 手册5.2节要求"每次check_before_trade()时更新仪表盘缓存"，但权益更新源头数据错误。
- **全链路**: `check_all_positions()` → `total_equity += abs(rec.volume) * rec.open_price` → `safety.on_equity_update(total_equity)` → 熔断器/日回撤判断基于错误值
- **影响系统**: 生产系统

### V1-003 [P1] SimplifiedSPAN最低保证金硬编码1000.0
- **文件**: `risk_service.py` 行2336
- **描述**: `SimplifiedSPAN.calc_margin()` 硬编码最低保证金为1000.0，无论组合规模大小。对于大资金账户（如100万以上），1000.0的最低保证金远低于实际风险敞口，形同虚设；对于小资金账户则可能过高。
- **手册对照**: 手册5.1节要求"情景压力库量化"确定保证金，但硬编码值未与账户规模挂钩。
- **影响系统**: 生产系统

### V1-004 [P1] 浮动盈亏未计入权益更新
- **文件**: `position_service.py` 行1079-1086
- **描述**: `check_all_positions()` 中权益计算仅使用 `open_price`，未获取当前市价计算浮动盈亏。在极端行情下，实际权益可能已大幅偏离开仓价计算的值，导致风控判断失效。
- **影响系统**: 生产系统

### V1-005 [P2] 弹簧策略手数计算未考虑已有持仓
- **文件**: `box_spring_strategy.py` 行689-709
- **描述**: `compute_equity_based_lots()` 使用 `account_equity * max_risk_ratio` 计算风险预算，但未扣除已有弹簧持仓占用的风险预算。当已有多个弹簧持仓时，新增持仓的风险预算应基于剩余可用风险额度，而非总额度。
- **手册对照**: 手册8.2节要求"半凯利×期权买方折扣系数"，但未考虑多持仓叠加风险。
- **影响系统**: 生产系统

## 二、风控可绕过路径

### V2-001 [P0] SafetyMetaLayer检查在锁外执行，存在TOCTOU窗口
- **文件**: `risk_service.py` 行217-267
- **描述**: `check_before_trade()` 中SafetyMetaLayer检查（行219-221）在 `with self._lock` 之前执行，后续的持仓限制检查等在锁内执行。在并发场景下，安全元层检查通过后、锁获取前，风控状态可能已变化（如另一线程触发了熔断），导致本应被阻断的交易通过。
- **全链路**: `check_before_trade()` → `_check_safety_meta_layer()` (锁外) → `with self._lock` → `_check_position_limit()` (锁内) → 窗口期风控状态可变
- **手册对照**: 手册5.2节safety_meta要求"每次check_before_trade()时更新"，但TOCTOU使更新不原子。
- **影响系统**: 生产系统

### V2-002 [P0] CLOSE信号绕过熔断器且无合法性验证
- **文件**: `risk_service.py` 行278-279
- **描述**: `_check_safety_meta_layer()` 中CLOSE动作直接跳过熔断器检查（`if action == 'CLOSE': return RiskCheckResult(...)`），但未验证该CLOSE信号是否合法。攻击者或异常代码可构造虚假CLOSE信号绕过熔断器，在熔断期间继续交易。
- **手册对照**: 手册9.2节补丁二b要求"硬终止效果：禁止所有交易（含平仓）"，但代码允许CLOSE绕过。
- **全链路**: `_check_safety_meta_layer()` → `if action == 'CLOSE': return` → 无CLOSE合法性校验 → 熔断期间可伪造CLOSE交易
- **影响系统**: 生产系统

### V2-003 [P0] confirm_daily_resume()无身份认证
- **文件**: `risk_service.py` 行1774-1820
- **描述**: `confirm_daily_resume()` 仅翻转 `_daily_hard_stop_triggered` 布尔值，无任何身份认证或授权检查。任何可访问该方法的代码路径（包括远程API调用）均可绕过日回撤硬停止。
- **手册对照**: 手册9.2节补丁二b要求"仅通过人工调用confirm_daily_resume()确认恢复"，但未实现"人工"约束。
- **全链路**: `_check_daily_drawdown()` → `_daily_hard_stop_triggered=True` → `confirm_daily_resume()` → `_daily_hard_stop_triggered=False`（无认证）
- **影响系统**: 生产系统

### V2-004 [P1] 运行时参数修改无审计追踪
- **文件**: `config_params.py` 行260-276
- **描述**: `update_cached_params()` 允许运行时修改参数缓存，但：1）无审计日志记录谁在何时修改了什么参数；2）`validate_params()` 的约束规则仅在 `rebuild_default_param_table()` 时检查，`update_cached_params()` 不触发验证；3）修改后无告警通知。
- **手册对照**: 手册十节要求"参数属性矩阵"含验证日期和重检条件，但运行时修改绕过了验证。
- **全链路**: `update_cached_params(key, value)` → `_param_cache[key]=value` → 无validate → 无audit → 无alert
- **影响系统**: 生产系统

### V2-005 [P1] TVF参数热重载无范围校验
- **文件**: `mode_engine.py` 行998-1037
- **描述**: `reload_tvf_params()` 从YAML热重载TVF参数到ModeConfig，但不对加载值做范围校验。恶意或错误的YAML配置可将TVF因子设为0（完全禁止交易）或极大值（放大仓位至危险水平）。
- **影响系统**: 生产系统

### V2-006 [P1] compute_mode_position_size()忽略TVF维度参数
- **文件**: `risk_service.py` 行1166-1184
- **描述**: `compute_mode_position_size()` 接受10个TVF维度参数（sortino, calmar, sharpe, ofi_score等），但调用 `ModeEngine.compute_position_size()` 时仅传递了equity/entry_price/stop_price/win_rate/win_loss_ratio，所有TVF参数被丢弃。手册5.3节定义的11维度风险评分体系在此路径完全失效。
- **全链路**: `compute_mode_position_size(sortino=..., calmar=..., ...)` → `ModeEngine.compute_position_size(equity, entry_price, stop_price, win_rate, win_loss_ratio)` → TVF参数丢失
- **手册对照**: 手册5.3节要求"综合评分驱动连续仓位缩放"，但TVF参数未传递。
- **影响系统**: 生产系统

### V2-007 [P2] 策略恢复resume_strategy()无冷却期
- **文件**: `strategy_ecosystem.py` 行1306-1314
- **描述**: `resume_strategy()` 直接将 `slot.paused=False`，无冷却期或渐进恢复机制。策略因EV<0被暂停后可立即恢复并全仓交易，无保护性观察期。
- **影响系统**: 生产系统

## 三、对冲逻辑缺陷

### V3-001 [P0] 无对冲比率计算机制
- **文件**: `box_spring_strategy.py` 全文
- **描述**: 弹簧策略的 `execute_spring_entry()` 仅检查跨策略风控和最大持仓数，无对冲比率计算。当弹簧策略与主策略/反向策略在同一标的上持有相反方向仓位时，无系统化的对冲比率管理，可能导致：1）对冲不足（风险敞口过大）；2）过度对冲（资金效率低下）。
- **手册对照**: 手册5.1节要求"希腊字母硬约束"管理净敞口，但弹簧策略独立于Greeks计算体系。
- **影响系统**: 生产系统

### V3-002 [P1] 弹簧策略对冲失败无回退机制
- **文件**: `box_spring_strategy.py` 行787-799
- **描述**: `execute_spring_entry()` 中跨策略风控检查失败时直接返回None，无回退机制（如降低手数重试、选择替代合约等）。在市场剧烈波动时，弹簧信号的有效期极短，直接放弃可能导致错过唯一的风险对冲窗口。
- **影响系统**: 生产系统

### V3-003 [P1] 跨策略Greeks聚合使用固定估算值
- **文件**: `position_service.py` 行1457-1504
- **描述**: `aggregate_greeks_exposure()` 使用固定每手delta/vega/gamma估算值（0.5/-0.5/0.15/0.05），而非从GreeksCalculator获取实际计算值。这些固定值：1）不反映实际行权价/到期日/波动率；2）深度虚值期权与平值期权的Greeks差异巨大；3）可能导致对冲比率严重失准。
- **手册对照**: 手册5.2节要求"实时希腊字母风险仪表盘"，但聚合层使用粗估值。
- **全链路**: `aggregate_greeks_exposure()` → 固定delta=0.5/vega=-0.5 → `CrossStrategyRiskGuard.check()` → 对冲判断基于错误值
- **影响系统**: 生产系统

### V3-004 [P2] 互斥规则仅检查方向，未检查Delta对冲
- **文件**: `strategy_ecosystem.py` 行734-779
- **描述**: `check_mutual_exclusion()` 仅检查策略间方向是否相同（`_is_same_delta_direction()`），未计算实际Delta对冲效果。例如：主策略持有Delta=0.8的CALL，反向策略持有Delta=-0.3的PUT，方向相反但净Delta=0.5仍有显著敞口，互斥规则不会阻断。
- **影响系统**: 生产系统

## 四、仓位计算漏洞

### V4-001 [P0] 开仓前无保证金充足性检查
- **文件**: `position_service.py` 行645-737
- **描述**: `_add_position()` 直接创建持仓记录，不检查保证金是否充足。虽然 `risk_service.check_before_trade()` 在上游执行了风控检查，但 `_add_position()` 本身无防御性检查，若上游检查被绕过（如V2-001/V2-002），将直接创建超限持仓。
- **全链路**: `signal → check_before_trade() → send_order() → _add_position()` → _add_position无保证金校验 → 超限持仓创建
- **手册对照**: 手册5.1节要求"希腊字母硬约束"，但仓位创建层无独立校验。
- **影响系统**: 生产系统

### V4-002 [P1] 部分平仓后止盈止损价未重算
- **文件**: `position_service.py` 行812-854
- **描述**: `_reduce_position()` 采用FIFO平仓，部分平仓后剩余持仓的 `stop_profit_price` 和 `stop_loss_price` 保持开仓时设定的值，不根据当前市场波动率或剩余仓位大小重新计算。在波动率显著变化后，原始止损价可能过于宽松或过于严格。
- **手册对照**: 手册6.2节定义了按开仓理由分化的TP/SL倍数，但未考虑部分平仓后的动态调整。
- **影响系统**: 生产系统

### V4-003 [P1] _calc_max_volume_from_capital()未考虑保证金
- **文件**: `position_service.py` 行1345-1354
- **描述**: `_calc_max_volume_from_capital()` 使用简单公式 `capital * leverage / price` 计算最大手数，未考虑保证金要求或已有持仓。实际最大可开仓手数应基于 `(可用资金 - 已占用保证金) / 单手保证金`，当前公式可能高估可开仓量。
- **影响系统**: 生产系统

### V4-004 [P1] CrossStrategyRiskGuard默认delta_limit=3.0过低
- **文件**: `position_service.py` 行1552-1597
- **描述**: `CrossStrategyRiskGuard` 默认 `delta_limit=3.0`，对于大资金账户（如100万以上），3手Delta限制过于保守，可能频繁阻断正常交易；且无动态调整机制根据账户规模或市场波动率调整限制。
- **手册对照**: 手册5.1节 `max_net_delta_pct=0.30` 对应的Delta限制应基于账户规模动态计算。
- **影响系统**: 生产系统

### V4-005 [P2] compute_position_size()无保证金检查
- **文件**: `mode_engine.py` 行817-867
- **描述**: `compute_position_size()` 计算仓位大小仅基于Kelly公式或固定风险比例，不检查计算出的手数是否在保证金约束内。返回的手数可能超出可用保证金支持的范围。
- **影响系统**: 生产系统

## 五、订单状态机漏洞

### V5-001 [P0] 订单状态无合法转换验证
- **文件**: `order_service.py` 行185-261
- **描述**: 订单状态（SUBMITTED/FILLED/FAILED/CANCELLED等）之间无合法转换矩阵。任何状态可转换为任何其他状态，例如：FILLED→SUBMITTED（已成交回退为已提交）、CANCELLED→FILLED（已取消变为已成交）等非法转换不会被检测或阻止。
- **全链路**: `send_order()` → `order['status']='SUBMITTED'` → `on_trade_update()` → `order['status']='ALL_FILLED'` → 无状态转换校验
- **手册对照**: 手册4.2节要求"订单生命周期管理"，但无状态机约束。
- **影响系统**: 生产系统

### V5-002 [P1] 部分成交仅记录日志无保护动作
- **文件**: `order_service.py` 行509-545
- **描述**: `on_trade_update()` 检测到部分成交时（行532-534），仅记录日志，不触发任何保护动作（如：调整剩余挂单量、更新风控状态、通知策略层）。部分成交可能导致：1）风控系统认为订单未成交而重复下单；2）策略层无法感知仓位变化。
- **影响系统**: 生产系统

### V5-003 [P1] CLOSE订单绕过重复检查
- **文件**: `order_service.py` 行197-204
- **描述**: `send_order()` 中CLOSE订单跳过限流器和重复订单检查（`if action != 'CLOSE' and not is_chase`），可能导致重复平仓订单。在并发场景下，多个CLOSE请求可能同时通过，导致超量平仓。
- **全链路**: `send_order(action='CLOSE')` → 跳过rate_limiter → 跳过duplicate_check → 可能重复平仓
- **影响系统**: 生产系统

### V5-004 [P1] 重试下单使用阻塞式sleep
- **文件**: `order_service.py` 行321-335
- **描述**: `_retry_send_order()` 使用 `time.sleep(2**retry)` 阻塞调用线程，最多阻塞14秒（2^3=8+2^2=4+2^1=2）。在HFT场景下，14秒阻塞可能导致大量信号积压和错过交易窗口。
- **影响系统**: 生产系统

### V5-005 [P2] 超时订单chase使用相同路径可能再次失败
- **文件**: `order_service.py` 行547-575
- **描述**: `check_pending_orders()` 对超时订单执行cancel+chase，但chase使用相同的 `send_order()` 路径。如果超时原因是市场流动性不足或价格剧烈波动，相同路径大概率再次失败，形成"超时→chase→再超时"的循环。
- **影响系统**: 生产系统

## 六、价格保护缺失

### V6-001 [P0] send_order()无价格偏离检查（胖手指防护）
- **文件**: `order_service.py` 行185-261
- **描述**: `send_order()` 接受任意价格，无价格偏离检查。缺少：1）与最新成交价的偏离度检查；2）与买卖盘口的偏离度检查；3）涨跌停价格检查。胖手指错误（如多输一个零）或极端滑点订单将直接提交。
- **手册对照**: 手册7.1节要求"订单流参数及过滤有效性检验"，但下单层无价格合理性校验。
- **全链路**: `send_order(price=X)` → 无max_price_deviation检查 → 平台下单 → 胖手指订单成交
- **影响系统**: 生产系统

### V6-002 [P1] execute_by_ranking()加价无最大偏离限制
- **文件**: `order_service.py` 行401-431
- **描述**: `execute_by_ranking()` 对BUY加tick_size、对SELL减tick_size以提高成交率，但无最大价格偏离限制。在流动性极差的市场中，多次加价可能使最终成交价严重偏离预期。
- **影响系统**: 生产系统

### V6-003 [P1] 止损价固定不随波动率调整
- **文件**: `position_service.py` 行686-701
- **描述**: `_add_position()` 中止盈止损价基于开仓价×固定比率计算，不随市场波动率变化调整。在高波动率环境下，固定止损可能过于狭窄导致频繁触发；在低波动率环境下，固定止损可能过于宽松导致单笔亏损过大。
- **手册对照**: 手册6.2节定义了按理由分化的TP/SL倍数，但未要求波动率自适应。
- **影响系统**: 生产系统

### V6-004 [P2] 极端行情下止损滑点无保护
- **文件**: `position_service.py` 行934-1020
- **描述**: `_trigger_close_position()` 使用市价单平仓，在极端行情下（如涨跌停、流动性枯竭），止损单可能以远超预期的价格成交，造成远超止损线设定的亏损。缺少最大滑点限制或分批平仓策略。
- **影响系统**: 生产系统

## 七、时间窗口漏洞

### V7-001 [P1] EOD平仓仅检查时分，不区分交易日类型
- **文件**: `position_service.py` 行1184-1200
- **描述**: `_check_eod_close()` 仅检查 `now.hour == EOD_CLOSE_HOUR and now.minute >= EOD_CLOSE_MINUTE`，不区分：1）半日交易（如春节前最后交易日）；2）交易所特定休市安排；3）夜盘交易时段。在半日交易日，下午的EOD检查不会触发，持仓可能过夜。
- **手册对照**: 手册9.2节补丁二b要求"会话周期硬终止"，但未覆盖非标准交易日。
- **影响系统**: 生产系统

### V7-002 [P1] Greeks计算Theta使用日历日而非交易日
- **文件**: `greeks_calculator.py` 行102-157
- **描述**: `_bs_greeks()` 中Theta计算除以365.0（日历日），但中国期权市场应使用约242个交易日。日历日计算导致Theta被低估约33.7%（365/242-1），可能低估时间衰减风险。
- **手册对照**: 手册5.1节要求"长假持仓时间损耗占比"用max_theta_ratio控制，但Theta计算基准错误。
- **影响系统**: 生产系统

### V7-003 [P1] time_to_expiry()使用datetime.now()非市场时间
- **文件**: `greeks_calculator.py` 行488-508
- **描述**: `TradingCalendar.time_to_expiry()` 使用 `datetime.now()` 计算到期时间，不区分交易时段和非交易时段。在非交易时间（如夜间、周末），计算出的到期时间包含非交易时间，导致Greeks值偏差。
- **影响系统**: 生产系统

### V7-004 [P2] 熔断器滚动窗口不考虑隔夜缺口
- **文件**: `risk_service.py` 行1715-1772
- **描述**: `SafetyMetaLayer._check_circuit_breaker()` 使用滚动权益序列（maxlen=60）计算1分钟回撤，但不考虑隔夜缺口。开盘跳空时，前一日收盘价与当日开盘价的差异被计入1分钟回撤，可能误触发熔断器。
- **影响系统**: 生产系统

### V7-005 [P2] SignalService收盘时段信号未完全阻断
- **文件**: `signal_service.py` 行59-61
- **描述**: `SignalService` 定义了 `MARKET_CLOSE_HOUR=15` 和 `MARKET_CLOSE_MINUTE_START=15`，但在 `generate_signal()` 中未使用这些常量进行收盘时段信号过滤。收盘前5分钟的信号可能导致无法成交的挂单。
- **影响系统**: 生产系统

## 八、资金安全漏洞

### V8-001 [P0] 透支交易可能：保证金检查与下单非原子
- **文件**: `risk_service.py` 行1934-1961 + `order_service.py` 行185-261
- **描述**: 保证金充足性检查（`check_capital_sufficiency()`）与实际下单（`send_order()`）不在同一原子操作中。并发场景下，多个线程可能同时通过保证金检查，但实际下单总额超过可用资金，导致透支。
- **全链路**: Thread A: `check_capital_sufficiency()` → PASS → Thread B: `check_capital_sufficiency()` → PASS → 两者均下单 → 总保证金超限
- **手册对照**: 手册5.1节要求"希腊字母硬约束"，但检查与执行非原子。
- **影响系统**: 生产系统

### V8-002 [P0] 负权益无硬停止
- **文件**: `risk_service.py` 行1774-1820
- **描述**: 日回撤硬停止基于5日均值×multiplier，但当5日均值为正且较小时，允许的日亏损可能不足以防止负权益。缺少独立的负权益硬停止机制：当equity≤0时，应立即强制平仓所有持仓并禁止任何新交易，但代码中无此检查。
- **手册对照**: 手册9.2节补丁二b要求"会话周期硬终止"，但未覆盖负权益场景。
- **影响系统**: 生产系统

### V8-003 [P1] 资金路由分配总和可能超过100%
- **文件**: `strategy_ecosystem.py` 行549-603
- **描述**: `route_capital()` 中资金分配经过状态联动乘子调整后，虽然做了归一化（行600-603），但在乘子应用前，各策略分配额可能临时超过1.0。若归一化逻辑因浮点精度问题失效（如total接近0），可能导致分配异常。
- **影响系统**: 生产系统

### V8-004 [P1] 再平衡摩擦成本仅扣除bps但未计入实际亏损
- **文件**: `strategy_ecosystem.py` 行610-616
- **描述**: `route_capital()` 中再平衡摩擦成本以bps形式从分配比例中扣除（`allocations[k] *= (1.0 - total_friction_pct)`），但这是对分配比例的调整，不是对实际资金的扣除。实际交易中的滑点和手续费未在此处反映，可能导致资金分配与实际可用资金不一致。
- **影响系统**: 生产系统

### V8-005 [P2] 策略EV底线检查使用缓存可能延迟
- **文件**: `strategy_ecosystem.py` 行1216-1225
- **描述**: `get_health_status()` 中EV检查使用5秒TTL缓存（`_ev_cache_ttl=5.0`）。在快速亏损场景下，缓存可能导致EV底线突破延迟最多5秒才被检测，期间可能继续开仓。
- **影响系统**: 生产系统

## 九、评判系统特有漏洞

### V9-001 [P1] CascadeJudge L3影子策略隐形度阈值方向错误
- **文件**: `evaluation/cascade_judge.py` 行438-446
- **描述**: L3关"影子策略隐形度"使用 `_check_constraint(direction="max")`，即要求 `l3_silhouette_invisibility <= 0.7`。但隐形度越高意味着影子策略越独立，应该是"越低越危险"。阈值方向应为 `direction="min"`（要求隐形度≥0.7），当前逻辑会阻断隐形度高的好策略。
- **全链路**: `judge()` → `_check_constraint(name="L3 影子策略隐形度", direction="max")` → `value <= threshold` → 隐形度0.9被判定为BLOCK
- **影响系统**: 评判系统

### V9-002 [P2] StrategyJudgmentEngine权重修改后未恢复异常路径
- **文件**: `策略评判/strategy_judgment_engine.py` 行545-653
- **描述**: `judge()` 方法在行563修改了 `self._weights` 和 `self._thresholds`，在行652-653恢复。但如果在563-652之间抛出异常（如deep_validation中的ImportError），权重和阈值不会被恢复，影响后续评判调用。虽然使用了try/except，但恢复代码在finally之外。
- **影响系统**: 评判系统

### V9-003 [P2] adapt_backtest_result() Sortino估算可能高估
- **文件**: `evaluation/cascade_judge.py` 行142-152
- **描述**: 当无Sortino直接值且无returns序列时，使用 `sharpe * 1.2` 或 `sharpe + 0.3` 估算Sortino。对于高Sharpe策略（如Sharpe>3），估算值可能显著高于实际Sortino，导致评判过于乐观。
- **影响系统**: 评判系统

## 十、跨系统共性问题

### V10-001 [P1] 三系统Greeks计算不一致
- **文件**: `greeks_calculator.py` / `position_service.py` / `策略评判/strategy_judgment_engine.py`
- **描述**: 生产系统使用GreeksCalculator计算Greeks，position_service使用固定估算值（0.5/-0.5/0.15/0.05），评判系统使用实时风险评分桥接。三系统的Greeks值来源不同，可能导致：1）生产风控基于粗估值放行，评判基于精算值否决；2）回测与实盘风险暴露不一致。
- **手册对照**: 手册5.3.7节"五对齐验证"要求三系统对齐，但Greeks计算未对齐。
- **影响系统**: 生产/测试/评判系统

### V10-002 [P1] 回测与实盘止盈止损逻辑差异
- **文件**: `position_service.py` 行686-701 vs 手册6.2节
- **描述**: 实盘 `_add_position()` 使用 `_get_tp_sl_ratios_by_reason()` 从StateParamManager读取TP/SL倍数，有P3→P2→P1三级降级路由。但回测引擎（task_scheduler）可能使用不同的TP/SL计算路径，导致回测结果与实盘表现不一致。
- **手册对照**: 手册6.2节定义了平仓规则分叉表，但回测与实盘的代码路径未统一。
- **影响系统**: 生产/测试系统
