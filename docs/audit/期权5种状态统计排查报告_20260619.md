## 一、致命问题（P0）

### BUG-1: `_classify_status` 装饰器与参数签名冲突 —— 五态分类完全失效

- **文件**: `data/width_cache_state_mixin.py`
- **位置**: 第173-176行
- **问题代码**:
  ```python
  @staticmethod
  def _classify_status(self, underlying_future_id: int, month: str, opt_type: str,
                       current_price: float, prev_price: float,
                       option_direction: Optional[bool] = None) -> str:
      future_state = self._get_future_runtime_state(underlying_future_id, month)
  ```
- **根因**: 方法被声明为 `@staticmethod`，但第一个参数名为 `self` 且在方法体内调用了实例方法 `self._get_future_runtime_state()`。在 Python 中，`@staticmethod` 不会自动绑定实例，因此通过 `self._classify_status(a, b, c, d, e)` 调用时，参数发生严重错位：`self` 形参接收的是 `underlying_future_id` 实参（一个整数），后续所有参数顺延错位，且 `prev_price` 永远缺失。
- **手动验证结果**:
  ```
  ERROR calling _classify_status: TypeError WidthCacheStateService._classify_status() missing 1 required positional argument: 'prev_price'
  ```
- **影响**: 
  - 所有期权状态分类调用均抛出 `TypeError`，五态统计功能**完全瘫痪**。
  - 调用点包括:
    - `width_cache_state_mixin.py:535` (首Tick初始化)
    - `width_cache_state_mixin.py:548` (常规Tick更新)
    - `width_cache_sort_mixin.py:86` (期货方向变化重算)
    - `width_cache_state_mixin.py:631` (重算全量状态)
  - `_status_counts` 中的五态计数无法更新，下游 `StateParamManager.update_state_from_width_cache()` 读取到的永远是空或旧数据。
  - `select_otm_targets_by_volume` 依赖的 `_status_counts` 和 `_sort_buckets` 也因状态无法分类而失效。
- **修复方案**: 移除 `@staticmethod` 装饰器，恢复为普通实例方法。
- **验收标准**:
  1. `svc._classify_status(1, '2605', 'CALL', 100.0, 99.0, option_direction=True)` 不再抛出 `TypeError`。
  2. 返回正确的五态字符串（如 `correct_rise` / `wrong_rise` / `correct_fall` / `wrong_fall` / `other`）。
  3. 通过 `on_option_tick` 触发后，`_status_counts` 中对应状态的计数正确增加。

### BUG-2: `select_otm_targets_by_volume` 完全没有实现五态全维排序框架

- **文件**: `data/width_cache_query_mixin.py`
- **位置**: 第231-306行
- **问题描述**:
  《期权五态全维排序框架设计报告》要求实现完整的五态全维排序系统，核心包括:
  - 5个月份权重体系: `MONTH_WEIGHTS_5 = (0.35, 0.25, 0.20, 0.12, 0.08)` + 自适应归一化
  - 核心指标: `correct_up_pct = Σ[w_i × CR_i/(CR_i+WR_i)]`
  - `noise_ratio = Σ[w_i × (WR_i+WF_i)/Total_i]`
  - `coverage = Σ[w_i × I(CR_i>0)]`
  - Wilson小样本校正 (`wilson_lower_bound`)
  - Tier 1-4 分级逻辑
  - 排序键: `(tier升序, correct_up_pct降序, wilson降序)`
  - 手数分配: Tier 1→2手, Tier 2→1手
  - Top期货参数化取数

  但实际代码仅做了最简单的 `correct_rise` 计数和比较:
  ```python
  cr = sum(m_data.get(opt_type, {}).get(target_status, 0) for m_data in month_data.values())
  wr = sum(m_data.get(opt_type, {}).get('wrong_rise', 0) for m_data in month_data.values())
  if cr > 0 and cr >= wr:
      future_scores.append({'future_internal_id': fid, 'correct_rise': cr, 'wrong_rise': wr})
  future_scores.sort(key=lambda x: -x['correct_rise'])
  top_futures = future_scores[:2]  # 硬编码
  ```
  月份权重、Wilson、Tier、correct_up_pct、noise_ratio、coverage 等核心逻辑**全部缺失**。
- **影响**:
  - 品种排序逻辑与设计报告严重不符，排序结果没有统计严谨性。
  - 无法利用五态中的 `wrong_rise` / `correct_fall` / `wrong_fall` 信息进行风险调整和噪声过滤。
  - 手数分配硬编码为 `2 if pidx == 0 else 1`，未按 Tier 分配。
  - Top期货取数硬编码为 `[:2]`，不可配置。
- **修复方案**:
  1. 在 `WidthCacheQueryService`（或新建模块）中实现 `_resolve_month_weights`、`compute_correct_up_pct`、`compute_noise_ratio`、`compute_coverage`、`wilson_lower_bound`、`determine_tier` 等方法。
  2. 重写 `select_otm_targets_by_volume`，按设计报告第5节流程实现：五态采集 → 按月计算指标 → Wilson校正 → Tier分级 → 排序 → 取Top N。
  3. 提取 `TOP_FUTURES_COUNT`、`TIER1_LOTS`、`TIER2_LOTS` 为类级常量。
  4. 月份来源使用 `month_mapping` 配置为主数据源（如设计报告4.4节）。
- **验收标准**:
  1. 使用设计报告第7节数值示例的数据调用 `select_otm_targets_by_volume`，返回的品种A为 Tier 1, correct_up_pct≈0.807；品种B为 Tier 4, correct_up_pct≈0.327。
  2. 手数分配遵循 Tier 规则（Tier 1→2手，Tier 2/3/4→1手）。
  3. 所有类级常量（`MONTH_WEIGHTS_5`、`TIER1_WILSON_THRESHOLD` 等）可在类定义处集中修改。

## 二、严重问题（P1）

### BUG-3: `print_status_diagnosis` 过滤条件永远为 False

- **文件**: `data/width_cache_diagnosis_mixin.py`
- **位置**: 第51行
- **问题代码**:
  ```python
  if future_internal_id is not None and future_id != future_id:
      continue
  ```
- **根因**: `future_id != future_id` 在数学上永远为 `False`。当调用方传入了 `future_internal_id` 参数时，循环不会过滤，所有期货的数据都会被统计，导致诊断输出错误。
- **影响**: 按品种过滤的诊断功能失效，运维人员看到的可能是全量聚合数据而非目标品种数据。
- **修复方案**: 将 `future_id != future_id` 改为 `future_id != future_internal_id`。
- **验收标准**:
  1. 传入 `future_internal_id=123` 时，只统计 `future_id==123` 的数据。
  2. 传入 `future_internal_id=None` 时，统计所有期货数据。

### BUG-4: `_recalc_all_state_for_underlying` 重复定义

- **文件**: `data/width_cache_sort_mixin.py` 第34行、`data/width_cache_state_mixin.py` 第579行
- **问题描述**: 同名方法在两个 mixin 文件中均有完整实现。`WidthStrengthCache` 继承 `WidthCacheStateService`，因此基类版本会被直接使用。`WidthCacheSortService` 的版本通过 `__getattr__` 反向访问 facade，但由于基类已存在同名方法，不会产生运行时错误，但造成了代码冗余和维护歧义。
- **影响**: 代码冗余，维护者不清楚哪个版本实际生效；如果未来调整继承顺序或 `__getattr__` 逻辑，可能引发不可预期的行为。
- **修复方案**: 
  1. 保留 `WidthCacheStateService` 中的版本（因为它是基类）。
  2. 删除 `WidthCacheSortService` 中的重复实现，或在其中显式委托给基类。
- **验收标准**: 
  1. 两个文件中不再存在逻辑重复的全量 `_recalc_all_state_for_underlying`。
  2. 期货方向变化时，状态重算链路正常触发，`_status_counts` 正确更新。

## 八、运行时验证阶段新发现问题

### BUG-7 (P1): `WidthCacheContextMixin.__getattr__` 委托逻辑缺陷

- **文件**: `data/width_cache_context_mixin.py`
- **位置**: 第25-33行
- **问题代码**:
  ```python
  def __getattr__(self, name):
      if self._facade is not None:
          for klass in type(self._facade).__mro__:
              if name in klass.__dict__:
                  attr = klass.__dict__[name]
                  if hasattr(attr, '__get__'):
                      return attr.__get__(self._facade, type(self._facade))
                  return attr
      raise AttributeError(...)
  ```
- **根因**: 该 `__getattr__` 只遍历 facade 的类字典 (`klass.__dict__`)，无法访问 facade 的**实例属性**（如 `_instrument_id_to_internal_id`、`_option_info`、`_lock` 等）。当 `_context_service` 中的方法（如 `_resolve_internal_id`）尝试访问这些属性时，委托链断裂，抛出 `AttributeError`。
- **影响**: 
  - `register_option` 中调用的 `_resolve_internal_id` 失败，期权无法注册。
  - `on_option_tick` 中调用的 `_get_future_runtime_state` 失败，Tick 处理中断。
  - 门面组合模式的委托链路存在根本缺陷。
- **修复方案**: 将 `__getattr__` 改为直接使用 `hasattr(self._facade, name)` + `getattr(self._facade, name)`，同时委托实例属性和类属性。
- **验收标准**:
  1. `cache._context_service._instrument_id_to_internal_id` 能正确访问到 facade 的实例属性。
  2. `cache.register_option()` 和 `cache.on_option_tick()` 完整链路不再抛出 `AttributeError`。

---

### BUG-8 (P1): `width_cache_state_mixin.py` / `width_cache_context_mixin.py` 缺少 `datetime` 导入

- **文件**: `data/width_cache_state_mixin.py`, `data/width_cache_context_mixin.py`
- **位置**: `_get_current_trading_date` 方法
- **问题代码**:
  ```python
  def _get_current_trading_date(self) -> str:
      return datetime.now(CHINA_TZ).strftime('%Y%m%d')
  ```
- **根因**: 两个文件均使用了 `datetime.now()`，但缺少 `from datetime import datetime` 导入。`_get_current_trading_date` 在 `on_option_tick` 中被调用，导致运行时 `NameError: name 'datetime' is not defined`。
- **影响**: `on_option_tick` 完整链路在运行时中断，期权 Tick 无法处理。
- **修复方案**: 在两个文件顶部添加 `from datetime import datetime`。
- **验收标准**:
  1. `python -m py_compile` 两个文件通过。
  2. `on_option_tick` 运行时不再抛出 `NameError`。

### BUG-9 (P0): `select_otm_targets_by_volume` 未过滤 Tier 4 期货 —— 违反"Tier 4 = 不交易"设计

- **文件**: `data/width_cache_query_mixin.py`
- **位置**: 第383-394行
- **问题代码**:
  ```python
  future_scores.append({
      'future_internal_id': fid,
      'tier': tier,           # 可能是 4
      ...
  })
  future_scores.sort(key=lambda x: (x['tier'], -x['correct_up_pct'], -x['wilson']))
  top_futures = future_scores[:self.TOP_FUTURES_COUNT]   # Tier 4 也被选入
  ```
- **根因**: 设计报告第二节明确 `Tier 4 | 其余 | 不交易`，第五节流程图过滤条件为 `correct_up_pct > 0 且 correct_up_pct ≥ noise_ratio`。但代码将所有期货（含 Tier 4）加入 `future_scores`，排序后直接取 Top N。当可交易品种不足 `TOP_FUTURES_COUNT` 时，Tier 4 期货会被选为交易目标。
- **影响**:
  - `runtime_verify_option_sort.py` 第149-153行验证了期货B（Tier 4）被返回并分配 lots=1，与设计报告"Tier 4 不交易"直接矛盾。
  - 实盘可能对统计不显著、噪声过高的品种开仓，造成亏损。
- **修复方案**: 在 `future_scores.append` 前增加 `if tier == 4: continue` 过滤。
- **验收标准**:
  1. Tier 4 期货不出现在返回的 `targets` 中。
  2. 当所有期货均为 Tier 4 时，`select_otm_targets_by_volume` 返回空列表。

### BUG-10 (P0): `TTypeService.select_otm_targets_by_volume` 参数语义错位

- **文件**: `data/t_type_service.py`
- **位置**: 第380-385行
- **问题代码**:
  ```python
  def select_otm_targets_by_volume(self, product: str = None, ...):
      wc = self._get_width_cache(strategy_id)
      if wc:
          return wc.select_otm_targets_by_volume(product)   # product(str) 传给 future_internal_id(int)
      return []
  ```
- **根因**: `TTypeService.select_otm_targets_by_volume` 的参数名为 `product`（字符串，如 'IF'），但委托给 `WidthCacheQueryService.select_otm_targets_by_volume(self, future_internal_id: int = None)` 时直接透传。下游过滤条件 `if future_internal_id is not None and fid != future_internal_id` 中 `fid` 为 int、`future_internal_id` 为 str，恒不相等，导致**所有期货被过滤掉**，返回空列表。
- **影响**:
  - 当前生产调用 `t_type.select_otm_targets_by_volume()` 不传参，故 `product=None` 时不触发。但一旦调用方传入品种名做过滤，将静默返回空结果，造成"无信号"假象。
  - 参数命名误导维护者，以为可以按品种过滤。
- **修复方案**: 将 `TTypeService.select_otm_targets_by_volume` 的参数改为 `future_internal_id: int = None`，与下游签名对齐。
- **验收标准**:
  1. 传入 `future_internal_id=1` 时，只返回该期货的候选。
  2. 传入 `future_internal_id=None` 时，返回全量候选。

### BUG-12 (P1): `resolve_month_weights` 当 n_months > 5 时返回等权重，违反设计报告4.3节

- **文件**: `data/width_cache_query_mixin.py`
- **位置**: 第259-261行
- **问题代码**:

- **验收标准**:
  1. `resolve_month_weights(6)` 返回 `(0.35, 0.25, 0.20, 0.12, 0.08)`。
  2. `resolve_month_weights(8)` 同上。

### BUG-14 (P1): `select_otm_targets_by_volume` 持仓过滤逻辑缺陷 —— 仅检查最优候选，已持仓则跳过整个品种

- **文件**: `data/width_cache_query_mixin.py`
- **位置**: 第425-430行
- **问题代码**:
  ```python
  best_candidates.sort(key=lambda x: x['volume'], reverse=True)
  best = best_candidates[0]                    # 只取第1名

      if pos_svc and ... and pos_svc.has_position(best['instrument_id']):
  ```

- **根因**: 当 `best_candidates[0]` 已有持仓时，`continue` 直接跳过该品种，未尝试 `best_candidates[1]`、`best_candidates[2]` 等次优候选。设计报告第五节阶段二 Step 3 "持仓过滤"应是从候选列表中**排除已持仓**后取最优，而非"首个已持仓则放弃全品种"。

- **验收标准**: 当 best_candidates[0] 已持仓但 best_candidates[1] 未持仓时，返回 best_candidates[1]。

### BUG-17 (P2): `select_otm_targets_by_volume` 未计算诊断指标 tc/th/ra

- **文件**: `data/width_cache_query_mixin.py`
- **问题**: 设计报告3.2节要求计算三个诊断指标：

- **影响**: 设计报告第十节"衰减监测"依赖 th/ra 指标，缺失后衰减预警无法触发。
- **修复方案**: 新增 `compute_th` 和 `compute_ra` 静态方法，在 `select_otm_targets_by_volume` 中调用并写入 target。
- **验收标准**: target 包含 th/ra 字段，数值符合设计报告公式。

### BUG-18 (P0): Tick数据未路由到TTypeService — 5态统计全other（运行时根因）

- **文件**: `strategy/tick_processing_service.py`
- **现象**: 重启后日志显示 `[5种状态统计] total=15230 | correct_rise=0 wrong_rise=0 correct_fall=0 wrong_fall=0 other=15230`，所有期权恒为 `other`。
- **根因链路**:
  1. `_classify_status` 返回 `'other'` 的条件：`future_state['initialized'] == False`

### 14.4 多品种组合排序算法

```python
def sort_options_cross_products(option_states: dict, product_config: dict) -> list:
    """
    顶级基金视角：跨品种期权5态排序
    
    Args:
        option_states: {option_id: {status, price, volume, ...}}
        product_config: {product_category: {weights, decay, ...}}
    
    Returns:
        sorted_targets: [(option_id, score, category), ...]
    """
    scored = []
    for opt_id, state in option_states.items():
        # 1. 识别品种类别
        product = extract_product(opt_id)
        category = classify_product(product)
        config = product_config[category]
        
        # 2. 计算状态权重分
        status = state['status']
        if status == 'other':
            continue
        weight = config['state_weights'][status]
        
        # 3. 计算时间衰减因子
        age = time.now() - state['last_update']
        decay = 0.5 ** (age / config['decay_half_life'])
        
        # 4. 计算流动性因子（成交量归一化）
        liquidity = min(1.0, state['volume'] / config['ref_volume'])
        
        # 5. 计算覆盖率因子
        coverage = state.get('coverage', 0)
        if coverage < config['min_coverage']:
            continue
        
        # 6. 综合评分
        score = weight * decay * liquidity * coverage
        scored.append((opt_id, score, category))
    
    # 7. 按品种类别分配风险预算（每类最多N个）
    budget = {cat: 3 for cat in product_config}  # 每类最多3个
    result = []
    for opt_id, score, cat in sorted(scored, key=lambda x: -x[1]):
        if budget[cat] > 0:
            result.append((opt_id, score, cat))
            budget[cat] -= 1
        if sum(budget.values()) == 0:
            break
    
    return result
```

### 15.2 对第十四章的批判式吸收

第十四章提出了"顶级基金视角"的跨品种差异化排序方案，其动机（品种异质性）具有工程直觉，但方案本身在数学严谨性、系统兼容性和落地可行性三个维度上存在显著缺陷。以下逐节批判。

#### 15.2.1 设计哲学（14.1）：方向正确，但与现有系统错位

**合理内核**：
- "品种特性优先"和"风险预算驱动"是机构级量化系统的共识，确实比当前"一刀切"的 `MONTH_WEIGHTS_5` 更接近实战。
- "多时间框架融合"在概念上是对现有单 tick 状态分类的天然扩展。

**批判**：
- **现有系统尚未跑稳，不宜叠加第二层复杂度**。第十三章记录显示，系统在 2026-06-22 刚刚完成 16 笔交易（此前为 0），说明基础链路（tick 路由 → 五态分类 → Tier 分级 → 排序 → 风控 → 下单）刚刚打通。此时引入跨品种差异化权重，相当于在尚未验证的公式上再套一层非线性变换，一旦出错将难以定位是底层框架问题还是上层参数问题。
- **"自适应衰减"与现有 `Tier` 体系功能重叠**。第十四章的 `decay_half_life` 旨在降低陈旧状态的权重，但现有 `Tier` 分级中已经通过 `coverage`（有效月份占比）和 `wilson`（小样本校正）实现了对数据质量和时效性的惩罚。两者叠加可能导致"双重衰减"，使分数过度保守。

#### 15.2.2 品种分类矩阵（14.2）：分类合理，权重主观

**合理内核**：
- 将品种按波动率/流动性/季节性分为 9 类，符合商品期货的微观结构常识。例如金融期货（IF/IH/IC/IM）确实具有低波、高流动性、弱季节性的特征。

**批判**：
- **权重数值缺乏实证支撑**。表中 `state_weights` 的赋值（如农产品高波的 `correct_rise: 0.35`）未给出任何历史回测、夏普比率或胜率统计的依据，属于"专家拍脑袋"。量化系统的参数如果不经过样本外检验，极易过拟合。
- **分类粒度与交易标的不匹配**。当前系统的交易单元是"期权"，而分类矩阵的单元是"期货品种"。同一期货品种下可能有多个月份、CALL/PUT、多档行权价的期权，它们的波动率特征并不相同。例如 IF 近月平值期权波动率可能显著高于远月虚值期权，统一按"金融期货-低波"处理会丢失关键信息。

#### 15.2.3 差异化排序方案（14.3）：与现有 Tier 体系的根本性冲突

**核心冲突**：现有框架的经济学含义 vs. 第十四章的直觉权重。

现有 `WidthCacheQueryService.select_otm_targets_by_volume` 的评分逻辑是：
1. `correct_up_pct = Σ[w_i × CR_i/(CR_i+WR_i)]` —— 衡量"上涨时猜对的概率"
2. `noise_ratio = Σ[w_i × (WR_i+WF_i)/Total_i]` —— 衡量"噪声占比"
3. `tier` 由 `correct_up_pct`、`noise_ratio`、`coverage`、`wilson` 联合决定 —— 是一个**基于统计显著性的硬阈值分级**。

第十四章的方案则直接给五态分配权重（如 `correct_rise: 0.35`、`wrong_fall: 0.20`），其隐含假设是：
- `wrong_fall`（期货下跌但期权猜错方向）具有正向交易价值，权重 0.20 意味着系统应该**奖励**这种状态。

**批判**：
- **`wrong_fall` 在现有框架中被定义为错误状态**。`width_cache_state_mixin.py` 中 `_classify_status` 返回 `wrong_fall` 的条件是期权方向与期货实际方向相反。将错误状态赋以正权重，等同于在排序公式中引入"反向指标"，这与 `Tier` 分级逻辑（噪声高 → 降级/过滤）直接矛盾。
- **如果确实需要利用 `wrong_fall` 做均值回归策略，不应在现有趋势跟随框架内混合**。现有框架的设计哲学是"趋势跟随"（CR/CF 为正，WR/WF 为噪声），而第十四章的农产品高波和有色金属方案带有明显的"均值回归"色彩。两种哲学在同一排序函数中混合，会导致策略信号打架、仓位方向冲突。
- **建议**：若需实现均值回归策略，应另建一套 `WidthCacheMeanReversionService`，独立维护状态、排序和风控，而非在现有 `WidthCacheQueryService` 中做权重补丁。

#### 15.2.4 多品种组合排序算法（14.4）：伪代码的数学缺陷

对 `sort_options_cross_products` 函数的逐项批判：

**缺陷 1：四个 [0,1] 因子连乘导致分数过度压缩**
```python
score = weight * decay * liquidity * coverage
```
假设各因子均值为 0.5，则 `score ≈ 0.0625`。在 0.06 量级上区分数百个期权的优劣，数值稳定性极差，且对任何一个因子的微小扰动极度敏感（乘积的相对误差等于各因子相对误差之和）。

**缺陷 2：`coverage` 被重复使用**
现有框架中 `coverage` 已经是 `Tier` 分级的核心输入（`Tier 1` 要求 `coverage ≥ 0.8`）。在通过 `Tier` 门槛后，再次将 `coverage` 乘入 `score`，相当于对同一信息重复计价。

**缺陷 3：`liquidity = min(1.0, volume / ref_volume)` 缺乏定义**
`ref_volume`（参考成交量）在代码和配置中均未定义，无法落地。

**缺陷 4：未继承现有核心逻辑**
伪代码完全丢弃了 `Wilson` 校正、`Tier` 分级、月份权重等已验证的统计组件，退化为一个简易加权乘积模型。从统计严谨性角度，这是倒退而非升级。

**缺陷 5：风险预算分配过于粗糙**
```python
budget = {cat: 3 for cat in product_config}
```
每类固定 3 个标的，未考虑：
- 类别内部的相关性（如 `i/j/jm` 同属黑色系，高度相关，同时选 3 个并不能分散风险）
- 类别间的波动率差异（农产品高波的一个标的可能比金融期货 3 个标的的风险贡献更大）
- 账户权益的动态变化

#### 15.2.5 实施路线图（14.5）：验收标准主观且不可量化

| 阶段 | 批判 |
|------|------|
| Phase 1 | "配置加载无异常"是必要非充分条件，不验证配置的经济含义 |
| Phase 2 | "权重和=1.0"是数学约束，不是策略约束；权重可以是负数（对冲），也可以不归一（回归系数） |
| Phase 3 | "状态超时自动降权"的实现需要确定衰减触发器和回测基准，未给出 |
| Phase 4 | "单类集中度<50%"是组合风控常识，但未说明如何与现有 `risk_service.py` 的保证金检查联动 |
| Phase 5 | **"月度回测Sharpe>1.5"是最严重的问题**。Sharpe>1.5 是一个结果目标，不是验收标准；若未达成，无法反推是权重问题、数据问题还是市场问题。且 1.5 的阈值在商品期权市场（高偏度、厚尾、低流动性）中过于激进，缺乏文献支撑。 |

#### 15.2.6 风险控制要点（14.6）：合理但缺少实现路径

14.6 的 5 条风控要点（单品种集中度、类别相关性、状态翻转检测、衰减熔断、流动性门槛）在原则上是正确的，但存在以下落地障碍：

1. **"状态翻转检测：CR→WR 时自动减仓50%"**：现有 `PositionService` 没有基于"状态翻转"的减仓接口。减仓触发器需要监听 `_classify_status` 的返回值变化，并在变化时跨服务调用 `PositionService.reduce_position()`，这会引入新的耦合点。
2. **"衰减熔断：覆盖率连续3分钟低于阈值"**：现有系统没有"连续 N 分钟低于阈值"的状态机。实现需要引入时间窗口缓存（如 `deque(maxlen=180)`），对 `width_cache` 的内存占用和性能都有影响。
3. **"流动性门槛：低于100手/日不参与排序"**：期权的日成交量在开盘和收盘时段差异巨大，使用"日级"阈值过滤会导致盘中可用标的骤减。应使用滚动窗口（如过去 30 分钟成交量）而非日累计。

#### 15.2.7 集成点（14.7）：低估了改造成本

14.7 列出的 5 个集成点（配置层、数据层、排序层、风控层、交易层）看似清晰，但实际改造量巨大：

- **数据层**：`_classify_status` 增加"品种类别参数"意味着每次调用都需要查表（`instrument_id → product → category`），而 `_classify_status` 在 tick 路径上每秒被调用数千次，查表延迟不可忽略。
- **排序层**：`select_otm_targets_by_volume` 当前返回的是 `List[Dict]`，其中字段与现有下游（`TTypeService` → `StrategyBusinessLayer.execute_option_trading_cycle`）强耦合。增加差异化权重后，下游所有解析逻辑都需要同步修改。
- **风控层**：`check_capital_sufficiency` 当前仅做保证金是否充足的二元判断。按品种类别分配风险预算需要重写为带约束的优化问题（如马科维茨或风险平价），远超现有方法的复杂度。

**综合评估**：第十四章是一个**愿景文档**，而非**实施文档**。它描述了"顶级基金可能想要什么"，但没有回答"在现有代码基础上如何不破坏既有功能地逐步实现"。若直接按 14.7 的集成点执行，预计会引入新一轮 P0 级回归缺陷。

### 15.3 可落地的改进建议

基于上述批判，提出三档改进方案，按优先级递增：

#### 15.3.1 短期（1-2 周）：在现有框架内做参数化封装

**目标**：在不改变排序算法数学结构的前提下，允许不同品种使用不同的 `Tier` 阈值。

**动作**：
1. 在 `WidthCacheQueryService` 中新增类方法 `_get_product_tier_thresholds(product: str) -> dict`，返回该品种的 `TIER1_WILSON_THRESHOLD`、`TIER2_COVERAGE_THRESHOLD` 等阈值。
2. 默认全部品种使用当前硬编码常量；对 IF/IC/IM 等金融期货，可单独降低 `TIER1_WILSON_THRESHOLD`（因为金融期货 tick 噪声小，Wilson 门槛可以放宽）。
3. 在 `select_otm_targets_by_volume` 中，`tier = self.determine_tier(...)` 改为 `tier = self.determine_tier(..., thresholds=self._get_product_tier_thresholds(product))`。

**优势**：
- 完全复用现有 `correct_up_pct` / `noise_ratio` / `wilson` / `Tier` 数学框架，不引入新的经济学假设。
- 改动范围集中在 `width_cache_query_mixin.py`，不影响 `width_cache_state_mixin.py` 的分类逻辑。
- 可通过 A/B 测试验证：同一交易日，对比默认阈值 vs. 金融期货降低阈值后的 `Tier` 分布差异。

#### 15.3.2 中期（1 个月）：引入状态时效性衰减作为叠加因子

**目标**：解决第十四章提出的"自适应衰减"需求，但以兼容现有框架的方式实现。

**动作**：
1. 在 `_status_counts` 的每个状态计数器中，不再只存 `int`，而是存 `(count: int, last_update_ts: float)` 元组。
2. 在 `select_otm_targets_by_volume` 计算 `correct_up_pct` 之前，对每个月份的 CR/WR/CF/WF 计数施加衰减：`count_eff = count * exp(-λ * (now - last_update_ts))`，其中 `λ = ln(2) / half_life`。
3. `half_life` 按品种类别配置（如金融期货 120s、黑色系 300s），但衰减仅作用于计数，不改变 `Tier` 分级的阈值体系。

**优势**：
- 衰减是"乘法修正"，作用于输入数据层，不改变 `correct_up_pct` 和 `Tier` 的定义，数学上自洽。
- 若衰减效果不佳，可将 `half_life` 设为极大值（如 86400s），等价于关闭衰减，具备回滚能力。

#### 15.3.3 长期（3 个月）：建立品种特征数据库，用数据驱动替代主观权重

**目标**：用历史回测结果校准品种参数，替代第十四章中"专家拍脑袋"的权重。

**动作**：
1. 建立 `product_feature_db`（ DuckDB 表或本地 Parquet），每日收盘后记录各品种期权的：
   - 五态分布（CR/WR/CF/WF/Other 占比）
   - 各 `Tier` 品种的次日收益率分布
   - 状态翻转频率（CR→WR、CF→WF 的转移矩阵）
2. 每月初运行回测脚本，计算不同 `Tier` 阈值组合下各品种的夏普比率、最大回撤、Calmar 比率。
3. 将回测输出的最优阈值自动写入 `config/product_tier_thresholds.json`，作为下一交易日的参数。

**验收标准**：
- 不是"Sharpe>1.5"这种结果指标，而是"回测系统能自动产出参数文件并热加载"的过程指标。
- 人工只需审核参数变化的合理性（如阈值不应从 0.8 突变到 0.2），不做主观赋值。

#### 15.3.4 明确否定：不建议在现有 `select_otm_targets_by_volume` 中引入第十四章的 `state_weights`

**理由**：
- `state_weights` 将错误状态（WR/WF）赋以正权重，与现有"趋势跟随"框架的设计哲学冲突。
- 若策略团队确有"均值回归"或"反向收割假突破"的需求，应**新建独立服务**（如 `MeanReversionOptionSelector`），独立维护状态、排序和持仓，与现有 `WidthCacheQueryService` 并行运行，通过策略层开关切换。
- 混合两种哲学于同一函数，是 BUG-2（框架缺失）级别的根本性错误，其危害大于不实现差异化排序。

1. **修复核查**：全量 26 项 BUG 及 4 项技术债务均已在代码层面修复，与报告记录一致。但 BUG-18 降级路径性能、BUG-22 硬编码映射的维护性、BUG-24 委托链深度仍需在后续迭代中关注。

### 16.2 对第十四章的再批判

第十五章（15.2）已经从数学严谨性、系统兼容性和落地可行性三个维度对第十四章进行了详尽批判。本轮再批判聚焦于第十五章**未深入覆盖或未提及**的视角，与第十五章形成互补而非重复。

#### 16.2.1 "顶级基金视角"的叙事包装与实质内容的落差

第十四章的标题和 14.1 节使用了"顶级基金视角"这一叙事框架，但其实际内容更接近一本《量化投资入门》教科书的课后习题：

- **14.2 品种分类矩阵**：波动率的高/中/低、流动性的高/中/低、季节性的强/中/弱，这些定性分类是任何商品期货入门教材的基础知识，不是"顶级基金"的差异化能力。
- **14.3 差异化权重**：给 CR/CF/WR/WF 分配 0.35/0.15/0.30/0.20 这样的权重，本质上是一个**线性评分卡**（linear scorecard），这是 1990 年代的量化方法，不是顶级基金在 2026 年使用的技术。
- **14.4 伪代码**：`score = weight * decay * liquidity * coverage` 是一个标准的**乘积效用函数**（multiplicative utility function），在金融工程教材中属于第 3 章内容。

**核心判断**：第十四章的价值在于其**方向性直觉**（品种异质性需要差异化处理），而非其**技术方案**。将第十四章称为"顶级基金视角"是一种叙事包装，不应被其标题误导而高估其技术含量。

#### 16.2.2 期权微观结构视角的缺失

第十五章主要从排序框架的数学结构角度批判第十四章，但未涉及期权市场的微观结构。第十四章存在以下微观结构层面的盲区：

**盲区 1：期权 Greeks 的完全缺失**

第十四章的评分公式仅使用五态分类（CR/WR/CF/WF/Other）和成交量，完全没有考虑期权的 Greeks：
- **Delta**：不同行权价的期权对标的期货价格变化的敏感度不同。两个同品种的 CALL 期权，一个 Delta=0.9（深度实值），一个 Delta=0.1（深度虚值），其 CR/WR 状态的统计意义完全不同。深度虚值期权的 CR 可能只是随机噪声，不应与深度实值期权的 CR 等权重。
- **Gamma**：近月平值期权的 Gamma 远高于远月虚值期权，意味着状态翻转速度更快。第十四章的 `decay_half_life` 是基于品种类别的静态值，但同一品种内不同期权的状态有效时间可以差一个数量级。
- **Vega**：隐含波动率的变化对期权价格的影响远大于标的期货价格变化对五态分类的影响。一个期权被分类为 `correct_rise`，可能不是因为方向判断正确，而是因为 IV 上升——五态分类本身无法区分这两种情况。

**盲区 2：做市商行为对五态分类的污染**

在低流动性期权市场，做市商的报价行为会显著影响五态分类：
- 做市商在临近收盘时撤单（宽 bid-ask spread），导致 `last_price` 出现虚假跳变，`_classify_status` 可能将正常的持仓分类为 `wrong_rise` 或 `wrong_fall`。
- 第十四章未提及任何做市商行为过滤或 tick 质量控制机制。

**盲区 3：不同交易所的微观结构差异**

第十四章将金融期货（IF/IH/IC/IM）归为一类，但它们在中金所交易，而商品期货在上期所/大商所/郑商所交易。不同交易所的：
- 最小价格变动单位（tick size）不同
- 涨跌停板幅度不同
- 期权行权价间距不同
- 集合竞价机制不同

这些差异会影响五态分类的粒度和频率，但第十四章的分类矩阵完全忽略了交易所层面的差异。

#### 16.2.3 第十四章与现有系统的数据流冲突（第十五章未覆盖的角度）

第十五章 15.2.3 讨论了 Tier 体系与 `state_weights` 的哲学冲突（趋势跟随 vs 均值回归），但未讨论**数据流**层面的冲突：

**冲突 1：五态分类是 tick 级，品种权重是配置级**

现有 `_classify_status` 在**每个 tick** 上运行，产生的是**流式、实时**的状态数据。第十四章的 `state_weights` 是**静态配置**（YAML 文件），在系统启动时加载一次。两者之间存在时间维度上的错配：
- 市场进入高波动状态（如农产品收获季），当日有效的 CR 权重应该与平日不同，但第十四章的 YAML 配置无法感知市场状态变化。
- 如果某个品种从"正常波动"切换为"高波动"，需要人工修改 YAML 并重启系统，这在实盘中是不现实的。

**冲突 2：第十四章引入了新的排序维度但未定义与现有维度的优先级**

现有排序键是 `(tier升序, correct_up_pct降序, wilson降序)`，这是一个**全序关系**（total order）。第十四章引入 `state_weights` 后，排序键变为：
- 按品种类别分组 → 类内按 `weight * decay * liquidity * coverage` 排序 → 跨类按风险预算分配

这是一个**偏序关系**（partial order），因为不同品种类别之间没有可比性。从全序到偏序的转变，意味着下游代码（`TTypeService` → `StrategyBusinessLayer`）的"取 Top N"逻辑需要重写为"按类别取 Top N_i"，这是一个**接口契约级别的破坏性变更**。

#### 16.2.4 统计显著性的根本性缺失

第十五章批评了第十四章"权重数值缺乏实证支撑"，但未从**统计显著性**角度深入分析。这一点值得单独强调：

**当前系统的样本量无法支撑差异化参数估计**：
- 第十三章记录显示，系统在 2026-06-22 重启后 37 分钟内完成了 16 笔交易，5 态统计中非 other 占比仅为 0.17%-1.82%。
- 以 1% 的非 other 占比计算，每分钟产生约 15000 个 tick，其中约 150 个产生非 other 状态。按 9 个品种类别均分，每个类别每分钟约 17 个有效状态。
- 要估计 5 个 `state_weights`（CR/WR/CF/WF/Other），每个类别需要至少数百个有效状态才能获得统计上显著的权重估计。这意味着需要数小时的连续交易数据，而当前系统刚跑通基础链路。

**第十四章的 9 类 x 5 权 = 45 个参数，在样本量不足的情况下，参数估计的方差将远大于参数本身的值**，这就是为什么第十五章称之为"专家拍脑袋"——不是因为专家不专业，而是因为在当前数据量下，任何方法都只能是拍脑袋。

#### 16.2.5 第十五章改进建议的补充与修正

第十五章 15.3 提出了三档改进方案（短期参数化阈值 → 中期状态衰减 → 长期数据驱动回测），这些建议方向正确，但以下两点需要补充：

**补充 1：在短期方案中增加"品种分组"而非"品种分类"**

第十五章建议的 `_get_product_tier_thresholds(product)` 是按单个品种配置阈值，但更好的做法是**按品种分组**（group）配置：
- 分组粒度：金融期货组 / 黑色系组 / 有色金属组 / 农产品组 / 能源化工组
- 每组只需 1 套 Tier 阈值，共 5 套，而非每个品种 1 套（当前系统涉及 20+ 品种）
- 分组的依据不是第十四章的"波动率/流动性/季节性"定性判断，而是**实测的五态分布相似度**（用 KL 散度或 Jensen-Shannon 距离聚类）

**补充 2：中期衰减方案需要考虑衰减的"因果方向"**

第十五章建议的 `count_eff = count * exp(-lambda * (now - last_update_ts))` 假设衰减是**对称的**（所有状态同等衰减）。但金融直觉上，`correct_rise` 和 `wrong_rise` 的衰减速度应该不同：
- `correct_rise`（判断正确）：趋势可能持续，衰减应慢
- `wrong_rise`（判断错误）：假突破信号快速失效，衰减应快

建议将对称衰减改为**非对称衰减**：
```python
count_eff = count * exp(-lambda_status * (now - last_update_ts))
# lambda_wrong_rise, lambda_wrong_fall > lambda_correct_rise, lambda_correct_fall
```

这与第十四章"农产品高波品种 WR/WF 权重提升"的直觉方向一致，但实现方式不同：不是在评分阶段给 WR/WF 赋正权重，而是在数据输入阶段让"错误状态"更快衰减，自然降低其对评分的污染。

#### 16.2.6 第十四章中最有价值的两个思想（值得保留）

尽管上述批判集中在第十四章的问题上，但其中有两个思想值得在后续迭代中保留：

**思想 1：品种差异化处理（14.2 的核心）**

这是第十四章最核心的贡献。即使当前不应实现完整的差异化权重体系，但"不同品种的行为特征不同"这一认知应该体现在代码架构中。具体做法：
- 在 `WidthCacheQueryService` 的类设计中预留 `product` 参数（当前 `select_otm_targets_by_volume` 的 `future_internal_id` 已经可以反查品种）
- 在 `_get_scoring_months` 中已经使用了 `month_mapping`（按品种配置月份），这是差异化处理的一个成功案例

**思想 2：风险预算驱动的仓位分配（14.4 的动机）**

第十四章的风险预算分配（每类最多 N 个标的）的动机是正确的——防止单一品种类别过度集中。但实现方式不应是固定预算（每类固定 3 个），而应是**动态风险预算**：
- 基于各品种类别的近期波动率（如过去 30 分钟 ATR）分配风险预算
- 高波动类别分配更多预算（因为要求更大的止损空间），但限制标的数量（因为波动大，少数标的就能消耗完预算）
- 低波动类别分配较少预算但允许更多标的（分散化）

这个方向可以作为中期目标，在基础链路稳定运行至少 1 个月后再启动。

#### 17.2.1 product_feature_daily

存储每个期货品种每个交易日的五态分布和基础统计特征。

```sql
CREATE TABLE IF NOT EXISTS product_feature_daily (
    trade_date DATE NOT NULL,
    product VARCHAR(10) NOT NULL,           -- 'IF', 'au', 'cu' 等
    future_internal_id INTEGER,

    -- 五态分布（日终聚合）
    total_options INTEGER,                   -- 该品种下期权总数
    count_cr INTEGER,                        -- correct_rise 计数
    count_wr INTEGER,                        -- wrong_rise 计数
    count_cf INTEGER,                        -- correct_fall 计数
    count_wf INTEGER,                        -- wrong_fall 计数
    count_other INTEGER,                     -- other 计数

    -- 比率特征
    pct_cr FLOAT,                            -- count_cr / total
    pct_wr FLOAT,
    pct_cf FLOAT,
    pct_wf FLOAT,
    pct_other FLOAT,

    -- 趋势强度
    trend_strength FLOAT,                    -- |pct_cr - pct_cf| / (pct_cr + pct_cf)
    direction_skew FLOAT,                    -- (pct_cr + pct_wr) - (pct_cf + pct_wf)

    -- 波动率与流动性
    realized_vol_20d FLOAT,                  -- 20日实现波动率
    avg_option_volume FLOAT,                 -- 期权日均成交量
    futures_volume INTEGER,                  -- 期货标的成交量
    bid_ask_spread_avg FLOAT,                -- 平均买卖价差（%）

    -- 月份结构
    active_months INTEGER,                   -- 活跃月份数
    term_structure_slope FLOAT,              -- 近月-远月隐含波动率斜率

    -- 元数据
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    PRIMARY KEY (trade_date, product)
);
```

#### 17.2.3 state_transition_matrix

马尔可夫链转移矩阵，用于判断状态持续性。

```sql
CREATE TABLE IF NOT EXISTS state_transition_matrix (
    trade_date DATE NOT NULL,
    product VARCHAR(10) NOT NULL,

    -- 从状态X到状态Y的转移次数（日级）
    cr_to_cr INTEGER, cr_to_wr INTEGER, cr_to_cf INTEGER, cr_to_wf INTEGER, cr_to_other INTEGER,
    wr_to_cr INTEGER, wr_to_wr INTEGER, wr_to_cf INTEGER, wr_to_wf INTEGER, wr_to_other INTEGER,
    cf_to_cr INTEGER, cf_to_wr INTEGER, cf_to_cf INTEGER, cf_to_wf INTEGER, cf_to_other INTEGER,
    wf_to_cr INTEGER, wf_to_wr INTEGER, wf_to_cf INTEGER, wf_to_wf INTEGER, wf_to_other INTEGER,
```

### 17.4 回测引擎设计

#### 17.4.1 核心设计原则

1. **向量化优先**：10年数据 × 数十品种 × 数百期权，事件驱动太慢。使用 NumPy/Pandas 向量化计算日级信号和收益。
2. **事件驱动验证**：对关键日期（如黑天鹅事件）使用事件驱动精细回放，验证向量化结果。
3. **无锁回测**：回测专用 `WidthCacheBacktestService` 去除线程锁，纯单线程循环。
4. **参数网格缓存**：同一组参数在不同品种上的回测结果缓存，避免重复计算。
5. **全量数据常驻内存**：依托 64GB 物理内存，可将 10 年日级特征数据（约 2-3GB）全部加载至内存，避免磁盘 I/O 瓶颈；支持多品种并行回测（`multiprocessing` 池），将全品种单次回测时间从 60 秒压缩至 < 15 秒。

#### 17.4.2 向量化回测核心逻辑

```python
# research/backtest_engine.py
import numpy as np
import pandas as pd
from typing import Dict, List, Tuple
from dataclasses import dataclass

@dataclass(frozen=True)
class TierThresholds:
    """不可变参数对象，支持缓存哈希。"""
    TIER1_WILSON: float = 0.6
    TIER2_COVERAGE: float = 0.5
    TIER2_CORRECT_UP: float = 0.5
    TIER3_CORRECT_UP: float = 0.4


class VectorizedBacktestEngine:
    """
    向量化回测引擎：基于日级快照数据，快速评估不同 Tier 阈值的表现。
    核心假设：日级数据足够支撑 Tier 阈值评估（Tick级精度由状态分类保证）。
    """

    def __init__(self, snapshot_df: pd.DataFrame, feature_df: pd.DataFrame):
        """
        Args:
            snapshot_df: option_state_snapshot 表数据
            feature_df: product_feature_daily + tier_performance_daily 合并数据
        """
        self.snapshot_df = snapshot_df
        self.feature_df = feature_df

    @staticmethod
    def compute_tier_vectorized(
        df: pd.DataFrame,
        thresholds: TierThresholds
    ) -> np.ndarray:
        """
        向量化计算 Tier。

        输入 df 列: correct_up_pct, noise_ratio, coverage, wilson
        输出: tier (1/2/3/4)
        """
        t1 = (df['wilson'] >= thresholds.TIER1_WILSON) & (df['coverage'] >= 0.8)
        t2 = (df['coverage'] >= thresholds.TIER2_COVERAGE) & \
             (df['correct_up_pct'] >= thresholds.TIER2_CORRECT_UP)
        t3 = df['correct_up_pct'] >= thresholds.TIER3_CORRECT_UP

        return np.select([t1, t2, t3], [1, 2, 3], default=4)

    def run_backtest(
        self,
        thresholds: TierThresholds,
        product_list: List[str] = None,
        top_n_per_day: int = 5
    ) -> Dict[str, float]:
        """
        执行向量化回测，返回绩效指标。

        策略逻辑模拟：
        1. 每日收盘后，按当日五态数据计算各品种 Tier
        2. 过滤 Tier 4，在 Tier 1/2/3 中按 correct_up_pct 排序
        3. 取前 top_n_per_day 个品种，等权配置
        4. 持有到次日收盘，计算日收益
        """
        df = self.feature_df.copy()
        if product_list:
            df = df[df['product'].isin(product_list)]

        # 向量化分级
        df['tier'] = self.compute_tier_vectorized(df, thresholds)

        # 过滤 Tier 4
        tradable = df[df['tier'] <= 3].copy()
        if tradable.empty:
            return {'calmar_ratio': -1.0, 'annualized_return': 0.0}

        # 模拟交易：按日期分组，每日选 top_n
        def _daily_select(g: pd.DataFrame) -> pd.DataFrame:
            g = g.sort_values(['tier', 'correct_up_pct'], ascending=[True, False])
            return g.head(top_n_per_day)

        selected = tradable.groupby('trade_date', group_keys=False).apply(_daily_select)

        # 计算加权日收益（使用 feature_df 中预计算的 next_day_return_mean）
        daily_returns = selected.groupby('trade_date')['next_day_return_mean'].mean()

        if daily_returns.empty or daily_returns.std() == 0:
            return {'calmar_ratio': -1.0, 'annualized_return': 0.0}

        # 绩效指标
        total_return = (1 + daily_returns).prod() - 1
        n_days = len(daily_returns)
        annualized_return = (1 + total_return) ** (252 / n_days) - 1
        volatility = daily_returns.std() * np.sqrt(252)
        sharpe = annualized_return / volatility if volatility > 0 else 0

        cumulative = (1 + daily_returns).cumprod()
        running_max = cumulative.cummax()
        drawdown = (cumulative - running_max) / running_max
        max_drawdown = drawdown.min()
        calmar = annualized_return / abs(max_drawdown) if max_drawdown != 0 else 0

        return {
            'annualized_return': annualized_return,
            'volatility': volatility,
            'sharpe_ratio': sharpe,
            'max_drawdown': max_drawdown,
            'calmar_ratio': calmar,
            'hit_rate': (daily_returns > 0).mean(),
            'total_trades': len(selected),
        }
```

### 17.5 参数校准算法

#### 17.5.1 为什么用 Calmar 比率而非 Sharpe？

| 指标 | 适用场景 | 本项目选择 |
|------|---------|---------|
| Sharpe | 收益近似正态分布、低偏度 | ❌ 商品期权收益厚尾、左偏，Sharpe 会低估尾部风险 |
| Sortino | 只惩罚下行波动 | ⚠️ 比Sharpe好，但仍假设下行波动可对称度量 |
| **Calmar** | **关注最大回撤、策略生存能力** | ✅ **与实盘风控（保证金、回撤限制）直接挂钩** |

**补充指标**：
- `hit_rate`：日胜率，验证信号稳定性
- `profit_factor`：盈亏比，验证盈亏分布
- `param_drift_from_prev`：参数漂移，验证校准稳定性

#### 17.5.2 贝叶斯优化（Optuna）

```python
# research/tier_optimizer.py
import optuna
from typing import Optional

class TierThresholdOptimizer:
    """
    使用 Optuna TPE（Tree-structured Parzen Estimator）算法优化 Tier 阈值。
    相比网格搜索，TPE 在高维连续空间（4维阈值）中效率更高。
    """

    def __init__(
        self,
        engine: VectorizedBacktestEngine,
        product: str,
        prev_thresholds: Optional[TierThresholds] = None
    ):
        self.engine = engine
        self.product = product
        self.prev_thresholds = prev_thresholds

    def objective(self, trial: optuna.Trial) -> float:
        # 定义搜索空间（基于品种特征动态调整边界）
        t1_wilson = trial.suggest_float('TIER1_WILSON', 0.30, 0.90, step=0.05)
        t2_coverage = trial.suggest_float('TIER2_COVERAGE', 0.30, 0.80, step=0.05)
        t2_correct_up = trial.suggest_float('TIER2_CORRECT_UP', 0.30, 0.70, step=0.05)
        t3_correct_up = trial.suggest_float('TIER3_CORRECT_UP', 0.20, 0.60, step=0.05)

        thresholds = TierThresholds(
            TIER1_WILSON=t1_wilson,
            TIER2_COVERAGE=t2_coverage,
            TIER2_CORRECT_UP=t2_correct_up,
            TIER3_CORRECT_UP=t3_correct_up,
        )

        # 硬约束：阈值单调性（Tier 1 > Tier 2 > Tier 3）
        if not (t1_wilson > t2_correct_up > t3_correct_up):
            return -1e6

        # 硬约束：Tier 2 覆盖率不能低于 Tier 1 覆盖率门槛的一半
        if t2_coverage < 0.4:
            return -1e6

        metrics = self.engine.run_backtest(thresholds, product_list=[self.product])
        return metrics['calmar_ratio']

    def optimize(self, n_trials: int = 200) -> Tuple[TierThresholds, Dict]:
        study = optuna.create_study(
            study_name=f"tier_opt_{self.product}",
            direction='maximize',
            sampler=optuna.samplers.TPESampler(seed=42)
        )
        study.optimize(self.objective, n_trials=n_trials, show_progress_bar=True)

        best = study.best_params
        best_thresholds = TierThresholds(
            TIER1_WILSON=best['TIER1_WILSON'],
            TIER2_COVERAGE=best['TIER2_COVERAGE'],
            TIER2_CORRECT_UP=best['TIER2_CORRECT_UP'],
            TIER3_CORRECT_UP=best['TIER3_CORRECT_UP'],
        )

        return best_thresholds, study.best_trial.user_attrs
```

#### 17.5.3 参数稳定性约束

避免参数在相邻月份剧烈抖动，导致实盘频繁换仓：

```python
# research/constraint.py
import math

DRIFT_LIMIT = 0.20  # 相对变化不超过 20%

def apply_drift_constraint(
    new_thresholds: TierThresholds,
    prev_thresholds: TierThresholds,
    max_drift: float = DRIFT_LIMIT
) -> TierThresholds:
    """
    限制参数相对上期变化不超过 max_drift。
    若超出，则向边界收缩（不做外推）。
    """
    fields = ['TIER1_WILSON', 'TIER2_COVERAGE', 'TIER2_CORRECT_UP', 'TIER3_CORRECT_UP']
    result = {}
    for f in fields:
        new_val = getattr(new_thresholds, f)
        old_val = getattr(prev_thresholds, f)
        max_change = old_val * max_drift
        clamped = max(old_val - max_change, min(old_val + max_change, new_val))
        result[f] = round(clamped, 4)
    return TierThresholds(**result)
```

### 17.8 风险控制与回滚机制

1. **参数回滚**：
   - 若新参数在实盘首周 Calmar 为负，自动回滚到上期参数。
   - 回滚触发器：`CalibrationReportGenerator` 每日检查实盘收益，连续3日负收益即触发。

2. **A/B测试**：
   - 对流动性好的品种（如 IF、cu），`product_tier_thresholds.json` 支持 `experiment_group` 字段：
     ```json
     {"IF": {"TIER1_WILSON": 0.55, "experiment_group": "treatment"}}
     ```
   - 实盘同时运行对照组（默认阈值）和实验组（优化阈值），2周后对比 Calmar。

3. **数据质量检查**：
   - ETL 阶段强制检查：`_status_counts` 总和 > 0；`next_day_return` 绝对值 < 20%；无重复主键。
   - 若检查失败，当日数据标记为 `dirty`，不参与优化，但保留用于排查。

4. **计算资源上限**：
   - 回测引擎单次运行最大内存 16GB（64GB 总内存 ÷ 4 并行进程），超限时自动降级为低分辨率网格搜索（step 从 0.05 放宽到 0.1）。
   - Optuna `n_trials` 上限 3000，防止无限运行；默认 2000 trials 可充分利用 64GB 内存完成 9 品种 × 2000 trials 的搜索任务。

---

### 17.9 本章小结

1. **数据库设计**：6 张 DuckDB 表覆盖品种特征、Tier 绩效、状态转移、期权快照、参数校准记录、回测汇总，形成完整的量化研究数据底座。物理隔离（`quant_research.duckdb`）保证生产安全。
2. **回测引擎**：向量化优先（NumPy/Pandas）+ Walk-Forward 滚动窗口验证，单品种单次回测 < 5 秒，支撑 10 年数据的快速迭代与参数敏感性分析。
3. **参数校准**：以 **Calmar 比率**（年化收益/|最大回撤|）为目标函数，Optuna TPE 贝叶斯优化，配合 ±20% 漂移约束和阈值单调性硬约束，避免过拟合与参数抖动。
4. **渐进实施**：7 个阶段从基建到月度自动循环，预计 2-3 个月完成核心功能，之后进入自动运维模式。明确回滚、A/B 测试、资源上限三重保护，确保从研究到实盘的平稳过渡。

## 十八、极简市场共振排序与分层架构设计

> 本章回应策略核心目标——通过排序捕捉市场整体强势共振。在第十七章复杂方案的基础上，提出极简市场共振排序算法（MRR），并论证复杂方案与极简方案的分层并存架构。

---

### 18.1 复杂方案的专业性

必须首先承认：第十七章设计的复杂方案（五态统计 + Wilson 校正 + Tier 分级 + 月份权重 + 品种差异化 + 贝叶斯优化）在**统计学严谨性和机构级风控框架**上，确实是更专业的排序体系。

| 专业维度 | 复杂方案的能力 | 极简方案的缺失 |
|---------|--------------|----------------|
| **小样本校正** | Wilson score interval 处理低成交量期权的统计显著性 | 无校正，小样本品种可能误导 |
| **风险分层** | Tier 1/2/3/4 明确区分高置信 / 中置信 / 低置信 / 噪声 | 只有"选中 / 不选中"二元结果 |
| **数据质量惩罚** | coverage（有效月份占比）过滤数据稀疏品种 | 无质量过滤，可能选中统计基础弱的品种 |
| **噪声识别** | noise_ratio 量化"错误状态"占比，主动惩罚 | 只统计正确率，不区分噪声结构 |
| **品种差异化** | 支持按品种类别使用不同阈值 | 一刀切，不区分金融期货与农产品 |
| **参数自适应** | 贝叶斯优化 + Walk-Forward，动态适应市场变化 | 完全静态，无法适应市场制度变化 |

**结论**：如果目标是构建一个可解释、可审计、可规模化的机构级量化基础设施，复杂方案是更专业的选择。

---

### 18.2 复杂方案在当前阶段的局限性

然而，"专业"不等于"当前阶段的最优解"。复杂方案在当前策略生命周期中存在三个致命短板：

#### 18.2.1 参数过多 → 过拟合风险

复杂方案涉及的核心参数：
- 4 个 Tier 阈值（Wilson、Coverage、Correct_up ×2）
- 5 个月份权重
- 品种级差异化参数（9 个品种 × 4 个阈值 = 36 个）
- 衰减半衰期

**总计 45+ 个参数**，用 10 年数据优化，在商品期权这种低信噪比市场中极易过拟合。参数越多，历史回测的表现越好看，样本外失效的概率越高。

#### 18.2.2 信号衰减

复杂方案的决策链路：
```
原始五态 → Wilson校正 → coverage惩罚 → noise_ratio过滤 → Tier分级 → 月份加权 → 品种差异化 → 最终排序
```
每一层都是对信号的"加工"，但**每一层也可能过滤掉真正的 Alpha**。当策略连续 6 周 0 交易时（第十三章记录），问题可能不是"信号不存在"，而是"信号在层层过滤中被杀死了"。

#### 18.2.3 可解释性陷阱

当策略亏损时，复杂方案很难回答"为什么亏"：
- 是 Wilson 阈值设高了？
- 是 coverage 惩罚太重？
- 是某品种的差异化参数失效？
- 还是月份权重分配不当？

**调试 45 个相互耦合的参数，比调试 0 个参数难得多。**

---

### 18.3 极简市场共振排序算法（Market Resonance Rank, MRR）

#### 18.3.1 核心思想

共振 = 市场本身处于强势状态（多数品种趋势正确）+ 个别品种显著强于市场平均。

只需两个指标：
1. `accuracy`：品种自身的正确趋势占比
2. `resonance`：该品种相对全市场平均的强势倍数

#### 18.3.2 算法（4步）

```python
def market_resonance_rank(width_cache, products: List[str], top_n: int = 3):
    """
    极简共振排序。输入当日五态统计，输出应交易品种。
    """
    # Step 1: 计算全市场基准正确率
    total_cr = sum(width_cache._status_counts[p]['CR'] for p in products)
    total_cf = sum(width_cache._status_counts[p]['CF'] for p in products)
    total_all = sum(width_cache._status_counts[p]['Total'] for p in products)
    market_accuracy = (total_cr + total_cf) / total_all if total_all > 0 else 0

    # Step 2 & 3: 计算各品种正确率与共振倍数
    scores = []
    for product in products:
        counts = width_cache._status_counts[product]
        cr = counts.get('CR', 0)
        cf = counts.get('CF', 0)
        total = counts.get('Total', 1)
        
        accuracy = (cr + cf) / total
        resonance = accuracy / market_accuracy if market_accuracy > 0 else 0
        
        scores.append({
            'product': product,
            'accuracy': accuracy,
            'resonance': resonance,
            'market_accuracy': market_accuracy,
        })

    # Step 4: 过滤 + 排序
    # 只选自身正确率>50%（趋势正确）且共振倍数>1（强于市场平均）的品种
    strong = [s for s in scores if s['accuracy'] > 0.5 and s['resonance'] > 1.0]
    strong.sort(key=lambda x: x['resonance'], reverse=True)
    
    return strong[:top_n]
```

#### 18.3.3 为什么这样定义"共振"？

| 场景 | market_accuracy | 某品种 accuracy | resonance | 策略行为 |
|------|----------------|----------------|-----------|---------|
| 市场整体强势 | 0.55 | 0.70 | **1.27** | ✅ 选中，共振领头羊 |
| 市场整体强势 | 0.55 | 0.45 | 0.82 | ❌ 过滤（虽市场好，但自身弱） |
| 市场整体震荡 | 0.35 | 0.60 | **1.71** | ⚠️ 选中，但建议降仓（市场基础弱） |
| 市场整体弱势 | 0.30 | 0.35 | 1.17 | ❌ 过滤（accuracy<0.5） |

#### 18.3.4 与市场状态联动的仓位管理

```python
def position_size_by_market_state(market_accuracy: float) -> float:
    """根据市场整体状态决定仓位系数"""
    if market_accuracy >= 0.50:
        return 1.0        # 市场强势共振，满仓参与
    elif market_accuracy >= 0.40:
        return 0.5        # 市场温和，半仓
    else:
        return 0.0        # 市场弱势，空仓观望
```

### 18.4 两种方案的分层并存架构

与其二选一，不如**分层使用**：

```
┌─────────────────────────────────────────┐
│  Layer 1: 交易排序（MRR极简共振）          │
│  - 日常选品种，计算简单，响应快            │
│  - 直接产出 target list                   │

                   ↓

│  - 不修改排序，只做 "veto / 放行"         │
└─────────────────────────────────────────┘
                   ↓
┌─────────────────────────────────────────┐
│  Layer 3: 研究监控（第十七章数据库）        │

│  - 监控各品种参数漂移                      │

```

```python
    else:
        return self._complex_tier_sort(...)  # 现有逻辑
```

#### 18.4.2 建议的演进路径

1. **第一阶段（立即）**：启用 MRR 作为默认排序算法，复杂方案降级为风控过滤器
2. **第二阶段（1-2个月后）**：对比 MRR 与复杂方案的实盘 Calmar 比率
3. **第三阶段（数据充足后）**：若复杂方案经 Walk-Forward 验证显著优于 MRR（Calmar 差距 > 0.3），再考虑切换回复杂方案作为主排序器

### 18.5 本章小结

1. **复杂方案（第十七章）的专业性不可否认**：Wilson 校正、Tier 分级、品种差异化在统计学上是严谨的，适合作为机构级风控底座和研究基础设施。但它有 45+ 个参数，在低信噪比商品期权市场中过拟合风险极高，且信号层层衰减可能导致连续无交易。

2. **MRR 极简算法直击"共振"核心**：**零参数、O(N) 复杂度、5 步完成**。核心公式 `resonance = accuracy / market_avg` 的直觉是——"比市场平均更正确的品种，就是共振中的领头羊"。算法已修正为正确读取 `_status_counts[fid][month][opt_type]` 四层嵌套结构，并仅计算方向匹配的期权类型（CALL/PUT）。

3. **MRR 与 Tier 框架的等价性**：`accuracy` 等价于不加权的 `correct_up_pct`，`total_signals >= 5` 等价于 Wilson 校正的硬阈值版本，`resonance` 等价于用相对排名替代绝对 Tier 分级。两者在数学上高度相关，但 MRR 的自适应特性（跟随市场平均值变化）更适合当前策略周期。

4. **最优架构是分层并存**：MRR 负责日常交易排序（Layer 1），复杂方案负责风控过滤与研究监控（Layer 2+3）。两者不是替代关系，而是**战术（赚钱）与战略（保命+积累数据）**的分工。

## 十九、对第十八章 MRR 极简方案的批判式吸收（2026-06-22）

> 第十八章提出了极简市场共振排序算法（MRR），并以分层架构作为复杂方案与极简方案的折中方案。本章从三个基本事实出发——（1）复杂方案实际参数仅 9 个而非 45+、（2）用户拥有 10 年历史数据、（3）策略目标为量化研究时期的长期标准方案——对第十八章进行逐条批判，并重新定位 MRR 的角色。

---

### 19.1 事实核查：第十八章的三处数据偏差

#### 19.1.1 "45+ 个参数"——被严重夸大

第十八章 18.2.1 称复杂方案涉及"45+ 个参数"，具体拆解为 4 个 Tier 阈值 + 5 个月份权重 + 36 个品种差异化参数 + 衰减半衰期。

**实际代码核查**（`width_cache_query_mixin.py:26-31`）：

```python
MONTH_WEIGHTS_5 = (0.35, 0.25, 0.20, 0.12, 0.08)  # 5 个值
TIER1_WILSON_THRESHOLD = 0.6                         # 1 个值
TIER2_COVERAGE_THRESHOLD = 0.5                       # 1 个值
TIER2_CORRECT_UP_THRESHOLD = 0.5                     # 1 个值
TIER3_CORRECT_UP_THRESHOLD = 0.4                     # 1 个值
```

**当前实际可调参数：4 个 Tier 阈值 + 5 个月份权重 = 9 个参数**。品种差异化（9 品种 × 4 阈值 = 36 个）是第十七章 Phase 6 的**未来扩展目标**，不是当前状态。衰减半衰期也尚未实现。

第十八章用"45+"来论证"过拟合风险极高"，但 9 个参数对于 10 年历史数据（约 2500 个交易日 × 20+ 品种 = 50,000+ 个样本日）而言，**参数-样本比约为 1:5500，过拟合风险在可控范围内**。

#### 19.1.2 "信号层层衰减"——缺乏实证

第十八章 18.2.2 称信号在 8 层处理中"层层衰减"，并以第十三章记录的"连续 6 周 0 交易"为证据。

**但第十三章记录的是修复前的状态**（BUG-1 到 BUG-26 尚未修复）。修复后（第十五章验证），`_classify_status` 正确运行、`determine_tier` 正确过滤、`_get_scoring_months` 正确选择月份。**修复后的系统是否仍然存在信号衰减问题，目前没有任何实证数据**。

"信号衰减"是一个需要数据验证的假设，不是可以用于论证的事实。

#### 19.1.3 MRR 的"零参数"——被隐藏的参数

第十八章称 MRR 是"零参数"算法，但实际存在三个隐式参数：

| 隐式参数 | 位置 | 值 | 影响 |
|---------|------|-----|------|
| 样本量门槛 | `total_signals >= 5` | 5 | 决定了哪些品种被排除；小样本品种直接出局 |
| 绝对正确率硬门槛 | `accuracy > 0.30` | 0.30 | 决定了市场整体走弱时是否交易；这个值从哪里来？ |
| 共振筛选门槛 | `accuracy > market_avg` | 动态 | 决定了"相对优秀"的标准；但 `market_avg` 本身不可控 |

**这三个参数实际上定义了 MRR 的整个行为边界**，但它们被隐藏在代码逻辑中，而非像 Tier 阈值那样以类常量的形式显式声明。当 MRR 表现不佳时，你无法用 Optuna 优化 `total_signals >= 5`，因为它不是一个可调参数——它是一个硬编码的魔法数字。

---

### 19.2 核心批判：10 年历史数据改变了游戏规则

第十八章的核心叙事是"复杂方案在样本不足时过拟合，MRR 更适合当前阶段"。但这是基于**实盘运行时分钟级样本量**的假设。用户明确声明：

> "现在策略处于量化研究时期，准备了10年历史数据进行量化"

这意味着：

**论据 1：样本量不再是瓶颈**

- 10 年 ≈ 2500 个交易日
- 20+ 个品种 × 2500 个交易日 = 50,000+ 个品种-日观测
- 每品种-日有数百个期权状态，非 other 占比约 1-2%，即每品种-日约 10-30 个有效状态
- 总有效状态数：50,000 × 20 = 1,000,000+ 个

**100 万+ 的有效状态，校准 9 个参数，参数-样本比超过 1:100,000**。在这个量级下，过拟合不再是主要风险。

**论据 2：复杂方案的设计初衷就是大样本**

第十七章设计的整个基础设施——`quant_research.duckdb` 的 6 张表、向量化回测引擎、Walk-Forward 验证——**都是为 10 年历史数据回测驱动的参数校准而设计的**。如果最终选择 MRR（零参数），第十七章的大部分基础设施将失去存在意义，因为 MRR 不需要任何数据库、不需要任何回测、不需要任何校准。

**论据 3：量化研究时期的目标是"发现"，不是"简化"**

量化研究时期的核心任务是：**用历史数据验证哪些因子（Wilson、Coverage、Noise_ratio、月份权重）对期权五态排序有预测力，以及各因子的最优权重组合**。MRR 跳过了这个"发现"过程，直接用一个简单公式替代了所有因子。这违背了量化研究的基本方法论——**在跳过因子之前，先验证因子**。

---

### 19.3 MRR 方法的固有缺陷

#### 19.3.1 相对标准的致命弱点

第十八章将 MRR 的"相对标准"（比市场平均好就行）描述为优势。但它在以下场景中完全失效：

**场景：全市场温和偏弱（最危险的情况）**

```
品种 A: accuracy = 0.33  (33% 正确率)
品种 B: accuracy = 0.31  (31% 正确率)
品种 C: accuracy = 0.29  (29% 正确率)
market_avg = 0.31

MRR 结果：
- 品种 A: resonance = 1.06 (高于 market_avg)
- 0.33 > 0.30 硬门槛 → 通过
- 最终结果：选中品种 A，但 33% 正确率意味着每 3 次交易就有 2 次错误
```

**Tier 框架的对比**：在同样的 33% 正确率下，`correct_up_pct = 0.33 < TIER3_CORRECT_UP_THRESHOLD = 0.4`，Tier 框架将其归为 Tier 4，不交易。**绝对标准在此场景下保护了资金**。

#### 19.3.2 信息损失：MRR 丢弃了三个关键维度

MRR 用单一 `accuracy` 指标替代了现有框架的 8 个指标。但被丢弃的指标中，有三个是风险控制的关键：

| 被丢弃的指标 | 为什么重要 | MRR 如何替代 |
|------------|----------|-------------|
| **Wilson 置信下界** | 小样本品种的 `accuracy` 极不稳定。5 个信号中 4 个正确 = 80% 正确率，但 Wilson 下界仅 0.38。MRR 的 `total_signals >= 5` 不区分 5 个信号和 500 个信号。 | 无法替代。`total_signals >= 5` 是二元开关，Wilson 是连续函数。 |
| **Coverage（有效月份占比）** | 某品种仅有 1 个主力月份有数据 ≠ 有 5 个月份都有数据。前者可能是临时流动性事件，后者是稳定的统计特征。 | 无法替代。MRR 不区分月份覆盖度。 |
| **Noise_ratio（错误信号占比）** | 品种 A: CR=50%, WR=50% → accuracy=50%，但 noise_ratio=50%。品种 B: CR=50%, WR=0%, CF=50% → accuracy=50%，但 noise_ratio=0%。两者 accuracy 相同，但噪声结构完全不同。 | 部分替代。`accuracy = 1 - noise_ratio` 仅当 `CF + WF = 0` 时成立。 |

#### 19.3.3 MRR 的仓位联动逻辑存在循环论证

第十八章的 `_resonance_lots` 根据 `market_avg` 决定仓位：

```python
if market_avg >= 0.50: return 2  # 满仓
elif market_avg >= 0.40: return 1  # 半仓
else: return 0  # 空仓
```

**问题**：`market_avg` 是品种正确率的平均值，而品种正确率本身又受到 `_future_rising` 方向判断的影响。如果 `_future_rising` 的判断本身有系统性偏差，`market_avg` 会系统性高估或低估，导致仓位决策被污染。

Tier 框架不存在这个问题，因为 Tier 分级是**独立于 `_future_rising` 的判断**——它只依据五态统计的客观分布，不依赖方向判断的质量。

---

### 19.4 MRR 的正确定位：研究基线，而非生产排序器

尽管上述批判集中在 MRR 的缺陷上，MRR 在量化研究体系中有一个**不可替代的价值**：作为基线模型（baseline）。

#### 19.4.1 基线模型的价值

在量化研究中，任何复杂模型都必须回答一个基本问题：**"比简单模型好多少？"** 如果复杂方案花费了大量计算资源优化出的参数，最终表现不如一个零参数的简单公式，那复杂方案就不值得部署。

MRR 正是这个"简单公式"——它代表了"不做任何优化"的基准表现。

#### 19.4.2 正确的分层架构（反转第十八章）

第十八章建议的分层架构是：

```
Layer 1: MRR → 交易排序
Layer 2: 复杂方案 → 风控过滤
Layer 3: 研究监控
```

**正确的分层架构应该是：**

```
┌─────────────────────────────────────────────────┐
│  Layer 1: 研究层（离线，10年历史数据）              │
│  ┌──────────────────┐  ┌───────────────────────┐ │
│  │ MRR Baseline     │  │ 复杂方案（9参数校准）    │ │
│  │ (零参数，直接计算) │  │ (Optuna + Walk-Forward) │ │
│  └──────────────────┘  └───────────────────────┘ │
│              ↓ 对比 Calmar 比率 ↓                 │
│     复杂方案胜出 → 部署复杂方案                     │
│     MRR 胜出 → 市场太嘈杂，复杂方案过拟合，用 MRR    │
└─────────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────┐
│  Layer 2: 生产层（在线，实盘交易）                   │
│  部署研究层胜出的排序器                             │
│  + 第十七章的热加载机制（config JSON）              │
└─────────────────────────────────────────────────┘
                      ↓
┌─────────────────────────────────────────────────┐
│  Layer 3: 监控层（在线，实盘监控）                   │
│  品种特征数据库（quant_research.duckdb）            │
│  日终 ETL 写入                                    │
│  月度 Walk-Forward 重校准                         │
│  参数漂移检测                                     │
└─────────────────────────────────────────────────┘
```

#### 19.4.3 研究流程

```
Phase A: 建立 Baseline（1 天）
  → 用 MRR 在 10 年历史数据上运行向量化回测
  → 记录 Calmar_baseline

Phase B: 校准复杂方案（第十七章 Phase 4-5，1-2 周）
  → 对每个品种，用 Optuna 搜索最优 Tier 阈值
  → Walk-Forward 验证（6 个月训练 / 2 个月测试）
  → 记录 Calmar_complex

Phase C: 对比决策（1 天）
  → 如果 Calmar_complex > Calmar_baseline + 0.3：
    部署复杂方案，品种差异化参数写入 JSON
  → 如果 Calmar_complex <= Calmar_baseline + 0.3：
    部署 MRR，继续积累数据，3 个月后重新校准
  → 如果 Calmar_complex < Calmar_baseline：
    复杂方案过拟合，直接用 MRR
```

**0.3 的 Calmar 差距阈值**：Calmar 比率 = 年化收益 / |最大回撤|。0.3 的差距意味着复杂方案要么年化收益高出 30%，要么最大回撤低 30%。低于这个阈值，复杂方案的增量价值不值得其维护成本。

---

### 19.5 MRR 中值得保留的两个思想

尽管本章对 MRR 进行了系统批判，第十八章中仍有两个思想值得保留：

**思想 1：共振倍数（resonance）作为辅助监控指标**

`resonance = accuracy / market_avg` 本身是一个有价值的**市场状态监控指标**，不应被丢弃。建议在第十七章的 `product_feature_daily` 表中增加 `resonance` 列，作为品种特征的一个维度，但不直接用于排序决策。

**思想 2：市场平均正确率作为仓位调节因子**

`market_avg` 作为全市场趋势强度的代理变量，可以用于调节整体仓位上限（而非像 MRR 那样调节单个品种的仓位）。例如：
- `market_avg >= 0.50`：允许最大仓位（当前配置的 100%）
- `market_avg >= 0.40`：允许最大仓位的 70%
- `market_avg < 0.40`：允许最大仓位的 50%

这个调节在 Tier 框架的**入口**（而非排序内部）执行，不影响排序逻辑本身的正确性。

### 19.6 本章小结

1. **第十八章的事实基础需要修正**：当前复杂方案仅有 9 个参数（4 个 Tier 阈值 + 5 个月份权重），而非 45+。`market_avg` 和 `total_signals >= 5` 在 MRR 中是隐式参数，MRR 并非真正的"零参数"。

2. **10 年历史数据是决定性变量**：100 万+ 的有效状态，9 个参数，过拟合风险极低。第十七章的全部基础设施（DuckDB、向量化回测、Walk-Forward）都是为这个场景设计的，选择 MRR 将使这些基础设施失去存在意义。

3. **MRR 的相对标准有致命缺陷**：在全市场温和偏弱时（如 market_avg = 0.31），MRR 可能选中正确率仅 33% 的品种，而 Tier 框架的绝对标准会将其归为 Tier 4 不交易。MRR 的仓位联动逻辑存在循环论证风险。

4. **MRR 的正确定位是研究基线**：作为量化研究的 baseline 模型，与复杂方案在 10 年历史数据上对比 Calmar 比率。胜出者进入生产层。这不是"二选一"，而是"先对比，后选择"的科学决策流程。

5. **最终建议**：立即启动第十七章的研究基础设施建设和第十八章 MRR 的 baseline 回测，**并行推进**。研究流程完成后（Phase A/B/C），基于数据（而非直觉）决定部署哪个排序器。**数据驱动决策，是量化研究的基本纪律。**

### 19.7 对第十九章自身的元批判（自我反思）

第十九章的上述批判看似逻辑严密，但在批判他者的同时，自身也存在值得审视的偏见与盲点。

#### 19.7.1 第十九章是否陷入了"复杂方案原教旨主义"？

第十九章的核心论点是："10 年数据 + 100 万状态 + 9 个参数 = 过拟合风险极低，因此复杂方案应当执行"。这一推导隐含了一个未被验证的假设：

> **"参数数量少 + 样本量大 = 策略有效"**

这是统计学上的**混淆相关与因果**。样本量大只能保证参数估计的方差小（估计精确），但不能保证参数估计的期望值接近真实值（估计准确）。如果 Tier 分级的基本假设（"correct_up_pct 越高 → 次日收益越高"）在商品期权市场中本身不成立，那么无论有多少数据、多少个 Walk-Forward 窗口，优化出来的阈值都只是对错误假设的精确拟合。

第十九章没有回答一个更根本的问题：**Tier 分级框架的经济学基础是否在商品期权市场中被验证过？** 在股票多因子领域，价值因子、动量因子的有效性有数十年文献支撑；但在商品期权五态统计领域，"correct_rise 占比高的品种次日会涨"这一假设，本身就需要用 MRR 这样的极简框架先做验证。

#### 19.7.2 第十九章对 MRR 的批判是否"稻草人论证"？

第十九章批判 MRR 时，使用了以下场景：

```
market_avg = 0.31
品种 A: accuracy = 0.33 → 被选中，但 33% 正确率意味着每 3 次交易 2 次错误
```

这个场景确实是 MRR 的弱点。但第十九章**没有同样苛刻地审视复杂方案的弱点**。例如：

- 当某品种长期处于 Tier 4（因 coverage 不足 0.3），复杂方案永远不对其交易，即使该品种正在经历一轮历史性行情
- 当 Wilson 校正过于保守时，高波动品种（如原油期权）的 correct_up_pct 可能永远达不到 Tier 1 门槛，导致策略系统性错过高波品种的行情
- 当品种差异化参数在 Walk-Forward 测试中表现"稳健"，但测试期恰好是市场低波动期，参数在高波动期会失效

第十九章对 MRR 的批判是**显微镜级别的**，对复杂方案的批判是**望远镜级别的**。这种不对称性削弱了批判的客观性。

#### 19.7.3 "研究层 → 生产层 → 监控层"架构是否过于理想化？

第十九章提出的三层架构（19.4.2）是一个教科书式的量化研究流程：先离线回测对比，再部署胜者，最后监控漂移。但在工程实践中，这个流程忽略了以下摩擦：

1. **数据清洗成本**：10 年历史数据不是"即插即用"的。期权合约的到期换月、行权价调整、分红除权、品种上市/退市，都需要大量的数据预处理。Phase A（MRR baseline）预估"1 天"完成，实际可能需要 2-3 周。
2. **回测与实盘的 gap**：向量化回测假设"日终收盘价成交"，但实盘可能面临流动性枯竭、开盘跳空、涨跌停无法成交等问题。Calmar 比率在回测中高估 30-50% 是常见现象。
3. **人员与维护成本**：复杂方案需要专人维护参数校准管道、监控 Walk-Forward 结果、处理品种阈值漂移。MRR 几乎不需要维护。如果团队只有 1-2 人，复杂方案的隐性成本可能吞噬其超额收益。

#### 19.7.4 "0.3 的 Calmar 差距阈值"是否武断？

第十九章在 19.4.3 中提出：

> "如果 Calmar_complex > Calmar_baseline + 0.3，部署复杂方案"

0.3 的阈值没有任何统计检验支撑。为什么是 0.3 而不是 0.2 或 0.5？这个阈值的设定本身就带有主观性，与第十九章批判的"专家拍脑袋"没有本质区别。

更科学的做法是：使用**假设检验**（如 Diebold-Mariano 检验）判断复杂方案与 MRR 的预测能力差异是否具有统计显著性，而非简单设定一个绝对阈值。

#### 19.7.5 第十九章是否存在"分析瘫痪"风险？

第十九章的批判链条是：

```
批判第十八章 → 提出修正 → 对修正进行自我批判 → 发现修正也有问题 → ...
```

这种无限递归的批判可能导致**分析瘫痪**（paralysis by analysis）：永远在研究，永远不部署。量化研究的核心目标不是写出完美的报告，而是**在可接受的风险下做出决策并执行**。

第十九章的最终建议"并行推进、数据驱动决策"是正确的，但它没有回答一个执行层面的问题：**如果 Phase A/B/C 完成后，复杂方案的 Calmar 仅比 MRR 高 0.15（低于 0.3 阈值），团队应该做什么？** 是坚持用 MRR 等待更好的数据？还是降低阈值部署复杂方案？还是同时运行两个方案各 50% 仓位？第十九章没有给出可操作的决策树。

---

### 19.8 元批判后的综合结论

经过对第十九章的自我批判，报告的最终立场应当是：

1. **承认认知边界**：无论是第十八章的"MRR 至上"，还是第十九章的"复杂方案应当执行"，都是基于不完备信息的判断。在 10 年历史数据实际跑完 baseline 对比之前，任何"应当"都是假说而非结论。

2. **接受不对称性**：复杂方案的验证成本高（需要建库、回测、校准、Walk-Forward），但潜在收益也高；MRR 的验证成本低（1 天即可跑完），但上限也低。**两者不是竞争关系，而是探索与利用（exploration vs. exploitation）的权衡。**

3. **建立决策触发器，而非决策结论**：
   - **触发器 1**：MRR 的 10 年 baseline Calmar < 0.3 → 市场本身太嘈杂，任何排序方案都难以盈利，策略需要重新设计
   - **触发器 2**：MRR baseline Calmar ≥ 0.3，但复杂方案仅比 MRR 高 < 0.2 → 部署 MRR，复杂方案降级为研究项目继续迭代
   - **触发器 3**：复杂方案比 MRR 高 ≥ 0.2，且 Walk-Forward 测试期 Calmar 标准差 < 0.3 → 部署复杂方案，MRR 作为监控基准
   - **触发器 4**：复杂方案 Walk-Forward 测试期出现 Calmar 为负的窗口 → 复杂方案过拟合，回退到触发器 2

4. **停止无限递归的批判**：第十九章的元批判已经揭示了两个方案的局限。继续撰写"第二十章：对第十九章元批判的再批判"只会陷入文字游戏。**报告到此应当收束，将笔墨转化为代码——用实际回测结果终结争论。**

---

### 19.9 本章最终结论（报告收束）

1. **第十八章过度简化了问题**：MRR 不是"零参数"，其隐式参数（market_avg 的计算方式、total_signals >= 5 的阈值）同样有主观性；在全市场温和偏弱时，MRR 的相对标准可能选中正确率仅 33% 的品种，存在资金安全风险。

2. **第十九章过度复杂化了问题**："100 万状态 + 9 参数 = 过拟合风险低"的推导混淆了估计精度与假设正确性；对 MRR 的批判存在稻草人论证；0.3 的 Calmar 阈值缺乏统计检验；理想化的三层架构忽略了工程摩擦。

3. **元批判揭示了三方共同的盲区**：三章（十五、十八、十九）都没有正面回答一个根本问题——**"correct_up_pct 越高 → 次日收益越高"这一假设本身是否成立？** 这是所有排序算法的经济学基础。如果这个假设不成立，无论是 MRR 还是复杂方案，都只是对随机性的精致拟合。

4. **唯一正确的下一步是执行**：
   - 用 3 天时间，对 10 年数据跑两个最简单的测试：
     - **测试 A**：按 `correct_up_pct` 分组（>0.6 / 0.5-0.6 / 0.4-0.5 / <0.4），计算每组的次日平均收益。如果 >0.6 组的收益不显著高于 <0.4 组，整个五态排序框架的经济学基础崩塌。
     - **测试 B**：计算 `market_accuracy` 与次日全市场平均收益的相关性。如果相关性不显著，"市场整体共振"的概念需要重新定义。
   - 这两个测试的结果，将决定是否需要继续投资第十七章的数据库建设和第十八章/第十九章的争论。

5. **报告的使命已经完成**：从 BUG 排查 → 框架修复 → 批判吸收 → 长期方案 → 极简替代 → 元批判，全链路已经梳理清楚。后续的工作不再是写文档，而是写代码、跑回测、看结果。**让数据说话。**
