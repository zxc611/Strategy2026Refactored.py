### P0-01：tick处理每次分配新dict（strategy_tick_handler.py）

- **文件**: `C:\Users\xu\AppData\Roaming\InfiniTrader_SimulationX64\pyStrategy\demo\ali2026v3_trading\strategy_tick_handler.py`

## 维度二：串行化瓶颈

### P0-03：全局锁串行化tick处理

- **文件**: `C:\Users\xu\AppData\Roaming\InfiniTrader_SimulationX64\pyStrategy\demo\ali2026v3_trading\risk_service.py`
- **位置**: L244 - `check_before_trade()` 方法
- **反模式代码**:
  ```python
  with self._lock:
      cl_result = self._check_consecutive_loss_protection(signal)
      if cl_result.is_block:
          return self._record_result(cl_result)
      status_result = self._check_strategy_status()
      ...
  ```
- **分析**: `_lock` 是一个 `threading.RLock()`（实例级别）。`check_before_trade()` 是每笔交易前串行调用的入口，其内部多个检查全部在全局锁保护下串行执行。当多个信号并发到达时，形成串行化瓶颈。虽然风控检查理论上需要串行化，但锁的粒度过粗（覆盖了整个检查链路），且锁争用可能导致信号处理延迟累积。
- **影响**: 在高并发场景下（数百个合约同时触发信号），全局锁形成硬串行化点，延迟与并发数成正比。
- **建议**: 使用读写锁（`shared_utils.RingBuffer` 已有类似设计）或分片锁减少争用；将无状态检查移出临界区。

## 维度三：缓存只增不删泄漏

### P0-05：信号去重缓存无上限清理

- **文件**: `C:\Users\xu\AppData\Roaming\InfiniTrader_SimulationX64\pyStrategy\demo\ali2026v3_trading\signal_service.py`
- **位置**: L119, L221-L233
- **反模式代码**:
  ```python
  self._signal_dedup_set[dedup_key] = dedup_now
  if len(self._signal_dedup_set) > 500:
      expired_keys = [k for k, v in self._signal_dedup_set.items() if dedup_now - v > self._signal_dedup_ttl]
      for k in expired_keys:
          del self._signal_dedup_set[k]
  ```
- **分析**: `_signal_dedup_set` 使用条件清理（超过500条才触发过期清理），但即便在500条限制内，仍可能内存泄漏——如果去重TTL很短但信号生成频率很高，超过500条后触发的 `expired_keys` 推导式（list comprehension）每次创建新list。且没有上限保护，极端情况下可能超过500条限制。
- **影响**: 长期运行时，每个新品种/新信号类型都会在 `_signal_dedup_set` 中留下条目。TTL过期后手动 `del` 虽释放条目，但整体设计上仍是"延迟清理"模式，在清理间隙存在内存峰值。
- **建议**: 使用LRU缓存的 `OrderedDict` 或 `cachetools.TTLCache`；或将清理策略改为定时器定期清理而非条件触发。
