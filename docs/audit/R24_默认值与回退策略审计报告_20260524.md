## 二、P0级发现（资金风险/错误交易决策）

### P0-01：YAML与Code回退在`correct_trending.max_risk_ratio`存在2.67倍差异

- **文件**：`参数池/state_param_sets.yaml:6` vs `config_params.py:690`
- **YAML值**：`max_risk_ratio: 0.8`（激进：80%权益风险暴露）
- **Code回退值**：`max_risk_ratio: 0.3`（保守：30%）
- **差异因子**：2.67倍
- **审计标准**：`parameter_attribute_matrix.yaml:303` 中 `correct_trending.max_risk_ratio=0.8`，确认YAML值为设计意图
- **后果**：系统正常加载YAML时，`correct_trending`状态使用80%风险敞口（等同于满仓），若`state_param_sets.yaml`因任何原因加载失败（磁盘故障、编码错误），系统静默回退到30%，导致仓位瞬间缩减至37.5%。反之，如果某人修改YAML降低了该值但未重启系统，旧缓存中的0.8将持续生效，形成"代码回退≠YAML权威值"的认知错位。
- **回退链**：`get_default_state_param_sets()` → YAML加载 → 失败时 `logging.debug` 级别记录 → **静默**回退到硬编码值（不同于YAML）
- **建议**：① 提升YAML加载失败日志级别至`logging.error`并触发告警；② 将Code回退值同步到YAML值（0.8），消除两套默认体系；③ 启动时校验`get_cached_params()`中的`state_param_sets`是否与YAML一致

### P0-02：`DEFAULT_PARAM_TABLE.signal_cooldown_sec=0.0` — 系统级信号冷却为零

- **文件**：`config_params.py:110`
- **代码**：`"signal_cooldown_sec": 0.0`
- **风险**：如果JSON外部配置文件缺失（`params_default.json`不存在），`get_cached_params()`回退到`DEFAULT_PARAM_TABLE`，此时`signal_cooldown_sec=0.0`。虽然`RiskService._get_signal_cooldown()`有自己的默认60.0（`risk_service.py:1134`），但如果有任何代码路径直接读取`params["signal_cooldown_sec"]`而绕过`safe_get_float`，将导致零冷却下的高频连续开仓。
- **审计标准**：V7手册 S1默认60s（因`risk_service.py:1134`和`task_scheduler.py:356`均默认60.0）
- **三系统不一致**：`DEFAULT_PARAM_TABLE=0.0` / `RiskService=60.0` / `box_detector=120.0` / `task_scheduler PARAM_DEFAULTS=60.0` / `task_scheduler RESONANCE_TIME_DEFAULTS=60.0`
- **get()陷阱**：`task_scheduler.py:1577` 使用 `params.get("signal_cooldown_sec", 0.0)` — 如果params来自`DEFAULT_PARAM_TABLE`，则默认值0.0被使用，而非期望的60.0
- **建议**：① 将`DEFAULT_PARAM_TABLE`中`signal_cooldown_sec`从0.0改为60.0；② 审计所有`dict.get("signal_cooldown_sec", X)`调用，确保default不为0.0

### P0-03：`close_stop_loss_ratio` CODE回退=0.5 vs V7手册S1=0.3 — 止损宽度偏差67%

- **文件**：`config_params.py:198` vs V7手册 section 21.1 S1参数默认表
- **DEFAULT_PARAM_TABLE**：`close_stop_loss_ratio: 0.5`（50%亏损才止损）
- **V7手册S1默认**：`close_stop_loss_ratio: 0.3`（30%止损）
- **后果**：当JSON配置缺失，系统使用50%止损阈值。在期权市场中，50%的单笔亏损意味着几乎翻倍的杠杆损失，远超标准30%。
- **task_scheduler中的正确默认**：`PARAM_DEFAULTS_HFT.close_stop_loss_ratio=0.3`（`task_scheduler.py:501`），说明回测系统使用正确默认，但生产配置表不一致
- **建议**：将`DEFAULT_PARAM_TABLE.close_stop_loss_ratio`从0.5改为0.3

### P0-04：YAML `correct_trending.close_stop_loss_ratio=0.05` 极端紧缩止损

- **文件**：`参数池/state_param_sets.yaml:5`
- **值**：`close_stop_loss_ratio: 0.05`（5%止损）
- **Code回退值**：`close_stop_loss_ratio: 0.4`（40%止损）
- **差异因子**：8倍
- **后果**：YAML正常加载时，`correct_trending`状态使用5%止损。期权市场5%的价格波动极为常见（1个行权价档位可能对应2-4%波动），这将导致大量正常仓位被过早止损。配合该状态的`max_risk_ratio=0.8`（满仓），形成"满仓+极紧止损"的不稳定组合。
- **审计标准**：`parameter_attribute_matrix.yaml:281`中`correct_trending.close_stop_loss_ratio=0.4`（而非0.05），说明YAML的0.05与属性矩阵矛盾
- **建议**：① 排查YAML中的0.05是否为笔误；② 如果确认为设计意图，需在文档中明确说明理由（如仅适用于特定品种）

### P0-05：`correct_trending`状态 Code回退缺乏`correct_trending_defensive`等YAML新状态

- **文件**：`config_params.py:657-712` vs `参数池/state_param_sets.yaml`
- **YAML包含的状态**：`correct_trending`, `correct_trending_defensive`, `incorrect_reversal`, `other`, `CORRECT_RESONANCE`, `CORRECT_DIVERGENCE`, `INCORRECT_OVERSHOOT`, `shadow_engine`, `capital_route`, `box_extreme` 等30+个状态
- **Code回退仅包含**：`correct_trending`, `incorrect_reversal`, `other`
- **后果**：YAML加载失败时，除3个核心状态外的所有状态参数丢失。`shadow_engine`（含`alpha_window_days`）、`capital_route`（含资金路由权重）、`box_extreme`（箱体极值参数）等关键配置全部缺失。系统将以空字典运行这些模块，导致对应功能退化到模块各自的类默认值（可能与YAML不一致）。
- **建议**：将Code回退扩展到覆盖全部YAML关键状态，或至少对YAML加载失败触发严重告警

### 维度4：get(key, default)的default陷阱

#### P1-15：`task_scheduler.py:1577` 使用 `params.get("signal_cooldown_sec", 0.0)` — 0.0作为fallback

### 维度5：回退链完整性

- **文件**：`config_params.py:224-242`

### 维度4：get(key, default)的default陷阱

#### P2-27：`params_service.py:1099-1106` 嵌套default回退可能返回空列表/空字典

- **文件**：`params_service.py:1099-1106`
- **代码**：
  ```python
  val = self._params.get(key, default or [])
  return list(val) if isinstance(val, (list, tuple)) else (default or [])
  ```
- **问题**：当`default=None`时，`default or []`返回`[]`。这意味着调用者显式传入`default=None`（表示"无默认值，期望None"）被覆盖为空列表
- **建议**：使用`default if default is not None else []`区分"未传默认"和"显式None"

#### P2-28：`width_cache.py:591,625` 使用 `getattr(params, 'month_mapping', {}) or {}`

- **文件**：`width_cache.py:591, 625`
- **代码**：`month_mapping = getattr(params, 'month_mapping', {}) or {}`
- **问题**：如果`month_mapping`存在但为空字典`{}`，则`{} or {}`仍返回`{}`（OK）。但如果`month_mapping`是一个falsy值（如`None`），`None or {}`返回`{}`。这里的`or {}`是冗余保护
- **建议**：保留现有逻辑，但添加注释说明防御意图

#### P2-29：`governance_engine.py:504` 使用 `self._config = config or {}`

- **文件**：`governance_engine.py:504`
- **代码**：`self._config = config or {}`
- **问题**：如果传入`config={}`（空字典），`{} or {}`返回`{}`。但如果传入`config=0`或`config=False`，也会被替换为`{}`。虽然实际场景中config总是dict或None，但类型不安全
- **建议**：改为`self._config = config if isinstance(config, dict) else {}`

## 五、全局性发现

### G-01：缺少中心化默认值常量体系

当前系统存在4层默认值来源，但缺乏统一管理：

| 层级 | 位置 | 优先级 | 问题 |
|:---|:---|:---|:---|
| L1 环境变量 | `PARAM_OVERRIDE_*` | 最高 | 无文档化的覆盖变量列表 |
| L2 YAML配置 | `state_param_sets.yaml` | 高 | 与Code回退值偏差大 |
| L3 JSON配置 | `params_default.json` | 中 | 可能不存在 |
| L4 代码硬编码 | `DEFAULT_PARAM_TABLE` | 低 | 多模块各自定义 |

**建议**：建立`defaults_constants.py`模块，将所有中心化默认值以常量形式导出，模块各自引用而非硬编码。

### G-02：`state_confirm_bars` 是默认值不一致的"重灾区"

该参数在9个不同位置有3种不同的默认值（2/3/5），是系统中默认值最不一致的参数，且涉及"P0-17修复"的文档追溯链断裂。

### G-03：`get_default_state_param_sets()` 是系统中最危险的静默回退点

该函数位于整个状态参数体系的根部，其YAML加载失败使用debug级别日志，且代码回退值与YAML值在`correct_trending`状态的`max_risk_ratio`（0.3 vs 0.8）和`close_stop_loss_ratio`（0.4 vs 0.05）上存在极端差异。
