## 1. CTP/交易所数据验证

### P0-TD-01: tick入口层缺少NaN/Inf/负价格过滤 ⚠️

**文件**: `strategy_tick_handler.py` 行 269–287 (`_process_tick`)  
**输入来源**: CTP/InfiniTrader 平台的 `tick` 对象  
**缺失的验证**: 
```python
last_price = self._get_tick_field(tick, 'last_price', 0.0)   # 无 NaN/Inf 检测
volume     = self._get_tick_field(tick, 'volume', 0)         # 无 NaN/Inf/负值检测
```
- `last_price` 可能为 `float('nan')`、`float('inf')`、`-float('inf')`、负值、0
- `volume` 可能为负、NaN、浮点数而非整数

**下游影响链**（当前无保护的路径）:
1. `last_price` → `_sub_mgr.on_tick(instrument_id, last_price, volume)` → `subscription_manager.py:1102`
2. `last_price` → `OrderFlowBridge.on_tick_feed()` → 订单流计算
3. `last_price` → `real_time_cache.update_tick()` → 缓存污染
4. `last_price` → `ProductionQuantSystem.update_tick()` → `TickAggregator.update_tick()` → K线合成（`quant_platform.py:95-98` 有 `price<=0` 检查，但 NaN 和 Inf 不会被该检查捕获）

**恶意/异常输入后果**: NaN 传播到趋势计算、HMM 状态推断、波动率过滤，导致全指标输出为 NaN，最终可能触发错误交易信号。

**现有局部保护**:
- `quant_platform.py:97`: `if price <= 0: return None` — 但 NaN > 0 为 False（不会触发），+Inf > 0 为 True（会通过）
- `subscription_manager.py:1109`: `if last_price in (None, ''): return` — 不拦截 NaN

**修复建议**: 在 `_process_tick` 开头增加 `math.isfinite(last_price) and last_price > 0` 检查。

### P0-TD-02: tick成交量未验证非负且未验证为整数

**文件**: `strategy_tick_handler.py` 行 287, `subscription_manager.py` 行 1102  
**输入来源**: CTP平台的 `Volume` 字段  
**缺失的验证**: 
- `volume` 未检查是否为负数
- `volume` 未检查是否 NaN/Inf
- `volume` 以浮点 `float` 形式传入 `on_tick`（声明类型为 `float = 0`），实际应为整数

**证据**: 
```python
# subscription_manager.py:1102
def on_tick(self, instrument_id: str, last_price: float, volume: float = 0) -> None:
```
类型声明为 `float`，且仅在第 1109 行检查 `last_price in (None, '')`，完全没有对 volume 做任何验证。

**恶意/异常输入后果**: 负 volume 导致合成K线的 delta 成交量计算错误（`quant_platform.py:104` 使用 `max(0, volume - self._last_cum_volume)`），但若 `_last_cum_volume = None` 则 `tick_vol = 0`，后续 bar 合成正常。然而负 volume 会影响下游统计、订单流 OFI 计算等。

**修复建议**: `on_tick` 入口处增加 `volume >= 0` 和 `isinstance(volume, (int, float)) and math.isfinite(volume)` 验证。

### P1-NB-04: Python float 比较陷阱——`price <= 0` 不等于 `not math.isfinite` 或 `price > 0`

**影响范围**: 全系统所有 `price <= 0` 检查  
**问题**: 
```python
float('nan') <= 0    # False — NaN 通过检查！
float('inf') <= 0    # False — Inf 通过检查！
float('-inf') <= 0   # True  — 仅 -Inf 被拦截
```
这意味着 `price = NaN` 的 tick 会通过所有用 `price <= 0` 做保护的检查点。

**影响模块**:
- `position_service.py:387`
- `quant_platform.py:97`
- `shared_utils.py:741`
- `strategy_tick_handler.py:1164`

**修复建议**: 全局替换 `price <= 0` 为 `not (math.isfinite(price) and price > 0)`，或使用 `safe_float()` 工具函数。

### P1-SQL-02: cleanup_old_data 的 condition 参数解析有注入残留风险

**文件**: `storage_core.py` 行 1604–1630  
**代码**:
```python
if condition:
    match = re.match(r"^\s+AND\s+(\w+)\s*=\s*['\"]?([^'\"]+)['\"]?$", condition, re.IGNORECASE)
    if match:
        col_name = match.group(1)
        col_value = match.group(2)
        sql += f" AND {col_name} = ?"    # 使用参数化查询 ✅
        params.append(col_value)
```

**分析**:
- `table` 名有白名单验证（`allowed_tables` set, 行 1608-1611）✅
- `condition` 有正则匹配验证（`re.match`）✅
- 列名 `col_name` 通过正则 `(\w+)` 提取，仍然拼接为 f-string ⚠️

**残余风险**: 虽然 `\w+` 限制了字符集，但如果 DuckDB 支持特殊列名（如引号包含空格），则可能被绕过。不过 DuckDB 参数化查询不支持参数化列名，这是架构限制。

**结论**: 输入源的 `condition` 来自 `self._cleanup_config` 字典中的预定义值（`storage_core.py` 初始化时硬编码），当前无外部输入路径——风险为 P1（架构防御不足）。

## 7. 跨系统数据验证

### P1-CS-01: 评判系统输出的参数值无输入验证 ⚠️

1. 止盈/止损比一致性: `close_take_profit_ratio > close_stop_loss_ratio`（当前 C1 仅检查 tp>1, C2 仅检查 sl<1，未做交叉比较）

## 8. 异常值检测与处理

### P0-AD-01: 无实时价格跳变检测 ⚠️

**文件**: 无对应功能模块（缺失）  
**输入来源**: 连续 tick 的 `last_price`  
**缺失的检测**:
