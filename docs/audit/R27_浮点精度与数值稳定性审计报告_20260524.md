## 二、维度1：浮点数相等比较

### P1-1: `profit <= 0` 在跟踪止损中的零判断

**位置**: `strategy_tick_handler.py` 第1301、1305行  
**代码**: `if profit <= 0: return avg_price`  
**分析**: `profit = current_price - avg_price`，当价格恰好等于均价时，因浮点精度 `profit` 可能为 `±1e-15`。`profit <= 0` 会将 `+1e-15` 视为有利润而进入跟踪逻辑，`-1e-15` 视为无利润返回均价。虽然1e-15级别的误差对期权价格无实质影响，但会导致跟踪止损行为不一致。  
**风险**: 极微小的正利润被用于计算 `current_price - profit * ratio`，结果与 `avg_price` 差异在 1e-15 级别，不影响止损触发。  
**建议**: 添加容差 `if profit <= 1e-12:` 替代 `profit <= 0`，提高逻辑一致性。

### P1-2: `pnl < 0` 在连续亏损计数中的零判断

**位置**: `risk_service.py` 第1213行  
**代码**: `if pnl < 0: self._consecutive_loss_counts[strategy_id] = count + 1`  
**分析**: 当盈亏恰好为0（例如因手续费导致盈亏=0.00000000001），浮点误差可能将本应为0的盈亏判断为微负。`pnl < 0` 会将 `pnl = -1e-15` 计入连续亏损，而 `pnl = 0.0` 或 `+1e-15` 不计数。这导致连续亏损计数被浮点噪声污染。  
**风险**: 极端情况下，多次微盈亏被误判为连续亏损，可能错误触发连亏暂停机制。  
**建议**: 使用 `if pnl < -1e-10:` 引入容差。

### P2-1: `composite > 0.05` 趋势方向判定

**位置**: `quant_core.py` 第181行  
**代码**: `'trend_direction': 'UP' if composite > 0.05 else ('DOWN' if composite < -0.05 else 'FLAT')`  
**分析**: 使用0.05作为趋势判断阈值。composite 是 `sum(w_i * s_i)`，其中 s_i 是 `trend_dir * adx/100`。当趋势刚好在边界时，浮点误差可能导致方向在 UP/DOWN/FLAT 之间抖动。0.05的阈值已经提供了足够的缓冲区间，风险较低。  
**建议**: 已有阈值防护，可接受。

### P2-2: `if price <= 0` 守卫检查

**位置**: 多处（`shared_utils.py:741`, `strategy_tick_handler.py:1164`, `order_service.py:412`, `task_scheduler.py` 多处）  
**分析**: `price <= 0` 是典型的守卫检查，用于拒绝无效价格。浮点误差不可能使正价格变为 ≤0。  
**风险**: 无实质风险。

## 三、维度2：灾难性抵消

### P1-3: PnL归因中 `total_pnl - abs(total_explained)` 的抵消

**位置**: `risk_service.py` 第800行  
**代码**: 
```python
total_pnl = abs(net_delta * 0.01) + abs(net_gamma * 0.0001) + abs(net_vega * 0.01) + abs(net_theta)
unexplained = total_pnl - abs(total_explained) if total_pnl > 0 else 0.0
```
**分析**: `total_pnl` 和 `total_explained` 由相同的Greek贡献项累加得到。当两者都非常大（如大仓位时），相减得到的小差值 `unexplained` 会受灾难性抵消影响，有效精度下降。例如 `total_pnl = 100000.0001` 与 `total_explained = 100000.0000` 相减，在float64下精度约为1e-11，但在float32下仅约1e-5。  
**风险**: 当 unexplained 接近 15% 阈值时，精度损失可能导致E7告警误触发或漏触发。  
**当前缓解**: 代码使用了 `abs(deviation) / max(abs(path_mean), 1e-10)` 的防护（第917行），在别处已有除法安全措施。  
**建议**: 改为逐项对比 `sum of (individual_contrib - individual_explained)` 而非先求总和再相减。

### P2-3: 跟踪止损中的 `current_price - profit * ratio` 抵消

**位置**: `strategy_tick_handler.py` 第1303行  
**代码**:
```python
profit = current_price - avg_price
if profit <= 0: return avg_price
return current_price - profit * self._stop_profit_trail_ratio
```
**分析**: 当 `profit` 很小时，`current_price - profit * ratio` → `current_price * (1-ratio) + avg_price * ratio`，这是两个相近大数（当价格和均价都很大时）的线性组合。由于 `ratio` 是精确浮点数（0.3），`current_price` 和 `avg_price` 均为价格量级（如4000），精度损失为 4000 × 2^-53 ≈ 4.4e-13，对期权价格无实质影响。  
**建议**: 已可接受。

## 四、维度3：累积误差

### P1-4: Greek贡献的多项 `+=` 累加

**位置**: `risk_service.py` 第756-757行（压力测试），第793-796行（PnL归因），第892行（Gamma保真度）  
**代码**:
```python
jump_1sigma_pnl += delta_contrib * sigma_1 + gamma_contrib * sigma_1 * sigma_1
# ...
delta_contrib += greeks.get('delta', 0.0) * multiplier * size * 0.01
gamma_contrib += 0.5 * greeks.get('gamma', 0.0) * multiplier * size * 0.0001
vega_contrib += greeks.get('vega', 0.0) * multiplier * size * (iv * 0.01)
theta_contrib += greeks.get('theta', 0.0) * multiplier * size
```
**分析**: 循环遍历所有持仓合约，对Greek贡献进行 `+=` 累加。备兑组合可能包含数十个期权合约，每个合约的Greek乘以multiplier×size后可能达到数百量级。假设累加50次，每次累加的精度损失约 `ulp(最大值) × 50`，在float64下ulp(10000)≈1.8e-12，累积后约9e-11，对金融计算无实质影响。  
**风险**: 在极端组合（数百个合约）或float32精度下降场景（如`to_float32`转换后）下，累积误差可能达到0.01级别。  
**建议**: 对于希腊字母聚合，使用 `math.fsum()` 替代循环 `+=`。

### P1-5: 已实现波动率的 `_rv_buffer.sum()` 

**位置**: `quant_core.py` 第605行  
**代码**: `self._current_rv = math.sqrt(self._rv_buffer.sum() / len(self._rv_buffer))`  
**分析**: `_rv_buffer` 是 `NumpyRingBuffer`，`sum()` 方法对应 NumPy 的 `ndarray.sum()`。NumPy 的 `.sum()` 在内部可能使用快速但不保证精确的累加算法（取决于BLAS实现和向量化路径）。对于 lookback=100 个元素，累积误差极小。  
**建议**: 如果 lookback 很大（>1000），可考虑使用 `np.sum(_rv_buffer, dtype=np.float64)` 显式指定精度。

### P2-4: 趋势评分的加权和

**位置**: `quant_core.py` 第178行  
**代码**: `composite = max(-1.0, min(1.0, sum(w * s for w, s in zip(self._weights, scores))))`  
**分析**: 3个权重×3个分数的加权和。weights总和已归一化为1。累加3项float64的误差可忽略。

### P2-5: 弹簧策略的盈亏比累加

**位置**: `box_spring_strategy.py` 第1290-1291行  
**代码**:
```python

avg_loss = sum(loss_pnls) / len(loss_pnls) if loss_pnls else 1.0
```

## 五、维度4：比较容差缺失（🔴 高风险）

### 🔴 P0-1: 动态追击引擎止损比较无容差

**位置**: `strategy_tick_handler.py` 第1256-1264行  
**代码**:
```python
if direction == 'BUY':
    if current_price >= pos.current_stop_profit:  # 止盈
        should_exit, reason = True, 'pursuit_take_profit'
    elif current_price <= pos.current_stop_loss:   # 止损
        should_exit, reason = True, 'pursuit_stop_loss'
else:
    if current_price >= pos.current_stop_loss:     # 止损
        should_exit, reason = True, 'pursuit_stop_loss'
    elif current_price <= pos.current_stop_profit: # 止盈
        should_exit, reason = True, 'pursuit_take_profit'
```
**分析**: 这是全系统最严重的浮点精度风险。止损/止盈价格通过 `_calc_trailing_stop` 等函数从浮点运算得出（乘法、减法链），而 `current_price` 来自tick数据。两者均可能是float64经过多步运算的产物。当 `current_price` 与 `stop_price` 之差在1e-12级别时：
- 使用 `>=` 和 `<=` 的比较，浮点误差1e-12的偏差不会改变比较结果（差值远大于float64精度）
- 但若 `stop_price` 是通过 `price * (1 - tight_sl_pct)` 计算（第1316行），当 `tight_sl_pct=0.15` 且 `price=1.0` 时结果=0.85，`current_price=0.8500000000000001` 将触发止损

**真正危险场景**: 回测中`current_price`来自K线合成的模拟价格（非真实tick），而`stop_price`来自理论计算。在极窄区间（如权利金<0.01的虚值期权），`price * (1 ± ratio)` 的浮点误差（约1e-16 × price）远小于最小价格变动单位（tick size），**当前极低风险**。

**风险等级调整**: 考虑到期权的最小价格变动单位（tick size）通常为0.5或1，远大于float64精度（2^-52 × price ≈ 1e-13），**当前P0标记是基于金融行业最佳实践的预防性标记**。在实际交易中，价格本身已经被tick size离散化，浮点误差不会跨越tick边界。

**建议**: 引入容差值 `max(current_price - stop_price, abs(stop_price) * 1e-12)`，或对止损价进行tick size舍入后再比较。

### 🔴 P0-2: 硬时间止损第二阶段斜率比较无容差

**位置**: `risk_service.py` 第2046-2047行  
**代码**:
```python

if profit_slope < stage2_slope_threshold:
```
**分析**: `profit_slope` 是通过 `(profit[-1]-profit[-5]) / time_interval` 计算的斜率。当盈利曲线接近水平时，斜率接近0。浮点误差使斜率为 `-0.0100000000000001` 时将触发止损，而本应是 `-0.0099999999999999` 时不触发。差值在1e-15级别。  
**当前防护**: threshold=-0.01已经预留了缓冲（即斜率必须明显下降），但差值仍在浮点噪声范围内。  
**风险**: 极低——profit_slope的浮点误差远小于0.01的实际阈值。差值至少3个数量级以上。  
**建议**: 如需要，将阈值设为 `-0.01 - 1e-12` 以消除边界歧义。

## 六、维度5：金融精度要求违反

### P1-9: `send_order` 中 `volume` 参数类型为 float

**位置**: `order_service.py` 第188行  
**代码**: `def send_order(self, instrument_id: str, volume: float, price: float, ...)`  
**分析**: 交易量/手数 `volume` 在金融交易中本质上是整数（整数手数），但函数签名声明为 `float`。实际调用时，下游有 `int()` 转换（如金字塔加仓 engine 第1374行 `int(base_volume * ratio)`），但在 `send_order` 内部可能未强转。  
**风险**: 如果传入 `volume=1.0000000000000002`，可能被下游视为 2 手（取决于 int 转换位置）。但实际上 tick volume 传入时已经是 int（见 `_dispatch_tick` 第643行 `volume: int`）。  
**建议**: 将 `volume` 参数类型改为 `int`，或在函数入口做 `volume = max(1, int(volume))` 防护。

### P2-11: `safe_int` 的 `int(float(value))` 可能丢失精度

**位置**: `shared_utils.py` 第181行  
**代码**: `return int(float(value))`  
**分析**: 先转 float 再转 int。对于大于 2^53 的整数（约9e15），float 转换会丢失低位精度。但对于交易量/手数等字段，值远小于 2^53。  
**建议**: 可接受。如果value本身已是int，直接转int更安全。

### P2-12: 资金路由分配使用 float 而非 Decimal

**位置**: `strategy_ecosystem.py` 第530行 `route_capital`  
**代码**: `Dict[str, float]` 返回值  
**分析**: 资金分配比例使用 float64 表示。0.05（5%）在 float64 下可精确表示。多个比例累加后总和可能偏离 1.0，代码末尾使用 `remaining` 变量补齐，抵消了分配误差。  
**建议**: 已有剩余补齐机制，可接受。

## 八、维度7：舍入模式不明确

### P2-14: 广泛使用的 `round()` 用于显示

**位置**: `risk_service.py` (多处), `strategy_ecosystem.py` (多处), `task_scheduler.py` (多处)  
**代码**: `round(net_delta, 4)`, `round(close_slippage, 2)` 等  
**分析**: Python的 `round()` 使用银行家舍入（round half to even），而非金融常用的四舍五入（round half up）。例如 `round(2.5) = 2`, `round(1.5) = 2`。  
**风险**: 仅用于日志输出和仪表盘显示，不影响交易决策。但若未来被用于金额计算，需改为 `Decimal.quantize()`。  
**建议**: 当前实现可接受（仅显示）。如需金融精确舍入，使用 `Decimal`。

### P2-15: `int(round(result))` 混用

**分析**: 先 `round()`（银行家舍入）再 `int()`（截断）。意味浮点值先舍入到整数再转int，避免了直接int截断导致的 1.9999999999→1 问题。
