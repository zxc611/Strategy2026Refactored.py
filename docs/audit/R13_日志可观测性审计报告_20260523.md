## 二、维度1：关键操作无日志

### D1-P0-01: mode_engine.compute_position_size() 完全无日志
- **文件**: `mode_engine.py`
- **行号范围**: 约820-900（compute_position_size方法）
- **问题描述**: 仓位计算是交易系统最核心的决策函数之一，返回0.0时（拒绝开仓）无任何日志记录。实盘中若仓位持续为0，运维人员无法判断是ModeEngine拒绝、TVF调整还是其他原因。
- **严重等级**: P0
- **影响系统**: 生产系统
- **手册依据**: V7.0 §11.1 结构化日志要求开仓信号、仓位变化写入JSONL

### D1-P0-02: position_service._check_stop_profit()/_check_stop_loss() 无直接日志
- **文件**: `position_service.py`
- **行号范围**: 约950-1010（_check_stop_profit/_check_stop_loss方法）
- **问题描述**: 止盈止损触发是持仓管理最关键的操作，但_check_stop_profit()和_check_stop_loss()内部无日志，仅委托_trigger_close_position()记录。当止盈止损因条件不满足而跳过时，无任何记录。实盘中止盈止损"不触发"与"触发但失败"无法区分。
- **严重等级**: P0
- **影响系统**: 生产系统

### D1-P0-03: order_service.check_pending_orders() 超时检测无日志
- **文件**: `order_service.py`
- **行号范围**: 约470-520（check_pending_orders方法）
- **问题描述**: 挂单超时检测是订单生命周期管理的关键环节。当订单超时被撤销时，无独立的超时日志记录，仅依赖cancel_order()的INFO日志。实盘中大量挂单超时可能被淹没在普通取消日志中。
- **严重等级**: P0
- **影响系统**: 生产系统

### D1-P1-01: mode_engine.filter_signal_by_mode() 不记录被过滤信号

- **行号范围**: 约750-783（filter_signal_by_mode方法）

- **问题描述**: 预测状态引擎的状态转换记录无任何日志输出，无法追踪预测模型的实时行为。

## 三、维度2：敏感信息泄露

### D2-P1-01: StructuredJsonlLogger 写入信号/订单数据无脱敏
- **文件**: `health_check_api.py`
- **行号范围**: 19-51（StructuredJsonlLogger.log_signal/log_order方法）
- **问题描述**: 结构化日志直接写入instrument_id、price、volume等交易敏感数据到JSONL文件，无脱敏处理。虽然交易系统日志通常需要这些字段，但JSONL文件无文件权限控制（umask默认），可能被同机器其他用户读取。
- **严重等级**: P1
- **影响系统**: 生产系统

### D2-P2-01: config_service.py 环境变量中可能包含数据库密码
- **文件**: `config_service.py`
- **行号范围**: 约60-70（环境变量读取）
- **问题描述**: DUCKDB_MAX_MEMORY等环境变量通过os.getenv读取，虽然当前未发现密码泄露，但配置加载路径中可能包含敏感连接字符串。日志中未发现直接打印密码/token的情况（PlatformAuthenticator的token未记录，这是好的）。
- **严重等级**: P2
- **影响系统**: 生产系统

### D2-P2-02: shadow_strategy_engine 日志包含模拟账户权益信息
- **文件**: `shadow_strategy_engine.py`
- **行号范围**: 约1030-1083（get_paper_account/update_paper_account方法）
- **问题描述**: 影子策略的模拟账户信息（初始权益100万、当前权益、持仓）通过get_stats()暴露，虽然影子策略是模拟的，但权益信息仍属于内部敏感数据。
- **严重等级**: P2
- **影响系统**: 生产系统

## 四、维度3：日志级别不当

### D3-P0-01: greeks_calculator.py 模块级StreamHandler导致重复日志
- **文件**: `greeks_calculator.py`
- **行号范围**: 约52-58（模块级logger配置）
- **问题描述**: greeks_calculator.py在模块级别创建StreamHandler并添加到logger，当模块被多次导入时会产生重复handler，导致每条日志输出多次。这违反了config_service.py的统一日志初始化原则，且在高频tick场景下会放大日志量。
- **严重等级**: P0
- **影响系统**: 生产系统

### D3-P1-01: signal_service.py 信号过滤使用DEBUG级别
- **文件**: `signal_service.py`
- **行号范围**: 218, 231, 241（PLR/ModeEngine/cooldown过滤日志）
- **问题描述**: 信号被PLR过滤、ModeEngine过滤、冷却过滤时均使用DEBUG级别。在实盘中，信号过滤是核心决策路径，DEBUG级别默认不输出，导致无法追踪信号被过滤的原因和频率。
- **严重等级**: P1
- **影响系统**: 生产系统

### D3-P1-02: event_bus.py 回调异常使用exc_info=False
- **文件**: `event_bus.py`
- **行号范围**: 约421（_invoke_all_callbacks异常处理）
- **问题描述**: 事件回调执行异常时仅记录异常消息字符串，不记录堆栈（exc_info=False）。在事件驱动架构中，回调异常的根因定位需要完整堆栈信息。
- **严重等级**: P1
- **影响系统**: 生产系统

### D3-P1-03: risk_service.py check_asymmetric_risk() 阻断用INFO级别
- **文件**: `risk_service.py`
- **行号范围**: 约1198（check_asymmetric_risk阻断日志）
- **问题描述**: 非对称风控阻断交易时使用INFO级别，而风控阻断是重要的安全事件，应使用WARNING级别。通过时使用DEBUG级别是合理的，但阻断应提升。
- **严重等级**: P1
- **影响系统**: 生产系统

### D3-P2-01: config_service.py 使用f-string格式化日志
- **文件**: `config_service.py`
- **行号范围**: 多处（如667行 `logging.info(f'[config_service] Paths configured: {_CACHED_PATHS}')`）
- **问题描述**: 使用f-string而非%格式化，在日志级别未启用时仍会计算f-string表达式。虽然config_service的日志量不大，但违反了Python logging最佳实践。
- **严重等级**: P2
- **影响系统**: 生产系统

### D3-P2-02: strategy_core_service.py 大量使用f-string日志
- **文件**: `strategy_core_service.py`
- **行号范围**: 多处（如188, 199, 213, 233, 693, 716等）
- **问题描述**: Strategy2026和StrategyCoreService中大量使用f-string格式化日志（如`logging.error(f"[Strategy2026.onStart] CRITICAL: {e}")`），在DEBUG级别未启用时仍会计算字符串。
- **严重等级**: P2
- **影响系统**: 生产系统

### D3-P2-03: signal_service.py 错误参数使用f-string
- **文件**: `signal_service.py`
- **行号范围**: 208（`logging.error(f"[SignalService] Invalid signal params: {instrument_id}, {signal_type}")`）
- **问题描述**: 无效信号参数使用f-string的ERROR日志，虽然频率低但违反统一规范。
- **严重等级**: P2
- **影响系统**: 生产系统

### D3-P2-04: subscription_manager.py 错误日志混用f-string和%格式
- **文件**: `subscription_manager.py`
- **行号范围**: 多处（如316行f-string, 228行%格式）
- **问题描述**: 同一文件中混用f-string和%格式化日志，不一致且f-string有性能开销。
- **严重等级**: P2
- **影响系统**: 生产系统

## 五、维度4：日志结构化缺失

### D4-P1-01: 全局日志格式为纯文本，非结构化JSON
- **文件**: `config_service.py`
- **行号范围**: 578-581（日志格式定义）
- **问题描述**: `setup_logging()`配置的日志格式为`'%(asctime)s - %(name)s - %(levelname)s - %(message)s'`纯文本格式，不符合V7.0手册§11.1要求的"结构化日志写入JSON lines格式"。纯文本格式不利于日志聚合、搜索和自动化分析。仅PositionService和SignalService集成了StructuredJsonlLogger，但主日志流仍为纯文本。
- **严重等级**: P1
- **影响系统**: 生产系统

### D4-P1-02: risk_service.py Greeks仪表盘输出为多行文本
- **文件**: `risk_service.py`
- **行号范围**: 约862-931（log_greeks_dashboard_periodic方法）
- **问题描述**: Greeks仪表盘每60秒输出一次，使用多行文本格式（分隔线+逐行字段），不符合结构化日志要求。虽然信息丰富，但无法被日志分析系统自动解析。
- **严重等级**: P1
- **影响系统**: 生产系统

### D4-P2-01: StructuredJsonlLogger 未集成到主日志流
- **文件**: `health_check_api.py`
- **行号范围**: 12-65（StructuredJsonlLogger类）
- **问题描述**: StructuredJsonlLogger是独立的结构化日志实现，但仅被SignalService和PositionService使用，未集成到config_service.py的统一日志框架中。订单服务使用独立的_persist_trade_log()方法写入JSONL，三套日志系统并存。
- **严重等级**: P2
- **影响系统**: 生产系统

### D4-P2-02: shadow_strategy_engine 日志文件命名不含策略组
- **文件**: `shadow_strategy_engine.py`
- **行号范围**: 约722-729（_enqueue_trade_log方法）
- **问题描述**: 影子策略日志文件名格式为`shadow_{shadow_type}_{date}.jsonl`，shadow_type包含策略组前缀（如s2_resonance_shadow_a），但所有策略组混在同一目录下，不利于按策略组筛选。
- **严重等级**: P2
- **影响系统**: 生产系统

### D4-P2-03: event_bus.py 事件持久化截断信息
- **文件**: `event_bus.py`
- **行号范围**: 约444-472（_record_event/_persist_event方法）
- **问题描述**: 事件记录截断event repr到200/500字符，丢失完整事件数据。虽然是为了控制日志大小，但截断后的事件数据无法用于完整的事后分析。
- **严重等级**: P2
- **影响系统**: 生产系统

## 六、维度5：日志性能问题

### D5-P1-01: greeks_calculator.py 错误路径使用f-string计算
- **文件**: `greeks_calculator.py`
- **行号范围**: 约99, 156（_bs_greeks/_bs_price错误路径）
- **问题描述**: Black-Scholes计算的错误路径使用f-string格式化日志消息。虽然错误路径频率低，但在高频tick场景下，IV计算的异常可能频繁触发（如vega过小），f-string的额外开销会累积。
- **严重等级**: P1
- **影响系统**: 生产系统

### D5-P1-02: event_bus.py _invoke_all_callbacks 记录完整回调对象
- **文件**: `event_bus.py`
- **行号范围**: 约410-430（_invoke_all_callbacks方法）
- **问题描述**: 回调执行时记录完整的callback和event对象repr，在高频事件场景下（如每秒数百tick），大量对象的字符串表示会产生显著的CPU和I/O开销。
- **严重等级**: P1
- **影响系统**: 生产系统

### D5-P2-01: strategy_core_service.py onStart 初始化步骤日志过于详细
- **文件**: `strategy_core_service.py`
- **行号范围**: 约783-958（onStart方法）
- **问题描述**: onStart()方法包含约20条日志输出，包括每一步的详细状态。虽然初始化仅执行一次，但日志量过大不利于快速定位关键问题。
- **严重等级**: P2
- **影响系统**: 生产系统

### D5-P2-02: subscription_manager.py 错误熔断计数器使用hasattr
- **文件**: `subscription_manager.py`
- **行号范围**: 约1164-1174（_on_tick_impl错误处理）
- **问题描述**: 每次tick处理错误时使用`hasattr(self, '_op_error_circuit')`检查属性存在性，在高频tick场景下hasattr有额外开销。应在__init__中初始化该属性。
- **严重等级**: P2
- **影响系统**: 生产系统

### D5-P2-03: signal_service.py 信号生成记录多个日志
- **文件**: `signal_service.py`
- **行号范围**: 约368-398（generate_signal信号生成后）
- **问题描述**: 每个信号生成后输出3条日志：标准INFO日志、StructuredJsonlLogger日志、快照采集日志。在高频信号场景下，3条日志/信号的I/O开销显著。
- **严重等级**: P2
- **影响系统**: 生产系统

### D5-P2-04: shadow_strategy_engine.py 日志写入使用ThreadPoolExecutor
- **文件**: `shadow_strategy_engine.py`
- **行号范围**: 约721-760（_enqueue_trade_log/_async_flush_log_queue方法）
- **问题描述**: 日志写入使用ThreadPoolExecutor(max_workers=1)异步执行，每100条刷盘一次。虽然减少了I/O阻塞，但ThreadPoolExecutor的线程调度开销在极高频率下可能成为瓶颈。建议使用Queue+专用写入线程。
- **严重等级**: P2
- **影响系统**: 生产系统

### D5-P2-05: risk_service.py Greeks仪表盘每60秒全量输出
- **文件**: `risk_service.py`
- **行号范围**: 约862-931（log_greeks_dashboard_periodic方法）
- **问题描述**: Greeks仪表盘每60秒输出一次完整的多行文本，包含所有Greeks指标。在持仓较多时，输出量较大。应考虑增量输出或仅输出变化部分。
- **严重等级**: P2
- **影响系统**: 生产系统

## 七、维度6：告警与监控缺失

### D6-P0-01: 无外部告警系统集成
- **文件**: 全系统
- **行号范围**: N/A
- **问题描述**: 整个交易系统无任何外部告警系统集成（无邮件/短信/Webhook/Prometheus/Grafana等）。SafetyMetaLayer熔断、绝对EV底线突破、每日回撤硬停止等P0级事件仅记录到日志文件，无主动通知机制。实盘中运维人员可能数小时后才在日志中发现熔断事件。
- **严重等级**: P0
- **影响系统**: 生产系统
- **手册依据**: V7.0 §12 监控与健康检查API要求"任一组件状态非OK触发告警"

### D6-P1-01: HealthCheckAPI 无主动推送能力
- **文件**: `health_check_api.py`
- **行号范围**: 67-160（HealthCheckAPI类）
- **问题描述**: HealthCheckAPI仅提供get_health_status()和check_l1_alerts()查询接口，无主动推送/定时检查能力。需要外部调用者主动轮询，但系统中无定时轮询的调度器。
- **严重等级**: P1
- **影响系统**: 生产系统

### D6-P1-02: EventBus 事件丢弃无外部告警
- **文件**: `event_bus.py`
- **行号范围**: 约300-350（背压/限流事件丢弃）
- **问题描述**: EventBus在背压或限流场景下丢弃事件时，仅更新内部计数器（dropped_events），无外部告警机制。持续的事件丢弃可能导致关键交易事件丢失而运维不知情。
- **严重等级**: P1
- **影响系统**: 生产系统

### D6-P1-03: subscription_manager 告警回调默认为None
- **文件**: `subscription_manager.py`
- **行号范围**: 128（SubscriptionConfig.alert_callback默认值）
- **问题描述**: SubscriptionConfig的alert_callback默认为None，即订阅失败率超过阈值时不触发任何告警。虽然架构上支持告警回调，但默认配置下告警功能未启用。
- **严重等级**: P1
- **影响系统**: 生产系统

### D6-P2-01: PerformanceMonitor 无阈值告警

- **行号范围**: 全文件（PathCounter类）
- **问题描述**: PathCounter仅记录调用次数和执行时间，无阈值告警机制。当某个路径的执行时间超过阈值或错误率过高时，无主动通知。
- **严重等级**: P2
- **影响系统**: 生产系统

### D6-P2-02: shadow_strategy_engine Alpha衰减无实时告警

- **严重等级**: P2
- **影响系统**: 生产系统

## 八、维度7：日志丢失风险

### D7-P0-01: enable_async_logging默认为False，无异步日志保护
- **文件**: `config_service.py`
- **行号范围**: 约159-171（LoggingConfig数据类）
- **问题描述**: LoggingConfig中`enable_async_logging: bool = False`，`log_queue_size: int = 1000`，但setup_logging()函数中完全未实现异步日志功能。在高频tick场景下，同步日志写入可能成为性能瓶颈，且进程崩溃时缓冲区中的日志会丢失。
- **严重等级**: P0
- **影响系统**: 生产系统

### D7-P0-02: EventBus shutdown 清除所有统计数据
- **文件**: `event_bus.py`
- **行号范围**: 约629-640（shutdown方法）
- **问题描述**: EventBus的shutdown()方法清除所有统计数据（_event_counts, _callback_stats等），导致停机时的最终状态无法被外部获取。如果shutdown后需要生成运行报告，所有统计信息已丢失。
- **严重等级**: P0
- **影响系统**: 生产系统

### D7-P1-01: StructuredJsonlLogger 每次写入都open/close文件
- **文件**: `health_check_api.py`
- **行号范围**: 33-34, 50-51, 63-64（log_signal/log_order/log_close方法）
- **问题描述**: StructuredJsonlLogger的每个log方法都使用`with open(...) as f`模式，每次写入都打开和关闭文件。在高频场景下，频繁的文件open/close操作有性能开销，且进程崩溃时最后一条记录可能未写入。
- **严重等级**: P1
- **影响系统**: 生产系统

### D7-P1-02: subscription_manager WAL写入无fsync
- **文件**: `subscription_manager.py`
- **行号范围**: 约473-477（_wal_writer_loop方法）
- **问题描述**: WAL写入线程使用`self._wal_file.flush()`但未调用`os.fsync()`，在操作系统崩溃时可能丢失最近写入的WAL记录。对比config_service.py的FlushHandler使用了os.fsync()。
- **严重等级**: P1
- **影响系统**: 生产系统

### D7-P2-01: 日志文件无权限控制
- **文件**: `config_service.py`
- **行号范围**: 约608-634（日志文件创建）
- **问题描述**: 创建日志文件时未设置文件权限（如0600），使用系统默认umask。在多用户服务器上，其他用户可能读取包含交易信息的日志文件。
- **严重等级**: P2
- **影响系统**: 生产系统

### D7-P2-02: shadow_strategy_engine 日志队列满时静默丢弃
- **文件**: `shadow_strategy_engine.py`
- **行号范围**: 约241（LOG_QUEUE_MAX_LEN = 50000）
- **问题描述**: 日志写入队列使用deque(maxlen=50000)，当队列满时旧记录被静默丢弃（deque的默认行为），无任何告警或计数。
- **严重等级**: P2
- **影响系统**: 生产系统

## 九、维度8：审计追踪缺失

### D8-P0-01: 无防篡改的审计日志机制
- **文件**: 全系统
- **行号范围**: N/A
- **问题描述**: 交易系统的审计日志（order_service的_persist_trade_log、position_service的结构化日志、shadow_strategy_engine的JSONL日志）均为普通文件写入，无哈希链或数字签名保护。日志文件可被任何人修改或删除，无法满足金融交易的审计合规要求。
- **严重等级**: P0
- **影响系统**: 生产系统
- **手册依据**: V7.0 §11.1 结构化日志要求，金融系统通常需要防篡改审计

### D8-P1-01: order_service._persist_trade_log 无文件权限控制
- **文件**: `order_service.py`
- **行号范围**: 约722-770（_persist_trade_log方法）
- **问题描述**: 交易日志写入JSONL文件时未设置文件权限，且无文件完整性校验。每日交易日志是最重要的审计记录，应设置严格权限并支持完整性验证。
- **严重等级**: P1
- **影响系统**: 生产系统

### D8-P1-02: SafetyMetaLayer 状态变更无独立审计日志
- **文件**: `risk_service.py`
- **行号范围**: 约1767-1819（SafetyMetaLayer熔断/硬停止方法）
- **问题描述**: SafetyMetaLayer的熔断触发、每日回撤硬停止等关键状态变更仅记录到主日志流，无独立的审计日志文件。当主日志文件被轮转或清理后，这些关键事件可能丢失。
- **严重等级**: P1
- **影响系统**: 生产系统

### D8-P1-03: EventBus 事件历史在shutdown时清除
- **文件**: `event_bus.py`
- **行号范围**: 约629-640（shutdown方法）
- **问题描述**: EventBus停机时清除所有事件历史和回调统计，导致运行期间的事件追踪链断裂。虽然_persist_event将事件写入JSON文件，但shutdown时的最终状态未持久化。
- **严重等级**: P1
- **影响系统**: 生产系统

### D8-P2-01: state_param_manager 状态切换历史有上限

- **问题描述**: 状态切换历史最大保留100条，超出后丢弃旧记录。在长时间运行中，早期的状态切换记录会丢失，影响长期趋势分析。
- **严重等级**: P2
- **影响系统**: 生产系统

### D8-P2-02: shadow_strategy_engine 交易记录有上限
- **文件**: `shadow_strategy_engine.py`

- **问题描述**: 影子策略的交易记录和权益曲线最大保留10000条，超出后丢弃旧记录。在长期运行中，早期的交易记录会丢失，影响Alpha衰减的长期分析。
- **严重等级**: P2
- **影响系统**: 生产系统

## 十、跨维度关联分析

### 关联1: 日志丢失 + 审计追踪 = 实盘故障不可追溯
- **涉及问题**: D7-P0-01(异步日志未实现) + D8-P0-01(无防篡改审计) + D7-P0-02(EventBus统计清除)
- **风险**: 当实盘出现异常交易或资金损失时，可能无法从日志中完整还原事件链。日志可能因同步写入阻塞而丢失，审计日志可被篡改，EventBus统计在停机时清除。

### 关联2: 关键操作无日志 + 日志级别不当 = 故障排查盲区
- **涉及问题**: D1-P0-01(仓位计算无日志) + D3-P1-01(信号过滤DEBUG级别) + D1-P1-03(治理引擎无日志)
- **风险**: 仓位计算拒绝开仓无日志、信号过滤用DEBUG级别、治理引擎完全静默，三者叠加导致实盘中"为什么没有交易"的问题无法排查。

### 关联3: 结构化缺失 + 告警缺失 = 运维效率低下
- **涉及问题**: D4-P1-01(纯文本日志) + D6-P0-01(无外部告警) + D6-P1-01(HealthCheckAPI无推送)
- **风险**: 纯文本日志无法被自动化系统解析，无外部告警无法主动发现问题，HealthCheckAPI需手动查询。运维完全依赖人工查看日志文件，效率极低。

### 关联4: greeks_calculator重复handler + f-string = 高频场景性能雪崩
- **涉及问题**: D3-P0-01(重复StreamHandler) + D5-P1-01(f-string错误路径) + D5-P1-02(完整对象repr)
- **风险**: greeks_calculator在高频tick场景下，重复handler导致每条日志输出N次，f-string和完整对象repr增加CPU开销，三者叠加可能在tick密集时造成显著延迟。

## 十三、审计结论

ali2026v3_trading交易系统在日志与可观测性方面存在**8个P0级、21个P1级、23个P2级**问题，共52项。

**核心风险**：
1. **可观测性缺口**：关键决策路径（仓位计算、止盈止损、信号过滤）缺乏日志，实盘故障无法追溯
2. **告警真空**：整个系统无外部告警机制，P0级事件（熔断、EV底线突破）仅记录到日志文件
3. **审计合规不足**：无防篡改审计日志，日志文件无权限控制，不符合金融交易审计要求
4. **日志架构碎片化**：三套日志系统并存（标准logging/StructuredJsonlLogger/_persist_trade_log），格式不统一

**积极发现**：
1. PlatformAuthenticator的token未被记录到日志（安全实践良好）
2. config_service.py使用FlushHandler确保日志即时刷盘
3. ShadowStrategyEngine实现了完整的独立JSONL日志+异步写盘
4. SubscriptionManager实现了WAL保护+重试队列+告警回调架构（虽然默认未启用）

**建议优先级**：P0修复 > 日志结构化统一 > 外部告警集成 > 审计防篡改
