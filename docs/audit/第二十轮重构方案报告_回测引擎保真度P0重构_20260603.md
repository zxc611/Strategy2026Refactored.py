### 0.2 顶级基金核心评判意见

**（一）整体架构级问题**

1. **BF-02与BF-03高度冗余**：BF-02（下一Bar执行）和BF-03（执行延迟）本质上是同一问题的两面——执行时序模型。顶级基金的实践中，这两者应合并为统一的"Execution Timing Model"（执行时序模型），区分三层时序：信号生成Bar → 信号确认Bar → 订单执行Bar。当前拆分为两个独立BF导致方案碎片化。

2. **跨BF耦合被低估**：BF-01（Bar内止损）与BF-02（延迟执行）存在强耦合——如果开仓延迟到下一Bar，那么持仓期间的Bar内止损必须同时生效。当前依赖图中BF-01标注为依赖Phase 1是正确方向，但方案未说明延迟执行后，平仓止损信号是否也延迟（会导致致命的前视偏差回归）。

3. **期权定价特殊性完全缺失**：BF-05（MTM权益）的方案仅用`(close - open_price) * multiplier`计算浮盈，这对线性品种（股票/期货）勉强可用，但对期权是完全错误的——期权浮盈必须通过Black-Scholes或二叉树模型逐Bar重定价，否则未实现P&L会严重失真。这是**最大的技术盲区**。

4. **缺失：跨Bar跳空（Gap）处理**：回测引擎完全没有讨论跨Bar跳空场景。实盘中过夜跳空/周末跳空是止损被穿透的主要来源。当开盘价直接跳过止损线时，成交价应该是开盘价而非止损线价格——这直接影响BF-01和BF-02的实现正确性。

**（二）模型级问题**

5. **Bar内路径推断过于简单**：当前方案仅检查`low <= stop_loss`就判断止损触发，这等价于"假设价格遍历了从high到low的全部区间"，实际上忽略了路径不确定性。顶级基金使用Brownian Bridge方法计算止损触发概率，而非简单的"触及即触发"。

6. **Almgren-Chriss模型仅引用未实现**：BF-06方案提到Almgren-Chriss框架，但实际实现的`_compute_market_impact()`函数中的`sigma`和`iv`参数来源未定义，且临时冲击和永久冲击的系数(`eta=0.1`, `gamma=eta*0.5`)是随意赋值而非校准值。

7. **缺失：TCA（Transaction Cost Analysis）闭环**：顶级基金的标准做法是重构后必须运行TCA分析——对比重构前后的实现成本（implementation shortfall）差异，确认保真度提升方向正确。

**（三）流程级问题**

8. **验收标准缺少压力测试**：当前验收标准全部基于正常市场状态。缺少极端市场（如2020年3月、2015年股灾）下的回测行为验证。

9. **参数校准缺少方法论**：4.3节的校准方案仅列了4项待校准参数和数据源，但未给出校准方法（如：用实盘交易日志做分布拟合？用Grid Search扫最优？用Bayesian Optimization？）和验收标准（如：校准后TCA偏差<5bp？）。

10. **缺失：回退SOP（标准操作流程）**：虽然提到了向后兼容策略，但未定义：当institutional模式结果异常时的回退流程、触发条件、责任人。

### BF-01：止损仅检查收盘价（未使用Bar内High/Low）

| 属性 | 值 |
|------|-----|
| **文件** | `param_pool/backtest_runner_base.py` |
| **行号** | L1950, L1972-1985 |
| **方法** | `_check_positions()` |
| **当前代码** | `price = bar.get("close", 0.0)` → 止损判断 `price <= pos.stop_loss_price` |
| **问题本质** | Bar内价格可能先触及止损线再回到收盘价上方，但回测仅用close判断，导致止损被"穿越"而不触发 |
| **实盘差异** | 实盘中止损单在价格触及止损线时立即成交，不需要等到Bar收盘 |

**精确代码**：
```python
# L1950: 止损/止盈判断使用的价格
price = bar.get("close", 0.0)  # ← 仅用收盘价

# L1972-1985: 止损判断逻辑
if pos.volume > 0:
    if price >= pos.stop_profit_price:  # ← 用close判断止盈
        should_close = True; close_reason = "StopProfit"
    elif price <= pos.stop_loss_price:  # ← 用close判断止损
        should_close = True; close_reason = "StopLoss"
elif pos.volume < 0:
    if price <= pos.stop_profit_price:
        should_close = True; close_reason = "StopProfit"
    elif price >= pos.stop_loss_price:
        should_close = True; close_reason = "StopLoss"
```

**顶级基金视角**：Citadel/Two Sigma的回测引擎均使用Bar内路径模拟（OHLC路径推断）来判断止损触发，而非仅用收盘价。这是回测-实盘偏差（backtest-live slippage）的最大来源之一。

**<评判补充>** 需区分两种止损穿透场景：①Bar内触及止损线（High/Low越过止损线，路径推断）；②跨Bar跳空穿透（开盘价直接越过止损线）。场景②需要用开盘价而非止损线价格作为成交价，当前方案未覆盖。

| 属性 | 值 |
|------|-----|
| **文件** | `param_pool/backtest_runner_base.py` |
| **行号** | L2020, L2039, L2500-2501 |

| **问题本质** | equity_curve记录的是"已实现权益"而非"标记市值权益"，导致回撤计算偏乐观 |
| **实盘差异** | 实盘中风控系统基于MTM权益计算回撤和保证金，浮亏同样触发风控 |

**精确代码**：
```python

pnl = (price - pos.open_price) * pos.volume * _mult
net_pnl = pnl - slip - commission
_safe_equity_add(bt, net_pnl)  # ← 仅平仓时更新

```

**顶级基金视角**：Bridgewater/Point72的风控系统基于MTM权益运行。回撤计算必须包含浮盈浮亏，否则策略可能在浮亏30%时仍不触发风控，而实盘中早已被强制减仓。这是回测-实盘偏差的第二大来源。

**<评判补充·关键>** 当前方案的最大盲区：**期权的MTM定价不能用`(close - open_price) * multiplier`**。期权的公允价值必须通过定价模型（Black-Scholes / Binomial Tree / Monte Carlo）逐Bar重新计算，输入参数包括：标的现价、行权价、剩余到期时间、隐含波动率、无风险利率。简单线性计算会显著低估深度虚值期权的浮亏（因为gamma效应被忽略），也会高估深度实值期权的浮盈。正确做法：
1. 逐Bar调用greeks_calculator或独立的期权定价模块
2. 使用bar.get("iv", ...)作为隐含波动率输入
3. 期权MTM权益 = sum(理论价 - 开仓理论价) × 合约乘数
4. 如无法获取实时IV，至少使用开仓IV作为代理变量

## 三、重构方案（优化版）

### 3.1 BF-01重构：Bar内止损路径推断 + 跨Bar跳空处理

**方案名称**：Intra-Bar Stop-Loss Path Simulation with Gap Handling

**设计原理**：
- 当Bar的High/Low跨越止损线时，推断止损触发时机
- 使用OHLC路径推断规则（Bar内方向推断：阳线(close>open)则low先触及，阴线(close<open)则high先触及）
- **关键补充**：增加跨Bar跳空检测：当`this_bar.open < prev_bar.close`跳过止损线时，成交价=this_bar.open（而非止损线价格）
- 止损成交价 = 止损线价格 + 滑点（Bar内触发）；= 开盘价 + 滑点（跳空穿透）
- **优先级规则**：止损 > 止盈（保守原则）；跳空穿透 > Bar内触及

**实现方案**：

```python
# 新增函数 _check_intra_bar_stop_loss()
# 位置: backtest_runner_base.py, _check_positions() 内部

def _check_intra_bar_stop_loss(pos, bar, prev_bar, params):
    """Bar内止损路径推断 + 跨Bar跳空处理 — BF-01重构（优化版）
    
    规则:
    1. 跳空穿透优先检测: 开盘价直接跳过止损线 → 成交价=开盘价
    2. Bar内检测: high/low跨越止损线 → 路径推断确定触发顺序
    3. 多头止损: low或open穿透止损线 → 触发
    4. 空头止损: high或open穿透止损线 → 触发
    5. 优先级: (跳空 > Bar内) × (止损 > 止盈)
    6. 路径推断: 阳线→low先触及(先检查止损), 阴线→high先触及(先检查止盈)
    """
    bar_high = bar.get("high", bar.get("close", 0.0))
    bar_low  = bar.get("low", bar.get("close", 0.0))
    bar_open = bar.get("open", bar.get("close", 0.0))
    bar_close = bar.get("close", 0.0)
    
    # ===== 补充: 跨Bar跳空穿透检测 =====
    # 场景: 开盘价直接跳过止损线 (例如: 昨日收盘100, 止损95, 今日开盘93)
    prev_close = prev_bar.get("close", bar_open) if prev_bar else bar_open
    gap_down = bar_open < prev_close      # 向下跳空
    gap_up   = bar_open > prev_close      # 向上跳空
    
    if pos.volume > 0:  # 多头持仓
        # 跳空穿透止损: 开盘价直接低于止损线
        if gap_down and bar_open <= pos.stop_loss_price:
            return True, "StopLoss_GapDown", bar_open
        # 跳空穿透止盈: 开盘价直接高于止盈线
        if gap_up and bar_open >= pos.stop_profit_price:
            return True, "StopProfit_GapUp", bar_open
        # Bar内路径推断
        is_bullish = bar_close >= bar_open  # 阳线: 先跌后涨
        if is_bullish:
            # 阳线: low先触及, 先检查止损
            if bar_low <= pos.stop_loss_price:
                return True, "StopLoss_IntraBar", pos.stop_loss_price
            if bar_high >= pos.stop_profit_price:
                return True, "StopProfit_IntraBar", pos.stop_profit_price
        else:
            # 阴线: high先触及, 先检查止盈
            if bar_high >= pos.stop_profit_price:
                return True, "StopProfit_IntraBar", pos.stop_profit_price
            if bar_low <= pos.stop_loss_price:
                return True, "StopLoss_IntraBar", pos.stop_loss_price
    
    elif pos.volume < 0:  # 空头持仓
        if gap_up and bar_open >= pos.stop_loss_price:
            return True, "StopLoss_GapUp", bar_open
        if gap_down and bar_open <= pos.stop_profit_price:
            return True, "StopProfit_GapDown", bar_open
        is_bullish = bar_close >= bar_open
        if is_bullish:
            if bar_high >= pos.stop_loss_price:
                return True, "StopLoss_IntraBar", pos.stop_loss_price
            if bar_low <= pos.stop_profit_price:
                return True, "StopProfit_IntraBar", pos.stop_profit_price
        else:
            if bar_low <= pos.stop_profit_price:
                return True, "StopProfit_IntraBar", pos.stop_profit_price
            if bar_high >= pos.stop_loss_price:
                return True, "StopLoss_IntraBar", pos.stop_loss_price
    
    return False, "", None
```

**修改点**：
1. `_check_positions()` L1950: 先调用`_check_intra_bar_stop_loss()`，传入prev_bar
2. 止损/止盈成交价区分两种场景（跳空用开盘价，Bar内用止损线价格）
3. 新增配置参数 `enable_intra_bar_stop_loss`（默认True）
4. 记录止损触发类型（StopLoss_GapDown / StopLoss_IntraBar）用于TCA分析

**策略影响分析**：Bar内止损开启后，预计止损触发频率提升50%-80%，净收益下降3%-8%。这反映的是真实风险暴露，而非策略退化。

### 3.2 BF-02 + BF-03合并重构：统一执行时序模型（Execution Timing Model）

**方案名称**：Unified Execution Timing Model（合并原BF-02和BF-03）

> **评判优化说明**：BF-02（开仓延迟）和BF-03（平仓延迟）本质上是执行时序模型的两个维度，顶级基金的实践是统一设计而非拆分。合并后消除方案碎片化，依赖关系更清晰。

**设计原理**：
- **三层时序模型**：
  - T层（信号生成Bar）：策略逻辑运行，产生开仓/平仓信号
  - T+1层（信号确认Bar）：信号在下一Bar开始时确认有效（消除前视偏差）
  - T+1或T+2层（订单执行Bar）：订单执行，成交价=该Bar开盘价+滑点+冲击成本
- **延迟分级**：
  - 开仓：信号Bar(T) → 执行Bar(T+1)，延迟=1 Bar
  - 止盈平仓：信号Bar(T) → 执行Bar(T+1)，延迟=1 Bar
  - 止损平仓：不延迟，即时执行（止损为预挂单，触及即成交）
- **与BF-01联动**：Bar内止损触发时，强制无视延迟立即平仓
- **与BF-06联动**：延迟执行队列中的订单在执行时需重新检查流动性约束

**数据模型**：

### 3.5 BF-05重构：逐Bar Mark-to-Market（含期权BS定价修正）

**方案名称**：Per-Bar Mark-to-Market with Option Pricing Model

> **评判优化说明**：原始方案最大的技术盲区已修正——期权的MTM计算必须通过定价模型，不能简单用`(close-open)*multiplier`。

**设计原理**：
- **线性品种**（股票/期货）：MTM = sum((current_price - open_price) * volume * multiplier)
- **非线性品种**（期权）：MTM = sum((bs_price(current) - bs_price(open)) * volume * multiplier)
- 风控指标（回撤、保证金率）基于MTM equity计算
- 如果无法获取实时IV，使用开仓时的IV作为代理变量（次优但可用）

**实现方案**：

```python
# _BacktestState 新增字段:
mtm_equity: float = 0.0

# 新增函数 _compute_option_mtm_price():
def _compute_option_mtm_price(symbol, underlying_price, strike_price, 
                                days_to_expiry, iv, option_type, risk_free_rate=0.025):
    """期权MTM定价 — BF-05重构（优化版）
    
    使用Black-Scholes模型计算期权的理论公允价值。
    如果无法获取实时IV, 使用开仓IV作为代理。
    
    Args:
        symbol: 期权合约代码
        underlying_price: 标的现价
        strike_price: 行权价
        days_to_expiry: 剩余到期天数
        iv: 隐含波动率（优先使用当前Bar的iv, fallback使用开仓iv）
        option_type: "call" / "put"
        risk_free_rate: 无风险利率（默认2.5%）
    Returns:
        期权理论价格
    """
    import numpy as np
    from scipy.stats import norm
    
    T = max(days_to_expiry / 365.0, 1.0/365.0)  # 至少1天
    sigma = max(iv, 0.01)  # iv下限1%
    
    if sigma <= 0 or T <= 0:
        return max(0, underlying_price - strike_price) if option_type == "call" else max(0, strike_price - underlying_price)
    
    d1 = (np.log(underlying_price / strike_price) + (risk_free_rate + sigma**2 / 2) * T) / (sigma * np.sqrt(T))
    d2 = d1 - sigma * np.sqrt(T)
    
    if option_type == "call":
        return underlying_price * norm.cdf(d1) - strike_price * np.exp(-risk_free_rate * T) * norm.cdf(d2)
    else:
        return strike_price * np.exp(-risk_free_rate * T) * norm.cdf(-d2) - underlying_price * norm.cdf(-d1)

def _update_mtm_equity(bt, bar, params):
    """逐Bar MTM更新 — BF-05重构（优化版：区分线性/非线性品种）"""
    unrealized_pnl = 0.0
    
    for sym, pos in bt.positions.items():
        if _is_option(sym):
            # === 期权: BS定价 ===
            underlying_price = bar.get("close", 0.0)
            current_iv = bar.get("iv", pos.open_iv)  # 优先当前Bar的iv, fallback开仓iv
            current_theo = _compute_option_mtm_price(
                symbol=sym,
                underlying_price=underlying_price,
                strike_price=pos.strike_price,
                days_to_expiry=pos.remaining_days,
                iv=current_iv,
                option_type=pos.option_type
            )
            pos_pnl = (current_theo - pos.open_theo_price) * pos.volume * _get_contract_multiplier(sym)
        else:
            # === 线性品种: 简单差价 ===
            price = bar.get("close", 0.0)
            _mult = _get_contract_multiplier(sym)
            pos_pnl = (price - pos.open_price) * pos.volume * _mult
        
        unrealized_pnl += pos_pnl
    
    bt.mtm_equity = bt.equity + unrealized_pnl
    bt.mtm_peak_equity = max(bt.mtm_peak_equity, bt.mtm_equity)
    mtm_dd = (bt.mtm_peak_equity - bt.mtm_equity) / bt.mtm_peak_equity if bt.mtm_peak_equity > 0 else 0
    bt.mtm_max_drawdown = max(bt.mtm_max_drawdown, mtm_dd)
    bt.mtm_equity_curve.append(bt.mtm_equity)
```

2. 扩展`_BacktestPosition`数据类，增加期权定价所需字段

### 3.6 BF-06重构：流动性约束与部分成交（优化版）

**方案名称**：Liquidity-Constrained Partial Fill with Calibrated Impact Model

> **评判优化说明**：原始A-C模型实现存在参数随意赋值、sigma来源不确定等问题。优化版明确标注待校准参数，区分一级模型（立即可用）和二级模型（待校准）。

**设计原理**：
- **一级模型（立即可用）**：基于参与率（Participation Rate）限制 + 固定冲击成本
- **二级模型（待数据校准）**：完整的Almgren-Chriss冲击成本拟合
- 参与率 = 订单手数 / Bar成交量（手）× 100%
- 当参与率超过阈值时，仅部分成交
- 未成交部分策略：放弃（本轮实现）/ 下一Bar追单（后续迭代）

**实现方案**：

```python
# === 一级模型: 参与率限制（立即可用）===

# 新增配置参数:
"max_participation_rate": 0.15,    # 最大参与率, 默认15%
"enable_partial_fill": True,       # 启用部分成交
"min_fill_lots": 1,                # 最小成交手数
"fixed_slippage_bps": 1.0,         # 固定滑点(bps), 默认1bp

def _compute_fill_quantity(order_lots, bar, params):
    """一级填充量计算 — BF-06重构
    
    基于参与率限制:
    1. participation_rate = order_lots / max(bar_volume_lots, 1)
    2. if participation_rate > max_participation_rate:
           fill_lots = int(bar_volume_lots * max_participation_rate)
    3. fill_lots = max(fill_lots, min_fill_lots) if fill_lots > 0 else 0
    """
    max_pr = params.get("max_participation_rate", 0.15)
    min_fill = int(params.get("min_fill_lots", 1))
    
    bar_volume = bar.get("volume", 0)
    bar_volume_lots = max(1, int(bar_volume))  # 假设volume已是手数
    
    pr = order_lots / bar_volume_lots
    
    if pr <= max_pr:
        return order_lots  # 全部成交
    
    fill_lots = int(bar_volume_lots * max_pr)
    if fill_lots < min_fill:
        return 0  # 成交量不足最小手数
    return min(fill_lots, order_lots)

# === 二级模型: Almgren-Chriss冲击成本（待校准）===

# 新增配置参数（待校准, 当前使用默认值标记为 TODO: CALIBRATE）:
"market_impact_eta": 0.1,            # TODO: CALIBRATE — 临时冲击系数
"market_impact_gamma": 0.05,         # TODO: CALIBRATE — 永久冲击系数
"enable_market_impact": False,       # 默认关闭, 校准后开启

def _compute_market_impact_v2(fill_lots, bar, price, params):
    """Almgren-Chriss市场冲击成本 — BF-06重构（二级模型, 待校准）
    
    临时冲击 = sigma * sqrt(participation_rate) * eta
    永久冲击 = sigma * participation_rate * gamma
    总冲击 = 临时冲击 + 永久冲击（仅永久部分加到成交价中）
    
    注意: sigma和eta/gamma均需从实盘Tick数据校准, 当前为占位值
    """
    if not params.get("enable_market_impact", False):
        return 0.0
    
    bar_volume = max(1, bar.get("volume", 1))
    pr = fill_lots / bar_volume
    
    # 日波动率转换为分钟波动率
    daily_vol = bar.get("daily_volatility", 0.015)  # 默认1.5%日波动率
    sigma = price * daily_vol * np.sqrt(1/240)       # 240分钟/交易日
    
    eta = params.get("market_impact_eta", 0.1)
    gamma = params.get("market_impact_gamma", 0.05)
    
    temp_impact = sigma * np.sqrt(pr) * eta
    perm_impact = sigma * pr * gamma
    
    # 冲击成本 = (临时+永久) × 成交手数 (单位: 元)
    return (temp_impact + perm_impact) * fill_lots * _get_contract_multiplier(None)
```

**未成交部分处理策略**：
```
当前实现: 放弃（fill_lots < order_lots 时, 仅部分成交, 剩余放弃）
后续迭代: 
  - Option A: 下一Bar追单（加入pending_orders队列, 避免无限追单-最多3次）
  - Option B: 降低订单量至参与率内（如果策略允许）
  - Option C: 使用TWAP算法拆单（需Tick数据支持）
```

## 八、TCA闭环设计（新增）

回测保真度重构的终极检验标准是能否准确复现实盘的Implementation Shortfall（IS）。因此设计以下TCA闭环：

### 8.1 Implementation Shortfall 定义

```
IS = 理论决策收益 - 实际执行收益

理论决策收益 = Σ(信号Bar收盘价 - 开仓价) × 手数  (假设完美执行)
实际执行收益 = Σ(实际成交价 - 开仓价 - 手续费) × 手数  (回测/实盘)

TCA偏差 = IS(回测) - IS(实盘)
```

### 8.2 TCA验证流程

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ standard模式  │ vs │institutional │ vs │  实盘交易日志 │
│ 回测结果      │    │ 回测结果      │    │  (最近1个月)  │
└──────┬───────┘    └──────┬───────┘    └──────┬───────┘
       │                   │                   │
       ▼                   ▼                   ▼
   IS_std              IS_inst             IS_live
       │                   │                   │
       └───────────────────┼───────────────────┘
                           │
                           ▼
                  ┌─────────────────┐
                  │  IS_inst vs IS_live  │
                  │  偏差评估:          │
                  │  <5bp  → 优秀        │
                  │  5-10bp → 可接受      │
                  │  10-20bp → 需校准     │
                  │  >20bp → 退回重做    │
                  └─────────────────┘
```
