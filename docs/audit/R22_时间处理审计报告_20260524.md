**代码片段**:
```python

# box_spring_strategy.py:1549
days_to_expiry = (expiry - datetime.now()).days  # naive! 时区差导致到期日误算

# order_service.py:548

# position_service.py:681

# .timestamp() 将 naive datetime 解释为本地时间——如果OS时区不等于CST则结果错8小时
```

**时间处理问题**: 全系统使用 `datetime.now()` 一律不携带 `tzinfo`。`datetime.now()` 返回的 naive datetime 隐式依赖操作系统时区。若服务器部署在 UTC 时区（AWS/GCP 默认）而交易所在 CST（UTC+8），所有时间戳将偏移8小时。

**实际/潜在后果**:
- **风险**: `risk_service.py:1779` 日切点判断 `datetime.now().strftime("%Y-%m-%d")` 在 UTC 服务器上于北京时间08:00才翻日，夜盘（21:00-02:30）持仓的日切逻辑全部错位
- **风险**: `order_service.py:934` 日统计 `datetime.now().strftime("%Y-%m-%d")` 分组错误
- **风险**: `box_spring_strategy.py:1549` 期权到期日 `(expiry - datetime.now()).days` 在UTC服务器上比实际少约0.33天
- **风险**: `risk_service.py:1923` `open_time.timestamp()` 将 naive datetime 按OS时区转换，与其他模块的时区假设不一致
- **置信度**: 高。该问题贯穿全部59个核心模块。

### P0-1.2: naive/aware datetime 混用——`ds_query_cache.py` 转换链不完整

**文件**: `ds_query_cache.py`  
**行号**: 41-57

**代码片段**:
```python
# ds_query_cache.py:41-48
@staticmethod
def _normalize_datetime_for_cache(dt: Optional[datetime]) -> Optional[str]:
    if dt is None:
        return None
    if dt.tzinfo is not None:
        dt = dt.astimezone(timezone.utc).replace(tzinfo=None)  # 强转UTC naive
    dt = dt.replace(microsecond=0)
    return dt.isoformat()

# ds_query_cache.py:51-57
@staticmethod
def _to_utc_naive(dt: datetime) -> datetime:
    if dt.tzinfo is not None:
        logger.warning(f"Converting timezone-aware datetime {dt} to UTC naive...")
        dt = dt.astimezone(timezone.utc).replace(tzinfo=None)
    return dt
```

**时间处理问题**: `ds_query_cache` 设计了 aware→naive 转换，但几乎所有调用方传入的都是 naive datetime（来自 `datetime.now()`）。同时 `_normalize_datetime_for_cache` 将 aware 转为 UTC naive 后用作缓存键——如果某天调用方传了 CST aware datetime，会被强转为 UTC，导致缓存键与查询 SQL 的 `BETWEEN` 参数出现时区语义不一致。

**实际/潜在后果**: 缓存键可能指向错误的时间范围数据，导致查询返回过时或空结果。

## 维度2: DST边界 (DST Boundary)

### P1-2.1: 无任何 DST 处理机制——部署在美区服务器时系统性错位

**涉及文件**: 所有使用 `datetime.now()` 的文件

**时间处理问题**: 中国不实施夏令时，但若系统部署在美国时区（AWS us-east-1等），`datetime.now()` 在3月和11月DST切换日会跳变1小时。系统无任何DST检测或防护机制。

**实际/潜在后果**:
- 美东时间3月第二个周日：02:00直接跳到03:00，1小时数据真空
- 美东时间11月第一个周日：01:00-02:00重复两次，tick/bar可能出现乱序
- `MarketTimeService.is_market_open()` 使用 CST aware `datetime.now(cst)`，不受DST影响——但其余所有模块都用 naive datetime，实际可能比较不同时区的"现在"

## 维度3: 时间戳精度不一致

### P0-3.1: `pd.Timestamp.now()` 作为 bar 缺失时的降级默认值——回测/实盘bar时间污染

**文件**: `参数池/task_scheduler.py`  
**行号**: 1332, 1519, 1654, 1689, 1911, 2042, 2218, 2387, 2549, 2859, 3070, 4290, 4834, 6838

**代码片段**:
```python
bar_time = bar.get("minute", pd.Timestamp.now())  # 如果bar无"minute"键,用当前时间！
```

**时间处理问题**: 当 bar 字典缺少 `"minute"` 键时，用 `pd.Timestamp.now()`（纳秒精度）作为降级默认值。这会：
1. 将回测中的 bar 时间污染为实时的"现在"
2. 导致 tick 与 bar 时间脱离，止盈止损阈值计算错误
3. `bar_time` 与其他模块中的 `datetime.now()` 时间戳精度不一致

**实际/潜在后果**: P0级——若 bar 字典键名不匹配（如字段名变更），大量 bar 会被用错误时间标记，触发错误的风控时间止损。

### P1-3.2: `risk_service.py` 毫秒字符串构造中两次调用 `datetime.now()` 导致微秒漂移

**文件**: `risk_service.py`  
**行号**: 561

**代码片段**:
```python
now_str = datetime.now().strftime("%Y-%m-%d %H:%M:%S.") + f"{datetime.now().microsecond // 1000:03d}"
```

**时间处理问题**: 第一段 `strftime` 取到 `datetime.now()` 的时刻T1，第二段取到 `datetime.now().microsecond` 的时刻T2。如果 T2 的秒数已经变化（跨越秒边界），毫秒部分的格式将是错误的（例如 T1=12:00:00.999，T2=12:00:01.000 → 输出 "12:00:00.000"）。

## 维度4: 时间比较安全性

### P0-4.1: `datetime.timestamp()` 的本地时区假设——硬止损检查

**文件**: `risk_service.py`  
**行号**: 1922-1947

**代码片段**:
```python
def check_position_hard_time_stop(self, position_id: str, open_time, ...):
    if isinstance(open_time, datetime):
        open_time = open_time.timestamp()  # 本地时区!
    # ...
    now = time.time()
    elapsed_min = (now - open_time) / 60.0
```

**时间处理问题**: `datetime.timestamp()` 将 naive datetime 按操作系统本地时区转换为 epoch 秒。如果系统时区不是 UTC 且 `open_time` 在其他时区创建，`elapsed_min` 计算错误。例如：
- `open_time` 由 `datetime.now()` 创建于北京服务器（UTC+8 naive）
- 代码部署到 UTC 服务器
- `.timestamp()` 将其解释为 UTC → 偏移8小时

**实际/潜在后果**: P0级。硬止损时间判断错8小时——如果服务器在UTC，所有持仓的实际持有时间被计算为比真实多8小时，会过早触发止损平仓；反之如果少算8小时，止损永远不触发。

## 维度5: 定时器与调度准确性

### P1-5.1: `SchedulerService._master_loop` 分段休眠导致漂移累积

**文件**: `scheduler_service.py`  
**行号**: 347-378

**代码片段**:
```python
def _master_loop(self) -> None:
    while self._running_event.is_set():
        try:
            now = time.time()
            # ... 清理、调度 ...
            for _ in range(int(self._tick_interval * 10)):
                if not self._running_event.is_set():
                    break
                time.sleep(0.1)  # 每0.1秒休眠一次
        except Exception as e:
            logging.error(f"[Scheduler] Master loop error: {e}")
            time.sleep(1)
```

**时间处理问题**: `time.sleep(0.1)` 的精度受系统调度器影响（Windows 默认精度 ~15ms，Linux ~1ms）。每0.1秒休眠额外有循环开销（range迭代 + Event检查），实际间隔 > 0.5s。累积漂移约 1-5% 每小时。

**实际/潜在后果**: 对于需要精确间隔的任务（如每30秒检查持仓），实际间隔可能漂移到33-35秒，1小时后累计延迟 ~3-5秒。

## 维度6: 日内时间边界

### P1-6.1: `config_params.py` 中 `close_overnight_check_time: "14:58"` 和 `close_daycut_time: "15:58"` 无时区锚定

**文件**: `config_params.py`
