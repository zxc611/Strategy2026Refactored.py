## P0 级缺陷（控制流导致错误交易决策/死锁/无限循环）

### P0-1: `get_health_status()` 中 `risk_dashboard_status` 变量在赋值前被引用 — NameError 必现崩溃

- **文件**: `strategy_core_service.py`
- **行号**: L447（引用）、L498（定义）
- **控制流结构**: 顺序执行（非分支）
- **缺陷类型**: 不可达代码 / 异常控制流
- **触发条件**: 任何对 `get_health_status()` 的调用
- **后果**: 函数执行到 L447 时，`risk_dashboard_status` 尚未定义，Python 解释器抛出 `NameError`，函数直接崩溃。由于 `get_health_status()` 是健康检查的核心入口（被 `execute_option_trading_cycle` L317、`Strategy2026` 等多处调用），此缺陷导致**健康检查完全不可用**，进而：
  - `execute_option_trading_cycle` 中 `health.get('health')` 异常（L318-321），实际效果是异常被外层 `except Exception` (L343) 捕获，**所有交易在健康检查阶段失败后静默终止**
  - P-34 新开仓暂停逻辑（L576-581）永远不会执行
  - 影子引擎、策略生态等组件的健康状态也不会被采集到

- **代码上下文**:
  ```python
  # L447: 使用 risk_dashboard_status — 此时变量尚未定义！
  components["risk_dashboard"] = risk_dashboard_status   # ← NameError!

  # ... 中间 50+ 行其他代码 ...

  # L498: 变量在此才被定义
  risk_dashboard_status = "UNAVAILABLE"
  try:
      # ...
      risk_dashboard_status = "OK"
  ```

- **修复建议**: 将 L498-517 的 `risk_dashboard_status` 初始化代码（以及 `try` 块）移动到 L447 之前。简单做法是在 L446 之后插入 `risk_dashboard_status = "UNAVAILABLE"`。

### P0-2: 模式切换中的 `_apply_to_*` 方法全部在 `except` 块中 `raise` — 单体故障触发灾难性回滚

- **文件**: `mode_engine.py`
- **行号**: L871-972（5个 `_apply_to_*` 方法）
- **控制流结构**: 顺序调用链
- **缺陷类型**: 异常控制流分区
- **触发条件**: `switch_mode()` 执行期间任一组件（RiskService/SignalService/StateParamManager/BoxSpringStrategy/PredictiveStateEngine）的初始化失败
- **后果**: 模式切换时，如果 5 个受传播组件中**任意一个**出现异常（例如 `get_risk_service()` 临时不可用），所有 5 个组件的配置变更全被回滚。这等于把"单体故障"放大为"系统级回滚"。特别严重的是 `BoxSpringStrategy`（非关键策略模块）的失败会让核心风控和信号服务也回滚。

- **受影响的异常抛出点**:
  | 方法 | 行号 | 组件 |
  |------|------|------|
  | `_apply_to_risk_service` | L893 | RiskService |
  | `_apply_to_predictive_state_engine` | L909 | PredictiveStateEngine |
  | `_apply_to_signal_service` | L937 | SignalService |
  | `_apply_to_state_param_manager` | L953 | StateParamManager |
  | `_apply_to_box_spring_strategy` | L972 | BoxSpringStrategy |

- **修复建议**: 为各组件配置分级容错策略：
  - RiskService / StateParamManager：`raise`（核心组件）
  - SignalService / PredictiveStateEngine：`logging.error` + 继续执行（重要但可降级）
  - BoxSpringStrategy：`logging.warning` + 继续执行（非关键组件）

## P1 级缺陷（控制流导致功能退化/数据丢失）

### P1-1: `check_before_trade()` 中影子引擎检查不在锁保护范围内 — 竞态条件

- **文件**: `risk_service.py`
- **行号**: L218-242（check_before_trade 入口）
- **控制流结构**: 顺序执行 → `with self._lock:` 上下文管理器
- **缺陷类型**: 回调与事件循环控制流
- **触发条件**: 两个线程同时对同一 signal 调用 `check_before_trade`，且影子引擎状态在 check 和 lock 之间变化
- **后果**: 
  1. Lines 224-242 的影子引擎检查（`is_absolute_ev_paused()`, `is_degradation_active()`）在锁外执行，可能与影子引擎的状态更新产生竞态
  2. 如果在线程A读取 `is_absolute_ev_paused() == False` 后、获取锁之前，线程B触发了 `_absolute_ev_pause = True`，线程A将绕过绝对EV暂停，发出不应被允许的交易信号
- **代码位置**:
  ```python
  def check_before_trade(self, signal):
      try:
          safety_result = self._check_safety_meta_layer(signal)  # 锁外
          if safety_result.is_block:
              return ...
          # L224-242: 影子引擎检查 — 锁外!
          with self._lock:   # L244 — 锁在此处才获取
              # ...风控链...
  ```

- **修复建议**: 将影子引擎检查移入 `with self._lock:` 块内，或在影子引擎中保证 `is_absolute_ev_paused()` 等方法使用原子操作读取。

### P1-2: `_ensure_order_service()` / `_ensure_position_service()` 异常静默吞掉 — 下游方法无防御

- **文件**: `strategy_core_service.py`
- **行号**: L180-200
- **控制流结构**: try/except → 设置 `None`
- **缺陷类型**: 异常控制流
- **触发条件**: OrderService 或 PositionService 导入失败
- **后果**: 当 `_ensure_order_service()` 捕获异常后（L188-189），`self._order_service` 被设为 `None`，但下游调用方（如 `execute_option_trading_cycle` L325-328）的防御模式不一致：

### P1-3: `_check_cross_strategy_risk()` 的 `except Exception: return 'BLOCK'` — 故障时永久阻断交易

- **文件**: `order_service.py`
- **行号**: L337-353
- **控制流结构**: try/except
- **缺陷类型**: 异常控制流
- **触发条件**: `position_service` 模块未加载、`get_cross_strategy_risk_guard()` 返回 None 时对其调用 `.check()` 导致 AttributeError、或任何瞬态异常（如网络超时导致 Greeks 计算失败）
- **后果**: `_check_cross_strategy_risk()` 被 `send_order` (L213-218) 和 `box_spring_strategy._check_cross_strategy_risk` (L1305-1325) 两处调用。任何异常（包括瞬态网络抖动）都会返回 `'BLOCK'`，**永久阻断所有 OPEN 方向订单**，直到异常原因被清除。在实盘交易中，这意味着即使是短暂的数据服务抖动也会导致无法开仓。

- **修复建议**: 实现异常计数器和自动恢复机制；区分持久性故障（如模块缺失）和瞬态故障（如网络超时）；持续失败超过阈值时才真正阻断。

### P1-4: `signal_service.py` 过滤管道中各过滤器之间无故障隔离

- **文件**: `signal_service.py`
- **行号**: L236-306（generate_signal 中的多个连续过滤块）
- **控制流结构**: 顺序 if 块（PLR → ModeEngine → 冷却 → 决策分数 → HFT → 自适应）
- **缺陷类型**: 分支覆盖缺口
- **触发条件**: 中间过滤器（如 `ModeEngine` L243-255）的 `except Exception: pass` 静默吞掉异常后，后续过滤器继续执行
- **后果**: 此处行为是双重性的——
  - **有利**: 单个过滤器故障不会阻断整个信号管道
  - **有害**: 如果 ModeEngine 因 `_config` 为 None（从未初始化）而静默失败，PLR 过滤和信号强度过滤都会被跳过，**低质量信号不被过滤直接进入下单**
- **代码**:
  ```python
  # L243-255: ModeEngine 过滤
  try:
      _me = ModeEngine.get_instance()
      _passed, _reason = _me.filter_signal_by_mode(...)
      if not _passed:
          return None  # 被过滤
  except Exception:
      pass  # ← 异常时静默跳过过滤！
  ```

- **修复建议**: 在 `except` 块中添加降级行为——例如 `logging.warning` 并采用保守过滤（直接阻断信号），而非完全跳过过滤。

### P1-5: `risk_service.py` 仪表盘定时器存在递归`Timer`链，停止后无清理导致资源泄漏

- **文件**: `risk_service.py`
- **行号**: L193-212
- **控制流结构**: `Timer` 回调 → 执行 → 创建新 `Timer`（递归调度链）
- **缺陷类型**: 无限循环风险 / 回调与事件循环
- **触发条件**: `RiskService` 实例被销毁后，`_dashboard_timer` 的引用丢失，但最后一个 Timer 仍在后台等待触发
- **后果**: 如果 `RiskService` 被停止/重建（如策略重启），旧的 Timer 线程可能继续运行并尝试调用已销毁对象的方法，导致：
  - `AttributeError`（访问已清空的属性）
  - 内存泄漏（Timer 线程永远不退出）
  - 新的 RiskService 实例启动后，存在两个 Timer 竞争，产生双倍日志输出

- **修复建议**: 为 RiskService 添加 `stop()` 方法，在其中调用 `self._dashboard_timer.cancel()` 并设置标志位；在 `_dashboard_timer_callback` 中检查标志位，避免已停止后继续调度。

## P2 级缺陷（代码可读性/维护性）

### P2-1: `mode_engine.py` `_rollback()` 使用 if/elif 链处理组件回滚 — 无默认分支

- **文件**: `mode_engine.py`
- **行号**: L974-1014
- **控制流结构**: if/elif/elif/elif 链（4个显式分支，无 else）
- **缺陷类型**: 分支覆盖缺口
- **触发条件**: 新增第5个受传播组件时，如果忘记在 `_rollback` 中为其添加 elif 分支，则该组件不会被回滚
- **后果**: 当前4个组件均有对应分支，无实际运行期影响。但如果未来扩展（如添加第5个 `_apply_to_*` 方法），回滚逻辑会遗漏新增组件。
- **修复建议**: 将回滚逻辑改为 **注册-闭包模式**：在 `_apply_to_*` 中同时注册回滚闭包，`_rollback` 只需遍历回滚列表执行即可。

### P2-2: `box_spring_strategy.py` `_infer_direction()` 硬编码魔法数字阈值

- **文件**: `box_spring_strategy.py`
- **行号**: L520-526
- **控制流结构**: if/elif/else
- **缺陷类型**: 死条件与矛盾条件（潜在风险，当前无矛盾）
- **触发条件**: `self._direction_buy_call_threshold` (0.45) > `self._direction_buy_put_threshold` (0.55) 的配置错误

### P2-4: `signal_service.py` 信号去重集合使用线性扫描清理过期键

- **文件**: `signal_service.py`
- **行号**: L230-233
- **控制流结构**: if 条件 + for 循环清理
- **缺陷类型**: 循环正确性
- **触发条件**: `_signal_dedup_set` 中的键数超过 500（高交易频率、多合约场景）
- **后果**: 每次超过 500 键时，遍历全量字典检查过期——500 个键的线性扫描对性能影响很小（<1ms），但在极端场景（>10000 键）下每次信号生成都可能触发 O(n) 清理，成为微秒级延迟敏感路径上的不可预测性能瓶颈。
- **当前评估**: P2 级别，因为 `_signal_dedup_ttl=5.0s` 且在大多数场景下键数不会远超 500。
- **修复建议**: 使用 `OrderedDict` 按插入时间排序，只清理最早插入的键。

### P2-7: `shadow_strategy_engine.py` `process_signals_batch()` 中 `_last_summary_time` 在锁外更新

- **行号**: L979-982
- **控制流结构**: 时间条件检查 + 无锁更新
- **缺陷类型**: 不可达代码（竞态条件下的双重汇总）
- **触发条件**: 两个线程几乎同时调用 `process_signals_batch()`
