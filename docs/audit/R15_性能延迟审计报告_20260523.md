## 2. 维度一：热路径性能瓶颈

### 2.1 [P0-BP-01] subscription_manager.on_tick 在 `_tick_lock` 内执行全部路由逻辑

- **文件**: `subscription_manager.py`
- **行号**: 1102-1115
- **问题描述**: `on_tick()` 方法在第1112行获取 `self._tick_lock` 后，在第1115行调用 `_on_tick_impl()`。`_on_tick_impl()` 内部执行了完整的 metadata 查找（`params_service.get_instrument_meta_by_id`）、T-Type Service 转发、错误熔断记录等。这意味着在 `_tick_lock` 持有期间执行了**字典查找**、**跨模块调用**（T-Type Service 的 `on_option_tick`/`on_future_instrument_tick`）、以及**错误计数与日志写入**。该锁成为整个 tick 处理链的串行化瓶颈。
- **影响系统**: 生产系统
- **严重等级**: **P0** — 数千个合约同时产生 tick 时，`_tick_lock` 持有时间过长会导致订阅管理器层面全局阻塞，使所有后续 tick 排队，直接增加处理延迟，导致后续 tick 积压。
- **实际代码**:
```python
# subscription_manager.py:1102-1115
def on_tick(self, instrument_id: str, last_price: float, volume: float = 0) -> None:
    with self._tick_lock:                          # line 1112: 全局锁
        if not self._bg_threads_started:
            self._start_background_threads()
        return self._on_tick_impl(instrument_id, last_price, volume)  # line 1115: 锁内执行全部路由
```

### 2.2 [P0-BP-02] strategy_tick_handler._dispatch_tick 中 position_service.on_tick 持有 instrument lock 时执行 np.polyfit

- **文件**: `position_service.py`
- **行号**: 377-436
- **问题描述**: `on_tick()` 在第396行获取合约级 `_get_instrument_lock(inst_id)` 后，迭代该合约的所有持仓记录（每 tick 对每个持仓都执行），并在 `_profit_history` 长度 ≥ 5 时执行 `np.polyfit()`（第417行）或等价的纯 Python 线性回归（第422-433行）。这意味着 **在锁内执行了 O(n×m) 操作**（n=持仓数，m=拟合计算量），直接阻塞了该合约的止盈止损检查和并发 tick 处理。

- **严重等级**: **P0** — 同一合约有大量持仓（5+ 个 position），每个 tick 都触发 `np.polyfit`，锁持有时间可达数百微秒甚至毫秒级，直接增加 tick 处理延迟，可能导致后续 tick 积压并造成滑点。

# position_service.py:396-433

### 2.3 [P1-BP-03] strategy_tick_handler._process_tick 中 on_tick_feed 到 OrderFlowBridge 每条 tick 都执行

- **文件**: `strategy_tick_handler.py`
- **行号**: 300-314
- **问题描述**: 每条 tick 都调用 `self._order_flow_bridge.on_tick_feed()`（第307行），该方法内部执行了：
  - `_infer_direction()` 方向推断
  - `MicrostructureAnalyzer.on_trade()` 微结构分析
  - `VolumeWeightedOrderFlow.on_trade()` 成交量加权分析
  - `MicrostructureArbitrageDetector` 套利检测

- **影响系统**: 生产系统
- **严重等级**: **P1** — 不影响丢单但显著增加每 tick 处理时间，影响整体吞吐量。

## 3. 维度二：数据库查询性能

### 3.1 [P1-DB-01] storage_core.py tick writer 循环中使用 `time.sleep(0.001)` 忙等

- **文件**: `storage_core.py`
- **行号**: 420-433
- **问题描述**: tick shard writer 循环（`_tick_shard_writer_loop`）中，当 batch 未满且 shard 队列非空时，使用 `time.sleep(0.001)`（1ms）进行忙等。在 Windows 上，`time.sleep(0.001)` 的实际精度约为 15.6ms（系统时钟粒度为 15.6ms），这意味着实际等待时间远大于预期，反而造成不必要的 15ms 延迟。同时，1000 ticks/sec 时该 writer 线程会频繁唤醒睡眠，增加系统调度开销。
- **影响系统**: 生产系统
- **严重等级**: **P1** — 在 Windows 平台下造成不稳定的写入延迟。应使用 `queue.get(timeout=0.1)` 或条件变量替代 `time.sleep`。
- **实际代码**:
```python
# storage_core.py:432-433
else:
    time.sleep(0.001)  # Windows下实际≈15.6ms
```

### 3.2 [P1-DB-02] storage_core.py K-line 批次间插入 `time.sleep(0.01)` 延迟

- **文件**: `storage_core.py`
- **行号**: 607-608
- **问题描述**: 在 `_flush_batch_to_db` 中，当 K-line 批次被拆分为多个子批次时（`DEFAULT_MAX_SUB_BATCH` 默认 500），每个子批次之间插入 `time.sleep(0.01)`。虽然在离线/回测场景影响不大，但在实时交易时如果 K-line 写入量大，这会在 3 层写入锁持有期间造成 10ms 的额外延迟。而在 Windows 下 `time.sleep(0.01)` 实际精度为 15.6ms。
- **影响系统**: 生产系统
- **严重等级**: **P1** — 影响实时 K-line 写入延迟。

### 3.3 [P2-DB-03] storage_query.py fetchdf 无显式 LIMIT

- **文件**: `storage_query.py`
- **行号**: 全文
- **问题描述**: DuckDB 的 `fetchdf()` 调用未检查查询是否包含显式 LIMIT 子句。在大数据集（如全量 tick 表、全量 K 线表）上执行不带 LIMIT 的 fetchdf 可能导致内存溢出和长时间阻塞。虽然 `storage_query.py` 主要用于非实时分析查询，但缺少防护措施。
- **影响系统**: 生产系统（存储查询模块）
- **严重等级**: **P2** — 离线分析场景可能导致 OOM。

### 3.4 [P2-DB-04] 缺少 DuckDB 索引审计

- **文件**: `storage_core.py`, `ds_db_connection.py`
- **行号**: 全局
- **问题描述**: 审计发现代码中没有显式的 DuckDB 索引创建语句（如 `CREATE INDEX`）。虽然 DuckDB 在某些列上自动创建索引（如主键），但对于高频查询维度（如 `timestamp`, `instrument_id`, `product`），缺少索引会导致全表扫描。特别是 tick 表在高频写入后可能有数百万行，查询时性能下降。

- **文件**: `storage_core.py`

## 4. 维度三：内存分配热点

### 4.1 [P0-MEM-01] position_service.py on_tick 中 `list(self.positions[inst_id].items())` 每 tick 创建列表副本

- **文件**: `position_service.py`
- **行号**: 398
- **问题描述**: 每 tick 对合约的所有持仓执行 `list(self.positions[inst_id].items())`，创建所有持仓的元组列表副本。这是一个 O(n) 的内存分配操作（n = 该合约的持仓数），每 tick 都会分配新列表。如果同一合约有两笔持仓且 tick 频率为 1000/s，则每秒创建 1000 个列表 → 数百万次内存分配/释放。
- **影响系统**: 生产系统
- **严重等级**: **P0** — 高频场景下直接导致显著的内存分配/GC 压力，延长 tick 处理延迟。
- **实际代码**:
```python
# position_service.py:396-398
with self._get_instrument_lock(inst_id):
    if inst_id in self.positions:
        for pid, record in list(self.positions[inst_id].items()):  # ← 每 tick 分配新 list
```

- **文件**: `strategy_tick_handler.py`
- **行号**: 669-679
- **问题描述**: 与 [P1-BP-05] 描述相同。每个 tick 都在堆上分配一个 9 字段的字典。1000 tick/s → 每秒 1000 个 dict 分配。
- **影响系统**: 生产系统
- **严重等级**: **P1**

### 4.3 [P1-MEM-03] position_service.py _profit_history 列表无界增长

- **文件**: `position_service.py`
- **行号**: 411-413
- **问题描述**: 每个持仓记录的 `_profit_history` 列表在每个 tick 都 `append(profit_pct)`，且没有大小限制（仅当长度 ≥ 5 时计算斜率）。如果持仓从开盘到收盘（例如 4 小时 = 14400 秒，每秒一个 tick），该列表将增长到 14400 个元素，每条记录约 115KB。且 `np.polyfit` 在每次 tick 时对全量历史数据重新拟合，复杂度 O(n) 随时间增长。
- **影响系统**: 生产系统
- **严重等级**: **P1** — 持仓时间长时，内存和计算时间线性增长。

### 4.4 [P1-MEM-04] strategy_ecosystem 中 6 个 deque 预分配 maxlen=10000

- **文件**: `strategy_ecosystem.py`
- **行号**: 290-295
- **问题描述**: 6 个策略槽（master, reverse, other, spring, arbitrage, market_making）各自预分配了 `maxlen=10000` 的 deque。即使某些策略永不激活，这些 deque 仍占用内存（每个 deque 初始为空，但 Python deque 的底层数组会预分配一定容量）。6 × 10000 槽位 = 60000 条记录的潜在存储，每条交易记录数十个字段。
- **影响系统**: 生产系统
- **严重等级**: **P1** — 内存预分配浪费，活跃策略少时尤其明显。

### 4.5 [P2-MEM-05] strategy_tick_handler._dispatch_tick_degraded 每 tick 创建 tick_data dict

- **文件**: `strategy_tick_handler.py`
- **行号**: 411-417
- **问题描述**: 在历史加载降级分发路径中，每条降级 tick 也创建字典。虽然降级路径只在历史加载期间触发，但历史加载期间可能有大量 tick，同样造成内存分配压力。
- **影响系统**: 生产系统

### 4.6 [P2-MEM-06] order_flow_bridge 缓存字典

- **文件**: `order_flow_bridge.py`
- **行号**: 36-38, 40-42

- **严重等级**: **P2** — 内存持续增长。

## 5. 维度四：算法复杂度问题

### 5.1 [P1-ALG-01] position_service.on_tick 中 np.polyfit 在每次 tick 重拟合全量利润历史

- **文件**: `position_service.py`
- **行号**: 414-433
- **问题描述**: 当 `_profit_history` 长度 ≥ 5 时，代码调用 `np.polyfit(range(len(history)), history, 1)`，对**全量**历史数据执行最小二乘拟合。该操作复杂度为 O(n)（n = profit_history 长度），且每个 tick 都重新计算。使用 Welford 在线算法或增量线性回归可以在 O(1) 时间内完成同样的计算（维护 sum_x, sum_y, sum_xy, sum_x2）。
- **影响系统**: 生产系统
- **严重等级**: **P1** — 持仓时间长时，每个 tick 的拟合计算量线性增长。

### 5.2 [P1-ALG-02] 缺少 Black-Scholes / IV 计算结果缓存

- **文件**: `greeks_calculator.py`
- **行号**: 105-157（_bs_greeks）、166-224（IVCalculator）、246-324（BinomialTreePricer）
- **问题描述**:

  - `IVCalculator.implied_volatility()` 使用牛顿迭代法（最多 100 次迭代），没有缓存 IV 结果。相同参数会重复进行数百次 `_bs_price` 和 `_bs_greeks` 调用。

- **影响系统**: 生产系统
- **严重等级**: **P1** — 期权 Greeks 计算在实时场景中可能频繁调用，无缓存导致重复计算。

### 5.3 [P1-ALG-03] position_service 止盈止损检查 O(n×m) 复杂度

## 6. 维度五：GC压力

### 6.1 [P1-GC-01] Tick 热路径中大量短生命周期对象

- **文件**: `strategy_tick_handler.py`, `position_service.py`
- **行号**: 647-735, 669-679, 398-433
- **问题描述**: 汇总上述发现，每条 tick 的热路径创建了以下短生命周期对象：

  6. f-string 格式化产生的临时字符串（如 logging 调用）
  这些对象的生命周期极短（一次 tick 处理），在 1000+ ticks/s 的场景下对 Python GC 造成持续压力。
- **影响系统**: 生产系统
- **严重等级**: **P1** — 高频场景下 GC 停顿可能达数十毫秒，导致 tick 延迟抖动。

### 6.2 [P1-GC-02] storage_core writer 循环中 dict 累积

- **文件**: `storage_core.py`
- **行号**: 458-494
- **问题描述**: `_async_writer_loop` 中的 `batch` 列表累积 tick 任务 dict，每次 flush 后调用 `batch.clear()`。但 `batch.clear()` 仅清空引用，已分配的内存不会立即释放给 OS。长时间运行后，worker 线程的 batch 列表内部数组可能一直维持在峰值大小（如 batch_tasks=1000）。
- **影响系统**: 生产系统
- **严重等级**: **P1** — 内存占用偏高。

### 6.3 [P2-GC-03] order_flow_bridge 高频缓存过期机制

- **文件**: `order_flow_bridge.py`
- **行号**: 41-42
- **问题描述**: `_flow_cache_ttl = 5.0` 秒的缓存 TTL 意味着每 5 秒缓存过期，所有新查询需要重新计算。但如果 tick 频率为 1000/s，5 秒内可能有 5000 次查询触发缓存重新计算，导致旧缓存对象被 GC 回收。使用时间戳而不是基于事件驱动的过期策略更高效。
- **影响系统**: 生产系统
- **严重等级**: **P2**

### 6.4 [P2-GC-04] 无显式 `__del__` 中的耗时操作

- **文件**: 全量 `.py` 文件
- **描述**: 审计全量代码后未发现 `__del__` 中包含 I/O 操作或其他耗时操作的实例。`__del__` 的使用是安全的。✅ **此项检查通过。**
- **影响系统**: 无
- **严重等级**: **P2** — 检查通过，无问题。

## 7. 维度六：线程/锁竞争

### 7.1 [P0-TH-01] subscription_manager._tick_lock 作为全局串行化锁持有时间过长

- **文件**: `subscription_manager.py`
- **行号**: 1112-1115
- **问题描述**: 与 [P0-BP-01] 从不同维度分析同一问题。`_tick_lock` 是一个 `threading.Lock()`（非 RLock），在持有期间执行完整的元数据查找、T-Type Service 转发、错误熔断等操作。单合约场景下问题不大，但多合约（数百种期货+数千期权）并发 tick 时，`_tick_lock` 成为系统级的**全局串行化点**。所有 tick 必须排队通过此锁。
- **影响系统**: 生产系统
- **严重等级**: **P0** — 全局锁是系统吞吐的根本性瓶颈。
- **实际代码**:
```python
# subscription_manager.py:1112-1115
with self._tick_lock:  # 全局串行化锁
    if not self._bg_threads_started:
        self._start_background_threads()
    return self._on_tick_impl(instrument_id, last_price, volume)
```

### 7.2 [P1-TH-02] strategy_tick_handler._dispatch_tick 中 shard_lock 内执行多层操作

- **文件**: `strategy_tick_handler.py`
- **行号**: 685-706
- **问题描述**: `shard_lock` 在持有期间执行了：
  - dict.setdefault 操作（第686行）
  - buffer.append（第686行）
  - len() 计算（第688行）
  - time.time() 调用（第692行）
  - 条件判断和 buffer clear（第697-698行）
  虽然不是 I/O 操作，但在 shard 被 flush 时锁持有时间延长。虽然有分片设计降低了竞争概率，但同 shard 内的 tick 仍需排队。
- **影响系统**: 生产系统
- **严重等级**: **P1** — 分片内竞争。

### 7.3 [P1-TH-03] storage_core 三级写入锁设计合理但 tick 写入锁粒度偏粗

- **文件**: `storage_core.py`
- **行号**: 619-628
- **问题描述**: 写入操作分为三级锁：`_db_tick_write_locks`（分片）、`_db_kline_write_lock`、`_db_maintenance_write_lock`。tick 分片锁设计是合理的（按 shard 索引分散竞争），但 `_flush_sub_batch` 在持有 tick write lock 期间执行了 merge 操作和多次 `method(*args, **kwargs)` 调用。merge 操作应尽可能在锁外完成。
- **影响系统**: 生产系统
- **严重等级**: **P1** — 写入锁内执行 merge 延长锁持有时间。

### 7.4 [P2-TH-04] strategy_tick_handler.on_tick 中使用 RLocks 的开销

- **文件**: `strategy_tick_handler.py`
- **行号**: 多处
- **问题描述**: `_probe_lock`, `_lock`, `_tick_stats_lock` 均为 `threading.RLock()`。RLock 比普通 Lock 有额外开销（需要跟踪持有线程和重入计数）。在热路径中，如果不需要重入特性，使用普通 Lock 更高效。
- **影响系统**: 生产系统
- **严重等级**: **P2** — 微小优化。

### 7.5 [P2-TH-05] position_service 使用 RLock 分片锁

- **文件**: `position_service.py`
- **行号**: 242-243, 297-299
- **问题描述**: `position_locks` 是 `Dict[str, threading.RLock]`（按合约分片），初始化使用 `self.global_lock`（Rlock）保护。分片设计降低了竞争，RLock 允许同线程重入。但在 `on_tick` 场景中不需要重入特性，使用普通 Lock 即可。
- **影响系统**: 生产系统
- **严重等级**: **P2** — 微小优化。
