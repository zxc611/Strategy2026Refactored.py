# ali2026v3_trading 类似功能优化方案与风险评估

**版本：** v1.0  
**日期：** 2026-06-09  
**依据：** 《类似功能排查文档报告.md》（22 项类似点）  

---

## 一、优化总体策略

采用 **"先内后外、先高后低、 Facade 先行、工具下沉"** 的四步策略：

1. **先内后外**：优先消除 M1 生产模块内部的高风险重复（S-04/05/19），再处理跨 M1/M2/M3 的重复。
2. **先高后低**：按风险等级排序，高风险项必须在本轮处理，中风险项纳入下一轮迭代。
3. **Facade 先行**：对现有 Facade 进行彻底薄壳化，删除内联实现，改为纯委托或纯 re-export。
4. **工具下沉**：将跨模块通用的算法（滑点、MaxDD、Cooldown、Tick 聚合）提取到 `infra/shared_utils.py` 或新建 `infra/calculation_engine.py`。

## 二、高风险项优化方案（4 项）

### OPT-1 S-04 WAL 预写日志服务统一

**现状：** `OrderWALStateService` 与 `OrderPersistenceService` 存在 30-35 行逐行重复的核心 WAL 方法。

**优化方案：**
1. **保留 `OrderWALStateService` 作为唯一实现**，删除 `OrderPersistenceService` 中的 WAL 相关方法（`wal_write`、`wal_read`、`wal_delete`、`_rotate_jsonl`）。
2. **`OrderPersistenceService` 转型为 Facade**，通过 `__getattr__` 将 WAL 调用委托到 `OrderWALStateService`。
3. **孤儿订单恢复逻辑统一到 `OrderWALStateService`**，删除 `OrderPersistenceService` 中的恢复方法。

**代码变更估算：**
- 删除/简化 `order_persistence.py`：约 -80 行
- 修改 `order_service.py` 的调用链：约 -5 行（直接引用 `OrderWALStateService`）

**验证方法：**
- `py_compile` 全量编译
- 12/12 子系统 import smoke test
- 订单状态恢复单元测试（模拟崩溃后恢复）

**风险评估：**

| 风险项 | 等级 | 说明 | 缓解措施 |
|--------|------|------|----------|
| 恢复行为变更 | 中 | 统一恢复逻辑后，边缘 case 处理可能不同 | 保留旧恢复逻辑作为注释对比，运行恢复单元测试 |
| 上层调用断裂 | 低 | `OrderService` 若直接调用 `OrderPersistenceService.wal_write` | Facade 委托保持接口兼容 |
| 性能回退 | 低 | 增加一次委托调用 | 实测 WAL 写入耗时（应 <1ms） |

---

### OPT-2 S-05 RiskService 彻底 Facade 化

**现状：** `RiskService` 保留了 20+ 个对 `RiskCheckService` 的纯委托方法，以及大量 SafetyMeta 子服务的内联调用。

**优化方案：**
1. **删除所有纯委托方法**：`check_before_trade`、`_check_safety_meta_layer`、`_check_operational_risks` 等 20+ 个方法全部删除。
2. **通过 `__getattr__` 动态委托**：在 `RiskService` 中实现 `__getattr__`，将未定义属性委托到 `self._check_service` 或 `self._safety_meta_layer`。
3. **公开 API 白名单化**：`RiskService.__all__` 仅保留业务入口（如 `evaluate_signal`、`on_position_update`），所有底层检查方法通过 `_check_service.xxx` 显式访问。

**代码变更估算：**
- 删除 `risk_service.py` 纯委托方法：约 -80 行
- 新增 `__getattr__` 动态委托：约 +10 行
- 修改外部调用者（如有直接调用 `_check_xxx` 的）：需全局搜索替换

**验证方法：**
- 14/14 关键符号可达性测试（特别关注 `check_before_trade` 的可达路径）
- 风控规则变更集成测试

**风险评估：**

| 风险项 | 等级 | 说明 | 缓解措施 |
|--------|------|------|----------|
| 外部调用断裂 | **高** | 其他模块可能直接调用 `RiskService.check_before_trade` | 全局 grep 确认所有调用点；`__getattr__` 保持运行时兼容；分两步走（先标记 deprecated，下轮删除） |
| 调试困难 | 中 | `__getattr__` 动态委托增加堆栈深度 | 添加 `__qualname__` 透传；文档明确委托链 |
| 循环依赖 | 低 | `__getattr__` 可能触发 `__getattr__` 递归 | 使用 `object.__getattribute__` 避免递归 |

### OPT-3 S-19 SafetyMeta 遗留代码清理

**现状：** `risk/risk_circuit_breaker.py` 保留了重构前完整的 `SafetyMetaLayer`（500+ 行），与拆分后的 4 个子文件重复。

**优化方案：**
1. **确认 `risk_circuit_breaker.py` 的引用关系**：全局搜索 `from risk.risk_circuit_breaker import SafetyMetaLayer` 的调用点。
2. **若调用点均为内部文件**：将 `risk_circuit_breaker.py` 中的 `SafetyMetaLayer` 改为纯 re-export（`from risk.safety_meta_layer import SafetyMetaLayer`）。
3. **若调用点为零**：直接删除 `risk_circuit_breaker.py` 中的 `SafetyMetaLayer` 类定义，保留文件作为 Facade 空壳或合并到 `safety_meta_layer.py`。

**代码变更估算：**
- 删除/简化 `risk_circuit_breaker.py`：约 -400 行
- 修改 import 路径：约 -5 处

**验证方法：**
- 全局 import smoke test（特别关注 risk/ 子系统）
- `py_compile` 全量编译

**风险评估：**

| 风险项 | 等级 | 说明 | 缓解措施 |
|--------|------|------|----------|
| 隐藏引用断裂 | 中 | 可能存在动态 import（`__import__`、字符串反射） | 全局搜索 `risk_circuit_breaker`、`SafetyMetaLayer` 字符串 |
| 行为差异 | 低 | 旧 `SafetyMetaLayer` 与新拆分实现可能存在细微差异 | 对比单元测试用例，确保覆盖所有分支 |

### OPT-4 S-03 滑点计算统一引擎

**现状：** 生产侧 `compute_slippage_bps` 与回测侧 `_compute_dynamic_slippage_bps` 核心公式一致但参数命名和扩展性不同，分散在 6 个文件中。

**优化方案：**
1. **新建 `infra/slippage_engine.py`**：提取统一滑点计算核心，支持以下参数：
   - `base_slippage_bps`
   - `spread_bps`
   - `quality_scale` / `liquidity_tier`
   - `instrument_mult`
   - `expiry_mult`
   - `premium_bps`（可选，回测专用）
   - `cascade_bps`（可选，回测专用）
2. **生产侧适配**：`infra/shared_utils.py` 的 `compute_slippage_bps` 改为对 `SlippageEngine` 的薄封装，保持接口不变。
3. **回测侧适配**：`param_pool/backtest_loop_slippage.py` 的 `_compute_dynamic_slippage_bps` 改为调用 `SlippageEngine`，保留回测特有的级联滑点和流动性分层逻辑作为预处理层。
4. **删除重复定义**：清理 `backtest_runner_utils.py`、`backtest_loop_core.py`、`backtest_position_manager.py`、`backtest_runner_base.py`、`backtest_state.py` 中的内联滑点函数，统一 import `SlippageEngine`。

**代码变更估算：**
- 新增 `infra/slippage_engine.py`：约 +40 行
- 简化 `infra/shared_utils.py`：约 -15 行
- 简化 `param_pool/backtest_loop_slippage.py`：约 -20 行
- 删除 5 个文件中的内联函数：约 -25 行

**验证方法：**
- 滑点计算单元测试（生产场景 + 回测场景对比基准值）
- 回测保真度评分验证（确保滑点模型行为不变）

**风险评估：**

| 风险项 | 等级 | 说明 | 缓解措施 |
|--------|------|------|----------|
| 回测保真度偏差 | **高** | 统一后回测滑点结果可能与旧实现存在浮点差异 | 建立滑点基准测试（golden master），对比统一前后输出 |
| 生产滑点突变 | 中 | 生产侧引入 `premium_bps` 等回测参数，可能误用 | `SlippageEngine` 默认参数保持生产旧行为，`premium_bps` 默认 0 |
| 级联依赖 | 低 | 6 个文件同时修改，可能遗漏 | 全局 grep `_compute_dynamic_slippage_bps` 和 `compute_slippage_bps`，确保全部替换 |

## 三、中风险项优化方案（6 项）

### OPT-5 S-08 Greeks 聚合统一服务

**方案：** 新建 `governance/greeks_aggregation_service.py`，统一实现 `aggregate_greeks(positions, source='cached')`。`position_greeks.py`、`greeks_calculator.py`、`risk_service.py` 均委托到此服务。

**变更估算：** +30 行，-30 行（三处简化）  
**风险：** 低（纯计算逻辑，无副作用）

### OPT-6 S-10 Config 参数读取统一

**方案：** `RiskConfigProvider` 删除独立缓存链，改为从 `ConfigService` 或 `get_cached_params` 读取。保留 `RiskConfigProvider` 作为 Facade，仅增加风险参数的语义封装（如 `_get_signal_cooldown` 变为 `return get_cached_params().get('signal_cooldown_sec', 60.0)`）。

**变更估算：** -20 行（删除独立缓存），+5 行（委托）  
**风险：** 低（配置读取无副作用）

---

### OPT-7 S-11 TickAggregator 核心提取

**方案：** 新建 `infra/tick_aggregation_core.py`，提取 OHLCV 计算、时间窗口/成交量窗口判断逻辑。生产侧和回测侧分别保留 I/O 适配层（实时 tick 流 vs 历史 tick 文件）。

**变更估算：** +25 行，-20 行  
**风险：** 中（回测数据预处理行为需严格一致）

---

### OPT-8 S-12 PositionManager 回测复用

**方案：** 评估 `param_pool/backtest_position_manager.py` 与 M1 `PositionCommandService` 的接口差异。若差异主要为数据访问层（回测内存状态 vs 生产数据库），则通过依赖注入统一业务逻辑，回测侧提供 `MockPositionDataAccess`。

**变更估算：** 视接口差异程度，预计 +50 行适配器，-40 行回测内联逻辑  
**风险：** 中（回测持仓行为变更可能影响历史回测结果可复现性）

### OPT-9 S-02 MaxDD 统一工具函数

**方案：** 提取 `infra/shared_utils.py` -> `compute_max_drawdown(equity_curve, use_numpy=True)`。向量化版作为默认，循环版作为 `numpy` 不可用的 fallback。

**变更估算：** +15 行，-10 行（两处简化）  
**风险：** 低（纯数学计算）

### OPT-10 S-06 Cooldown 统一

**变更估算：** +5 行，-5 行

### OPT-11 S-15 Commission 统一计算器

**变更估算：** +30 行，-10 行  
**风险：** 中（生产 PnL 计算可能因费率精度变化产生微小差异）

## 五、整体风险评估

### 5.1 不优化的风险（维持现状）

| 风险项 | 等级 | 说明 |
|--------|------|------|
| 维护成本递增 | **高** | 同一变更需修改 3-6 处，人为遗漏概率指数上升 |
| 生产/回测偏差 | **高** | 滑点、佣金、Cooldown 等核心参数在生产与回测中实现不同，导致回测失真 |
| 代码理解成本 | 中 | 新开发者难以判断应在哪个模块实现功能，容易在错误位置新增重复代码 |
| 技术债务累积 | 中 | 每次重构都保留旧实现作为"兼容层"，系统体积持续膨胀 |

### 5.2 优化的风险（执行变更）

| 风险项 | 等级 | 说明 | 缓解措施 |
|--------|------|------|----------|
| 功能回归 | **高** | Facade 化过程中可能误删仍在使用的接口 | 全局 grep 所有调用点；分两步 deprecated + 删除；全量单元测试 |
| 回测结果不可复现 | 中 | 滑点、佣金、Tick 聚合统一后，历史回测结果可能产生微小差异 | 建立 golden master 基准测试；记录版本差异说明 |
| 性能回退 | 低 | 增加委托链和统一引擎可能引入微小延迟 | 对 WAL 写入、滑点计算、Greeks 聚合进行性能基准测试 |
| 循环依赖 | 低 | 统一引擎下沉到 infra 可能引发新的跨模块依赖 | 严格遵循 infra 不依赖业务模块的原则；新建引擎仅依赖基础类型 |

### 5.3 风险矩阵

```
            影响低    影响中    影响高
概率高      [性能]    [回归]    [维护成本↑]
概率中      [循环依赖] [不可复现] [生产/回测偏差]
概率低      —         [理解成本] [技术债务]
```

**结论：** 不优化的长期风险远高于优化执行的短期风险。建议按 Phase 1→4 分阶段实施，每阶段结束后进行全量验证，确保零回归。

## 六、验收标准

1. **编译验证**：819 文件零编译错误（维持现有水平）。
2. **Import 验证**：12/12 子系统 import PASS，14/14 关键符号可达性 PASS。
3. **消减量验证**：累计消减 600-800 行重复代码，不引入新的重复点。
4. **行为一致性验证**：
   - 滑点计算：生产与回测同一组输入输出偏差 < 0.01 bps
   - MaxDD 计算：向量化版与循环版同一权益曲线输出偏差 < 1e-9
   - Greeks 聚合：三模块同一持仓输出偏差 < 1e-6
5. **回测保真度**：分钟 Bar 回测保真度维持 75-80 分，不下降。
