## 二、发出交易信号环节（新发现）

### CHAIN-BUG-1 (P1): signal 字典 `volume` 字段读取市场成交量而非下单手数

- **文件**: `strategy/strategy_business_layer.py:221`
- **问题代码**:
  ```python
  signal = {
      'symbol': t.get('instrument_id', ''),
      'direction': t.get('direction', 'BUY'),
      'price': t.get('price', 0.0),
      'volume': t.get('volume', 0),   # ← 读取的是市场成交量，不是下单手数
      ...
      'action': t.get('action', '') or 'OPEN',
      'signal_id': _sig_id,
  }
  check_result = rs.check_before_trade(signal)
  ```
- **根因**: `select_from_sort_bucket` 返回的候选 dict 中 `volume` 是**期权市场成交量**（`width_cache_query_mixin.py:210`，来自 `entry.volume`，即日内成交量，可能数千）。而真正下单手数在 `lots` 字段（`width_cache_query_mixin.py:545`，Tier1=2/Tier2=1）。`execute_by_ranking` 正确读取 `target.get('lots', 1)`（`order_executor.py:978`），但**风控检查** `check_before_trade(signal)` 读取的 `signal['volume']` 却是市场成交量。
- **影响**:
  - 风控按市场成交量（数千）评估单笔风险，极易触发"单笔手数超限/保证金不足"而**误阻断**合法的小手数开仓订单。
  - 反之，若某期权市场成交量很小（个位数），风控可能**误放行**本应阻断的订单。
  - 风控检查与实际下单量语义脱节，是开仓链路的真实断点。
- **修复方案**: 将 `signal['volume']` 改为读取 `t.get('lots', 1)`，与 `execute_by_ranking` 的下单量保持一致。
- **验收标准**:
  1. 构造 target（`volume=5000, lots=2`），`check_before_trade` 收到的 `signal['volume']` 为 2。
  2. 风控不再因市场成交量大而误阻断 2 手开仓。

### CHAIN-BUG-2 (P2): target 未显式设置 `direction`，开仓方向隐式默认 'BUY'

- **文件**: `data/width_cache_query_mixin.py:543-555`（targets.append）、`data/width_cache_query_mixin.py:205-215`（select_from_sort_bucket 返回字段）
- **问题**: `select_from_sort_bucket` 返回的 9 个字段中**不含 `direction`**；`select_otm_targets_by_volume` 的 `targets.append({**best, ...})` 也**从未添加 `direction`**。因此：
  - `execute_option_trading_cycle:219` 的 `t.get('direction', 'BUY')` 恒为 `'BUY'`
  - `execute_by_ranking:982` 的 `target.get('direction', direction)` 恒为函数默认 `'BUY'`
- **当前是否正确**: 本策略为**买权策略**（买入 OTM CALL 或买入 OTM PUT，均为 `direction='BUY'`，CALL/PUT 区别编码在 `instrument_id` 中），故**当前功能正确**。
- **风险**: 隐式依赖"永远只买不卖"的不变量，无任何断言或注释保护。一旦未来加入卖权策略或调用方显式传 `direction`，将静默出错。
- **修复方案**: 在 `select_otm_targets_by_volume` 的 `targets.append` 中显式设置 `'direction': 'BUY'`，使买权策略意图自文档化。
- **验收标准**: 返回的每个 target 含 `'direction': 'BUY'` 字段。

但存在以下断点/死代码：

### CHAIN-BUG-3 (P2): `StateTransitionCapture` 状态转换平仓为死代码

- **文件**: `config/state_param.py:365`（`check_position_exit`）、`config/state_param.py:441`（`register_position`）
- **问题**: `check_position_exit` 实现了基于状态转换的止盈/止损/超时平仓（返回 `{'action':'CLOSE', ...}`），但**全代码库无任何调用方**。`register_position` 同样无调用方，导致 `_active_positions` 永远为空，即使调用 `check_position_exit` 也返回 None。
- **影响**: 状态转换平仓层完全失效。系统实际依赖 `position_pnl_service` 的价格SL/TP 和 `position_check_service` 的时间止损，状态转换维度的平仓信号丢失。
- **修复方案**: 本系统已有完善的价格/时间SL/TP 机制，状态转换平仓属冗余设计。**建议保留死代码但添加明确注释**标注"未启用"，避免维护者误以为生效；或后续评估是否接入。本次不删除（删除可能影响测试）。
- **验收标准**: 代码中该处有明确"未启用/死代码"注释，维护者不会被误导。

---

### CHAIN-BUG-4 (P2): `check_last_trading_day` 到期前N天平仓为死代码

- **文件**: `risk/risk_service.py:1271`（`check_last_trading_day`）
- **配置**: `strategy_config_layer.py:590` `last_trading_day_close_ahead_days: 3`
- **问题**: `check_last_trading_day` 方法存在、配置项存在，但**从未被任何平仓触发路径调用**。当前期权平仓仅靠 `_check_option_expiry`（`position_pnl_service.py:104`）在 `days_to_expiry <= 0`（已到期）时触发，**不会在到期前N天提前平仓**。
- **影响**: "到期前3天提前平仓"的风险管理设计未落地。期权临近到期 Gamma 风险放大，未提前平仓可能造成尾部亏损。
- **修复方案**: 本次**不擅自接入**（`last_trading_day_close_ahead_days` 语义可能针对期货最后交易日，需产品确认是否适用于期权）。**建议在 `_check_option_expiry` 中读取该配置，当 `days_to_expiry <= ahead_days` 时触发提前平仓**，由产品确认后实施。本次先以注释形式标注缺口。
- **验收标准**: 缺口在报告中明确记录，待产品确认后实施。

---

### CHAIN-BUG-5 (P3): `signal_id` 未写入 PositionRecord，平仓订单丢失开仓 signal_id

- **文件**: `position/position_service.py:87-112`（PositionRecord 无 signal_id 字段）、`position/position_command_service.py:113`（on_trade 未传 signal_id）、`position/position_command_service.py:451`（平仓 signal_id 回退为 `CLOSE_{instrument_id}`）
- **问题**:
  - `_add_position` 已有 `signal_id: str = ''` 参数（`position_command_service.py:187`），但 `on_trade` 调用 `_add_position(...)` 时**未传 signal_id**（`:113`）。
  - `PositionRecord` 数据类**无 `signal_id` 字段**，构造时（`:306-313`）也未传入。
  - 平仓时 `_trigger_close_position` 执行 `getattr(record, 'signal_id', '') or f"CLOSE_{record.instrument_id}"`（`:451`），因 record 无 signal_id，平仓订单 signal_id 恒为 `CLOSE_{instrument_id}`。
- **影响**: 平仓订单无法通过 signal_id 回溯到原始开仓信号，审计链路断裂（开仓 signal_id 已贯穿 target→order，但平仓端丢失）。
- **修复方案**:
  1. `PositionRecord` 新增 `signal_id: str = ''` 字段
  2. `_add_position` 构造 PositionRecord 时传入 `signal_id=signal_id`
  3. `on_trade` 通过 order_id 查询订单 signal_id 并传入 `_add_position`
  4. `to_dict` 补充 signal_id
- **验收标准**:
  1. 开仓后 `PositionRecord.signal_id` 等于开仓 signal_id。
  2. 平仓订单 signal_id 等于开仓 signal_id（而非 `CLOSE_{instrument_id}`）。

### CHAIN-BUG-6 (P3): 平仓重试与新增开仓的竞态

- **文件**: `position/position_command_service.py:480`（`_schedule_close_retry`，独立线程池执行）
- **问题**: 平仓下单失败后，`_schedule_close_retry` 在独立线程中重试发送 CLOSE 订单，重试前**未重新检查持仓是否已被其他路径平仓**。`record._closing` 标志仅在成功下单后置位（`:464`），在失败到重试之间，止损/止盈可能已在新的 tick 上平掉该持仓。
- **缓解**: `order_executor.py:364` 的 5 秒 CLOSE 去重（`CLOSE_{instrument_id}_{direction}`）提供部分保护，但仅 5 秒窗口。
- **修复方案**: 重试前重新检查 `record.volume != 0` 且 `not record._closing`，已平仓则跳过重试。
- **验收标准**: 持仓已被其他路径平仓时，重试任务不再发送 CLOSE 订单。
