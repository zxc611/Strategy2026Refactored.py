## 一、除零风险审计

### P0-01: 回测引擎Sharpe计算中std(returns)可能为零

- **文件**: `参数池/task_scheduler.py`
- **行号**: 2898
- **问题描述**: `sharpe = np.sqrt(252 * 240) * np.mean(returns) / np.std(returns) if np.std(returns) > 1e-10 else 0.0`
  虽有1e-10保护，但在做市策略回测中（run_backtest做市分支），当所有tick的spread_pnl相同时，returns数组标准差为零，sharpe返回0.0而非合理值。更严重的是，equity_arr中相邻值完全相等时np.diff(equity_arr)/_safe_prev可能产生全零returns数组。
- **严重等级**: P0（实战中可能导致策略评分为0而误杀正常策略）
- **影响系统**: 测试系统（回测引擎）

### P0-02: equity_curve[0]为0时年化收益率计算无保护

- **文件**: `参数池/task_scheduler.py`
- **行号**: 1763-1768
- **问题描述**:
  ```python
  total_ret = (equity_curve[-1] / equity_curve[0] - 1) if equity_curve[0] > 0 else 0.0
  ```
  此处有保护，但1766行 `max_dd_pct = float(np.min(equity_curve / cummax - 1))` 中，
  若equity_curve全为0（策略未交易或初始资金为0），cummax也为0，产生0/0=NaN。
- **严重等级**: P0（回测引擎在极端情况下输出NaN指标，传播到评判系统）
- **影响系统**: 测试系统

### P1-01: Calmar计算中abs(max_dd_pct)可能为零

- **文件**: `参数池/task_scheduler.py`
- **行号**: 1767-1768
- **问题描述**: `calmar = annualized_ret / abs(max_dd_pct)` 使用了 `if abs(max_dd_pct) > 1e-10` 保护，
  但max_dd_pct来自 `np.min(equity_curve / cummax - 1)`，如果该表达式本身产生NaN（见P0-02），
  则abs(NaN) > 1e-10为False，calmar=0.0，掩盖了上游错误。
- **严重等级**: P1（掩盖了上游NaN错误，指标失真）
- **影响系统**: 测试系统

### P1-02: recovery_efficiency计算中equity_curve[0]保护不完整

- **文件**: `参数池/task_scheduler.py`
- **行号**: 1757-1758
- **问题描述**: `recovery_efficiency = dd_depth / (avg_recovery_bars * float(equity_curve[0])) if equity_curve[0] > 0 else 0.0`

### P1-03: governance_engine中线性回归分母可能为零

- **文件**: `governance_engine.py`

## 二、数值溢出审计

### P0-03: equity累加无溢出保护

- **文件**: `参数池/task_scheduler.py`
- **行号**: 1335, 1377, 1409, 1606, 1679等多处
- **问题描述**: `bt.equity += pnl - slip - commission` 在多个平仓路径中执行。
  pnl = (price - pos.open_price) * pos.volume，当pos.volume极大或price差值极大时，
  pnl可能超出float64精度范围。无inf/nan检测直接累加到equity，
  若equity变为inf则后续所有计算崩溃。
- **严重等级**: P0（极端行情下资金计算崩溃）

- **行号**: 120-123
- **问题描述**: `self.cvd += delta` 持续累加，无上限/重置机制。

- **严重等级**: P1（长周期运行精度丢失）
- **影响系统**: 生产系统

### P1-07: _cumulative_weighted_flow指数衰减累加

- **文件**: `order_flow_analyzer.py`
- **行号**: 890-891
- **问题描述**: `self._cumulative_weighted_flow[product] = self._cumulative_weighted_flow.get(product, 0.0) * self._decay + delta`
  decay=0.95，理论上不会溢出，但delta = volume * weight，

### P1-08: HMM观察概率计算中diff*diff可能溢出

## 三、未来数据泄漏审计

### P0-04: evaluate_state_accuracy中使用lookahead_bars访问未来数据

- **文件**: `参数池/task_scheduler.py`
- **行号**: 3998-4033
- **问题描述**:
  ```python
  for i in range(confirm_bars, n - lookahead_bars):
      future_close = closes[min(i + lookahead_bars, n - 1)]
      future_return = (future_close - current_close) / current_close
  ```
  该函数用于L-2参数独立验证，计算状态判定准确率。
  `future_close = closes[min(i + lookahead_bars, n - 1)]` 使用min保护不会越界，
  但i+lookahead_bars可能超过当前Bar的实际数据范围（如果bar_data包含不完整的最新Bar）。
  边界情况：当n - lookahead_bars <= confirm_bars时循环不执行，
  但循环内的min(i + lookahead_bars, n - 1)确保不越界。
  **关键问题**：该函数被`optimize_l2_params()`调用（4118行），而`optimize_l2_params()`
  被参数扫描流程调用。如果bar_data中混入了未来数据（如滚动窗口未正确截断），
  则lookahead_bars指向的确实是"未来"。
- **严重等级**: P0（L-2参数优化可能使用未来数据，导致参数过拟合）
- **影响系统**: 测试系统（参数优化）

## 四、浮点精度误差审计

### P0-05: money = price * multiplier * volume 无溢出保护

- **文件**: `参数池/task_scheduler.py`
- **行号**: 1329
- **问题描述**: `pnl = (price - pos.open_price) * pos.volume`
  期权价格差×合约乘数×手数。当price和open_price差异极大时（极端行情），
  pnl可能超出预期范围。更关键的是，该计算未使用合约乘数（multiplier），
  直接使用了名义价格差×volume。对于期权，实际PnL应该是：
  `(price - open_price) * contract_multiplier * lots`。
  当前实现可能遗漏了contract_multiplier因子。
- **严重等级**: P0（PnL计算遗漏合约乘数，金额错误）
- **影响系统**: 测试系统

### P1-11: 价格比较使用==而非容差

- **文件**: `box_spring_strategy.py`
- **行号**: 62-65
- **问题描述**:
  ```python
  def contains_price(self, price: float) -> bool:
      if self.box_bottom <= 0 or self.box_top <= 0:
          return False
      return self.box_bottom <= price <= self.box_top
  ```
  使用`<=`比较浮点数，在边界情况下（price恰好等于box_bottom或box_top），
  由于浮点精度问题可能判定错误。例如box_bottom=4120.0，price=4120.00000000001，
  应判定为在箱体内，但实际可能被判定为在箱体外。
- **严重等级**: P1（箱体边界判定精度问题）
- **影响系统**: 生产系统

### P1-12: float32精度转换丢失信息

- **文件**: `shared_utils.py`
- **行号**: 52-64
- **问题描述**: `to_float32(value)` 将float64转换为float32。
  对于价格精度（如4120.12），float32的7位有效数字足够，
  但对于微小价格变动（如0.00001）或大数值（如累计PnL 1e8），
  float32精度丢失严重（相对误差~1e-7）。
  代码中to_float32被width_cache使用，如果期权价格使用float32存储，
  在计算价差时可能丢失精度。

## 十、结论

### 总体评价

ali2026v3_trading交易系统在数值精度与数据泄漏方面存在**7项P0级风险**和**26项P1级问题**。

### 最严重发现

1. **PnL计算遗漏合约乘数**（P0-05, P1-14）：回测引擎中PnL计算未乘入期权合约乘数，
   导致盈亏金额量级错误，直接影响策略评判结果。
2. **多进程随机种子共享**（P0-07）：ProcessPoolExecutor环境下numpy随机种子可能共享，
   导致回测结果不可复现或伪随机。
3. **NaN被静默替换**（P0-06）：相关性计算的NaN被替换为0.0，语义错误掩盖了数据质量问题。
4. **未来数据泄漏风险**（P0-04, P1-09, P1-10）：L-2参数优化和walk-forward中可能引入未来信息。

### 建议修复优先级

1. **立即修复（P0）**：合约乘数、NaN处理、随机种子隔离、除零保护
2. **短期修复（P1）**：类型转换安全、累加精度、影子策略种子、全链路对齐
3. **长期改进（P2）**：decimal引入、统一随机种子管理、边界条件完善
