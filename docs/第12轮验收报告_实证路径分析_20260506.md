## 三、根因结论

### Python代码层结论

1. **路径1（init注册）对所有产品完全一致**，成功注册了全部2752个期权
2. **路径2（后台预加载）对所有产品都因 `row['last_price']` 列未命中而失败**
3. **路径3（tick路由）代码完全相同于所有产品**，但在**tick到达 `_on_tick_impl` 层面存在不对称性**
4. **路径4（延迟重订阅）对所有产品都执行了**

**Python代码中没有任何产品相关的分支逻辑能解释为什么AL2606和CFFEX期权收到tick而CU/AU/ZN期权收不到。**

### 关于用户的观察

用户指出"AL期权和指数期权总是成功，其它金属总是失败"——如果这是**可重复的规律**，那说明问题出在平台的**tick推送层面**：

- CFFEX数据源对指数期权(HO/IO/MO)推送tick
- SHFE数据源对部分商品期权(al2606)推送tick，对其他(cu/au/zn)不推送

这与CU/AU期货是否比AL期货更活跃**无关**——期权tick的生成取决于**期权本身是否有做市商报价或成交**，而非对应的期货合约活跃度。

### 修复建议（不再建议等平台修）

既然问题根源不在Python代码能控制的范围内，**以下防御性措施能让系统更健壮**:

1. **TTypeService后台线程改用正确的DB列名**，修复 `row['last_price']` → `row['option_last_price']`，使路径2恢复正常
2. **width_cache.on_option_tick入口加DEBUG日志**，每个tick记录 instrument_id → internal_id
3. **在 init 注册完成后，对所有2752个期权的价格进行交叉验证**，对价格为0的告警
4. **对连续60秒_price == _prev_price的期权发出僵死告警**
