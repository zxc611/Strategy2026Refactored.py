## 三、CRITICAL模块详细评判

### 3.1 data_service.py (2694行/102函数/2类)

- **SRP 1/5**: 严重违反单一职责。同时承担: (1)RealTimeCache缓存管理(2)DuckDB全生命周期管理(连接/建表/索引/视图/Schema迁移)(3)查询缓存(LRU+TTL)(4)数据写入(batch_insert)(5)数据加载(incremental_load)(6)期权同步状态计算+物化视图刷新(200+行SQL)(7)性能监控线程(8)全局配置解析(9)单例工厂+模块级便捷函数。至少应拆分为4-5个独立模块。
- **并发安全 3/5**: RealTimeCache使用分片锁(16)+全局RLock; DataService使用类级RLock单例双重检查。但: `_cache_hits`/`_cache_misses`在分片锁外递增(竞态); `_all_connections`列表无锁保护(并发append/remove, L682); close()中裸except吞异常(L2443)。
- **耦合度 2/5**: 循环依赖: data_service→config_service(模块级L37)→data_service(ensure_products); 延迟导入subscription_manager(L1056)/params_service(L580)。模块级全局变量DB_FILE/PARQUET_PATH在导入时执行I/O。
- **错误处理 2/5**: 1处裸except(L2443); 6处`except Exception: pass`(L630/633/635/639/641/643)静默吞掉DuckDB配置异常; 约50处宽泛`except Exception`; `_get_data_paths_config`(L39)用`except Exception: return None`吞掉所有异常。
- **可测试性 2/5**: 单例模式使隔离测试不可能; 模块级全局变量导入时I/O无法mock; atexit.register副作用; daemon监控线程难以控制。
- **关键问题**:
  1. 裸except(L2443)在close()中吞掉所有异常,连接关闭失败无感知
  2. 6处`except Exception: pass`静默吞掉DuckDB配置异常
  3. `_all_connections`列表无锁保护,多线程并发append/remove
  4. 硬编码路径`D:\ali2026_data\ticks.duckdb`(L50)
  5. 循环依赖: data_service↔config_service
  6. 单例模式使测试隔离和故障恢复困难
- **改进建议**: [P0]消除裸except和`except Exception: pass`; [P0]`_all_connections`加锁; [P1]拆分为RealTimeCache/DBConnectionPool/OptionSyncService等; [P1]消除硬编码路径

- **关键问题**:

  3. `_enqueue_write`直接访问`_params_service._instrument_meta_by_id`(L505)
