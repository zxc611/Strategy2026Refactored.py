## 三、核心问题R-24深度分析：五态统计total从15898降到1189

### 3.1 时间线

| 时间 | 事件 | total | other |
|------|------|-------|-------|
| 08:34:51 | 首次五态统计（第1次启动后） | 15898 | 15898 |
| 09:28:51 | 第1次启动稳定后 | 15898 | 15898 |
| 09:44:55 | 第2次重启，注册期权15898/15898 | - | - |
| 09:47:55 | 第2次重启后首次五态统计 | 1294 | 931 |
| 09:50:55 | 第2次重启后稳定 | 1188 | 825 |
| 10:26:40 | 第3次重启，注册期权15898/15898 | - | - |
| 10:29:40 | 第3次重启后首次五态统计 | 1189 | 825 |
| 11:17:40 | 最新五态统计 | 1189 | 825 |

### 3.2 根因分析

**直接原因**：`_recalc_all_state_for_underlying`（width_cache_state_mixin.py:712-807）在期货首次收到tick时被调用，用`temp_status_counts`**替换**`_status_counts[future_id]`（行806）。

**替换逻辑**：
```python
# 行806
self._status_counts[future_internal_id] = defaultdict(
    lambda: defaultdict(self._new_status_bucket), 
    {k: defaultdict(self._new_status_bucket, v) for k, v in temp_status_counts.items()}
)
```

`temp_status_counts`只包含snapshot中的期权。snapshot来自`_options_by_future_type[future_id]`（行728-729）。

**关键问题**：替换后，`_status_counts[future_id]`只包含snapshot中的期权。如果snapshot不完整（某些期权未被包含），这些期权就从`_status_counts`中消失了。

### 3.3 可能的snapshot不完整原因

| 假设 | 分析 | 可能性 |
|------|------|--------|
| A. _options_by_future_type缺少期权 | register_option行474-476填充，应该完整 | 低 |
| B. _option_info被清理导致snapshot跳过 | MAX_OPTION_CACHE_SIZE=20000>15898，不会触发 | 低 |
| C. underlying_future_id不匹配导致snapshot跳过 | 行734检查underlying_future_id!=future_id则跳过 | **中** |
| D. _option_info中underlying_future_id为None | 行757-758跳过 | **中** |
| E. 期货无期权时temp_status_counts为空 | 某些期货确实没有期权（如金融期货IF/IH只有少量期权） | **高** |

### 3.4 数据实证

- `_option_info`中有15898个期权（cache stats: total_options=15898）
- `_status_counts`的total只有1189
- 差值=14709个期权丢失
- FutureScoreDiagnosis显示total_futures=361（_status_counts中有361个future_id）
- 注册了390个期货，差29个

**推论**：361个期货在`_status_counts`中有条目（因为收到了tick触发了_recalc），29个期货没有收到tick所以`_status_counts`中没有条目（保持注册时的初始other计数）。但361个期货的`_status_counts`被替换后，total只有1189，说明每个期货平均只有1189/361≈3.3个期权被计入。这远低于15898/390≈40.8的平均值。

**最可能的根因**：`_recalc_all_state_for_underlying`中snapshot只包含了**有tick数据的期权**（`_option_price`非零的期权），而不是所有已注册的期权。但源码分析显示snapshot包含所有`_options_by_future_type`中的期权，无论是否有价格。

**需要进一步验证**：添加诊断日志打印snapshot大小与`_options_by_future_type`大小的对比，确认是否一致。

### 3.5 其他可能原因

另一个可能：`_classify_status`中`current_price <= 0`返回'other'（行243-244），但`_set_current_status`在行309-310中`old_status in bucket`检查。如果bucket是新创建的（defaultdict），`old_status`不在bucket中（因为_new_status_bucket返回的dict中key存在但值为0），`old_status in bucket`为True（因为key存在），所以会执行`bucket[old_status] = max(0, bucket[old_status] - 1)`，将other从0减到0（不变），然后`bucket[new_status] += 1`将other加到1。

这个逻辑是正确的，不会导致total减少。

### 11.1 逆向损耗点1：_status_counts total=1189 vs _option_info=15898

**现象**：五态统计total=1189，但_option_info中有15898个期权。

**根因链（已确认）**：

1. `register_option`将15898个期权全部加入`_status_counts`（通过`_set_current_status`），初始状态='other'，total=15898 ✅
2. 期货收到tick后，`_recalc_all_state_for_underlying`被调用（width_cache_state_mixin.py:533-535）
3. `_recalc`从snapshot重建`temp_status_counts`，然后**替换**`_status_counts[future_id]`（行806）
4. 替换后total降到1189

**snapshot构建逻辑分析**（行726-743）：
- 从`_options_by_future_type[future_id]`获取`target_internal_ids`
- 遍历每个iid，从`_option_info`获取`opt_info`
- 过滤条件：`not opt_info or opt_info.get('underlying_future_id') != future_internal_id`则跳过

**关键时序问题**：
- 第1次启动（08:31）：register_option完成后，期货tick才到达，_recalc触发时snapshot完整，total=15898（全部other）
- 第2次重启（09:44）：register_option在09:44:55完成，五态统计在09:47:55显示total=1294→1188
- 第3次重启（10:26）：register_option完成后，五态统计显示total=1381→1189

**核心推理**：3次启动register_option都成功注册15898个期权。但第2、3次重启后total大幅下降。差异在于**第1次启动时所有期权都是other（无tick数据），而第2、3次重启后部分期权收到了tick，_recalc被触发替换了_status_counts**。

**待验证假设**：`_recalc`替换时，snapshot中的期权数量少于register_option时_set_current_status添加的数量。可能原因：
- A. `_options_by_future_type[future_id]`中缺少某些期权（register_option时未正确添加）
- B. `_option_info`中某些期权的`underlying_future_id`与`_options_by_future_type`的key不一致
- C. `_recalc`在register_option完成之前被触发（期货tick先于期权注册到达），snapshot为空

**需要添加的诊断日志**：
```python

### 18.2 场景排查结果

#### 场景1：on_init与on_tick的ParamsService缓存读写竞态

| 项目 | 结论 |
|------|------|
| 是否存在竞态 | **否** |
| 分析 | load_caches_from_db的原子替换在锁内（_params_instrument_cache.py:1270-1276），get_instrument_meta和get_internal_id也在同一把锁内读取。读写同锁，不存在读到半初始化状态的可能 |

#### 场景2：_recalc_all_state_for_underlying的快照-替换竞态

| 项目 | 结论 |
|------|------|
| 是否存在竞态 | **否（当前）/可能（未来）** |
| 分析 | _recalc的唯一调用点是on_future_tick（行535），而on_future_tick在with self._lock内（行507）。由于self._lock是RLock，_recalc内部的with self._lock会重入成功。因此"锁内快照→锁外重算→锁内替换"实际上**全部在锁内执行**，不存在增量更新被覆盖的可能 |
| 风险 | 如果未来出现非持锁调用路径，则"锁外重算"阶段将真正在锁外执行，存在覆盖增量更新风险。**设计脆弱但当前安全** |

#### 场景3：_products_loaded的竞态

| 项目 | 结论 |
|------|------|
| 是否存在竞态 | **否** |
| 分析 | 三个写入点都是幂等操作（False→True），Python简单属性赋值受GIL保护是原子操作。_ensure_instruments_loaded设True后，_lazy_load_from_db不会读到旧值 |

#### 场景4：register_option与on_option_tick的竞态

|------|------|

#### 场景5：t_type_service的原子引用竞态

| 项目 | 结论 |
|------|------|
| 是否存在竞态 | **可能（逻辑竞态）** |
| 严重度 | **P1** |
| 分析 | strategy_config_layer.py:38在策略重置时将t_type_service设为None，到lifecycle_init.py:245重新赋值之间存在窗口期。此期间所有tick回调中的tts=self.t_type_service读到None，虽然代码有回退逻辑（get_t_type_service()），但若回退也返回None则tick丢失 |

#### 场景6：_enforce_cache_size_limit淘汰逻辑不一致

| 项目 | 结论 |
|------|------|

| 编号 | 环节 | 问题描述 | 影响 | 严重度 |

### 29.3 R-62详细分析（P0级）

**文件**: `width_cache_state_mixin.py`，on_future_tick方法（行498-538）

**问题**: on_future_tick不检查price是否为0，直接写入`_future_prices_by_month[future_id][month] = price`。后续`_is_out_of_the_money`中，期货价格为0时：
- CALL: `future_price < strike_price` → `0 < strike` → 恒True → 全部判为虚值
- PUT: `future_price > strike_price` → `0 > strike` → 恒False → 全部判为实值

**影响范围**: _sync_otm_count、_sort_buckets、select_otm_targets_by_volume的全部结果

**修复建议**: on_future_tick中增加`if price <= 0: return`或`if price <= 0: logger.warning(...); return`

### 29.4 R-63详细分析（P1级）

**文件**: `width_cache_state_mixin.py`，register_option方法（行399-496）

**问题**: internal_id参数没有强制int()转换。如果上游传入numpy.int64，则`_option_info[numpy.int64(42)]`和`_option_info[42]`是不同的key，导致on_option_tick中查找失败，期权tick被静默丢弃。

**当前缓解**: t_type_service.register_option_contract（行323-331）已做int()转换，所以主路径不会触发。但缺少防御性编程，如果未来有其他调用路径未做转换则会触发。

**修复建议**: register_option入口处增加`internal_id = int(internal_id)`、`underlying_future_id = int(underlying_future_id)`

### 29.5 R-64详细分析（P1级）

**文件**: `width_cache_state_mixin.py`

**问题**: `_options_by_future_type`的key是future_id，如果某处用numpy.int64、另一处用int，则同一个期货的期权会被分到两个不同的bucket中，导致_recalc_all_state_for_underlying只能看到部分期权。

**修复建议**: 所有写入_options_by_future_type的地方统一int()转换

### 29.6 R-65详细分析（P1级）

**文件**: `width_cache_context_mixin.py`（行32-35）vs `subscription_service.py`（行164-175）

**问题**: `_normalize_instrument_id`只处理`.`分隔符（如`SHFE.ag2501`），不处理`:`分隔符（如`ag2501:SHFE`）。而`_strip_exchange_prefix`处理两种格式。如果某模块传入`:`格式，normalize后无法匹配，导致查找失败。

**修复建议**: _normalize_instrument_id增加对`:`分隔符的处理，或统一调用_strip_exchange_prefix

### 30.5 R-68详细分析（P1级）

**文件**: `width_cache_state_mixin.py`，register_option方法

**修复建议**: register_option入口校验并标准化opt_type（upper()后映射）

### 30.6 R-69详细分析（P1级）

**文件**: `width_cache_state_mixin.py`，on_future_tick方法

**问题**: future_info的month为None时，`_future_prices_by_month[future_id][None]`写入None key的bucket，与register_option中用字符串month写入的bucket不匹配，导致后续期权查期货价格时查不到。

**修复建议**: on_future_tick中增加`if not month: logger.warning(...); return`

_enforce_cache_size_limit → 发现 _option_info_timestamps[iid] < _expired_cutoff
→ 删除 _option_info[iid]、_current_status[iid]、_sync_flag[iid] 等
→ 但该期权仍在接收tick！
on_option_tick → _option_info.get(internal_id) → None（已被TTL淘汰）
→ return（静默丢弃tick）
```

**影响范围**: `_CACHE_TTL_SECONDS = 86400`（24小时），策略运行超过24小时后，所有在初始化时注册但未重新注册的活跃期权都会被TTL淘汰。这是**数据流守恒的核心断裂点**，直接导致长时间运行后期权数据大量丢失。

**修复建议**: on_option_tick中每次成功处理tick后，更新`_option_info_timestamps[internal_id] = time.time()`，确保活跃期权不被TTL淘汰。

### 32.5 R-70详细分析（P1级）

**文件**: `width_cache_state_mixin.py`，行119-124

**问题**: 容量淘汰与TTL淘汰的清理范围不一致：
- TTL淘汰（行109-118）：清理`_option_info`、`_option_prev_price`、`_option_last_distinct_price`、`_option_last_direction`、`_option_volume`、`_option_daily_volume`、`_sync_flag`、`_current_status`（8个dict）
- 容量淘汰（行119-124）：**只清理**`_option_info`和`_option_info_timestamps`（2个dict）

残留的6个dict数据会导致：
1. 内存泄漏（_option_price等dict持续增长）
2. 计数不一致（_current_status残留但_option_info已删除）
3. 后续tick处理异常（_option_price残留但_option_info缺失，on_option_tick在行597-612直接return）

**修复建议**: 容量淘汰时同步清理所有8个关联dict，与TTL淘汰保持一致。

### 32.6 R-71详细分析（P1级）

**文件**: `width_cache_state_mixin.py`，行480-481

**问题**: `preserve_runtime_state=True`时（`existing_info is not None and initial_price <= 0 < existing_price`），直接return，不执行`_set_current_status`。

场景：策略重启时，register_option被重新调用，如果期权已有价格且新注册时initial_price=0，则跳过状态更新。但此时`_status_counts`可能已经因为`_recalc`替换而不一致，跳过更新意味着不一致无法被修复。

**修复建议**: preserve_runtime_state时仍需执行`_set_current_status`，或至少验证当前状态一致性。

### 32.7 R-72详细分析（P1级）

**文件**: `width_cache_state_mixin.py`，行136-146

**问题**: 期货缓存淘汰按`_future_prev_price`的dict插入顺序淘汰最旧的。但390个期货合约全部活跃，如果加上期权underlying_future_id指向的期货ID，总数可能超过`MAX_FUTURE_CACHE_SIZE=500`。

淘汰活跃期货时，行142删除`_status_counts[_fid]`，导致该期货下所有期权的五态统计被永久删除。后续on_option_tick只能增量更新被tick触及的期权，无法恢复完整的五态统计。

**修复建议**: 期货缓存淘汰前检查该期货是否有活跃期权（`_options_by_future_type.get(_fid)`），有则跳过。

## 三十四、第十六轮排查：状态机/生命周期角度

### 34.1 排查方法

从状态机/生命周期角度审查34+环节，追踪期权从注册→首次tick→状态分类→方向变化→淘汰的完整生命周期，验证状态转换是否完备、是否有死状态或不可达状态。核心问题：期权在每个生命周期阶段的状态是否可恢复？淘汰后是否有残留数据导致不一致？

### 34.2 期权状态机定义

```
注册(register_option) → other
  ↓ 首次tick(on_option_tick, prev_price=None)
  ├─ 期货已初始化 → _classify_status → correct_rise/wrong_rise/correct_fall/wrong_fall/other
  └─ 期货未初始化 → other（不可自动恢复，需等待后续tick或期货初始化）
  ↓ 后续tick(on_option_tick, prev_price≠None)
  → _classify_status → correct_rise/wrong_rise/correct_fall/wrong_fall/other
  ↓ 期货方向变化 → _recalc_all_state_for_underlying
  → 整体重算 → 替换_status_counts（PG-2: 替换丢失非snapshot期权）
  ↓ TTL/容量淘汰 → _enforce_cache_size_limit
  → 删除_option_info等 → 期权从系统中消失
  ↓ 后续tick → _option_info.get(iid)=None → return（静默丢弃）
```

### 34.3 发现的问题

| R-75 | _compute_sync_flag | future_state['rising']为None时（期货TTL淘汰后），bool(None)=False，导致PUT+option_direction=True被错误标记为同步 | 同步虚值计数错误 | P1 |

### 34.4 R-74详细分析（P1级）

**文件**: `width_cache_state_mixin.py`，行136-146

**问题**: 期货TTL淘汰删除`_future_prev_price`、`_future_rising`、`_future_initialized`等7个dict中的数据，但**不删除**`_future_prices_by_month`中该期货的价格数据。`_future_prices_by_month`用product字符串作key，与future_internal_id无关，所以期货TTL淘汰不会清理它。

**影响**: `_get_future_runtime_state`返回`initialized=False, rising=None, price=有效值`。后续：
1. `_classify_status`中行246-247：`if not future_state['initialized']: return 'other'` — 期货被淘汰后所有期权变为'other'
2. 但`_get_future_price_by_id_and_month`仍返回有效价格，造成语义不一致

**恢复路径**: 后续期货tick到来时，`on_future_tick`重新设置`_future_initialized[_fid]=True`和`_future_rising[_fid]`，并触发`_recalc_all_state_for_underlying`。但如果`_options_by_future_type[_fid]`也被淘汰（行146），则`_recalc`的snapshot为空，无法恢复任何期权状态。

**修复建议**: 期货TTL淘汰时，同步清理`_future_prices_by_month`中该期货product的所有月份价格；或者不淘汰有活跃期权的期货。

### 34.5 R-75详细分析（P1级）

**文件**: `width_cache_state_mixin.py`，行359

**问题**: `_compute_sync_flag`中行359：`future_rising = bool(future_state['rising'])`。如果`future_state['rising']`是None（期货被TTL淘汰后`_future_rising`被删除），`bool(None)` = False。

这导致：
- CALL + option_direction=True + future_rising=False → `is_target = (False and True and True) = False` ✅ 正确
- PUT + option_direction=True + future_rising=False → `is_target = (True and True and True) = True` ❌ 错误

期货方向未知时，PUT期权被错误地标记为同步，因为`not future_rising = not False = True`，满足PUT的同步条件。

**修复建议**: `_compute_sync_flag`中增加`if future_state['rising'] is None: return False`，在行353-354之后。

### 34.6 R-76详细分析（P1级）

**文件**: `width_cache_state_mixin.py`，行109-118

**问题**: TTL淘汰清理了8个dict（`_option_info`、`_option_prev_price`、`_option_last_distinct_price`、`_option_last_direction`、`_option_volume`、`_option_daily_volume`、`_sync_flag`、`_current_status`），但**不清理**`_option_price`。

影响：
1. 内存泄漏：`_option_price`持续增长，永不缩减
2. 重新注册时状态不一致：register_option中行447 `existing_price = float(self._option_price.get(internal_id, 0.0) or 0.0)`，如果`_option_price`残留旧价格，则`preserve_runtime_state`判断为True，跳过状态更新（R-71叠加）
3. `_option_price`残留但`_option_info`已删除，后续on_option_tick在行597-612直接return（静默丢弃tick）

**修复建议**: TTL淘汰时增加`self._option_price.pop(_eid, None)`。

### 34.7 R-77详细分析（P1级）

**文件**: `width_cache_state_mixin.py`，行109-118

**问题**: TTL淘汰不清理`_options_by_future_type`中该期权的internal_id。`_options_by_future_type[future_id][opt_type]`是一个列表，TTL淘汰后仍保留已淘汰期权的internal_id。

影响：
1. `_recalc_all_state_for_underlying`中行732-735：遍历`_options_by_future_type`中的iid，但`_option_info.get(iid)`返回None（已淘汰），该期权被跳过，不计入temp_status_counts
2. 行806替换后，该期权的计数丢失（与PG-2叠加，加重92.7%损耗）
3. `_options_by_future_type`列表持续增长，包含大量无效iid

**修复建议**: TTL淘汰时，从`_options_by_future_type[future_id][opt_type]`中移除该iid。或在`_recalc`中过滤掉`_option_info`为None的iid。

## 三十六、第十七轮排查：锁/并发安全角度

### 36.1 排查方法

从锁/并发安全角度审查34+环节，检查所有共享状态的并发访问是否安全，特别是`_recalc`的"快照+锁外重算+锁内替换"模式是否存在TOCTOU问题，以及锁内调用外部服务是否导致持锁时间过长。

### 36.2 发现的问题

| 编号 | 环节 | 问题描述 | 影响 | 严重度 |
|------|------|----------|------|--------|
| R-78 | _recalc | "快照+锁外重算+锁内替换"模式存在TOCTOU竞态：阶段2锁外重算期间，on_option_tick的增量更新被阶段3的替换覆盖 | 短暂计数不一致 | P2 |

### 36.3 R-78详细分析（P2级）

**文件**: `width_cache_state_mixin.py`，行712-807

**问题**: `_recalc`分三个阶段执行：
1. 阶段1（锁内快照）：读取所有期权的`_option_price`、`_current_status`等
2. 阶段2（锁外重算）：调用`_classify_status`、`_compute_sync_flag`等
3. 阶段3（锁内替换）：用temp_status_counts替换`_status_counts[future_id]`

阶段2期间，其他线程的`on_option_tick`可以获取锁并更新`_current_status`、`_status_counts`等。阶段3替换时，这些增量更新被覆盖。

**实际影响评估**：
- `_recalc`只在期货方向变化时触发（行534-535），频率低
- 阶段2的重算时间短（主要是`_classify_status`调用）
- 竞态窗口很小，且Python GIL进一步缩小窗口
- 被覆盖的增量更新会在下一个tick到来时被修复

**严重度**：P2（实际影响小，但设计上不严谨）

**修复建议**：阶段3替换时，合并阶段2期间的增量更新，而非直接替换。或将阶段2也放入锁内（牺牲性能换取正确性）。

### 37.3 R-79详细分析（P1级）

**文件**: `width_cache_state_mixin.py`，行402

**问题**: `underlying_future_id: Optional[int] = None` — 类型标注为Optional，默认值为None。但行422-423：`if underlying_future_id is None: raise ValueError("underlying_future_id required")` — 实际是必填参数。

类型标注与运行时行为不一致，调用方可能误以为可传None，导致运行时异常。

**修复建议**: 将`underlying_future_id`改为必填参数（去掉`Optional`和`= None`），或在docstring中明确标注"运行时必填"。

### 37.4 R-80详细分析（P1级）

**文件**: `width_cache_state_mixin.py`，行433-439

**问题**: `internal_id`为None且`_resolve_internal_id`也返回None时，行438-439 `logging.error(...); return None`。调用方（如`register_option_contract`，行334-338）不检查返回值，无法感知注册失败。

**影响**: 期权注册静默失败，该期权后续tick全部被丢弃（`_option_info`中无该期权）。

**修复建议**: `register_option`在`internal_id`无法解析时应抛异常而非静默返回None，或`register_option_contract`应检查返回值。

### 37.5 R-81详细分析（P1级）

**文件**: `t_type_service.py`，行267-282 + `subscription_service.py`，行1464

**问题**: `TTypeService.on_future_instrument_tick(future_internal_id, price)` 无None校验。在`subscription_service.py`行1464：`tts.on_future_instrument_tick(int(internal_id), float(last_price))` — `internal_id`已做`is not None`检查（行1463），但`int(internal_id)`在`internal_id`为非数字类型时会抛ValueError。

虽然`int()`转换在调用方（subscription_service.py）执行，但`on_future_instrument_tick`本身不校验参数，如果其他调用方传入None，会导致`wc.on_future_instrument_tick(None, price)` → `wc._get_params().get_instrument_meta(None)` → 可能异常。

**修复建议**: `on_future_instrument_tick`入口增加`if future_internal_id is None: return`防御检查。

### 37.6 R-82详细分析（P2级）

**文件**: `t_type_service.py`，行284-287 + 行267-282

**问题**: `on_option_tick`和`on_future_instrument_tick`的`price`参数无NaN/Inf/负值校验，直接透传给WidthCache。

上游`subscription_service.py`的`on_tick`入口已做NaN/Inf/负值过滤（行1384-1386），但`tick_processing_service.py`的直连调用路径可能未过滤。如果NaN/Inf/负值price进入WidthCache，会导致`_classify_status`逻辑异常。

**修复建议**: `on_option_tick`和`on_future_instrument_tick`入口增加`if not isinstance(price, (int, float)) or math.isnan(price) or math.isinf(price) or price <= 0: return`防御检查。

### 40.3 R-96详细分析（P1级 — 容量淘汰内存泄漏）

**文件**: `width_cache_state_mixin.py`，行119-124

**问题**: 容量淘汰（行119-124）只清理`_option_info`和`_option_info_timestamps`两个dict，不清理`_option_price`、`_option_prev_price`、`_option_last_distinct_price`、`_option_last_direction`、`_option_volume`、`_option_daily_volume`、`_sync_flag`、`_current_status`等8个关联dict。

与TTL淘汰（行109-118，清理8个dict但遗漏`_option_price`）相比，容量淘汰遗漏更严重。

**内存影响**: 16288个合约，每个key-value约50 bytes，8个dict累计泄漏 16288 * 8 * 50 ≈ 6.5MB。加上TTL淘汰遗漏的`_option_price`约0.8MB，总计约7.3MB。

**修复建议**: 容量淘汰时同步清理所有9个关联dict（与TTL淘汰一致+补齐`_option_price`）。

### 40.4 R-92详细分析（P2级）

**文件**: `subscription_service.py`，行1019-1020

**问题**: `_last_tick_seq`和`_last_tick_seq_ts`以instrument_id为key，16288个合约产生16288个key，无淘汰机制。

**内存影响**: 两个dict合计约3.5MB，策略运行期间永不释放。

**修复建议**: 添加LRU淘汰机制，或定期清理长时间未更新的key（如超过1小时无tick的合约）。

## 四十七、真实行情覆盖失败的根因修正：不能归因于平台未推送

### 47.1 结论修正

上一节把真实tick低覆盖作为全链路失败点是正确的，但把它表述为“平台/模拟真实覆盖低”仍然不够严谨。根据本轮继续排查，当前证据不能直接归因于平台没有推送。更准确的根因链路是：

1. `subscription_confirmed_count=16288`并不是平台ack，只是本地订阅调用/本地分母。
2. 6/30 21:12窗口没有任何可识别的平台订阅ack/订阅应答日志。
3. 6/30 21:12窗口没有`SubscribeTickGapAudit`输出，原本应该直接暴露“本地订阅调用数 vs 收到tick合约数”的缺口。
4. 6/30 21:12窗口没有行情读取队列/出队/消费日志，无法证明平台回调后的队列读取链路正常。
5. `PROBE_SUB`显示本地订阅调用存在明显重复，不能把本地调用数当成有效订阅确认数。

因此，本轮根因不得写为“平台没有推送”。正确判定是：**订阅ack与行情读取队列不可观测，且本地订阅调用存在重复；真实tick覆盖低发生在“本地订阅调用之后、onTick入口之前或入口队列观测缺失处”，目前不能越过该边界归因于平台。**

## 四十九、继续排查：DataService直接写入模式的tick入队真实断点

### 49.1 结论

本轮继续沿34+链路从`onTick`之后往后查，发现一个真实数据流失点，不是单纯日志不可观测：

**`DataService`当前MRO实际使用`DataWriterMixin._enqueue_write()`，但该方法只支持`_save_kline_impl`，遇到`_save_tick_impl`直接返回`False`。**

因此，凡是走以下路径的tick都会在写入入口被判失败：

```text
process_tick()
  -> _enqueue_write('_save_tick_impl', internal_id, instrument_type, [tick])
  -> DataWriterMixin._enqueue_write()
  -> 旧逻辑不识别 _save_tick_impl
  -> return False
  -> [process_tick] Tick 入队失败，已跳过
```

这是34+链路中“tick入口之后、写入队列/落库之前”的真实断点。

### 48.3 根因分析

#### 根因1（P0）：dispatch_tick_degraded降级路径无flush逻辑

**文件**：`ali2026v3_trading/strategy/tick_dispatch.py`
**函数**：`dispatch_tick_degraded`（行730-881）

**问题**：降级路径 `dispatch_tick_degraded` 写入shard_buffers（行822），但**没有flush逻辑**，没有batch_insert_ticks调用。与正常路径 `_dispatch_tick_inner`（行1122-1180）的flush逻辑不一致。

**影响**：如果 `_historical_load_in_progress=True`，所有tick走降级路径，shard_buffers中的tick永远不flush到DB，导致ticks_raw=0行。

#### 根因2（P0）：_callback_group为None导致flush_tick_buffer未注册定时回调

**文件**：`ali2026v3_trading/strategy/tick_processing_service.py`
**行号**：272-276

**问题**：日志显示 `[STARTUP-DEP] R17-P1-CFG-P1-05: 关键缺失: _callback_group`，说明 `_callback_group` 为None。`flush_tick_buffer` 不会注册为定时回调。如果tick推送停止（如收盘后），shard_buffers中可能还有未flush的tick。

**影响**：shard_buffers中的tick在tick推送停止后无法被flush，导致数据流失。

#### 根因3（P1）：_historical_load_in_progress状态不同步

**文件**：`ali2026v3_trading/strategy/tick_dispatch.py:581` 和 `ali2026v3_trading/strategy/strategy_historical.py:696/731`

**问题**：`strategy_historical.py` 设置 `self._historical_load_in_progress`（策略实例属性），但 `tick_dispatch.py:581` 从 `svc._state_store.get('_historical_load_in_progress')` 读取。两者不同步，`state_store` 中从未设置此值。

**影响**：`tick_dispatch.py` 永远读取到False，所有tick走正常路径。虽然这"意外"避免了根因1的降级路径问题，但不是设计意图。
