## 三、根因分析

### m1-01/m1-05: date字段使用UTC日历日期而非trade_date (核心根因)

**代码位置:**
- `ds_data_writer.py:400` — `build_tick_arrow_batch` 中 `'date': dt.date()`
- `ds_data_writer.py:122` — `_enrich_tick_option_metadata` 中 `tick_row['date'] = _date_src.date()`

**根因链路:**
1. 夜盘21:00 CST的tick，timestamp为`2026-06-30 21:00:00`
2. `build_tick_arrow_batch`将其解析为UTC时间`2026-06-30 13:00:00 UTC`
3. `dt.date()`返回`2026-06-30`（当日）
4. 查询时`_current_trade_date_for_coverage()`调用`ExchangeTime.get_trade_date()`，21:00 CST返回`2026-07-01`（次日）
5. SQL `WHERE date = '2026-07-01'`查不到`date = '2026-06-30'`的夜盘tick
6. **结果: 所有夜盘tick的date字段与查询条件不匹配，ticks_raw_today=0**

**影响范围:**
- health_monitor.py `_load_full_chain_entry_counts` — ticks_raw_today=0
- ds_data_writer.py `ensure_config_coverage_for_today` — before_tick_actual偏低
- ds_data_writer.py `refresh_chain_coverage_audit` — real_tick_count=0
- chain_coverage_audit — tick_return_count冻结在启动快照

### m1-02: tick_return_count冻结在启动快照

**代码位置:** `ds_data_writer.py:721-732`

**根因:**
1. `ensure_config_coverage_for_today`仅在启动时调用一次(`product_initializer.py:566`)
2. FIX-M1补丁添加了`refresh_chain_coverage_audit`每30秒刷新
3. 但refresh继承了m1-01的日期bug，查询`WHERE date = trade_date`仍查不到夜盘tick
4. **结果: refresh刷新后real_tick_count仍为0，审计表不更新**

### m1-06: price<=0的return丢弃期权tick (注释与代码不一致)

**代码位置:** `subscription_service.py:1383-1390`

**根因:**
1. 注释声明"FIX-20260701: price<=0不再丢弃（仅影响TType路由）"
2. 但代码实际执行`return`，**完全丢弃了price<=0的tick**
3. 期权合约在无成交时last_price经常为0（尤其是深度虚值期权）
4. **结果: 大量期权tick被完全丢弃，期权路由占比仅8.5%**

### P0-问题5: _classify_status因price<=0返回'other'

**代码位置:** `width_cache_state_mixin.py:348-349`

**根因:**
1. `_classify_status`函数在price<=0时直接返回'other'
2. 导致大量期权被错误分类为'other'，无法进入correct_rise/wrong_rise等有效状态
3. 影响五态统计准确性，导致correct_up_pct偏低，大量期货被判定为Tier 4
4. **结果: 97.1%期权被错误分类为'other'**

**修复方案:**
1. 移除`if current_price <= 0 or prev_price <= 0: return 'other'`的直接返回逻辑
2. 改为`if current_price > 0 and prev_price > 0:`条件下才计算option_direction
3. 如果price<=0，尝试使用已提供的option_direction参数进行分类
4. 仅当无法确定方向时才返回'other'

### P0-问题07: tick去重窗口过宽导致大量合法tick被误杀

**代码位置:** `subscription_service.py:1409`

**根因:**
1. tick去重基于`(instrument_id, last_price, volume)`三元组+1秒时间窗口
2. 在盘口静止期间（价格不变、成交量不变），1秒内多个合法tick的三元组完全相同
3. 高流动性合约（如br2609）在活跃时段1秒内可能收到数十个tick，大量被误杀
4. **结果: br2609累计丢弃10001个tick，全合约累计丢弃数十万tick**

**修复方案:**
1. 将时间窗口从1秒缩短为100ms
2. 仅过滤真正的重复推送（同一tick在100ms内到达两次）

### P0-问题08: _cache_temp_instrument_info key未标准化

**代码位置:** `_params_instrument_cache.py:1039, 1084`

**根因:**
1. `cache_instrument_info`（运行时缓存）使用`normalize_instrument_id`标准化key
2. `_cache_temp_instrument_info`（DB批量加载）直接使用原始instrument_id作为key
3. 如果DB中存储了带交易所前缀的ID（如`SHFE.al2608`），查找时使用标准化后的`al2608`，导致查找失败
4. **结果: tick路由时metadata查找失败，期权tick无法路由到TTypeService**

**修复方案:**
1. 在`_cache_temp_instrument_info`中也使用`normalize_instrument_id`标准化key
2. 确保`temp_instrument_id_to_internal_id`使用标准化key

### P0-问题09: _compute_sync_flag因price<=0返回False

**代码位置:** `width_cache_state_mixin.py:463`

**根因:**
1. `_compute_sync_flag`在`current_price <= 0`时直接返回False
2. 导致price<=0的期权sync_flag始终为False，无法进入排序桶
3. 与P0-问题05级联：price<=0 → sync=False → 排序桶移除 → 选股无目标
4. **结果: 大量期权被排除在排序桶之外**

**修复方案:**
1. 移除`if current_price <= 0`的检查
2. 仅当`option_direction is None`时才返回False（方向未知无法判断同步性）

### P0-问题10: _update_option_direction因price<=0清空方向

**代码位置:** `width_cache_state_mixin.py:426-429`

**根因:**
1. `_update_option_direction`在price<=0时清空`_option_last_direction`
2. 导致后续`_classify_status`和`_compute_sync_flag`使用None方向
3. 与P0-问题05/09级联：方向清空 → 分类为'other' → sync=False → 排序桶移除
4. **结果: 大量期权方向被清空，级联效应导致选股返回空列表**

**修复方案:**
1. price<=0时仅清空锚点价格，保留已有方向
2. 下一笔有效价格到达时重新设为锚点，方向在下一笔不同价格时更新

### P1-问题11: Price jump期权识别逻辑缺陷

**代码位置:** `tick_dispatch.py:305`

**根因:**
1. Price jump检测使用内联启发式`len(instrument_id) > 6 and any(c.isdigit() for c in instrument_id[-4:])`判断期权
2. 该启发式对国内期货/期权合约ID格式不可靠，期货可能被误判为期权
3. 同文件存在更可靠的正式函数`is_option_instrument()`，但未被调用
4. **结果: 期权跳变告警统计失真，期货跳变可能被期权阈值吞掉**

**修复方案:**
1. 将内联启发式替换为正式函数`is_option_instrument(instrument_id)`

### P0-问题12: _do_update_sort_bucket过滤条件不一致

**根因**: `width_cache_state_mixin.py:963` 只允许 `correct_rise`/`correct_fall` 进入排序桶，而 `width_cache_types.py:701` 允许四种有效状态（含 `wrong_rise`/`wrong_fall`）。由于 `WidthStrengthCache` 继承 `WidthCacheStateService`，MRO优先使用 `state_mixin` 版本，`types.py` 中的 `WidthCacheSortService` 版本成为死代码。两个文件不一致本身就是bug。

**影响**: `wrong_rise`/`wrong_fall` 期权在 `state_mixin` 版本中被排除在排序桶外（设计正确——噪音期权不应进入交易候选），但 `types.py` 版本过宽会导致不一致行为。

**修复**: 统一 `types.py` 的过滤条件与 `state_mixin` 一致，只允许 `correct_rise`/`correct_fall` 进入排序桶。

**修改文件**: `width_cache_types.py:701`

### P0-问题13: _compute_sync_flag只覆盖correct_rise不覆盖correct_fall（核心根因）

**根因**: `_compute_sync_flag` 中的 `is_target` 逻辑只判断"同步上涨"（correct_rise），不判断"同步下跌"（correct_fall）：

```python
# 修复前: 只覆盖correct_rise
is_target = (
    (future_rising and opt_type == 'CALL' and option_direction)       # CALL上涨+期货上涨
    or (not future_rising and opt_type == 'PUT' and option_direction) # PUT上涨+期货下跌
)
```

这导致所有 `correct_fall` 期权的 `sync_flag` 为 `False`，不会被计入 `_sync_otm_count`，也不会进入排序桶。

**级联效应**: 
- `_sync_otm_count` 严重偏低（只计correct_rise，不计correct_fall）

## 十四、第七轮深度排查与修复（FIX-M8系列）— K线合成链路根因定位

> 排查时间: 2026-07-01
> 排查对象: 7月1日19:54:22重启后运行日志，聚焦"klines_real=0"根因
> 排查方法: 从tick落库→K线合成全链路审计，定位5个级联根因

### 排查背景

第十三轮排查确认16288→756的tick缺口主因是平台限制，但发现另一个独立问题：**即使756个合约的tick已正常落库，klines_real仍为0**。这意味着K线合成链路存在独立于tick覆盖率的阻断点。

证据链:
1. PROBE_TICK_SUMMARY显示736个合约的tick已Saved落库
2. 但20合约诊断持续显示klines_real=0
3. 历史K线加载日志显示success=0, failed=250（平台返回无数据）
4. 合成K线（aggregate_kline_only）从未生成任何K线

### 根因链路（5个级联根因）

```
历史K线加载失败(platform无数据)
        ↓
M8-5: empty_batch_limit=5过早终止(仅尝试250合约)
        ↓
M8-1: K线fallback guard阻止所有合成K线(_ext_kline_load_in_progress=True)
        ↓
M8-2: _KlineAggregator双重过滤price<=0(拒绝深度虚值期权)
        ↓
M8-3: 无flush机制，稀疏tick场景K线永不完成(736/16288≈0.045 tick/合约)
        ↓
M8-4: 历史加载期间tick背压误触发，丢弃合法tick
        ↓
klines_real=0
```

### FIX-M8-1: K线fallback guard阻断所有合成K线（核心根因）

**代码位置**: `data/storage_snapshot_mixin.py:331-345`

**根因**:
1. `_is_ext_kline_missing`函数在`_ext_kline_load_in_progress=True`时直接`return False`
2. 这阻止了`aggregate_kline_only`生成任何合成K线
3. 但历史K线加载可能完全失败（success=0, failed=250），此时`_last_ext_kline`为空
4. 当外部K线不存在时，合成K线是唯一K线来源，却被guard阻止
5. **结果: 历史加载期间(58秒)所有756个合约的合成K线被阻止生成**

**修复**: 仅在已有外部K线数据（`last_time is not None`）时，加载期间才跳过合成K线（避免重复）。当`last_time`为`None`时（无外部K线），无论是否在加载中，都允许合成K线作为唯一来源。

```python
# 修复前: 加载期间无条件阻止合成K线
if self._ext_kline_load_in_progress:
    return False  # ← 阻断所有合成K线

# 修复后: 仅当已有外部K线时才跳过
if self._ext_kline_load_in_progress and last_time is not None:
    return False  # 已有外部K线，加载期间跳过避免重复
if last_time is None:
    return True   # 无外部K线，允许合成K线作为唯一来源
```

**修改文件**: `data/storage_snapshot_mixin.py:332-345`

### FIX-M8-2: _KlineAggregator双重过滤price<=0

**代码位置**: `data/query_service.py:135-145`

**根因**:
1. `aggregate_kline_only`上游已尝试用bid/ask中间价替代price<=0
2. 当bid/ask也无效时，price仍为<=0传入`_KlineAggregator._update_impl`
3. `_update_impl`再次检查`price <= 0`并直接拒绝（双重过滤）
4. 深度虚值期权快照的last_price=0是合法数据（无成交但有报价）
5. **结果: 大量期权tick无法生成K线，即使已正常落库到ticks_raw**

**修复**: 将`price <= 0`改为`price < 0`（仅拒绝负价格），允许price=0通过。price=0的K线虽然OHLC均为0，但能保证K线时间轴连续性。

```python
# 修复前: price<=0直接拒绝(双重过滤)
if price <= 0:
    return None

# 修复后: 仅拒绝NaN/Inf和负价格，允许price=0
if isinstance(price, float) and (_kline_math.isnan(price) or _kline_math.isinf(price) or price < 0):
    self.logger.warning("[CC-05] K线聚合拒绝NaN/Inf/负价格: ...")
    return None
```

**修改文件**: `data/query_service.py:139-145`

### FIX-M8-3: 无flush机制导致稀疏tick场景K线永不完成

**代码位置**: `data/query_service.py:200-214`（新增flush方法）, `data/storage_data_write_mixin.py:526-556`（新增flush_incomplete_klines）, `strategy/tick_dispatch.py:1016-1059`（周期调用）

**根因**:
1. `_KlineAggregator`仅在tick跨越K线周期边界时返回completed_kline
2. 736个real ticks / 16288 instruments ≈ 0.045 tick/instrument
3. 大多数instrument只有1个tick，`update()`初始化后永不返回completed_kline
4. 导致klines_real=0，即使tick数据已正常落库
5. **结果: 756个有tick的合约全部无K线生成**

**修复**: 
1. 在`_KlineAggregator`新增`flush()`方法，强制返回当前未完成的K线并重置聚合器
2. 在`storage_data_write_mixin`新增`flush_incomplete_klines()`方法，遍历所有`_aggregators`调用`flush()`
3. 在`tick_dispatch.flush_tick_buffer()`（周期性调用）末尾调用`flush_incomplete_klines()`
4. **结果: 每个收到tick的instrument至少有一根K线落库**

```python
# query_service.py 新增flush方法
def flush(self) -> Optional[Dict[str, Any]]:
    with self._lock:
        if self.kline_start_time is None or self.open_price is None:
            return None
        kline = self._get_current_kline()
        # 重置聚合器，开始新的K线周期
        self.open_price = self.high_price = self.low_price = self.close_price = None
        self.volume = 0
        self.amount = 0.0
        self.open_interest = None
        self.kline_start_time = None
        return kline

# storage_data_write_mixin.py 新增flush_incomplete_klines方法
@requires_phase(InitPhase.READY)
def flush_incomplete_klines(self) -> int:
    """遍历所有_aggregators，调用flush()强制返回未完成K线。"""
    flushed_count = 0
    with self._agg_lock:
        aggregators = list(self._aggregators.items())
    for (instrument, period), agg in aggregators:
        kline = agg.flush()
        if kline:
            if self._is_ext_kline_missing(instrument, period, time.time()):
                self.save_external_kline(kline)
                flushed_count += 1
    return flushed_count

# tick_dispatch.py flush_tick_buffer末尾调用
_storage_flush = svc._state_store.get_ref('storage') if svc._state_store else None
# ... 正常flush逻辑 ...
# FIX-M8-3: 周期性flush未完成的K线
if _storage_flush and hasattr(_storage_flush, 'flush_incomplete_klines'):
    _storage_flush.flush_incomplete_klines()
```

**附带修复**: `_storage_flush`变量原定义在for循环内，循环外不可访问。已移到函数开头定义。

**修改文件**: `data/query_service.py:200-214`, `data/storage_data_write_mixin.py:526-556`, `strategy/tick_dispatch.py:1018-1059`

**关键发现: #1/#2 与 #3/#4 是顺序执行，非互斥**

```

├── [步骤A] lines 489-557: _sub_mgr.on_tick() → _on_tick_impl → 调用点#1 (TTypeService路由)
│   └── (若_sub_mgr为None) → 调用点#2 (fallback直接路由TTypeService)
```

## 十五、第七轮深度排查与修复（M9系列）— 16288→756 tick缺口真实根因定位

> 排查时间: 2026-07-02
> 排查对象: 推翻第十三节"平台限制"错误归因，定位策略侧真实根因
> 排查方法: 源码考古 + 数学验证 + 调用链静态分析
> 触发原因: 用户明确指出"没有确定证据归因于平台是十分错误"

### 排查背景

第十三节（M8系列）得出"16288→756 tick缺口是平台/订阅层限制"的结论，依据是：
1. `success=16288, failed=0`（订阅全部成功）
2. 仅756个合约收到tick
3. `platform_ack_observed=false`（无法验证平台ack）

**该结论的根本性错误**: 上述3条依据中，第1条（success=16288）被策略侧Bug污染为虚假统计，第3条（无法验证平台ack）恰恰是Bug的表现而非平台问题。在订阅成功率统计本身不可信的前提下，"平台限制"结论缺乏证据基础，属于错误归因。

### 真实根因定位：策略侧三个Bug

#### FIX-M9-01 (P0，根因): lifecycle_bind.py 20ms per-call 速率限制器

**代码位置**: `lifecycle_bind.py:40`（_do_bind_platform_apis 内 _subscribe 包装器）

**根因**:
```python
_sub_min_interval_sec = 0.02  # 20ms per call
# ...
if _sub_last_call_at[0] > 0 and elapsed < _sub_min_interval_sec:
    _sub_time.sleep(_sub_min_interval_sec - elapsed)
```
- 每次 `sub_market_data` 调用前强制 sleep 20ms
- `subscribe_all_instruments` 对 16288 合约逐一调用 → 总阻塞 16288 × 0.02s = **325.76s**
- 实测 on_start 阻塞 **342s**（5m42s），与数学计算吻合（差异来自锁开销和函数调用开销）
- 该速率限制器于 7月1日 引入，与"7月1日重启后342s阻塞"时间点完全吻合

**历史基线对比**:
| 日期 | 订阅耗时 | 说明 |
|------|---------|------|
| 6月26-30日 | ~8s | 无速率限制器（历史基线） |
| 7月1日 16:32 | ~342s | 引入20ms限制器（40x减速） |
| 7月1日 18:38 | ~342s | 同上 |
| 7月1日 19:54 | ~342s | 同上（M8排查对象） |

**为何"平台限制"是错误归因**:
- on_start 阻塞342s 期间，平台推送的 tick 全部排队等待
- 342s 后 on_start 返回，排队 tick 突发到达，首tick 476ms JIT 初始化触发5s背压窗口
- 此后策略进入正常运行，但前342s的tick已全部错过（开盘关键时段）
- M8原结论将"342s阻塞"列为"次要因素"，实际这是**主因**

**修复**: `_sub_min_interval_sec = 0.0`（完全禁用per-call限速）
- `subscribe_all_instruments` 已有批级节流（每100合约 `time.sleep(0.05)`），per-call限速冗余且有害
- 禁用后预期订阅耗时回归 ~8s 基线

**修改文件**: `lifecycle/lifecycle_bind.py:40-44`

#### FIX-M9-02 (P0，共因): DataService.subscribe 静默吞噬所有异常

**代码位置**: `data_service.py:181-192`

**根因**:
```python
def subscribe(self, instrument_id, callback=None):
    # ...
    try:
        return _fn(instrument_id)
    except Exception as e:  # 吞噬所有异常
        logger.warning(...)  # 仅warning日志
        return None          # 返回None，调用方无法区分成功/失败
```
- 平台 `sub_market_data` 失败时（如合约不存在、订阅超限、参数错误），异常被 `except Exception` 静默捕获
- 返回 `None`，调用方 `_do_subscribe` 的检查 `if result is None and not self._is_subscribe_ready()` 在API已绑定时不触发
- 结果：失败的订阅被计为成功，`success=16288/failed=0` 是**虚假统计**
- M8原结论基于此虚假统计得出"全部订阅成功"，再因"仅756收tick"反推"平台限制"，逻辑链从第一步就崩塌

**修复**:
1. 订阅成功返回 `True`，`subscribe_fn` 未绑定返回 `False`
2. 平台调用失败：递增类级失败计数器 `_subscribe_fail_count`，**重新抛出异常**（由 `_subscribe_single_with_retry` 捕获入重试队列）
3. 新增 `get_subscribe_fail_count()` / `reset_subscribe_fail_count()` 诊断方法
4. 同步更新 `_do_subscribe`：检查 `result is False` 而非 `result is None`

**修改文件**: `data/data_service.py:181-214`, `infra/subscription_service.py:1000-1012`

#### FIX-M9-03 (P0，共因): subscribe_all_instruments 无条件 success_count 自增

**代码位置**: `infra/subscription_service.py:1147, 1180`

**根因**:
```python
self._subscribe_single_with_retry(inst_id, 'tick')  # 失败时入重试队列，不抛异常
success_count += 1  # 无条件自增！即使订阅失败也计为成功
```
- `_subscribe_single_with_retry` 捕获异常入重试队列后**不重新抛出**
- 调用方在 try 块内无条件 `success_count += 1`
- 即使订阅失败（已入重试队列），仍被计为成功
- 最终日志 `success=16288, failed=0` 完全掩盖真实失败数

**修复**:
1. `_subscribe_single_with_retry` 返回 `bool`（True=成功，False=失败入重试）
2. `subscribe_all_instruments` 仅在返回 True 时 `success_count += 1`
3. 返回 False 时计入 `failed_tasks` 并触发失败探针
4. 期货和期权两条路径同步修复

**修改文件**: `infra/subscription_service.py:1144-1181（期货）, 1190-1225（期权）, 1234-1249（_subscribe_single_with_retry）`
