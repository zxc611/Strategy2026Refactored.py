## 三、根因链

### 根因 1：过期行情未被淘汰，导致 15:00 后仍用 14:58 行情交易

代码位置：`strategy/tick_processing_service.py`

原逻辑只处理数值型 tick 时间：

```python
_tick_epoch = float(_tick_ts) if isinstance(_tick_ts, (int, float)) else 0.0
```

如果平台 tick 的 `datetime` 是 `datetime` 对象或 ISO 字符串，`_tick_epoch` 会变成 0，后续过期校验被跳过。

实际数据库显示生产 tick 最大时间停在 `2026-06-27 14:58:36`，但 17:24 仍出现下单与影子交易，说明策略使用了过期价格继续触发交易。这是本轮交易链路的 P0 根因。

### 根因 2：首次期权 tick 在五态锁内同步 sleep 500ms，放大 tick 超时与过载

代码位置：`data/width_cache_state_mixin.py`

旧逻辑在 `on_option_tick` 持锁路径内执行 5 次 `time.sleep(0.1)`，单个期权首次 tick 最坏阻塞 500ms。多个期权并发到达时，会放大为：

```text
五态锁等待 -> tick处理超时 -> 过载丢弃窗口 -> tick去重丢弃 -> 状态继续 other
```

这与日志中的 tick 处理超限、严重超时、过载丢弃高度一致。

### 根因 3：ParamsService 元数据懒加载会隐式创建完整 DataService

代码位置：`config/_params_instrument_cache.py`

`get_instrument_meta_by_id()` 在缓存为空时调用 `get_data_service()`，这会创建完整 `DataService`。本地复现中单次元数据查询耗时 62 秒，且最终因 `_products_loaded=False` 返回空。

该路径不应由元数据查询触发完整数据库服务启动，否则 SubscriptionManager、诊断、风控等轻量读取都会被数据库初始化拖慢或超时。

### 根因 4：DataService 初始化无条件做重型 DuckDB 维护

代码位置：`data/ds_schema_manager.py`

旧逻辑每次初始化都会：

1. `DROP TABLE IF EXISTS option_sync_otm_stats`
2. 刷新 `option_sync_otm_stats`
3. 全表更新 `ticks_raw` 状态列
4. `ANALYZE ticks_raw`

在 1200 万 tick 量级下，这些操作不应处于策略启动和元数据懒加载路径。

### 根因 5：tick 泛化写入路径缺 date/timestamp 补齐

代码位置：`data/ds_data_writer.py`

批量 tick 不传 `instrument_id` 参数时，代码依赖每条 tick 自带字段。旧逻辑对 ISO 字符串时间只写 `ts`，但 Arrow schema 只认 `timestamp`，导致 `timestamp` 空；同时未补齐 `date`，导致按日查询失真。

### 根因 6：20 合约诊断 DuckDB 成功存在假阳性

代码位置：`infra/health_monitor.py`

旧逻辑在真实 tick/kline 计数为 0 时，会用 `SubscriptionManager._subscription_success['kline_instruments']` 作为 `duckdb_inserted=True` 的替代证据。这会把“订阅过”误报成“已落库”。

## 四、修复清单

| 文件 | 修复内容 |
|------|----------|
| `strategy/tick_processing_service.py` | 新增 `_tick_time_to_epoch()`，支持数值、`datetime`、ISO 字符串；单调性校验与过期淘汰统一使用该函数，阻断 stale tick 继续交易。 |
| `data/width_cache_state_mixin.py` | 移除首次期权 tick 持锁 sleep 500ms；改为非阻塞标记 `other`，后续 tick 自动重分类。 |
| `config/_params_instrument_cache.py` | `InstrumentCacheService` 显式初始化 `RLock`；元数据懒加载改用 `get_existing_data_service()`，不再隐式创建完整 DataService。 |

| `infra/health_monitor.py` | 移除“订阅 kline 即视为 DuckDB 已落库”的假阳性判定。 |

## 六、修复后预期

1. 15:00 后如无新 tick，过期行情会被淘汰，不再触发基于旧价格的开仓。
2. 五态处理不再因首次期权 tick 持锁 sleep 造成 500ms 级阻塞，tick 处理超时和过载窗口应显著下降。
3. 元数据查询不会隐式拉起完整 DataService，SubscriptionManager 和诊断链路不会被数据库初始化拖慢。
4. DataService 启动不会默认扫描千万级 `ticks_raw` 做维护任务，启动和懒加载更稳定。
5. 新 tick 落库具备 `date` 和 `timestamp`，按日统计、15:05 后查询、最新行情判断恢复可信。
6. 20 合约诊断的 `12-DuckDB` 不再把订阅状态误报为落库成功。

## 七、仍需运行时复验的指标

需要重启策略后，在真实交易时段观察：

1. `tick处理超限`、`tick处理严重超时`、`过载丢弃窗口` 是否显著下降。
2. 15:05 之后或夜盘期间，`ticks_raw.max(timestamp)` 是否持续推进。
3. 过期 tick 是否出现淘汰日志，而不是继续触发 `OTHER_SCALP` 开仓。
4. IO/MO/HO 是否有当前交易日 tick；若仍为 0，边界应定位到平台推送/交易所行情源，而非本地配置。
5. `market_state=other` 占比是否下降，五态分类是否恢复到非 other 状态。
6. `OrderService result=None trading=False` 是否随 stale tick 开仓减少而下降。

## 八、结论

本轮核心根因不是旧报告中的 SubscriptionManager 断裂，也不是 CFFEX 期权配置仍为 0；当前配置已补齐。真正导致 17:24 仍异常交易的根因是：生产 tick 已停在 14:58:36，但 tick 时间解析未识别 `datetime`/字符串格式，过期行情没有被淘汰，策略继续用 stale price 触发交易。

同时，五态首次 tick 持锁等待、元数据查询隐式创建 DataService、DataService 启动重型全表维护、tick date/timestamp 缺失、诊断 DuckDB 假阳性共同放大了性能、可观测性和数据可信度问题。本轮已逐一修复，并通过编译、元数据查询、时间解析、订阅配置、真实库统计、最小落库读回验证。
