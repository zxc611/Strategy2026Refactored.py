## 二、深度验证发现的隐患及修复

### 隐患1：排序桶重算后被清空（致命）

- **发现阶段**: 深度验证V3
- **严重度**: 🔴 致命
- **根因**: `_sort_buckets[future_internal_id].clear()` 放在重算循环之后，把循环中 `_update_sort_bucket` 填充的结果全部清掉，等于白算了
- **影响**: 方向变化后排序桶为空，`select_otm_targets_by_volume` 返回空列表，无法开仓
- **修复**: 将 `_sort_buckets` 清空移到循环之前

**修复前**:
```python
# 循环中调用 _update_sort_bucket 填充排序桶
for iid in target_internal_ids:
    ...
    self._update_sort_bucket(iid, opt_info, future_internal_id, month, opt_type)

# 循环后清空 —— 把刚填充的结果全删了！
self._status_counts[future_internal_id] = dict(temp_status_counts)
self._sync_otm_count[future_internal_id] = dict(temp_sync_otm_count)
if future_internal_id in self._sort_buckets:
    self._sort_buckets[future_internal_id].clear()
```

**修复后**:
```python
# 循环前清空旧数据
if future_internal_id in self._sort_buckets:
    self._sort_buckets[future_internal_id].clear()

# 循环中重建
for iid in target_internal_ids:
    ...
    self._update_sort_bucket(iid, opt_info, future_internal_id, month, opt_type)

# 循环后替换计数字典（排序桶保留）
self._status_counts[future_internal_id] = dict(temp_status_counts)
self._sync_otm_count[future_internal_id] = dict(temp_sync_otm_count)
```

### 隐患2：首次tick不触发重算（致命）

- **发现阶段**: 深度验证V5
- **严重度**: 🔴 致命
- **根因**: `direction_changed = old_rising is not None and old_rising != new_rising`，首次tick时 `old_rising=None`，条件永远为False，导致初始状态永远不被计算
- **影响**: 策略启动后第一个期货tick不触发重算，期权五态全部为空，无法开仓
- **修复**: 增加 `not was_initialized` 条件

**修复前**:
```python
direction_changed = old_rising is not None and old_rising != new_rising
...
if direction_changed:
    self._recalc_all_state_for_underlying(future_internal_id)
```

**修复后**:
```python
was_initialized = self._future_initialized.get(future_internal_id, False)
...
direction_changed = old_rising is not None and old_rising != new_rising
...
need_recalc = direction_changed or (not was_initialized and self._future_initialized[future_internal_id])
if need_recalc:
    self._recalc_all_state_for_underlying(future_internal_id)
```

### 隐患3：冷却键格式不一致（中等）

- **发现阶段**: 深度验证V6
- **严重度**: 🟡 中等
- **根因**: `generate_signal` 中 `signal_type=''` 时冷却键为 `f"{instrument_id}_"`（末尾带下划线），但 `set_cooldown` 中 `signal_type=''` 时冷却键为 `instrument_id`（无下划线），两者不匹配导致冷却查找失败
- **影响**: 无signal_type的信号冷却检查失效，可能产生重复信号
- **修复**: 统一为 `if signal_type else instrument_id`

**修复前**:
```python
# generate_signal 中
cooldown_key = f"{instrument_id}_{signal_type}"  # signal_type='' → "inst_"

# set_cooldown 中
cooldown_key = f"{instrument_id}_{signal_type}" if signal_type else instrument_id  # → "inst"
```

**修复后**:
```python
# 两处统一
cooldown_key = f"{instrument_id}_{signal_type}" if signal_type else instrument_id
```

### 隐患4：补偿注册永远不生效（致命）

- **发现阶段**: 深度验证V8
- **严重度**: 🔴 致命
- **根因**: APScheduler 3.x 的 `scheduler.get_job(job_id)` 对不存在的job返回 `None` 而非抛异常，原代码 `try: get_job(); return; except: pass` 永远走 `return` 路径，补偿注册代码永远不会执行
- **影响**: `check_pending_orders` 定时任务永远不被补偿注册，挂单超时无人检查
- **修复**: 检查 `get_job()` 返回值是否为 `None`，且 `add_job` 包裹在 `try/except` 中

**修复前**:
```python
try:
    self._scheduler_manager.scheduler.get_job(job_id)
    return  # get_job返回None也走这里，永远return！
except Exception:
    pass
self._scheduler_manager.add_job_with_owner(...)  # 永远执行不到
```

**修复后**:
```python
try:
    existing = self._scheduler_manager.scheduler.get_job(job_id)
    if existing is not None:  # 只有job已存在才跳过
        return
except Exception:
    pass
try:
    self._scheduler_manager.add_job_with_owner(...)
    logging.info("[StrategyCoreService] 补偿注册check_pending_orders job成功")
except Exception as e:
    logging.error("[StrategyCoreService] 补偿注册check_pending_orders job失败: %s", e)
```

### 隐患5：止损重试全败后锁死（致命）

- **发现阶段**: 深度验证V13
- **严重度**: 🔴 致命
- **根因**: 首次下单失败时设置 `record._closing = True`，3次重试也全部失败后 `_closing` 仍为 `True`，导致下个风控周期检查 `if record._closing: return` 跳过该持仓，永远无法再尝试平仓
- **影响**: 止损失败后持仓被永久锁死，无法再次触发平仓，可能导致巨额亏损
- **修复**: 全部重试失败后重置 `_closing = False`

**修复前**:
```python
# 首次失败设_closing=True
else:
    record._closing = True
    need_retry = True

# 重试循环
if need_retry:
    for _retry in range(1, 4):
        ...
        if order_id:
            record._closing = True
            break
        # 重试全败后 _closing 仍为 True，下次风控周期跳过！
```

**修复后**:
```python
if need_retry:
    retry_success = False
    for _retry in range(1, 4):
        ...
        if order_id:
            record._closing = True
            retry_success = True
            break
    if not retry_success:
        with self._get_instrument_lock(record.instrument_id):
            record._closing = False  # 重置，允许下次风控重试
        logging.error("[PositionService._trigger_close_position] 全部重试失败，重置_closing以允许下次风控重试: %s", record.instrument_id)
```

### 隐患6：反向映射内存泄漏（轻微）

- **发现阶段**: 深度验证V11
- **严重度**: 🟢 轻微
- **根因**: `_platform_id_to_order_id` 反向映射只增不减，随着订单量增长会持续占用内存
- **影响**: 长时间运行后内存缓慢增长
- **修复**: `check_pending_orders` 中清理已完结订单的反向映射

**修复前**:
```python
def check_pending_orders(self) -> None:
    with self._lock:
        for order_id, order in list(self._orders_by_id.items()):
            if order['status'] != 'SUBMITTED':
                continue  # 跳过非提交状态，但不清理映射
```

**修复后**:
```python
def check_pending_orders(self) -> None:
    with self._lock:
        for order_id, order in list(self._orders_by_id.items()):
            if order['status'] not in ('SUBMITTED', 'PENDING'):
                pid = order.get('platform_order_id')
                if pid and str(pid) in self._platform_id_to_order_id:
                    del self._platform_id_to_order_id[str(pid)]
                continue
```
