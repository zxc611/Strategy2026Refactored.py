### 1.3 五态分类逻辑（核心Alpha引擎）

```
期货上涨 (future_rising=True)
  ├─ CALL 价格上涨 → correct_rise  (正确上涨)
  ├─ CALL 价格下跌 → wrong_rise    (错误上涨)
  ├─ PUT  价格上涨 → correct_fall  (正确下跌) [期货上涨时PUT价涨=正确下跌]
  └─ PUT  价格下跌 → wrong_fall    (错误下跌)

期货下跌 (future_rising=False)
  ├─ CALL 价格下跌 → correct_fall  (正确下跌)
  ├─ CALL 价格上涨 → wrong_fall    (错误下跌)
  ├─ PUT  价格上涨 → correct_rise  (正确上涨)
  └─ PUT  价格下跌 → wrong_rise    (错误上涨)
```

**排序公式**:
- **等权**: `cr_score = correct_rise - wrong_rise`（加权取整）
- **成交量加权**: `score = (correct_rise × w1 - wrong_rise × w2) × volume_factor`
- **宽度强度**: `width_strength = correct_rise / max(1, wrong_rise)`

## 阶段2: 参数分层

### 2.1 双分类体系

本策略参数采用**两套正交分类法**同时标注每个参数：

| 分类法 | 来源 | 维度 | 用途 |
|--------|------|------|------|
| **A/B/C/D** | 方案甲 | 代码可达性——参数在代码中的"读取层级" | 决定参数能否通过配置表生效 |
| **L1/L2/L3/L4** | 方案丁 | 决策层次——参数影响的策略决策类型 | 决定参数对绩效的一阶/二阶影响 |

### 2.2 A/B/C/D 代码可达性分类（方案甲）

> **核心原则**: "在默认表中存在" ≠ "主执行链已生效"。一个参数可能同时满足「是 DEFAULT_PARAM_TABLE 的真键」和「主执行链的代码并未读取该值」。本分类体系的两步验证法正是为了区分这两种状态。

#### 两步验证法

```
Step 1 — 表键确认: 该参数是否为 DEFAULT_PARAM_TABLE 的真键？
                → YES → 进入 A层候选池
                → NO  → 进入 B/C/D层判断

Step 2 — 读取确认: 主执行链的代码是否确实通过 _get_param_value()
                   或等价路径读取该键并用于决策？
                → YES → A-已确认 (confirmed, 可安全进入主参数池)
                → NO  → A-待确认 (pending, 需代码追踪补检)
```

| 层级 | 含义 | 判定依据 | 参数示例 | 实验适用性 |
|------|------|----------|----------|-----------|
| **A-已确认** | 表键存在 + 主链读取路径已通过代码追溯验证 | 代码体中可见 `_get_param_value(..., '该键')` 且该值最终被传递到策略决策分支（如排序/过滤/开仓判断） | `option_width_min_threshold`, `signal_cooldown_sec` | ✅ 直接进入主参数池 |
| **A-待确认** | 表键存在 + 主链读取路径**尚未完成代码追溯** | 在 DEFAULT_PARAM_TABLE 中能搜到该键，但在主执行链（tick→分类→排序→信号→下单）中未确认有明确的 `_get_param_value` 调用 | `close_max_hold_days`, `position_limit_max_ratio`, `close_max_chase_attempts` | ⚠️ **降级处理**：可纳入实验矩阵但结论需标注不确定性 |
| **B层** | 业务代码读取但未在默认表显式声明 | 代码有读取但未在`DEFAULT_PARAMS`中定义 | `_cooldown_durations`, 某些内部计数器 | ⚠️ 需确认读取路径后再入池 |
| **C层** | 测试注入变量 | 作为函数参数传入，非配置表键 | `top3_rows`/`target_top_n`/`ranking_mode` | ⚠️ 需Harness注入，不能靠JSON覆盖 |
| **D层** | 条件能力池参数 | 仅在被明确激活的模块中使用 | Greeks计算器的`price_change_pct`/`iv_change_pct` | ⚠️ 默认不启用，需确认模块已激活 |

### 2.5 主链验证方法论（A-待确认参数的补救流程）

> 本方法论是 §2.2"两步验证法"中 Step 2 的执行细则。目标是将每个 A-待确认参数逐一提升为 A-已确认或降级到正确的层级。

#### 验证流程（每个参数独立执行）

```
1. grep 键名 → 找到所有 _get_param_value / get_param / params.get 调用点
2. 追溯调用链 → 该值从 param_service 取出后，传递给了哪个变量？
3. 追踪变量 → 该变量最终被哪个主执行链路的方法/条件使用？
4. 对照主链路 → 该方法/条件是否属于
   tick→分类→排序→信号→下单→持仓 这条主执行链？
5. 判定:
   ├→ 在主链中被使用 → 升级为 A-已确认
   ├→ 在被使用但非主链（如仅用于日志/诊断）→ 降级为 D层
   ├→ 仅存在于表但代码无读取路径 → 降级为 E1淘汰候选
   └→ 搜索结果不明确 → 保持 A-待确认 + 标注不确定性
```

## 阶段3: 参数约束

### 3.1 约束体系概述（方案丁独有贡献）

本方案建立19条参数间约束关系（C1-C19），确保每组实验参数在**逻辑上合法**。违反约束的参数组将在**生成阶段即被拦截**，不进入实验执行。

### 3.2 约束全表

#### L1 信号生成层约束（4条）

| ID | 约束表达式 | 违反后果 | 严重度 |
|----|-----------|----------|--------|

| C7 | `sort_bucket_max_size ≥ top_n` | 排序桶溢出 | 🟡 高 |

| C14 | `option_width_month_count ≤ 5` | 查询不存在的月份 | 🟡 中 |

#### L2 信号过滤层约束（6条）

```python
"""
策略运行器 — 参数覆盖JSON→策略启动→日志解析→指标提取
来源: 方案丙 §2.2.2
"""

import json
import os
import re
import time
import logging
from typing import Dict, Any, Optional
from datetime import datetime

class StrategyRunner:
    """策略运行的封装，通过 param_override JSON 注入参数"""
    
    def __init__(self, config_dir: str = ".arts"):
        self.config_dir = os.path.abspath(config_dir)
        self.override_path = os.path.join(self.config_dir, "override_params.json")
        self.log_path = os.path.join(
            os.path.dirname(os.path.abspath(__file__)),
            "logs", "strategy.log"
        )
    
    def write_override(self, params: Dict[str, Any]) -> None:
        """写入参数覆盖文件"""
        os.makedirs(self.config_dir, exist_ok=True)
        with open(self.override_path, 'w', encoding='utf-8') as f:
            json.dump(params, f, indent=2, ensure_ascii=False)
    
    def remove_override(self) -> None:
        """清理覆盖文件"""
        if os.path.exists(self.override_path):
            os.remove(self.override_path)
    
    def run_strategy(self, duration_seconds: int = 900) -> float:
        """
        启动策略并运行指定时长
        
        Returns:
            目标函数值（越大越好）
        """
        start_time = time.time()
        time.sleep(duration_seconds)
        metrics = self._parse_performance_metrics()
        return self._compute_objective(metrics)
    
    def _parse_performance_metrics(self) -> Dict[str, float]:
        """从策略日志中提取性能指标"""
        metrics = {
            'total_signals': 0, 'total_trades': 0, 'win_rate': 0.0,
            'profit_loss_ratio': 0.0, 'sharpe_ratio': 0.0,
            'max_drawdown': 0.0, 'total_pnl': 0.0, 'avg_hold_minutes': 0.0,
        }
        if not os.path.exists(self.log_path):
            return metrics
        with open(self.log_path, 'r', encoding='utf-8') as f:
            log_content = f.read()
        patterns = {
            'total_signals': r'total_signals[:\s=]+(\d+)',
            'total_trades': r'total_trades[:\s=]+(\d+)',
            'win_rate': r'win_rate[:\s=]+([\d.]+)',
            'sharpe_ratio': r'sharpe[_\s]*ratio[:\s=]+([\d.]+)',
            'max_drawdown': r'max[_\s]*drawdown[:\s=]+([\d.]+)',
            'total_pnl': r'total[_\s]*pnl[:\s=]+([\d.-]+)',
        }
        for key, pattern in patterns.items():
            match = re.search(pattern, log_content, re.IGNORECASE)
            if match:
                try:
                    metrics[key] = float(match.group(1))
                except ValueError:
                    pass
        return metrics
    
    def _compute_objective(self, metrics: Dict[str, float]) -> float:
        """综合目标函数"""
        sharpe = metrics.get('sharpe_ratio', 0.0)
        trades = metrics.get('total_trades', 0)
        dd = metrics.get('max_drawdown', 0.0)
        if trades == 0:
            return -1000.0
        return sharpe * (1.0 - min(dd, 0.9))
```

```python
    # 方案丁约束检查 → 核心守卫
    is_valid, violations = validate_params(params)
    if not is_valid:
        raise optuna.TrialPruned()
    
    # 执行
    runner = StrategyRunner()
    runner.write_override(params)
    score = runner.run_strategy(duration_seconds=900)
    runner.remove_override()
    
    return score

def run_bayesian():
    sampler = TPESampler(n_startup_trials=10, seed=42, multivariate=True)
    study = optuna.create_study(
        study_name=STUDY_NAME, storage=STORAGE_URL,
        sampler=sampler, direction='maximize')
    study.optimize(objective, n_trials=30, show_progress_bar=True)
    
    print(f"Best trial: #{study.best_trial.number}, score={study.best_value:.4f}")
    for k, v in study.best_params.items():
        print(f"  {k} = {v}")
    return study

if __name__ == "__main__":
    run_bayesian()
```

### 5.4 引擎B: RSM响应面法（方案丙完整脚本）

```python
"""
Phase 2 RSM 响应面法 — 围绕贝叶斯最优邻域做统计验证
来源: 方案丙 §1.3
"""

import numpy as np
from sklearn.preprocessing import PolynomialFeatures
from sklearn.linear_model import LinearRegression
from sklearn.metrics import r2_score

# CCD 设计矩阵 (3因子 × 17组)
# 假设已从贝叶斯优化得到最优邻域 P* = (W*, C*, T*)
# CCD 围绕 (W*, C*, T*) 做 ±1 编码

def run_ccd_around_optimum(w_opt, c_opt, t_opt, half_width=1.0):
    """
    围绕贝叶斯最优 (w_opt, c_opt, t_opt) 做 CCD
    
    Args:
        w_opt: 宽度最优值
        c_opt: 冷却最优值(秒)
        t_opt: 止盈最优值
        half_width: 编码半距（实际值）
    """
    coded_to_real = {
        'W': lambda c: w_opt + c * half_width,
        'C': lambda c: c_opt + c * (c_opt * 0.4),
        'T': lambda c: t_opt + c * (t_opt * 0.2),
    }
    
    designs = []
    
    # 8 阶乘点
    for w in [-1, +1]:
        for c in [-1, +1]:
            for t in [-1, +1]:
                designs.append((w, c, t))
    
    # 6 轴点 (面心 α=±1)
    for axis, fix_val in [('W', 0), ('C', 0), ('T', 0)]:
        for sign in [-1, +1]:
            pt = [0, 0, 0]
            idx = ['W', 'C', 'T'].index(axis)
            pt[idx] = sign
            designs.append(tuple(pt))
    
    # 3 中心点
    for _ in range(3):
        designs.append((0, 0, 0))
    
    # 执行17组
    runner = StrategyRunner()
    real_values = []
    results = []
    
    for coded in designs:
        params = FIXED_PARAMS.copy()  # 来自贝叶斯脚本的固定参数
        params['option_width_min_threshold'] = coded_to_real['W'](coded[0])
        params['signal_cooldown_sec'] = int(coded_to_real['C'](coded[1]))
        params['close_take_profit_ratio'] = coded_to_real['T'](coded[2])
        
        is_valid, _ = validate_params(params)
        if not is_valid:
            continue
        
        runner.write_override(params)
        score = runner.run_strategy(duration_seconds=900)
        runner.remove_override()
        
        real_values.append([params['option_width_min_threshold'],
                           params['signal_cooldown_sec'],
                           params['close_take_profit_ratio']])
        results.append(score)
    
    # 拟合二阶模型
    poly = PolynomialFeatures(degree=2, include_bias=False)
    X_poly = poly.fit_transform(np.array(real_values))
    model = LinearRegression()
    model.fit(X_poly, results)
    R2 = r2_score(results, model.predict(X_poly))
    
    print(f"CCD 模型 R² = {R2:.4f}")
    feat_names = poly.get_feature_names_out(['W', 'C', 'T'])
    for name, coef in sorted(zip(feat_names, model.coef_),
                              key=lambda x: abs(x[1]), reverse=True):
        print(f"  {name}: {coef:.6f}")
    
    return model, R2, real_values, results

if __name__ == "__main__":
    # 从贝叶斯最优点出发
    run_ccd_around_optimum(w_opt=4.5, c_opt=25, t_opt=1.8)
```

```python
FACTOR_BOUNDS = {
    'option_width_min_threshold': (1.0, 8.0),

def generate_lhs(n_samples=36):
    n_factors = len(FACTOR_BOUNDS)
    sampler = qmc.LatinHypercube(d=n_factors, seed=42, optimization='random-cd')
    unit_samples = sampler.random(n=n_samples)
    
    names = list(FACTOR_BOUNDS.keys())
    real_samples = np.zeros_like(unit_samples)
    
    for i, name in enumerate(names):
        low, high = FACTOR_BOUNDS[name]
        real_samples[:, i] = low + unit_samples[:, i] * (high - low)
    
    return names, real_samples
```

```python
"""
Walk-forward 验证脚本
来源: 方案甲 §13
"""

def run_walk_forward(params_to_test: list[dict], data_days: int = 50):
    """
    Args:
        params_to_test: 从Phase 2筛选出的候选参数列表（每组一个dict）
        data_days: 可用数据总天数
    """
    window_size = 20
    valid_size = 5
    step = 5
    n_windows = (data_days - window_size - valid_size) // step + 1
    
    results = {f"P{idx}": {f"W{w}": None for w in range(n_windows)}
               for idx in range(len(params_to_test))}
    
    runner = StrategyRunner()
    
    for w in range(n_windows):
        train_start = w * step
        train_end = train_start + window_size
        valid_start = train_end
        valid_end = valid_start + valid_size
        
        print(f"\n=== Window {w}: Train[{train_start}:{train_end}] Valid[{valid_start}:{valid_end}] ===")
        
        for idx, params in enumerate(params_to_test):
            # 调整参数中的日期范围
            window_params = params.copy()
            window_params['simulation_start_date'] = f"2025-{train_start//30+1:02d}-{train_start%30+1:02d}"
            
            is_valid, violations = validate_params(window_params)
            if not is_valid:
                continue
            
            runner.write_override(window_params)
            score = runner.run_strategy(duration_seconds=600)
            runner.remove_override()
            
            results[f"P{idx}"][f"W{w}"] = score
    
    # 淘汰分析
    for idx in range(len(params_to_test)):
        scores = [results[f"P{idx}"][f"W{w}"] for w in range(n_windows)
                  if results[f"P{idx}"][f"W{w}"] is not None]
        n_入选 = len([s for s in scores if s > 0])
        
        if n_入选 < 2:
            print(f"P{idx}: ❌ WF-1 伪优解淘汰 ({n_入选}/6窗入选)")
        else:
            print(f"P{idx}: ✅ {n_入选}/6窗入选, 均值={np.mean(scores):.4f}")
    
    return results
```

### 常见问题 (FAQ)

**Q1: 贝叶斯优化得到的参数和L18正交的最优参数不一致怎么办？**

A: 相信贝叶斯优化。L18只有18组扫描8个因子，分辨率远低于贝叶斯30组的自适应收敛。但是**必须通过Walk-forward验证**——如果贝叶斯最优在6窗中只入选2窗以下，说明它只是单点偶然最优，应以Walk-forward结论为准。

**Q2: validate_params() 拦截了大量参数组，组数不够怎么办？**

A: 这是正常的——约束拦截的比例通常在5-15%之间。如果拦截超过30%，检查：
- 是否搜索空间范围过宽（如 `signal_cooldown_sec` 边界到了600，但 `rate_limit_window_sec=120` → C5约束大量触发）
- 应缩小搜索空间到合理范围再跑

**Q3: 什么时候应该重新从头跑完整参数池？**

A: 以下情况触发全量重跑：
- 策略代码有超过2个核心链路模块的大幅修改
- 新品种追加 > 5个
- 季度重跑发现最优参数漂移 > 50%
- 交易时段/规则变更（如夜盘取消）

**Q4: Walk-forward的6窗需要6组参数分别独立运行，能否并行？**

A: 技术上可以并行（如果仿真环境支持多个策略实例），但因为不同窗口用不同时间段的历史数据和相同参数的即时表现，并没有强依赖关系。**注意**: 6窗结果需基于**同一组参数**在不同时间段的运行表现。

**Q5: 生产可调参数的白名单区间如何确定？**

A: 
1. 取Walk-forward 6窗中入选次数 ≥ 4窗的前2-3组参数值
2. 取这些参数值的 min 和 max
3. 如果 max-min < 1档 → 升级为生产固定值
4. 如果 max-min ≥ 2档 → 保留为生产可调区间
5. 记录到治理表并同步到变更日志

**Q6: 治理总表中有些参数标注为「⚠️ 待确认」，这意味着什么？我该怎么处理？**

A: 「⚠️ 待确认」表示该参数**已确认存在于 DEFAULT_PARAM_TABLE 中**（表键验证 Step 1 通过），但**尚未通过 §2.5 的代码调用链追溯**（Step 2 待完成）。

具体到当前状态：
- `close_max_hold_days`、`position_limit_max_ratio`、`close_max_chase_attempts`、`close_take_profit_ratio` 这4个参数在默认表中都有明确的默认值定义，但你无法 100% 确定主执行链（tick→分类→排序→信号→下单→持仓）是否真的通过 `_get_param_value` 读取了它们。
- **它们不阻断冒烟池通过**（冒烟池假定表键默认值即生效）。
- **但它们影响 DOE 优化结论的可信度**——如果代码实际上不读取 `close_max_hold_days`，那么"调整持仓天数到7天发现收益率提升"这个结论就是虚假的（因为不管你设什么值，代码执行的结果都一样）。
- **处置流程**: 执行 §2.5 的 `grep` 验证 → 确认读取路径 → 升级状态 → 更新治理总表。如果验证发现某参数代码确实不读，则按 E1 规则淘汰出参数池。

**关键区别**: 「⚠️ 待确认」≠「参数无效」。它只是诚实地说"我们还缺一式验证"。在生产上线前，所有待确认参数必须完成代码追溯并在此表中更新状态。
