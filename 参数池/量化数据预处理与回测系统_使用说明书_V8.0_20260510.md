## 一、系统总览

本系统由四个模块组成，形成从原始Tick到P0验证绿灯的完整流水线：

```
原始Tick(Parquet)
      │
      ▼  preprocess_ticks.py
preprocessed.duckdb（分钟Bar + 真实五态 + BS Greeks + bid_ask_spread）
      │
      ▼  task_scheduler.py
quant_results.duckdb（回测绩效 + 最优参数 + 动态滑点 + 逻辑反转平仓）
      │
      ▼  sensitivity_analysis.py
sensitivity_results（参数敏感性报告）
      │
      ▼  参数池统一执行方案
生产策略参数锁定（P0绿灯）
```

| 模块 | 文件 | 功能 | 输入 | 输出 |
|------|------|------|------|------|
| 数据预处理 | `preprocess_ticks.py` | Tick→分钟Bar + **真实**五态/Greeks/bid_ask_spread | Parquet Tick | `preprocessed.duckdb` |
| 任务调度 | `task_scheduler.py` | 两轮分层网格扫描 + 多进程回测 + **动态滑点+逻辑反转** | `preprocessed.duckdb` | `quant_results.duckdb` |
| 敏感性分析 | `sensitivity_analysis.py` | OAT扰动 + 敏感度排名 + DuckDB持久化 | `preprocessed.duckdb` | `sensitivity_results` |
| 参数池 | `params_service.py` + `parameter_attribute_matrix.yaml` | 属性矩阵校验 + 审计日志 + 路由优先级 + **动态参数+source校验** | YAML | 运行时校验 + 审计日志 |

## 二、三大真实化优化（V8.0核心变更）

### 2.1 优化A：五态与Greeks真实化

**问题：** V7.0的`preprocess_ticks.py`中，`compute_option_state_vectorized()`和`compute_greeks_vectorized()`均为占位符实现，与生产代码的`width_cache._classify_status()`和`greeks_calculator.py`语义完全不一致。

**V8.0方案：**

#### 五态分类真实化

移植`width_cache._classify_status()`的核心判定表到向量化实现：

| 期权类型 | 期货上涨 | 期权上涨 | 五态结果 |
|----------|----------|----------|----------|
| CALL | ✓ | ✓ | **correct_rise** |
| CALL | ✗ | ✓ | **wrong_rise** |
| CALL | ✗ | ✗ | **correct_fall** |
| CALL | ✓ | ✗ | **wrong_fall** |
| PUT | ✗ | ✓ | **correct_rise** |
| PUT | ✓ | ✓ | **wrong_rise** |
| PUT | ✓ | ✗ | **correct_fall** |
| PUT | ✗ | ✗ | **wrong_fall** |

**所需数据列：** `underlying_price`（期货价）, `option_type`（CALL/PUT）, `close`（期权价）  
**降级策略：** 缺少关键列时自动降级为价格动量简化分类，并输出DEBUG日志

#### Greeks真实化

移植`greeks_calculator.py`的BS解析解+IV牛顿迭代到预处理流程：

| 函数 | 算法 | 移植源 |
|------|------|--------|
| `_bs_price_scalar()` | Black-Scholes定价公式 | `greeks_calculator.py:76-96` |
| `_bs_greeks_scalar()` | BS解析解delta/gamma/theta/vega | `greeks_calculator.py:98-153` |
| `_implied_volatility_scalar()` | 牛顿迭代法，容差1e-6，范围[0.01,5.0] | `greeks_calculator.py:162-220` |

**所需数据列：** `close`（期权市价）, `underlying_price`（标的价格）, `strike_price`（行权价）, `expire_date`（到期日）, `option_type`（CALL/PUT）  
**降级策略：** 缺少关键列时降级为日内波幅估算

### 2.2 优化B：动态滑点模型

**问题：** `SLIPPAGE_BPS = 1.0` 对流动性好的品种或许合理，但对远月或深度虚值期权，真实滑点远不止1bps。

**V8.0方案：**

预处理阶段从Tick数据的`bid_price1`/`ask_price1`聚合分钟级`bid_ask_spread`，回测中按流动性动态调整：

```python
def _compute_dynamic_slippage_bps(price, bid_ask_spread, base=1.0):
    """
    流动性好(spread小) → 使用base=1bps
    流动性差(spread大) → spread/price转为bps，取半
    """
    spread_bps = bid_ask_spread / price * 10000
    return max(base, spread_bps * 0.5)
```

**示例：**
| 场景 | bid_ask_spread | price | 动态滑点(bps) |
|------|----------------|-------|---------------|
| 平值近月 | 0.5 | 5000 | 1.0（取base） |
| 远月虚值 | 20 | 200 | 50.0（spread主导） |
| 深度虚值 | 50 | 50 | 500.0（spread主导） |

### 2.3 优化C：开平仓严格配对（逻辑反转平仓）

**问题：** V7.0的持仓检查仅等待止盈/止损/硬时间止损/EOD平仓，不会因信号逻辑反转而主动离场。

**V8.0方案：**

`_check_logic_reversal()` 在每个Bar循环中检测：如果持仓基于`CORRECT_RESONANCE`开仓，但当前Bar的五态信号强烈反转（`wrong_pct > correct_pct * threshold`且`wrong_pct > 0.3`），立即平仓离场。

```python
# 逻辑反转判定
if open_reason == "CORRECT_RESONANCE":
    if wrong_pct > correct_pct * logic_reversal_threshold and wrong_pct > 0.3:
        → 立即平仓（不等止损）
```

**参数：** `logic_reversal_threshold`（默认1.5，已加入YAML属性矩阵）  
**原理：** 顶级基金操作——信号逻辑消失的第一时间离场，而非等待价格止损。这避免了"共振→反转→持仓→止损"的大幅回撤路径。

## 三、数据预处理：preprocess_ticks.py

### 3.1 功能概要

将原始Tick数据（Parquet格式）聚合为分钟级Bar，同时计算**真实**衍生指标，写入DuckDB。

**衍生指标一览：**

| 类别 | 字段 | 来源 | V8.0变更 |
|------|------|------|----------|
| OHLCV | open, high, low, close, volume, tick_count | Tick聚合 | — |
| 资金流 | turnover, vwap | Tick聚合 | — |
| 持仓量 | open_interest | Tick聚合 | — |
| **流动性** | **bid_ask_spread** | **Tick聚合(ask1-bid1均值)** | **V8.0新增** |
| **期权结构** | **strike_price, expire_date, option_type, underlying_price** | **Tick聚合(取last)** | **V8.0新增** |
| 五态 | correct_rise/fall_pct, wrong_rise/fall_pct, other_pct, strength | `compute_option_state_vectorized()` | **V8.0真实化** |
| 订单流 | imbalance, consistency | `compute_order_flow_vectorized()` | — |
| Greeks | iv, delta, gamma, vega, theta | `compute_greeks_vectorized()` | **V8.0真实化(BS+IV牛顿迭代)** |

### 3.2 配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `TICK_DATA_DIR` | `/data/ticks` | 原始Tick Parquet文件目录 |
| `OUTPUT_DB` | `preprocessed.duckdb` | 输出DuckDB文件 |
| `MIN_DATE` | `2020-01-01` | 数据起始日期 |
| `MAX_DATE` | `2026-05-09` | 数据截止日期 |
| `SYMBOLS` | 49品种（已去重） | 待处理品种列表 |
| `MAX_WORKERS` | `min(cpu//2, len(SYMBOLS))` | 并行进程数 |
| `ROWS_PER_CHUNK` | 500,000 | 分块行数阈值 |

### 3.3 运行方法

```bash
# 直接运行
python ali2026v3_trading/preprocess_ticks.py

# 指定数据目录（修改脚本顶部 TICK_DATA_DIR）
# 或通过环境变量
export TICK_DATA_DIR=/data/ticks_prod
python ali2026v3_trading/preprocess_ticks.py
```

### 3.4 架构关键设计

**1. 多进程安全写入**

```
子进程1 → 临时Parquet_1
子进程2 → 临时Parquet_2      → 主进程串行合并 → preprocessed.duckdb
...
子进程N → 临时Parquet_N
```

**2. 分钟边界安全分块** — `_minute_safe_chunk_indices()`

**3. 数据质量过滤** — `_filter_bars()` 严格执行volume>0, OHLC>0, OHLC一致性, 日期范围

**4. 大品种按年拆分子任务** — `_build_task_list()` 超过2GB按年份拆分

**5. Tick数据期权结构字段自动传播（V8.0新增）**

聚合时自动检测并传播`strike_price`, `expire_date`, `option_type`, `underlying_price`列到分钟Bar，为真实Greeks计算提供必要输入。

### 3.5 输出验证

## 四、任务调度与回测：task_scheduler.py

### 4.1 功能概要

两轮分层参数网格扫描，多进程并行回测，结果存入DuckDB，输出Top5最优参数及样本外衰减。V8.0增加**动态滑点**和**逻辑反转平仓**。

### 4.2 两轮分层优化策略

| 轮次 | 扫描参数 | 组合数 | 任务数 | 策略 |
|------|----------|--------|--------|------|
| **Round1粗扫** | 6个核心交易参数 | 972 | 1,944 | 全排列，找Top10 |
| **Round2精扫** | 10个辅助参数 | 1,024/Top | 20,480 | 固定Round1核心，扫描辅助 |
| **合计** | 16参数全覆盖 | — | **22,424** | ~30分钟@14核 |

### 4.3 配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `PREPROCESSED_DB` | `preprocessed.duckdb` | 预处理数据源 |
| `RESULTS_DB` | `quant_results.duckdb` | 回测结果输出 |
| `INITIAL_EQUITY` | 1,000,000 | 初始权益 |
| `COMMISSION_PER_LOT` | 1.5 | 每手佣金 |
| `SLIPPAGE_BPS` | 1.0 | 基础滑点(bps)，动态模型下限 |
| `ROUND1_TOP_K` | 10 | Round1保留Top-K进入Round2 |
| `logic_reversal_threshold` | 1.5 | 逻辑反转平仓阈值（V8.0新增） |

### 4.4 运行方法

```bash
# 前提：preprocessed.duckdb 已生成
python ali2026v3_trading/task_scheduler.py
```

### 4.5 回测引擎逻辑（V8.0）

每个参数组合的回测执行完整V7策略闭环 + V8.0三大优化：

```
逐Bar循环:
  1. 日切重置（权益/断路器）
  2. 状态路由（五态→三态 + 确认机制）
  3. 逻辑反转平仓检测（V8.0新增）
     - CORRECT_RESONANCE持仓 + 信号反转 → 立即平仓
  4. 持仓检查（止盈 → 止损 → 硬时间止损 → 收盘平仓）
     - 平仓滑点: _compute_dynamic_slippage_bps(price, bid_ask_spread)
  5. 安全检查（日回撤硬停止 + 断路器暂停）
  6. 信号生成（strength > 0.3 + 冷却 + 持仓上限 + 一致性共振配对过滤）
  7. 开仓执行（风险比率约束手数 + 动态滑点 + 佣金扣减）
  8. 权益曲线追加
```

## 六、参数池基础设施

### 6.1 parameter_attribute_matrix.yaml

V8.0属性矩阵定义37个参数（V7.0为28个），每个参数均有`source`/`evidence`/`last_validated`字段：

**量化来源分类（铁律：无量化来源的参数不得锁定为生产值）：**

| 来源 | 含义 | 可否锁定生产值 |
|------|------|---------------|
| `quantified` | 已通过独立数据集优化+反事实验证 | ✓ |
| `backtest_grid` | 已通过参数网格回测优选 | ✓ |
| `goldilocks` | 已通过量化金发姑娘测试 | ✓ |
| `stress_scenario` | 已通过情景压力库扫描 | ✓ |
| `design` | V7架构设计决定（逻辑约束） | ✓ |
| `intuition` | 基于经验/直觉设定，待量化 | ✗ |

**当前intuition参数（5个，不可锁定生产值）：**
- `min_profit_threshold`, `max_theta_ratio`, `option_width_min_threshold`, `option_buy_lots_max`, `shadow_alpha_threshold`

### 6.2 ParamsService V8.0新增

| 功能 | 方法 | 说明 |
|------|------|------|
| **动态参数表达式** | `eval_dynamic_param(key, context)` | 根据YAML的`dynamic_expr`+`dynamic_depends`实时计算参数值，range clamp+类型转换+降级兜底 |
| **source=intuition生产锁定警告** | `validate_with_attribute_matrix()` | 自动检测source=intuition参数，输出INTUTION_LOCK警告 |
| **task_scheduler联动** | `_validate_params_via_params_service()` | 回测前通过ParamsService API加载校验参数，确保回测-实盘参数逻辑一致 |

### 6.3 动态参数表达式用法

在YAML中为参数添加`dynamic_expr`和`dynamic_depends`：

```yaml
signal_cooldown_sec:
  dynamic_expr: "base * (1 + (iv_percentile - 50) / 100)"
  dynamic_depends: [base, iv_percentile]
```

运行时调用：

```python
ps = get_params_service()
result = ps.eval_dynamic_param("signal_cooldown_sec", {"iv_percentile": 65})
# result = 60.0 * (1 + (65 - 50) / 100) = 60.0 * 1.15 = 69.0
```

## 七、P0最终绿灯检验（V8.0新增）

### 7.1 设计理念

回测结果只是"结果"，P0绿灯检验才是"证据"。`_run_final_checks()`是自动化的法官，对最佳参数组合执行不可妥协的硬编码检验，直接输出通过/不通过指令。

### 7.2 八项P0硬编码检验

| # | 检验项 | 阈值 | 性质 | 判定逻辑 |
|---|--------|------|------|----------|
| 1 | 样本外衰减 | < -30% | **不可妥协** | 衰减≥-30%通过，-20%~-30%警告 |
| 2 | 训练夏普最低门槛 | < 0.5 | **不可妥协** | 夏普≥0.5才满足可交易最低要求 |
| 3 | 样本外夏普正期望 | < 0.3 | **不可妥协** | 样本外必须正期望才可信 |
| 4 | 最大回撤生存红线 | < -50% | **不可妥协** | 超过-50%回撤不可生存 |
| 5 | 最少信号数 | < 30 | **不可妥协** | 信号数<30统计显著性不足 |
| 6 | 逻辑反转阈值合理性 | < 0.8 | **不可妥协** | 阈值<0.8频繁误平仓，<1.0警告 |
| 7 | 风险收益比 | 止盈≤止损 | **不可妥协** | 止盈/止损>1，否则长期必亏 |
| 8 | 参数来源intuition | 存在 | **不可妥协铁律** | 无量化来源的参数不得锁定生产值 |

### 7.3 三级判决

| 判决 | 条件 | 行动 |
|------|------|------|
| **P0绿灯** | 8项全部PASS，无WARN | 可进入小资金实盘测试 |
| **P0黄灯** | 8项全部PASS，但有WARN | 建议处理警告后进入实盘 |
| **P0红灯** | 任一FAIL | 必须调整参数或补充量化证据 |

### 7.4 输出示例

```
======================================================================
P0 最终绿灯检验 — 从结果到证据的法官审判
======================================================================
  [PASS] 样本外衰减=-15.2% >= -30%: 样本外稳健
  [PASS] 训练夏普=2.345 >= 0.5: 满足可交易门槛
  [PASS] 样本外夏普=1.989 >= 0.3: 样本外正期望
  [PASS] 样本外最大回撤=-0.182 >= -50%: 回撤可控
  [PASS] 信号数=256 >= 30: 满足统计显著性最低要求
  [PASS] 逻辑反转阈值=1.5 >= 1.0: 合理
  [PASS] 止盈/止损=1.5/0.5=3.0:1 风险收益比>1
  [PASS] 无intuition参数: 所有锁定参数均有量化来源
----------------------------------------------------------------------
  *** P0绿灯: 全部通过。可进入小资金实盘测试。***

```

### 7.5 集成位置

`_run_final_checks()` 在 `main_scheduler()` 末尾自动调用——回测完成后立即执行审判，无需手动触发。

## 八、故障排查

| 现象 | 原因 | 解决 |
|------|------|------|
| `preprocess_ticks.py` 报OOM | 单个Parquet文件过大 | 减小 `ROWS_PER_CHUNK`（如250000） |
| DuckDB `CatalogException` | 多进程并发写 | 确认使用临时Parquet架构（已修复） |
| 回测sharpe全为0 | `preprocessed.duckdb` 为空 | 先运行预处理，确认行数>0 |
| 五态降级为价格动量 | 缺少`underlying_price`/`option_type`列 | 确认Tick数据包含期权结构字段 |
| Greeks降级为波幅估算 | 缺少`strike_price`/`expire_date`列 | 确认Tick数据包含期权结构字段 |
| `bid_ask_spread`全为0 | Tick数据缺少`bid_price1`/`ask_price1` | 确认Tick数据包含盘口字段 |
| `INTUTION_LOCK` 警告 | source=intuition参数不可锁定生产值 | 为该参数补充量化来源（回测/金发姑娘/压力测试） |
| `Attribute matrix validation FAILED` | 参数值违反YAML约束 | 检查日志中的VIOLATION详情，修正参数 |
| Round2耗时过长 | `ROUND1_TOP_K` 过大 | 减小为5（默认10） |
