### 0.2 四条铁律（升级版）

1. **代码优先于文档，状态优先于参数** – 所有参数取值必须由当前市场状态和持仓希腊字母状态驱动，参数表仅提供初始边界。
2. **主试验与诊断层试验强制分离** – L0状态诊断准确性与稳定性必须单独验证，不得与策略寻优混淆。
3. **收益不可解释即淘汰** – 任何参数组合产生的PnL必须可归因为Delta、Gamma、Vega、Theta及订单流特征，否则标记E7并立刻退回。
4. **先治理后上线，先压力后实盘** – 必须通过新增的E7-E9淘汰编码、WF-6状态单调性检验，方可进入生产分级。

## 一、L-1 安全元层（最高权限）

### 1.1 三条硬规则

| 规则           | 参数                          | 默认值          | 代码位置                                              |
| :----------- | :-------------------------- | :----------- | :------------------------------------------------ |
| **速率断路器**    | `circuit_breaker_pause_sec` | 180.0        | `SafetyMetaLayer._check_circuit_breaker()`        |
| **持仓时间硬止损**  | `hard_time_stop_minutes`    | 90.0         | `SafetyMetaLayer.check_position_hard_time_stop()` |
| **持仓时间硬止损**  | `min_profit_threshold`      | 0.002 (0.2%) | 同上                                                |
| **日最大回撤硬停止** | `daily_drawdown_multiplier` | 2.0          | `SafetyMetaLayer._check_daily_drawdown()`         |

### 1.2 触发逻辑

**速率断路器**：

- 维护最近60个权益快照（deque）
- 计算1分钟内权益回撤百分比
- 若回撤 > max(2.5σ阈值, 2%硬底)，暂停交易180秒
- 暂停期间仅允许平仓，禁止新开仓

**持仓时间硬止损**：

- 在`position_service._check_time_stop()`中检查
- 读取`record._max_profit_pct`（tick回调中实时更新）
- 若持仓≥90分钟且最大浮盈从未达到0.2%，强制平仓
- 此逻辑独立于策略参数，不可被覆盖

**日最大回撤硬停止**：

- 每日重置，追踪日内峰值权益
- 若已知5日平均日收益：当前亏损 ≥ 5日均值 × 2.0 → 禁止新开仓
- 若未知5日平均日收益：回撤 ≥ 5% → 禁止新开仓

### 1.3 数据流

```
position_service.on_tick()
    └── 更新 record._max_profit_pct     ← 实时追踪最大浮盈

position_service.check_all_positions()   ← 周期性调用
    ├── _check_time_stop()
    │   ├── TimeStop@Nmin               ← 原有超时平仓
    │   └── HardTimeStop@Nmin           ← L-1硬时间止损
    └── safety.on_equity_update()        ← 驱动断路器+日回撤

risk_service.check_before_trade()
    └── _check_safety_meta_layer()       ← L-1最高权限检查
```

## 二、状态诊断层（L0）参数与路由

### 2.1 五态分类 → 三态路由

底层分类由`width_cache._classify_status()`实现，返回5种原始状态：

| 原始状态           | 含义                   | 路由映射                 |
| :------------- | :------------------- | :------------------- |
| `correct_rise` | 期货涨+Call涨/Put跌（方向正确） | `correct_trending`   |
| `correct_fall` | 期货跌+Put涨/Call跌（方向正确） | `correct_trending`   |
| `wrong_rise`   | 期货跌+Call涨/Put跌（方向错误） | `incorrect_reversal` |
| `wrong_fall`   | 期货涨+Call跌/Put涨（方向错误） | `incorrect_reversal` |
| `other`        | 无法判定或数据不足            | `other`              |

路由由`state_param_manager.StateParamManager`实现，映射规则定义在`_STATE_MAP`常量中。

### 2.2 状态选择逻辑

```
if 非other占比 ≥ 40%:
    从 correct_trending / incorrect_reversal 中选计数最大者
else:
    选择 other（防御模式）
```

此逻辑确保：当市场结构清晰（≥40%期权方向正确或错误）时选择对应策略，当市场混沌时进入防御模式。

### 2.3 状态参数集

参数集定义在`state_param_sets.yaml`中，运行时由`StateParamManager`加载：

| 参数                           | correct\_trending | incorrect\_reversal | other |
| :--------------------------- | :---------------- | :------------------ | :---- |
| `option_width_min_threshold` | 2.0               | 4.0                 | 4.0   |
| `signal_cooldown_sec`        | 15                | 120                 | 300   |
| `close_take_profit_ratio`    | 2.5               | 1.3                 | 1.1   |
| `close_stop_loss_ratio`      | 0.4               | 0.6                 | 0.8   |
| `max_risk_ratio`             | 0.8               | 0.3                 | 0.2   |
| `max_signals_per_window`     | 10                | 3                   | 2     |
| `lots_min`                   | 5                 | 2                   | 1     |
| `option_buy_lots_max`        | 100               | 30                  | 10    |

**切换连续性处理**：

- 切换瞬间**不强制平仓已有头寸**
- 切换至`other`状态时，触发防御性减仓模式（日志WARNING标记）
- PyYAML未安装时自动降级为内置默认参数集
- YAML还预置`correct_trending_defensive`参数集（介于trending和other之间），当前未接入路由，留待Phase 2状态细化后启用

### 2.4 L0 验证要求（阶段 0，强制前置）

任何策略进入后续测试前，必须先完成 L0 状态诊断的以下验证：

- **无未来函数**：`_classify_status()`仅使用当前tick价格和前一个tick价格，不使用未来数据
- **区分度检验**：至少三种状态下的后市收益率或波动率分布有显著差异（K-S检验 p < 0.05）
- **切换稳定性**：在随机时段内，状态切换次数不超过K线数量的30%，避免过度灵敏
- **覆盖完整性**：历史全部交易时段的状态分布中，`other` 类占比不超过40%，否则需细化状态定义

## 三、风险自适应层（L1-L3）参数深度动态化

### 3.1 希腊字母硬约束（已实现）

以下参数已在`risk_service._check_greeks_limits()`中实现，订单生成前强制检查：

| 参数                  | 类型    | 说明                | 基线值    | config\_params键名    |
| :------------------ | :---- | :---------------- | :----- | :------------------ |
| `max_net_delta_pct` | L1硬约束 | 组合净Delta占净值的百分比上限 | `0.30` | `max_net_delta_pct` |
| `max_net_gamma_pct` | L1硬约束 | 组合净Gamma的绝对值上限    | `0.08` | `max_net_gamma_pct` |
| `max_net_vega_bps`  | L1硬约束 | 净Vega上限（基点）       | `0.02` | `max_net_vega_bps`  |
| `max_theta_ratio`   | L1硬约束 | 最大负Theta与日预期收益的比率 | `0.5`  | `max_theta_ratio`   |

**实现细节**：

- 调用`greeks_calculator.GreeksCalculator()`实例的`check_risk_limits()`方法
- 传入当前持仓字典，计算组合Greeks
- 任一超限则返回`RiskCheckResponse.block_result("greeks_limit_exceeded")`
- 平仓订单（action="CLOSE"）跳过Greeks检查
- `GreeksCalculator`导入失败时静默跳过（不影响主链路）

### 3.2 决策得分协同计算（已实现）

`t_type_service.compute_decision_score()`实现期权状态强度与订单流方向的联合评分：

```
决策得分 = 期权状态强度(0-1) × 0.6 + 订单流方向一致性(-1到1) × 0.4
```

**期权状态强度**：由`_calc_option_state_strength()`计算，基于`width_cache._status_counts`中`correct_rise`+`correct_fall`占比

**决策规则**：

| 期权状态强度      | 订单流一致性    | 动作                      | 仓位缩放 |
| :---------- | :-------- | :---------------------- | :--- |
| 高 (>0.7)    | 正 (>0.3)  | `normal_open`           | 100% |
| 高 (>0.7)    | 负 (<-0.3) | `divergence_warning`    | 50%  |
| 中 (0.4-0.7) | 正         | `small_open_tight_stop` | 30%  |
| 低 (<0.4)    | 任意        | `no_open_wait`          | 0%   |

## 七、治理与淘汰编码最终版

| 编码   | 含义       | 自动判定条件                                  |
| :--- | :------- | :-------------------------------------- |
| E7   | 收益不可解释   | PnL归因分析中，不可解释残差占比 > 15%                 |
| E8   | 过度拟合尾部风险 | 极端事件贡献利润 > 总利润的 40%，且预案滑点修正后利润转负        |
| E9   | Minsky风险 | 低波动累积的杠杆或负Gamma仓位，在模拟波动率跳升2倍时净值回撤 > 20% |
| E10  | 状态依赖性过强  | 策略收益的80%以上来自某单一状态，其他状态持续亏损且无法通过参数隔离改善   |
| WF-6 | 状态单调恶化   | 见上述                                     |
| WF-7 | 参数脆弱性    | 参数值扰动 ±5% 导致业绩排名变化超过3位                  |

符合任意一条即触发自动降级，不得进入生产上线流程。

## 八、使用手册与场景执行路径

### 8.1 场景 A：新策略首次建立参数体系

1. 完成 L0 状态诊断开发与验证（阶段0）
2. 在`state_param_sets.yaml`中定义状态参数集
3. 按阶段 A → B(B1,B2,B3) → C → D 顺序执行，时间不少于 15 个交易日
4. 输出包含状态分层的绩效报表，标注所有淘汰标记

### 8.2 场景 B：季度调优

1. 重跑 L0 区分度检验，若市场状态分布漂移超过 20%，优先修正状态定义
2. 修改`state_param_sets.yaml`中的参数值（无需改代码，热加载）
3. 对生产参数函数中的系数进行小幅网格重寻优（±10%范围）
4. 若新参数未通过 WF-6 或 E7 检验，维持原生产值

### 8.3 场景 C：加入新品种（如指数期权、商品期权）

1. 验证新品种的 Greeks 计算准确性（`greeks_calculator.py`）
2. 检查新品种在不同状态下的流动性分布
3. 进行新品种独立冒烟，再与现有品种联合运行至少 1 周

### 8.4 场景 D：启动全自动 Phase 2 贝叶斯优化

1. L0-L3 所有动态函数系数均已提炼为连续可优化变量
2. 订单流参数范围稳定，微结构成本可建模
3. 建立仿真沙盒，能够实时计算希腊字母约束下的目标函数
4. 自动化迭代结果必须通过阶段 D 的 walk-forward 和治理编码检查后才可进入生产观察

### 8.5 场景 E：运行时状态路由调参

**修改参数集**（无需重启策略）：

```python
from ali2026v3_trading.state_param_manager import get_state_param_manager
spm = get_state_param_manager()
spm.reload_param_sets()  # 重新加载 state_param_sets.yaml
```

**查看当前状态**：

```python
spm.get_current_state()       # 'correct_trending' / 'incorrect_reversal' / 'other'
spm.get_params()              # 当前状态对应的参数字典
spm.get_state_distribution()  # {'correct_trending': 0.6, 'incorrect_reversal': 0.3, 'other': 0.1}
spm.get_stats()               # 完整统计信息
```

**查看安全元层状态**：

```python
from ali2026v3_trading.risk_service import get_safety_meta_layer
safety = get_safety_meta_layer()
safety.is_trading_paused()    # (False, '') 或 (True, '速率断路器: ...')
safety.is_new_open_blocked()  # 日回撤硬停止是否激活
safety.get_stats()            # {'circuit_breaker_triggers': 0, ...}
```

**计算决策得分**（自动从OrderFlowBridge获取订单流一致性）：

```python
from ali2026v3_trading.t_type_service import get_t_type_service
tts = get_t_type_service()
result = tts.compute_decision_score(
    future_internal_id=None,       # None=全品种聚合
)
# result = {'score': 0.58, 'state_strength': 0.72, 'flow_consistency': 0.35,
#           'action': 'normal_open', 'position_scale': 1.0, 'flow_source': 'auto'}

# 也可手工传入订单流值：
# result = tts.compute_decision_score(order_flow_imbalance=0.5)
```
