## 一、总原则

执行本手册必须遵守四条原则：

1. 代码优先于文档。参数是否进入主试验，以当前代码读取链为准，不以文档设想为准。
2. 主试验与条件试验分离。主试验只使用已验证读取点；排序模式、Greeks、注入变量作为条件能力试验单独统计。
3. 先功能后优化。先冒烟，再筛选，再压力，再 walk-forward，最后才进入 Phase 2 自动化寻优。
4. 先治理后上线。任何“实验最优”参数，都必须通过 E1-E6、区间稳定性和生产分级后，才能进入上线候选。

## 三、执行前守卫

每一组参数必须先通过守卫检查，再允许启动运行。

| 编码 | 约束 |
| --- | --- |
| C1 | close_take_profit_ratio > 1.0 |
| C2 | 0 < close_stop_loss_ratio < 1.0 |
| C3 | 0 < max_risk_ratio <= 1.0 |
| C4 | option_width_min_threshold >= 1.0 |
| C5 | signal_cooldown_sec <= rate_limit_window_sec |
| C6 | max_signals_per_window >= 1 |
| C7 | history_minutes >= 60 |
| C8 | history_load_batch_size >= 1 |
| C9 | history_load_max_workers >= 1 |
| C10 | max_hold_minutes >= 1 |
| C11 | subscription_batch_size >= 1 |
| C12 | subscription_interval > 0 |

补充规则：

1. 若参数属于观察参数或条件能力参数，必须加标签，不纳入主评分。
2. 任一组违反守卫，直接标记为“生成即淘汰”，不进入运行队列。

### 5.4 阶段 D，Walk-forward 验证

目标：淘汰伪优解，保留跨窗口稳健解。

当前推荐执行版：

1. 从阶段 B 选前 3 名。
2. 每组做 3 个滚动窗口，形成 9 次验证运行。
3. 若后续数据规模允许，再升级到 6 窗版本。

基础保留规则：

1. 3 窗中至少 2 窗收益为正或收益回撤比达标。
2. 参数排名不发生完全反转。
3. 回撤不出现失控放大。

升级版 6 窗规则，吸收治理方案：

1. WF-1：6 窗入选次数小于 2，退回研究池。
2. WF-2：最优窗与次优窗差异大于 2 倍标准差，标记不稳健。
3. WF-3：连续 2 窗以上目标函数单调下降，标记漂移。
4. WF-4：6 窗最优值漂移大于 1 档或 20%，退回研究池。
5. WF-5：任一验证窗生存能力显著劣化，标记致命。

### 5.5 阶段 E，Phase 2 可选升级

前提：

1. 阶段 A-D 已完成。
2. 已识别 5 个以内关键因子。
3. 已具备稳定的覆盖写入和结果汇总脚本。

推荐路径：

1. 快速寻优：贝叶斯优化 15 到 30 组。
2. 统计显著性：RSM CCD 15 组左右。
3. 空间补充：LHS 或 Sobol。

重要边界：

1. 当前仓库尚未验证完整自动注入与执行闭环，Phase 2 引擎应视为“可选升级蓝图”。
2. 只有在回测/仿真脚本链打通后，才建议切入自动化引擎。

## 七、使用手册

### 7.1 场景 A，首次建立参数基线

1. 跑阶段 A 12 组，确认链路无断点。
2. 跑阶段 B 18 组，筛出前 3 名。
3. 跑阶段 C 8 组，确认工程边界稳定。
4. 跑阶段 D 的 9 次验证，过滤伪优解。
5. 输出治理分级表，再决定是否进入 Phase 2。

### 7.2 场景 B，季度调优

1. 对比当前生产值与最近一个验证周期的候选区间。
2. 若偏离超过 20%，触发小范围重跑。
3. 先重跑阶段 B 的局部矩阵，再做 2 到 3 个窗口快速验证。
4. 若不优于当前生产值，不更新治理表。

### 7.3 场景 C，加入新品种

1. 先确认订阅、路由、tick_size、持仓和风控是否兼容。
2. 只对最可能受影响的参数做窄范围重跑。
3. 新旧品种共存跑一整天，确认旧品种表现未显著退化。

### 7.4 场景 D，准备自动化 Phase 2

1. 先确认参数覆盖文件、日志解析、结果汇总链打通。
2. 把 E1、E6 参数剔除出搜索空间。
3. 只保留 3 到 5 个核心参数给贝叶斯或 RSM。
4. 自动化结果仍需回到 walk-forward 和治理表，不得直接上线。

## 八、FAQ

### Q1 为什么不直接采用全生命周期治理方案里的完整引擎？

因为它的治理框架很成熟，但自动化执行链在当前仓库里仍属于可选蓝图，不宜直接当成现成能力。

### Q2 为什么一些参数在默认表里，却不纳入主执行清单？

因为“存在于默认表”不等于“已被主链稳定读取”。主执行清单只收录已验证读取链的参数。

### Q3 为什么等权排序不放进主清单？

因为当前主链仍以 `select_otm_targets_by_volume()` 到 `execute_by_ranking()` 为核心。等权排序更适合作为专项 A/B 条件试验。

### Q4 什么时候需要从头全量重跑？

1. 核心模块发生明显变更。
2. 品种结构明显扩张。
3. 生产最优区间漂移超过 50%。
4. 交易规则或时间结构发生实质变化。

## 九、执行检查清单

### 9.1 启动前

1. Python 运行环境正常。
2. 策略启动环境正常。
3. 日志路径可访问。
4. 参数覆盖流程可用。
5. 守卫校验函数已准备。

### 9.2 每组运行前

1. 守卫校验通过。
2. 覆盖内容已更新，无残留旧值。
3. 上次日志已归档或清理。
4. 本组属于主试验、观察试验还是条件能力试验，标签已标明。

### 9.3 结束后

1. 指标已汇总。
2. 淘汰编码已判定。
3. 是否进入下一阶段已记录。
4. 若涉及上线候选，治理表已更新。
