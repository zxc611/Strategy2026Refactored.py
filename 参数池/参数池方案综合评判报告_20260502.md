## 一、执行摘要

### 1.1 总结论

三份文档不是同一层级的替代关系，而是分别处于三个不同方法层：

1. [strategy_parameter_pool_20260501.md](strategy_parameter_pool_20260501.md) 是当前最完整、最接近可执行落地的“端到端研究方案”，强项是代码口径一致、DOE 与 walk-forward 已经成形。
2. [策略参数池测试方案_20260502.md](策略参数池测试方案_20260502.md) 是覆盖面最广、工程测试意识最强的“Phase 1 广谱测试方案”，强项是模块链路覆盖和压力测试设计，但参数口径混杂，部分映射仍偏概念化。
3. [策略参数池Phase2进阶方案详细设计_20260502.md](策略参数池Phase2进阶方案详细设计_20260502.md) 是方法论最先进、统计设计最强的“Phase 2 优化方案库”，强项是 RSM / 贝叶斯 / LHS 三套框架齐全，但它本身不是独立可执行方案，而是建立在 Phase 1 成果和额外自动化基建之上的二阶段方案。

补充看，新增的 [../../docs/策略参数池设计方案_20260501.md](../../docs/策略参数池设计方案_20260501.md) 不是当前最佳主方案，但它在“参数约束显式化”“三池执行颗粒度”“速查表和执行顺序”三个方面，具有很强的实操价值，值得被吸收进最终执行版清单。

进一步看，[参数池全生命周期治理方案_20260502.md](参数池全生命周期治理方案_20260502.md) 是目前“治理框架最完整、使用手册最像正式 SOP”的文档。它最强的不是新增参数，而是把定义、约束、实验、验证、淘汰编码、上线分级和 FAQ 串成了完整生命周期。

### 1.2 最关键判断

如果只能选一份作为当前主工作底稿，优先级应为：

1. [strategy_parameter_pool_20260501.md](strategy_parameter_pool_20260501.md)
2. [策略参数池测试方案_20260502.md](策略参数池测试方案_20260502.md)
3. [策略参数池Phase2进阶方案详细设计_20260502.md](策略参数池Phase2进阶方案详细设计_20260502.md)

原因不是后两份不专业，而是当前阶段最稀缺的不是“更复杂的方法”，而是“代码口径一致且能马上执行的分层试验方案”。

### 1.3 建议采用方式

最佳做法不是三选一，而是组合采用：

1. 用 [strategy_parameter_pool_20260501.md](strategy_parameter_pool_20260501.md) 作为主方案与主报告框架。
2. 从 [策略参数池测试方案_20260502.md](策略参数池测试方案_20260502.md) 吸收模块覆盖、压力池和执行模板。
3. 把 [策略参数池Phase2进阶方案详细设计_20260502.md](策略参数池Phase2进阶方案详细设计_20260502.md) 作为 Phase 2 优化方法库，而不是 Phase 1 主方案。
4. 从 [../../docs/策略参数池设计方案_20260501.md](../../docs/策略参数池设计方案_20260501.md) 吸收参数约束校验、冒烟/研究/压力三池的执行颗粒度，以及快速参考表写法。
5. 从 [参数池全生命周期治理方案_20260502.md](参数池全生命周期治理方案_20260502.md) 吸收 E1-E6 淘汰编码、生产固定/可调/研究/注入四类治理边界，以及使用说明手册结构。

## 二、主要发现

以下发现按重要程度排序。

### F1 高优先级发现

[策略参数池测试方案_20260502.md](策略参数池测试方案_20260502.md) 在“排序模式”描述上更像研究假设，不完全等于当前生产实现。

原因：

1. 当前主链路已经明确由 `select_otm_targets_by_volume()` 到 `execute_by_ranking()` 驱动。
2. 代码中未检索到 `equal_weight_round_robin`、`width_only`、`lots_mode`、`product_rank_weight` 这类可直接切换的生产实现。
3. 因此文档中“等权 vs 成交量加权”的表述更适合作为实验基线设计，而不应被写成“已存在的配置式切换”。

影响：

1. 作为研究文档是合理的。
2. 作为直接执行清单时，会高估当前系统已有的排序切换能力。

### F2 高优先级发现

[策略参数池Phase2进阶方案详细设计_20260502.md](策略参数池Phase2进阶方案详细设计_20260502.md) 的统计方法最强，但当前自动化落地链条并不完整。

证据：

1. 文档中给出了 `optuna_override.json`、策略运行器、日志解析器、Study 存储等完整框架。
2. 当前代码只检索到 `use_param_overrides_in_debug` 和 `param_override_table` 两个配置键定义，未检索到完整的参数覆盖读取与执行调用链。
3. 这说明该文档更像“高质量实施蓝图”，而不是已经能直接调用现有系统运行的脚本化方案。

影响：

1. 方法先进度高。
2. 但若把它直接当成现成执行方案，会低估二次开发成本。

### F3 中优先级发现

[strategy_parameter_pool_20260501.md](strategy_parameter_pool_20260501.md) 在代码口径、DOE、walk-forward、治理表四方面最均衡，但广谱覆盖面弱于 [策略参数池测试方案_20260502.md](策略参数池测试方案_20260502.md)。

表现为：

1. 该方案对主收益链路、排序、风控、walk-forward 和上线治理的描述最完整。
2. 但在 UI、监控、异常恢复、长时浸泡测试、多交易所极限订阅等工程性压力设计上，没有后者覆盖得广。

影响：

1. 更适合作为主研究方案。
2. 如果目标是上线前的系统级压测，需要补入后者的压力设计元素。

### F4 中优先级发现

[策略参数池测试方案_20260502.md](策略参数池测试方案_20260502.md) 的优点是“先全链路覆盖后做研究”，但其参数空间偏宽，容易把 Phase 1 与 Phase 2 混在一起。

表现为：

1. 一份文档同时覆盖模块地图、冒烟池、研究池、压力池、进阶方法建议。
2. 在阅读体验上非常完整。
3. 但执行时容易出现“还没完成主效应筛选，就已经把过多维度推进到研究池”的情况。

影响：

1. 适合团队快速建立全局认知。
2. 不适合直接作为最小闭环执行脚本。

### F5 中优先级发现

[../../docs/策略参数池设计方案_20260501.md](../../docs/策略参数池设计方案_20260501.md) 在执行工程化层面有明显增量价值，尤其是“先约束校验，再分池执行，再按阶段升级”的组织方式。

表现为：

1. 它显式列出 19 条参数约束关系，把非法组合排除前置化，而不是等运行时报错。
2. 它把冒烟池、研究池、压力池拆得更细，且每组测试目标都写得更像任务单。
3. 它提供了速查表、推荐实施路径和约束验证清单，便于转成实际执行 checklist。

影响：

1. 它适合被吸收为“执行层模板”。
2. 但它本身仍不是最佳主方案，因为部分参数维度与能力链假设仍需按当前代码口径再收敛。

### F6 高优先级发现

[参数池全生命周期治理方案_20260502.md](参数池全生命周期治理方案_20260502.md) 是当前最完整的生命周期治理文档，但它对“哪些参数已经被主链稳定读取”仍有少量高估。

证据：

1. 文档把 `top3_rows`、`close_max_hold_days`、`position_limit_max_ratio` 放入较强执行语境。
2. 当前代码检索显示，这三个键在 [ali2026v3_trading/config_service.py](ali2026v3_trading/config_service.py#L1412) 到 [ali2026v3_trading/config_service.py](ali2026v3_trading/config_service.py#L1465) 中已存在默认表定义，但未检索到当前主交易链中的稳定读取点。
3. 与之相对，`option_width_min_threshold`、`signal_cooldown_sec`、`max_signals_per_window`、`rate_limit_window_sec`、`max_risk_ratio`、`total_position_limit`、`close_take_profit_ratio`、`close_stop_loss_ratio`、`max_hold_minutes`、`history_minutes` 等参数有更清晰的读取链路。

影响：

1. 这份文档非常适合做治理总纲和使用手册。
2. 但若不做代码口径收敛，就会把“存在于默认表”误当成“已在主链生效”。

## 三、逐份专业评判

## 3.1 对 [strategy_parameter_pool_20260501.md](strategy_parameter_pool_20260501.md) 的评判

### 优点

1. 代码口径最一致。参数池明确区分 A/B/C/D 四层，能区分统一配置键、扩展参数、测试注入变量和条件能力池。
2. 研究路径最闭环。从参数依据、DOE 矩阵、walk-forward、参数治理表，一直到上线建议，逻辑完整。
3. 对当前生产链路尊重度最高。明确承认当前主排序链路是 `select_otm_targets_by_volume()` 到 `execute_by_ranking()`，没有把等权模式写成现成实现。
4. 可解释性最强。每一层都回答“为什么要测这个”“怎么知道测够了”“什么是伪优解”。
5. 最接近生产研究流程。已经把 DOE 和样本外验证串起来，不是停留在“枚举参数”阶段。

### 缺点

1. 工程压测颗粒度不如 [策略参数池测试方案_20260502.md](策略参数池测试方案_20260502.md) 细，缺少长时浸泡、死循环、日志膨胀等更具体的极限场景脚本化描述。
2. 对全模块地图、功能矩阵、异常注入路径的说明不如对比方案直观。
3. DOE 是正式化的，但仍以手工矩阵为主，缺少直接可运行的调度脚本模板。

### 适用场景

1. 作为主研究方案。
2. 作为上线前参数治理底稿。
3. 作为团队统一讨论“什么叫参数有效、什么叫伪优解”的标准文档。

### 综合判断

这是三份文档里当前最值得作为主版本维护的方案。

## 3.2 对 [策略参数池测试方案_20260502.md](策略参数池测试方案_20260502.md) 的评判

### 优点

1. 全局感最强。27 模块矩阵、核心数据流、五态分类逻辑、测试分层、执行规划一体化呈现，阅读门槛低。
2. 工程测试意识最强。压力池设计非常具体，覆盖高频零冷却、追单死循环、内存泄漏、跨日持仓、日志膨胀等真实故障模式。
3. Phase 1 思维清晰。冒烟池、研究池、压力池的结构明确，便于团队先做功能性验证再做研究。
4. 执行模板友好。给出了参数 JSON 模板和执行阶段规划，利于协作交接。

### 缺点

1. 参数口径不够纯。它把“代码已读参数”“可能存在的业务参数”“研究假设变量”放在同一大表里，容易让执行者误以为都能直接配置。
2. 排序逻辑描述存在前移。文档把“等权排序 vs 成交量加权”写得较强，但当前代码检索没有给出直接可切换的现成实现，更多仍需测试层注入或改造。
3. 从 Phase 1 到 Phase 2 的边界不够硬。对于真正的统计识别和样本外验证，它给出了方向，但没有像 [strategy_parameter_pool_20260501.md](strategy_parameter_pool_20260501.md) 那样落成正式 DOE 和 walk-forward 规则。
4. 维度较宽，容易把一次迭代做成“大而全”，导致单轮执行成本偏高。

### 适用场景

1. 作为测试总纲或团队 onboarding 文档。
2. 作为系统级冒烟与压力测试手册。
3. 作为补充 [strategy_parameter_pool_20260501.md](strategy_parameter_pool_20260501.md) 工程覆盖面的副文档。

### 综合判断

这份文档在“全局覆盖”和“压力测试设计”上优于主方案，但在参数治理严谨性和代码口径一致性上弱于主方案。

## 3.3 对 [策略参数池Phase2进阶方案详细设计_20260502.md](策略参数池Phase2进阶方案详细设计_20260502.md) 的评判

### 优点

1. 统计方法最专业。RSM、PBD-12、CCD、Optuna、LHS、决策矩阵、执行检查清单都给到了操作级细节。
2. 方法库属性极强。不是只讲概念，而是给了公式、设计矩阵、代码框架和输出物定义。
3. 对“什么时候用什么方法”讲得最好。它不是只推一种方法，而是给了因子数量、目标类型、运行预算不同情况下的方案选择路径。
4. 对后续深度优化非常有价值。尤其适合作为 Phase 2 和 Phase 3 的专项优化文档。

### 缺点

1. 不是独立主方案。它明确依赖 Phase 1 结果，包括因子敏感性排序、Top 3 因子确认、目标函数定义确认。
2. 自动化闭环尚未就位。文中的 Optuna 运行器、覆盖文件、日志解析、study 恢复、目标函数计算，目前更像待接入的脚手架，不是现成能直接运行的仓库能力。
3. 对当前代码链路的直接约束较少。它关注的是优化方法，而不是当前哪些参数已经确认真实可达、哪些只是未来可注入变量。
4. 若在 Phase 1 未完成前过早采用，会产生“方法很先进，但输入不可靠”的问题。

### 适用场景

1. 在 Phase 1 已经筛出关键因子后，进入精细优化阶段。
2. 有资源做自动化回测管线时，作为实施蓝图。
3. 团队需要统计显著性和优化自动化时，作为二阶段方法总库。

### 综合判断

这份文档不是当前阶段的主执行方案，但它是最有研究升级价值的文档。它应该被定位为“Phase 2 方法论与实施蓝图”，而不是直接替代 Phase 1 主方案。

## 3.4 对 [../../docs/策略参数池设计方案_20260501.md](../../docs/策略参数池设计方案_20260501.md) 的评判

### 优点

1. 参数约束意识最强。它把 19 条参数约束显式列出，这一点对避免无效回测和提前拦截非法组合很有价值。
2. 执行清单感最强。冒烟池、研究池、压力池以及速查表的写法，非常接近测试任务单，便于直接分派执行。
3. 控制变量思想明确。压力池强调每次只压 1 到 2 个极端维度，便于归因，这一点值得保留。
4. 对“为什么不是全排列”解释得直观，适合作为团队说明材料。

### 缺点

1. 它对参数空间的统计和模块归类有价值，但并非所有维度都已严格映射到当前主代码的统一配置键。
2. 部分 lots、Greeks、排序等设计仍包含实验层假设，不能直接视为现成生产开关。
3. 与 [strategy_parameter_pool_20260501.md](strategy_parameter_pool_20260501.md) 相比，它在 DOE、walk-forward、治理闭环上仍然偏弱。

### 适用场景

1. 作为执行清单模板。
2. 作为参数约束校验附录。
3. 作为把研究方案落地成任务单时的中间文档。

### 综合判断

这份文档最适合被“吸收”，而不是被“替代采用”。它对最终执行版清单的价值高于对主方案排序的价值。

## 3.5 对 [参数池全生命周期治理方案_20260502.md](参数池全生命周期治理方案_20260502.md) 的评判

### 优点

1. 生命周期最完整。它把参数定义、双分类、约束、实验设计、执行引擎、walk-forward、治理分级、FAQ、检查清单全部串成一条线。
2. 治理语言最成熟。E1-E6 剔除编码、四类生产治理边界、治理输出模板，已经接近正式投产文档格式。
3. 使用手册最实用。首次上线、季度调优、加入新品种三种场景写得清楚，适合团队协作和交接。
4. 文档整合能力最强。它把前面几份文档的强项几乎都吸了进去，结构上也最像最终统一版。

### 缺点

1. 代码口径仍需二次收敛。部分参数虽然在默认表存在，但未验证为当前主链稳定读取点，不宜直接写成主执行参数。
2. 执行引擎部分仍偏蓝图。Optuna、RSM、LHS 和 `StrategyRunner` 的价值更接近未来自动化脚手架，而非当前仓库现成能力。
3. 个别表述仍会给人“等权排序和某些 lots/持仓参数已可直接开关”的印象，这一点需要继续收敛到已验证链路。

### 适用场景

1. 作为最终统一版文档的骨架。
2. 作为团队使用手册和治理总纲。
3. 作为后续把执行清单、治理表、变更流程合并时的主模板。

### 综合判断

这是目前最适合被升级成“正式统一手册”的文档骨架，但前提是把主执行参数严格收敛到已验证读取点，把自动化引擎明确标注为可选升级路径。
