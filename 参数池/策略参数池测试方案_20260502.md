### 1.3 五态分类逻辑（核心引擎）

```
期货上涨 (future_rising=True)
  ├─ CALL 价格上涨 → correct_rise  (正确上涨，最优状态)
  ├─ CALL 价格下跌 → wrong_rise    (错误上涨)
  ├─ PUT  价格上涨 → correct_fall  (正确下跌) [注：期货上涨时PUT正确下跌]
  └─ PUT  价格下跌 → wrong_fall    (错误下跌)

期货下跌 (future_rising=False)
  ├─ CALL 价格下跌 → correct_fall  (正确下跌)
  ├─ CALL 价格上涨 → wrong_fall    (错误下跌)
  ├─ PUT  价格上涨 → correct_rise  (正确上涨)
  └─ PUT  价格下跌 → wrong_rise    (错误上涨)
```

**排序逻辑**:
- 等权排序: 仅按 `cr_score = correct_rise - wrong_rise` 排序
- 成交量加权: 在等权基础上乘 `volume` 权重因子
  - `score = (correct_rise * w1 - wrong_rise * w2) * volume_factor`
  - 宽度强度: `width_strength = correct_rise / max(1, wrong_rise)`

## 二、参数池总体设计

### 2.1 设计方法论

**参数选取原则**:
1. **覆盖关键决策分支**: 每个参数取值必须改变策略实际的决策路径
2. **参数正交化**: 尽量减少参数间的强相关（减少冗余组合）
3. **边界-常态-边界**: 覆盖下限、默认值、上限三个区段
4. **模块联动**: 考虑跨模块参数联动的组合爆炸风险

**排列组合规模控制**:
- 全排列会达到 `3^15 ≈ 14 million` → 不可行
- 采用 **正交表 + 关键路径覆盖** 策略，控制在可执行范围内

### 2.2 参数分类与维度

| 维度 | 参数族 | 关键参数 | 值域 | 控制逻辑 |
|------|--------|---------|------|---------|
| **P1 时间驱动** | K线周期 | `kline_style` | M1/M5/M15/H1 | 信号触发频率 |
| **P2 数据窗口** | K线窗口 | `max_kline` | 50/200/500 | 趋势判定基准 |
| **P3 排序算法** | 排序模式 | `min_option_width`+`lots_min` | 权重比 | 等权 vs 量加权 |
| **P4 宽度阈值** | 宽度过滤 | `option_width_min_threshold` | 1.0~8.0 | 信号质量门槛 |
| **P5 月份跨度** | 跨月范围 | `option_width_month_count` | 2~5 | 到期周覆盖 |
| **P6 冷却控制** | 信号冷却 | `signal_cooldown_sec` | 0~300 | 交易频率 |
| **P7 风控约束** | 风险敞口 | `max_daily_loss`+`position_limit_max_ratio` | 宽松/标准/严格 | 持仓上限 |
| **P8 出场策略** | 止盈止损 | `close_take_profit_ratio`+`close_max_hold_days` | 快/中/慢 | 盈亏比控制 |
| **P9 追单策略** | 成交追击 | `close_max_chase_attempts` | 1/3/5 | 成交保障 |

| **P11 Tick生成** | 数据模式 | `use_tick_kline_generator`+`backtest_tick_mode` | True/False | Tick源切换 |
| **P12 希腊计算** | IV更新 | `price_change_pct`+`iv_change_pct` | 0.1~5.0 | 希腊刷新频率 |

### 5.2 压力池详细说明

**STR-01「全品种饱和压测」**
> 配置: 6交易所全部启用的期货→对应全部期权(~40+品种 × 2月 × N行权价)  
> 预期: 订阅队列饱和 → 验证 subscription_batch_size 和 rate_limit 机制是否正常削峰  
> 监控: 内存占用 < 2GB, CPU < 80%, 订阅不超时  
> 风险: 平台API限流触发 → 验证降级/重试策略

**STR-02「高频零冷却压测」**
> 配置: M1 K线 + 零冷却 + 最低阈值 + 极小max_kline(10) → 模拟信号洪水  
> 预期: 信号限频 + 风控阻塞 → 确认 rate_limit 和 risk_service 通路  
> 监控: signals_emitted vs signals_filtered 的过滤比  
> 风险: 信号队列溢出 → EventBus背压(DEFAULT_BACKPRESSURE_LIMIT=5000)

**STR-03「内存泄漏找茬」**
> 配置: STR-01全品种 + 运行4小时 + 不停止  
> 重点检查: `_cooldown_times/_cooldown_durations` 僵尸键、`_sort_buckets` 死月份  
> 验证: memory_profiling (enable_memory_profiling=True) 趋势线是否平  
> 风险: `_option_volume` 累计量永不清理→单调增长

**STR-04「追单死循环压测」**
> 配置: 极小tick_size品种 + 10次追单 + 零冷却  
> 预期: chase_count上限生效，不会无限追单  
> 监控: _chase_tasks 字典不会无限增长  
> 风险: 追单任务堆积未清理→内存泄漏

**STR-05「超长持仓+日切压测」**
> 配置: close_max_hold_days=30, 跨天/周末模拟  
> 预期: close_overnight_check_time="14:58"日切逻辑触发

## 六、设计依据说明

### 6.1 为什么选取这12个参数维度？

选取原则遵循 **「决策分支覆盖率」**：

1. **P1-P3 (时间-数据-排序)**: 这三个参数控制策略**何时**发出信号。改变它们直接改变`select_otm_targets_by_volume`被调用的频率和输入数据量。
   - `kline_style`: 直接控制 `StrategyScheduler` 添加的交易周期Job间隔
   - `max_kline`: 决定 `HistoricalKlineMixin` 的趋势判定基准长度
   - `min_option_width` + `lots_min`: 等权(=1,=1) vs 量加权(>1,>3)的开关

2. **P4-P5 (宽度-月份)**: 这两个参数控制**选什么**期权。改变 `option_width_min_threshold` 和 `option_width_month_count` 直接影响 `WidthStrengthCache` 的过滤和跨月逻辑。这是与 H7 修复(跨月迭代)强相关的参数。

3. **P6-P8 (冷却-风控-出场)**: 控制**能否交易**和**何时平仓**。改变冷却时间 (`signal_cooldown_sec`) 和风控参数直接改变 `RiskService.check_before_trade` 的返回结果。

4. **P9 (追单)**: 控制**订单执行**的顽强程度。与 E3(tick_size溢价) + H21(递增加价)联动。

5. **P10 (订阅范围)**: 压力测试专用，验证多品种下订阅管理器 (`subscription_manager`) 的批量订阅和限频机制。

6. **P11 (Tick源)**: 控制数据进入 `WidthStrengthCache` 的源头——实tick vs Tick合成K线再分解。这是策略TickHandlerMixin的最上游开关。

7. **P12 (希腊计算频率)**: 影响 `GreeksCalculator` 的计算开销和Delta/Gamma暴露准确性。

### 6.2 排列组合够不够？

**答案：在当前方法论下足够；但从纯统计完备性角度不够。** 具体分析：

**够了的部分**:
- 冒烟池 3组 → 覆盖 全链路通 + 默认参数通 + 实tick路由通 ✅
- 研究池 L18(8因子×3水平×18组) → 覆盖所有两因子组合至少一次 ✅
- 压力池 6组 → 每种极限边界至少1组 ✅

**不够的部分**:
- 三因子及以上交互效应未被L18正交表覆盖（L18只能保证所有两两组合出现）
- 品种级别的差异化参数（每个品种的行权价数量、tick_size差异）未被纳入参数池
- **强烈建议**: 通过研究池结果做**响应面分析**，定位关键因子后做 2-3 因子的 **全因子实验** (Full Factorial, 3×3×3=27组) 针对最关键的前3个因子

**具体缺口估算**:

| 场景 | 缺失 | 影响 |
|------|------|------|
| 三因子交互 | `A×B×C` → 8组合仅3组出现 | 中等：K线周期×窗口×宽度阈值的三联动未探测 |
| 品种差异化 | 每品种独立参数(如`tick_size=0.2` vs `0.05`) | 低：品种level差异体现在压力池全品种 |
| 希腊联动 | `price_change_pct × iv_change_pct` + 五态宽度 | 低：希腊变化通过宽度间接影响信号筛选 |
| 跨日/周末 | 连续运行7×24 | 建议增加**浸泡测试** (Soak Test) |

### 6.3 更好更专业的参数池设计方案

以下提供 **三种进阶方案**，可逐步采纳：

#### 方案1: 响应面法 (RSM, Response Surface Methodology) ⭐ 推荐

**流程**: 冒烟池 → 筛选实验(Screening, Plackett-Burman设计, 8组) → 识别关键2-3因子 → 中心复合设计(CCD, 15组) → 响应面拟合 → 最优参数定位

**优势**: 
- 用最少的组数(8+15=23组)获得精确的最优参数预测
- 统计显著性可量化(P值、R²)
- 学术/工程界工业标准

**劣势**: 需要统计软件(JMP/Minitab/Python scipy.optimize)辅助分析

#### 方案2: 贝叶斯优化 (Bayesian Optimization)

**流程**: 先验参数分布 → 每组运行后更新后验 → TPE(Tree-structured Parzen Estimator)选择下一组参数 → 迭代收敛

**优势**:
- 每组运行都学习，后组更聪明
- 适合参数空间大但每组运行成本高(>30分钟)的场景
- Python库支持: `optuna`, `hyperopt`, `scikit-optimize`

**推荐配置**:
```python
import optuna
study = optuna.create_study(direction='maximize')
study.optimize(objective_strategy, n_trials=50)
```

#### 方案3: 拉丁超立方 (LHS, Latin Hypercube Sampling)

适用场景: 当需要**均匀覆盖高维参数空间**而非正交覆盖时。

**优势**: 在15维空间中用100组参数获得比正交表均匀得多的覆盖
**劣势**: 100组运行成本高（6所全品种×M1周期≈15分钟/组 → 总25小时）

**推荐规模**: 
- LHS-36: 6因子 × 6分层 → 36组（预计9小时）
- LHS-64: 8因子 × 8分层 → 64组（预计16小时）
