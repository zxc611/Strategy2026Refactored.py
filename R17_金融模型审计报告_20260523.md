## 维度一：Black-Scholes模型假设验证

### 问题 R17-01 [P1] Theta归一化与use_trading_days标志不一致

- **文件**: `greeks_calculator.py`
- **行号**: 137-145
- **影响系统**: 生产系统
- **问题描述**: 当`use_trading_days=True`时，T使用交易日计算（T = trading_days/252），但theta仍然除以365.0（日历日）。这导致理论不一致：如果T以交易日年化，theta应当除以252（交易日）而非365（日历日）。

  ```python
  # 行137-139 (CALL theta)
  theta = (-math.exp(-q * T) * S * _norm_pdf(d1) * sigma / (2 * math.sqrt(T))
           - r * K * math.exp(-r * T) * _norm_cdf(d2)
           + q * S * math.exp(-q * T) * _norm_cdf(d1)) / 365.0  # ← 硬编码/365
  # 行143-145 (PUT theta)
  theta = (-math.exp(-q * T) * S * _norm_pdf(d1) * sigma / (2 * math.sqrt(T))
           + r * K * math.exp(-r * T) * _norm_cdf(-d2)
           - q * S * math.exp(-q * T) * _norm_cdf(-d1)) / 365.0  # ← 硬编码/365
  ```

  当`use_trading_days=True`时，BS公式中的T以交易日为基准，theta的"每日衰减"语义也应为"每交易日衰减"。当前实现导致theta被低估约30.6%（365/252-1≈44.8%的差异在时间基准上的投影）。

- **理论依据**: BS模型中theta = ∂V/∂t，当T以交易日年化时，∂t的增量应为1/252而非1/365。

### 问题 R17-02 [P2] 无BS假设失效检测与降级机制

- **文件**: `greeks_calculator.py`
- **行号**: 全文件
- **影响系统**: 生产系统
- **问题描述**: BS模型假设对数正态分布、无摩擦市场、连续交易、常数波动率，但代码中没有任何机制检测这些假设何时失效。在极端行情（如涨跌停、闪崩）中，BS假设全部失效，但系统仍使用BS Greeks进行风控决策，可能导致风险低估。

  缺失的检测维度：
  1. 无厚尾检测（收益率峰度>3时标记非正态）
  2. 无波动率聚类检测（GARCH效应显著时标记非恒定波动率）
  3. 无跳跃风险检测（日内价格跳变>3σ时标记非连续路径）

### 问题 R17-03 [P2] IV牛顿迭代初始猜测值固定且无二分法兜底

- **文件**: `greeks_calculator.py`
- **行号**: 167-224
- **影响系统**: 生产系统
- **问题描述**: `IVCalculator.implied_volatility`的初始猜测值硬编码为0.2，对于深度虚值/实值期权，0.2可能远离真实IV，导致牛顿迭代收敛缓慢甚至发散。虽然vega过小时返回initial_guess（行208-211），但缺少二分法兜底机制。

## 维度二：Greeks对冲理论一致性

### 问题 R17-04 [P1] 组合Greeks未考虑交叉Gamma效应

- **文件**: `greeks_calculator.py`
- **行号**: 1062-1102
- **影响系统**: 生产系统
- **问题描述**: `get_portfolio_greeks`方法简单线性加权求和各持仓的Greeks，未考虑交叉Gamma效应。当组合中同时持有不同行权价/到期日的期权时，交叉Gamma（∂²V/(∂Sᵢ∂Sⱼ)）可能显著影响对冲精度。

  ```python
  # 行1091-1094: 简单加权累加，忽略交叉项
  portfolio['delta'] += greeks['delta'] * multiplier * size
  portfolio['gamma'] += greeks['gamma'] * multiplier * size
  portfolio['theta'] += greeks['theta'] * multiplier * size
  portfolio['vega'] += greeks['vega'] * multiplier * size
  ```

  对于同一标的的不同期权，线性叠加在理论上正确（同一标的的Greeks可加），但跨标的组合的交叉Gamma被忽略。

### 问题 R17-05 [P2] 无离散对冲调整

- **文件**: `greeks_calculator.py`、`risk_service.py`
- **行号**: 全文件
- **影响系统**: 生产系统
- **问题描述**: BS模型假设连续对冲，但实际对冲是离散的（每N秒/每tick）。离散对冲产生的追踪误差（tracking error）未被量化或补偿。Whalley-Wilmott近似给出最优对冲带宽 ΔH ≈ (3/2 × κ × e^(-rT) × |Γ| × S² / λ)^(1/3)，其中κ为交易成本率，λ为风险厌恶系数。当前系统未实现此调整。

### 问题 R17-06 [P2] 二叉树Greeks的有限差分精度问题

- **文件**: `greeks_calculator.py`
- **行号**: 294-324
- **影响系统**: 生产系统
- **问题描述**: `BinomialTreePricer.price_with_greeks`使用有限差分法计算Greeks，其中theta的步长使用1/365（日历日），但二叉树模型的T参数是以年为单位的时间。当T较小时（如到期前1天），`max(T - 1.0/365.0, 0.0)`可能直接归零，导致theta计算失败。

  ```python
  price_T = BinomialTreePricer.price(S, K, max(T - 1.0/365.0, 0.0), ...)
  theta = (price_T - price) / (1.0 / 365.0)  # 行315
  ```

## 维度三：Kelly公式实现正确性

### 问题 R17-07 [P1] Kelly公式未扣除交易成本

- **文件**: `mode_engine.py`
- **行号**: 573-577
- **影响系统**: 生产系统、测试系统
- **问题描述**: Kelly公式实现为 `kelly = (win_rate * win_loss_ratio - (1 - win_rate)) / win_loss_ratio`，数学形式正确，但未扣除交易成本。含交易成本的Kelly公式应为：

  f* = (p(b-c) - q) / (b-c)

  其中c为单位交易成本占赌注的比例。忽略交易成本会导致Kelly值系统性偏大，尤其在HFT策略中（交易频率高、单笔利润薄），偏差可能超过50%。

  ```python
  # 行573-577: 未考虑交易成本
  def kelly_fraction(win_rate: float, win_loss_ratio: float, fraction: float = 1.0) -> float:
      """计算Kelly仓位比例"""
      kelly = (win_rate * win_loss_ratio - (1 - win_rate)) / win_loss_ratio
      return max(0.0, kelly * fraction)
  ```

### 问题 R17-08 [P2] Kelly公式在多策略组合中未考虑策略相关性

- **文件**: `mode_engine.py`
- **行号**: 573-590
- **影响系统**: 生产系统
- **问题描述**: Kelly公式独立应用于每个策略，未考虑策略间的相关性。对于N个策略的组合，最优Kelly向量应通过求解以下优化问题获得：

  max Σᵢ fᵢμᵢ - 1/2 Σᵢⱼ fᵢfⱼσᵢⱼ

  其中σᵢⱼ为策略i和j的收益协方差。当前实现等价于假设策略间完全独立（σᵢⱼ=0, i≠j），这在策略间存在正相关时会高估总仓位，增加组合风险。

### 问题 R17-09 [P2] kelly_fraction参数命名歧义

- **文件**: `mode_engine.py`
- **行号**: 79, 573
- **影响系统**: 生产系统、测试系统
- **问题描述**: `ModeConfig.kelly_fraction`（行79）表示"使用Kelly的比例"（如0.33=1/3 Kelly），而函数`kelly_fraction()`（行573）计算的是"完整Kelly值"。两者同名但语义不同，容易导致混淆。建议将ModeConfig中的参数重命名为`kelly_usage_fraction`或`kelly_cap_fraction`。

## 维度四：风险调整收益计算

### 问题 R17-10 [P0] Sharpe比率未扣除无风险利率

- **文件**: `shadow_strategy_engine.py`（行552）、`task_scheduler.py`（行1941, 2124, 2282, 2457, 2579）、`advanced_validation.py`（行124）
- **影响系统**: 生产系统、测试系统、评判系统
- **问题描述**: 所有Sharpe比率计算均未扣除无风险利率，实际计算的是"信息比率"的简化版而非Sharpe比率。

  ```python
  # shadow_strategy_engine.py 行552
  sharpe = (mean_r / std_r) * math.sqrt(annualize_factor)  # ← 未减rf

  # task_scheduler.py 行1941
  sharpe = np.sqrt(252 * 240) * mean_r / std_r  # ← 未减rf

  # advanced_validation.py 行124
  sharpe = float(np.mean(returns) / np.std(returns) * np.sqrt(252))  # ← 未减rf
  ```

  正确的Sharpe比率公式为：SR = (E[R] - Rf) / σ(R)

  **影响评估**: 
  - 在当前低利率环境（Rf≈2%），年化收益5%的策略Sharpe被高估约0.13（0.02/0.15）
  - 系统中`greeks_calculator.py`已定义`DEFAULT_RISK_FREE_RATE = 0.02`，但未被任何Sharpe计算引用
  - 这导致策略评估系统性偏乐观，可能引发过度杠杆

### 问题 R17-11 [P1] Sortino比率使用均值而非MAR作为下行偏差阈值

- **文件**: `evaluation/cascade_judge.py`
- **行号**: 144-150
- **影响系统**: 评判系统
- **问题描述**: Sortino比率的下行偏差计算使用收益率均值（mean_ret）作为阈值，而非MAR（Minimum Acceptable Return）。

  ```python
  # 行146-150
  mean_ret = sum(returns) / len(returns)
  downside = [min(0, ret - mean_ret) ** 2 for ret in returns]  # ← 使用mean_ret而非MAR
  ```

  标准Sortino比率定义：Sortino = (E[R] - MAR) / σ_downside，其中σ_downside = √(E[min(0, R-MAR)²])

  使用mean_ret作为阈值会导致：
  1. Sortino比率永远大于Sharpe比率（因为下行偏差永远小于总标准差）
  2. 失去Sortino比率的原始语义——衡量"低于可接受最低收益"的风险
  3. 当策略收益率为正时，几乎所有收益都高于mean_ret，下行偏差被人为缩小

### 问题 R17-12 [P1] Sharpe比率年化因子不一致

- **文件**: `task_scheduler.py`（行1941）、`shadow_strategy_engine.py`（行552）、`cascade_judge.py`（行150）
- **影响系统**: 生产系统、测试系统、评判系统
- **问题描述**: 不同模块使用不同的Sharpe年化因子，导致同一策略在不同模块中Sharpe值不一致：

  | 模块 | 年化因子 | 假设频率 |
  |------|---------|---------|
  | task_scheduler.py | √(252×240) ≈ 245.9 | 分钟Bar |
  | shadow_strategy_engine.py | √252 ≈ 15.87 | 日频 |
  | cascade_judge.py | √252 ≈ 15.87 | 日频 |

  分钟Bar和日频的Sharpe值相差约15.5倍，可能导致评判系统使用日频Sharpe（较低）评估策略，而回测系统使用分钟Bar Sharpe（较高）优化参数，产生系统性偏差。

### 问题 R17-13 [P1] Calmar比率年化收益计算假设分钟Bar频率

- **文件**: `参数池/task_scheduler.py`
- **行号**: 1807-1814

  ```python
  # 行1810: 硬编码252*240=60480，假设分钟Bar
  ```

### 问题 R17-14 [P2] Sharpe比率使用总体标准差而非样本标准差

- **文件**: `shadow_strategy_engine.py`
- **行号**: 548-549
- **影响系统**: 生产系统
- **问题描述**: `_compute_sharpe`使用总体标准差（除以N）而非样本标准差（除以N-1），在样本量较小时会低估标准差，从而高估Sharpe。

  ```python
  # 行548-549: 使用总体方差（除以N）
  ```

## 维度五：期权定价模型一致性

### 问题 R17-15 [P1] Call-Put Parity校验使用PnL而非价格

- **文件**: `greeks_calculator.py`
- **行号**: 1268-1304
- **影响系统**: 生产系统、评判系统
- **问题描述**: `compute_call_put_parity_pnl`方法使用PnL（价格变化量）而非绝对价格来校验Call-Put Parity。标准Call-Put Parity关系为 C - P = S - K×exp(-rT)，应使用期权和标的的当前价格进行校验。

  ```python
  # 行1293-1295: 使用PnL而非价格
  call_pnl = call_price - (call_greeks.get('entry_price', call_price) if call_greeks else 0.0)
  put_pnl = put_price - (put_greeks.get('entry_price', put_price) if put_greeks else 0.0)

  ```

  问题：
  1. `call_pnl`和`put_pnl`是相对于entry_price的变化量，不是当前价格
  2. `underlying_pnl`直接赋值为theoretical_forward，语义混乱
  3. Parity校验应直接比较 |C_market - P_market - (S - K×exp(-rT))| 与阈值

## 维度六：波动率模型正确性

### 问题 R17-18 [P1] EWMA衰减因子α=0.05缺乏理论依据

- **文件**: `quant_core.py`
- **行号**: 219, 583
- **影响系统**: 生产系统
- **问题描述**: `IVSurfacePCA`和`VolatilityRegimeFilter`均使用`ewma_alpha=0.05`，但未提供理论依据。

  ```python
  # IVSurfacePCA 行219
  ewma_alpha: float = 0.05

  # VolatilityRegimeFilter 行583
  ewma_alpha: float = 0.05
  ```

  J.P. Morgan RiskMetrics框架推荐的EWMA衰减因子λ=0.94（对应α=1-λ=0.06），这是基于全球多资产数据集的实证最优值。当前α=0.05对应λ=0.95，比RiskMetrics更平滑，对近期波动率变化的响应更慢，可能延迟波动率regime切换的检测。

### 问题 R17-19 [P2] 已实现波动率计算使用简单均值而非指数加权

- **文件**: `quant_core.py`
- **行号**: 605
- **影响系统**: 生产系统
- **问题描述**: `VolatilityRegimeFilter`的已实现波动率使用简单均值计算（`self._rv_buffer.sum() / len(self._rv_buffer)`），而EWMA已实现波动率使用指数加权。两者混合使用可能导致regime判断不一致——简单均值RV已切换到HIGH regime，但EWMA RV仍处于NORMAL。

  ```python
  # 行605: 简单均值RV
  self._current_rv = math.sqrt(self._rv_buffer.sum() / len(self._rv_buffer))
  # 行607: EWMA RV
  self._ewma_rv = a * self._current_rv + (1 - a) * self._ewma_rv
  # 行615: regime判断使用EWMA RV
  new_regime = 0 if self._ewma_rv <= self._p25 else (2 if self._ewma_rv >= self._p75 else 1)
  ```

  建议：统一使用EWMA RV进行regime判断，或明确说明两者用途差异。

### 问题 R17-20 [P2] IV曲面PCA未显式建模波动率微笑/偏斜

- **文件**: `quant_core.py`
- **行号**: 202-319
- **影响系统**: 生产系统
- **问题描述**: `IVSurfacePCA`使用PCA分解IV曲面为Level/Slope/Curvature三个主成分，这是一种数据驱动的降维方法，但未显式建模波动率微笑（volatility smile）和偏斜（volatility skew）。

  在中国期权市场中，50ETF期权的波动率微笑呈现典型的"微笑"形态（虚值认沽IV高于虚值认购），而股指期权的波动率偏斜呈现"负偏"（低行权价IV高于高行权价）。PCA的Slope成分可能部分捕捉偏斜，但无法区分微笑和偏斜的不同形态。

### 问题 R17-21 [P2] ADF检验临界值表精度不足

- **文件**: `quant_core.py`
- **行号**: 647-651
- **影响系统**: 生产系统
- **问题描述**: `_ADF_CRITICAL_5PCT`临界值表仅包含11个样本量节点（25-500），使用线性插值。MacKinnon (1996)的精确临界值表包含更多节点和更高精度。对于样本量在节点之间的场景，线性插值可能导致ADF检验结论错误（尤其是边界情况）。

## 维度七：资金管理理论正确性

### 问题 R17-22 [P1] 11维度风险评分未考虑策略相关性对资金分配的影响

- **文件**: `risk_service.py`
- **行号**: 1240-1375
- **影响系统**: 生产系统
- **问题描述**: `compute_decision_score`的11维度风险评分中，D11（cross_correlation）仅度量策略间PnL的绝对相关性，未将相关性纳入资金分配决策。当多个策略高度正相关时，应降低总仓位以避免集中风险，但当前实现仅将相关性作为一个评分维度（权重0.08），未直接影响资金分配比例。

  理论正确做法：使用风险预算（risk budgeting）框架，根据策略间协方差矩阵分配资金，使每个策略的风险贡献相等（ERC模型）或与目标风险预算匹配。

### 问题 R17-23 [P2] 无等风险贡献（ERC）模型

- **文件**: `risk_service.py`、`mode_engine.py`
- **行号**: 全文件
- **影响系统**: 生产系统
- **问题描述**: 系统使用固定模式配置（small/medium/large）进行资金分配，未实现ERC（Equal Risk Contribution）模型。ERC模型要求每个策略对组合总风险的边际贡献相等，数学表达为：

  wᵢ × (Σw)ᵢ / √(w'Σw) = 常数, ∀i

  其中w为权重向量，Σ为协方差矩阵。ERC模型在理论上优于等权重分配（当策略波动率差异大时）和等波动率分配（当策略相关性差异大时）。

### 问题 R17-24 [P2] DrawdownManager未动态约束资金分配

- **文件**: `mode_engine.py`（DrawdownManager相关代码）
- **影响系统**: 生产系统
- **问题描述**: DrawdownManager跟踪回撤状态并触发动作（REDUCE_SIZE/HALT_NEW/FULL_STOP），但未将当前回撤深度直接映射到资金分配比例。理论正确做法是使用回撤约束下的动态资金分配：

  w(t) = w₀ × max(0, 1 - |DD(t)|/DD_max)

  其中DD_max为最大允许回撤。当前实现使用离散的动作（减仓/暂停/全停）而非连续的资金缩放，可能导致资金利用率不足或风险暴露过度。

### 问题 R17-25 [P2] 非对称风控乘数缺乏理论推导

- **文件**: `risk_service.py`
- **行号**: 175-177, 1211-1217
- **影响系统**: 生产系统
- **问题描述**: 非对称风控的盈利乘数1.5和亏损乘数0.7为硬编码值，缺乏理论推导。

  ```python
  # 行175-177
  self._asymmetric_profit_multiplier: float = 1.5
  self._asymmetric_loss_multiplier: float = 0.7
  ```

  理论上，非对称风控乘数应基于效用函数推导。例如，使用CRRA效用函数 u(W) = W^(1-γ)/(1-γ)，当风险厌恶系数γ=2时，盈利时的回撤容忍度应为亏损时的约1.5倍。当前硬编码值可能不适用于所有风险偏好。

## 维度八：统计检验正确性

### 问题 R17-26 [P1] Sharpe比率标准误近似过于粗糙

- **文件**: `shadow_strategy_engine.py`
- **行号**: 1179-1182
- **影响系统**: 生产系统、评判系统
- **问题描述**: Sharpe比率的标准误使用 `SE ≈ 1/√N` 的近似，这仅在Sharpe=0且收益独立同分布正态时成立。实际中，Sharpe>0且收益可能存在偏度和峰度，导致标准误被低估。

  ```python
  # 行1179-1180
  # Sharpe的标准误近似: SE ≈ sqrt(1/N) (简化假设)
  main_se = 1.0 / math.sqrt(master_n) if master_n > 0 else 1.0
  ```

  根据Lo (2002)和Bailey & Lopez de Prado (2014)，正确的标准误公式为：

  SE(SR) = √((1 + SR²/2 - skew×SR + (kurt-3)/4×SR²) / N)

  当SR=2、skew=-0.5、kurt=4时，正确SE约为1/√N的1.87倍，当前实现严重低估了Sharpe的不确定性。

### 问题 R17-27 [P2] 收益自相关性未在统计检验中考虑

- **文件**: `shadow_strategy_engine.py`、`advanced_validation.py`
- **行号**: 全文件
- **影响系统**: 生产系统、评判系统
- **问题描述**: 所有统计检验（Sharpe CI、Walk-Forward验证等）均假设收益独立同分布，未考虑自相关性。当收益存在正自相关时，有效样本量从N降低到N/(1+2×Σρₖ)，其中ρₖ为k阶自相关系数。

  期权策略的收益通常存在显著的自相关性（趋势跟踪策略尤为明显），忽略自相关会导致：
  1. Sharpe比率的置信区间过窄
  2. Walk-Forward验证的p值偏低（假阳性增加）
  3. 策略Alpha的统计显著性被高估

### 问题 R17-28 [P2] BH校正实现缺少单调性保证

- **文件**: `参数池/advanced_validation.py`
- **行号**: 72-85
- **影响系统**: 测试系统
- **问题描述**: Benjamini-Hochberg校正的实现中，步骤2（行81-84）的单调性强制使用了从后向前的传递，但初始adjusted值可能不满足单调性。标准BH算法要求先排序、再从最大p值开始向前强制单调递减。当前实现虽然最终结果正确，但中间步骤可能产生负值或>1的值。

  ```python
  # 行78-84
  for rank_i, (orig_i, p) in enumerate(indexed):
      bh_p = p * n / (rank_i + 1)  # ← 可能>1
      adjusted[orig_i] = min(1.0, bh_p)
  for rank_i in range(len(indexed) - 2, -1, -1):
      orig_i = indexed[rank_i][0]
      next_orig_i = indexed[rank_i + 1][0]
      adjusted[orig_i] = min(adjusted[orig_i], adjusted[next_orig_i])  # ← 单调性强制
  ```

  实现逻辑正确，但建议添加注释说明BH算法的两步过程。

### 问题 R17-29 [P2] 蒙特卡洛破产检验的i.i.d.假设被Block Bootstrap修正但未成为默认

- **文件**: `参数池/statistical_validation.py`

### 问题 R17-30 [P2] 反事实检验的Call-Put Parity校验阈值过松

- **文件**: `参数池/statistical_validation.py`
- **行号**: 57-89
- **影响系统**: 评判系统
- **问题描述**: Call-Put Parity校验使用10%偏差阈值（行83：`parity_threshold = max(abs(underlying_pnl) * 0.1, 1.0)`），且允许20%的打乱违反Parity（行89：`parity_violation_rate <= 0.20`）。
